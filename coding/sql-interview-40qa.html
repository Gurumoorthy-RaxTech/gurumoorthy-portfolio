Query Optimization

Questions 1–8 | Performance tuning, execution plans, bottlenecks
1

How do you optimize a SQL query that takes several minutes to execute?

▼

Query slow-ஆ run ஆகும்போது, இந்த steps follow பண்ணு:

  1. Execution Plan பாரு — SSMS-ல Ctrl+M → actual plan check பண்ணு. "Table Scan" இருந்தா index missing.
  2. SELECT * avoid பண்ணு — தேவையான columns மட்டும் select பண்ணு.
  3. WHERE clause-ல functions avoid பண்ணு — Index use ஆகாது.
  4. Proper indexes add பண்ணு — WHERE, JOIN, ORDER BY columns-ல.
  5. JOIN order optimize பண்ணு — Smaller result set முதல்ல.
  6. Subqueries → JOINs/CTEs ஆக மாத்து — Better performance.
  7. Statistics update பண்ணு — Outdated stats → wrong execution plan.
-- Bad (function on column = no index)
SELECT * FROM Students WHERE YEAR(AdmissionDate) = 2024

-- Good (sargable query = index can be used)
SELECT StudentName, RollNo FROM Students
WHERE AdmissionDate >= '2024-01-01' AND AdmissionDate < '2025-01-01'
💡 First action: Always check Execution Plan. It shows exactly where time is wasted.
2

How do you identify performance bottlenecks in SQL Server?

▼

Performance bottleneck identify பண்ண இந்த tools & DMVs use பண்ணு:

  • Activity Monitor — SSMS → Activity Monitor → CPU, waits, expensive queries live-ஆ பாரு
  • sys.dm_exec_query_stats — Most expensive queries DMV
  • sys.dm_os_wait_stats — Server எதுக்கு wait பண்றதுன்னு பாரு
  • SQL Server Profiler / Extended Events — Slow queries capture பண்ணு
  • Performance Dashboard Reports — SSMS → Reports → Standard Reports
-- Top 10 most expensive queries (CPU)
SELECT TOP 10
    total_worker_time/execution_count AS AvgCPU,
    execution_count,
    SUBSTRING(st.text, 1, 200) AS QueryText
FROM sys.dm_exec_query_stats qs
CROSS APPLY sys.dm_exec_sql_text(qs.sql_handle) st
ORDER BY AvgCPU DESC
💡 Wait stats-ல CXPACKET அதிகமா இருந்தா parallelism issue. PAGEIOLATCH_SH இருந்தா disk I/O issue.
3

What is an Execution Plan, and how do you analyze it?

▼

Execution Plan = SQL Server query-ஐ எப்படி execute பண்றதுன்னு show பண்ற visual diagram. SQL Server Query Optimizer தேர்ந்தெடுக்கற best path.

Types:

  • Estimated Plan (Ctrl+L) — Execute பண்ணாம plan மட்டும் பாரு
  • Actual Plan (Ctrl+M) — Execute பண்ணி real stats பாரு (accurate)

Analyze பண்றது எப்படி:

  • 🔴 Table Scan / Clustered Index Scan — Entire table scan → Index missing
  • 🟢 Index Seek — Specific rows find → Efficient
  • ⚠️ Key Lookup — Extra data fetch → Covering index தேவை
  • ⚠️ Hash Match / Sort — Costly operations → Optimize join/order
  • ❗ Thick arrows — Lots of rows moving → Bottleneck
  • ❗ Yellow warning icon — Statistics outdated or missing index
💡 Right-click any node → Properties → Cost % பாரு. Highest cost node-ஐ முதல்ல fix பண்ணு.
4

How do you troubleshoot slow-running queries in production?

▼

Step-by-step production troubleshooting:

  1. Identify the query — Activity Monitor or sp_who2
  2. Check wait types — sys.dm_os_waiting_tasks
  3. Get execution plan — sys.dm_exec_query_plan
  4. Check statistics — Outdated statistics → wrong plan
  5. Check blocking — sp_lock or sys.dm_exec_requests
  6. Parameter Sniffing check — Wrong cached plan use ஆகுதா?
  7. Index fragmentation check — DBCC SHOWCONTIG
-- Currently running slow queries
SELECT r.session_id, r.status, r.wait_type,
       r.total_elapsed_time/1000 AS elapsed_sec,
       SUBSTRING(t.text, 1, 200) AS QueryText
FROM sys.dm_exec_requests r
CROSS APPLY sys.dm_exec_sql_text(r.sql_handle) t
WHERE r.total_elapsed_time > 5000  -- 5 seconds+
ORDER BY r.total_elapsed_time DESC
⚠️ Production-ல DBCC FREEPROCCACHE பண்ணாதே — Plan cache clear ஆகும், performance worse ஆகும்.

What techniques do you use to optimize queries on large tables?

▼
  • Proper Indexing — Clustered + Non-clustered indexes on filter/join columns
  • Table Partitioning — Large tables-ஐ partition key-ல divide பண்ணு (e.g., by year)
  • Pagination — OFFSET-FETCH use பண்ணி batch-ஆ data load பண்ணு
  • Avoid SELECT * — Only needed columns
  • Filtered Indexes — WHERE condition உள்ள index (smaller, faster)
  • Columnstore Indexes — Analytics/reporting queries-க்கு
  • Archiving — Old data separate archive table-க்கு move பண்ணு
  • Query Hints — NOLOCK (dirty reads ok-ஆ இருந்தா), FORCESEEK
-- Pagination example
SELECT StudentName, RollNo, Class
FROM Students
ORDER BY StudentName
OFFSET 0 ROWS FETCH NEXT 100 ROWS ONLY

-- Filtered index (only active students)
CREATE INDEX IX_Students_Active
ON Students(StudentName) WHERE IsActive = 1
💡 Millions of rows → Columnstore index use பண்ணு → 10x faster for analytics.

How do you reduce IO and CPU consumption in SQL Server?

▼

IO Reduction:

  • Proper indexes → Table scan-ஐ Seek-ஆ convert பண்ணு
  • Covering indexes → Key lookup eliminate பண்ணு
  • Read only needed columns (avoid SELECT *)
  • Buffer Pool — Hot data memory-ல cache ஆகும்
  • SSD storage use பண்ணு tempdb-க்கு

CPU Reduction:

  • Set-based operations → Cursor avoid பண்ணு
  • MAXDOP settings — Parallelism control பண்ணு
  • Stored procedures → Compilation overhead குறையும்
  • Avoid scalar functions in SELECT/WHERE
  • Temp tables instead of repeated subqueries
-- Avoid scalar function (CPU hog)
-- Bad: SELECT dbo.GetStudentGrade(StudentId) FROM Students
-- Good: JOIN Grades g ON g.StudentId = s.StudentId
✅ Tip: MAXDOP = 0 (unlimited parallelism) avoid பண்ணு. OLTP servers-க்கு MAXDOP = 1 or 4.

How do you optimize queries involving millions of records?

▼
  • Batch Processing — Millions-ஐ ஒரே time-ல process பண்ணாம batch-ஆ
  • Partitioned Tables — Partition elimination use ஆகும்
  • Columnstore Indexes — Aggregation queries-க்கு blazing fast
  • Materialized results — Frequently needed results pre-compute பண்ணு
  • Indexed Views — Complex join results pre-aggregate பண்ணு
  • Statistics maintenance — Regular UPDATE STATISTICS
-- Batch delete (don't delete millions at once)
WHILE 1=1
BEGIN
    DELETE TOP(10000) FROM ArchivedLogs
    WHERE LogDate < '2023-01-01'
    IF @@ROWCOUNT = 0 BREAK
    WAITFOR DELAY '00:00:01'  -- 1 sec pause
END
💡 Million rows delete → Always batch-ஆ பண்ணு. Log file overflow avoid ஆகும்.

What is the difference between Clustered and Non-Clustered Indexes?

▼
FeatureClustered IndexNon-Clustered Index

Count per tableOnly 1Up to 999
Speed (Range scan)FastestSlower (extra lookup)

-- Clustered index (Primary Key = auto clustered)
CREATE TABLE Students (
    StudentId INT PRIMARY KEY,  -- Clustered by default
    StudentName VARCHAR(100)
)

-- Non-Clustered index
CREATE NONCLUSTERED INDEX IX_Students_Name
ON Students(StudentName)
💡 Analogy: Clustered = Library books sorted by call number (physically). Non-Clustered = Index card pointing to book location.
10

When should you create a Composite Index?

▼

Composite Index = Multiple columns-ஐ ஒரே index-ல include பண்றது.

Create பண்ண வேண்டிய situations:

  • WHERE clause-ல multiple columns always together use ஆகும்போது
  • One column-ல selectivity குறைவாக இருந்தா (e.g., Gender alone → poor index)
  • JOIN + WHERE combination-ல
-- Query: WHERE ClassId = 10 AND Section = 'A'
CREATE INDEX IX_Students_ClassSection
ON Students(ClassId, Section)  -- Composite!

-- Column ORDER matters!
-- Most selective column FIRST
-- ClassId (fewer values) before Section
⚠️ Column order matters! Leading column filter இல்லாம query போனா index use ஆகாது.
💡 Rule: (ClassId, Section) index — WHERE ClassId=10 ✅, WHERE ClassId=10 AND Section='A' ✅, WHERE Section='A' only ❌
11

How do you determine whether an index is actually being used?

▼ 
-- Index usage stats (since last SQL Server restart)
SELECT
    i.name AS IndexName,
    ius.user_seeks, ius.user_scans, ius.user_lookups,
    ius.user_updates,
    ius.last_user_seek
FROM sys.indexes i
LEFT JOIN sys.dm_db_index_usage_stats ius
    ON i.object_id = ius.object_id AND i.index_id = ius.index_id
WHERE i.object_id = OBJECT_ID('Students')
  • user_seeks = 0, user_scans = 0 → Index never used → Drop பண்ணலாம்
  • user_updates high → Index maintain-ஆக overhead → Worth it-ஆ check பண்ணு
  • Execution Plan → Index Seek icon இருந்தா use ஆகுது
💡 Unused indexes-ஐ drop பண்ணு — INSERT/UPDATE/DELETE slowdown ஆகும் otherwise.
12

What are Covering Indexes and Included Columns?

▼

Covering Index = Query-க்கு தேவையான எல்லா columns-உம் index-லயே இருக்கும். Table-க்கு போக வேண்டாம் (Key Lookup இல்ல).

Included Columns = Index key-ல இல்லாம, leaf level-ல extra columns add பண்றது.

-- Query needs: WHERE StudentId = 5, returns Name + Email
-- Without covering index: Index Seek + Key Lookup (2 operations)
-- With covering index: Index Seek only (1 operation) ✅

CREATE NONCLUSTERED INDEX IX_Students_Covering
ON Students(StudentId)
INCLUDE (StudentName, Email)  -- Included columns!
✅ Key Lookup eliminate பண்றதுதான் biggest performance win. Execution plan-ல Key Lookup இருந்தா Covering index add பண்ணு.
13

What are Index Fragmentation and Index Rebuild/Reorganize?

▼

Fragmentation = INSERT/UPDATE/DELETE operations-ஓட காரணமா index pages out-of-order ஆகுது → Slower reads.

Fragmentation %ActionLocking
< 10%Do nothing—
10% – 30%REORGANIZEOnline (no lock)
> 30%REBUILDOffline (with ONLINE=ON option)
 
-- Check fragmentation
SELECT index_id, avg_fragmentation_in_percent
FROM sys.dm_db_index_physical_stats
    (DB_ID(), OBJECT_ID('Students'), NULL, NULL, 'LIMITED')

-- Reorganize (online)
ALTER INDEX IX_Students_Name ON Students REORGANIZE

-- Rebuild (offline or online)
ALTER INDEX IX_Students_Name ON Students REBUILD WITH (ONLINE = ON)
💡 Weekly maintenance job: Reorganize + Update Statistics schedule பண்ணு. Monthly full Rebuild.
14

How many indexes are too many?

▼

No hard rule, but context-ல பாரு:

  • OLTP systems — 3-5 indexes per table. More = INSERT/UPDATE slow ஆகும்
  • OLAP/Reporting — More indexes ok. Reads > Writes
  • General rule — Every write operation, ALL indexes update ஆகும்

Too many indexes signs:

  • INSERT/UPDATE queries slow ஆகும்
  • Index maintenance time அதிகமாகும்
  • Disk space waste
  • Query Optimizer wrong plan choose பண்ணலாம்
⚠️ 10 indexes per table-க்கு மேல போனா red flag. Unused indexes drop பண்ணு.
💡 Rule of thumb: "Every index makes SELECT faster, but INSERT/UPDATE/DELETE slower."
15

Why can indexes sometimes reduce performance?

▼
  • Write overhead — INSERT/UPDATE/DELETE → All indexes update ஆகும் → Slower writes
  • Wrong index chosen — Optimizer bad index select பண்ணலாம் → Table scan faster-ஆ இருந்தாலும்
  • Index fragmentation — Fragmented index → Poor performance
  • Statistics outdated — Wrong cardinality estimate → Bad plan
  • Index on low-selectivity column — e.g., Gender column (only M/F) → Index useless, adds overhead
  • Memory pressure — Too many indexes → Buffer pool pressure
✅ Solution: Regularly audit indexes using sys.dm_db_index_usage_stats. Drop unused indexes.
16

How do you design indexing strategies for high-volume transactional systems?

▼
  • Clustered index on sequential key — Identity or sequential GUID (avoid random GUIDs → fragmentation)
  • Minimal non-clustered indexes — Only on frequently queried columns
  • Covering indexes — Eliminate Key Lookups
  • Filtered indexes — Only active/recent records
  • Avoid over-indexing — Write performance critical in OLTP
  • Fill Factor — 70-80% for high-insert tables (leaves room for growth)
  • Partitioning — Large tables partition-ஆ split பண்ணி index smaller ஆகும்
-- Fill factor example
CREATE INDEX IX_Orders_Date
ON Orders(OrderDate)
WITH (FILLFACTOR = 75)  -- 75% full, 25% room for inserts
💡 ZenCampus-ல Students table busy ஆ இருந்தா — StudentId (clustered), AdmissionDate (NC), ClassId+Section (composite) மட்டும் போதும்.

How do you prevent Deadlocks?

▼

Deadlock = Two transactions ஒண்ணோட ஒண்ணு resource-க்கு wait பண்றது → Neither can proceed.

Prevention strategies:

  • Same order table access — எல்லா transactions-லயும் same table order-ல access பண்ணு
  • Short transactions — Transaction-ல minimum work பண்ணு, quickly commit பண்ணு
  • SNAPSHOT isolation — Readers don't block writers
  • Index optimization — Fewer locks needed with good indexes
  • Avoid user interaction in transaction — Transaction open-ஆ இருக்கும்போது user input wait பண்ணாதே
  • SELECT before UPDATE — Needed data select பண்ணி validate பண்ணி then update
⚠️ SQL Server automatically detects deadlocks and kills the "victim" transaction (lower priority).
20

How do you identify and resolve Deadlocks in production?

▼

Identify:

  • SQL Server Error Log — "Deadlock" error 1205
  • Extended Events — deadlock_graph event capture பண்ணு
  • Trace Flag 1222 → Detailed deadlock info to error log
  • System Health Session → SSMS → Management → Extended Events
-- Enable deadlock trace
DBCC TRACEON(1222, -1)

-- Read system_health deadlock graph
SELECT
    xdr.value('(event/@timestamp)[1]', 'datetime2') AS DeadlockTime,
    xdr.query('.') AS DeadlockGraph
FROM (
    SELECT CAST(target_data AS XML) AS TargetData
    FROM sys.dm_xe_session_targets
    WHERE target_name = 'ring_buffer'
) AS Data
CROSS APPLY TargetData.nodes('//RingBufferTarget/event[@name="xml_deadlock_report"]') AS XEventData(xdr)
💡 Deadlock graph → SSMS-ல open பண்ணினா visual diagram show ஆகும். Which tables, which processes involved-ன்னு clear-ஆ தெரியும்.

What is Row Versioning?

▼

Row Versioning = Row update ஆகும்போது old version-ஐ tempdb-ல save பண்ணுவது. Readers old version read பண்ணலாம் — Writers block ஆகாது.

Used by:

  • SNAPSHOT isolation level
  • READ COMMITTED SNAPSHOT isolation (RCSI)
-- Enable RCSI (recommended for most apps)
ALTER DATABASE ZenCampusDB
SET READ_COMMITTED_SNAPSHOT ON

-- Enable Snapshot isolation
ALTER DATABASE ZenCampusDB
SET ALLOW_SNAPSHOT_ISOLATION ON
✅ RCSI enable பண்ணினா — Blocking dramatically குறையும். Azure SQL-ல default-ஆவே ON.
⚠️ tempdb usage அதிகமாகும். tempdb-ஐ fast disk-ல வையுங்க.

How do you handle concurrent updates from multiple users?

▼

Two approaches:

  • Pessimistic Concurrency — Lock the row before updating (UPDLOCK hint)
  • Optimistic Concurrency — rowversion/timestamp column use பண்ணி conflict detect பண்ணு
-- Optimistic concurrency with rowversion
CREATE TABLE Students (
    StudentId INT PRIMARY KEY,
    StudentName VARCHAR(100),
    RowVer ROWVERSION  -- Auto-updated on every change
)

-- Update with version check
UPDATE Students
SET StudentName = 'Updated Name'
WHERE StudentId = 1 AND RowVer = @OriginalVersion

IF @@ROWCOUNT = 0
    RAISERROR('Record was modified by another user', 16, 1)
💡 ASP.NET Core EF Core-ல [Timestamp] attribute → automatic optimistic concurrency.

Database Design

25

How do you design a database for scalability?

▼
  • Proper normalization — Data redundancy குறைக்கணும் (3NF)
  • Strategic indexing — Over/under indexing avoid
  • Table partitioning — Large tables horizontal split
  • Archiving strategy — Old data archive tables-க்கு move
  • Read replicas — Read-heavy workload → Secondary replica
  • Caching — Redis/In-memory cache for frequent reads
  • Connection pooling — DB connections efficiently manage
  • Avoid EAV pattern — Entity-Attribute-Value → performance nightmare
  • Use appropriate data types — INT not BIGINT unnecessarily, VARCHAR not NVARCHAR
💡 ZenCampus-ல Students table — Partition by AcademicYear. Old students archive table-க்கு move.

When should you normalize or denormalize a database?

▼

Normalize when:

  • OLTP systems (transactions heavy)
  • Data integrity critical
  • Storage space concern
  • Data changes frequently

Denormalize when:

  • Read performance critical (reporting, analytics)
  • Complex JOINs causing slowness
  • OLAP/Data warehouse
  • Caching/pre-computed results store பண்ண
💡 "Normalize till it hurts, denormalize till it works" — Start normalized, denormalize specific bottlenecks.
⚠️ Denormalize carefully — data update anomalies risk. Application layer-ல sync manage பண்ணணும்.

How do you partition very large tables?

▼

Table Partitioning = Large table-ஐ logical sections-ஆ divide பண்றது (physically separate filegroups-ல).

-- Step 1: Partition Function (how to split)
CREATE PARTITION FUNCTION PF_Year(INT)
AS RANGE RIGHT FOR VALUES (2022, 2023, 2024, 2025)

-- Step 2: Partition Scheme (where to store)
CREATE PARTITION SCHEME PS_Year
AS PARTITION PF_Year
TO (FG_2021, FG_2022, FG_2023, FG_2024, FG_2025)

-- Step 3: Create partitioned table
CREATE TABLE StudentAttendance (
    AttId INT, StudentId INT,
    AttYear INT, AttDate DATE
) ON PS_Year(AttYear)  -- Partition by AttYear

Benefits:

  • Partition elimination — Query only relevant partition scan
  • Faster archiving — Old partition switch out
  • Parallel query across partitions
💡 ZenCampus-ல Attendance table year-ஆ partition பண்ணினா current year data மட்டும் fast ஆ query ஆகும்.
30

How do you archive historical data without affecting performance?

▼
  • Archive tables — Separate table: Students_Archive
  • Partition switching — Old partition → Archive table (metadata only, instant)
  • Batch archiving — Off-peak hours-ல batch-ஆ move பண்ணு
  • Temporal Tables — SQL Server 2016+ → Automatic history tracking
  • Stretch Database — Azure-க்கு cold data automatically move
-- Temporal Table (auto-archives history)
CREATE TABLE Students (
    StudentId INT PRIMARY KEY,
    StudentName VARCHAR(100),
    ValidFrom DATETIME2 GENERATED ALWAYS AS ROW START,
    ValidTo   DATETIME2 GENERATED ALWAYS AS ROW END,
    PERIOD FOR SYSTEM_TIME (ValidFrom, ValidTo)
) WITH (SYSTEM_VERSIONING = ON)

-- Query historical data
SELECT * FROM Students
FOR SYSTEM_TIME AS OF '2024-01-01'
✅ Temporal Tables = Built-in time travel. No custom archiving code needed.

31

What are Common Table Expressions (CTEs), and when do you use them?

▼

CTE = Named temporary result set, query duration-க்கு மட்டும் exist ஆகும். WITH keyword use பண்ணி define பண்றோம்.

Use when:

  • Complex query-ஐ readable-ஆ break பண்ண
  • Recursive queries (hierarchy/tree structures)
  • Same subquery multiple times reference பண்ண
-- Simple CTE
WITH TopStudents AS (
    SELECT StudentId, StudentName, TotalMarks
    FROM Students
    WHERE TotalMarks > 90
)
SELECT s.*, c.ClassName
FROM TopStudents s
JOIN Classes c ON s.ClassId = c.ClassId

-- Recursive CTE (org hierarchy)
WITH OrgHierarchy AS (
    SELECT EmpId, EmpName, ManagerId, 0 AS Level
    FROM Employees WHERE ManagerId IS NULL
    UNION ALL
    SELECT e.EmpId, e.EmpName, e.ManagerId, h.Level+1
    FROM Employees e
    JOIN OrgHierarchy h ON e.ManagerId = h.EmpId
)
SELECT * FROM OrgHierarchy

What are Window Functions, and where have you used them?

▼

Window Functions = GROUP BY-ஓட summary result-க்கு பதிலா, individual row-ஐ keep பண்ணி aggregation பண்றது. OVER() clause use பண்றோம்.

-- Running total (each row + cumulative)
SELECT
    StudentName,
    FeeAmount,
    SUM(FeeAmount) OVER(ORDER BY PaymentDate) AS RunningTotal,
    AVG(FeeAmount) OVER(PARTITION BY ClassId)   AS ClassAvgFee,
    ROW_NUMBER()   OVER(ORDER BY TotalMarks DESC) AS Rank
FROM Students

Common Window Functions:

  • ROW_NUMBER(), RANK(), DENSE_RANK(), NTILE()
  • SUM(), AVG(), COUNT(), MIN(), MAX() OVER()
  • LAG(), LEAD() — Previous/Next row values
  • FIRST_VALUE(), LAST_VALUE()
💡 Real use: Student fee running total, class rank, previous month comparison — ZenCampus reports-ல அதிகமா use ஆகும்.

What is the difference between CROSS APPLY and OUTER APPLY?

▼

APPLY operator = Table-valued function-ஐ each row-க்கும் apply பண்றது. Correlated subquery மாதிரி.

APPLY TypeBehaviourLike
CROSS APPLYNo result return பண்ண rows excludeINNER JOIN
OUTER APPLYNo result ஆனாலும் NULL-ஆ includeLEFT JOIN
 
-- Get top 3 marks per class
SELECT c.ClassName, s.StudentName, s.Marks
FROM Classes c
CROSS APPLY (
    SELECT TOP 3 StudentName, Marks
    FROM Students
    WHERE ClassId = c.ClassId
    ORDER BY Marks DESC
) s
-- Classes with no students won't appear (CROSS APPLY)
-- Use OUTER APPLY to include empty classes with NULLs
💡 "Top N per group" pattern-க்கு APPLY is perfect. ROW_NUMBER() alternative but APPLY cleaner.

How do you optimize Stored Procedures?

▼
  • SET NOCOUNT ON — Unnecessary rowcount messages stop → reduces network traffic
  • Avoid SELECT * — Specific columns only
  • Parameter Sniffing fix — OPTION(RECOMPILE) or local variables
  • Schema prefix — dbo.ProcName not just ProcName
  • Proper error handling — TRY/CATCH
  • Avoid dynamic SQL when possible (SQL injection risk + plan cache pollution)
  • Break large SPs — Single responsibility principle
-- Parameter sniffing fix
CREATE PROCEDURE dbo.GetStudents @ClassId INT
AS
BEGIN
    SET NOCOUNT ON  -- Always add this!
    DECLARE @LocalClassId INT = @ClassId  -- Local var = sniff fix
    SELECT StudentName, RollNo
    FROM Students
    WHERE ClassId = @LocalClassId
END
✅ SET NOCOUNT ON = Every SP-ல add பண்ணு. Slight performance improvement + cleaner output.

A query that normally runs in 2 seconds suddenly takes 2 minutes. How would you troubleshoot it?

▼

Systematic troubleshooting steps:

  1. Is it blocked? — sp_who2 or sys.dm_exec_requests → blocking_session_id check
  2. Execution plan changed? — Query Store → Compare recent plan vs historical plan
  3. Statistics outdated? — UPDATE STATISTICS → Re-run query
  4. Parameter sniffing? — DBCC FREEPROCCACHE (dev only!) → Re-run
  5. Data volume changed? — Table-ல data suddenly much more?
  6. Index fragmentation? — sys.dm_db_index_physical_stats
  7. Server resources? — CPU, Memory, Disk I/O check
  8. Tempdb pressure? — Sort/hash spills to disk?
-- Quick check: Is it blocking?
SELECT session_id, blocking_session_id, wait_type,
       total_elapsed_time, SUBSTRING(t.text,1,100) AS Query
FROM sys.dm_exec_requests r
CROSS APPLY sys.dm_exec_sql_text(r.sql_handle) t
WHERE total_elapsed_time > 10000

-- Fix: Force recompile
EXEC dbo.SlowProc @Param WITH RECOMPILE
💡 Most common cause: Parameter sniffing or statistics outdated. Query Store-ல forced plan → quick fix for production.

How would you migrate a database with minimal downtime?

▼

Minimal downtime migration strategies:

  1. Log Shipping — Source DB logs continuously ship to target → Cutover fast
  2. Database Mirroring / Always On — Real-time sync → Failover seconds
  3. Transactional Replication — Changes replicate in near-realtime
  4. Backup/Restore with tail-log — Small downtime window
  5. Azure Database Migration Service — Online migration to Azure SQL

Steps for minimal downtime:

  • Phase 1: Offline backup → Restore on target (hours ahead)
  • Phase 2: Transaction log backup + restore (differential catch-up)
  • Phase 3: Brief app downtime → Final tail-log backup → Restore → Cutover
  • Phase 4: Connection string update → App back online
✅ Goal: Downtime in minutes, not hours. Always test full migration on staging first.

How do you investigate CPU spikes caused by SQL Server?

▼
  1. Task Manager / Resource Monitor — sqlservr.exe CPU % confirm
  2. Activity Monitor — Recent Expensive Queries tab
  3. DMV query — Top CPU consuming queries
  4. Extended Events — CPU > threshold captures
  5. Check for: Missing indexes, Table scans, Compilation spikes, Parallelism abuse
-- Top CPU queries right now
SELECT TOP 5
    qs.total_worker_time/1000 AS TotalCPU_ms,
    qs.execution_count,
    qs.total_worker_time/qs.execution_count/1000 AS AvgCPU_ms,
    SUBSTRING(t.text, 1, 200) AS QueryText
FROM sys.dm_exec_query_stats qs
CROSS APPLY sys.dm_exec_sql_text(qs.sql_handle) t
ORDER BY qs.total_worker_time DESC

-- Check plan cache compilation spikes
SELECT sql_compilations_per_sec, batch_requests_per_sec
FROM sys.dm_os_performance_counters
WHERE counter_name IN ('SQL Compilations/sec', 'Batch Requests/sec')
⚠️ Compilations/Batch ratio > 10% → Plan cache issues. Ad-hoc queries parameterize பண்ணு.
