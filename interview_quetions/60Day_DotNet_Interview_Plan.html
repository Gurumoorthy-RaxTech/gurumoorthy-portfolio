Day 1-2
LINQ Deep Dive + Performance
🎬 Vikram (2022): Vikram — different identities, different roles. LINQ-um same — oru data-ku Where, Select, GroupBy, Join — different operations, oru chain-la!
Deferred Execution IEnumerable vs IQueryable ToList() trap Any() vs Count() SelectMany
🏫 ZenCampus Example:
Student attendance filter panna:
students.Where(s => s.IsActive).Select(s => new { s.Name, s.Class })
DB-la run aagum vs memory-la — ivvavvu difference!
💡 Interview Trick: "IEnumerable vs IQueryable difference?" — IEnumerable = memory-la filter, IQueryable = DB-la filter. ZenCampus-la StudentService-la ithai use panrom nu sollu!
❓ Interview Q:
Q: What is deferred execution in LINQ?
A: Query create panna execute aagaadu. .ToList() or foreach call panna execute aagum. Ithunale unnecessary DB calls avoid pannalaam.
Day 3-4
Async/Await + Task Parallel Library
🎬 Enthiran (2010): Chitti oru nanosecond-la 1000 calculations panuvan — parallel processing! Task.WhenAll() — multiple APIs simultaneously call panna, Chitti maari!
async/await Task.WhenAll Task.WhenAny CancellationToken ConfigureAwait(false) Deadlock
🏫 ZenCampus Example:
Student details load panna — simultaneously student info + attendance + fees fetch panna:

💡 Deadlock Danger: .Result or .Wait() use pannaa — UI thread block aagum. Always await use pannu. ZenCampus-la Controller-la ithaye pattern follow panrom.
❓ Interview Q:
Q: ConfigureAwait(false) yen use panuvu?
A: Library code-la original thread-ku return aaga vendam — performance improve aagum. UI project-la vendaam, Web API-la always use pannu.

Day 10-12
Query Optimization + Execution Plans
🎬 Ratsasan (2018): Detective methodically clues collect panruvan — slow queries-um same, execution plan-la Table Scan vs Index Seek — serial killeraa find panra maari!
Table Scan Index Seek Covering Index Query Cost SET STATISTICS TIME N+1 Problem
🏫 ZenCampus Example:
Student list load — N+1 problem: oru loop-la each student-ku separate DB call. Fix: JOIN use panni single query-la fetch pannu. ZenCampus StudentService-la ithai already fix pannirukkom.
❓ Interview Q:
Q: What is the N+1 problem and how to fix it?
A: 1 query returns N records, then N additional queries run. Fix: Use JOIN or include navigation properties. In ZenCampus, we fixed this in attendance reporting.

SOLID Principles
🎬 Magadheera (2009): Every warrior has ONE purpose — S = Single Responsibility. Bairavan kodukkaadaan — O = Open/Closed. Character never changes core identity — L = Liskov!
S - Single Responsibility O - Open/Closed L - Liskov I - Interface Segregation D - Dependency Inversion
🏫 ZenCampus Example:

O: New fee type add panna existing code change vendam — new class add pannuvu
D: Controller depends on IStudentService (interface), not StudentService (concrete)
❓ Interview Q:
Q: Give a real example of Open/Closed Principle.
A: ZenCampus-la fee calculation — Term fee, Bus fee, Hostel fee — each one separate class implements IFeeCalculator. New fee type add panna existing code touch vendam!

Day 31-33
CQRS + MediatR Pattern
🎬 Thalapathy (1991): Surya (write) = commands execute pannuvan. Dev (read) = information gather pannuvan. CQRS = Commands (write) and Queries (read) separated — same maari!
CQRS Command Query MediatR IRequest IRequestHandler Pipeline Behavior
🏫 ZenCampus Example:
Command: CreateStudentCommand → CreateStudentHandler
Query: GetStudentQuery → GetStudentHandler
Controller sirf mediator.Send(command) — handler veruppu!
❓ Interview Q:
Q: Why use CQRS?
A: Read and write different scales, optimizations needed. Read = denormalized views, Write = validated commands. In large systems like ZenCampus multi-school, reporting queries separate DB use panalaam.
