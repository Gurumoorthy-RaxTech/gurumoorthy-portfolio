STAR Example 1 — Performance Problem

SituationZenCampus fee collection page was loading in 8 seconds for schools with 2,000+ students. Teachers were complaining during peak morning hours when 200 teachers logged in simultaneously.

TaskI was responsible for identifying and fixing the performance bottleneck without changing the UI or breaking existing functionality.

ActionI used Application Insights to find the slow query — a N+1 problem loading 2,000 students with their fee records one by one (2,000 separate SQL queries). I replaced it with a single Include() query with proper pagination, added an IDistributedCache (Redis) layer for fee structures that rarely change, and created a covering index on SchoolId + ClassId on the database.

ResultPage load time dropped from 8 seconds to 0.4 seconds (95% improvement). Teacher complaints dropped to zero. Database load reduced by 80% during peak hours. Implemented the same pattern across 3 other slow pages using the same cache-aside approach.

STAR Example 2 — Architecture Decision

SituationZenCampus was sending attendance SMS directly in the HTTP request. During 6 PM bulk SMS (50,000 parents), requests were timing out and teachers were getting errors saying "SMS failed" even though attendance was saved.

TaskRedesign the SMS flow to be reliable and non-blocking without changing the teacher's experience.

ActionI introduced Hangfire with SQL Server backing. Attendance save completes → BackgroundJob.Enqueue() enqueues SMS (5ms). HTTP request returns success immediately. Hangfire workers send SMS asynchronously with automatic retry on failure. Added the Outbox pattern to guarantee SMS is always enqueued even if the app restarts mid-request.

ResultZero teacher-visible SMS errors. Hangfire retries successfully when Msg91 SMS provider has transient failures. Throughput improved from 500 SMS/minute (synchronous) to 10,000 SMS/minute (parallel workers). System now reliably delivers 50,000 SMS within 8 minutes of attendance submission.

STAR Example 3 — Security Fix

SituationDuring a code review, I found that ZenCampus API endpoints were missing school-level authorization. A teacher logged into School A could theoretically access School B's student data by changing the schoolId parameter.

TaskFix the authorization gap without breaking any existing functionality.

ActionI implemented a CurrentSchoolContext service that extracts SchoolId from the JWT claims. Added a global EF Core query filter: HasQueryFilter(s => s.SchoolId == _currentSchool.SchoolId). All queries automatically filter by the authenticated school. Added integration tests verifying cross-school data is blocked. Deployed to production using staging slot with smoke tests before swap.

ResultSecurity gap closed across all entities with minimal code change. Zero cross-school data access possible even with manipulated API parameters. Security audit passed. Pattern documented for new developers.

🎯 10 Final Interview Scenarios

1
How do you explain Clean Architecture to a non-technical interviewer?Communication

"Imagine ZenCampus as an onion. The heart is pure business rules — like 'a student can't have duplicate admission numbers.' That's the Domain layer. Next ring: Application — what the system CAN DO (admit student, collect fee). Outer rings: how it stores data (SQL Server) and how it talks to users (Web API). The rule is: outer rings depend on inner rings, never the reverse. We can swap SQL Server for PostgreSQL without touching business rules. This makes the system testable and maintainable."

2
Describe a time you disagreed with a team decision.STAR

Situation: Team decided to put connection strings in Docker image environment variables baked into docker-compose.yml committed to git. I disagreed on security grounds. Task: Convince the team to change approach without creating conflict. Action: I researched Azure Key Vault, showed how DefaultAzureCredential works with Managed Identity, created a branch demonstrating the change with documentation. I shared how similar leaks caused production incidents at other companies. Result: Team adopted Key Vault approach. No stored credentials in git. The pattern now applies to all secrets in the project.

3
How do you stay updated with .NET and cloud technology?Growth

"I follow a structured approach: Microsoft's .NET blog and Azure Blog for official announcements. Nick Chapsas on YouTube for advanced .NET patterns. Azure certification learning paths. I apply everything to ZenCampus immediately — not just theory. When .NET 8 released Frozen Collections and output caching improvements, I implemented them in our production feed endpoints. I also follow system design discussions on engineering blogs from companies like Netflix and Cloudflare to understand how large-scale systems solve the same problems we face."

4
You join a team with messy legacy code. How do you approach it?Practical

"First 30 days: understand, don't change. Read existing code, run it locally, read tests (if any). Talk to the team — why did decisions get made this way? Map the pain points. Then: propose small targeted improvements with clear ROI. Add tests for critical paths before touching anything. Refactor in the edit path — when you have to change code anyway, clean it up at that point. Never big-bang rewrite — death march risk. Boy Scout rule: leave code slightly better than you found it, every PR."

Designed and implemented CurrentSchoolContext that reads SchoolId from JWT, wired it into EF Query Filters, added integration tests, and deployed with zero downtime using staging slot. This pattern protected 50,000 student records from potential cross-school data access.

6
How do you handle a production incident?Process

"1. Acknowledge and communicate first — inform stakeholders ASAP. 2. Mitigate before fixing — if possible, roll back or enable feature flag. 3. Investigate with Application Insights: check exceptions, slow queries, error spikes. 4. Fix with minimum change — smallest diff to restore service. 5. Post-incident review — what broke, why, detection gap, prevention. For ZenCampus: if fee payment fails, immediately check payment gateway webhook logs + Application Insights. Rollback via deployment slot swap if needed — takes 30 seconds."

7
What would you do in your first 90 days at a new company?Practical

"Days 1-30: Understand. Set up dev environment, read the codebase, run tests, understand the business domain. Ask 'why' more than 'how'. No major code changes. Days 31-60: Contribute. Take ownership of a small feature or bug fix. Follow existing patterns. Build relationships with teammates. Days 61-90: Impact. Identify one meaningful improvement — performance, testing, developer experience. Propose with data. Ship with confidence. The goal: be trusted with progressively larger work by day 90."

8
What is your approach to code review?Process

"As reviewer: check correctness first (does it do what it should?), then security (SQL injection? JWT bypass? cross-tenant access?), then performance (N+1? missing index?), then maintainability. Phrase feedback as questions ('Did you consider...?') not commands. Approve when 90% good — don't block on style. As author: small PRs, good description with 'why', test evidence, flag areas I'm uncertain about. Code review is knowledge transfer as much as quality control."
