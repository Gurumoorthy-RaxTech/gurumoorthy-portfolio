The 3 DI Lifetimes

⏳ When is a new instance created? That is the lifetime question.

Wrong lifetime register pannombodhu — data leak between requests, DbContext reused across threads, stale data. Most common source of hard-to-reproduce bugs in .NET apps.

TRANSIENT
New instance every GetService call
Safest but most allocations. Use for lightweight stateless services.
AddTransient<T>()
SCOPED
New instance per HTTP request
Same object within one request. Different requests get different instances.
AddScoped<T>()
SINGLETON
One instance for app lifetime
Shared by ALL requests. Must be thread-safe. Created at first resolve.
AddSingleton<T>()
ZenCampus ServiceLifetimeReason
ZenCampusDbContextScopedOne DB connection per request, tracks changes for that request only
IStudentRepositoryScopedUses DbContext — must match its lifetime (dependency chain rule)
IFeeServiceScopedUses IStudentRepository — match the whole chain
ILogger<T>SingletonThread-safe, stateless, expensive to create once
IMemoryCacheSingletonMust be shared across all requests to be useful
ISmsValidator (stateless)SingletonNo state, pure functions — safe to share
IHttpClientFactorySingletonManages pooled socket handlers; typed clients are transient
AuditEntry (per-operation)TransientUnique mutable state per service call, not per request

Captive Dependency — The Silent Data Leak

🔢 Singleton holding Scoped = scoped object lives forever = data corruption

Singleton-la scoped service inject panna — scoped object singleton-oda lifetime-la live pannum — app lifetime! Same DbContext forever. Request 1 student data, request 2 reads stale change-tracked state. Cross-user data leak — serious GDPR violation.

The ZenCampus Captive Dependency Bug:
SmsNotificationService (Singleton) injects IStudentRepository (Scoped)
Same DbContext held for the entire app lifetime
Request A and Request B share the same change-tracking context
Parent B receives Parent A's fee receipt — data breach! 
// WRONG -- Singleton capturing a Scoped dependency
public class SmsNotificationService(
    IStudentRepository _repo,   // Scoped -- injected into Singleton!
    IMsg91Client _msg91)        // Transient -- also captured forever
    : ISmsNotificationService
{
    public async Task SendReceiptAsync(int studentId)
    {
        var student = await _repo.GetByIdAsync(studentId);
        // _repo.DbContext is from the FIRST request that triggered this singleton's creation
        // ALL subsequent requests share this same DbContext -- not thread-safe!
    }
}
builder.Services.AddSingleton<ISmsNotificationService, SmsNotificationService>(); // BUG

// FIX 1: Change lifetime to Scoped (simplest)
builder.Services.AddScoped<ISmsNotificationService, SmsNotificationService>();

// FIX 2: Keep Singleton, create a scope per operation via IServiceScopeFactory
public class SmsNotificationService(IServiceScopeFactory _factory, IMsg91Client _msg91)
    : ISmsNotificationService
{
    public async Task SendReceiptAsync(int studentId, CancellationToken ct)
    {
        await using var scope = _factory.CreateAsyncScope();
        var repo    = scope.ServiceProvider.GetRequiredService<IStudentRepository>();
        var student = await repo.GetByIdAsync(studentId, ct);
        await _msg91.SendSmsAsync(student.ParentPhone, "Fee collected", ct);
        // scope disposed here -- fresh DbContext per call -- thread-safe
    }
}
builder.Services.AddSingleton<ISmsNotificationService, SmsNotificationService>(); // Now safe
// IServiceScopeFactory is itself Singleton -- always safe to inject into Singleton
Captive Dependency Rules (memorise):
Singleton → Singleton: safe  |  Singleton → Transient (stateless): safe
Scoped → Scoped: safe  |  Scoped → Transient: safe  |  Transient → anything: safe
Singleton → Scoped: CAPTIVE DEPENDENCY BUG
General rule: never inject a shorter-lifetime service into a longer-lifetime consumer. 
// Catch captive dependencies at startup -- not at runtime under load
builder.Host.UseDefaultServiceProvider(opts =>
{
    opts.ValidateScopes  = true;  // Singleton->Scoped detected at Build() time
    opts.ValidateOnBuild = true;  // Missing registrations detected at Build() time
});

// In Production it is disabled (performance) -- but fix issues in Development first

Keyed Services — .NET 8+

🔑 Multiple implementations of same interface — resolve by string/enum key

ZenCampus-la multiple notification channels irukku — SMS (fee receipts), Email (teachers), Push (parent app). Same interface, different implementations. Before .NET 8: dictionary hack or factory. Now: AddKeyedScoped and [FromKeyedServices].

// Common interface
public interface INotificationChannel
{
    Task SendAsync(string recipient, string message, CancellationToken ct);
}

public class SmsChannel   : INotificationChannel { /* Msg91 */ }
public class EmailChannel : INotificationChannel { /* SendGrid */ }
public class PushChannel  : INotificationChannel { /* Firebase FCM */ }

// Program.cs -- register with keys
builder.Services.AddKeyedScoped<INotificationChannel, SmsChannel>("sms");
builder.Services.AddKeyedScoped<INotificationChannel, EmailChannel>("email");
builder.Services.AddKeyedScoped<INotificationChannel, PushChannel>("push");

// Inject by key -- [FromKeyedServices] attribute on constructor parameter
public class FeeCollectionService(
    [FromKeyedServices("sms")]   INotificationChannel _sms,
    [FromKeyedServices("email")] INotificationChannel _email)
{
    public async Task CollectAsync(FeeCommand cmd, CancellationToken ct)
    {
        // save fee to DB...
        await _sms.SendAsync(cmd.ParentPhone, "Fee received: Rs.{Amount}", ct);
        await _email.SendAsync(cmd.TeacherEmail, "Fee collected notification", ct);
    }
}

// Runtime key resolution -- school config says which channel to use
public class NotificationDispatcher(IServiceProvider _sp)
{
    public INotificationChannel GetForSchool(School school)
    {
        var key = school.PreferredChannel; // "sms" | "email" | "push" from DB
        return _sp.GetRequiredKeyedService<INotificationChannel>(key);
    }
}
Before .NET 8 workaround: Dictionary pattern — AddSingleton<IDictionary<string, INotificationChannel>>(sp => new Dictionary ... { ["sms"] = sp.GetRequiredService<SmsChannel>() }). Keyed services is the clean replacement — mention both in interviews to show awareness of evolution.

Factory Pattern in DI

🏭 When you need to create services at runtime with dynamic parameters

Factory pattern = runtime-la decide panna — school A different DB connection, school B different fee calculator. DI container compile-time-la register panna mudiyathu — factory use pannrom.

// Func<T> factory -- simplest form
builder.Services.AddScoped<IFeeCalculator>(sp =>
{
    var config  = sp.GetRequiredService<ISchoolConfigService>();
    var schoolId = sp.GetRequiredService<IHttpContextAccessor>()
                     .HttpContext?.User.FindFirst("schoolId")?.Value;
    return config.IsStateBoard(schoolId)
        ? sp.GetRequiredService<StateBoardFeeCalculator>()
        : sp.GetRequiredService<CbseFeeCalculator>();
});
// Different fee calculation rules per board type -- resolved at request time
// Abstract factory -- create DbContext with school-specific connection string
public interface ISchoolDbContextFactory
{
    ZenCampusDbContext CreateForSchool(int schoolId);
}

public class SchoolDbContextFactory(ISchoolConnectionResolver _resolver)
    : ISchoolDbContextFactory
{
    public ZenCampusDbContext CreateForSchool(int schoolId)
    {
        var connStr = _resolver.GetConnectionString(schoolId);
        var opts    = new DbContextOptionsBuilder<ZenCampusDbContext>()
                         .UseSqlServer(connStr)
                         .Options;
        return new ZenCampusDbContext(opts);
    }
}
// Usage: using var ctx = _factory.CreateForSchool(schoolId); -- caller owns lifetime
// IServiceProvider.GetService vs GetRequiredService
var service = sp.GetService<IOptionalFeature>();      // returns null if not registered
var service = sp.GetRequiredService<IFeeService>();   // throws if not registered
// Always prefer GetRequiredService -- fail fast at startup, not at runtime under load

Options Pattern — IOptions / IOptionsSnapshot / IOptionsMonitor

⚙ Strongly-typed, validated configuration — no more magic strings

Configuration["Msg91:ApiKey"] everywhere pannadheenga — refactor pannum bodhu miss aagum. Options pattern use pannunga — strongly typed, validated at startup, testable.

// Settings classes -- one per logical section
public class Msg91Settings
{
    public const string SectionName = "Msg91";

    [Required]
    public string ApiKey    { get; init; } = string.Empty;

    [Required]
    public string SenderId  { get; init; } = string.Empty;

    [Range(1, 5)]
    public int    MaxRetries { get; init; } = 3;
}

public class PaymentGatewaySettings
{
    public const string SectionName = "PaymentGateway";
    [Required] public string MerchantId  { get; init; } = string.Empty;
    [Required] public string SecretKey   { get; init; } = string.Empty;
    public bool              IsSandbox   { get; init; }
}

// Program.cs -- register with validation
builder.Services
    .AddOptions<Msg91Settings>()
    .BindConfiguration(Msg91Settings.SectionName)
    .ValidateDataAnnotations()          // [Required], [Range] validated
    .ValidateOnStart();                  // Fail at startup, not first SMS send!

builder.Services
    .AddOptions<PaymentGatewaySettings>()
    .BindConfiguration(PaymentGatewaySettings.SectionName)
    .ValidateDataAnnotations()
    .Validate(s => !s.IsSandbox || !string.IsNullOrEmpty(s.SecretKey),
               "SecretKey required when not in Sandbox mode")
    .ValidateOnStart();
// Three IOptions variants -- critical interview topic
public class Msg91Client(
    IOptions<Msg91Settings>         _opts,       // Singleton: config read ONCE at startup
    IOptionsSnapshot<Msg91Settings> _snapshot,   // Scoped: reloads per request (dev reload)
    IOptionsMonitor<Msg91Settings>  _monitor)    // Singleton + change notification
{
    public async Task SendAsync(string phone, string msg, CancellationToken ct)
    {
        // Use IOptions for production -- config doesn't change at runtime
        var key = _opts.Value.ApiKey;

        // IOptionsMonitor.CurrentValue -- always latest (useful for feature flags)
        var retries = _monitor.CurrentValue.MaxRetries;
    }
}

// IOptionsMonitor -- react to config changes (Azure App Config, hot reload)
public class FeatureFlagService(IOptionsMonitor<FeatureFlags> _monitor)
{
    private FeatureFlags _current = default!;

    public FeatureFlagService(IOptionsMonitor<FeatureFlags> monitor)
    {
        _monitor = monitor;
        _current = monitor.CurrentValue;
        monitor.OnChange(updated => _current = updated); // live reload without restart
    }
    public bool IsEnabled(string flag) => _current.EnabledFlags.Contains(flag);
}

Decorator Pattern in DI

🎨 Wrap an existing implementation to add cross-cutting behaviour

Decorator pattern = existing service-a wrap panrom — logging, caching, retry — original code touch pannama. ZenCampus: CachedStudentRepository wraps StudentRepository — cache check before every DB call.

// Original implementation
public class StudentRepository(ZenCampusDbContext _db) : IStudentRepository
{
    public async Task<Student?> GetByIdAsync(int id, CancellationToken ct)
        => await _db.Students.FindAsync([id], ct);
}

// Decorator -- adds caching without changing StudentRepository
public class CachedStudentRepository(
    IStudentRepository _inner,   // wraps the real repository
    IDistributedCache  _cache) : IStudentRepository
{
    public async Task<Student?> GetByIdAsync(int id, CancellationToken ct)
    {
        var key    = $"student:{id}";
        var cached = await _cache.GetStringAsync(key, ct);
        if (cached is not null) return JsonSerializer.Deserialize<Student>(cached);

        var student = await _inner.GetByIdAsync(id, ct); // delegate to real repo
        if (student is not null)
            await _cache.SetStringAsync(key, JsonSerializer.Serialize(student),
                new() { AbsoluteExpirationRelativeToNow = TimeSpan.FromMinutes(10) }, ct);
        return student;
    }
}

// Program.cs -- register decorator manually (no built-in AddDecorator)
builder.Services.AddScoped<StudentRepository>();   // register concrete
builder.Services.AddScoped<IStudentRepository>(sp =>
    new CachedStudentRepository(
        sp.GetRequiredService<StudentRepository>(),   // inner
        sp.GetRequiredService<IDistributedCache>()));  // cache
// Or use Scrutor NuGet: services.Decorate<IStudentRepository, CachedStudentRepository>()

Service Locator Anti-Pattern

🚫 Injecting IServiceProvider everywhere = hiding dependencies

IServiceProvider everywhere inject panna — class-oda dependencies hidden aagum, tests mock panna mudiyathu, DI container-la validate aagaadhu. Legitimate use: middleware InvokeAsync parameter, abstract factory, BackgroundService (Day 21).

Anti-pattern (Service Locator) 
public class FeeService(IServiceProvider _sp)
{
    public async Task CollectAsync(...)
    {
        // Dependencies hidden inside method
        var repo = _sp.GetRequiredService<IFeeRepo>();
        var sms  = _sp.GetRequiredService<ISmsClient>();
        // Unit test can't see these deps
        // ValidateOnBuild won't catch missing reg
    }
}
Constructor injection (correct) 
public class FeeService(
    IFeeRepository _repo,
    ISmsClient     _sms)
{
    public async Task CollectAsync(...)
    {
        // Dependencies visible at a glance
        // Mockable in unit tests
        // ValidateOnBuild checks them
        // DI graph fully analysable
    }
}
Legitimate IServiceProvider uses: Middleware (IMiddleware pattern or InvokeAsync param), abstract factory (runtime key resolution), BackgroundService (IServiceScopeFactory to create scope per job), extension methods bootstrapping DI itself. Outside these cases — always prefer constructor injection.

🎯 10 Interview Scenarios

Q1. What are the three DI lifetimes? Give a ZenCampus example for each.
Concept

Transient — new instance every time GetService is called. ZenCampus: PhoneNumberFormatter (pure function, no state, called from multiple places — new instance each call is cheapest to reason about).

Scoped — new instance per HTTP request. All services within that request share the same instance. ZenCampus: ZenCampusDbContext (one DB connection and change-tracker per request), IStudentRepository, IFeeService (anything that uses DbContext must be scoped).

Singleton — one instance for the app lifetime, shared across all requests and threads. ZenCampus: IMemoryCache (must be shared to be useful), ILogger<T> (thread-safe by design, expensive to create), ISchoolCodeGenerator (stateless, thread-safe utility).

Rule of thumb: start with Scoped. Move to Singleton only when you need app-wide state and can prove thread safety. Use Transient for stateful helpers that must not be shared.

Q2. PRODUCTION BUG: Random students getting wrong school's data. No pattern, intermittent. What is the root cause?
Production Bug

Root cause: Captive Dependency. A Singleton service holds a Scoped IStudentRepository (and therefore a Scoped ZenCampusDbContext). The DbContext was created for one request and captured for the app lifetime. Concurrent requests from different schools share the same DbContext instance, which is not thread-safe and leaks change-tracked entities across requests.

// Diagnosis: enable ValidateScopes in Development
builder.Host.UseDefaultServiceProvider(opts => opts.ValidateScopes = true);
// This throws at startup: "Cannot consume scoped service IStudentRepository
// from singleton SomeService" -- catches bug before it hits production

// Fix: IServiceScopeFactory in the Singleton
public async Task DoWorkAsync(int schoolId, CancellationToken ct)
{
    await using var scope = _factory.CreateAsyncScope();
    var repo = scope.ServiceProvider.GetRequiredService<IStudentRepository>();
    // fresh DbContext per operation -- thread-safe, no cross-request leak
}

Key lesson: ValidateScopes = true should ALWAYS be on in Development. It turns a runtime data-corruption bug into a startup exception.

Q3. IOptions vs IOptionsSnapshot vs IOptionsMonitor — when do you use each?
Concept

IOptions<T>: Singleton. Config read once at startup, never refreshes. Use for settings that cannot change without a restart — DB connection strings, API keys. Cannot inject into Scoped or Transient services without concern (it is singleton).

IOptionsSnapshot<T>: Scoped. Re-reads config each HTTP request. Use in dev when you want hot-reload without restarting the app, or for per-request config experiments. Cannot inject into Singleton — captive dependency trap!

IOptionsMonitor<T>: Singleton. Live refresh via OnChange callback. Use for feature flags, rate limits, dynamic thresholds that ops team needs to change without deploying. ZenCampus: IOptionsMonitor<FeatureFlags> — ops flips "EnableNewAdmissionFlow" in Azure App Config, it propagates without restart within seconds.

ZenCampus defaults: IOptions for Msg91Settings and PaymentGatewaySettings (secrets, can't change without rotation). IOptionsMonitor for FeatureFlags and rate limit thresholds.

Q5. What is the Service Locator anti-pattern? Why is it bad?
Concept

Service Locator = injecting IServiceProvider and calling GetService<T>() inside methods instead of declaring dependencies in the constructor.

Problems:

  1. Dependencies are hidden inside method bodies — a class with 10 GetService calls looks like it has zero dependencies from the outside
  2. Unit tests cannot mock the dependencies easily — must set up a full DI container or mock IServiceProvider (painful)
  3. ValidateOnBuild cannot detect missing registrations for services resolved this way
  4. DI graph cannot be analysed by tools — no dependency visualisation, no cycle detection

Legitimate uses: Middleware InvokeAsync parameters, abstract factory that resolves by runtime key, BackgroundService with IServiceScopeFactory. Outside these patterns — always use constructor injection.

Q6. How do you validate all DI registrations and configuration at startup instead of getting runtime errors?
Design 
// 1. ValidateOnBuild -- detects missing service registrations at Build() time
builder.Host.UseDefaultServiceProvider(opts =>
{
    opts.ValidateOnBuild = true; // throws if any registered service can't be resolved
    opts.ValidateScopes  = true; // throws on Singleton->Scoped captive dependency
});

// 2. ValidateOnStart -- Options validation at startup, not first use
builder.Services
    .AddOptions<Msg91Settings>()
    .BindConfiguration("Msg91")
    .ValidateDataAnnotations()  // [Required], [Range] validated
    .ValidateOnStart();          // Msg91:ApiKey missing? App crashes at startup, not at 2AM SMS

// 3. IStartupFilter or IHostedService for custom startup checks
public class StartupValidationService(ZenCampusDbContext _db) : IHostedService
{
    public async Task StartAsync(CancellationToken ct)
    {
        var canConnect = await _db.Database.CanConnectAsync(ct);
        if (!canConnect)
            throw new InvalidOperationException("Cannot connect to ZenCampus database. Check connection string.");
    }
    public Task StopAsync(CancellationToken ct) => Task.CompletedTask;
}

Philosophy: "Fail fast at startup." Every minute of running with wrong config is a minute where SMS is silently failing or DB is being hit with wrong credentials. Startup validation converts 2 AM production incidents into deployment failures that block bad config from reaching production.

Q7. BackgroundService (IHostedService) needs to use DbContext every 5 minutes. How?
Design

BackgroundService is a Singleton (registered by the framework). DbContext is Scoped. Cannot inject DbContext in constructor — captive dependency. Must create a scope for each operation.

public class AttendanceSyncService(IServiceScopeFactory _factory, ILogger<AttendanceSyncService> _log)
    : BackgroundService
{
    protected override async Task ExecuteAsync(CancellationToken ct)
    {
        while (!ct.IsCancellationRequested)
        {
            try
            {
                await using var scope = _factory.CreateAsyncScope();  // fresh scope per tick
                var db      = scope.ServiceProvider.GetRequiredService<ZenCampusDbContext>();
                var service = scope.ServiceProvider.GetRequiredService<IAttendanceService>();
                await service.SyncAllSchoolsAsync(ct);
                // scope disposed here -- DbContext lifetime correctly bounded to this tick
            }
            catch (Exception ex) when (ex is not OperationCanceledException)
            {
                _log.LogError(ex, "Attendance sync failed");
            }
            await Task.Delay(TimeSpan.FromMinutes(5), ct);
        }
    }
}
// IServiceScopeFactory is registered by the framework as Singleton -- always safe to inject

Q9. How do you add caching to IStudentRepository without modifying its source code?
Design

Decorator pattern — create a wrapper that implements the same interface and delegates to the real implementation:

public class CachedStudentRepository(
    IStudentRepository _inner,
    IDistributedCache  _cache,
    ILogger<CachedStudentRepository> _log) : IStudentRepository
{
    public async Task<Student?> GetByIdAsync(int id, CancellationToken ct)
    {
        var key    = $"student:{id}";
        var cached = await _cache.GetStringAsync(key, ct);
        if (cached is not null)
        {
            _log.LogDebug("Cache HIT {Key}", key);
            return JsonSerializer.Deserialize<Student>(cached);
        }
        var student = await _inner.GetByIdAsync(id, ct);
        if (student is not null)
            await _cache.SetStringAsync(key, JsonSerializer.Serialize(student),
                new() { AbsoluteExpirationRelativeToNow = TimeSpan.FromMinutes(10) }, ct);
        return student;
    }
    // other IStudentRepository methods delegate to _inner similarly
}

// Registration

    new CachedStudentRepository(
        sp.GetRequiredService<StudentRepository>(),
        sp.GetRequiredService<IDistributedCache>(),
        sp.GetRequiredService<ILogger<CachedStudentRepository>>()));

Q10. "Tell me about your DI experience in ZenCampus." — Full structured answer
Experience

"In ZenCampus we serve 200 schools and 50,000 students across 3 load-balanced API instances, so getting DI lifetimes right is critical for both correctness and performance.

Lifetime discipline: All services that touch DbContext are registered as Scoped — repositories, services, command handlers. Stateless utilities like phone validators and code generators are Singleton. We follow the chain rule: if you depend on something Scoped, you must also be Scoped or create a new scope per operation.

Captive dependency incident: Early on, our background notification service was registered as Singleton and accidentally held a Scoped IStudentRepository. This went undetected until load testing — concurrent requests started getting mixed data. We fixed it using IServiceScopeFactory to create a fresh scope per notification job. We also turned on ValidateScopes = true in Development, which now catches this class of bug at startup, not at runtime.

Keyed services: We have SMS, Email, and Push notification channels implementing INotificationChannel. Using .NET 8 keyed services — AddKeyedScoped with "sms", "email", "push" keys — and a NotificationDispatcher that resolves by school preference stored in the DB.

Options pattern: All external API settings (Msg91, payment gateway) use AddOptions + ValidateDataAnnotations + ValidateOnStart. Missing API key means the app crashes at deployment, not at 2 AM when the first fee SMS fails.

Decorator: CachedStudentRepository wraps StudentRepository — added Redis caching across the codebase with zero changes to existing service code."

🧠 Quick Quiz

Q1. SmsNotificationService is registered as Singleton. It needs ZenCampusDbContext (Scoped). What is the safest fix?
IServiceScopeFactory — it is itself a Singleton, safe to inject into a Singleton. Each SMS send creates a fresh scope via CreateAsyncScope(), resolves a fresh DbContext from that scope, and disposes it when done. Making DbContext Singleton would break all EF Core change-tracking (it holds state). IServiceProvider resolves from the root scope = same captive dependency bug. Static DbContext = guaranteed concurrency bugs.
Q2. IOptionsSnapshot<Msg91Settings> is injected into a Singleton class. What happens?
IOptionsSnapshot is Scoped — injecting it into a Singleton is a captive dependency bug. With ValidateScopes=true it throws at startup. Use IOptions (Singleton, static config) or IOptionsMonitor (Singleton + live change notification) inside Singleton services. IOptionsSnapshot is for Scoped/Transient services that want per-request config reload.
Q3. ZenCampus needs to pick between SmsChannel and PushChannel at runtime based on each school's preference stored in DB. Best DI approach?
Keyed services (.NET 8) is the clean solution: register SmsChannel with key "sms", PushChannel with key "push". A NotificationDispatcher resolves GetRequiredKeyedService<INotificationChannel>(school.PreferredChannel) at runtime. IEnumerable + GetType() works but is fragile. new() in controller bypasses DI entirely. Injecting both and using if/else works but scales poorly (10 channels = 10 injected parameters).
Q4. ValidateOnBuild = true is set. A new IReportService is added to a controller but never registered in Program.cs. When does the error surface?
ValidateOnBuild causes builder.Build() to resolve all registered controllers and validate their dependency graphs. Missing IReportService registration throws InvalidOperationException during startup — the app never starts. Without ValidateOnBuild, the error only surfaces when a request hits that specific controller action, potentially hours or days after deployment.
