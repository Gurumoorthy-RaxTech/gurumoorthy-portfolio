📄 Production Dockerfile for ZenCampus API

# Good Dockerfile for .NET 8 Web API
FROM mcr.microsoft.com/dotnet/sdk:8.0 AS build
WORKDIR /src

# Copy csproj and restore first (layer caching)
COPY ["ZenCampusWeb/ZenCampusWeb.csproj", "ZenCampusWeb/"]
RUN dotnet restore "ZenCampusWeb/ZenCampusWeb.csproj"

# Copy all source, then build
COPY . .
WORKDIR /src/ZenCampusWeb
RUN dotnet build "ZenCampusWeb.csproj" -c Release -o /app/build

---
# .dockerignore (avoid copying unnecessary files)
bin/
obj/
*.user
.vs/
appsettings.Development.json
# Never include secrets in the image
NEVER bake secrets into Docker images. No connection strings, API keys, or passwords in Dockerfile or image layers. Use environment variables, Azure Key Vault, or Docker secrets at runtime.

🏗️ Multi-Stage Build — Production-Grade

# Stage 1: Build
FROM mcr.microsoft.com/dotnet/sdk:8.0 AS build
WORKDIR /src
COPY ["ZenCampusWeb.csproj", "."]
RUN dotnet restore
COPY . .
RUN dotnet publish "ZenCampusWeb.csproj" \
    -c Release \
    -o /app/publish \
    --no-restore \
    /p:UseAppHost=false

# Stage 2: Final runtime image (much smaller!)
FROM mcr.microsoft.com/dotnet/aspnet:8.0 AS final

# Run as non-root user (security best practice)
RUN adduser --disabled-password --gecos "" appuser
USER appuser

WORKDIR /app
EXPOSE 8080
ENV ASPNETCORE_URLS=http://+:8080
ENV ASPNETCORE_ENVIRONMENT=Production

# Copy only published output from build stage
COPY --from=build /app/publish .

# Health check
HEALTHCHECK --interval=30s --timeout=10s --start-period=5s --retries=3 \
  CMD curl -f http://localhost:8080/health || exit 1

ENTRYPOINT ["dotnet", "ZenCampusWeb.dll"]
Why multi-stage? SDK image (~800MB) only used for building. Final image uses aspnet runtime (~220MB). Production containers are 4x smaller, faster to pull, less attack surface.

🎼 docker-compose for ZenCampus Local Dev

# docker-compose.yml — ZenCampus local environment
version: '3.9'

services:
  zencampus-api:
    build:
      context: .
      dockerfile: ZenCampusWeb/Dockerfile
      target: final
    container_name: zencampus-api
    ports:
      - "8080:8080"
    environment:
      - ASPNETCORE_ENVIRONMENT=Development
      - ConnectionStrings__ZenCampusDb=Server=sqlserver;Database=ZenCampusDb;User Id=sa;Password=${SQL_PASSWORD};TrustServerCertificate=True
      - Redis__ConnectionString=redis:6379
    depends_on:
      sqlserver:
        condition: service_healthy
      redis:
        condition: service_healthy
    networks:
      - zencampus-net

  sqlserver:
    image: mcr.microsoft.com/mssql/server:2022-latest
    container_name: zencampus-sql
    ports:
      - "1433:1433"
    environment:
      - SA_PASSWORD=${SQL_PASSWORD}
      - ACCEPT_EULA=Y
    healthcheck:
      test: ["CMD", "/opt/mssql-tools/bin/sqlcmd", "-S", "localhost", "-U", "sa", "-P", "${SQL_PASSWORD}", "-Q", "SELECT 1"]
      interval: 10s
      timeout: 3s
      retries: 10
      start_period: 30s
    volumes:
      - sqldata:/var/opt/mssql
    networks:
      - zencampus-net

  redis:
    image: redis:7-alpine
    container_name: zencampus-redis
    ports:
      - "6379:6379"
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 5s
      timeout: 3s
      retries: 5
    networks:
      - zencampus-net

volumes:
  sqldata:

networks:
  zencampus-net:
    driver: bridge
# .env file (never commit to git!)
SQL_PASSWORD=ZenCampus@123!

🎯 10 Interview Scenarios

1
What is the difference between a Docker image and a container?Concept

Image: read-only, layered blueprint with app + dependencies. Container: running instance of an image — adds a writable layer on top. One image can spawn many containers. Analogy: image = class definition, container = object instance. Images are stored in registry (Docker Hub, ACR). Containers are ephemeral — delete container, image stays.

2
Explain multi-stage Docker builds and why you'd use them for ZenCampus.Code

Multi-stage: multiple FROM statements in one Dockerfile. Build stage uses SDK image (800MB) with full compiler. Final stage uses aspnet runtime (220MB) — COPY --from=build only copies the published output. Final image has no build tools, no source code, no SDK. Result: 4x smaller image, faster pulls, less security attack surface. For ZenCampus: dev builds take 800MB SDK; prod image only 220MB.

3
Why should connection strings NOT be in the Dockerfile?Security

Docker image layers are permanent — once a secret is added to a layer, docker history can reveal it even if later layers delete it. Images are often pushed to registries. Anyone with registry access sees the secret. Correct approach: pass via environment variables at runtime (docker run -e), Docker secrets, Kubernetes secrets, or Azure Key Vault with Managed Identity. Never bake secrets into the image.

4
What is .dockerignore and why does it matter?Practical

.dockerignore tells Docker which files to exclude from the build context sent to the daemon. Without it: bin/, obj/, node_modules/, .git/, and appsettings.Development.json (with dev secrets) all get copied to the daemon — huge context, slow builds, potential secret leak. With .dockerignore: only source files sent, build is faster, secrets stay off the image.

5
Why do we COPY csproj files and RUN dotnet restore BEFORE copying all source code?Optimization

Docker layer caching: each instruction is cached. If only C# source files change but csproj hasn't changed, Docker reuses the cached restore layer. NuGet packages (often 100–300MB) are not re-downloaded on every build — only when package versions change. Without this pattern: every code change triggers full dotnet restore = very slow CI builds.

6
What does depends_on with condition: service_healthy do in docker-compose?Code

depends_on alone only waits for the container to START — not for the service to be READY. SQL Server takes 15–30 seconds after starting before accepting connections. With condition: service_healthy, docker-compose waits until the healthcheck passes before starting the dependent service. For ZenCampus: API container waits until SQL Server health check (SELECT 1) succeeds before starting.

7
Explain the difference between COPY and ADD in Dockerfile.Concept

COPY: simply copies files from build context to the image. Transparent and predictable. ADD: does everything COPY does, plus can auto-extract tar.gz archives and download from URLs. Best practice: always use COPY unless you specifically need ADD's extra features. ADD's implicit behaviors (auto-extract, URL download) can cause unexpected build behavior.

8
What is a Docker volume and when would ZenCampus need one?Concept

Docker containers have ephemeral writable layers — data is lost when container stops. Volumes provide persistent storage outside the container. ZenCampus needs volumes for: SQL Server data files (sqldata:/var/opt/mssql), uploaded student photos/documents (if stored locally), Hangfire job database. Without volumes: restart SQL Server container = all student data gone!

9
Why run the container as non-root user?Security

Container breakout: if an attacker exploits the app and escapes the container, running as root means they have root on the host. Non-root user limits blast radius. Best practice: create a dedicated user (adduser appuser), switch to it before ENTRYPOINT. Azure App Service on Linux does this by default. Also: helps with compliance requirements in education sector.

10
How would you apply EF Core migrations in a containerized ZenCampus setup?Practical

Options: 1) Run migrations at startup in Program.cs (simple but risky — parallel instances all try to migrate). 2) Separate migration job container that runs and exits before API starts. 3) Use idempotent migration scripts applied by DBA/CI pipeline. Best: option 3 for production (migrations are audited, reviewed). Option 1 OK for dev/staging. docker-compose: add a migration service that depends_on sqlserver: healthy.
