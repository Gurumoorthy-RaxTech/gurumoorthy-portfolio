ASP.NET Core Request Pipeline

🔩 Every request passes through a chain of middleware — order is everything

Request vandha — pipeline-la oru by oru middleware pass aagum. Each middleware can: process and pass to next, short-circuit and return early, or modify request/response on the way in AND out. Order matters critically — authentication MUST run before authorization.

REQUEST PIPELINE — order in Program.cs = order of execution
→ExceptionHandlercatch all unhandled exceptions globally
→HSTS / HTTPS Redirectionsecurity headers
→Static Filesshort-circuit for .js/.css — never hits controllers
→Routingmatch URL → select endpoint
→CORSmust be BEFORE Auth
→Authenticationwho are you? → populate User
→Authorizationare you allowed? — MUST be after Auth
→TenantResolutionZenCampus: which school?
→RequestTimingZenCampus: log slow requests
→Endpoints (Controllers)
← Response flows back through each middleware in REVERSE order (response side of the pipeline) 
// Program.cs — correct order (CRITICAL for interviews)
var app = builder.Build();

app.UseExceptionHandler("/error");       // 1. Catch everything — must be FIRST
app.UseHsts();                              // 2. Security headers
app.UseHttpsRedirection();                 // 3. HTTP → HTTPS redirect
app.UseStaticFiles();                      // 4. Short-circuit for static assets
app.UseRouting();                           // 5. Match URL to endpoint
app.UseCors();                              // 6. CORS — after routing, BEFORE auth
app.UseAuthentication();                   // 7. Who are you? (parse JWT → User)
app.UseAuthorization();                    // 8. Are you allowed? — MUST be after 7
app.UseMiddleware<TenantResolutionMiddleware>(); // 9. Custom — after auth
app.UseMiddleware<RequestTimingMiddleware>();     // 10. Custom
app.MapControllers();                      // 11. Route to controllers
❌ Wrong order — classic interview trap:
app.UseAuthorization() before app.UseAuthentication() → Authorization runs without knowing who the user is → every request appears anonymous → all [Authorize] endpoints return 401

app.UseCors() after app.UseAuthorization() → CORS headers missing on 401 responses → browser can't read the 401 → confusing CORS errors in frontend instead of auth errors

Custom Middleware

🔧 Two patterns: convention-based delegate vs IMiddleware interface
// Pattern 1: Convention-based (minimal, no DI constructor injection for scoped)
public class RequestTimingMiddleware(RequestDelegate _next, ILogger<RequestTimingMiddleware> _log)
{
    public async Task InvokeAsync(HttpContext ctx)
    {
        var sw = Stopwatch.StartNew();
        await _next(ctx);   // ← Call next middleware in pipeline
        sw.Stop();

        var elapsed = sw.ElapsedMilliseconds;
        if (elapsed > 500)  // Log slow requests > 500ms
            _log.LogWarning(
                "SLOW REQUEST: {Method} {Path} took {ElapsedMs}ms",
                ctx.Request.Method, ctx.Request.Path, elapsed);
    }
}
// Register: app.UseMiddleware<RequestTimingMiddleware>();
// ILogger injected from constructor — singleton scope (safe for ILogger)
// Pattern 2: IMiddleware — injected by DI, supports SCOPED services
public class TenantResolutionMiddleware(ILogger<TenantResolutionMiddleware> _log) : IMiddleware
{
    public async Task InvokeAsync(HttpContext ctx, RequestDelegate next)
    {
        // IMiddleware gets HttpContext + RequestDelegate — can inject scoped services
        var tenantService = ctx.RequestServices.GetRequiredService<ITenantService>();

        // Resolve school from JWT claim (already populated by UseAuthentication)
        var schoolIdClaim = ctx.User.FindFirst("schoolId")?.Value;

        if (schoolIdClaim is not null && int.TryParse(schoolIdClaim, out var schoolId))
        {
            var tenant = await tenantService.GetSchoolAsync(schoolId, ctx.RequestAborted);
            if (tenant is null)
            {
                // Short-circuit — school doesn't exist in DB
                ctx.Response.StatusCode = StatusCodes.Status403Forbidden;
                await ctx.Response.WriteAsJsonAsync(
                    new { error = "Invalid school context" });
                return;  // ← DON'T call next — short-circuit here
            }
            ctx.Items["Tenant"] = tenant;  // Available to all downstream middleware + controllers
            _log.LogDebug("Tenant resolved: {SchoolId} {SchoolName}", tenant.Id, tenant.Name);
        }
        await next(ctx);  // Call next — continue pipeline
    }
}
// Must register in DI: builder.Services.AddScoped<TenantResolutionMiddleware>();
// Then: app.UseMiddleware<TenantResolutionMiddleware>();
Convention vs IMiddleware — key difference:
Convention (RequestDelegate in constructor): Constructor called ONCE — singleton lifetime. Cannot inject scoped services in constructor. Works with ILogger<T>, IOptions<T> (singletons). Access scoped via ctx.RequestServices.GetRequiredService<T>().

IMiddleware: DI creates instance per request (transient/scoped). Can inject scoped services directly in constructor. More testable (DI-friendly). Requires builder.Services.AddScoped<MyMiddleware>().

Short-Circuit Pattern

⚡ Not calling await next(ctx) = short-circuit — stop the pipeline here

Short-circuit = pipeline-a stop pannum — next middleware, controller — nothing runs. Response middleware writes panna — client-ku return aagum. API key invalid aana short-circuit pannrom — controller-e reach aagaadhu.

// Security headers middleware — adds headers to EVERY response (response side)
public class SecurityHeadersMiddleware(RequestDelegate _next)
{
    public async Task InvokeAsync(HttpContext ctx)
    {
        // Request side — runs BEFORE next middleware
        await _next(ctx);  // ← pipeline executes here

        // Response side — runs AFTER everything downstream completes
        ctx.Response.Headers.Append("X-Content-Type-Options", "nosniff");
        ctx.Response.Headers.Append("X-Frame-Options",        "DENY");
        ctx.Response.Headers.Append("X-XSS-Protection",      "1; mode=block");
        ctx.Response.Headers.Append("Referrer-Policy",        "strict-origin-when-cross-origin");
    }
}

// Rate limiter — short-circuit before hitting controller
public async Task InvokeAsync(HttpContext ctx)
{
    var key    = $"rate:{ctx.Connection.RemoteIpAddress}";
    var count  = await _redis.StringIncrementAsync(key);
    if (count == 1) await _redis.KeyExpireAsync(key, TimeSpan.FromMinutes(1));

    if (count > 100)
    {
        ctx.Response.StatusCode = StatusCodes.Status429TooManyRequests;
        ctx.Response.Headers.Append("Retry-After", "60");
        return;  // Short-circuit — no next() call
    }
    await _next(ctx);
}

MVC Filter Pipeline

🔍 Filters run INSIDE MVC — access to route data, action context, model state

Middleware runs outside MVC — route data, controller name, action name — nothing visible. Filters run inside MVC — ActionContext, ModelState, return value — all accessible. Choose wisely!

MVC FILTER PIPELINE (inside app.MapControllers)
→Authorization Filters[Authorize], [RequireScope] — runs first, short-circuit if denied
→Resource Filters (before)BEFORE model binding — cache hit → short-circuit
→Model Bindingdeserialize request → action parameters
→Action Filters (before)validate, check access, log entry
→Action Executesyour controller code runs here
←Action Filters (after)modify result, add headers
←Exception FiltersMVC-only exceptions (prefer IExceptionHandler)
←Result Filters (before/after)wrap IActionResult execution
←Resource Filters (after)store response in cache

Filter Scope — Global → Controller → Action (outer to inner):

// Global — applies to ALL controllers and actions
builder.Services.AddControllers(opts =>
{
    opts.Filters.Add<AuditLogFilter>();          // Every API call logged
    opts.Filters.Add<ValidateModelStateFilter>(); // Auto 400 on invalid model
});

// Controller-level — applies to all actions in this controller
[ServiceFilter(typeof(ValidateSchoolAccessFilter))]
[ApiController]
public class StudentsController : ControllerBase { ... }

// Action-level — one specific action
[ServiceFilter(typeof(CacheSchoolDataFilter))]
[HttpGet("{schoolId}/students")]
public async Task<IActionResult> GetStudents(int schoolId) { ... }
ServiceFilter vs TypeFilter:
[ServiceFilter(typeof(MyFilter))] — filter resolved from DI. Must be registered (builder.Services.AddScoped<MyFilter>()). Supports scoped services. Preferred.

[TypeFilter(typeof(MyFilter))] — filter instantiated by ASP.NET, NOT from DI container. Can pass constructor args: [TypeFilter(typeof(MyFilter), Arguments = new object[] { "arg1" })]. Does NOT support scoped services from DI.
🎬 Action Filters — validate, check access, log — before/after action
// ZenCampus: ValidateSchoolAccessFilter
// Ensures a user can only access their OWN school's data
// Applied at controller level — no need to repeat in every action
public class ValidateSchoolAccessFilter(
    ILogger<ValidateSchoolAccessFilter> _log) : IAsyncActionFilter
{
    public async Task OnActionExecutionAsync(
        ActionExecutingContext ctx, ActionExecutionDelegate next)
    {
        // Read route value — {schoolId} from URL
        if (!ctx.ActionArguments.TryGetValue("schoolId", out var routeSchoolId))
        {
            await next(); // No schoolId in route — let it pass (admin endpoints)
            return;
        }

        // Read schoolId from JWT claim
        var claimSchoolId = ctx.HttpContext.User.FindFirst("schoolId")?.Value;
        var isAdmin       = ctx.HttpContext.User.IsInRole("SuperAdmin");

        if (!isAdmin && claimSchoolId != routeSchoolId?.ToString())
        {
            _log.LogWarning(
                "Cross-school access attempt: UserSchool={ClaimSchool} RouteSchool={RouteSchool} User={UserId}",
                claimSchoolId, routeSchoolId, ctx.HttpContext.User.Identity?.Name);

            // Short-circuit — set result directly, don't call next()
            ctx.Result = new ForbidResult();
            return;  // Action never executes
        }

        // ── Before action executes ──────────────────────────────────
        var executed = await next();  // ← Action runs HERE
        // ── After action executes ───────────────────────────────────

        if (executed.Exception is not null)
            _log.LogError(executed.Exception,
                "Action failed for SchoolId={SchoolId}", routeSchoolId);
    }
}
// AuditLogFilter — global, logs every API call with user + action + duration
public class AuditLogFilter(ILogger<AuditLogFilter> _log) : IAsyncActionFilter
{
    public async Task OnActionExecutionAsync(
        ActionExecutingContext ctx, ActionExecutionDelegate next)
    {
        var sw         = Stopwatch.StartNew();
        var controller = ctx.RouteData.Values["controller"];
        var action     = ctx.RouteData.Values["action"];
        var userId     = ctx.HttpContext.User.Identity?.Name;

        var executed   = await next();
        sw.Stop();

        var statusCode = ctx.HttpContext.Response.StatusCode;
        _log.LogInformation(
            "AUDIT: {Controller}/{Action} by {UserId} → {StatusCode} in {ElapsedMs}ms",
            controller, action, userId, statusCode, sw.ElapsedMilliseconds);
        // Every API call: AUDIT: Students/GetStudents by teacher@school7 → 200 in 45ms
    }
}

Resource Filters — Cache at MVC Layer

📦 Resource filters run BEFORE model binding — cheapest short-circuit point

Model binding = request deserialize pannum — JSON body parse, route values extract — all that work pannuvathu illa cache hit aana. Resource filter = model binding-kku munnaadi cache check panna. Cache hit? Short-circuit — controller-e reach aagaadhu!

// CacheSchoolDataFilter — return cached response without hitting controller or DB
// Applied on: GET /api/v1/schools/{schoolId}/students (read-heavy, rarely changes)
public class CacheSchoolDataFilter(IDistributedCache _cache) : IAsyncResourceFilter
{
    public async Task OnResourceExecutionAsync(
        ResourceExecutingContext ctx, ResourceExecutionDelegate next)
    {
        // Build cache key from route + query string
        var key = "action:" + ctx.HttpContext.Request.Path
                  + ctx.HttpContext.Request.QueryString;

        // ── BEFORE model binding ─────────────────────────────────────
        var cached = await _cache.GetStringAsync(key);
        if (cached is not null)
        {
            // Cache HIT — short-circuit. Controller + DB = skipped entirely
            ctx.Result = new ContentResult
            {
                Content     = cached,
                ContentType = "application/json",
                StatusCode  = 200
            };
            return;  // Don't call next() — pipeline stops here
        }

        var executed = await next();  // ← Model binding + Action runs here

        // ── AFTER action, BEFORE result execution ────────────────────
        if (executed.Exception is null && ctx.Result is ObjectResult { Value: not null } result)
        {
            // Cache the response for 5 minutes
            var json = JsonSerializer.Serialize(result.Value);
            await _cache.SetStringAsync(key, json,
                new() { AbsoluteExpirationRelativeToNow = TimeSpan.FromMinutes(5) });
        }
    }
}
// Resource filter is BETTER than action filter for caching:
// Action filter: model binding already done before "before" runs
// Resource filter: catches it BEFORE model binding — zero wasted work on cache hit

Result Filters — Wrap the Response

📤 Result filters — add headers, wrap response, transform output
// AddPaginationHeadersFilter — adds pagination metadata to response headers
// Instead of wrapping every response in PagedResult<T> wrapper object
public class PaginationHeaderFilter : IAsyncResultFilter
{
    public async Task OnResultExecutionAsync(
        ResultExecutingContext ctx, ResultExecutionDelegate next)
    {
        // Before result executes — add headers if action returned paged data
        if (ctx.HttpContext.Items["X-Total-Count"] is int total)
        {
            ctx.HttpContext.Response.Headers.Append("X-Total-Count",   total.ToString());
            ctx.HttpContext.Response.Headers.Append("X-Page-Size",    ctx.HttpContext.Items["X-Page-Size"]?.ToString() ?? "20");
            ctx.HttpContext.Response.Headers.Append("X-Has-Next-Page", ctx.HttpContext.Items["X-Has-Next"]?.ToString() ?? "false");
            // Client reads headers: X-Total-Count: 1250, X-Page-Size: 20
        }

        await next();  // Execute the result (write response body)
    }
}

// In action — set values on HttpContext.Items for the filter to pick up
[ServiceFilter(typeof(PaginationHeaderFilter))]
[HttpGet]
public async Task<IActionResult> GetStudents([FromQuery] int page = 1, [FromQuery] int size = 20)
{
    var (students, total) = await _service.GetPagedAsync(page, size);

    HttpContext.Items["X-Total-Count"] = total;
    HttpContext.Items["X-Page-Size"]   = size;
    HttpContext.Items["X-Has-Next"]    = (page * size) < total;

    return Ok(students);  // Clean response body — no pagination wrapper
}

🎯 10 Interview Scenarios

Q1. What's the difference between Middleware and Filters? Which do you use for: (a) rate limiting, (b) school access validation?
Concept

Key distinction: Middleware runs outside MVC — before routing decisions, before model binding, no access to route values or action context. Filters run inside MVC — have full access to controller name, action name, route data, model state, and return value.

(a) Rate limiting → Middleware
Runs before routing → applies to ALL requests (static files, health checks, API). Needs to fire before any MVC processing wastes CPU. Can check IP-based or API-key-based limits without knowing which controller will handle it.

(b) School access validation → Action Filter
Needs the schoolId route value from URL (/api/students/{schoolId}/...). This is only available after routing resolves. Also only applies to MVC controllers — not static files. [ServiceFilter(typeof(ValidateSchoolAccessFilter))] on the controller.

Q2. PRODUCTION BUG: All [Authorize] endpoints returning 401. Unauthenticated endpoints work. What's wrong?
Production Bug

Root cause: UseAuthorization() is registered before UseAuthentication() in Program.cs. Authorization runs first — User is not yet populated (JWT not yet parsed) — appears anonymous — every [Authorize] check fails.

// ❌ Wrong order
app.UseAuthorization();    // Runs BEFORE auth — User.Identity.IsAuthenticated = false
app.UseAuthentication();   // Too late — authorization already decided

// ✅ Correct order — immutable rule
app.UseAuthentication();   // 1. Parse JWT → populate HttpContext.User
app.UseAuthorization();    // 2. Check HttpContext.User claims → allow/deny

Why unauthenticated endpoints work: Endpoints without [Authorize] don't trigger the authorization check — they pass through regardless of order. Only [Authorize] endpoints consult the User object.

Q3. How does short-circuiting work in middleware? Write TenantResolutionMiddleware that short-circuits for invalid schools.
Design

Short-circuit = don't call await next(ctx). Instead write a response directly and return. Everything downstream (more middleware, controllers) never runs.

public async Task InvokeAsync(HttpContext ctx, RequestDelegate next)
{
    var schoolIdClaim = ctx.User.FindFirst("schoolId")?.Value;

    if (schoolIdClaim is null)
    {
        await next(ctx); // Anonymous user — let auth middleware handle it
        return;
    }

    var school = await _tenantService.GetSchoolAsync(int.Parse(schoolIdClaim));
    if (school is null)
    {
        ctx.Response.StatusCode = 403;
        await ctx.Response.WriteAsJsonAsync(new { error = "Invalid school" });
        return;  // ← SHORT-CIRCUIT: no next(), no controller, no nothing
    }

    ctx.Items["Tenant"] = school;
    await next(ctx);  // CONTINUE: valid school, pass to controller
}

Key rule: Either call next(ctx) (continue) or write response and return (short-circuit). Never do both — writing to an already-started response causes errors.

Q4. Why use a Resource Filter for caching instead of an Action Filter?
Performance

Resource filter runs BEFORE model binding. Action filter runs AFTER model binding.

Model binding in a complex request: deserialize JSON body (allocations), extract route values, validate types, map to action parameters. This work happens BEFORE the action filter's "before" method.

For a cache hit: why do model binding at all? Resource filter avoids the wasted work entirely.

Filter TypeCache Hit — what still runs?Wasted work?
Action Filter (OnActionExecuting)Authorization + Resource + Model Binding + Action Filter (before)❌ Model binding still happens
Resource Filter (OnResourceExecuting)Authorization filter only✅ Model binding skipped entirely

For ZenCampus GET /students — model binding is cheap. But for POST with 50KB JSON body — resource filter caching saves real allocations.

Q6. Global filter vs controller filter vs action filter — what's the execution order when all three apply?
Concept

Outer to inner on the way IN, inner to outer on the way OUT.

// Order: Global → Controller → Action (before)
//        Action → Controller → Global (after)

// 1. [Global] AuditLogFilter.OnActionExecuting     ← first
// 2. [Controller] ValidateSchoolAccessFilter.OnActionExecuting
// 3. [Action] CacheSchoolDataFilter.OnActionExecuting
// ── ACTION EXECUTES ──
// 4. [Action] CacheSchoolDataFilter.OnActionExecuted  ← innermost first on way out
// 5. [Controller] ValidateSchoolAccessFilter.OnActionExecuted
// 6. [Global] AuditLogFilter.OnActionExecuted       ← outermost last on way out

// Set Order property to override default (lower Order = runs earlier on way in)
public class AuditLogFilter : IAsyncActionFilter, IOrderedFilter
{
    public int Order => -1000;  // Very low = runs first in, last out
}
Q7. How do you inject scoped services into middleware? What's the trap with conventional middleware?
Design

Trap: Conventional middleware is a singleton (constructor called once at startup). Injecting a scoped service (DbContext, IStudentRepository) in the constructor causes captive dependency bug — scoped object held for the app lifetime → data leaks between requests, potential concurrency errors.

// ❌ WRONG — scoped DbContext injected in constructor (singleton lifetime!)
public class TenantMiddleware(RequestDelegate _next, ZenCampusDbContext _db) { ... }

// ✅ Fix Option 1: InvokeAsync parameter injection (DI creates per-request)
public class TenantMiddleware(RequestDelegate _next)
{
    public async Task InvokeAsync(HttpContext ctx, ZenCampusDbContext db)
    // ↑ DI injects scoped DbContext per-request here — SAFE
    {
        var school = await db.Schools.FindAsync(...);
        await _next(ctx);
    }
}

// ✅ Fix Option 2: IMiddleware — registered as scoped, full constructor DI
public class TenantMiddleware(ZenCampusDbContext _db) : IMiddleware
// Register: builder.Services.AddScoped<TenantMiddleware>()

// ✅ Fix Option 3: ctx.RequestServices (explicit scope)
var db = ctx.RequestServices.GetRequiredService<ZenCampusDbContext>();
Q8. ValidateModelStateFilter — how do you avoid writing ModelState check in every action?
Design

Global Action Filter checks ModelState before every action — if invalid, returns 400 automatically.

// ❌ Without filter — repeated in every action
public IActionResult AdmitStudent([FromBody] AdmitStudentDto dto)
{
    if (!ModelState.IsValid) return BadRequest(ModelState); // Repeated 100 times!
    ...
}

// ✅ Global filter — write once, applies everywhere
public class ValidateModelStateFilter : IActionFilter
{
    public void OnActionExecuting(ActionExecutingContext ctx)
    {
        if (!ctx.ModelState.IsValid)
            ctx.Result = new UnprocessableEntityObjectResult(ctx.ModelState); // 422
    }
    public void OnActionExecuted(ActionExecutedContext ctx) { }
}

// Program.cs
builder.Services.AddControllers(opts => opts.Filters.Add<ValidateModelStateFilter>());

// OR use [ApiController] — it already does this automatically!
// [ApiController] attribute enables automatic ModelState validation → 400 response
// If you have [ApiController], ValidateModelStateFilter is redundant

Important: [ApiController] already auto-validates ModelState. Manual ValidateModelStateFilter only needed when NOT using [ApiController].

Q9. You need to add X-School-Name header to ALL responses from SchoolsController. Best approach?
Design

Result Filter at controller level — runs after action completes, access to HttpContext to add headers.

public class SchoolNameHeaderFilter : IAsyncResultFilter
{
    public async Task OnResultExecutionAsync(
        ResultExecutingContext ctx, ResultExecutionDelegate next)
    {
        // HttpContext.Items["Tenant"] set by TenantResolutionMiddleware
        if (ctx.HttpContext.Items["Tenant"] is School school)
            ctx.HttpContext.Response.Headers.Append("X-School-Name", school.Name);

        await next(); // Write the response body
    }
}

// Apply to controller
[ServiceFilter(typeof(SchoolNameHeaderFilter))]
public class SchoolsController : ControllerBase { ... }

// Why not Action Filter? Result filter runs after result object is set — headers

Q10. "Describe your middleware and filter usage in ZenCampus." — Full structured answer
Experience

"ZenCampus serves 200 schools and 50,000 students across multiple tenants. We built a layered pipeline using both middleware and MVC filters based on what each needs to access.

Middleware layer: RequestTimingMiddleware logs any request exceeding 500ms — runs for all requests including health checks. TenantResolutionMiddleware (IMiddleware pattern, scoped) resolves the school from the JWT claim and populates HttpContext.Items["Tenant"] — this runs after UseAuthentication so the JWT is already parsed. SecurityHeadersMiddleware adds X-Frame-Options, X-Content-Type-Options to every response on the way out.

Action Filter layer: ValidateSchoolAccessFilter is applied at controller level on all school-scoped controllers — it compares the schoolId route parameter with the JWT claim. If a school 7 teacher tries accessing school 8's data, it logs a warning and short-circuits with 403. This saved us from a cross-tenant data exposure risk we caught in testing.

Resource Filter: CacheSchoolDataFilter on GET student list endpoints — checks Redis before model binding. Cache hit = controller never runs. On miss, stores the result after action completes.

Global filters: AuditLogFilter is registered globally — logs every controller/action/user/status/duration. ValidateModelStateFilter was removed when we added [ApiController] — which does it automatically."

🧠 Quick Quiz

Q1. You need to validate that the schoolId in the URL matches the user's JWT claim. Best implementation?
Action Filter — schoolId route parameter is only accessible after routing resolves, inside the MVC pipeline. Middleware runs BEFORE routing so route values are unavailable. Resource filter also has route values, but action filter is the standard choice for pre-action validation. Result filter runs too late — action already executed.
Q2. You want to cache entire action responses in Redis. For a cache HIT, what should NOT run at all?
Model binding AND controller action — both skipped via Resource Filter. Resource filter runs BEFORE model binding so on cache hit: authentication runs (always), authorization runs (always), then resource filter short-circuits — model binding and controller action never execute. This is why Resource Filter is more efficient than Action Filter for caching.
Q3. IMiddleware vs convention-based middleware — why does IMiddleware safely support scoped DbContext injection?
IMiddleware is resolved by DI per request — the container creates a new instance (or scoped instance) for each request, so scoped services like DbContext get proper per-request lifetime. Convention middleware constructor is called ONCE at startup (singleton), so injecting scoped services causes captive dependency — the scoped object is held forever, leaking data between requests.
Q4. Three filters applied: Global AuditLogFilter, Controller ValidateSchoolFilter, Action CacheFilter. What runs FIRST when request arrives?
Global filter runs first on the way IN (outer → inner: Global → Controller → Action). On the way OUT it's reversed (inner → outer: Action → Controller → Global). This is the "matryoshka doll" / Russian nesting doll model — outermost wraps everything. Exception: you can override this with the Order property (lower Order number = runs first).
