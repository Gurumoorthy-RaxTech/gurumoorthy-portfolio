💡 Interview Rule — Catch specific, not general

Specific exceptions first, general last. catch(SqlException) before catch(Exception). Compiler enforces order — base class catch before derived = compile error!

2 try/catch/finally Patterns — 5 Rules

✅ Correct Rethrow — preserve stack trace

// ❌ WRONG — stack trace LOST!
catch (Exception ex)
{
    _logger.LogError(ex, "Error");
    throw ex; // resets stack trace!
}

// ✅ CORRECT — stack trace preserved
catch (Exception ex)
{
    _logger.LogError(ex, "Error");
    throw; // bare throw — preserves!
}

// ✅ Wrap with context (InnerException)
catch (SqlException ex)
{
    throw new DataAccessException(
        $"Failed loading student {id}", ex);
    // original SqlException = InnerException
}

✅ finally — always runs

SqlConnection conn = null;
try
{
    conn = new SqlConnection(cs);
    conn.Open();
    // ... work ...
}
catch (SqlException ex)
{
    _logger.LogError(ex, "DB error");
    throw;
}
finally
{
    conn?.Dispose(); // ALWAYS runs!
    // even if exception thrown
    // even if return in try
}

// Modern: using replaces try/finally
using var conn = new SqlConnection(cs);

When to catch, when NOT to:

SituationActionWhy
You can HANDLE itcatch + handleRetry, fallback, user message
Need to LOG + rethrowcatch + log + throwAdd context, preserve trace
Need to add contextcatch + wrap new ex(inner)Business message + original cause
Cannot handleDON'T catchLet it bubble up — don't swallow!
Resource cleanup onlyfinally (or using)No catch needed for cleanup
⚠️ Exception Anti-patterns — Never Do These!
// ❌ 1. Swallow exception — worst!
catch (Exception) { } // silent failure — impossible to debug

// ❌ 2. Catch Exception for control flow
try { int n = int.Parse(input); }
catch { n = 0; } // use int.TryParse instead!

// ❌ 3. throw ex — loses stack trace
catch (Exception ex) { throw ex; } // use bare throw!

// ❌ 4. return in finally
finally { return null; } // swallows exception silently!

// ❌ 5. Catch Exception catch-all in library
catch (Exception) { return null; } // hides bugs from callers

Exception Filters — when keyword (C# 6+):

// when = catch only if condition true
catch (SqlException ex) when (ex.Number == -2)
{
    // Only catch SQL timeout (error -2)
    await _retryPolicy.ExecuteAsync(() => _repo.QueryAsync());
}

// ZenCampus — catch only transient errors
catch (HttpRequestException ex)
    when (ex.StatusCode is HttpStatusCode.ServiceUnavailable
       or HttpStatusCode.TooManyRequests)
{
    await Task.Delay(1000);
    throw; // retry upstream
}
3 Custom Exceptions — ZenCampus Pattern
🎬 Vikram (2022) — Specific villain, specific response

Villain type பொறுத்து Vikram different tactics use பண்ணுவான். Custom exceptions same — caller specific exception type catch பண்ணி specific response குடுக்கலாம். Generic Exception = all villains same response — useless!

// ✅ ZenCampus Custom Exception Hierarchy
public class ZenCampusException : Exception
{
    public string ErrorCode { get; }

    public ZenCampusException(string message, string errorCode)
        : base(message) => ErrorCode = errorCode;

    public ZenCampusException(string message, string errorCode, Exception inner)
        : base(message, inner) => ErrorCode = errorCode;
}

public class StudentNotFoundException : ZenCampusException
{
    public int StudentId { get; }
    public StudentNotFoundException(int id)
        : base($"Student {id} not found", "STU_404")
        => StudentId = id;
}

public class FeeValidationException : ZenCampusException
{
    public decimal AttemptedAmount { get; }
    public FeeValidationException(decimal amount, string reason)
        : base($"Fee validation failed: {reason}", "FEE_INVALID")
        => AttemptedAmount = amount;
}

public class DuplicateAdmissionException : ZenCampusException
{
    public string RollNumber { get; }
    public DuplicateAdmissionException(string roll)
        : base($"Student {roll} already admitted", "ADM_DUP")
        => RollNumber = roll;
}

// Usage in Service
public async Task<Student> GetStudentAsync(int id)
{
    var student = await _repo.GetByIdAsync(id);
    return student ?? throw new StudentNotFoundException(id);
}

// Controller — specific catches → specific HTTP responses
try { return Ok(await _svc.GetStudentAsync(id)); }
catch (StudentNotFoundException ex)  { return NotFound(ex.Message); }
catch (FeeValidationException ex)    { return BadRequest(ex.Message); }
catch (ZenCampusException ex)        { return StatusCode(500, ex.ErrorCode); }
💡 Exception Middleware — Don't catch everywhere!

ASP.NET Core-la UseExceptionHandler middleware — one place exception handle pannuvu. Controller-la catch pannaa vendam! Exception filter or global middleware = cleaner.

4 Generics <T> — Type-safe Reusable Code Interview Fav!
🎬 Dasavathaaram — One Kamal, 10 types

Same actor, different roles — generic method same! Repository<T> one class — Student, Teacher, Fee — all serve pannuvu. No duplicate code!

// Without generics — duplicate code nightmare!
public Student GetById(int id) { ... }
public Teacher GetById(int id) { ... } // duplicate!
public FeeRecord GetById(int id) { ... } // more duplicate!

// ✅ Generic Repository — one class, all types!
public class Repository<T> where T : class
{
    private readonly DbContext _ctx;
    public Repository(DbContext ctx) => _ctx = ctx;

    public async Task<T?> GetByIdAsync(int id)
        => await _ctx.Set<T>().FindAsync(id);

    public async Task<List<T>> GetAllAsync()
        => await _ctx.Set<T>().ToListAsync();

    public async Task AddAsync(T entity)
    {
        await _ctx.Set<T>().AddAsync(entity);
        await _ctx.SaveChangesAsync();
    }
}

// Usage — same class, different types!
var studentRepo = new Repository<Student>(ctx);
var teacherRepo = new Repository<Teacher>(ctx);
var feeRepo     = new Repository<FeeRecord>(ctx);

public class AuditRepository<T>
    where T : class, IEntity, new()
{
    public T CreateWithAudit()
    {
        var entity = new T(); // new() constraint allows this
        // entity.Id, entity.CreatedAt — IEntity constraint allows this
        return entity;
    }

// Generic method with constraint
public T GetOrThrow<T>(T? value, string name)
    where T : class
    => value ?? throw new ArgumentNullException(name);

// Generic Result pattern — no exceptions for expected failures
public class Result<T>
{
    public T? Value { get; }
    public string? Error { get; }
    public bool IsSuccess => Error == null;

    public static Result<T> Ok(T val) => new() { Value = val };
    public static Result<T> Fail(string err) => new() { Error = err };
}

// No exception thrown — caller checks Result
public Result<Student> AdmitStudent(AdmissionDto dto)
{
    if (_repo.Exists(dto.RollNo))

5 IDisposable + using — Resource Management ♻️
🎬 Kaithi (2019) — Escape before dawn

Dilli resources (vehicles, phone) use பண்ணிட்டு mission முடிஞ்சதும் release பண்ணுவான். IDisposable same — DB connection, file handle, HTTP client — use பண்ணிட்டு Dispose() call பண்ணலைன்னா leak!

using statement — auto Dispose

// using block — Dispose at end of block
using (var conn = new SqlConnection(cs))
{
    conn.Open();
    // use conn
} // conn.Dispose() called here!

// Modern C# 8+ — using declaration
using var conn = new SqlConnection(cs);
conn.Open();
// Dispose at end of enclosing scope
// cleaner — no extra nesting!

// Multiple using
using var conn   = new SqlConnection(cs);
using var cmd    = conn.CreateCommand();
using var reader = await cmd.ExecuteReaderAsync();

IDisposable Implementation

public class ZenCampusDbContext : IDisposable
{
    private SqlConnection _conn;
    private bool _disposed;

    public void Dispose()
    {
        Dispose(true);
        GC.SuppressFinalize(this);
    }

    protected virtual void Dispose(bool disposing)
    {
        if (_disposed) return;
        if (disposing)
            _conn?.Dispose(); // managed
        // unmanaged resources here
        _disposed = true;
    }

    // Finalizer — last resort (GC calls)
    ~ZenCampusDbContext()
        => Dispose(false);
}

IAsyncDisposable — .NET 6+ async cleanup:

public class SmsService : IAsyncDisposable
{
    private readonly HttpClient _httpClient;

    public async ValueTask DisposeAsync()
    {
        await _httpClient.GetAsync("/flush"); // async cleanup!
        _httpClient.Dispose();
    }
}

// await using — for IAsyncDisposable
await using var sms = new SmsService();
await sms.SendAsync(mobile, msg);
// DisposeAsync() called automatically!

6 🎯 Scenario-Based Interview Questions
1
throw ex vs throw — Stack Trace difference Easy
Topic: Most asked basic exception question
📋 Scenario: Junior developer இந்த code எழுதினான். Production-la exception varuthu — stack trace line numbers wrong school point பண்ணுது! Root cause enna? 
try
{
    await _repo.SaveStudentAsync(student); // line 45
}
catch (Exception ex)
{
    _logger.LogError(ex, "Save failed");
    throw ex; // ← problem here!
}
✅ ANSWER

Problem: throw ex — stack trace reset பண்ணிடுது! Original exception-a line 45-la throw aachu — aaana throw ex stack trace-a catch block line-ku reset பண்ணுது. Stack trace useless aagiduchu!

// ❌ throw ex — resets stack trace to THIS line
catch (Exception ex) { throw ex; }

// ✅ throw — preserves original stack trace
catch (Exception ex)
{
    _logger.LogError(ex, "Save failed");
    throw; // bare throw = preserve line 45!
}

// ✅ Or wrap with inner (add context)
catch (SqlException ex)
{
    throw new DataException("Student save failed", ex);
    // ex = InnerException, original trace preserved inside
}
⭐ Interview One-liner

"throw ex = stack trace reset. throw = stack trace preserved. Always use bare throw when rethrowing. Wrap in new Exception when adding context — original becomes InnerException."

Exception Swallowing — Silent Bug Medium
Topic: Worst anti-pattern — catch + ignore
📋 Scenario: ZenCampus-la fee payment page — sometimes payment process aagiduchu aaana student fee still shows unpaid! No error log, no exception visible. Code: 
public async Task ProcessPaymentAsync(PaymentDto dto)
{
    try
    {
        await _paymentGateway.ChargeAsync(dto);
        await _feeRepo.MarkPaidAsync(dto.StudentId);
    }
    catch (Exception) { } // ← PROBLEM!
}
✅ ROOT CAUSE + FIX

Root Cause: Empty catch block = silent exception swallow. Payment gateway charge pannuthu, MarkPaid throw pannuthu → silently ignored → student fee = unpaid. No log, no trace, no alert!

// ✅ Fix — never swallow exceptions
public async Task<PaymentResult> ProcessPaymentAsync(PaymentDto dto)
{
    try
    {
        await _paymentGateway.ChargeAsync(dto);
        await _feeRepo.MarkPaidAsync(dto.StudentId);
        return PaymentResult.Success();
    }
    catch (PaymentGatewayException ex)
    {
        _logger.LogError(ex, "Gateway failed for {StudentId}", dto.StudentId);
        return PaymentResult.GatewayError(ex.Message);
    }
    catch (Exception ex)
    {
        _logger.LogError(ex, "Unexpected payment error");
        throw; // re-throw unexpected — don't swallow!
    }
}

Generic Repository — No Duplicate Code Medium
Topic: Generics for DRY principle
📋 Scenario: ZenCampus-la StudentRepository, TeacherRepository, FeeRepository — all same CRUD methods! Code duplicate. Generic base repository design pannu + DI-la register panna epdi?
✅ ANSWER
// Base entity interface
public interface IEntity { int Id { get; set; } }

// Generic interface
public interface IRepository<T> where T : IEntity
{
    Task<T?> GetByIdAsync(int id);
    Task<List<T>> GetAllAsync();
    Task AddAsync(T entity);
    Task UpdateAsync(T entity);
    Task DeleteAsync(int id);
}

// Generic implementation — ONE class for all!
public class EfRepository<T> : IRepository<T>
    where T : class, IEntity
{
    private readonly AppDbContext _ctx;
    private readonly DbSet<T> _set;

    public EfRepository(AppDbContext ctx)
    {
        _ctx = ctx;
        _set = ctx.Set<T>();
    }

    public async Task<T?> GetByIdAsync(int id)
        => await _set.FindAsync(id);

    public async Task<List<T>> GetAllAsync()
        => await _set.ToListAsync();

    public async Task AddAsync(T entity)
    {
        await _set.AddAsync(entity);
        await _ctx.SaveChangesAsync();
    }
    // UpdateAsync, DeleteAsync same pattern...
}

// DI Registration — Program.cs
builder.Services.AddScoped(typeof(IRepository<>), typeof(EfRepository<>));
// One line registers for ALL types — Student, Teacher, Fee, etc.!

// Specific repo with extra methods — inherit!
public class StudentRepository : EfRepository<Student>
{
    public async Task<List<Student>> GetByClassAsync(string className)
        => await _set.Where(s => s.Class == className).ToListAsync();
}

DB Connection Leak — using fix Easy
Topic: IDisposable + connection leak
📋 Scenario: ZenCampus app — 2 hours run aana connection pool exhausted! "All pooled connections are in use." Log-la "Maximum pool size was reached" varuthu. Code: 
public async Task<List<Student>> GetStudentsAsync()
{
    var conn = new SqlConnection(_cs); // not disposed!
    await conn.OpenAsync();
    var result = await conn.QueryAsync<Student>("SELECT...");
    return result.ToList();
    // conn NEVER disposed → pool exhausted!
}
✅ FIX
// ✅ using var — auto Dispose even on exception
public async Task<List<Student>> GetStudentsAsync()
{
    await using var conn = new SqlConnection(_cs);
    await conn.OpenAsync();
    var result = await conn.QueryAsync<Student>("SELECT...");
    return result.ToList();
} // conn.DisposeAsync() here — returns to pool!
⭐ SqlConnection — pooling magic

SqlConnection.Dispose() doesn't close physically — returns to pool! Pool = reusable connections. Not disposing = pool exhaustion. Always using var SqlConnection!

7
Generic Result Pattern — No Exception for Expected Failures Senior
Topic: Railway-oriented programming concept
📋 Scenario: Interview-la: "Exceptions are expensive. Expected failures-ku (validation fail, not found) exception throw பண்றீங்க — better approach enna?" Senior answer expected.
✅ SENIOR ANSWER — Result<T> Pattern
// Exceptions = for UNEXPECTED failures (DB down, network error)
// Result<T> = for EXPECTED failures (validation, not found)
public record Result<T>
{
    public T? Value { get; init; }
    public string? Error { get; init; }
    public bool IsSuccess => Error == null;

    public static Result<T> Ok(T val)       => new() { Value = val };
    public static Result<T> Fail(string err) => new() { Error = err };
}

public async Task<Result<Student>> AdmitStudentAsync(AdmissionDto dto)
{
    if (string.IsNullOrEmpty(dto.Name))
        return Result<Student>.Fail("Name required"); // no exception!

    if (await _repo.ExistsAsync(dto.RollNo))
        return Result<Student>.Fail("Already admitted");

    var student = _mapper.Map<Student>(dto);
    await _repo.AddAsync(student); // this CAN throw (unexpected DB fail)
    return Result<Student>.Ok(student);
}

// Controller — no try/catch for business failures
var result = await _svc.AdmitStudentAsync(dto);
return result.IsSuccess
    ? Ok(result.Value)
    : BadRequest(result.Error);
⭐ Why Result pattern is senior-level

"Exceptions are expensive — stack unwind, heap allocation. For expected failures (validation, business rules) → Result<T>. For unexpected failures (DB down, null bug) → exceptions. This distinction = clean error handling design."

Exception Filter — Catch only specific SQL errors Medium
Topic: when keyword — exception filters
📋 Scenario: ZenCampus DB layer — SQL timeout-ku retry பண்ணணும், aaana other SQL errors (constraint violation, connection fail) retry பண்ணக்கூடாது. when filter use பண்ணு.
✅ ANSWER
private const int SqlTimeoutError  = -2;
private const int SqlDeadlockError  = 1205;

public async Task<List<Student>> GetStudentsAsync()
{
    int retryCount = 0;
    while (true)
    {
        try
        {
            return await _repo.GetAllAsync();
        }
        // ONLY catch timeout or deadlock — other SQL errors bubble up!
        catch (SqlException ex)
            when ((ex.Number == SqlTimeoutError
               || ex.Number == SqlDeadlockError)
               && retryCount < 3)
        {
            retryCount++;
            await Task.Delay(retryCount * 500); // 500ms, 1s, 1.5s
            _logger.LogWarning("Retry {n} for SQL {err}", retryCount, ex.Number);
        }
    }
}

Interview — "Explain your exception strategy in ZenCampus" Senior
Topic: Complete design-level answer
📋 Scenario: "Your project-la exception handling எப்படி design பண்ணீங்க? Global vs local — explain."
✅ COMPLETE INTERVIEW ANSWER

"ZenCampus-la 3-layer exception strategy:"

Layer 1 — Global Middleware: UseExceptionHandler middleware — all unhandled exceptions catch பண்ணி log பண்ணி 500 return பண்ணும். Controllers-la no try/catch for unexpected errors!

Layer 2 — Domain Exceptions: StudentNotFoundException, FeeValidationException — specific exceptions throw பண்ணுவோம். Controller-la specific catch → specific HTTP status code (404, 400, 409).

Layer 3 — Result<T> for expected: Validation failures, business rule violations — exceptions போடாம Result<T> return பண்ணுவோம். No exception overhead for expected outcomes.

Golden rules: throw not throw ex. Never swallow. Always using for IDisposable. Custom exceptions for domain errors. Exception filters for specific retry logic.

7 ⚡ Quick Quiz

Q1: throw ex vs throw — difference?

A) No difference
B) throw ex resets stack trace; bare throw preserves it
C) throw ex is faster
D) throw ex wraps in new exception
✅ throw ex = new stack trace from catch block. throw = original stack trace preserved. ALWAYS use bare throw when rethrowing!
❌ throw ex resets! Stack trace now points to catch line not original throw site. throw (bare) = preserve. Senior difference!

Q2: using var conn — when does Dispose() get called?

A) When GC runs
B) Manually only
C) End of enclosing scope — always, even on exception
D) After method returns
✅ using var = Dispose() at end of enclosing scope. Exception thrown in the middle? Still Dispose()! Same as try/finally but cleaner syntax.
❌ using var = end of enclosing scope ALWAYS. Exception or return — doesn't matter. Guaranteed cleanup!

Q3: Generic constraint where T : new() — enna pannum?

A) T must be new keyword
B) T must have a parameterless constructor — allows new T() in code
C) T cannot be created
D) T is always reference type
✅ new() constraint = T has parameterless constructor. Allows var x = new T() inside generic class/method. Without it = compiler error on new T()!
❌ new() = parameterless constructor required. Allows you to write new T() in generic code. Without this constraint, compiler won't allow new T().

Q4: finally block — எப்போது run ஆகாது?

A) Always runs
B) Environment.FailFast() or power failure — process killed
C) When exception is caught
D) When return is in try
✅ finally skipped only on: Environment.FailFast() (immediate process kill), StackOverflowException, ExecutionEngineException, or system crash. Normal exceptions + return = finally ALWAYS runs!
❌ finally = almost always runs! Exceptions, returns, break — doesn't matter. Only NOT runs on: FailFast(), fatal CLR errors, or OS kill. Normal code = always runs.
