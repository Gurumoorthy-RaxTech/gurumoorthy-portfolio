2 IHostedService — Startup & Shutdown Hooks
// IHostedService — lifecycle hooks into ASP.NET Core host
public class CacheWarmupService : IHostedService
{
    private readonly IServiceProvider _sp;
    private readonly ILogger<CacheWarmupService> _log;

    // StartAsync — called when app starts (before first HTTP request!)
    public async Task StartAsync(CancellationToken ct)
    {
        _log.LogInformation("Warming up ZenCampus caches...");

        // IHostedService is Singleton — use IServiceScope for Scoped services!
        using var scope   = _sp.CreateScope();
        var schoolSvc      = scope.ServiceProvider.GetRequiredService<ISchoolService>();

        var activeSchools   = await schoolSvc.GetAllActiveAsync();
        await Parallel.ForEachAsync(activeSchools,
            new ParallelOptions { MaxDegreeOfParallelism = 5, CancellationToken = ct },
            async (s, _) => await schoolSvc.WarmCacheAsync(s.Id));

        _log.LogInformation("Cache warm-up complete: {Count} schools", activeSchools.Count);
    }

    // StopAsync — called on graceful shutdown (SIGTERM, app stop)
    public Task StopAsync(CancellationToken ct)
    {
        _log.LogInformation("ZenCampus shutting down gracefully");
        return Task.CompletedTask;
    }
}

// Register
builder.Services.AddHostedService<CacheWarmupService>();
⚠️ IHostedService lifetime trap — Scoped services!

IHostedService is registered as Singleton. You CANNOT inject Scoped services (DbContext, repositories) directly — lifetime mismatch! Always use IServiceProvider.CreateScope() to resolve scoped services inside. This is the most common mistake in background service code!

3 BackgroundService — Long-Running Loops
🎬 Ratsasan — Systematic background investigation, never stops, checks evidence periodically.

Ratsasan investigation loop = BackgroundService. Runs forever. Checks evidence (queue) every N seconds. Processes what's found. Sleeps. Repeats. CancellationToken = case closed (app shutting down).

// BackgroundService — abstract class, override ExecuteAsync
public class AttendanceSyncService : BackgroundService
{
    private readonly IServiceProvider _sp;
    private readonly ILogger<AttendanceSyncService> _log;

    // ExecuteAsync — runs on a background thread, loop until cancellation
    protected override async Task ExecuteAsync(CancellationToken ct)
    {
        _log.LogInformation("Attendance sync service started");

        while (!ct.IsCancellationRequested)
        {
            try
            {
                await ProcessPendingAttendanceAsync(ct);
            }
            catch (OperationCanceledException) when (ct.IsCancellationRequested)
            {
                break; // graceful shutdown — don't log as error!
            }
            catch (Exception ex)
            {
                // Log but KEEP RUNNING — don't let one failure kill the service!
                _log.LogError(ex, "Attendance sync failed — retrying in 30s");
            }

            // Wait before next poll (use ct so Delay exits on shutdown!)
            await Task.Delay(TimeSpan.FromSeconds(30), ct);
        }

        _log.LogInformation("Attendance sync service stopped");
    }

    private async Task ProcessPendingAttendanceAsync(CancellationToken ct)
    {
        using var scope = _sp.CreateScope(); // new scope per iteration!
        var svc         = scope.ServiceProvider.GetRequiredService<IAttendanceSyncService>();
        var pending     = await svc.GetPendingAsync(ct);

        if (!pending.Any()) return; // nothing to do

        _log.LogInformation("Processing {Count} attendance records", pending.Count);
        await svc.ProcessBatchAsync(pending, ct);
    }
}

builder.Services.AddHostedService<AttendanceSyncService>();
💡 Key Rules for BackgroundService

1. Always pass CancellationToken to Task.Delay — else app hangs on shutdown waiting for delay!
2. New IServiceScope per iteration — prevents DbContext connection leaks
3. Catch exceptions inside loop — one failure should NOT kill the entire service
4. Separate OperationCanceledException from real errors — graceful shutdown ≠ error

5 Channel<T> — In-Process Producer/Consumer Queue
🎬 Vikram — Message relay between agents. Producer drops message, consumer picks up. No direct interaction needed.

Amar drops attendance data into channel. Vikram (consumer) picks it up, processes it. HTTP request (producer) doesn't wait for consumer. Decoupled! Channel = in-memory, thread-safe, backpressure-aware queue.

// Attendance event model
public record AttendanceEvent(
    int      StudentId,
    int      SchoolId,
    DateOnly Date,
    bool     IsPresent,
    string?  Remarks
);

// Register Channel as Singleton (shared between producer and consumer)
builder.Services.AddSingleton(Channel.CreateBounded<AttendanceEvent>(
    new BoundedChannelOptions(capacity: 1000)
    {
        FullMode          = BoundedChannelFullMode.Wait, // backpressure!
        SingleReader      = true,                         // 1 consumer
        SingleWriter      = false                         // many producers
    }));

// Producer — in the controller/API handler
public class AttendanceController : ControllerBase
{
    private readonly ChannelWriter<AttendanceEvent> _writer;

    [HttpPost]
    public async Task<IActionResult> MarkAttendanceAsync(AttendanceDto dto)
    {
        var evt = new AttendanceEvent(
            dto.StudentId, dto.SchoolId,
            DateOnly.FromDateTime(DateTime.Today),
            dto.IsPresent, dto.Remarks);

        // Non-blocking write — returns immediately!
        await _writer.WriteAsync(evt); // waits only if channel full (backpressure)

        return Accepted(); // 202 — queued, processing in background
    }
}

// Consumer — BackgroundService
public class AttendanceProcessorService : BackgroundService
{
    private readonly ChannelReader<AttendanceEvent> _reader;
    private readonly IServiceProvider _sp;

    protected override async Task ExecuteAsync(CancellationToken ct)
    {
        // ReadAllAsync — async stream, blocks until item available
        await foreach (var evt in _reader.ReadAllAsync(ct))
        {
            try
            {
                using var scope = _sp.CreateScope();
                var svc         = scope.ServiceProvider
                    .GetRequiredService<IAttendanceService>();

                await svc.SaveAsync(evt);

                // Notify parent if absent (fire-and-forget SMS)
                if (!evt.IsPresent)
                    BackgroundJob.Enqueue<ISmsService>(s =>
                        s.SendAbsentAlertAsync(evt.StudentId));
            }
            catch (Exception ex)
            {
                _log.LogError(ex, "Failed to process attendance for Student {Id}", evt.StudentId);
                // Continue — don't let one failure block the channel!
            }
        }
    }
}

            CreatedAt = jobDetails?.CreatedAt,
        });
    }

// Report service — update progress in Redis
public async Task GenerateAsync(int schoolId, string year, PerformContext? ctx)
{
    // PerformContext — access to Hangfire job context
    ctx?.WriteLine($"Starting report for school {schoolId}");

    var students = await _ctx.Students.Where(s => s.SchoolId == schoolId).ToListAsync();
    var total    = students.Count;

    for (int i = 0; i < total; i++)
    {
        await ProcessStudentReportAsync(students[i]);

        // Update progress every 100 students
        if (i % 100 == 0)
        {
            var pct = (i * 100) / total;
            ctx?.WriteLine($"Progress: {pct}% ({i}/{total})");
            await _redis.StringSetAsync($"job:progress:{ctx?.BackgroundJob.Id}", pct);
        }
    }
}

Monthly fee reminder — 50,000 parents, SMS gateway rate limit Medium
📋 Scenario: "1st of every month — 50,000 parents-ku fee pending SMS remind பண்ணணும். SMS gateway = max 100 SMS/second. Naive approach = all 50k at once → rate limit error. Design the job."
✅ BATCH + THROTTLED RECURRING JOB
// Recurring job — triggers orchestrator on 1st of month
[AutomaticRetry(Attempts = 0)] // orchestrator shouldn't retry
public async Task SendMonthlyRemindersAsync()
{
    // Load all pending fees — students who owe money
    var pending = await _ctx.Students
        .Where(s => s.Status == "Active" && s.PendingFees > 0)
        .Select(s => new { s.Id, s.ParentPhone, s.PendingFees })
        .ToListAsync();

    _log.LogInformation("Scheduling {Count} fee reminder SMSes", pending.Count);

    // Batch into groups of 100 (gateway rate = 100/sec)
    // Each batch = 1 Hangfire job, delay = batchIndex seconds
    var batches = pending.Chunk(100);
    int delaySeconds = 0;

    foreach (var batch in batches)
    {
        var studentIds = batch.Select(s => s.Id).ToList();
        BackgroundJob.Schedule<ISmsService>(
            sms => sms.SendBatchReminderAsync(studentIds),
            TimeSpan.FromSeconds(delaySeconds));

        delaySeconds++; // next batch 1 second later
    }
    // 50k students → 500 batches → 500 jobs → 500 seconds = ~8 min total
    // Gateway never sees more than 100 SMS/sec. Zero rate limit errors!
}

BackgroundService crash — IHostedService stops working Hard
📋 Scenario: "Attendance sync BackgroundService — unhandled exception threw out of ExecuteAsync. Service stopped completely! Attendance from SMS gateway no longer syncing. How to make it crash-resilient?"
✅ RESILIENT EXECUTEASYC — NEVER THROW OUT!
// Default behavior: uncaught exception in ExecuteAsync
// = BackgroundService STOPS. ASP.NET doesn't restart it!
// (unless IHostOptions.BackgroundServiceExceptionBehavior = StopHost)

protected override async Task ExecuteAsync(CancellationToken ct)
{
    _log.LogInformation("[AttendanceSync] Service started");

    while (!ct.IsCancellationRequested)
    {
        try
        {
            await ProcessAsync(ct);
        }
        catch (OperationCanceledException) when (ct.IsCancellationRequested)
        {
            _log.LogInformation("[AttendanceSync] Graceful shutdown");
            return; // exit cleanly
        }
        catch (Exception ex)
        {
            // NEVER propagate — log and continue!
            _log.LogError(ex, "[AttendanceSync] Error — retrying in 30s");
            _metrics.IncrementCounter("background_service.errors");

            try
            {
                await Task.Delay(TimeSpan.FromSeconds(30), ct);
            }
            catch (OperationCanceledException) { return; }
        }
    }
}

// Configure: unhandled exception in ANY hosted service = stop host
builder.Services.Configure<HostOptions>(opts =>
    opts.BackgroundServiceExceptionBehavior =
        BackgroundServiceExceptionBehavior.Ignore); // don't crash host!

Year-end promotion — app restart mid-job, 25,000 students processed! 🔥 Production Senior
📋 Scenario: "Year-end: promote 50,000 students to next grade. Job started — 25,000 done. Server restart (Azure deployment). All 50,000 get double-promoted! How to design idempotent resumable job?"
✅ IDEMPOTENT JOB — SAFE TO RUN MULTIPLE TIMES
// Idempotency key — track processed students
public async Task PromoteStudentsAsync(string academicYear)
{
    // Get ONLY students not yet promoted (idempotent!)
    var toPromote = await _ctx.Students
        .Where(s =>
            s.Status        == "Active"       &&
            s.AcademicYear  != academicYear  && // not already promoted!
            s.PromotionDate  == null             // not yet processed
        )
        .ToListAsync();

    _log.LogInformation("Promoting {Count} students (already done: {Done})",
        toPromote.Count,
        await _ctx.Students.CountAsync(s => s.AcademicYear == academicYear));

    // Process in small batches with per-batch commit
    foreach (var batch in toPromote.Chunk(100))
    {
        foreach (var student in batch)
        {
            student.Grade         = student.Grade + 1;
            student.AcademicYear  = academicYear;
            student.PromotionDate = DateTime.UtcNow; // mark processed!
        }
        await _ctx.SaveChangesAsync(); // commit each batch
        await Task.Delay(100);          // breathe!
    }
}
// Restart after 25k done:
// WHERE AcademicYear != newYear AND PromotionDate IS NULL
// → 0 students matched (already done) if previous batch committed
// → picks up remaining 25k if mid-batch crash
// Idempotent = safe to re-run anytime!

IHostedService vs BackgroundService vs Worker — interview question Easy Real Interview
📋 Scenario: "'IHostedService, BackgroundService, Worker Service — difference? When to use each in ZenCampus?'"
✅ CLEAR ANSWER

IHostedService: Interface with StartAsync + StopAsync. Full control over lifecycle. Use for startup tasks (cache warm-up, DB migration check, seed data). Not looping — runs once on start.

BackgroundService: Abstract class that implements IHostedService. Adds ExecuteAsync — your long-running loop. Use for: periodic polling (attendance sync every 30s), queue consumers, Channel readers. Lives in the same process as the web app.

Worker Service: Separate project (dotnet new worker) — standalone exe with its own host. No HTTP pipeline. Runs as Windows Service / Linux systemd. Use when background work is heavy enough to warrant its own process and independent scaling. ZenCampus example: EmailWorker.exe — 5 instances on separate VMs, processes email queue independently from web app.

ZenCampus: IHostedService for cache warm-up. BackgroundService for attendance sync + Channel consumer. Worker Service for EmailWorker. Hangfire for everything else (reliably persisted, retryable, dashboard).

{

        .Select(i => ConsumeAsync(i, ct));

9 ⚡ Quick Quiz

Q1: IHostedService-la DbContext inject பண்ணலாமா directly?

A) Yes — AddDbContext registers as Singleton by default
B) No — DbContext is Scoped, IHostedService is Singleton. Use IServiceProvider.CreateScope()!
C) Yes if using AddDbContextFactory
D) Only if AddDbContext is called before AddHostedService
✅ IHostedService = Singleton lifetime. DbContext = Scoped. Injecting Scoped into Singleton = captive dependency = same DbContext shared forever = concurrency issues + connection leaks! Always CreateScope() and resolve DbContext fresh per operation!
❌ Lifetime mismatch! IHostedService = Singleton (lives forever). DbContext = Scoped (per-request). Direct injection = one DbContext shared for app lifetime = race conditions! Use IServiceProvider.CreateScope() to get fresh DbContext per operation!

Q2: Task.Delay(30000) — no CancellationToken. App shutdown ஆனா?

A) App shuts down immediately — Delay is cancelled automatically
B) App waits 30 seconds before shutting down — delay ignores shutdown signal!
C) Task.Delay always respects application shutdown
D) Task.Delay throws on app shutdown
✅ Task.Delay without CancellationToken = ignores shutdown! App will wait the full 30 seconds. ASP.NET Core has a 5-second shutdown timeout — if delay exceeds it, process is killed hard. Always: Task.Delay(TimeSpan, cancellationToken) in BackgroundService loops!
❌ Task.Delay without ct = shutdown-blind! App waits full duration. Always pass the CancellationToken: await Task.Delay(TimeSpan.FromSeconds(30), ct) — this throws OperationCanceledException on shutdown, exits cleanly!

Q3: Hangfire vs BackgroundService — main difference for reliability?

A) Hangfire is faster — uses in-memory queue
B) Hangfire persists jobs to DB — survives app restart. BackgroundService = in-memory, lost on restart!
C) Both are equally reliable
D) BackgroundService is more reliable — closer to runtime
✅ Key difference! Hangfire → SQL Server → job survives crash/restart/deploy. BackgroundService → in-memory → app restart = all queued work lost. For SMS/email after fee payment, always Hangfire — never lose a notification due to deploy!
❌ Persistence is the key! BackgroundService = in-memory queue. Deploy, crash, or restart = pending work LOST. Hangfire stores every job in SQL Server — process it was enqueued, picked up after restart. For critical work (SMS, email) = always Hangfire!

Q4: Channel<T> BoundedChannelFullMode.Wait — producer behavior when channel is full?

A) Throws ChannelFullException immediately
B) Drops the message silently
C) WriteAsync awaits until consumer reads an item and space frees up (backpressure!)
D) Writes to overflow buffer automatically
✅ BoundedChannelFullMode.Wait = backpressure! WriteAsync suspends caller until consumer reads. This slows the producer naturally — prevents OOM. API response will be slightly delayed when channel is full, but no data is lost. Alternative: TryWrite + fallback to Hangfire!
❌ Wait mode = producer waits! WriteAsync is async and suspends until a slot opens. This is backpressure — the consumer's processing speed naturally caps the producer. No data loss, but producer (API request) feels the pressure. Use TryWrite for non-blocking check!
