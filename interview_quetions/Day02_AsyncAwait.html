1 Async yen venuum? — Thread Story புரிஞ்சுக்கோ
🎬 Enthiran (2010) — Chitti Multi-tasking

Chitti oru secondla 1000 calculations — parallel! Sync code = oru waiter oru table மட்டும் serve panruvan — customer wait panuvaanga. Async = oru waiter 10 tables manage panruvan — order place pannitu kitchen-ku pona, other tables attend pannuvan. Thread = Waiter. I/O wait = Kitchen time.

❌ Sync — Thread Block

// Thread 10 seconds BLOCK!
public IActionResult GetStudent()
{
    var data = _service.GetStudent(); // blocks
    return Ok(data);
}
// 100 users = 100 threads = server out of threads!

✅ Async — Thread Free

// Thread released during DB wait!
public async Task<IActionResult> GetStudentAsync()
{
    var data = await _service.GetStudentAsync();
    return Ok(data);
}
// Thread does OTHER work while DB runs!
💬 Tanglish — 1 line interview answer

Async/await = I/O operation (DB, API, File) wait panna thread block aagaadu — thread pool-la other requests handle panuvu. Throughput increases, latency same — but scalability huge!

Returns: Task / Task<T> / ValueTask<T>

Caller: also awaits
// ✅ CORRECT async method signature
public async Task<Student> GetStudentAsync(int id)
{
    var student = await _repo.GetByIdAsync(id);  // awaitable
    return student;
}

// Return types:
async Task                 // → void equivalent (fire & forget avoid)
async Task<Student>        // → returns Student
async ValueTask<Student>   // → perf: no heap alloc if sync path common
async void                 // → ONLY for event handlers! Never use in API!

Naming Convention:

GetStudentAsync() SaveFeesAsync() SendSmsAsync() Async suffix = convention CancellationToken last param

ConfigureAwait(false) — When & Why:

// Library / Service layer — NO UI context needed
public async Task<Student> GetAsync(int id)
{
    var result = await _db.QueryAsync(sql)
        .ConfigureAwait(false); // ← don't capture context → faster
    return result;
}

// ASP.NET Core Controller — ConfigureAwait(false) needed illa
// (no SynchronizationContext in Core) — but still good practice in libs
💡 Interview Answer: ConfigureAwait(false) yen use panuvu?

await complete aana, original thread context-ku return aagum — that's context capture. Library code-la context vendam — ConfigureAwait(false) skip panuvu → small perf gain + deadlock risk reduce. ASP.NET Core-la SynchronizationContext illai — but library code-la always use pannu (portable).

3 Task Parallel Library — WhenAll, WhenAny, Parallel
🎬 Vikram (2022) — Three Agents Simultaneously

Vikram, Amar, Karnan — same time different places investigate pannaanga. Task.WhenAll() = ellaarum same time work panuvaanga — ellaarum finish aana than next step. Task.WhenAny() = yaaraavathu first finish aana — proceed!

// ❌ Sequential — 3 seconds total (1+1+1)
var student    = await GetStudentAsync(id);    // 1 sec
var attendance = await GetAttendanceAsync(id); // 1 sec
var fees       = await GetFeesAsync(id);       // 1 sec

// ✅ WhenAll — 1 second total (parallel!)
var (student, attendance, fees) = await (
    GetStudentAsync(id),
    GetAttendanceAsync(id),
    GetFeesAsync(id)
); // ValueTuple deconstruct — clean!

// Or classic WhenAll:
var t1 = GetStudentAsync(id);
var t2 = GetAttendanceAsync(id);
var t3 = GetFeesAsync(id);
await Task.WhenAll(t1, t2, t3);
var s = t1.Result; // already done — safe to use .Result here

WhenAny — First to finish wins:

// Two data sources — use whichever responds first
var cacheTask = GetFromCacheAsync(id);
var dbTask    = GetFromDbAsync(id);

var first = await Task.WhenAny(cacheTask, dbTask);
var result = await first; // get the winner's result

CancellationToken — Timeout & Cancel:

// ZenCampus API — cancel if user navigates away / timeout
public async Task<IActionResult> GetReportAsync(
    int classId,
    CancellationToken cancellationToken)  // ASP.NET auto injects this
{
    using var cts = CancellationTokenSource
        .CreateLinkedTokenSource(cancellationToken);
    cts.CancelAfter(TimeSpan.FromSeconds(5)); // 5s timeout

    var data = await _service
        .GetReportAsync(classId, cts.Token);
    return Ok(data);
}

// Service — honor the token
public async Task<Report> GetReportAsync(
    int id, CancellationToken ct)
{
    ct.ThrowIfCancellationRequested(); // check before heavy work
    return await _repo.GetAsync(id, ct); // pass down!
}

4 💀 Deadlocks & Traps — Loop-la Maatikichu!
🎬 Ratsasan (2018) — Trap Scene

Killer carefully trap set panruvan — victim escape-e pannala. .Result and .Wait() use panna — deadlock trap! Thread A waits for Thread B, Thread B waits for Thread A — infinite wait — application hang!

Deadlock #1 — .Result or .Wait() in sync context:

// ❌ DEADLOCK — Classic trap!
public Student GetStudent(int id)
{
    // Main thread (.Result) waits for async to complete
    // async completion needs main thread (sync context)
    // DEADLOCK — both waiting for each other!
    return _repo.GetAsync(id).Result; // ← BOMB!
}

// ✅ Fix Option 1 — Go async all the way
public async Task<Student> GetStudentAsync(int id)
    => await _repo.GetAsync(id);

// ✅ Fix Option 2 — ConfigureAwait false breaks deadlock
public Student GetStudent(int id)
    => _repo.GetAsync(id).ConfigureAwait(false)
           .GetAwaiter().GetResult(); // last resort only!

Deadlock #2 — async void fire and forget:

// ❌ async void — exception swallowed, can't await!
public async void SendSmsOnAdmission(Student s)
{
    await _sms.SendAsync(s.Mobile); // exception here → app crash!
}

// ✅ Task return — awaitable + exception catch possible
public async Task SendSmsOnAdmissionAsync(Student s)
{
    await _sms.SendAsync(s.Mobile);
}

Deadlock #3 — await inside lock:

// ❌ NEVER await inside lock{} — deadlock risk!
lock(_lockObj)
{
    var data = await _repo.GetAsync(id); // ← COMPILER ERROR (good!)
}

// ✅ Use SemaphoreSlim for async locking
private readonly SemaphoreSlim _semaphore = new(1, 1);

await _semaphore.WaitAsync();
try
{
    var data = await _repo.GetAsync(id); // safe!
}
finally { _semaphore.Release(); }

Trap — parallel loop with async:

// ❌ WRONG — Parallel.ForEach + async void = fire and forget!
Parallel.ForEach(students, async s =>
{
    await _sms.SendAsync(s.Mobile); // NOT awaited by Parallel!
});

// ✅ CORRECT — Task.WhenAll with Select
var tasks = students.Select(s => _sms.SendAsync(s.Mobile));
await Task.WhenAll(tasks); // all SMS sent, all awaited!

// ✅ With throttling — max 10 parallel (don't overload SMS API)
var semaphore = new SemaphoreSlim(10);
var tasks = students.Select(async s =>
{
    await semaphore.WaitAsync();
    try { await _sms.SendAsync(s.Mobile); }
    finally { semaphore.Release(); }
});
await Task.WhenAll(tasks);

1
API Slow Problem — Student Dashboard Medium
Topic: Sequential vs Parallel async calls
📋 Scenario: ZenCampus student dashboard API 3 seconds time edukuthu. Code review-la ithana pathingom:

public async Task<DashboardDto> GetDashboardAsync(int studentId)
{
    var profile    = await _studentSvc.GetProfileAsync(studentId);   // 1s
    var attendance = await _attendSvc.GetSummaryAsync(studentId);  // 1s
    var fees       = await _feeSvc.GetPendingAsync(studentId);     // 1s
    return new DashboardDto { Profile=profile, Attendance=attendance, Fees=fees };
}
Problem identify panni fix pannu. Interviewer follow-up: "Exception handle panuva epdi?"
✅ ANSWER

Problem: Sequential await — 3 independent calls one after another = 3 seconds. These calls don't depend on each other — parallel run panalaam!

public async Task<DashboardDto> GetDashboardAsync(int studentId)
{
    // Start all 3 — DON'T await yet!
    var profileTask    = _studentSvc.GetProfileAsync(studentId);
    var attendanceTask = _attendSvc.GetSummaryAsync(studentId);
    var feesTask       = _feeSvc.GetPendingAsync(studentId);

    // Wait for ALL — slowest one = total time (~1s)
    await Task.WhenAll(profileTask, attendanceTask, feesTask);

    return new DashboardDto
    {
        Profile    = profileTask.Result,    // already done — safe!
        Attendance = attendanceTask.Result,
        Fees       = feesTask.Result
    };
}

Result: 3s → ~1s. 66% improvement!

⭐ Follow-up: Exception handle epdi?
try
{
    await Task.WhenAll(profileTask, attendanceTask, feesTask);
}
catch (Exception)
{
    // WhenAll — ONE exception surfaces. Others may also failed.
    // Check each task individually:
    if (profileTask.IsFaulted)
        _logger.LogError(profileTask.Exception, "Profile failed");
    throw; // or return partial data
}

// Graceful partial — return whatever succeeded
var profile = profileTask.IsCompletedSuccessfully
    ? profileTask.Result : null;
2
Application Hang — Deadlock Debug Hard
Topic: .Result deadlock — classic interview bomb
📋 Scenario: ZenCampus login page — production-la suddenly hang aaguthu. Request varuthu, response varaadu — no error, no timeout for 30 seconds. Code: 
// In a service called from old sync code
public class AuthService
{
    public User ValidateUser(string username, string password)
    {
        return _userRepo.GetByUsernameAsync(username).Result;
    }
}
1) What exactly causes the hang? 2) Fix panna solutions enna?
✅ DEADLOCK EXPLAINED

What happens step by step:

1. ValidateUser() runs on Thread A (ASP.NET request thread)
2. .Result blocks Thread A — "wait for async to complete"
3. GetByUsernameAsync() finishes DB query
4. Tries to continue on the original SynchronizationContext (Thread A)
5. Thread A is BLOCKED waiting for step 4
6. Step 4 needs Thread A which is blocked by step 2
7. DEADLOCK — both waiting for each other! 
// ✅ Fix 1 — Best: Go async all the way
public async Task<User> ValidateUserAsync(string username, string password)
{
    return await _userRepo.GetByUsernameAsync(username);
}

// ✅ Fix 2 — ConfigureAwait breaks context requirement
public User ValidateUser(string username, string password)
{
    return _userRepo.GetByUsernameAsync(username)
        .ConfigureAwait(false)  // don't need original thread
        .GetAwaiter().GetResult();
}

// ✅ Fix 3 — GetAwaiter().GetResult() alone (ASP.NET Core no SC)
// ASP.NET Core-la SynchronizationContext illai — Core-la deadlock less likely
// But still avoid in production — bad practice!
⭐ Interview Gold Answer

"Classic deadlock — sync context captures the awaiting thread. Fix: async all the way. If legacy sync code — ConfigureAwait(false) breaks the context dependency. ZenCampus migration-la intha pattern pathom, async all the way refactor panninom."

3
Loop-la Maatikichu — Bulk SMS Send Medium
Topic: Parallel.ForEach async trap + throttling
📋 Scenario: Fee reminder day — 5000 parents-ku SMS pathaikanom. Code ezuthingom — run aana SMS reach aagala, error-um illai, idhu enna problem? 
public async Task SendReminderAsync(List<Parent> parents)
{
    Parallel.ForEach(parents, async parent =>
    {
        await _smsService.SendAsync(parent.Mobile, "Fee pending");
    });
}
✅ ANSWER

Bug: Parallel.ForEach with async lambda = async void! ForEach doesn't await the tasks — method returns immediately, SMS tasks running in background, exceptions swallowed!

// ✅ Fix 1 — Task.WhenAll (all at once — risky if 5000!)
var tasks = parents
    .Select(p => _smsService.SendAsync(p.Mobile, "Fee pending"));
await Task.WhenAll(tasks);

// ✅ Fix 2 — Throttled parallel (BEST for production)
var semaphore = new SemaphoreSlim(20); // max 20 parallel
var tasks = parents.Select(async parent =>
{
    await semaphore.WaitAsync();
    try
    {
        await _smsService.SendAsync(parent.Mobile, "Fee pending");
    }
    catch (Exception ex)
    {
        _logger.LogError(ex, "SMS failed: {Mobile}", parent.Mobile);
    }
    finally { semaphore.Release(); }
});
await Task.WhenAll(tasks);

// ✅ Fix 3 — Chunk panni send (memory efficient)
foreach (var chunk in parents.Chunk(100)) // .NET 6+ Chunk
{
    await Task.WhenAll(chunk.Select(p =>
        _smsService.SendAsync(p.Mobile, "Fee pending")));
}
⭐ Senior Level Add-on

"Production-la 5000 SMS oru time-la blast panna SMS provider rate limit hit aagum. Better approach: Hangfire background job + batches + retry policy + dead letter queue."

CancellationToken — User Navigates Away Medium
Topic: Request cancellation, wasted DB queries
📋 Scenario: Student report page load panna 5 seconds aaguthu. User report click pannitu wait panna mudiyaama back press pannuvan — aaana DB-la query still running! Server resources waste. Epdi fix panriya?
✅ ANSWER — CancellationToken propagation
// Controller — ASP.NET auto-cancels when client disconnects
[HttpGet("report/{classId}")]
public async Task<IActionResult> GetReportAsync(
    int classId,
    CancellationToken cancellationToken)  // ← ASP.NET injects this
{
    try
    {
        var report = await _reportService
            .GetClassReportAsync(classId, cancellationToken);
        return Ok(report);
    }
    catch (OperationCanceledException)
    {
        _logger.LogInformation("Report request cancelled by client");
        return StatusCode(499); // Client Closed Request
    }
}

// Service — pass it down
public async Task<Report> GetClassReportAsync(
    int classId, CancellationToken ct)
{
    return await _repo.GetReportAsync(classId, ct); // pass down!
}

// Repository — Dapper with cancellation
public async Task<Report> GetReportAsync(int classId, CancellationToken ct)
{
    using var conn = _connectionFactory.Create();
    return await conn.QueryFirstOrDefaultAsync<Report>(
        new CommandDefinition(sql, parameters,
            cancellationToken: ct)); // Dapper supports CT!
}
⭐ Key Insight

CancellationToken is a chain — Controller → Service → Repository → DB. If any link misses it, DB query still runs after client leaves. Always pass CT down the entire chain!

Task.Run vs async/await — CPU vs I/O Hard
Topic: When to use Task.Run
📋 Scenario: ZenCampus-la Excel report generate panurom — 10,000 rows, complex calculations, 4-5 seconds CPU time. Junior developer Task.Run() use pannuvan. Senior review-la "Controller-la Task.Run unnecessary" nu solluvan. Yaar correct, yen?
✅ ANSWER

Senior is correct for Web API context.

// ❌ Controller-la Task.Run — pointless in ASP.NET Web API
public async Task<IActionResult> GenerateReport()
{
    var result = await Task.Run(() => _reportSvc.GenerateExcel());
    // Takes a thread pool thread to do CPU work
    // ASP.NET already uses thread pool!
    // You're just moving work thread → thread — no benefit!
    return File(result, "application/xlsx");
}

// ✅ Task.Run VALID in Desktop/WPF — offload to keep UI responsive
private async void Button_Click(object sender, EventArgs e)
{
    progressBar.Visible = true;
    var result = await Task.Run(() => _svc.HeavyCalculation());
    // UI thread free — progress bar spins!
    progressBar.Visible = false;
}

// ✅ Right solution for Web API — Background service or queue
// Long CPU work → Hangfire job → user gets job ID → polls for status

Rule of Thumb:

• I/O bound (DB, API, File, Network) → async/await directly ✅

• CPU bound in Desktop/WPF → Task.Run() ✅

• CPU bound in Web API → Background job (Hangfire) ✅

• CPU bound in Web API with Task.Run → Just wastes a thread ❌

Retry Logic — External API Failure Medium
Topic: Task.Delay retry pattern + Polly
📋 Scenario: ZenCampus-la payment gateway call panurom — intermittently fail aaguthu (network blip). 3 retry + exponential backoff implement pannu. Sleep() use pannalama?
✅ ANSWER
// ❌ Thread.Sleep — blocks thread! Never in async code!
Thread.Sleep(1000); // BLOCKS thread for 1 second

// ✅ Manual retry with Task.Delay (exponential backoff)
public async Task<PaymentResult> ProcessPaymentAsync(PaymentRequest req)
{
    int maxRetries = 3;
    int delay = 500; // ms

    for (int attempt = 1; attempt <= maxRetries; attempt++)
    {
        try
        {
            return await _paymentGateway.ChargeAsync(req);
        }
        catch (HttpRequestException ex) when (attempt < maxRetries)
        {
            _logger.LogWarning("Attempt {n} failed: {msg}", attempt, ex.Message);
            await Task.Delay(delay * attempt); // 500, 1000, 1500ms
        }
    }
    throw new Exception("Payment failed after 3 retries");
}

// ✅ Better — Polly library (production standard)
// Program.cs:
services.AddHttpClient<IPaymentGateway, PaymentGateway>()
    .AddTransientHttpErrorPolicy(p =>
        p.WaitAndRetryAsync(
            3,
            attempt => TimeSpan.FromMilliseconds(500 * attempt),
            (result, time, attempt, ctx) =>
                _logger.LogWarning("Retry {n}", attempt)));

{

WhenAll Exception — All Errors Capture Hard
Topic: AggregateException in WhenAll
📋 Scenario: 5 external API calls parallel panurom Task.WhenAll() la. 2 calls fail aana — aaana catch-la only ONE exception varuthu. Why? All errors epdi paaruva?
✅ ANSWER
var t1 = CallApiAAsync();
var t2 = CallApiBAsync(); // fails!
var t3 = CallApiCAsync(); // fails!

try
{
    await Task.WhenAll(t1, t2, t3);
}
catch (Exception ex)
{
    // ex = ONLY FIRST exception from WhenAll!
    // t3's exception is LOST here!
}

// ✅ Get ALL exceptions:
var allTasks = new[] { t1, t2, t3 };
var whenAll = Task.WhenAll(allTasks);
try
{
    await whenAll;
}
catch
{
    // whenAll.Exception is AggregateException with ALL errors
    if (whenAll.Exception != null)
    {
        foreach (var inner in whenAll.Exception.InnerExceptions)
            _logger.LogError(inner, "Task failed");
    }
    // Or check each task:
    var failed = allTasks.Where(t => t.IsFaulted).ToList();
    _logger.LogError("{n} tasks failed", failed.Count);
}

ValueTask vs Task — Hot Path Optimization Senior
Topic: When to use ValueTask for performance
📋 Scenario: ZenCampus-la student permission check — 90% of the time cached result return aaguthu. Senior "ValueTask use pannu" nu solluran. Task vs ValueTask difference enna, intha case-la yen ValueTask better?
✅ ANSWER
// Task — always heap allocation, even if sync path
public async Task<bool> HasPermissionAsync(int userId, string permission)
{
    if (_cache.TryGetValue(userId, out bool cached))
        return cached; // sync path — but Task still allocates!
    return await _repo.CheckPermissionAsync(userId, permission);
}

// ValueTask — no heap alloc on sync path!
public ValueTask<bool> HasPermissionAsync(int userId, string permission)
{
    if (_cache.TryGetValue(userId, out bool cached))
        return new ValueTask<bool>(cached); // NO heap alloc!
    return new ValueTask<bool>(
        _repo.CheckPermissionAsync(userId, permission));
}

When ValueTask:

• Sync path common (cache hit, early return)

• Hot path called millions of times (GC pressure real)

When Task (default):

• Always async (DB, API, File)

• Task awaited multiple times (ValueTask = await only once!)

6 ⚡ Quick Quiz

Q1: Thread.Sleep(1000) vs Task.Delay(1000) — async method-la which one use pannuva?

A) Thread.Sleep — both same
B) Task.Delay — Thread.Sleep blocks the thread
C) Thread.Sleep — faster
D) Neither — just use while loop
✅ Task.Delay awaitable — thread released during wait. Thread.Sleep blocks thread completely — other requests can't use that thread. Never Thread.Sleep in async code!
❌ Thread.Sleep = thread blocked. Task.Delay = thread released. In Web API, blocking threads = fewer concurrent requests!

Q2: async void eppoluthe safe?

A) Always safe
B) Never use it
C) Only for event handlers, with try/catch inside
D) Use when no return value needed
✅ Event handlers only! Signature fixed. But ALWAYS wrap in try/catch — exceptions in async void crash the app (unobserved)!
❌ async void = only for event handlers + must try/catch inside. Exceptions otherwise crash app with no way to catch externally!

Q3: Task.WhenAll varuthu — 3 tasks la 2 fail aana, catch-la how many exceptions?

A) 1 exception surfaces in catch, but task.Exception has all
B) 2 exceptions — one per failed task
C) AggregateException automatically in catch
D) First failure stops others immediately
✅ await WhenAll — first exception only in catch. But WhenAll task itself has .Exception (AggregateException) with ALL inner exceptions. Check each task.IsFaulted for full picture!
❌ catch sees first exception only. Full picture: await the WhenAll task variable, check .Exception.InnerExceptions for all failures.

Q4: Parallel.ForEach with async lambda — what happens?

A) All tasks run in parallel correctly
B) Compilation error
C) async void — tasks not awaited, exceptions swallowed, method returns early
D) Works fine, Parallel knows about async
✅ Parallel.ForEach doesn't understand async — treats it as async void. Use Task.WhenAll(items.Select(async i => await ...)) instead!
❌ Parallel.ForEach + async lambda = async void trap! Tasks fire and forget, exceptions vanish, method returns before tasks complete!
