1 Repository + Unit of Work Pattern
🎬 Baasha — Manickam says "Naan oru thadava sonna, nooru thadava sonna maari" (once I say it, it's done 100 times). Repository = same. Write data access logic once → use everywhere. Controller doesn't know if it's EF Core, Dapper, or API. Change DB → change only repository.

ZenCampus problem WITHOUT Repository: StudentController does DbContext.Students.Where(...).ToList() directly. Now if we add caching, logging, change from EF to Dapper — we edit EVERY controller. Repository = single place for data access. Controllers talk only to interfaces.

// Generic repository interface
public interface IRepository<T> where T : class
{
    Task<T?>              GetByIdAsync(int id, CancellationToken ct = default);
    Task<IReadOnlyList<T>> GetAllAsync(CancellationToken ct = default);
    Task<T>              AddAsync(T entity, CancellationToken ct = default);
    Task                 UpdateAsync(T entity, CancellationToken ct = default);
    Task                 DeleteAsync(int id, CancellationToken ct = default);
}

// Specific interface extends generic — add domain-specific queries
public interface IStudentRepository : IRepository<Student>
{
    Task<IReadOnlyList<Student>> GetBySchoolAsync(int schoolId, CancellationToken ct = default);
    Task<IReadOnlyList<Student>> SearchAsync(string name, int schoolId, CancellationToken ct = default);
    Task<bool>                     IsParentOfStudentAsync(int parentId, int studentId);
}

// EF Core implementation
public class StudentRepository(ZenCampusDbContext ctx) : IStudentRepository
{
    public async Task<Student?> GetByIdAsync(int id, CancellationToken ct = default) =>
        await ctx.Students.AsNoTracking().FirstOrDefaultAsync(s => s.Id == id, ct);

    public async Task<IReadOnlyList<Student>> GetBySchoolAsync(int schoolId, CancellationToken ct = default) =>
        await ctx.Students
            .Where(s => s.SchoolId == schoolId && s.IsActive)
            .AsNoTracking()
            .ToListAsync(ct);

    public async Task<IReadOnlyList<Student>> SearchAsync(string name, int schoolId, CancellationToken ct = default) =>
        await ctx.Students
            .Where(s => s.SchoolId == schoolId &&
                        EF.Functions.Like(s.Name, $"%{name}%"))
            .AsNoTracking()
            .ToListAsync(ct);

    public async Task<Student> AddAsync(Student s, CancellationToken ct = default)
    {
        ctx.Students.Add(s);
        await ctx.SaveChangesAsync(ct);
        return s;
    }
}

// Unit of Work — wrap multiple repositories in one transaction
public interface IUnitOfWork : IAsyncDisposable
{
    IStudentRepository Students  { get; }
    IFeeRepository     Fees      { get; }
    ISchoolRepository  Schools   { get; }
    Task<int>           SaveChangesAsync(CancellationToken ct = default);
}

public class UnitOfWork(ZenCampusDbContext ctx) : IUnitOfWork
{
    public IStudentRepository Students { get; } = new StudentRepository(ctx);
    public IFeeRepository     Fees     { get; } = new FeeRepository(ctx);
    public ISchoolRepository  Schools  { get; } = new SchoolRepository(ctx);
    public Task<int> SaveChangesAsync(CancellationToken ct = default) =>
        ctx.SaveChangesAsync(ct);
    public ValueTask DisposeAsync() => ctx.DisposeAsync();
}

// Usage — admit new student + create fee record in one transaction
public class StudentAdmissionService(IUnitOfWork uow)
{
    public async Task<int> AdmitStudentAsync(StudentAdmissionDto dto)
    {
        var student    = new Student { Name = dto.Name, SchoolId = dto.SchoolId };
        await uow.Students.AddAsync(student);

        var feeRecord  = new FeeRecord { StudentId = student.Id, Amount = dto.AnnualFee };
        await uow.Fees.AddAsync(feeRecord);

        await uow.SaveChangesAsync(); // ONE transaction — both succeed or both fail!
        return student.Id;
    }
}

2 CQRS + MediatR Pattern
🎬 Vikram Vedha — Vikram (cop, queries/reads) and Vedha (action, commands/writes) — two different people for two different jobs! CQRS = Commands (write, change state) go one way. Queries (read, return data) go another. Never mix them!

ZenCampus benefit: Fee collection command validates, saves, sends SMS, publishes event. Student list query just reads — no locks, no validation, can use cache. Mixing them: every fee query runs validation overhead. Separating: queries can be read-replicas (10x faster), commands have full business logic.

// NuGet: MediatR, MediatR.Extensions.Microsoft.DependencyInjection

// COMMAND — changes state, returns result
public record CollectFeeCommand(
    int     StudentId,
    decimal Amount,
    string  PaymentMode
) : IRequest<CollectFeeResult>;

public record CollectFeeResult(int ReceiptNo, DateTime PaidAt);

// Command Handler — all business logic here
public class CollectFeeCommandHandler(
    IFeeRepository  _feeRepo,
    ISmsService     _sms,
    IPublisher      _publisher)
    : IRequestHandler<CollectFeeCommand, CollectFeeResult>
{
    public async Task<CollectFeeResult> Handle(CollectFeeCommand cmd, CancellationToken ct)
    {
        if (cmd.Amount <= 0) throw new ValidationException("Amount must be positive");

        var receipt = new FeeReceipt
        {
            StudentId  = cmd.StudentId,
            Amount     = cmd.Amount,
            PaidAt     = DateTime.UtcNow,
            PaymentMode = cmd.PaymentMode
        };
        await _feeRepo.SaveReceiptAsync(receipt, ct);
        await _sms.SendReceiptAsync(receipt.ReceiptNo, cmd.Amount, ct);
        await _publisher.Publish(new FeeCollectedEvent(receipt.ReceiptNo), ct);

        return new CollectFeeResult(receipt.ReceiptNo, receipt.PaidAt);
    }
}

// QUERY — read only, no side effects!
public record GetStudentsBySchoolQuery(int SchoolId, string? Search = null)
    : IRequest<IReadOnlyList<StudentDto>>;

public class GetStudentsBySchoolQueryHandler(IStudentRepository _repo)
    : IRequestHandler<GetStudentsBySchoolQuery, IReadOnlyList<StudentDto>>
{
    public async Task<IReadOnlyList<StudentDto>> Handle(
        GetStudentsBySchoolQuery qry, CancellationToken ct)
    {
        var students = string.IsNullOrEmpty(qry.Search)
            ? await _repo.GetBySchoolAsync(qry.SchoolId, ct)
            : await _repo.SearchAsync(qry.Search, qry.SchoolId, ct);

        return students.Select(s => new StudentDto(s.Id, s.Name, s.Class))
                       .ToList().AsReadOnly();
    }
}

[ApiController, Route("api/[controller]")]
public class FeesController(IMediator _mediator) : ControllerBase
{

    public async Task<IActionResult> CollectFee([FromBody] CollectFeeCommand cmd) =>
        Ok(await _mediator.Send(cmd));  // controller is dumb, handler is smart

// MediatR Pipeline Behavior — cross-cutting concerns (logging, validation)
public class LoggingBehavior<TRequest, TResponse>(ILogger<LoggingBehavior<TRequest, TResponse>> _log)
    : IPipelineBehavior<TRequest, TResponse> where TRequest : notnull
{
    public async Task<TResponse> Handle(
        TRequest request, RequestHandlerDelegate<TResponse> next, CancellationToken ct)
    {
        _log.LogInformation("Handling {RequestType}", typeof(TRequest).Name);
        var result = await next();  // call actual handler
        _log.LogInformation("Handled {RequestType}", typeof(TRequest).Name);
        return result;
    }
}

3 Strategy Pattern
🎬 Thuppakki — Vijay switches between different combat strategies based on the situation. Same goal (defeat enemy), different approach each time. Strategy pattern = same interface, swappable algorithms. ZenCampus fee calculation: different strategy for each school type (CBSE/state board/international)! 
// Problem: Fee discount calculation differs per school type
// Without strategy: giant if-else/switch in service — violates Open/Closed Principle

// Strategy interface
public interface IFeeDiscountStrategy
{
    decimal CalculateDiscount(Student student, decimal baseFee);
}

// Concrete strategies
public class StaffChildDiscountStrategy : IFeeDiscountStrategy
{
    public decimal CalculateDiscount(Student s, decimal fee) =>
        s.IsStaffChild ? fee * 0.50m : 0m;   // 50% for staff children
}

public class SiblingDiscountStrategy : IFeeDiscountStrategy
{
    public decimal CalculateDiscount(Student s, decimal fee) =>
        s.SiblingCount > 0 ? fee * (0.10m * s.SiblingCount) : 0m; // 10% per sibling
}

public class MeritScholarshipStrategy : IFeeDiscountStrategy
{
    public decimal CalculateDiscount(Student s, decimal fee) =>
        s.Rank == 1 ? fee * 0.75m :   // rank 1 → 75% off
        s.Rank <= 5 ? fee * 0.50m :   // top 5 → 50% off
        s.Rank <= 10 ? fee * 0.25m :  // top 10 → 25% off
        0m;

}

public class NoDiscountStrategy : IFeeDiscountStrategy
{
    public decimal CalculateDiscount(Student s, decimal fee) => 0m;
}

// Context — uses strategy
public class FeeCalculationService
{
    public decimal CalculateNetFee(Student student, decimal baseFee,
        IFeeDiscountStrategy strategy)
    {
        var discount = strategy.CalculateDiscount(student, baseFee);
        return Math.Max(0, baseFee - discount);
    }
}

// Strategy selector — chooses right strategy based on student
public class DiscountStrategySelector
{
    public IFeeDiscountStrategy SelectFor(Student student)
    {
        if (student.IsStaffChild)    return new StaffChildDiscountStrategy();
        if (student.Rank <= 10)       return new MeritScholarshipStrategy();
        if (student.SiblingCount > 0) return new SiblingDiscountStrategy();
        return new NoDiscountStrategy();
    }
}

// Adding new discount type: create new class implementing IFeeDiscountStrategy
// Don't touch existing code = Open/Closed Principle ✅

5 Decorator Pattern
🎬 7aum Arivu — Aravind goes through layers of training: memory → combat → virus resistance. Each layer wraps the previous and adds capability. Decorator pattern = same. Wrap a service with caching → wrap that with logging → same interface, stacked behaviors!

ZenCampus use: IStudentRepository (base) → wrap with CachedStudentRepository (adds Redis) → wrap with LoggedStudentRepository (adds timing). Controller uses IStudentRepository — doesn't know it's cached and logged!

// Base decorator wraps a real implementation
public class CachedStudentRepository(IStudentRepository _inner, IMemoryCache _cache)
    : IStudentRepository  // ← implements SAME interface as what it wraps!
{
    public async Task<IReadOnlyList<Student>> GetBySchoolAsync(int schoolId, CancellationToken ct)
    {
        var key = $"students:school:{schoolId}";
        return await _cache.GetOrCreateAsync(key, async entry =>
        {
            entry.AbsoluteExpirationRelativeToNow = TimeSpan.FromMinutes(5);
            return await _inner.GetBySchoolAsync(schoolId, ct); // delegate to real repo
        }) ?? [];
    }

    public async Task<Student> AddAsync(Student s, CancellationToken ct)
    {
        var result = await _inner.AddAsync(s, ct);       // delegate write
        _cache.Remove($"students:school:{s.SchoolId}"); // invalidate cache!
        return result;
    }

    // All other methods delegate to _inner (no caching needed for search etc.)
    public Task<Student?> GetByIdAsync(int id, CancellationToken ct) =>
        _inner.GetByIdAsync(id, ct);

    public Task<IReadOnlyList<Student>> SearchAsync(string name, int schoolId, CancellationToken ct) =>
        _inner.SearchAsync(name, schoolId, ct);

    public Task<bool> IsParentOfStudentAsync(int p, int s) =>
        _inner.IsParentOfStudentAsync(p, s);

    public Task<IReadOnlyList<Student>> GetAllAsync(CancellationToken ct) =>
        _inner.GetAllAsync(ct);

    public Task UpdateAsync(Student s, CancellationToken ct) => _inner.UpdateAsync(s, ct);
    public Task DeleteAsync(int id, CancellationToken ct) => _inner.DeleteAsync(id, ct);
}

// Logging decorator wraps cached decorator — layer 2!
public class LoggedStudentRepository(IStudentRepository _inner,
    ILogger<LoggedStudentRepository> _log) : IStudentRepository
{
    public async Task<IReadOnlyList<Student>> GetBySchoolAsync(int schoolId, CancellationToken ct)
    {
        var sw = Stopwatch.StartNew();
        var result = await _inner.GetBySchoolAsync(schoolId, ct);
        _log.LogInformation("GetBySchool {SchoolId}: {Count} students in {Ms}ms",
            schoolId, result.Count, sw.ElapsedMilliseconds);
        return result;
    }
    // ... other methods delegate to _inner
}

// DI registration — wire up the chain (Scrutor NuGet makes this easier)
// Layer: Controller → LoggedRepo → CachedRepo → StudentRepository (real EF Core)
builder.Services.AddScoped<StudentRepository>();  // concrete
builder.Services.AddScoped<IStudentRepository>(sp =>
    new LoggedStudentRepository(
        new CachedStudentRepository(
            sp.GetRequiredService<StudentRepository>(),
            sp.GetRequiredService<IMemoryCache>()),
        sp.GetRequiredService<ILogger<LoggedStudentRepository>>()));

1
Why Repository over direct DbContext in controllers? Medium Most Asked
📋 Controller-la directly DbContext use pannrom. "What's wrong with that, and when would you add a Repository layer?"
✅ TESTABILITY + SINGLE RESPONSIBILITY
// Direct DbContext in controller — 3 problems:

// 1. TIGHT COUPLING: Controller depends on EF Core. Swap to Dapper/API → rewrite controller.
// 2. NOT TESTABLE: Unit test needs real SQL Server running. TestContainers helps but slow.
// 3. DUPLICATION: ctx.Students.Where(s => s.SchoolId == id && s.IsActive) scattered everywhere.

// Repository solves all 3:
// 1. Controller depends on IStudentRepository (interface) — swap impl anytime
// 2. Unit test: var mockRepo = new Mock<IStudentRepository>() — no DB needed!
// 3. GetBySchoolAsync() defined once — DRY

// WHEN NOT to add Repository (pragmatic answer):
// - Small CRUD app with no complex queries — EF Core IS the repository
// - Quick prototype — add it when tests become painful
// - Generic Repository (IRepository<T>) is often over-engineering for CRUD

// ZenCampus answer: "We added Repository because StudentController tests ran
// 45 seconds (SQL Server startup). After mocking IStudentRepository, same tests
// run in 2 seconds. Also changed attendance module from EF to Dapper for reports —
// only AttendanceRepository changed, zero controller changes."

CQRS vs traditional service — when to use? Hard
📋 "Interview-la: Describe CQRS and explain when you wouldn't use it. What problem does MediatR solve?"
✅ CQRS TRADEOFFS — HONEST ANSWER WINS
// CQRS benefit: Commands and Queries have different needs
// Command (CollectFee): validate → save → SMS → event → audit log → complex
// Query (GetStudents): just read → fast → can cache → can use read replica DB
// Mixing them: every read goes through validation overhead. 50,000 students × overhead = slow!

// MediatR problem: Controller knows about every service it needs:
// public class FeeController(IFeeService _fee, ISmsService _sms, IAuditService _audit,
//     IEventBus _events) — constructor bloat!
// MediatR: Controller knows only IMediator. Handler knows specific services.
// Thin controller, fat handler — separation of concerns!

// Pipeline Behaviors solve cross-cutting concerns:
// Every request → LoggingBehavior → ValidationBehavior → Handler → LoggingBehavior
// No duplicate try-catch in every handler!

// WHEN NOT to use CQRS:
// - Simple CRUD app (Todo list, basic admin) — massive over-engineering
// - Small team: CQRS adds 2-3 files per operation. 50 endpoints = 150+ files!
// - No read/write performance difference — pointless separation

// ZenCampus answer: "Used CQRS for fee module (complex writes + read-heavy reports).
// Did NOT use for school/class CRUD — simple EF Core service sufficient.
// Added MediatR pipeline: LoggingBehavior catches handler timing, ValidationBehavior
// runs FluentValidation before every command — zero duplicate code!"

// Validation Pipeline Behavior — runs BEFORE every command handler
public class ValidationBehavior<TRequest, TResponse>(
    IEnumerable<IValidator<TRequest>> _validators)
    : IPipelineBehavior<TRequest, TResponse> where TRequest : notnull
{
    public async Task<TResponse> Handle(
        TRequest req, RequestHandlerDelegate<TResponse> next, CancellationToken ct)
    {
        if (!_validators.Any()) return await next();

        var context = new ValidationContext<TRequest>(req);
        var failures = _validators
            .Select(v => v.Validate(context))
            .SelectMany(r => r.Errors)
            .Where(f => f is not null)
            .ToList();

        if (failures.Count > 0)
            throw new ValidationException(failures); // stops pipeline, 400 auto

        return await next(); // validation passed → call real handler
    }
}

Repository + Unit of Work: Controllers were calling DbContext directly. Unit tests needed SQL Server — 45 seconds startup. Added IStudentRepository, IFeeRepository behind UnitOfWork. Strategy: Fee discount had switch statement. Product team added 3 new discount types in one sprint — every new type edited the switch, regression risk. 7 ⚡ Quick Quiz

Q1: CQRS-la Commands and Queries separate பண்றோம். Main benefit என்ன?

A) Commands are faster than Queries by default
B) Queries can be independently scaled/cached/read-replica optimized while Commands have full business logic. Different performance and complexity needs, different code paths.
C) CQRS reduces the number of classes in a project
D) Commands don't need validation because they write data
✅ CQRS reads and writes have different profiles. Reads: high frequency, can be stale (cache), can use read replica, no side effects. Writes: low frequency, need validation, save, emit events, send notifications. Mixing them: query overhead from write-side logic. ZenCampus: fee report query uses Dapper + Redis cache (milliseconds). Fee collection command validates + SMS + event (200ms). Right tool per job!
❌ CQRS separates READ concerns from WRITE concerns. Benefit: queries can be cached, use read replicas, have no validation overhead. Commands have full business logic, validation, event publishing. Separately they're optimized for their use cases. In ZenCampus: GetStudentsQuery returns cached list in 5ms. CollectFeeCommand validates + saves + sends SMS in 200ms. Same service would compromise both!

Q2: Decorator pattern — what's the key structural rule?

A) Decorator must inherit from the class it decorates
B) Decorator implements the SAME interface as the decorated class AND holds a reference to it. This allows transparent wrapping and stacking.
C) Decorator replaces the original implementation completely
D) Only one Decorator can wrap a class at a time
✅ CachedStudentRepository : IStudentRepository (same interface) + takes IStudentRepository _inner in constructor (wraps it). Calling code sees IStudentRepository — can't tell it's cached. Stacking: Logged(Cached(EFCore)) — all three implement IStudentRepository, each wraps the previous. Composition over inheritance!
❌ Decorator pattern rule: implement the SAME interface AND hold a reference to the wrapped object. CachedStudentRepository implements IStudentRepository AND takes IStudentRepository _inner. This means callers can't distinguish decorator from real impl. Multiple decorators can be stacked: Logged(Cached(EFCore)) — all implement same interface!

Q3: Unit of Work pattern solves what specific problem?

A) Makes repositories faster with caching
B) Allows multiple databases to be queried simultaneously
C) Ensures multiple repository operations (student save + fee record create) happen in ONE database transaction — all succeed or all fail atomically.
D) Prevents N+1 query problems
✅ Without UoW: _studentRepo.AddAsync() calls SaveChanges internally → student saved. Exception hits → fee record never created → partial data corruption! With UoW: both Add() calls without SaveChanges, then uow.SaveChangesAsync() once → one transaction → atomic. EF Core DbContext already IS a UoW — UoW pattern wraps it to share across repositories!
❌ Unit of Work solves atomicity across multiple repositories. Without it: StudentRepo.Add() + SaveChanges → done. FeeRepo.Add() crashes → student exists but no fee record = data inconsistency! With UoW: track all changes across repos, ONE SaveChanges = ONE transaction. All succeed or all rollback. EF Core's DbContext is a built-in Unit of Work!

Q4: Strategy pattern vs if-else switch — when does Strategy pay off?

A) Strategy is always better than if-else regardless of complexity
B) When new variants are added frequently and you need Open/Closed Principle — add new strategy class without modifying existing code. Switch with 2-3 stable cases: keep the switch!
C) Strategy is used when you have more than 10 if conditions
D) Strategy should replace all boolean flags in the codebase
✅ Strategy shines when: (1) algorithms vary and are added frequently, (2) Open/Closed Principle needed — add without modifying, (3) algorithms need to be testable in isolation. ZenCampus: new discount type every sprint → if-else becomes a merge conflict nightmare. Strategy: new class, new test, zero existing code change. For stable 2-3 cases → switch is simpler!
❌ Strategy pays off when: new variants are added frequently (Open/Closed Principle), algorithms need independent testing, multiple algorithms need to coexist. ZenCampus discount: product team adds 5 types per sprint. Switch = risky edits every sprint. Strategy = new class per sprint, zero existing code touched. But 2-3 stable types → switch is simpler and less over-engineering!
