REST Principles + HTTP Verbs + Status Codes

🌐 REST = Resource (noun) + HTTP Verb + Representation

URL-la action போடாதே — URL is the resource identity, HTTP method is the verb. This is the most common anti-pattern in junior code.

❌ Anti-pattern (RPC-style)
POST /api/getStudents
GET /api/deleteStudent?id=5
POST /api/createFeePayment
GET /api/getAllSchoolsList
✅ RESTful Design
GET /api/v1/students
DELETE /api/v1/students/5
POST /api/v1/fee-payments
GET /api/v1/schools

ZenCampus URL Design — Resource Hierarchy:

// School → Student → Fee (nested resources)
GET    /api/v1/schools/{schoolId}/students            // List (paginated)
GET    /api/v1/schools/{schoolId}/students/{id}       // Single student
POST   /api/v1/schools/{schoolId}/students            // Admit new student
PUT    /api/v1/schools/{schoolId}/students/{id}       // Full replace
PATCH  /api/v1/schools/{schoolId}/students/{id}       // Partial update
DELETE /api/v1/schools/{schoolId}/students/{id}       // Remove

GET    /api/v1/students/{id}/fee-payments             // Student fee history
POST   /api/v1/students/{id}/fee-payments             // Collect fee

// Actions that don't fit CRUD → verb as sub-resource noun
POST   /api/v1/students/{id}/promotions               // Promote to next class
POST   /api/v1/students/{id}/transfers                // Transfer school
POST   /api/v1/users/{id}/password-resets             // Reset password

Problem Details — RFC 7807

⚠️ Structured Error Responses — Industry Standard

Random error format return panna frontend team-ku one error handler write panna mudiyathu. RFC 7807 = standardized error contract — every API speaks the same language.

❌ Random format (before) 
{
  "error": "Student not found",
  "code": 404,
  "ok": false
}
✅ RFC 7807 ProblemDetails 
{
  "type":     "https://zencampus.in/errors/404",
  "title":    "Student Not Found",
  "status":   404,
  "detail":   "Student 42 not in school 7",
  "instance": "/api/v1/schools/7/students/42",
  "traceId":  "00-abc123-def456-00"
}
// Program.cs
builder.Services.AddProblemDetails();
builder.Services.AddExceptionHandler<ZenCampusExceptionHandler>();
app.UseExceptionHandler();
// Single IExceptionHandler — maps ALL domain exceptions to Problem Details
public class ZenCampusExceptionHandler : IExceptionHandler
{
    public async ValueTask<bool> TryHandleAsync(
        HttpContext ctx, Exception ex, CancellationToken ct)
    {
        var (status, title, detail) = ex switch
        {
            StudentNotFoundException      e => (404, "Student Not Found",      e.Message),
            DuplicateAdmissionException   e => (409, "Admission Conflict",     e.Message),
            SchoolQuotaExceededException  e => (422, "Quota Exceeded",         e.Message),
            UnauthorizedSchoolException   e => (403, "Access Denied",          e.Message),
            FluentValidation.ValidationException e => (400, "Validation Failed", e.Message),
            _                                  => (500, "Internal Error",        "An unexpected error occurred")
        };

        ctx.Response.StatusCode = status;

        await ctx.Response.WriteAsJsonAsync(new ProblemDetails
        {
            Type     = $"https://zencampus.in/errors/{status}",
            Title    = title,
            Status   = status,
            Detail   = detail,
            Instance = ctx.Request.Path,
            Extensions = { ["traceId"] = Activity.Current?.Id ?? ctx.TraceIdentifier }
        }, ct);

        return true; // true = handled, stop pipeline
    }
}
// ValidationProblemDetails — for FluentValidation field errors (400)
{
  "type":   "https://tools.ietf.org/html/rfc7231#section-6.5.1",
  "title":  "One or more validation errors occurred.",
  "status": 400,
  "errors": {
    "Name":    ["Name must contain only letters and spaces"],
    "Phone":   ["Phone must be 10 digits starting with 6-9"],
    "ClassId": ["ClassId must be between 1 and 12"]
  },
  "traceId": "00-abc123-00"
}

Pagination — Cursor vs Offset

📄 50,000 Students — Cursor Pagination is the Answer

Oru shot-la 50,000 students return panna try panna — server crash, network timeout, mobile app freeze. Pagination mandatory.

FeatureOffset (SKIP/TAKE)Cursor (WHERE Id > x)
URL?page=3&pageSize=20?cursor=BASE64&take=20
DB QueryScans and discards N rowsIndex seek — reads only needed rows
Page 500 perfReads 10,000 rows to skip themSame speed as page 1 — O(log n)
Jump to page✅ ?page=50❌ Sequential only
Real-time stable❌ New admission shifts all pages✅ Cursor anchors to last seen ID
ZenCampus useAdmin paged reports (small, needs jump)✅ Attendance/fee lists (scroll-heavy)
// Generic wrapper
public record PagedResult<T>(
    IReadOnlyList<T> Items,
    string?          NextCursor,   // null = last page
    bool             HasNextPage,
    int              TotalCount);  // 0 = not counted (expensive COUNT(*), only if UI needs it)
// CQRS Query Handler — cursor pagination
public async Task<PagedResult<StudentDto>> Handle(
    GetStudentsQuery q, CancellationToken ct)
{
    var take = Math.Min(q.Take, 100); // Never allow unbounded — max 100

    var query = _db.Students
        .Where(s => s.SchoolId == q.SchoolId && !s.IsDeleted)
        .OrderBy(s => s.Id);

    // Decode cursor → get last seen Id
    if (q.Cursor is not null)
    {
        var cursor = JsonSerializer.Deserialize<CursorData>(
            Convert.FromBase64String(q.Cursor));
        query = query.Where(s => s.Id > cursor.Id); // Index seek!
    }

    // Fetch one extra to detect if next page exists
    var items = await query
        .Take(take + 1)
        .Select(s => new StudentDto(s.Id, s.Name, s.Class, s.RollNo))
        .ToListAsync(ct);

    var hasNext = items.Count > take;
    if (hasNext) items.RemoveAt(items.Count - 1); // Remove the extra item

    string? nextCursor = null;
    if (hasNext)
    {
        var json = JsonSerializer.Serialize(new { id = items[^1].Id });
        nextCursor = Convert.ToBase64String(Encoding.UTF8.GetBytes(json));
    }

    return new PagedResult<StudentDto>(items, nextCursor, hasNext, 0);
}

// Response:
{
  "items":       [/* 20 students */],
  "nextCursor":  "eyJpZCI6MTIwfQ",
  "hasNextPage": true,
  "totalCount":  0
}

// Next page request:
GET /api/v1/schools/1/students?cursor=eyJpZCI6MTIwfQ&take=20
Production trap: Never GetAllStudentsAsync().Skip(n).Take(m) in memory — loads all 50,000 students into RAM first! Always paginate at DB level. EF Core's .Skip(n).Take(m) does translate to SQL OFFSET/FETCH — but cursor approach is still faster at large offsets.

🎯 10 Interview Scenarios

Q1. What is REST? Is every HTTP API a REST API? What makes an API truly RESTful?
Concept

REST (Representational State Transfer) is an architectural style, not a protocol. Many APIs claim to be REST but are actually HTTP-RPC (verbs in URLs, always POST).

6 REST Constraints:

  1. Client-Server — UI and API separated, evolve independently
  2. Stateless — server holds no session; JWT carries state per request
  3. Cacheable — GET responses declare cacheability (Cache-Control headers)
  4. Uniform Interface — resources identified by URI, manipulated via representations
  5. Layered System — client doesn't know if it hits load balancer, cache, or app server
  6. Code on Demand (optional) — server can send executable code to client

ZenCampus answer: "Our API follows REST conventions — resource-based URLs, HTTP verbs, stateless JWT. We're at Richardson Maturity Level 2. We skip HATEOAS (Level 3) — it adds complexity without practical benefit for our team-owned mobile clients."

Q2. Production Bug: Fee payment API returns 200 OK even when payment fails. Mobile app never retries. What's wrong, how to fix?
Production Bug

Root cause: Controller swallows exceptions, returns 200 {"success": false, "error": "DB failed"} — the "200 with error body" anti-pattern.

Why dangerous:

  • HTTP retry clients check status code only — 200 = "all good", no retry
  • Azure App Insights doesn't count 200 as failure → no alert fires
  • Frontend error interceptors skip 200 responses
  • Parents think payment succeeded when it didn't → double payment attempts
// ❌ WRONG
try { await _feeService.ProcessAsync(dto); return Ok(new { success = true }); }
catch { return Ok(new { success = false }); }  // ← NEVER do this

// ✅ CORRECT — let domain exceptions bubble to IExceptionHandler
var receipt = await _mediator.Send(new CollectFeeCommand(dto));
return CreatedAtAction(nameof(GetReceipt), new { id = receipt.Id }, receipt);
// Domain exceptions → ZenCampusExceptionHandler → 422/500 ProblemDetails

Q4. Attendance report API times out loading 50,000 students. Fix it without breaking existing clients.
Performance
  1. Add optional ?cursor and ?take=20 params — old clients omit them, get first 20 (default). Response envelope grows new nextCursor field — old clients ignore unknown fields (non-breaking).
  2. Never in-memory: GetAll().Skip(n) = loads 50,000 into RAM. Always paginate at DB level.
  3. For full-dataset exports (Excel reports): POST to /api/v1/reports/attendance → returns 202 Accepted + Location: /reports/job-123 → client polls → eventually GET returns download URL.
// Backward-compatible: add optional params
[HttpGet]
public async Task<IActionResult> GetStudents(
    int     schoolId,
    [FromQuery] string? cursor = null,  // old clients omit → first page
    [FromQuery] int     take   = 20)    // old clients omit → default 20
{
    var result = await _mediator.Send(new GetStudentsQuery(schoolId, cursor, take));
    return Ok(result); // old clients: items[] still present, nextCursor ignored
}
Q5. Explain RFC 7807 Problem Details. What's better about it versus custom error formats?
Concept

RFC 7807 is an IETF standard for HTTP error responses. Fields:

  • type — URI identifying error class (links to documentation)
  • title — human-readable error name (stable, same for all instances of type)
  • status — HTTP status code (redundant but useful for logging)
  • detail — specific message for this error instance
  • instance — URI of the request that triggered it
  • Custom: traceId, errors dict for validation

Over custom formats:

  • Frontend writes one generic error handler for all APIs
  • OpenAPI documents error shape standardly — no guesswork
  • Azure App Insights + monitoring tools understand the shape
  • traceId = direct correlation between parent's phone call and server logs
Q6. Why is SKIP/TAKE pagination bad for large datasets? Explain cursor pagination with SQL.
Concept 
-- OFFSET pagination — Page 500, size 20:
SELECT * FROM Students ORDER BY Id
OFFSET 9980 ROWS FETCH NEXT 20 ROWS ONLY;
-- SQL Server reads rows 1-10000 first, discards 9980, returns 20
-- Performance: O(n) — page 500 is 500x slower than page 1

-- Cursor pagination — same "page":
SELECT TOP 20 * FROM Students
WHERE Id > 9980  -- Index seek on clustered PK!
ORDER BY Id;
-- SQL Server: 1 B-tree lookup → read 20 rows → done
-- Performance: O(log n) — same speed as page 1

Bonus trap: With offset, if a new student is admitted between page 1 and page 2 loads, every student shifts forward by 1 — app shows duplicate student on page 2 that already appeared at end of page 1. Cursor is immune — anchors to last seen Id.

Q7. Security audit: Swagger UI is accessible at production URL. What's the risk and what's your fix?
Security

Risks:

  • Attacker sees all endpoints, parameters, expected input shapes
  • Reduces reconnaissance effort — no need to reverse-engineer the API
  • Reveals internal patterns (schoolId formats, auth endpoints, admin routes)
  • Swagger UI lets attackers invoke APIs directly in browser with a JWT
// Fix 1 — Dev only (simplest, correct for most cases)
if (app.Environment.IsDevelopment())
{
    app.UseSwagger();
    app.UseSwaggerUI();
}

// Fix 2 — Internal network only (for internal team tools in prod)
app.UseWhen(
    ctx => ctx.Request.Path.StartsWithSegments("/swagger"),
    branch => branch.Use(async (ctx, next) =>
    {
        var ip = ctx.Connection.RemoteIpAddress;
        if (ip is null || !ip.IsPrivate())
        {
            ctx.Response.StatusCode = StatusCodes.Status403Forbidden;
            return;
        }
        await next(ctx);
    }));
Q8. Parent wants to update only their phone number. PUT or PATCH? What's the difference?
API Design

PUT = full replacement. Client must send ALL fields. Missing fields → set to null/default.
PATCH = partial update. Client sends only changed fields.

Phone-only update → PATCH:

// PATCH /api/v1/parents/{id}
// Body: {"phone": "9876543210"}  ← only phone, nothing else
public record PatchParentDto
{
    public string? Phone   { get; init; }  // null = "don't touch"
    public string? Address { get; init; }
    public string? Email   { get; init; }
}

[HttpPatch("{id}")]
public async Task<IActionResult> PatchParent(int id, [FromBody] PatchParentDto dto)
{
    var parent = await _repo.GetAsync(id) ?? throw new NotFoundException();
    if (dto.Phone   is not null) parent.Phone   = dto.Phone;
    if (dto.Address is not null) parent.Address = dto.Address;
    if (dto.Email   is not null) parent.Email   = dto.Email;
    await _repo.UpdateAsync(parent);
    return Ok(parent);
}
// PUT would require sending name, address, email, district, pin — all fields — wasteful
Q9. What is HATEOAS? Should ZenCampus implement it?
Concept

HATEOAS (Hypermedia As Engine Of Application State) = API responses include _links to available actions. Client discovers what it can do next from the response — doesn't need hardcoded URLs.

  "id": 42, "name": "Ravi Kumar", "feeBalance": 5000,

Most production REST APIs are Richardson Maturity Level 2, not Level 3."

Q10. "Tell me about your API design experience." — Full ZenCampus structured answer
Experience

"In ZenCampus — our school ERP serving 50,000 students across 200 schools — I designed and evolved the REST API with several production-critical decisions:

Versioning: URL-path versioning — /api/v1/ and /api/v2/. When we added mandatory parentPhone in v2, 180 schools still ran old mobile apps. We kept v1 with [ApiVersion("1.0", Deprecated = true)] and a 6-month sunset window, so schools could upgrade without breaking existing operations.

Error handling bug: Fee payment returned 200 OK on failure. Mobile retry logic never triggered — parents thought payments succeeded. We migrated to RFC 7807 IExceptionHandler — single class maps all domain exceptions to proper 4xx/5xx ProblemDetails with traceId. Support team now resolves issues in minutes instead of hours.

Pagination performance: Attendance report was timing out. Replaced SKIP/TAKE offset with cursor pagination — WHERE Id > cursor LIMIT 20. Page 500 went from 2s timeout to consistent 8ms.

Documentation: Swashbuckle with XML comments and [ProducesResponseType] — frontend never asks 'what error shape does this return?' Swagger is blocked in production.

Core principle: API is a contract — once clients depend on it, you can only add, never break."

🧠 Quick Quiz

Q1. New student admitted successfully via POST. What HTTP status code should the controller return?
201 Created — POST success should return 201 + Location header pointing to the new resource URL. 200 is for GET/PUT. 204 is for DELETE (no body). 202 is for async operations where result isn't ready yet.
Q2. 50,000 students, mobile app loads list by infinite scroll. Which pagination?
Cursor pagination — infinite scroll is its exact use case. Offset at page 500+ scans 10,000 rows to discard them. Streaming doesn't solve payload size. GraphQL would be a major rewrite for this one problem.
Q3. Client sends name with special characters, FluentValidation rejects it. What RFC 7807 status?
400 Bad Request — input format is invalid (validation failed at boundary). 422 is for business rule violations when format is valid but business logic rejects it (e.g., student already admitted in same school). Know this distinction — it's a common interview trap.
Q4. V2 needs mandatory GPS field in student. 180 schools still on V1 app. Best approach?
API versioning — create V2 with mandatory GPS, keep V1 running. Option A delays the problem. Option B creates hidden conditional behavior — hard to test and maintain. Option D is manual versioning — no deprecated-version headers, no Swagger version groups, no framework support.
