3 Moq — Mock Dependencies, Isolate Logic
🎬 7aum Arivu — Subbu learns to impersonate Bodhidharma (fake everything). Moq = same. Mock IFeeRepository, fake its behavior, test FeeService logic in isolation. Real repository never called!

FeeService depends on IFeeRepository (DB), IEmailService, ISmsService. Unit test pannanam → Mock all dependencies → fake what they return → test only FeeService logic. If test fails → FeeService logic bug, not DB bug. Isolation = clear failure reason!

// NuGet: Moq

public class FeeServiceTests
{
    private readonly Mock<IFeeRepository>      _feeRepo    = new();
    private readonly Mock<ISmsService>         _sms        = new();
    private readonly Mock<ILogger<FeeService>>  _log        = new();
    private readonly FeeService               _sut;

    public FeeServiceTests()
    {
        _sut = new FeeService(_feeRepo.Object, _sms.Object, _log.Object);
        // .Object = actual IFeeRepository instance (the fake)
    }

    [Fact]
    public async Task CollectFeeAsync_WhenValid_SavesReceiptAndSendsSms()
    {
        // Arrange — fake what dependencies return
        var dto = new FeeCollectDto(StudentId: 101, Amount: 5000m, ParentPhone: "9876543210");
        var student = new Student { Id = 101, Name = "Ravi", SchoolId = 42 };

        _feeRepo
            .Setup(r => r.GetStudentAsync(101))
            .ReturnsAsync(student);  // fake: when asked for student 101, return this

        _feeRepo
            .Setup(r => r.SaveReceiptAsync(It.IsAny<FeeReceipt>()))
            .ReturnsAsync(1);         // fake: save returns rowsAffected=1

        _sms
            .Setup(s => s.SendReceiptAsync(It.IsAny<string>(), It.IsAny<string>(), It.IsAny<decimal>()))
            .Returns(Task.CompletedTask);

        // Act
        var result = await _sut.CollectFeeAsync(dto);

        // Assert — did service do what it should?
        Assert.True(result.IsSuccess);
        Assert.NotNull(result.ReceiptNo);

        // Verify SMS was sent exactly ONCE with correct phone
        _sms.Verify(
            s => s.SendReceiptAsync("9876543210", It.IsAny<string>(), 5000m),
            Times.Once);

        // Verify receipt was saved
        _feeRepo.Verify(
            r => r.SaveReceiptAsync(It.Is<FeeReceipt>(rec => rec.Amount == 5000m && rec.StudentId == 101)),
            Times.Once);
    }

    [Fact]
    public async Task CollectFeeAsync_WhenStudentNotFound_ThrowsNotFoundException()
    {
        _feeRepo
            .Setup(r => r.GetStudentAsync(999))
            .ReturnsAsync((Student?)null); // student not found!

        // Act + Assert — exception expected
        await Assert.ThrowsAsync<NotFoundException>(
            () => _sut.CollectFeeAsync(new FeeCollectDto(StudentId: 999, 5000m, "9876543210")));

        // Verify SMS was NOT sent (student didn't exist)
        _sms.Verify(
            s => s.SendReceiptAsync(It.IsAny<string>(), It.IsAny<string>(), It.IsAny<decimal>()),
            Times.Never); // never called!
    }
}
// Moq — Setup patterns reference

// Match any argument
.Setup(r => r.GetAsync(It.IsAny<int>())).ReturnsAsync(student);

// Match specific predicate
.Setup(r => r.GetAsync(It.Is<int>(id => id > 0))).ReturnsAsync(student);

// Throw exception from mock
.Setup(r => r.SaveAsync(It.IsAny<FeeReceipt>()))
.ThrowsAsync(new DbException("connection failed"));

// Capture what was passed (useful for asserting saved values)
FeeReceipt? capturedReceipt = null;
_feeRepo.Setup(r => r.SaveReceiptAsync(It.IsAny<FeeReceipt>()))
    .Callback<FeeReceipt>(r => capturedReceipt = r)  // captures what was passed!
    .ReturnsAsync(1);
// After Act: Assert.Equal("ZEN-2024-001", capturedReceipt?.ReceiptNo);

// Times.Exactly, Times.AtLeast, Times.AtMost, Times.Between
_sms.Verify(s => s.SendAsync(It.IsAny<string>()), Times.Exactly(3)); // 3 SMS sent
_feeRepo.VerifyNoOtherCalls();  // strict: assert nothing else was called!
4 FluentAssertions — Readable, Expressive Assertions
🎬 Anniyan — Ambi punishes lawbreakers with precise, descriptive sentences. FluentAssertions = same. Not just "Assert.Equal failed" — "Expected amount to be 5000 but found 4900 because late charge not applied." 
// NuGet: FluentAssertions
using FluentAssertions;

// Value assertions
result.Amount.Should().Be(5000m);
result.Amount.Should().BeGreaterThan(0);
result.ReceiptNo.Should().StartWith("ZEN-");
result.ReceiptNo.Should().NotBeNullOrEmpty();
result.IsSuccess.Should().BeTrue(
    "because fee was paid with valid student and amount"); // failure message!

// Collection assertions
students.Should().HaveCount(3);
students.Should().Contain(s => s.Name == "Ravi");
students.Should().AllSatisfy(s => s.SchoolId.Should().Be(42));
students.Should().BeInAscendingOrder(s => s.Name);
students.Should().NotContainNulls();

// Object deep equality — compares ALL properties!
result.Should().BeEquivalentTo(new FeeReceiptDto
{
    StudentId = 101, Amount = 5000m, SchoolId = 42
}, opts => opts
    .Excluding(r => r.ReceiptNo)   // exclude auto-generated fields
    .Excluding(r => r.CreatedAt));  // exclude timestamps

// Exception assertions
var act = () => _sut.CollectFeeAsync(new FeeCollectDto(StudentId: 0, -1m, ""));
await act.Should().ThrowAsync<ValidationException>()
    .WithMessage("*Amount must be positive*"); // wildcard match!

// Date assertions
receipt.CreatedAt.Should().BeCloseTo(DateTime.UtcNow, TimeSpan.FromSeconds(5));
receipt.DueDate.Should().BeAfter(DateTime.UtcNow);

// Why BeEquivalentTo > Assert.Equal for objects?
// Assert.Equal: "Expected: ZenCampus.Models.FeeReceipt, Actual: ZenCampus.Models.FeeReceipt"  ← useless!
// BeEquivalentTo: "Expected Amount to be 5000m but found 4900m at receipt.Amount" ← actionable!
5 TestContainers — Real SQL Server in Docker for Integration Tests
🎬 Enthiran — Vasi built real Chitti robot, not a plastic model. TestContainers = same. Real SQL Server in Docker, not EF Core InMemory fake. Real queries, real indexes, real constraints — catches bugs InMemory misses!

EF Core InMemory DB: fast but dangerous. LINQ that works in-memory fails in SQL (no LOWER(), no DATEDIFF). TestContainers: spin up real SQL Server Docker container before tests, run against it, tear down after. 500ms startup. Worth every millisecond — real DB catches real bugs!

⛔ EF Core InMemory — why it's dangerous

InMemory ignores: FK constraints, unique indexes, computed columns, SQL functions (DATEDIFF, DATEADD), raw SQL queries, transactions. Tests pass → production fails. Real incident: InMemory test passed for duplicate receipt — SQL unique constraint caught it in UAT. TestContainers instead!

// NuGet: Testcontainers.MsSql

public class SqlServerFixture : IAsyncLifetime
{
    private readonly MsSqlContainer _container = new MsSqlBuilder()
        .WithImage("mcr.microsoft.com/mssql/server:2022-latest")
        .WithPassword("ZenCampus_Test123!")
        .Build();

    public string ConnectionString => _container.GetConnectionString();
    public ZenCampusDbContext? DbContext { get; private set; }

    public async Task InitializeAsync()
    {
        await _container.StartAsync(); // pulls SQL Server image, starts container (~5s first run)

        var options = new DbContextOptionsBuilder<ZenCampusDbContext>()
            .UseSqlServer(ConnectionString, sql =>
                sql.EnableRetryOnFailure(3))
            .Options;

        DbContext = new ZenCampusDbContext(options);
        await DbContext.Database.MigrateAsync(); // apply all EF migrations!
        await SeedTestDataAsync();                // insert schools, students
    }

    public async Task DisposeAsync()
    {
        await DbContext!.DisposeAsync();
        await _container.StopAsync();  // container removed automatically
    }

    private async Task SeedTestDataAsync()
    {
        DbContext!.Schools.Add(new School { Id = 42, Name = "Test School" });
        DbContext.Students.Add(new Student { Id = 101, Name = "Ravi", SchoolId = 42 });
        await DbContext.SaveChangesAsync();
    }
}

TestContainers vs EF InMemory — which to use and why? 🔥 Most Asked Medium
📋 Scenario: "'Should I use EF Core InMemory database or TestContainers with real SQL Server for integration tests in ZenCampus?'"
✅ ALWAYS TESTCONTAINERS — AND HERE'S WHY FROM PRODUCTION

EF Core InMemory limitations (real bugs we hit):

1. FK constraints not enforced → test passes, prod throws FK violation
2. Unique index not enforced → duplicate receipt saved in test, fails in prod
3. Raw SQL (FromSqlRaw) not supported → 0 rows returned in test, real query in prod
4. No transactions → test passes, prod deadlock on concurrent fee saves
5. Case-sensitive in-memory vs case-insensitive SQL Server → queries behave differently

TestContainers: Real SQL Server. Same constraints, same collation, same query plan. Migrations applied. If integration test passes, prod behavior is guaranteed.

When InMemory is OK: Testing non-DB code that happens to use DbContext. Read-only LINQ projections with no raw SQL. Speed is critical and no constraints involved.

8 ⚡ Quick Quiz

Q1: Moq-la Times.Once vs Times.Never — when to use each?

A) Times.Once = only one mock created. Times.Never = mock not used anywhere
B) Times.Once = method should be called exactly once. Times.Never = method should NOT be called (e.g., SMS not sent for failed fee)
C) Both are same — just different syntax
D) Times.Never throws exception if method is called
✅ Times.Once = Verify method was called exactly once. Times.Never = Verify method was NOT called. Critical: "student not found → SMS not sent" test needs Times.Never on ISmsService.SendAsync. If SMS is called when it shouldn't be = production bug sending receipts to random parents!
❌ Times is used in Verify() call. Times.Once = assert method was called exactly 1 time. Times.Never = assert method was NEVER called (useful for: inactive student → fee not collected → SMS not sent). Times.Exactly(3) = called exactly 3 times.

Q2: xUnit-la [Theory] + [InlineData] — what's the benefit over multiple [Fact] methods?

A) Theory tests run faster than Fact tests
B) One test method + multiple InlineData = multiple test cases with less code duplication, each shown as separate test in the runner
C) InlineData tests can't fail — they always pass
D) Theory tests share the same result across all InlineData
✅ [Theory] + [InlineData] = data-driven tests! One method, many input/output combinations. Each InlineData runs as a SEPARATE test case in the test runner — clear failure message: "CalculateLateCharge_Theory(5000, 30, 250) FAILED". Test late charge for 1, 10, 30 days — 4 lines of InlineData vs 4 separate [Fact] methods!
❌ [Theory] with [InlineData] creates one test method that runs once per InlineData. Each InlineData appears as a separate test in VS Test Explorer. When one fails, you see exactly WHICH data combination failed. 10 InlineData = 10 test cases from 1 method. Much less duplication than 10 separate [Fact] methods!

Q3: TestContainers-la IAsyncLifetime implement பண்ணணும் — why?

A) Required interface for all xUnit test classes
B) InitializeAsync() starts Docker container before tests; DisposeAsync() stops and removes it after — async setup/teardown for fixture
C) Makes tests run in parallel
D) Provides access to appsettings.json in tests
✅ IAsyncLifetime gives async InitializeAsync() (called before any test in the fixture) and DisposeAsync() (called after all tests). Starting Docker container = async operation. Constructor is synchronous — can't await in constructor! IAsyncLifetime = the xUnit async setup/teardown hook.
❌ IAsyncLifetime is xUnit's async lifecycle interface. InitializeAsync() = async version of constructor (await container.StartAsync() here). DisposeAsync() = async cleanup (await container.StopAsync()). Without it, you can't start Docker containers asynchronously before tests run.

Q4: BeEquivalentTo vs Assert.Equal for comparing objects — difference?

A) No difference — both compare objects the same way
B) BeEquivalentTo only works with strings, Assert.Equal works with objects
C) BeEquivalentTo does deep property-by-property comparison with readable failure messages showing which property failed; Assert.Equal uses Equals() and shows generic "Expected: Type, Actual: Type"
D) BeEquivalentTo requires IEquatable implementation
✅ BeEquivalentTo compares all properties recursively, and when it fails, tells you exactly WHICH property is different: "Expected Amount to be 5000, but found 4900 at receipt.Amount." Assert.Equal uses reference equality or Equals() — for objects, usually shows "FeeReceipt vs FeeReceipt" with no useful detail. BeEquivalentTo also supports .Excluding(x => x.GeneratedField) to skip auto-generated properties!
❌ BeEquivalentTo (FluentAssertions) does deep structural comparison + gives property-level failure messages. Assert.Equal uses object.Equals() — if Equals not overridden, compares references. For DTO comparison: BeEquivalentTo wins every time. "Expected Amount to be 5000 but found 4900" >> "Expected: FeeReceipt, Actual: FeeReceipt"!
