2. Service Contract, Operation Contract, Data Contract

[ServiceContract]
public interface IStudentService
{
    [OperationContract]
    StudentDto GetStudent(int id);

    [OperationContract]
    [FaultContract(typeof(StudentNotFoundFault))]
    void UpdateStudent(StudentDto student);
}

[DataContract]
public class StudentDto
{
    [DataMember] public int Id { get; set; }
    [DataMember] public string Name { get; set; }
    [DataMember] public decimal PendingFee { get; set; }
}

[ServiceContract] marks an interface as a WCF service boundary. [OperationContract] marks which methods on that interface are actually callable remotely — a service interface can have plain helper methods without this attribute, and those stay internal/non-exposed. [DataContract]/[DataMember] control exactly which class and which properties get serialized over the wire (XML/SOAP by default) — unlike plain public properties, you explicitly opt in per member, giving you control over the wire format and versioning.

3. Bindings — Which One, When

BindingTransportUse case
BasicHttpBindingHTTP, SOAP 1.1Interop with old/non-WCF clients, simplest, least secure by default
WSHttpBindingHTTP, SOAP 1.2 + WS-* standardsWeb-facing services needing message security, reliable sessions, transactions
NetTcpBindingTCP, binary encodingFastest — WCF-to-WCF only, same LAN/intranet (e.g., internal service-to-service calls)
NetNamedPipeBindingNamed pipesFastest possible — but ONLY same-machine communication (e.g., two processes on one server)
NetMsmqBindingMSMQ (queued)Disconnected/offline messaging — sender and receiver don't need to be online at the same time
Rule of thumb interviewers expect: same machine → NamedPipe; same LAN, WCF-to-WCF → NetTcp; cross-platform/web-facing → (Ws)HttpBinding; guaranteed delivery without both sides online → Msmq.

5. Instance Management — InstanceContextMode

ModeBehaviorUse when
PerCall (default)New service instance created for EVERY call, disposed right afterStateless services, best scalability — default and usually correct choice
PerSessionOne instance per client session (multiple calls from same client reuse it)Need to remember state across a sequence of calls from one client (e.g., a multi-step wizard)
SingleONE instance for the entire service lifetime, shared by ALL clientsRare — shared in-memory cache/counter; must be made thread-safe yourself

6. Concurrency Mode

[ServiceBehavior(InstanceContextMode = InstanceContextMode.Single,
                 ConcurrencyMode = ConcurrencyMode.Multiple)]
public class StudentService : IStudentService
{
    private readonly object _lock = new object();
    private int _activeSessionCount;

    public void RegisterSession()
    {
        lock (_lock) { _activeSessionCount++; } // must self-protect — Multiple mode allows concurrent calls in
    }
}

Single concurrency (default) — WCF allows only ONE thread into the instance at a time (auto-locks for you), safest but least scalable. Multiple — many threads can enter concurrently; YOU must add your own locking for any shared state (as above). Reentrant — single-threaded like Single, but allows the current call to call OUT to another service and back in without deadlocking itself.

7. Faults — Proper Error Handling Across the Wire

[DataContract]
public class StudentNotFoundFault
{
    [DataMember] public int StudentId { get; set; }
    [DataMember] public string Message { get; set; }
}

// Service implementation:
public StudentDto GetStudent(int id)
{
    var student = _repo.Find(id);
    if (student == null)
        throw new FaultException<StudentNotFoundFault>(
            new StudentNotFoundFault { StudentId = id, Message = "Student not found" });
    return student;
}

// Client:
try { var s = client.GetStudent(999); }
catch (FaultException<StudentNotFoundFault> fex)
{
    Console.WriteLine(fex.Detail.Message);
}

A plain C# exception thrown inside a WCF service does NOT cross the wire as-is — by default it gets converted into a generic, detail-stripped fault (for security, so internal exception details/stack traces aren't leaked to clients) unless includeExceptionDetailInFaults is enabled (dev/debug only, never in production). FaultContract + FaultException<T> is the correct way to deliberately communicate a specific business error back to the client with strongly-typed detail.

9. Duplex Contracts (Server Push / Callbacks)

public interface IAttendanceCallback
{
    [OperationContract(IsOneWay = true)]
    void OnAttendanceMarked(int studentId, string status);
}

[ServiceContract(CallbackContract = typeof(IAttendanceCallback))]
public interface IAttendanceService
{
    [OperationContract]
    void Subscribe();
}

Duplex contracts let the SERVICE call back INTO the client (push notifications) — needs a binding that supports duplex, like WSDualHttpBinding or NetTcpBinding (HTTP request/response bindings like BasicHttpBinding can't do this natively). Conceptually this is WCF's older answer to what SignalR solves more elegantly in modern .NET — a fair interview talking point if asked to compare.

10. Throttling & Metadata Exchange (MEX)

<serviceBehaviors>
  <behavior>
    <serviceThrottling maxConcurrentCalls="200" maxConcurrentSessions="100" maxConcurrentInstances="200" />
    <serviceMetadata httpGetEnabled="true" />
  </behavior>
</serviceBehaviors>

ServiceThrottlingBehavior caps concurrent calls/sessions/instances to protect the service from being overwhelmed — tune these based on load testing, not defaults blindly. The MEX (metadata exchange) endpoint is what lets a client's "Add Service Reference" fetch the WSDL to auto-generate a proxy — it must be explicitly enabled (httpGetEnabled or a dedicated mex endpoint) and is typically disabled in production for security once clients already have their generated proxies.

🛠️ Windows Service Core Concepts

11. What Is a Windows Service, and the ServiceBase Lifecycle

public class AttendanceSyncService : ServiceBase
{
    private Timer _timer;

    public AttendanceSyncService() { ServiceName = "AttendanceSyncService"; }

    protected override void OnStart(string[] args)
    {
        _timer = new Timer(SyncAttendance, null, TimeSpan.Zero, TimeSpan.FromMinutes(5));
    }

    protected override void OnStop()
    {
        _timer?.Dispose();
    }

    private void SyncAttendance(object state)
    {
        // do the actual work; wrap in try/catch — an unhandled exception here can kill the service
    }
}

static class Program
{
    static void Main()
    {
        ServiceBase.Run(new AttendanceSyncService());
    }
}

A Windows Service is a long-running executable managed by the Windows Service Control Manager (SCM) — no user interaction, no UI, can start automatically at boot before any user logs in, and runs under a configurable service account. ServiceBase (.NET Framework) is the base class you override: OnStart (kick off background work — must return quickly, ideally <30 seconds, or the SCM reports a start failure), OnStop (clean shutdown), plus optional OnPause/OnContinue if CanPauseAndContinue = true.

13. Debugging a Windows Service

You cannot just hit F5 on a Windows Service project — the SCM must start it, so debugging requires either: (1) attach Visual Studio's debugger to the running service process (Debug → Attach to Process) after starting it via services.msc/net start, or (2) write the app so it can ALSO run as a plain console app when passed a command-line flag (e.g., if (Environment.UserInteractive) RunAsConsole(); else ServiceBase.Run(...)) — letting you debug the exact same logic directly in Visual Studio without installing it as a service at all. Modern Worker Service templates make this trivial since the Generic Host runs the same way whether hosted as a console app or a Windows Service.

15. Recovery Options (Auto-Restart on Failure)

sc failure AttendanceSyncService reset= 86400 actions= restart/60000/restart/60000/run/1000

Configurable in services.msc (Recovery tab) or via sc failure — tells the SCM what to do if the service crashes: restart immediately, restart after a delay, run a recovery program, or take no action. Production services should always have at least "restart the service" configured for the first and second failure, since an unattended crash otherwise means silent downtime until someone notices.

16. Logging — No Console, So Where Do Errors Go?

protected override void OnStart(string[] args)
{
    try { StartWork(); }
    catch (Exception ex)
    {
        EventLog.WriteEntry("AttendanceSyncService", ex.ToString(), EventLogEntryType.Error);
        throw;
    }
}

A Windows Service has no console/UI to write to — Console.WriteLine goes nowhere useful. Standard practice: write to the Windows Event Log (System.Diagnostics.EventLog) for critical start/stop/failure events, and use a proper logging framework (Serilog/NLog writing to rolling files) for detailed operational logs — this is exactly why every unhandled-exception path inside the service loop must be caught and logged, or failures become invisible.

17. Hosting a WCF Service Inside a Windows Service (The Classic Combo)

public class StudentServiceHost : ServiceBase
{
    private ServiceHost _host;

    protected override void OnStart(string[] args)
    {
        _host = new ServiceHost(typeof(StudentService));
        _host.Open(); // opens all configured endpoints (e.g., net.tcp://.../StudentService)
    }

    protected override void OnStop()
    {
        _host?.Close(); // graceful close — completes in-flight calls, then shuts endpoints down
    }
}

This is the pattern that ties both halves of this guide together: ServiceHost (WCF) is opened inside OnStart and closed inside OnStop (ServiceBase, Windows Service). The Windows Service gives the WCF endpoint a durable, auto-starting home without needing IIS — common for internal LAN services (e.g., NetTcpBinding) on a factory/school server that must be always-on. Always call _host.Close() rather than letting it be abandoned — Close() allows outstanding calls to finish within a timeout, whereas an abrupt process kill drops them.

💻 Coding & Debugging Problems

1
Self-host a WCF service (NetTcpBinding) inside a Windows Service, with graceful start/stop.Combo
[ServiceContract]
public interface IStudentService
{
    [OperationContract]
    StudentDto GetStudent(int id);
}

[ServiceBehavior(InstanceContextMode = InstanceContextMode.PerCall)]
public class StudentService : IStudentService
{
    public StudentDto GetStudent(int id) => StudentRepository.GetById(id);
}

public partial class StudentServiceHostWindows : ServiceBase
{
    private ServiceHost _host;
    public StudentServiceHostWindows() { ServiceName = "StudentWcfHost"; }

    protected override void OnStart(string[] args)
    {
        _host = new ServiceHost(typeof(StudentService),
            new Uri("net.tcp://localhost:8090/StudentService"));
        _host.AddServiceEndpoint(typeof(IStudentService), new NetTcpBinding(), "");
        try
        {
            _host.Open();
        }
        catch
        {
            _host.Abort(); // Abort (not Close) on a failed Open — Close assumes a healthy channel
            throw;
        }
    }

    protected override void OnStop()
    {
        if (_host.State == CommunicationState.Faulted) _host.Abort();
        else _host.Close(TimeSpan.FromSeconds(10));
    }
}

Key detail interviewers look for: use Abort() instead of Close() when the host is in a Faulted state or failed to open — calling Close() on a faulted channel throws a secondary exception masking the real one.

2
Fix this WCF client code that leaks connections — it never properly closes the proxy.Debugging
// BEFORE — "using" on a WCF proxy is a well-known trap
using (var client = new StudentServiceClient())
{
    var s = client.GetStudent(1); // if this throws (e.g., timeout), Dispose() calls Close(),
}                                  // which itself throws inside a faulted channel -> original exception is swallowed

// AFTER — correct pattern
var client = new StudentServiceClient();
try
{
    var s = client.GetStudent(1);
    client.Close();
}
catch (CommunicationException)
{
    client.Abort();
    throw;
}
catch (TimeoutException)
{
    client.Abort();
    throw;
}
catch
{
    client.Abort();
    throw;
}

A generated WCF client proxy implements IDisposable, but its Dispose() internally calls Close() — if the channel faulted (e.g., the call itself threw a CommunicationException/TimeoutException), Close() throws again while unwinding, and that second exception can hide the original cause. Correct pattern: call Close() explicitly on success, and Abort() explicitly in every catch block, never relying on `using`/Dispose for a WCF proxy.

4
Configure the service so it survives being called by 500 concurrent clients without falling over.Throttling
var behavior = new ServiceThrottlingBehavior
{
    MaxConcurrentCalls = 400,
    MaxConcurrentSessions = 400,
    MaxConcurrentInstances = 400
};
_host.Description.Behaviors.Add(behavior);

Default throttling limits in older WCF versions are quite low (e.g., 16 concurrent calls) — under real load this silently queues/blocks callers well before any hardware limit is hit. Explicitly configuring ServiceThrottlingBehavior based on load-testing results (not guessed numbers) is a standard production hardening step.

Write a Windows Service that also runs as a console app for local debugging (no install needed).Debugging
static class Program
{
    static void Main(string[] args)
    {
        var service = new AttendanceSyncService();
        if (Environment.UserInteractive)
        {
            // Running from Visual Studio / a console — simulate SCM calls directly
            service.StartConsole(args);
            Console.WriteLine("Running as console. Press ENTER to stop.");

public class AttendanceSyncService : ServiceBase
{
    public void StartConsole(string[] args) => OnStart(args);
    public void StopConsole() => OnStop();

Environment.UserInteractive is false when running as an actual Windows Service (no desktop session) and true when run directly from a console/IDE — branching on it lets the exact same class be debugged with F5 with zero install/attach ceremony.

🎯 Interview Scenarios

1
What are the ABC of WCF?Core

Address (where — the URI the service listens on), Binding (how — transport protocol, encoding, and security used to communicate), Contract (what — the operations/interface the service exposes). Every WCF endpoint is defined by exactly these three things, either in config (web.config/app.config <endpoint>) or in code.

2
Why would you choose NetTcpBinding over BasicHttpBinding for two internal services talking to each other?Bindings

NetTcpBinding uses TCP directly with binary encoding — significantly faster and lower overhead than HTTP+SOAP text/XML encoding, and supports session, reliable messaging, and transport security out of the box. The tradeoff: it's WCF-to-WCF only (can't be called from a non-.NET client or a plain HTTP tool like Postman) and typically restricted to same-LAN/intranet use — fine for internal service-to-service calls, wrong choice for anything public-facing or needing broad interop.

3
What happens if OnStart() in a Windows Service takes too long?Lifecycle

The SCM (Service Control Manager) expects OnStart to return within a default timeout (historically ~30 seconds); if it takes too long, Windows reports the service failed to start (even if the work is technically still progressing) and may mark it as stopped/failed, which can also trigger any configured Recovery actions. Long-running initialization should be kicked off asynchronously (a background thread/Task started in OnStart, with OnStart itself returning immediately) rather than blocking inside OnStart.

4
Client vs Abort vs Close on a WCF proxy — why does it matter which one you call?Client Lifecycle

Close() performs a graceful shutdown — only safe to call when the channel is healthy (not Faulted). If the channel already faulted (e.g., a timeout or communication exception occurred), calling Close() throws a NEW exception during the graceful handshake, masking the real underlying error. Abort() immediately and forcefully terminates the channel without attempting a graceful handshake — always safe to call regardless of channel state, which is why the standard pattern calls Close() only in the success path and Abort() in every catch block.

5
How would you troubleshoot a Windows Service that starts but then immediately stops in production?Troubleshooting

First check the Windows Event Log (Application and System logs) for the service's own error entries and any .NET unhandled-exception entries. Verify the service account has the required permissions (file paths, DB connection, network resources it needs) — a common cause is running under a low-privilege account that can't reach a config file or database. If OnStart is doing real work before returning, wrap it in try/catch to log the actual exception (an unhandled exception during OnStart is otherwise easy to miss). Also confirm any dependent services (e.g., SQL Server, another Windows Service it depends on) are set as a formal Service Dependency so it doesn't start before its dependencies are ready.

6
Instancing mode PerCall vs Single — what's the tradeoff, and which is the safer default?Instancing

PerCall creates a brand-new, stateless instance for every single call and disposes it right after — no shared state, so no thread-safety concerns by design, and it scales best under load (WCF just spins up more instances as needed). Single keeps exactly ONE instance alive for the service's whole lifetime, shared across every client and every call — enables shared in-memory state (like a cache or counter) but makes YOU fully responsible for thread-safety (locks) since concurrent calls can hit the same instance simultaneously. PerCall is the safer, more scalable default; reach for Single only when you have a specific, deliberate reason to share state.
