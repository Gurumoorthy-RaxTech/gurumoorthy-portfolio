📚 Core Concepts

1. DbContext & DbSet — The Starting Point

public class SchoolDbContext : DbContext
{
    public SchoolDbContext(DbContextOptions<SchoolDbContext> options) : base(options) { }

    public DbSet<Student> Students { get; set; }
    public DbSet<ClassRoom> Classes { get; set; }
    public DbSet<Fee> Fees { get; set; }
}

// Registered in Program.cs (ASP.NET Core):
builder.Services.AddDbContext<SchoolDbContext>(options =>
    options.UseSqlServer(builder.Configuration.GetConnectionString("Default")));

DbContext = the session with the database — wraps a connection, tracks changes, translates LINQ to SQL. DbSet<T> = a table, queryable like a collection.

3. Relationships & Navigation Properties

public class Student
{
    public int Id { get; set; }
    public string Name { get; set; }

    public int ClassId { get; set; }              // FK
    public ClassRoom Class { get; set; }           // navigation property (many-to-one)

    public ICollection<Fee> Fees { get; set; }     // one-to-many (one student, many fee records)
}

public class ClassRoom
{
    public int Id { get; set; }
    public string Name { get; set; }
    public ICollection<Student> Students { get; set; }   // one-to-many (one class, many students)
}

One-to-many: ClassRoom → Students (used above). Many-to-many: Students ↔ Subjects, via an implicit join table (EF Core 5+ auto-creates it) or an explicit join entity if you need extra columns (like enrollment date). One-to-one: Student ↔ StudentProfile, configured with HasOne().WithOne().HasForeignKey<T>().

4. Fluent API vs Data Annotations

// Data Annotations (on the entity class itself)
public class Student
{
    [Key] public int Id { get; set; }
    [Required, MaxLength(100)] public string Name { get; set; }
}

// Fluent API (in OnModelCreating — better for complex config, keeps entity classes clean)
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    modelBuilder.Entity<Student>(e =>
    {
        e.HasKey(s => s.Id);
        e.Property(s => s.Name).IsRequired().HasMaxLength(100);
        e.HasOne(s => s.Class).WithMany(c => c.Students).HasForeignKey(s => s.ClassId);
    });
}

6. Change Tracking & Entity States

StateMeaning
AddedNew entity, will INSERT on SaveChanges()
UnchangedLoaded from DB, no modifications — no SQL sent
ModifiedLoaded then changed — will UPDATE (only changed columns) on SaveChanges()
DeletedMarked for removal — will DELETE on SaveChanges()
DetachedNot tracked by this context at all
var student = context.Students.Find(1);   // state = Unchanged
student.Name = "Updated Name";             // EF detects the change → state = Modified
context.SaveChanges();                     // generates: UPDATE Students SET Name=@p0 WHERE Id=@p1

7. Lazy Loading vs Eager Loading vs Explicit Loading

TypeHowRisk
Eager Loading.Include(s => s.Class) — loads related data in the SAME querySafest, most predictable — preferred default
Lazy LoadingRelated data auto-fetched the moment you access the navigation property (requires virtual + proxies package)Causes the N+1 query problem if you loop over a list and touch a nav property each time
Explicit Loadingcontext.Entry(student).Reference(s => s.Class).Load() — load on demand, manuallyFull control, but more code
// Eager loading — ONE query with a JOIN
var students = context.Students.Include(s => s.Class).ToList();

// N+1 PROBLEM with lazy loading — 1 query for students + N queries (one per student) for Class
foreach (var s in context.Students.ToList())
    Console.WriteLine(s.Class.Name);   // each access triggers a NEW query if lazy loading is on

8. LINQ to Entities

var results = context.Students
    .Where(s => s.SchoolId == schoolId && s.IsActive)
    .OrderBy(s => s.Name)
    .Select(s => new StudentDto { Id = s.Id, Name = s.Name, ClassName = s.Class.Name })
    .ToList();
// EF translates this whole chain into ONE SQL SELECT with a JOIN — not C# loops after fetch

9. AsNoTracking() — Read-Only Performance

// For read-only queries (no updates planned), skip change tracking overhead:
var students = context.Students.AsNoTracking().Where(s => s.SchoolId == schoolId).ToList();
// Faster, less memory — EF doesn't snapshot the original values for comparison

10. SaveChanges() — Unit of Work

All tracked changes (Added/Modified/Deleted entities across multiple DbSets) are sent to the DB in ONE call to SaveChanges(), wrapped in an implicit transaction — either all succeed or none do. This is EF's built-in Unit of Work pattern; you rarely need to hand-roll one on top of it.

11. Optimistic Concurrency (RowVersion)

public class Student
{
    public int Id { get; set; }
    public string Name { get; set; }
    [Timestamp] public byte[] RowVersion { get; set; }   // auto-updated by SQL Server on every row change
}

try
{
    context.SaveChanges();
}
catch (DbUpdateConcurrencyException)
{
    // another user changed/deleted this row since you loaded it — handle: reload and retry, or show conflict to user
}

Interview point: "how do you handle two users editing the same record at once?" → optimistic concurrency with a RowVersion/Timestamp column, catch DbUpdateConcurrencyException.

12. Transactions in EF Core

using var txn = context.Database.BeginTransaction();
try
{
    context.Students.Add(newStudent);
    context.SaveChanges();
    context.Fees.Add(new Fee { StudentId = newStudent.Id, Amount = 5000 });
    context.SaveChanges();
    txn.Commit();
}
catch { txn.Rollback(); throw; }
// Note: a SINGLE SaveChanges() call covering both adds is already atomic —
// explicit BeginTransaction is only needed across MULTIPLE SaveChanges() calls.

13. Raw SQL When You Need It

var students = context.Students
    .FromSqlRaw("SELECT * FROM Students WHERE SchoolId = {0}", schoolId)
    .ToList();
// Escape hatch for complex queries LINQ struggles to express cleanly — still parameterized/safe

3
Implement full CRUD (Add, Get, Update, Delete) for Student using EF Core.CRUD
public class StudentService
{
    private readonly SchoolDbContext _context;
    public StudentService(SchoolDbContext context) => _context = context;

    public Student Add(Student s) { _context.Students.Add(s); _context.SaveChanges(); return s; }
    public Student Get(int id) => _context.Students.Include(s => s.Class).FirstOrDefault(s => s.Id == id);
    public void Update(Student s) { _context.Students.Update(s); _context.SaveChanges(); }
    public void Delete(int id)
    {
        var s = _context.Students.Find(id);
        if (s != null) { _context.Students.Remove(s); _context.SaveChanges(); }
    }
}
4
This code has an N+1 query problem. Fix it.Debugging
// BEFORE — N+1 problem
var students = context.Students.ToList();          // 1 query
foreach (var s in students)
    Console.WriteLine(s.Class.Name);                 // N more queries (lazy loading fires each time)

// AFTER — fixed with eager loading, ONE query total
var students = context.Students.Include(s => s.Class).ToList();
foreach (var s in students)
    Console.WriteLine(s.Class.Name);                 // no extra queries — already loaded

Say this in the interview: "N+1 means 1 query for the parent list plus N extra queries, one per related row accessed in a loop. Fix is .Include() for eager loading, or disable lazy loading and be deliberate about what you fetch."

5
Add optimistic concurrency control to the Student entity and handle the conflict.Concurrency
public class Student
{
    public int Id { get; set; }
    public string Name { get; set; }
    [Timestamp] public byte[] RowVersion { get; set; }
}

public bool UpdateStudentName(int id, string newName, byte[] rowVersionFromClient)
{
    var student = context.Students.Find(id);
    context.Entry(student).Property(s => s.RowVersion).OriginalValue = rowVersionFromClient;
    student.Name = newName;
    try
    {
        context.SaveChanges();
        return true;
    }
    catch (DbUpdateConcurrencyException)
    {
        return false;  // someone else updated this row first — tell the user to refresh and retry
    }
}

7
Spot the bug — a Web API controller injects DbContext as Singleton and gets random cross-user data errors under load. Why, and fix?Debugging

Bug: services.AddSingleton<SchoolDbContext>() — DbContext is NOT thread-safe. One shared instance across ALL concurrent requests means multiple threads mutate its internal change tracker simultaneously, corrupting state and causing one user's changes to bleed into another's.

Fix: Register with services.AddDbContext<SchoolDbContext>(...), which defaults to Scoped lifetime — a new DbContext instance per HTTP request, disposed at the end of that request. Never manually register DbContext as Singleton.

Implement soft delete — "deleting" a student should just flag it, never actually remove the row, and normal queries should automatically exclude it.Query Filters
public class Student
{
    public int Id { get; set; }
    public string Name { get; set; }
    public bool IsDeleted { get; set; }
}

protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    modelBuilder.Entity<Student>().HasQueryFilter(s => !s.IsDeleted);
}

// "Delete" becomes an update, not a real DELETE:
public void SoftDelete(int id)
{
    var student = context.Students.Find(id);
    student.IsDeleted = true;
    context.SaveChanges();
}

// Every normal query automatically excludes soft-deleted rows now:
var active = context.Students.ToList();               // IsDeleted = false only

// To explicitly see deleted rows anyway (e.g. an admin "recycle bin" screen):
var all = context.Students.IgnoreQueryFilters().ToList();
10
Write a LINQ query that returns total fees collected per class, grouped and summed.LINQ / Aggregation
var feesSummary = context.Fees
    .Include(f => f.Student)
    .GroupBy(f => f.Student.Class.Name)
    .Select(g => new
    {
        ClassName = g.Key,
        TotalFees = g.Sum(f => f.Amount),
        StudentCount = g.Select(f => f.StudentId).Distinct().Count()
    })
    .OrderByDescending(x => x.TotalFees)
    .ToList();

EF translates the whole GroupBy/Sum chain into a single SQL GROUP BY + SUM() query — it does not pull all rows into memory and group in C#, as long as the query stays fully translatable.

13
A WinForms form loads a Student, the user edits it offline, then saves. The original DbContext is long gone. How do you update it?Disconnected Update
// The "student" object here was loaded, detached, edited in a WinForms form,
// and is now being saved with a BRAND NEW DbContext instance:
public void SaveEditedStudent(Student editedStudent)
{
    using var context = new SchoolDbContext(options);
    context.Entry(editedStudent).State = EntityState.Modified;   // attach + mark whole entity as changed
    context.SaveChanges();   // generates an UPDATE for every column, not just the ones that changed
}

Because the entity was never tracked by THIS context, EF doesn't know which individual properties changed — marking the whole entity Modified updates every column. This is the standard pattern for desktop apps (WinForms/WPF) where the DbContext doesn't stay alive across the whole user editing session, unlike a web request.

14
SaveChanges() throws — extract the actual SQL error hiding inside the DbUpdateException.Error Handling
try
{
    context.SaveChanges();
}
catch (DbUpdateException ex) when (ex.InnerException is SqlException sqlEx)
{
    if (sqlEx.Number == 2627 || sqlEx.Number == 2601)
        MessageBox.Show("Duplicate value violates a unique constraint.");
    else
        MessageBox.Show("Database error: " + sqlEx.Message);
}
catch (DbUpdateException ex)
{
    MessageBox.Show("Save failed: " + ex.Message);
}

EF wraps the real ADO.NET/SqlException inside DbUpdateException.InnerException — the top-level message is often generic ("An error occurred while updating the entries"), so you have to drill into InnerException to get the actual SQL Server error code and message.

🎯 Interview Scenarios

1
When would you choose Dapper over EF Core, and vice versa, in a project like Zen Campus?Architecture

EF Core: standard CRUD entities (Student, Fee, Attendance) where you want change tracking and migrations to evolve the schema safely with the team. Dapper: complex multi-table reports, performance-critical bulk reads, or hand-tuned queries where LINQ-generated SQL isn't good enough. Many real projects — including yours — use both side by side: EF Core for entity CRUD, Dapper for reporting.

2
What's the danger of lazy loading in a Web API, and how do you avoid it?Performance

Two dangers: (1) the N+1 query problem — accessing a navigation property in a loop fires one extra query per item; (2) if the DbContext is already disposed (e.g., after the HTTP response starts serializing), lazy loading throws an ObjectDisposedException. Avoid it by preferring explicit eager loading (.Include()) or projecting directly to a DTO with .Select(), and generally not enabling lazy-loading proxies in API projects at all.

3
Why is DbContext registered as Scoped, not Singleton or Transient, in ASP.NET Core?DI Lifetime

DbContext is not thread-safe and holds per-request state (the change tracker). Scoped = one instance per HTTP request, shared by everything within that request, then disposed — correct isolation between requests without the overhead of creating one per class (Transient) or the corruption risk of one shared across the whole app (Singleton).

4
How do migrations get applied safely in a production environment?DevOps

Generate migrations locally/in CI, review the generated SQL (never blindly trust auto-generated ALTER TABLE on a live table with data), then apply via a controlled release step — either dotnet ef database update run as a deployment step, or by generating a SQL script (dotnet ef migrations script) that a DBA reviews and runs manually for high-risk changes. Avoid calling context.Database.Migrate() automatically on app startup in production — it can race across multiple app instances starting simultaneously.

5
Two users open the same Student record and both hit Save — what happens, and how do you prevent bad data?Concurrency

Without concurrency control ("last write wins"), the second SaveChanges() silently overwrites the first user's change. To prevent this: add a [Timestamp] RowVersion column — EF includes it in the WHERE clause of the UPDATE, so if the row changed since it was loaded, zero rows match and EF throws DbUpdateConcurrencyException. Catch it and show the user a "this record was changed by someone else, please refresh" message rather than silently overwriting.

6
What's the difference between AsNoTracking() and a normal tracked query, and when do you use it?Performance

Normal queries snapshot the original values of every loaded entity so EF can detect changes and generate UPDATE statements later. AsNoTracking() skips that snapshot entirely — faster, less memory, but you can't call SaveChanges() to persist edits to those entities afterward. Use it for read-only display data (dashboards, reports, GET endpoints that never modify the result) — a very common and expected performance optimization to mention.
