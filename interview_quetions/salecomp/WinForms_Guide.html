3. Events & Delegates — Event-Driven Programming

// WinForms is entirely event-driven — the UI thread sits idle until an event fires
private void btnSave_Click(object sender, EventArgs e)
{
    // this method is a delegate wired to the Click event —
    // Designer.cs has: this.btnSave.Click += new System.EventHandler(this.btnSave_Click);
    SaveStudent();
}

Interview point: "what is an event in .NET?" → a special multicast delegate — a "publish/subscribe" mechanism where a control (publisher) raises an event, and one or more handler methods (subscribers) run in response. += subscribes, -= unsubscribes.

// Simple binding: DataGridView directly to a DataTable

5. Cross-Thread UI Updates — The #1 WinForms Gotcha

❌ CRASHES: updating a control directly from a background thread throws InvalidOperationException: Cross-thread operation not valid: Control 'lblStatus' accessed from a thread other than the thread it was created on.
// WRONG — crashes if called from a non-UI thread
private void UpdateStatus(string text)
{
    lblStatus.Text = text;
}

// RIGHT — check InvokeRequired, marshal back to the UI thread
private void UpdateStatus(string text)
{
    if (lblStatus.InvokeRequired)
        lblStatus.Invoke(new Action(() => lblStatus.Text = text));
    else
        lblStatus.Text = text;
}

// MODERN — async/await keeps you on the UI thread automatically after "await"
private async void btnImport_Click(object sender, EventArgs e)
{
    lblStatus.Text = "Importing...";
    await Task.Run(() => ImportLargeFile());   // heavy work off the UI thread
    lblStatus.Text = "Done!";                    // back on UI thread automatically — no Invoke needed
}

Why this happens: WinForms controls have "thread affinity" — only the thread that created a control may touch its properties. Control.Invoke() marshals the call back onto that UI thread. BackgroundWorker is the older pattern (has built-in ProgressChanged/RunWorkerCompleted events that auto-marshal); async/await with Task.Run is the modern, preferred approach.

8. MDI vs SDI

SDI (Single Document Interface): each form is its own independent top-level window — most business apps use this. MDI (Multiple Document Interface): one parent container form (IsMdiContainer = true) hosts multiple child forms inside it, like an old-school Office/Photoshop-style app. MDI is less common in modern apps but still asked about conceptually.

9. UserControl — Reusable Custom Controls

A UserControl is a composite of existing controls packaged as one reusable unit — e.g., a NumericTextBox with built-in range validation, or a StudentSearchPanel reused across multiple forms. Build it once, drop it on any form like a built-in control.

10. Menu, ToolStrip, StatusStrip

MenuStrip: top File/Edit/View menu bar. ToolStrip: toolbar with buttons/icons. StatusStrip: bottom bar for status text, progress bars. All are event-driven the same way as buttons — a Click handler per menu item.

6
Build a login form: validate credentials against SQL Server with proper error handling.Full Flow
private void btnLogin_Click(object sender, EventArgs e)
{
    if (string.IsNullOrWhiteSpace(txtUser.Text) || string.IsNullOrWhiteSpace(txtPass.Text))
    {
        MessageBox.Show("Username and password required.", "Validation", MessageBoxButtons.OK, MessageBoxIcon.Warning);
        return;
    }
    try
    {
        btnLogin.Enabled = false;
        bool valid = _authRepo.ValidateLogin(txtUser.Text, HashPassword(txtPass.Text));
        if (valid)
        {
            var main = new MainForm();
            main.Show();
            this.Hide();
        }
        else
        {
            MessageBox.Show("Invalid username or password.", "Login Failed", MessageBoxButtons.OK, MessageBoxIcon.Error);
        }
    }
    catch (SqlException ex)
    {
        MessageBox.Show("Could not reach the database. Please try again.", "Connection Error");
        // log ex somewhere — never show raw exception details to the end user
    }
    finally
    {
        btnLogin.Enabled = true;
    }
}

// ValidateLogin uses a parameterized ADO.NET query — never string-concatenate credentials into SQL
public bool ValidateLogin(string username, string passwordHash)
{
    const string sql = "SELECT COUNT(*) FROM Users WHERE Username=@U AND PasswordHash=@P";
    using var conn = new SqlConnection(_connStr);
    using var cmd = new SqlCommand(sql, conn);
    cmd.Parameters.AddWithValue("@U", username);
    cmd.Parameters.AddWithValue("@P", passwordHash);
    conn.Open();
    return (int)cmd.ExecuteScalar() > 0;
}
7
Spot the bug — this form leaks memory every time it's opened and closed. Why, and fix?Debugging
// BEFORE — leaks
public partial class StudentEditForm : Form
{
    public StudentEditForm()
    {
        InitializeComponent();
        AppEvents.StudentListChanged += OnStudentListChanged;   // subscribes to a STATIC/long-lived event
    }
    // no unsubscribe anywhere!
}

Bug: the form subscribes to a static/long-lived event (AppEvents.StudentListChanged) but never unsubscribes. The static event holder keeps a reference to the form's handler alive forever, which keeps the ENTIRE form object alive in memory — even after the user closes it and it should be garbage collected. Open/close this form 100 times and you leak 100 form instances.

// AFTER — fixed
public partial class StudentEditForm : Form
{
    public StudentEditForm()
    {
        InitializeComponent();
        AppEvents.StudentListChanged += OnStudentListChanged;
        this.FormClosed += (s, e) => AppEvents.StudentListChanged -= OnStudentListChanged;
    }
}

9
Export the currently displayed DataGridView rows to a CSV file.Export
private void btnExportCsv_Click(object sender, EventArgs e)
{
    using var sfd = new SaveFileDialog { Filter = "CSV files (*.csv)|*.csv", FileName = "students.csv" };
    if (sfd.ShowDialog() != DialogResult.OK) return;

    var sb = new StringBuilder();

    var headers = dgvStudents.Columns.Cast<DataGridViewColumn>().Select(c => c.HeaderText);
    sb.AppendLine(string.Join(",", headers));

    foreach (DataGridViewRow row in dgvStudents.Rows)
    {
        if (row.IsNewRow) continue;
        var cells = row.Cells.Cast<DataGridViewCell>()
            .Select(c => "\"" + (c.Value?.ToString() ?? "").Replace("\"", "\"\"") + "\"");
        sb.AppendLine(string.Join(",", cells));
    }

    File.WriteAllText(sfd.FileName, sb.ToString());
    MessageBox.Show("Exported to " + sfd.FileName, "Done", MessageBoxButtons.OK, MessageBoxIcon.Information);
}

Wrapping each cell value in quotes (and escaping embedded quotes as "") keeps commas/quotes inside a student's name or email from corrupting the CSV structure.

11
Build a reusable NumericTextBox UserControl with a configurable Min/Max range.Custom Control
public partial class NumericTextBox : UserControl
{
    private TextBox _textBox = new TextBox();

    public int MinValue { get; set; } = 0;
    public int MaxValue { get; set; } = 100;

    public int Value
    {
        get => int.TryParse(_textBox.Text, out int v) ? v : 0;
        set => _textBox.Text = value.ToString();
    }

    public NumericTextBox()
    {
        _textBox.Dock = DockStyle.Fill;
        _textBox.KeyPress += (s, e) =>
        {
            if (!char.IsDigit(e.KeyChar) && !char.IsControl(e.KeyChar))
                e.Handled = true;
        };
        _textBox.Validating += (s, e) =>
        {
            if (Value < MinValue || Value > MaxValue)
            {
                MessageBox.Show($"Enter a value between {MinValue} and {MaxValue}.");
                e.Cancel = true;
            }
        };
        this.Controls.Add(_textBox);
    }
}
// Drop it on any form like a built-in control: numericTextBox1.MinValue = 1; numericTextBox1.MaxValue = 100;

This is a very common WinForms interview coding exercise — building one reusable control that packages validation logic, instead of copy-pasting KeyPress/Validating handlers onto every numeric field across every form.

12
Auto-refresh the DataGridView every 30 seconds without blocking the UI or fighting the user's current selection.Timer
private System.Windows.Forms.Timer _refreshTimer = new System.Windows.Forms.Timer { Interval = 30000 };

private void Form1_Load(object sender, EventArgs e)
{
    _refreshTimer.Tick += async (s, e) => await RefreshGridAsync();
    _refreshTimer.Start();
}

private async Task RefreshGridAsync()
{
    int? selectedId = _selectedId != 0 ? _selectedId : (int?)null;

    var students = await Task.Run(() => _repository.GetAll());   // DB call off the UI thread
    LoadGrid(students);                                            // back on UI thread automatically (async/await)

    if (selectedId.HasValue)
        foreach (DataGridViewRow row in dgvStudents.Rows)
            if (row.DataBoundItem is Student st && st.Id == selectedId.Value)
                row.Selected = true;   // restore selection after refresh
}

System.Windows.Forms.Timer (not System.Threading.Timer) fires its Tick event already on the UI thread, so no Invoke is needed around simple UI updates — but the actual DB call is still pushed to a background thread via Task.Run so a slow query doesn't freeze the form for those 30-second refreshes.

13
The DataGridView flickers badly every time it refreshes with new data. Fix it.Rendering
// A DataGridView subclass with double buffering enabled
public class BufferedDataGridView : DataGridView
{
    public BufferedDataGridView()
    {
        this.DoubleBuffered = true;   // DataGridView.DoubleBuffered is protected - only accessible via a subclass
    }
}

// Or, without subclassing, via reflection (quick fix, less clean):
typeof(DataGridView).InvokeMember(
    "DoubleBuffered",
    System.Reflection.BindingFlags.SetProperty | System.Reflection.BindingFlags.Instance | System.Reflection.BindingFlags.NonPublic,
    null, dgvStudents, new object[] { true });

DoubleBuffered is protected on Control/DataGridView by design — Microsoft expects you to enable it via a subclass, which is the cleaner approach for a real project; the reflection trick is a fast one-liner for a quick fix under interview time pressure.

🎯 Interview Scenarios

1
Walk through the WinForms Form lifecycle events in order, and where you'd load data.Lifecycle

Constructor (InitializeComponent runs, controls exist but not visible) → Load (form about to display — this is where you fetch and bind data, since controls are fully initialized but the user hasn't seen a flash of empty UI) → Shown (form is now visible — good for focusing a control or showing a first-run tip) → FormClosing (cancellable — check for unsaved changes here) → FormClosed (cleanup, unsubscribe from any external/static events) → Dispose (resource cleanup).

2
Why can't you update a Label's Text property directly from a background thread?Threading

WinForms controls have "thread affinity" — the underlying Win32 window handle can only be safely touched by the thread that created it (the UI thread). Direct access from another thread throws InvalidOperationException (in debug builds; in release it can silently corrupt UI state, which is worse). Fix: Control.Invoke()/BeginInvoke() to marshal the call onto the UI thread, or simply use async/await — code after an await automatically resumes on the original UI thread's SynchronizationContext.

3
ShowDialog() vs Show() — when would using the wrong one cause a bug?Forms

If you use Show() for an Edit dialog and then immediately read its result on the next line, you'd read stale/default data — Show() doesn't block, so the code after it runs before the user has even seen the dialog, let alone clicked OK. ShowDialog() blocks until the dialog closes and gives you a real DialogResult to branch on. Conversely, using ShowDialog() for something that should stay open alongside the main form (like a floating tool palette) would incorrectly block/disable the main window.

4
How do you keep the UI responsive while running a long-running operation, like importing 10,000 rows?Threading

Never run long work directly on the UI thread's event handler — it freezes the whole form (no repaint, no click response) until it finishes. Options: async/await with Task.Run() for the heavy work (modern, preferred); BackgroundWorker (older, has built-in ProgressChanged and RunWorkerCompleted events that already marshal to the UI thread for you); or a raw Thread with manual Invoke calls (most control, most boilerplate). For a progress bar during import, BackgroundWorker's ReportProgress() is still a clean fit.

5
Why would a form not get garbage collected even after the user closes it?Memory

Most common cause: the form subscribed to an event on a longer-lived object (a static event, a singleton service, or a parent form) and never unsubscribed. The event's invocation list holds a reference to the form's handler method, which keeps the form instance reachable from GC's perspective — "the tail wags the dog." Fix: unsubscribe in FormClosed, or use weak event patterns for long-lived publishers.

6
How would you structure a WinForms app so business logic isn't tangled into the Form code-behind?Architecture

Keep the Form as thin as possible — it only handles UI events and delegates to separate classes: a Repository/DataAccess layer for ADO.NET calls, and optionally a Presenter/Controller class if following MVP (Model-View-Presenter) — the Form implements a simple interface (IStudentView) with properties/events, and the Presenter contains the actual logic and orchestrates calls to the Repository. This makes the logic unit-testable without needing to instantiate a real Form.
