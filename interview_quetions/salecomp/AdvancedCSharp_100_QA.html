⚡ A. Async / Await & Task Parallel Library (Q1–10)

1
Why does C# need async/await — what problem does it solve?Fundamentals

Synchronous I/O (DB call, HTTP call, file read) blocks the calling thread until the operation finishes — that thread can't do anything else. async/await lets the thread return to the pool while the I/O completes in the background (via the OS), and resume execution on a thread from the pool once the result is ready. This means a web server can handle thousands of concurrent requests with a small thread pool instead of one blocked thread per request.

2
What does "await" actually do under the hood?Internals

The compiler rewrites the async method into a state machine (a struct/class implementing IAsyncStateMachine). When await hits an incomplete Task, the method returns control to its caller immediately (freeing the thread), and registers a continuation on the awaited Task. When the Task completes, the continuation resumes the state machine at the point right after the await — on a thread pool thread (or the captured SynchronizationContext, unless ConfigureAwait(false) is used).

3
Difference between Task and Task<T>?Fundamentals

Task represents an asynchronous operation that returns no value (like void, but awaitable). Task<T> represents an asynchronous operation that produces a result of type T, accessed via .Result (blocking) or await (non-blocking). Both expose .IsCompleted, .Status, continuations via .ContinueWith, etc.

4
What causes a deadlock with .Result / .Wait(), and how do you avoid it?Gotcha

In contexts with a captured SynchronizationContext (classic ASP.NET, WPF/WinForms UI thread), calling .Result/.Wait() synchronously blocks that thread while it waits for the async method to complete. But the async method's continuation needs to run back on that SAME captured context — which is blocked — so neither ever finishes: deadlock. Fix: use await all the way up ("async all the way"), or call ConfigureAwait(false) inside library code so the continuation doesn't need the original context. ASP.NET Core has no SynchronizationContext, so this specific deadlock is rarer there, but blocking async code is still bad practice (thread pool starvation).

5
Task.Run vs Task.Factory.StartNew — which should you default to?API Choice

Task.Run is a simplified, safe wrapper — always schedules on the thread pool and auto-unwraps nested Tasks. Task.Factory.StartNew has more options (custom TaskScheduler, TaskCreationOptions) but defaults can surprise you — e.g., it does NOT auto-unwrap a Task<Task>, and may run on the calling thread's context in some overloads. Default recommendation: use Task.Run unless you specifically need StartNew's extra control.

6
Task.WhenAll vs Task.WhenAny — when do you use each?Composition

Task.WhenAll(tasks) awaits until ALL tasks complete, running them concurrently — e.g., call 3 microservices in parallel and combine results. If any fail, it aggregates exceptions but waits for all to finish first. Task.WhenAny(tasks) completes as soon as the FIRST task finishes — useful for timeout patterns (race an operation against a delay task) or picking the fastest of several redundant calls.

7
Why is "async void" dangerous, and when is it the only option?Best Practice

async void methods can't be awaited by the caller, so the caller has no way to know when it finishes or catch its exceptions — an unhandled exception in an async void method crashes the process instead of propagating normally. Always prefer async Task. The one legitimate case for async void is a top-level UI event handler (e.g., a Button.Click handler) because the event signature requires void — even there, wrap the body in try/catch.

8
How does exception handling work across multiple awaited/parallel tasks?Error Handling

A single await rethrows the original exception directly — you can try/catch it normally. But Task.WhenAll on multiple failing tasks wraps ALL of them into an AggregateException if you inspect .Exception; if you await the WhenAll task itself, only the FIRST exception is rethrown (though all tasks did fault) — so to see every failure, loop through task.Exception.InnerExceptions or check each task individually after WhenAll completes.

9
What is a CancellationToken and how do you wire it into a long-running async operation?Cancellation
public async Task ProcessLargeFileAsync(CancellationToken ct)
{
    foreach (var line in GetLines())
    {
        ct.ThrowIfCancellationRequested(); // cooperative cancellation
        await ProcessLineAsync(line, ct);
    }
}
// caller:
var cts = new CancellationTokenSource(TimeSpan.FromSeconds(30));
await ProcessLargeFileAsync(cts.Token);

A CancellationTokenSource produces a CancellationToken that's passed down through the call chain. Cancellation is cooperative — the token doesn't forcibly kill anything; your code must check IsCancellationRequested or call ThrowIfCancellationRequested() periodically, and pass the token into any awaited APIs (HttpClient, EF Core queries, Task.Delay) that accept one so they self-cancel too.

10
CPU-bound vs I/O-bound async work — how should each be implemented differently?Design

I/O-bound (DB call, HTTP call, file/network) — use async/await directly against the async API (e.g., SqlCommand.ExecuteReaderAsync); no thread is blocked while waiting, it's genuinely non-blocking at the OS level. CPU-bound (heavy computation, image processing) — there's no I/O to "wait" on, so wrap it in Task.Run(() => ...) to push it onto a thread-pool thread and free the calling thread (e.g., the UI thread), but it still consumes a real CPU core/thread while running. Using Task.Run for I/O-bound work wastes a thread pool thread for nothing; using plain await for CPU-bound work blocks whatever thread called it.

🧵 B. Threading & Concurrency (Q11–20)

11
Difference between Thread and Task?Fundamentals

Thread is a raw OS-level thread — expensive to create (~1MB stack), no built-in way to return a result or compose with other work. Task is a higher-level abstraction representing a unit of work that runs (usually) on the thread pool, supports return values, composition (ContinueWith, WhenAll), cancellation, and exception propagation. In modern C#, you almost never create a raw Thread directly — use Task.Run instead, unless you need a dedicated long-lived foreground/background thread.

12
What's a race condition, and how does "lock" prevent it?Concurrency
private readonly object _sync = new object();
private int _counter;
public void Increment()
{
    lock (_sync) { _counter++; } // only one thread inside at a time
}

A race condition happens when two or more threads read-modify-write shared state concurrently without coordination, producing a result that depends on unpredictable timing (e.g., two threads both read counter=5, both write 6, one increment is lost). lock (syntactic sugar over Monitor.Enter/Exit) ensures only one thread executes the critical section at a time, making the increment atomic relative to other locked access.

13
lock vs Monitor vs Mutex — what's the difference?Synchronization

lock is compiler sugar for Monitor.Enter/Monitor.Exit wrapped in try/finally — in-process only, fast. Monitor is the full API behind lock, also offering Wait/Pulse for signaling between threads. Mutex is a heavier, kernel-level synchronization object that can be named and shared ACROSS PROCESSES (e.g., ensure only one instance of an app runs system-wide) — but much slower than Monitor for in-process locking, so only use it when cross-process coordination is actually needed.

14
What is SemaphoreSlim and when would you use it over lock?Synchronization
private readonly SemaphoreSlim _sem = new SemaphoreSlim(3); // max 3 concurrent
public async Task CallExternalApiAsync()
{
    await _sem.WaitAsync();
    try { await _httpClient.GetAsync(url); }
    finally { _sem.Release(); }
}

lock allows exactly ONE thread in at a time and has no async-friendly wait (you can't await inside a lock's acquire). SemaphoreSlim allows up to N concurrent callers (throttling — e.g., limit concurrent outbound API calls to avoid overwhelming a downstream service) and exposes WaitAsync(), making it safe to use in async code, unlike lock.

15
Give an example of a thread-safe collection and when you'd reach for one.Collections

ConcurrentDictionary<TKey,TValue> — safe for concurrent reads/writes from multiple threads without external locking (e.g., an in-memory cache updated by multiple request threads). Also ConcurrentQueue, ConcurrentBag, and BlockingCollection<T> (bounded producer/consumer queue, great for a background-worker pipeline). Regular Dictionary<TKey,TValue> is NOT thread-safe — concurrent writes can corrupt its internal structure or throw.

16
What does the "volatile" keyword do, and when is it needed?Memory Model

volatile tells the compiler/JIT/CPU not to cache a field's value in a register or reorder reads/writes around it — every read fetches the latest value, every write is immediately visible to other threads. It's needed for simple flags read/written by multiple threads without a lock (e.g., a volatile bool _stopRequested checked in a loop). It does NOT make compound operations (like x++) atomic — for that you still need lock or Interlocked.

17
What is the Interlocked class used for?Atomics

Interlocked.Increment(ref counter), Interlocked.CompareExchange, etc. perform simple operations (increment, add, exchange) as a single atomic CPU instruction — no lock needed, much faster than lock for simple counters. Use it for hit-counters, simple flags, or lock-free algorithms; for anything beyond a single-variable atomic operation, a proper lock is usually clearer and safer.

18
Parallel.For / Parallel.ForEach vs a plain loop with Tasks — when to use which?Parallelism

Parallel.For/ForEach is designed for CPU-bound, synchronous, data-parallel work (e.g., processing 10,000 in-memory records) — it automatically partitions the work across cores and manages the degree of parallelism. It is NOT async-friendly (the body should be synchronous). For async work per item (e.g., calling an API per record), use Task.WhenAll with a list of tasks, optionally throttled with SemaphoreSlim, instead of Parallel.ForEach.

19
What causes a classic multi-lock deadlock, and how do you avoid it?Deadlocks

Thread A locks Resource1 then waits for Resource2; Thread B locks Resource2 then waits for Resource1 — neither can proceed, deadlock. Avoid by always acquiring multiple locks in a single, consistent global order across the whole codebase, keeping lock scopes as small as possible, and preferring higher-level constructs (ConcurrentDictionary, a single coarser lock) over multiple fine-grained nested locks when possible.

20
What is thread pool starvation, and how can blocking async code cause it?Performance

The thread pool has a limited number of worker threads (grows slowly under load). If many requests call blocking code (.Result, Thread.Sleep, synchronous DB calls) instead of truly async I/O, those threads sit idle-but-occupied waiting, and the pool can't service new incoming work fast enough — requests queue up and latency spikes even though CPU usage looks low. Fix: use async I/O APIs end-to-end so waiting threads are actually released back to the pool.

21
What is IHostedService, and when do you implement it directly instead of using BackgroundService?Fundamentals

IHostedService is the core interface (StartAsync/StopAsync) that the .NET Generic Host uses to run background tasks alongside the app (registered via services.AddHostedService<T>()). Implement it directly when you only need simple start/stop hooks (e.g., warm up a cache once at startup, open a message-bus connection) without a continuously running loop. For anything that needs to keep running in a loop, BackgroundService (which implements IHostedService for you) is simpler.

22
What is BackgroundService, and how does it differ from raw IHostedService?Fundamentals
public class EmailQueueWorker : BackgroundService
{
    private readonly ILogger<EmailQueueWorker> _logger;
    public EmailQueueWorker(ILogger<EmailQueueWorker> logger) => _logger = logger;

    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        while (!stoppingToken.IsCancellationRequested)
        {
            _logger.LogInformation("Checking email queue at {time}", DateTimeOffset.Now);
            await Task.Delay(TimeSpan.FromSeconds(30), stoppingToken);
        }
    }
}

BackgroundService is an abstract base class that implements IHostedService for you — you just override ExecuteAsync(CancellationToken stoppingToken) with your long-running loop logic. It handles wiring StartAsync to kick off ExecuteAsync as a background Task, and StopAsync to signal the token and await graceful completion. Raw IHostedService requires you to manage that loop/Task bookkeeping yourself.

23
How do you register a BackgroundService in Program.cs (minimal hosting)?Setup
var builder = WebApplication.CreateBuilder(args);
builder.Services.AddHostedService<EmailQueueWorker>();
var app = builder.Build();
app.Run();

AddHostedService<T>() registers the worker as a Singleton IHostedService — the host starts it automatically when the app starts, and calls StopAsync during graceful shutdown. Multiple hosted services can be registered; they all start/stop together with the app's lifetime.

24
How would you implement a periodic job (e.g., run every 5 minutes) inside a BackgroundService?Timer Pattern
protected override async Task ExecuteAsync(CancellationToken stoppingToken)
{
    using var timer = new PeriodicTimer(TimeSpan.FromMinutes(5));
    while (await timer.WaitForNextTickAsync(stoppingToken))
    {
        await ProcessOverdueFeesAsync(stoppingToken);
    }
}

PeriodicTimer (modern, .NET 6+) is the cleanest way — WaitForNextTickAsync is truly async and honors cancellation cleanly. Older style uses Task.Delay(interval, stoppingToken) in a while loop, or System.Threading.Timer with a callback (needs manual re-entrancy guarding since the callback can fire again before the previous run finishes).

25
How do you gracefully stop a BackgroundService during app shutdown?Lifecycle

When the host shuts down, it calls StopAsync(CancellationToken) on every hosted service, which (in BackgroundService's base implementation) signals the stoppingToken passed into your ExecuteAsync loop and then awaits the ExecuteAsync Task up to the shutdown timeout (default 30s, configurable via HostOptions.ShutdownTimeout). Your loop should check stoppingToken.IsCancellationRequested (or pass it into awaited calls like Task.Delay) so it exits promptly instead of being forcibly abandoned.

26
How do you access a Scoped service like DbContext from inside a BackgroundService (which is a Singleton)?DI Gotcha
public class FeeReminderWorker : BackgroundService
{
    private readonly IServiceScopeFactory _scopeFactory;
    public FeeReminderWorker(IServiceScopeFactory scopeFactory) => _scopeFactory = scopeFactory;

    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        while (!stoppingToken.IsCancellationRequested)
        {
            using (var scope = _scopeFactory.CreateScope())
            {
                var db = scope.ServiceProvider.GetRequiredService<SchoolDbContext>();
                await db.SendOverdueFeeRemindersAsync(stoppingToken);
            }
            await Task.Delay(TimeSpan.FromHours(1), stoppingToken);
        }
    }
}

You can never inject a Scoped service (like a DbContext) directly into a Singleton's constructor — that would create a "captive dependency" living far longer than intended. Instead inject IServiceScopeFactory, and create a new scope per iteration of the loop, resolving scoped services from that scope and disposing it each time.

27
BackgroundService (in-process) vs a Windows Service / standalone Worker Service template — what's the real difference?Deployment

BackgroundService is just a class that runs INSIDE your existing app's process (e.g., alongside your ASP.NET Core web app) — same deployment, same lifetime as the web host. The "Worker Service" project template creates a STANDALONE process (no web server) whose entire purpose is running hosted services — typically deployed as a Windows Service or Linux systemd daemon via UseWindowsService()/UseSystemd(). Choose standalone Worker Service when the background work needs to run independently of a web app (its own deployment/scaling/restart lifecycle).

28
How do you keep an unhandled exception inside a BackgroundService from crashing the whole application?Resilience
protected override async Task ExecuteAsync(CancellationToken stoppingToken)
{
    while (!stoppingToken.IsCancellationRequested)
    {
        try
        {
            await DoWorkAsync(stoppingToken);
        }
        catch (Exception ex) when (!stoppingToken.IsCancellationRequested)
        {
            _logger.LogError(ex, "Worker iteration failed, will retry");
        }
        await Task.Delay(TimeSpan.FromSeconds(30), stoppingToken);
    }
}

By default, an exception that escapes ExecuteAsync stops that hosted service and (depending on BackgroundServiceExceptionBehavior, default in .NET 6+) can bring down the whole host. Wrap the per-iteration work in try/catch, log it, and continue the loop — so one bad iteration doesn't kill the whole background process.

29
How would you build a queue-based background processor (e.g., process uploaded files asynchronously)?Producer/Consumer
public class FileProcessingQueue
{
    private readonly Channel<string> _channel = Channel.CreateUnbounded<string>();
    public ValueTask EnqueueAsync(string filePath) => _channel.Writer.WriteAsync(filePath);
    public IAsyncEnumerable<string> DequeueAllAsync(CancellationToken ct) => _channel.Reader.ReadAllAsync(ct);
}

public class FileProcessorWorker : BackgroundService
{
    private readonly FileProcessingQueue _queue;
    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        await foreach (var filePath in _queue.DequeueAllAsync(stoppingToken))
            await ProcessFileAsync(filePath);
    }
}

System.Threading.Channels.Channel<T> is the modern producer/consumer primitive — a controller endpoint (producer) calls EnqueueAsync, and a singleton BackgroundService (consumer) drains it with await foreach. This decouples request handling (fast, returns immediately) from slow processing (runs in the background), and is far simpler than manually juggling BlockingCollection plus a dedicated thread.

30
How do you unit test a BackgroundService?Testing

Don't test the private ExecuteAsync loop directly — instead, extract the actual business logic into a separate injectable class/method that ExecuteAsync merely calls, and unit test THAT in isolation with mocked dependencies. For the worker itself, an integration-style test can construct it with test doubles, call StartAsync with a token that's cancelled shortly after (e.g., via CancellationTokenSource(TimeSpan)), and assert the expected side effects occurred (e.g., verify the mock repository's method was called).

31
What is SignalR and what problem does it solve?Fundamentals

SignalR is an ASP.NET Core library for real-time, bidirectional communication between server and clients (browser, mobile, desktop) — the server can PUSH data to connected clients instantly (e.g., a new notification, live attendance update, chat message) instead of the client having to poll an API repeatedly. It abstracts away the underlying transport and connection management, so you write plain C# method calls instead of managing raw WebSocket frames yourself.

32
What transports does SignalR use, and how does it pick one?Transports

SignalR negotiates the best available transport in order of preference: WebSockets (full-duplex, most efficient) → Server-Sent Events (SSE, server-to-client streaming over HTTP) → Long Polling (client repeatedly asks "any updates?", fallback for old browsers/proxies that block WebSockets). The negotiation happens automatically at connection start unless you force a specific transport via client configuration.

33
What is a Hub in SignalR?Core Concept
public class NotificationHub : Hub
{
    public async Task SendToAll(string message)
    {
        await Clients.All.SendAsync("ReceiveNotification", message);
    }
}
// Program.cs: app.MapHub<NotificationHub>("/hubs/notifications");

A Hub is the server-side pipeline endpoint that clients connect to — it's a high-level RPC abstraction: clients can invoke public methods on the Hub, and the Hub can invoke methods registered on connected clients via the Clients property (Clients.All, Clients.Caller, Clients.Group(...), Clients.User(...)). Each Hub instance is transient — created per method call, so don't store per-connection state as instance fields; use a separate connection-tracking service instead.

34
How does a client invoke a server Hub method, and how does the server push to the client?Two-Way
// JS client
const connection = new signalR.HubConnectionBuilder().withUrl("/hubs/notifications").build();
connection.on("ReceiveNotification", msg => console.log(msg)); // server -> client
await connection.start();
await connection.invoke("SendToAll", "Attendance updated");   // client -> server

Client → Server: connection.invoke("MethodName", args) calls a public method on the Hub by name. Server → Client: inside the Hub, await Clients.Caller.SendAsync("EventName", data) invokes a named handler the client registered with connection.on("EventName", handler). It's not strongly-typed by default (string method names), though you can use strongly-typed hubs (Hub<TClient>) for compile-time safety on the server-to-client direction.

35
What are SignalR Groups, and how would you use them (e.g., notify only one class/room)?Groups
public override async Task OnConnectedAsync()
{
    var classId = Context.GetHttpContext().Request.Query["classId"];
    await Groups.AddToGroupAsync(Context.ConnectionId, $"class-{classId}");
    await base.OnConnectedAsync();
}
public async Task NotifyClass(string classId, string message)
    => await Clients.Group($"class-{classId}").SendAsync("ClassNotice", message);

Groups let you organize connections into named buckets (e.g., "class-10A") so a message can target just that subset instead of everyone (Clients.All) or one connection. Groups.AddToGroupAsync/RemoveFromGroupAsync manage membership; group membership is NOT persisted — it's tied to the current connection, so a reconnect must re-join groups.

36
How do you send a message to one specific logged-in user (not just one connection)?Targeting

Clients.User(userId) sends to ALL connections belonging to that authenticated user identity (a user could have multiple tabs/devices open) — SignalR maps connections to users via the ClaimTypes.NameIdentifier claim by default (or a custom IUserIdProvider). Use Clients.Client(connectionId) instead when you specifically need to target one single connection rather than all of a user's sessions.

37
How does SignalR scale across multiple server instances (load-balanced)?Scaling

Out of the box, a Hub only knows about connections on its OWN server instance — if User A connects to Server 1 and you call Clients.User(A) from code running on Server 2, it won't reach them. A "backplane" solves this: a shared pub/sub layer (Redis backplane, or the managed Azure SignalR Service) that relays messages between all server instances, so any instance can broadcast to any connected client regardless of which server it's attached to. Azure SignalR Service also offloads the persistent client connections entirely from your app servers.

38
How do you secure a SignalR Hub with authentication/authorization?Security
[Authorize]
public class NotificationHub : Hub
{
    [Authorize(Roles = "Admin")]
    public async Task BroadcastAnnouncement(string msg) => await Clients.All.SendAsync("Announcement", msg);
}

Standard ASP.NET Core [Authorize] attributes work directly on Hubs and individual Hub methods, same as controllers. The tricky part is the access token: browser WebSocket/SSE connections can't set an Authorization header, so SignalR's JS client sends the JWT as a query string parameter (accessTokenFactory) on the negotiate/connect request, which the server is configured to read via OnMessageReceived in the JWT bearer options for the Hub's path.

39
How do you handle automatic reconnection when a SignalR connection drops?Resilience
const connection = new signalR.HubConnectionBuilder()
    .withUrl("/hubs/notifications")
    .withAutomaticReconnect([0, 2000, 5000, 10000])
    .build();
connection.onreconnected(connectionId => rejoinGroups(connectionId));

The client's .withAutomaticReconnect() makes the JS client retry with the given backoff delays on connection loss. Server-side, on reconnect you get a brand-new ConnectionId, so any Group memberships are lost and must be re-established (typically by re-sending the "join group" call, or handling it again in OnConnectedAsync using info like a query-string classId).

40
Give a concrete example: real-time attendance/notification system for a student management app using SignalR.Applied

When a teacher marks attendance via a controller action, after saving to the DB the controller resolves IHubContext<AttendanceHub> (not the Hub class itself, since controllers aren't Hub clients) and calls await hubContext.Clients.Group($"class-{classId}").SendAsync("AttendanceMarked", studentId, status). Parents/students subscribed to that class group receive an instant push — no polling needed — updating their dashboard live. This is the standard pattern: business logic lives in controllers/services, they inject IHubContext<T> to push notifications, while the Hub class itself just handles direct client RPC and group membership.

🧩 D. Delegates, Events & Generics (Q41–50)

41
What is a delegate in C#?Fundamentals

A delegate is a type-safe function pointer — an object that holds a reference to a method (or several, for multicast) matching a specific signature, which can be invoked, passed as a parameter, stored in a field, or returned from a method. It's the foundation for callbacks, events, and LINQ's functional-style APIs (Func/Action are just predefined generic delegates).

42
Difference between delegate, Func, Action, and Predicate?Built-ins
TypeSignatureExample
delegateCustom, any signature you definedelegate int Calc(int a, int b);
Action<T...>Takes params, returns voidAction<string> log = msg => Console.WriteLine(msg);
Func<T...,TResult>Takes params, returns TResult (last type param)Func<int,int,int> add = (a,b) => a+b;
Predicate<T>Takes T, returns boolPredicate<int> isEven = n => n % 2 == 0;

Func/Action/Predicate are just generic delegate types the BCL already defines, so you rarely need to declare a custom delegate keyword type anymore — reach for a custom delegate mainly when you want a specifically named type for readability/domain clarity.

43
What is a multicast delegate?Fundamentals

All C# delegates are inherently multicast — using += you can attach multiple method targets to one delegate instance; invoking it calls each in the order added. If the delegate has a return value, only the LAST invoked method's return value is what you get back from a direct call (others' results are discarded) — this is a classic gotcha, so multicast is mainly useful for void-returning notification-style callbacks like events.

44
What is an event, and how does it differ from a plain public delegate field?Encapsulation
public class FeeService
{
    public event EventHandler<FeePaidEventArgs> FeePaid;
    protected virtual void OnFeePaid(FeePaidEventArgs e) => FeePaid?.Invoke(this, e);
}

A plain public delegate field can be invoked, replaced entirely (=), or set to null from OUTSIDE the class — dangerous. An event restricts external code to only +=/-= (subscribe/unsubscribe) — only the declaring class can raise it (call Invoke) or clear it. This encapsulation is exactly why events exist instead of just using public delegate fields directly.

45
How do C# events implement the Observer pattern?Design Pattern

The Observer pattern is: a subject notifies multiple interested observers when its state changes, without the subject needing to know concrete observer types. A C# event IS this pattern built into the language — the publishing class is the "subject," each += subscriber is an "observer," and raising the event notifies all of them. It's a common interview point to connect this classic GoF pattern directly to a language feature you already use daily.

46
Why use List<T> (generics) instead of the old non-generic ArrayList?Generics Basics

ArrayList stores everything as object — adding a value type causes boxing (heap allocation) and every read requires an unboxing cast, with no compile-time type safety (you could accidentally add mismatched types and only find out at runtime). List<T> is type-safe at compile time, avoids boxing for value types, and is faster because the JIT can generate specialized code per concrete type. There's essentially no reason to use ArrayList/Hashtable/etc. in new code.

47
What are generic constraints, and why would you use "where T : IComparable<T>"?Constraints
public class SortedBag<T> where T : IComparable<T>
{
    private readonly List<T> _items = new();
    public void Add(T item)
    {
        int i = _items.BinarySearch(item);
        _items.Insert(i < 0 ? ~i : i, item);
    }
}

Constraints (where T : class, struct, new(), a base class, or an interface) restrict which types can be substituted for T, which in turn UNLOCKS operations on T that wouldn't otherwise compile — e.g., without where T : IComparable<T>, you can't call .CompareTo on a generic T because the compiler only knows it's "some type," not that it's comparable.

48
What is covariance/contravariance (in/out) in generics?Variance
IEnumerable<Student> students = GetStudents();
IEnumerable<object> asObjects = students; // OK — covariant (out T)

Action<object> logAny = obj => Console.WriteLine(obj);
Action<string> logString = logAny;        // OK — contravariant (in T)

Covariance (out T, e.g. IEnumerable<out T>) lets you use a MORE DERIVED type where a less derived one is expected — safe because the interface only produces/returns T, never accepts it as input. Contravariance (in T, e.g. Action<in T>) lets you use a LESS DERIVED type where a more derived one is expected — safe because the interface only consumes T as input. Both only apply to interface/delegate generic type parameters, not classes, and only in the direction that keeps type-safety intact.

49
What is a lambda expression, and how does it relate to delegates?Syntax

A lambda (x => x * 2) is a concise inline syntax for defining a method body without formally declaring a named method — the compiler converts it into either a delegate instance (Func/Action/custom delegate) or an Expression Tree, depending on the target type expected (e.g., LINQ-to-Objects wants a delegate, LINQ-to-Entities/IQueryable wants an Expression Tree so it can translate it to SQL).

50
Anonymous method vs lambda expression — is there any real difference today?History
Func<int,int> anon = delegate(int x) { return x * 2; }; // C# 2.0 style
Func<int,int> lambda = x => x * 2;                        // C# 3.0+ style, preferred

Anonymous methods (delegate(...) { ... }) were C# 2.0's way to write an inline method without a name. Lambdas (C# 3.0+) are strictly more capable and concise — they support expression-bodied syntax, can be converted to Expression Trees (anonymous methods cannot), and support implicit parameter typing. There's no reason to write anonymous-method syntax in new code; lambdas fully superseded it.

🔎 E. LINQ, Expression Trees & Memory/GC (Q51–70)

51
What is deferred execution in LINQ, and how can it "bite you"?Gotcha
var query = students.Where(s => s.IsActive); // NOT executed yet
students.Add(new Student { IsActive = true }); // mutate source
foreach (var s in query) { ... } // executes NOW — includes the new student!

Most LINQ operators (Where, Select, OrderBy) build up a query definition but don't actually run until you enumerate it (foreach, ToList, etc.) — this is deferred execution. The gotcha: if the underlying source changes between defining the query and enumerating it, the results reflect the LATEST state, not a snapshot from when you wrote the query — surprising bugs if you assumed otherwise. Calling .ToList() immediately forces execution and captures a snapshot.

52
IEnumerable vs IQueryable — what's the practical difference?Core

IEnumerable<T> LINQ operators work against in-memory objects, compiling lambdas into actual delegates executed in .NET (LINQ-to-Objects). IQueryable<T> (used by EF Core) builds an Expression Tree instead of executing anything — the provider (EF Core) translates that expression tree into SQL and only queries the database when enumerated. Practical trap: calling .AsEnumerable() or .ToList() too early switches from IQueryable to IEnumerable, meaning subsequent filters run in application memory instead of in the database — potentially pulling the whole table across the wire first.

53
First() vs FirstOrDefault() vs Single() vs SingleOrDefault() — when do you use each?API Choice

First() throws if the sequence is empty, returns the first element otherwise (use when you're SURE at least one exists, and want a loud failure if that assumption breaks). FirstOrDefault() returns default(T)/null instead of throwing on empty — use for optional lookups. Single() throws if there's zero OR more than one match — use when exactly one is a business invariant (e.g., a primary key lookup). SingleOrDefault() returns default on zero matches but still throws on more than one — good for "should be unique if it exists" lookups like a user by unique email.

54
How does GroupBy work in LINQ?Aggregation
var byClass = students.GroupBy(s => s.ClassId)
    .Select(g => new { ClassId = g.Key, Count = g.Count(), Names = g.Select(s => s.Name) });

GroupBy(keySelector) produces a sequence of IGrouping<TKey,TElement> — each grouping acts like a mini-collection of the matching elements, keyed by .Key. It's the LINQ equivalent of SQL's GROUP BY; against IQueryable (EF Core) it translates into an actual SQL GROUP BY, but you must be careful — some complex GroupBy shapes don't translate well and EF may need to pull data client-side or throw a translation exception.

55
What is the N+1 query problem, and how does it happen with LINQ-to-Entities?EF Performance
// BAD: N+1 — one query for students, then one query PER student for fees
foreach (var s in db.Students.ToList())
    Console.WriteLine(s.Fees.Count); // lazy-loads fees per student = N extra queries

// GOOD: one query total
var students = db.Students.Include(s => s.Fees).ToList();

N+1 happens when you fetch N parent rows in one query, then lazily trigger a separate query for each row's related data inside a loop — 1 query becomes 1 + N queries total, which is disastrous for performance at scale. Fix with eager loading (.Include()) or a single projection that gets everything needed up front.

56
What is an Expression Tree, and how is it different from a compiled delegate (Func<T>)?Advanced
Expression<Func<Student,bool>> expr = s => s.IsActive; // DATA describing the logic
Func<Student,bool> compiled = expr.Compile();            // now it's runnable code

A Func<T> lambda compiles directly to IL — it's executable code you can only call, not inspect. An Expression<Func<T>> instead builds a TREE OF OBJECTS describing the logic (a BinaryExpression, MemberExpression, etc.) that can be inspected, transformed, or translated to another language (like SQL) at runtime — this is exactly how EF Core turns your C# lambda into a SQL WHERE clause. You can call .Compile() on an expression tree to turn it into an invokable delegate if needed.

57
How does "yield return" work internally?Iterators
public IEnumerable<int> GetEvenNumbers(int max)
{
    for (int i = 0; i <= max; i += 2)
        yield return i; // pauses here, resumes on next MoveNext()
}

The compiler transforms a method using yield return into a hidden state-machine class implementing IEnumerator<T>. Each call to MoveNext() resumes execution exactly where the last yield return left off, runs until the next yield (or the method ends), and produces one value lazily — nothing runs until the caller actually iterates, which is why iterator methods support infinite sequences and avoid materializing a whole collection in memory upfront.

58
Select vs SelectMany — what's the difference?Projection
var perClassLists = classes.Select(c => c.Students);       // IEnumerable<List<Student>> — nested
var allStudentsFlat = classes.SelectMany(c => c.Students);  // IEnumerable<Student> — flattened

Select projects each item to a new shape 1-to-1, preserving the outer sequence structure — if each item maps to a collection, you get a sequence-of-sequences (nested). SelectMany projects each item to a collection AND flattens all those collections into one single flat sequence — the standard tool whenever you need "all X across all Y" in one flat list.

How would you join two in-memory collections using LINQ?Joins
var result = students.Join(fees,
    s => s.Id, f => f.StudentId,
    (s, f) => new { s.Name, f.Amount });

// or query syntax:
var result2 = from s in students
              join f in fees on s.Id equals f.StudentId
              select new { s.Name, f.Amount };

Enumerable.Join performs an inner join between two sequences on matching keys, similar to SQL JOIN. For a LEFT JOIN equivalent, combine GroupJoin with DefaultIfEmpty() on the grouped results.

60
Why does WHERE you call .ToList() in a LINQ chain matter for performance?Performance
// BAD (against EF): pulls ALL rows into memory, THEN filters in C#
var bad = db.Students.ToList().Where(s => s.IsActive);

// GOOD: filter translates to SQL WHERE, only matching rows come over the wire
var good = db.Students.Where(s => s.IsActive).ToList();

Calling .ToList() too early against an IQueryable source forces immediate execution of everything up to that point — any filtering/projection chained AFTER it now runs in application memory (LINQ-to-Objects) instead of being pushed down to the database as SQL. Always finish building the full IQueryable expression (Where/Select/OrderBy) and call ToList()/ToListAsync() LAST.

61
How does the .NET Garbage Collector work (generations 0/1/2)?GC Fundamentals

The GC uses a generational model based on the observation that most objects die young. New objects start in Gen 0 (small, collected frequently and cheaply). Objects that survive a Gen 0 collection get promoted to Gen 1 (a buffer zone, collected less often), and survivors of that get promoted to Gen 2 (long-lived objects — collected rarely, but a Gen 2 collection is the most expensive since it scans everything). There's also a separate Large Object Heap (LOH) for objects >85KB, collected only during Gen 2 collections.

62
Value types vs reference types — how are they stored in memory?Memory Model

Value types (struct, int, bool, enum, DateTime) hold their data directly — when declared as a local variable, they live on the STACK (fast to allocate/deallocate, no GC involvement) unless boxed or captured in a class/closure, in which case they move to the heap alongside their containing object. Reference types (class, string, array, delegate, interface-typed variable) always live on the HEAP, and the variable just holds a reference (pointer) to that heap location — assignment copies the reference, not the data, which is why mutating an object through one reference is visible through another.

63
What is IDisposable and the Dispose pattern, and why does it matter?Resource Cleanup

The GC only knows how to reclaim managed memory — it has no idea how to release unmanaged resources (file handles, network sockets, database connections, OS handles). IDisposable.Dispose() gives a deterministic, explicit way to release those resources exactly when you're done, rather than waiting for an unpredictable GC pass (or never, if the object is long-lived). Always wrap disposable objects in using so Dispose runs even on exceptions.

64
Dispose() vs a Finalizer (~ClassName) — what's the difference?Cleanup

Dispose() is called explicitly (usually via using) — deterministic, immediate, cheap. A finalizer (~MyClass()) is called by the GC nondeterministically, only if the object wasn't already disposed — it exists as a safety net for unmanaged resources in case someone forgot to call Dispose, but finalization is expensive (objects with finalizers survive at least one extra GC generation) and delayed (runs on a dedicated finalizer thread, timing not guaranteed). Best practice: implement the full Dispose pattern (Dispose + finalizer + GC.SuppressFinalize(this) in Dispose) only for classes that directly own unmanaged handles; for everything else, just implement Dispose without a finalizer.

65
Give an example of a "memory leak" in .NET even though it has a GC.Gotcha
publisher.SomethingHappened += subscriber.OnSomethingHappened; // subscribe
// ... subscriber logically goes out of scope, but is never explicitly removed
// publisher (long-lived) still holds a reference to subscriber via the event
// -> subscriber can NEVER be garbage collected while publisher is alive

The GC only collects objects with NO reachable references — it can't tell the difference between a "real" reference and one you forgot about. The classic leak: a long-lived publisher object holds an event subscription to a short-lived object that never unsubscribes (-=) — the subscriber is kept alive forever by that reference, even though logically nobody "uses" it anymore. Same pattern with static collections that only ever get items added, never removed.

What are weak references, and when would you use one?Advanced Memory
var weak = new WeakReference<LargeCacheEntry>(entry);
if (weak.TryGetTarget(out var cached)) { /* still alive, use it */ }
else { /* was collected, recreate it */ }

A WeakReference<T> holds a reference to an object WITHOUT preventing the GC from collecting it — useful for caches where you want to reuse an object if it's still around, but don't want the cache itself to be the reason it never gets collected under memory pressure. Rare in everyday business code; mostly seen in caching layers or large-object pooling scenarios.

67
What is boxing/unboxing, and why is it a performance concern?Performance
int i = 42;
object boxed = i;        // boxing: heap allocation, copies the int into a new object
int back = (int)boxed;   // unboxing: copies the value back out, with a type check

Boxing wraps a value type in a heap-allocated object so it can be treated as object (or an implemented interface) — this costs a heap allocation and a memory copy every single time it happens, plus extra GC pressure. It commonly sneaks in via non-generic collections (ArrayList), string.Format with value-type args (mitigated in modern .NET), or passing a struct where an interface/object parameter is expected. Generics (List<T>, Func<T>) avoid this entirely for value types.

68
Server GC vs Workstation GC — what's the difference?GC Modes

Workstation GC is optimized for desktop apps — runs mostly on the same thread as the app with minimal pause impact on responsiveness, single heap. Server GC (used by default for ASP.NET Core web apps) creates one heap PER CPU CORE and collects them in parallel on dedicated GC threads — much higher throughput for multi-core server workloads at the cost of higher baseline memory usage. Configurable via ServerGarbageCollection in the project's runtime config.

69
What is Span<T>, and why is it useful for performance?Modern Perf
ReadOnlySpan<char> span = "Hello, World".AsSpan(7, 5); // "World" — no new string allocated

Span<T> is a stack-only, type-safe view over a contiguous region of memory (an array, a string, or stack-allocated memory) that lets you slice/manipulate data WITHOUT allocating a new copy — e.g., substring-like operations without allocating new string objects. It's a big win in hot paths (parsing, string processing) for reducing GC pressure, though the "stack-only" restriction (it's a ref struct) means you can't store it in a field of a heap-allocated class or use it across await boundaries.

70
How can a static collection cause an unintended memory leak?Gotcha

A static field lives for the entire lifetime of the application domain. If you keep adding items to a static List<T>/Dictionary (e.g., a "cache" that never expires or evicts) without ever removing entries, it grows unbounded — every item added is reachable forever via that static root, so the GC can never reclaim any of it, no matter how much memory pressure exists. Fix with bounded caches (e.g., MemoryCache with eviction/expiration policies) instead of raw static collections.

🏗️ F. Exception Handling, Design Patterns & Dependency Injection (Q71–100)

"throw;" vs "throw ex;" — why does it matter?Gotcha
catch (Exception ex)
{
    LogError(ex);
    throw;      // GOOD: preserves original stack trace
    // throw ex;  BAD: resets stack trace to THIS line, losing where it actually happened
}

throw; rethrows the current exception while preserving its original stack trace, so you can still see exactly where it was first thrown. throw ex; throws the SAME exception object but resets its stack trace to the current catch block's location — you lose the original failure location, making production debugging much harder.

72
.NET doesn't have Java-style checked exceptions — how do you decide what to catch?Philosophy

Since the compiler never forces you to declare or catch specific exceptions, the convention is: catch SPECIFIC exception types you can meaningfully recover from or add context to (e.g., catch (SqlException ex) when (ex.Number == 2601) for a duplicate key), and let everything else propagate up to a top-level/global handler (like ASP.NET Core's exception-handling middleware) that logs it and returns a generic error response. Avoid blanket catch (Exception) that silently swallows errors — that hides real bugs.

How does "using" relate to exception safety for resource cleanup?Best Practice
using (var conn = new SqlConnection(connStr))
{
    conn.Open();
    DoWork(conn); // if this throws...
} // ...Dispose() STILL runs here, connection is still released

using compiles down to a try/finally block, guaranteeing Dispose() runs whether the block completes normally OR an exception is thrown partway through — this is the whole reason it exists: manual try/finally cleanup is easy to forget, using makes it automatic and impossible to skip accidentally.

Explain the Singleton pattern and a thread-safe implementation in C#.Pattern

    private ConfigManager() { }

Singleton ensures only ONE instance of a class ever exists, with a single global access point. The naive implementation (check-then-create in a getter) isn't thread-safe — two threads can both pass the null-check simultaneously and create two instances. Lazy<T> handles thread-safe lazy initialization for you (default mode uses a lock internally), which is far simpler and safer than hand-rolling double-checked locking.

Explain the Repository and Unit of Work patterns.Pattern

Repository abstracts data access behind an interface (e.g., IStudentRepository.GetById, .Add) so business logic doesn't depend directly on ADO.NET/EF Core specifics — easier to unit test with a fake/mock repository. Unit of Work coordinates multiple repository operations into a single atomic transaction/SaveChanges call (e.g., add a student AND their first fee record, commit both together or roll back both). In EF Core, DbContext itself already acts as a Unit of Work (its SaveChanges() commits all tracked changes together), which is why some teams consider a separate Repository layer over EF Core redundant — a valid interview talking point either way.

What is the Dependency Inversion Principle, and how does a DI container implement it?SOLID

What is the "captive dependency" problem when injecting a Scoped service into a Singleton?DI Gotcha

If a Singleton captures a Scoped service in its constructor, that Scoped instance (e.g., a DbContext meant to live only per-request) gets held onto for the ENTIRE APPLICATION LIFETIME instead of being released after its intended request — it becomes a de-facto singleton itself, leading to stale/shared state bugs and potential thread-safety issues since DbContext isn't thread-safe. What is a record type, and how does it differ from a class?Modern C#

.NET Framework vs .NET Core vs .NET 5+/8 — what should you say if asked?Ecosystem

== vs .Equals() — and why override both Equals AND GetHashCode together?Equality
public class Money
{
    public decimal Amount { get; }
    public override bool Equals(object? obj) => obj is Money m && m.Amount == Amount;
    public override int GetHashCode() => Amount.GetHashCode();
}

For reference types, == defaults to reference equality (same object in memory) unless overloaded; .Equals() is virtual and can be overridden per-class for value-based comparison. If you override Equals without also overriding GetHashCode, you break the contract that "equal objects must have equal hash codes" — hash-based collections like Dictionary/HashSet will then behave incorrectly (two "equal" objects can land in different buckets and appear as distinct entries).

struct vs class — when would you actually choose struct?Type Design

Choose struct for small (generally <16-24 bytes), immutable, short-lived, value-semantic data (a Point, a Money amount, a coordinate) where avoiding heap allocation/GC pressure matters — structs live on the stack (or inline in containing objects) and are copied by value on assignment/passing. Default to class for everything else, especially anything with identity, mutable shared state, or that's larger — copying a large struct by value on every assignment/parameter pass is actually SLOWER than passing a reference.

Abstract class vs interface — and what changed with default interface methods?OOP

An abstract class can hold state (fields), constructors, and a mix of implemented + abstract members, but a class can inherit from only ONE abstract class (single inheritance). An interface traditionally defined only a contract (no implementation, no fields) but a class can implement MANY interfaces. Since C# 8, interfaces can have default method implementations — narrowing the gap somewhat — but interfaces still can't hold instance fields/state, which remains the key practical distinction.

What is Channel<T>, and how is it used for producer-consumer scenarios?Modern Concurrency

lock vs SemaphoreSlim.WaitAsync() — why can't you "await" inside a lock block?Async + Locking

lock is built on Monitor, which is tied to the CURRENT THREAD — the thread that entered the lock must be the one that exits it. But an await can resume on a DIFFERENT thread pool thread after the awaited operation completes, which would violate that thread-affinity and the compiler flags it as an error (you literally can't write await inside a lock block). SemaphoreSlim.WaitAsync()/Release() has no such thread affinity — it's designed to be safely awaited and released from different threads, making it the correct choice for protecting a critical section that contains async work.

    public async Task<List<Student>> GetRosterAsync(int classId)
    {
        return await _cache.GetOrCreateAsync($"roster-{classId}", async entry =>
        {
            entry.SlidingExpiration = TimeSpan.FromMinutes(10);
            return await _db.Students.Where(s => s.ClassId == classId).ToListAsync();
        });
    }
}

IMemoryCache (in-process, per-server) is registered via AddMemoryCache() and offers GetOrCreateAsync to fetch-or-populate a cache entry atomically, with expiration policies (absolute, sliding) so entries don't live forever. For a multi-server deployment where all instances need the same cached data, use a distributed cache (Redis via IDistributedCache) instead — IMemoryCache is per-instance and won't be consistent across servers.

How do you implement global exception handling in ASP.NET Core?Error Handling
app.UseExceptionHandler(errApp =>
{
    errApp.Run(async context =>
    {
        var feature = context.Features.Get<IExceptionHandlerFeature>();
        _logger.LogError(feature?.Error, "Unhandled exception");
        context.Response.StatusCode = 500;
        await context.Response.WriteAsJsonAsync(new { error = "Something went wrong" });
    });
});

UseExceptionHandler middleware (registered early in the pipeline) catches any unhandled exception thrown further down the pipeline, logs it centrally, and returns a consistent error response — instead of leaking a stack trace to the client or crashing the request. .NET 8 also introduced IExceptionHandler as a more testable, DI-friendly alternative to the inline delegate approach.

[Authorize] // on controllers/actions that require a valid token

On login, the server issues a signed JWT containing claims (user id, roles, expiry) after validating credentials. The client sends it as Authorization: Bearer <token> on subsequent requests. AddJwtBearer configures the middleware to validate the token's signature, issuer, audience, and expiry on every request — if valid, HttpContext.User is populated from its claims and [Authorize]/role checks work against it, all without a server-side session store.

How would you debug a production .NET app with high CPU or a memory leak?Diagnostics

Use the dotnet-counters global tool for a live view of GC, thread pool, and request metrics without attaching a debugger. For high CPU, dotnet-trace collect captures a CPU profile you can open in a flame-graph viewer to see which methods are hot. For suspected memory leaks, dotnet-gcdump/dotnet-dump collect captures a memory snapshot analyzable in Visual Studio or PerfView to see which object types and roots are holding the most memory (often revealing exactly the "forgotten event subscription" or "unbounded static collection" pattern from Q65/Q70).
