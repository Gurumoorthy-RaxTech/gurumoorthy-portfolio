📚 Core Concepts

1. Connected vs Disconnected Architecture

Connected: connection open pannitu irukkanum, appadiye read pannanum (SqlDataReader) — fast, but connection busy-a irukkum. Disconnected: data-va DataSet/DataTable-la load pannitu connection close pannidalam — connection free aagidum, but data stale aagalam (namba manually refresh pannanum).

AspectConnected (SqlDataReader)Disconnected (DataSet/DataAdapter)
Connection stateOpen throughout readOpen only during Fill(), then closed
DirectionForward-only, read-onlyCan navigate any direction, editable in memory
SpeedFastest — no extra memory overheadSlower — builds in-memory tables
Best forSimple read-and-bind, high-throughput APIsOffline editing, multiple related tables, WinForms grids

2. SqlConnection & Connection Pooling

public class StudentRepository
{
    private readonly string _connStr;
    public StudentRepository(string connStr) => _connStr = connStr;

    public List<Student> GetActiveStudents(int schoolId)
    {
        var list = new List<Student>();
        using (var conn = new SqlConnection(_connStr))
        {
            conn.Open();
            using (var cmd = new SqlCommand(
                "SELECT Id, Name, ClassId FROM Students WHERE SchoolId=@SchoolId AND IsActive=1", conn))
            {
                cmd.Parameters.AddWithValue("@SchoolId", schoolId);
                using (var reader = cmd.ExecuteReader())
                {
                    while (reader.Read())
                    {
                        list.Add(new Student
                        {
                            Id = reader.GetInt32(0),
                            Name = reader.GetString(1),
                            ClassId = reader.GetInt32(2)
                        });
                    }
                }
            }
        } // connection auto-closed and RETURNED TO POOL here, not destroyed
        return list;
    }
}

Connection Pooling — enna nu
conn.Close() or using block exit aagumbodhu, actual TCP connection destroy aagadhu — ADO.NET adha oru "pool"-la vachukum. Next time nee same connection string-oda open pannumbodhu, pool-la irundhu already-open connection-a reuse pannum. Idhu vera oru new TCP handshake pannradha vida ​romba fast. Min Pool Size / Max Pool Size connection string-la configure pannalam.
Common mistake: Forgetting using or not calling Close() — connections leak, pool exhausts, and you get "Timeout expired. The timeout period elapsed prior to obtaining a connection from the pool" in production under load.

3. SqlCommand — ExecuteReader vs ExecuteScalar vs ExecuteNonQuery

MethodReturnsWhen to use
ExecuteReader()SqlDataReader (row-by-row stream)SELECT that returns multiple rows/columns
ExecuteScalar()object (first column of first row)SELECT COUNT(*), single aggregate value, SCOPE_IDENTITY()
ExecuteNonQuery()int (rows affected)INSERT / UPDATE / DELETE
// ExecuteScalar example — get total fee count
using var cmd = new SqlCommand("SELECT COUNT(*) FROM Fees WHERE StudentId=@Id", conn);
cmd.Parameters.AddWithValue("@Id", studentId);
int count = (int)cmd.ExecuteScalar();

// ExecuteNonQuery example — insert and return rows affected
using var cmd2 = new SqlCommand(
    "UPDATE Students SET IsActive=0 WHERE Id=@Id", conn);
cmd2.Parameters.AddWithValue("@Id", studentId);
int rowsAffected = cmd2.ExecuteNonQuery();

4. Parameterized Queries — SQL Injection Prevention

❌ DANGEROUS: "SELECT * FROM Students WHERE Name = '" + nameInput + "'" — user types ' OR '1'='1 and gets every row, or worse, drops a table.
✅ SAFE: cmd.Parameters.AddWithValue("@Name", nameInput) with WHERE Name = @Name in SQL — DB driver sends the value separately, never concatenated into the SQL text.

Interview la idhu almost guaranteed question — "how do you prevent SQL injection in ADO.NET" nu kekkalam. Always answer: parameterized queries, never string concatenation.

5. SqlDataAdapter + DataSet/DataTable (Disconnected Architecture)

public DataTable GetStudentsTable(int schoolId)
{
    var dt = new DataTable();
    using var conn = new SqlConnection(_connStr);
    using var da = new SqlDataAdapter(
        "SELECT Id, Name, ClassId FROM Students WHERE SchoolId=@SchoolId", conn);
    da.SelectCommand.Parameters.AddWithValue("@SchoolId", schoolId);
    da.Fill(dt);          // opens connection, fills DataTable, closes connection
    return dt;             // dt can now be bound to a DataGridView, edited offline
}

// Pushing edits back to the database:
public void SaveChanges(DataTable dt)
{
    using var conn = new SqlConnection(_connStr);
    using var da = new SqlDataAdapter("SELECT Id, Name, ClassId FROM Students", conn);
    using var builder = new SqlCommandBuilder(da);   // auto-generates Insert/Update/Delete
    da.Update(dt);          // pushes only CHANGED rows back to DB
}

6. Transactions — SqlTransaction

using var conn = new SqlConnection(_connStr);
conn.Open();
using var txn = conn.BeginTransaction();
try
{
    using (var cmd1 = new SqlCommand(
        "INSERT INTO Students (Name, SchoolId) VALUES (@Name, @SchoolId); SELECT SCOPE_IDENTITY();",
        conn, txn))
    {
        cmd1.Parameters.AddWithValue("@Name", "Arun");
        cmd1.Parameters.AddWithValue("@SchoolId", 1);
        int newStudentId = Convert.ToInt32(cmd1.ExecuteScalar());

        using var cmd2 = new SqlCommand(
            "INSERT INTO Fees (StudentId, Amount) VALUES (@StudentId, @Amount)", conn, txn);
        cmd2.Parameters.AddWithValue("@StudentId", newStudentId);
        cmd2.Parameters.AddWithValue("@Amount", 5000);
        cmd2.ExecuteNonQuery();
    }
    txn.Commit();     // both inserts succeed together
}
catch
{
    txn.Rollback();   // if fee insert fails, student insert is undone too — no orphan student
    throw;
}

Interview point: enna transaction venum-nu kekka, "atomicity" solradhu — multiple related writes ella-um succeed pannanum, illa ellam rollback aganum. ZenCampus example: admit student + create fee record — rendume ஒரே transaction-la.

7. Stored Procedure Call with Output Parameter

using var cmd = new SqlCommand("sp_GetFeeSummary", conn)
{
    CommandType = CommandType.StoredProcedure
};
cmd.Parameters.AddWithValue("@SchoolId", schoolId);

var totalParam = new SqlParameter("@TotalCount", SqlDbType.Int)
{
    Direction = ParameterDirection.Output
};
cmd.Parameters.Add(totalParam);

using (var reader = cmd.ExecuteReader())
{
    while (reader.Read()) { /* read fee rows */ }
}
int total = (int)totalParam.Value;   // read AFTER reader is closed
Common gotcha: output parameter value is only populated correctly after the SqlDataReader is fully read and closed — reading it too early gives garbage or DBNull.

8. Async ADO.NET

public async Task<List<Student>> GetActiveStudentsAsync(int schoolId)
{
    var list = new List<Student>();
    await using var conn = new SqlConnection(_connStr);
    await conn.OpenAsync();
    await using var cmd = new SqlCommand("SELECT Id, Name FROM Students WHERE SchoolId=@Id", conn);
    cmd.Parameters.AddWithValue("@Id", schoolId);
    await using var reader = await cmd.ExecuteReaderAsync();
    while (await reader.ReadAsync())
        list.Add(new Student { Id = reader.GetInt32(0), Name = reader.GetString(1) });
    return list;
}

Async methods free up the thread while waiting for I/O (network round-trip to SQL Server) — important for Web API scalability under load. WinForms desktop apps use async too, to avoid freezing the UI thread (details in WinForms guide).

10. MARS — Multiple Active Result Sets

By default one SqlConnection can only have ONE open SqlDataReader at a time. MARS (MultipleActiveResultSets=True in connection string) allows multiple open readers/commands on the same connection simultaneously — useful in some interleaved-read scenarios, but generally EF Core enables it by default; in raw ADO.NET, prefer opening a second connection or fully reading one result set before starting the next.

public (List<Fee> fees, int total) GetFeeSummary(int schoolId)

public void AdmitStudentWithFee(string name, int schoolId, decimal feeAmount)
{

    conn.Open();
    using var txn = conn.BeginTransaction();
    try
    {

            "INSERT INTO Students (Name, SchoolId, IsActive) VALUES (@Name,@SchoolId,1); SELECT CAST(SCOPE_IDENTITY() AS INT);",

}
6
Spot the bugs — this code has 3 problems. Find them before revealing.Debugging
public DataTable Search(string name)
{
    SqlConnection conn = new SqlConnection(_connStr);
    conn.Open();
    string sql = "SELECT * FROM Students WHERE Name = '" + name + "'";
    SqlCommand cmd = new SqlCommand(sql, conn);
    SqlDataReader reader = cmd.ExecuteReader();
    DataTable dt = new DataTable();
    dt.Load(reader);
    return dt;
}

Bug 1: SQL injection — name is string-concatenated directly into the query. Fix: parameterize with @Name.
Bug 2: No using/Dispose() on conn, cmd, or reader — connection is never closed, leaks the pool. Fix: wrap each in using.
Bug 3: No try/catch around conn.Open() or execution — if it throws, the connection is left open forever (compounds Bug 2). Fix: using blocks handle this automatically even on exception, so fixing Bug 2 also fixes Bug 3.

Bulk-insert 5000 attendance records efficiently instead of looping row-by-row.Performance
public void BulkInsertAttendance(DataTable attendanceRows)
{
    using var conn = new SqlConnection(_connStr);
    conn.Open();
    using var bulkCopy = new SqlBulkCopy(conn) { DestinationTableName = "Attendance" };

➕ More Practice Problems

8
Check if a student's email already exists before inserting (avoid duplicates).Validation
public bool EmailExists(string email)
{
    const string sql = "SELECT COUNT(*) FROM Students WHERE Email = @Email";
    using var conn = new SqlConnection(_connStr);
    using var cmd = new SqlCommand(sql, conn);
    cmd.Parameters.AddWithValue("@Email", email);
    conn.Open();
    return (int)cmd.ExecuteScalar() > 0;
}

public int InsertStudent(string name, string email, int schoolId)
{
    if (EmailExists(email))
        throw new InvalidOperationException("A student with this email already exists.");

    const string sql = @"INSERT INTO Students (Name, Email, SchoolId)
                          VALUES (@Name, @Email, @SchoolId);
                          SELECT CAST(SCOPE_IDENTITY() AS INT);";
    using var conn = new SqlConnection(_connStr);
    using var cmd = new SqlCommand(sql, conn);
    cmd.Parameters.AddWithValue("@Name", name);
    cmd.Parameters.AddWithValue("@Email", email);
    cmd.Parameters.AddWithValue("@SchoolId", schoolId);
    conn.Open();
    return (int)cmd.ExecuteScalar();
}

Note: this two-step check-then-insert has a small race-condition window under heavy concurrency — the bulletproof fix is a UNIQUE constraint on the Email column in SQL Server, and catching the resulting SqlException (error 2601/2627) instead of relying only on the pre-check.

9
Implement paging — return page 3 of students, 20 per page, without loading the whole table.Performance
public List<Student> GetPage(int pageNumber, int pageSize)
{
    var list = new List<Student>();
    const string sql = @"SELECT Id, Name, ClassId FROM Students
                          ORDER BY Name
                          OFFSET @Skip ROWS FETCH NEXT @Take ROWS ONLY";

    using var conn = new SqlConnection(_connStr);
    using var cmd = new SqlCommand(sql, conn);
    cmd.Parameters.AddWithValue("@Skip", (pageNumber - 1) * pageSize);
    cmd.Parameters.AddWithValue("@Take", pageSize);
    conn.Open();
    using var reader = cmd.ExecuteReader();
    while (reader.Read())
        list.Add(new Student { Id = reader.GetInt32(0), Name = reader.GetString(1), ClassId = reader.GetInt32(2) });
    return list;
}

OFFSET ... FETCH NEXT requires an ORDER BY clause — SQL Server needs a defined order before it can skip/take rows deterministically.

11
Distinguish a SQL-specific error (e.g., duplicate key) from a generic exception when inserting.Error Handling
try
{
    InsertStudent(name, email, schoolId);
}
catch (SqlException ex) when (ex.Number == 2627 || ex.Number == 2601)
{
    // 2627 = PK/unique constraint violation, 2601 = duplicate key on unique index
    MessageBox.Show("A student with this email already exists.", "Duplicate", MessageBoxButtons.OK, MessageBoxIcon.Warning);
}
catch (SqlException ex)
{
    // any other SQL Server error (timeout, connection, permissions, etc.)
    MessageBox.Show("Database error: " + ex.Message, "Error", MessageBoxButtons.OK, MessageBoxIcon.Error);
}
catch (Exception ex)
{
    MessageBox.Show("Unexpected error: " + ex.Message, "Error", MessageBoxButtons.OK, MessageBoxIcon.Error);
}

Catching SqlException specifically (before the generic Exception catch) lets you give the user a precise, actionable message instead of a raw stack trace — ex.Number is SQL Server's error code.

12
Write an UPSERT — insert the student if new, update if the email already exists — in one round-trip using MERGE.Advanced
const string sql = @"
    MERGE Students AS target
    USING (SELECT @Email AS Email) AS source
    ON target.Email = source.Email
    WHEN MATCHED THEN
        UPDATE SET Name = @Name, ClassId = @ClassId
    WHEN NOT MATCHED THEN
        INSERT (Name, Email, ClassId) VALUES (@Name, @Email, @ClassId);";

using var conn = new SqlConnection(_connStr);
using var cmd = new SqlCommand(sql, conn);
cmd.Parameters.AddWithValue("@Name", name);
cmd.Parameters.AddWithValue("@Email", email);
cmd.Parameters.AddWithValue("@ClassId", classId);
conn.Open();
cmd.ExecuteNonQuery();

One round-trip instead of "SELECT to check, then INSERT or UPDATE" (two round-trips plus a race-condition window). Interview point: mention MERGE has had subtle concurrency edge cases in some SQL Server versions under very high concurrency — for simple single-row upserts like this it's fine and commonly used.

13
Add simple retry logic for a transient connection failure (e.g., brief network blip to 192.168.1.52).Resilience
public List<Student> GetAllWithRetry(int maxAttempts = 3)
{
    for (int attempt = 1; attempt <= maxAttempts; attempt++)
    {
        try
        {
            return GetActiveStudents(1);
        }
        catch (SqlException) when (attempt < maxAttempts)
        {
            System.Threading.Thread.Sleep(500 * attempt);   // simple backoff: 500ms, 1000ms, ...
        }
    }
    throw new Exception("Could not reach the database after " + maxAttempts + " attempts.");
}

Say out loud: transient faults (brief network blips, DB failover) are common enough in real systems that a small retry-with-backoff is a reasonable, low-risk safety net — but it should NOT retry non-transient errors like a permission failure or a bad query, which is why this only retries SqlException, not every exception type.

🎯 Interview Scenarios

1
What's the difference between DataReader and DataSet — when would you use each?Core

DataReader: connected, forward-only, read-only, streams row-by-row — fastest, lowest memory, but connection must stay open. Use for a quick read-and-map-to-list operation, like an API endpoint. DataSet: disconnected, holds full data in memory, editable, connection closes after Fill() — use when you need to edit data offline, bind to a WinForms grid, or work with multiple related tables together (DataSet can hold multiple DataTables with relations).

2
How does connection pooling work, and what happens if you forget to close a connection?Performance

ADO.NET maintains a pool of open physical connections per unique connection string. Calling Close()/Dispose() doesn't destroy the connection — it returns it to the pool for reuse, avoiding the cost of a new TCP handshake + login each time. If you forget to close/dispose, the connection is never returned to the pool — under load, the pool exhausts (hits Max Pool Size, default 100) and new requests throw "Timeout expired waiting for a connection from the pool." Always use using to guarantee disposal even on exceptions.

3
How do you prevent SQL injection in ADO.NET?Security

Always use parameterized queries — @ParamName placeholders with cmd.Parameters.AddWithValue() or cmd.Parameters.Add(new SqlParameter(...)). The database driver sends the SQL text and parameter values separately over the wire — user input is never interpreted as part of the SQL syntax, so it can't break out of the query. Never build SQL with string concatenation or interpolation of raw user input.

4
What isolation levels are available in ADO.NET transactions, and what's the default?Transactions

Common levels: Read Uncommitted (dirty reads allowed, fastest, least safe), Read Committed (default in SQL Server — can't read uncommitted data, but non-repeatable reads possible), Repeatable Read (locks read rows until transaction ends), Serializable (strictest, fully isolated, most locking/blocking), Snapshot (row-versioning based, no blocking). Set via conn.BeginTransaction(IsolationLevel.ReadCommitted). Most business apps stay on the default Read Committed unless there's a specific concurrency problem to solve.

5
Why is the "using" statement so important for SqlConnection, SqlCommand, and SqlDataReader?Best Practice

All three implement IDisposable — using guarantees Dispose() runs even if an exception is thrown mid-method, releasing the connection back to the pool and freeing unmanaged resources (network handles). Without it, a single unhandled exception between Open() and Close() permanently leaks that connection for the app's lifetime, silently degrading the app until the pool exhausts.

6
Command Timeout vs Connection Timeout — what's the difference?Config

Connection Timeout (set in the connection string, e.g. Connection Timeout=30) is how long to wait while trying to establish the initial connection to SQL Server before giving up. Command Timeout (cmd.CommandTimeout = 30, in seconds, default 30) is how long to wait for a specific query/command to finish executing once connected — a long-running report query might need this raised, independently of the connection timeout.
