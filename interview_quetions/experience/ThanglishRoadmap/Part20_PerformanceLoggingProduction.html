20A Performance & Query Tuning
1. In-memory Cache vs Distributed Cache
IMemoryCache use pannina, data process oda memory kulla thaan store aagum — semaya fast, aana adhu andha oru instance ku mattum theriyum, restart aana poidum. Redis madhiri distributed cache na, app velila irukkum, so multiple instances ellame same cache ah share pannalam, restart aanalum survive aagum. Single instance service ku, or fee structure/dropdown madhiri rarely-change reference data ku IMemoryCache pothum. Multiple instance la scale pannumbodhu ella instance-um sync ah irukanum na (session data), Redis thaan sariya varum. Zen Campus la academic year config, fee category lookup, class-section mapping madhiri static-ah irukka data ku IMemoryCache aggressive ah use pannirukom — adhu thaan 40% performance improvement la periya part.
🗣️ Interview la eppadi sollanumI use IMemoryCache for data that's read constantly but rarely changes and doesn't need to be identical across instances — things like fee category lookups, class-section mappings, academic year config. It lives right in the process, so it's basically zero latency. Once you scale out to multiple instances, in-memory caches drift out of sync with each other, so anything that needs to be consistent across instances — like session data — needs a distributed cache like Redis instead. On Zen Campus we haven't needed Redis everywhere yet, but caching those static lookups was a big chunk of the 40% performance improvement I got on that module.
🔁 Follow-up Qs they might fire next
Q: Redis eppo compulsory aagum? — Multiple instance la ஒரே cache state venum na, or restart aana survive aaganum na, Redis thaan option.
Q: In-memory cache stale data problem eppadi solve pannuvinga? — Time-based expiry or explicit invalidation on write — adhu thaan next topic.
Q: Session data ku Redis use pannama vachaal enna aagum? — Load balancer vera instance ku route pannina, andha instance ku andha user session theriyathu — login state poidum.
2. Cache Invalidation
Cache-la periya problem data cache pannurathu illa — eppo andha data stale aaguthu nu correct ah kandu pidichu, ella place-lum same time la update/clear pannurathu thaan kashtam. Over-aggressive ah invalidate pannina cache-oda whole point poidum; lazy ah invalidate pannina wrong data user ku kaatuvom, adhuvum exception illama silent ah. Money/attendance madhiri critical data ku idhu romba careful ah handle pannanum, holiday list madhiri data konjam stale aanalum problem illa. Zen Campus la fee category data cache pannitirunthen, admin oru fee amount update pannuraappo, create flow ku mattum invalidation trigger vachirunthen, update flow ku illa — so 10 nimisham receipt la old amount kaatichi. Fix — time-based expiry mela depend aagama, update service method kulla explicit ah andha cache key ah clear pannuren, write nadakkum moment andha next read guaranteed fresh ah varum.
🗣️ Interview la eppadi sollanumCache invalidation is hard because you have to know exactly when data goes stale and clear every copy at the same moment — too aggressive and you lose the benefit of caching, too lazy and users silently see wrong data with no exception to point at. I care about this most for anything tied to money or attendance. I actually hit this on Zen Campus — I'd cached fee category data but only wired invalidation on the create path, not the update path, so a fee change didn't show up for about ten minutes. I fixed it by explicitly clearing that cache key right inside the update service method instead of relying on time-based expiry, so a write immediately guarantees a fresh next read.
🔁 Follow-up Qs they might fire next
Q: Time-based expiry use pannama explicit invalidation eppadi better? — Write nadakkum exact moment cache clear aagum, expiry window varaikkum wait pannum doubt vera irundha use aagathu.
Q: Idhu eppadi kandu pudichinga bug ah? — Silent ah irunthu, accounts/admin oru number wrong ah kaanumbodhu thaan report pannanaanga, exception onnum throw aagathu.
Q: Distributed environment la cache invalidation innum kashtama-a? — Aama, multiple instance ella pearum clear pannanum, adhukku pub/sub madhiri mechanism (Redis invalidation channel) venum.
3. Diagnosing a Memory Leak
Memory leak na, app memory usage hours/days ku steady ah rise aagi, quiet time la kooda thirumba varathu — final ah app pool recycle aaguvaraikkum poidum. Idhu diagnose panna, first memory trend ah watch pannanum, appuram edhu service nu narrow pannitu, memory dump eduthu heap la enna reference hold pannitirukku nu paakanum. Common reason — event handler unsubscribe pannama irukkathu, ever-growing static collection, eviction illatha cache. Zen Campus la bulk PDF export back-to-back run pannumbodhu memory creep aana madhiri thondhichuchu — real leak illa, periya byte array gen2 varaikkum live ah irunthathu adhanaal slow ah collect aachu. Memory stream ah using block end la depend aagama, file write aana udane explicit ah dispose pannitten, appuram normal sawtooth pattern ku thirumba varuthu.
🗣️ Interview la eppadi sollanumA real memory leak shows up as memory climbing steadily and never dropping back down, even during quiet periods, until the app pool eventually recycles or crashes — you diagnose it with a memory dump to see what's actually holding references it shouldn't. I haven't had to pull a full dump on Zen Campus, but I chased something that looked like a leak on our report-generation flow, where memory crept up during back-to-back bulk PDF exports. It turned out the PDF byte arrays were just living long enough to reach gen2 before getting collected. I fixed it by explicitly disposing the memory streams right after the file write instead of trusting the using block at the end of a longer method, and memory went back to a normal sawtooth pattern.
🔁 Follow-up Qs they might fire next
Q: Real leak-um, ippo pesina scenario-um difference enna? — Real leak la memory GC aanalum varathu; idhu la periya object gen2 varaikkum poi late ah collect aana adhu thaan slow ah kaanichuchu.
Q: Memory dump eppadi eduthu analyze pannurom? — Production process oda snapshot eduthu, tool vachu heap la enna type-oda object athigam irukku nu paaklam.
Q: Static collection leak eppadi create aagum? — Item add pannitte poi, remove panna marandhuduchu, andha collection endlessly grow aagum, GC-kooda touch panna mudiyave mudiyathu.
4. Diagnosing High CPU in Production
High CPU na, edhavathu code repeatedly athiga velai pannurathu — tight loop, memory kulla inefficient LINQ periya collection mela run pannurathu, alathu background job expected-ku athiga frequent ah fire aagurathu. First edhu process/endpoint CPU spike-oda correlate aaguthu nu check pannitu, andha specific code path ah profile pannanum. Zen Campus la attendance report whole school ku generate pannumbodhu app pool CPU spike aagurathu — first guess SQL query, aana SSMS la andha query fine ah run aachu. Turned out raw attendance rows ellame memory ku eduthu, C# nested loop la grouping/aggregation pannikittirunthom — couple thousand students ku absurd amount of iterations. Aggregation ah stored procedure kulla push pannitten, CPU spike basically poiruthu.
🗣️ Interview la eppadi sollanumHigh CPU usually means something's doing repeated work it shouldn't — a tight loop, or LINQ aggregation running in memory instead of being pushed down to SQL. I measure before I guess, because CPU spikes get blamed on the database way too often. On Zen Campus, an attendance report was spiking app pool CPU for a whole school's worth of students. The SQL query itself ran fine in SSMS — the real issue was we were pulling all the raw rows into memory and doing grouping and aggregation in C# with nested loops. I pushed that aggregation into the stored procedure instead, and the CPU spike basically disappeared.
🔁 Follow-up Qs they might fire next
Q: SQL query thaan first suspect nu solringale, apparam eppadi rule out pannitinga? — SSMS la standalone ah query run panni paathen — fast ah odina apparam application code side thaan problem nu confirm aachu.
Q: C# side aggregation vs SQL side aggregation — eppo edhu better? — Periya dataset ku SQL server itself group/aggregate panna better, network round trip + memory both save aagum.
Q: Idhu production la eppadi kandu pudichinga? — App pool CPU spike monitoring alert + andha specific endpoint call panra time correlate panninen.
5. Slow API Endpoint Only Under Load
Load-dependent slowness oda tricky part — namba manual ah test pannumbodhu fine ah irukkum, so production timing data thaan paakanum — request duration logs, DB call ah, external API ah, serialization ah, illna request queue la wait pannurathaa nu breakdown venum. Low traffic la kooda slow na, athu straight inefficiency; load la mattum slow na, contention — connection pool limit, lock contention, thread pool starvation — vera fix venum. Zen Campus attendance-marking endpoint la 8:45-9:15 AM la ella class teacher-um login aagi attendance mark pannumbodhu, response time 1 second-la irundhu 8-10 seconds ku poiruthu. DB call chutta timing log podren, connection pool configure panna vida athiga connections open aagi irunthadu theriyuthu — request queue la wait pannuthu. Pool size increase pannitten, appuram connection ah properly close panniraathan thaan sertu fix aachu.
🗣️ Interview la eppadi sollanumIf it's slow at low traffic too, that's a straightforward inefficiency, but if it's only slow under load, that points at contention — connection pool limits, lock contention, or thread pool starvation — a completely different fix. I look for that time-of-day clustering, like "fine in the morning, terrible at 9 AM." We had exactly that on the attendance-marking endpoint on Zen Campus, between 8:45 and 9:15 when every class teacher logs in at once — response times went from under a second to eight or ten seconds. Timing logs around the DB call showed we were opening more connections than the pool allowed, so requests were queuing for a free connection. I bumped the pool size and made sure connections were being closed promptly instead of held open across the whole request.
🔁 Follow-up Qs they might fire next
Q: Timing logs eppadi add pannirukeenga exact ah? — DB call start/end time ah log pannitu, duration calculate pannitten — adhu clear ah kaatiruthu connection wait time thaan periya chunk nu.
Q: Pool size increase pannrathe permanent fix-a? — Illa, root cause connection properly close pannama vachiruntha, adhuvum fix pannanum — size increase mattum symptom ah temporary hide pannum.
Q: Idhu eppadi reproduce pannirukeenga testing la? — Load test tool vachu concurrent requests simulate pannitten, staging la same pattern kaatichi.
6. Database Connection Pool Exhaustion
Connection pool exhaustion na, pool la irukka connections-ku vida athiga requests DB connection kettu wait pannurathu — konjam neram wait pannitu mudiyaatha bodhu "timeout expired" error varum. Usually reason — genuine ah connection athigam venum illai, edhavathu connection ah venda neram varaikkum hold pannitirukkum — DbConnection properly dispose pannama illatha, unrelated velai pannumbodhu-um transaction connection ah hold pannitirukkum. Zen Campus la raw ADO.NET section la connection method top la open pannitu, validation failure early return path la properly close aagathu — most time fine (GC eventually clean pannum), aana morning peak la minutes kulla pool exhaust aagi ella downstream-um timeout aagi poichu. Fix — every connection usage ah using block kulla properly wrap pannitten, rest of codebase-ellaam same pattern ku audit panninen.
🗣️ Interview la eppadi sollanumConnection pool exhaustion means more requests want a DB connection than the pool has available, so new requests queue up and eventually time out. The usual cause isn't needing more connections — it's something holding a connection open longer than it should. On Zen Campus, a raw ADO.NET method opened a connection at the top, and an early return on a validation-failure path meant it wasn't reliably closed. It worked fine most of the time because GC eventually cleaned it up, but under morning peak load it exhausted the pool within minutes and everything downstream started timing out. I wrapped every connection usage in proper using blocks and audited the rest of the codebase for the same pattern.
🔁 Follow-up Qs they might fire next
Q: Idhu eppadi identify pannirukeenga first ah? — "Timeout expired, the timeout period elapsed prior to obtaining a connection" error burst ah ella same time la varuvathu thaan clue.
Q: using block vachaal connection eppadi guarantee ah close aagum? — Dispose() automatic ah call aagum scope exit aana udane, exception vandhalum varaadhalum — finally madhiri guaranteed.
Q: Pool size increase pannurathu idhukku fix-a? — Symptom mattum mask aagum, real leak vachiruntha innum periya load la thirumba varum — root fix thaan sariya path.
7. Thread Pool Starvation & Async Deadlocks
Thread pool starvation na, available threads ellame edhavathu wait pannikkittirukku (usually .Result alathu .Wait() madhiri blocking calls async method mela) — so pudhu requests thread kettu queue la nikkum, CPU edhum busy ah illamalum. Deadlock na innum specific — synchronous ah wait panra thread, same context venum async call ku wait panren nu irundha, rendum oru periya other ah wait panni forever nikkum. Confuse aagra part — CPU graph calm ah irukkum, so "server fine" nu first thonum, thread count/queue length thaan andha real signal. Zen Campus la refactor pannumbodhu legacy code oru async Dapper call ku .Result call pannitirunthu, dev la 1-2 requests ku fine ah irunthadu. Real load la requests pile up aagi timeout aachu, CPU low ah thaan irunthadu adhu thaan confuse pannirichu — thread pool queue length check pannina udane clear aachu, proper await ku maathinen udane fix aachu.
🗣️ Interview la eppadi sollanumThread pool starvation happens when threads are stuck on blocking calls like .Result or .Wait() on async methods, so new requests can't get a thread even though the CPU looks idle. I hit a version of this on Zen Campus — some ported legacy code called .Result on an async Dapper call instead of awaiting it, which worked fine with one or two requests in dev. Under real load, requests piled up and timed out, but CPU stayed low the whole time, which threw me off initially since I was looking at the wrong metric. Once I checked thread pool queue length instead of CPU, it was obvious, and switching to a proper await fixed it immediately.
🔁 Follow-up Qs they might fire next
Q: CPU normal ah irukkumna enna metric first check pannanum? — Thread pool queue length / active thread count — CPU alathu, thread availability thaan idhu ah suspect pannanum.
Q: .Result use pannurathu eppo problematic aagum? — ASP.NET Core request context la, blocking call sync context ku wait panna thread deadlock aaga chance irukku, high concurrency-la especially.
Q: Fix pannita apparam eppadi verify pannirukeenga? — Load test thirumba run pannitten, thread queue length normal ah irunthadu, response time consistent ah kammiyaaguruthu.
8. N+1 Query Problem
N+1 problem na, oru list edukka oru query aduthu, appuram andha list mela loop pannitu ovvoru record ku separate ah related data fetch pannurathu — 100 records na 101 round trips aagum, oru joined/batched query ku pathila. Profile pannurathukku query logging on pannitu alathu SQL Profiler vachu, oru page load/API call ethanai query fire pannurom nu paaklam. Zen Campus la student roster screen la — ovvoru student ku attendance status fetch panna separate Dapper call pannitirunthom, 40 students na 41 queries — small section ku fine, periya class ku laggy. WHERE IN vachu ella student ID-kkum ore query la attendance fetch pannitu, memory la map pannna maathinen — adhu thaan resume la irukka 40% query-time improvement oda periya part.
🗣️ Interview la eppadi sollanumAn N+1 problem is running one query for a list, then firing a separate query per record for related data instead of one batched query — so a hundred records becomes a hundred-and-one round trips. This hit us on the student roster screen on Zen Campus — for each student, we were making a separate call to fetch their latest attendance status, so a class of forty kids meant forty-one queries. I rewrote it into a single query using a WHERE IN on the student IDs and mapped the results back in memory. That's part of what fed into the 40% query-time improvement on my resume — once I started looking, N+1 patterns were hiding in a few other list-heavy screens too.
🔁 Follow-up Qs they might fire next
Q: Idhu eppadi kandu pudichinga first ah? — Periya class open panna page noticeably laggy ah irunthadu, list size ku proportional-ku vida athigama slow ah — adhu thaan clue.
Q: WHERE IN approach scale aagumaa periya list ku? — Reasonable batch size varaikkum fine, romba periya na chunk pannitu multiple batched queries pannalam.
Q: Dapper la idhu eppadi query panreenga exactly? — Student IDs oda list ah parameter ah pass pannitu, single SQL query la WHERE StudentId IN @Ids madhiri fetch pannitu, Dictionary ah map pannitten.
9. Capacity Planning for 1,000+ Concurrent Users
Capacity planning panna, flat number vachi start pannama, actual usage pattern-la irundhu start pannanum — school ERP ku concurrency whole day evenly spread aagathu, morning attendance alathu fee due date madhiri specific windows la cluster aagum. So average traffic ku vida, andha peak window ku size pannanum, DB connections/thread pool/cache edhu actual bottleneck nu kandu pidiyanum. Zen Campus la Ocelot gateway vachu microservices architecture design pannnom — attendance, billing madhiri services ah independent ah scale panna mudiyum madhiri, rendum completely vera load pattern kaatrathanaal. 1000+ concurrent user figure periya client schools ah look pannitu, ella teacher/konjam parents ore 15 nimisham window la active ah irundha enna aagum nu estimate pannitu vandhathu, appuram connection pool sizing + caching andha peak ah handle panna verify panninom, quiet afternoon test mattum pannama.
🗣️ Interview la eppadi sollanumI'd size for peak concurrent load in the windows that actually cluster hard — like morning attendance or fee due dates — not average traffic across the day, because that's where you'll actually get blindsided. On Zen Campus, we designed the microservices architecture with Ocelot as the gateway partly for this reason, so we could scale attendance or billing independently since they see very different load patterns. The 1,000+ concurrent users number came from looking at our bigger client schools and estimating what happens when every teacher and a chunk of parents are active in the same fifteen-minute window, then making sure connection pool sizing and caching could hold up under that peak, not just a quiet afternoon test.

Q: Ocelot gateway capacity planning ku eppadi help pannuthu? — Ovvoru service ah independent ah scale panna vidurathanaal, attendance peak time la billing ah touch pannama andha service mattum scale pannalam.
Q: 1000+ number eppadi arrive pannirukeenga? — Bigger client school teacher + parent count vachu, ella pearum same 15 nimisham window la active nu worst-case estimate pannitom.
20B Logging, Exception Handling & Observability
10. Centralized Exception-Handling Middleware
Idhu oru middleware — pipeline early la sit pannitu, downstream ella controller-um try-catch kulla wrap pannum madhiri — so ovvoru controller separate ah error handle pannama, exception ella ore place ku bubble up aagum. Andha place la stack trace, request path, correlation ID, user info full context ah log pannitu, consistent JSON error shape ku map pannitu client ku anuppurom — client ku raw .NET exception alathu body illama 500 varathu. Idhukku munnadi, ovvoru team member vera vera way la try-catch pannaanga — konjam per log pannaanga, konjam stack trace straight ah response la leak pannaanga (security problem kooda). Zen Campus la idhu build pannirukken thaan naan proud ah irukka item — 35% incident resolution time cut idhu oda directly related, exception vandhalum correlation ID + clean stack trace already right service ku tag aagi varum, logs search pannna mattum pothum ippo.
🗣️ Interview la eppadi sollanumThis is one piece of middleware sitting early in the pipeline that wraps everything downstream, so instead of every controller having its own scattered error handling, exceptions bubble up to one place. It logs the exception with full context — stack trace, request path, correlation ID — and maps it to a consistent JSON error shape before it goes back to the client, so nobody ever sees a raw exception or a bare 500. Before this on Zen Campus, every team member handled errors differently, some leaking stack traces straight into responses. After I built this middleware, every error came in tagged to the right service with a correlation ID, and it's directly tied to that 35% cut in incident resolution time.
🔁 Follow-up Qs they might fire next
Q: Idhu security angle-la eppadi help pannuthu? — Raw exception message/stack trace client ku pogathu, so internal implementation details leak aagathu.
Q: Ella exception-um same way ah handle pannuvingala? — Illa, custom business exceptions (like OtpExpiredException) ku specific status code/message map pannurom, unknown exception ku generic safe message.
Q: Idhu eppadi test pannirukeenga? — Deliberate ah different exception types throw pannitu, response shape consistent ah varuthaa nu verify pannnen.
11. Structured Logging
Structured logging na, plain sentence ku pathila key-value fields ah log pannurathu — "user 4521 logged in from 10.0.0.5" nu string pathila, UserId=4521, Ip=10.0.0.5, Module=Auth nu separate properties ah log pannurom. Idhu query, filter, aggregate panna easy ah varum, grep pannama. Thousands of lines per day, multiple services na, plain text la "show me every failed login for this student ID last hour" nu kekka ugly regex venum — structured log field mela mattum filter pannalam. Zen Campus la exception middleware + service logging ah structured format ku maathinen — CorrelationId, ServiceName, Endpoint, UserRole separate fields ah — adhu thaan issue ah fast ah trace panna mudinjadhu, ServiceName=BillingService and CorrelationId=X nu filter pannitu, request full story seconds la kidaikkum.
🗣️ Interview la eppadi sollanumStructured logging means logging key-value fields instead of a plain sentence, so I can filter and aggregate on individual properties later instead of grepping through text. On Zen Campus I moved the exception middleware and service-level logging to a structured format — CorrelationId, ServiceName, Endpoint, UserRole as separate fields. That's genuinely what made tracing fast — I could filter by ServiceName equals BillingService and CorrelationId equals whatever the user reported, and get the whole request's story in seconds instead of scrolling through plain text.
🔁 Follow-up Qs they might fire next
Q: Plain text logging vachiruntha enna problem varum? — Regex vachu search pannanum, query/filter/aggregate panna kashtam, especially multiple services la thousands of lines irundha.
Q: Ella field-um log pannalama, PII irundhu? — Illa, phone number/payment details madhiri sensitive field mask pannanum alathu log pannama vidanum — adhu vera topic.
Q: Structured log store panna edhu approach use pannirukeenga? — Log ah key-value format la store pannitu, ServiceName/CorrelationId madhiri fields mela filter pannurom.
12. Correlation IDs Across Microservices
Correlation ID na, oru unique value (usually GUID) — request entry point la (API Gateway) generate pannitu, ovvoru downstream call kooda header/context la pass pannurom. Andha request ah touch panra ovvoru service, same ID vachu log pannum. Illna, request gateway → student service → billing service poi billing la fail aana, three vera log source la timestamp compare panni guess pannanum. ID irundha, ore search la whole chain order ah kidaikkum. Zen Campus la Ocelot gateway entry la correlation ID stamp pannitu, headers vazhi downstream services ku flow aagum madhiri set pannirukken. Oru parent fee payment "disappeared" nu complain pannaar, support ticket la correlation ID vachu paathen — payment service success aachu, aana notification service confirm panna munnadi timeout aachu, adhanaal UI la broken madhiri kaatichi, actual ah money poichu.
🗣️ Interview la eppadi sollanumA correlation ID is a unique GUID generated at the entry point, like the API Gateway, and passed through every downstream call so every service that touches a request logs with the same ID. Without it, you're stuck comparing timestamps across separate log sources. On Zen Campus, Ocelot stamps a correlation ID on the way in, and it flows through headers into whichever service handles it. One case — a parent said a fee payment "disappeared" — I pulled up the correlation ID from the support ticket and could see the payment service actually succeeded, but the notification service timed out before confirming back, so it looked broken on the UI even though the money went through fine.

14. Graceful Degradation

🗣️ Interview la eppadi sollanumGraceful degradation means when part of the system fails, the rest keeps working, maybe with reduced functionality, instead of everything going down together. 15. Rate Limiting at the Gateway

🗣️ Interview la eppadi sollanumWe used Ocelot's built-in rate limiting at the gateway level, configuring requests-per-second thresholds per route, so a client hitting an endpoint too aggressively gets throttled with a 429 instead of slamming straight into the backend. 25. A Cascading Failure
Cascading failure na, oru service slow/down aagraappodhu, andha service ah call panra caller-um back up aagi fail aagrathu, appuram andha caller-oda caller-um fail aagrathu — oru corner problem system-full spread aagrathu. Zen Campus la notification service (SMS/email confirmation) third-party gateway hiccup vachi slow ah respond panna start pannichu, billing service payment process aana apparam synchronous ah idha call pannuchu, aggressive timeout illama. Billing requests pile up aagi wait panna start pannichu andha core-alathu service ku, billing itself down madhiri kaatichu, aana billing logic fine ah irunthichu. Fix — andha call ku strict timeout add pannnen, notification step ah fire-and-forget-ish ah maathinen — log pannitu separate ah retry pannrathu, payment response ah adhu mela block pannama — graceful degradation idea-oda thaan connect aagrathu.
🗣️ Interview la eppadi sollanumA cascading failure is when one service's slowness causes its callers to also back up and fail, spreading a corner problem system-wide — usually because there's no timeout on a synchronous call to the struggling dependency. On Zen Campus, the notification service started responding slowly because of a third-party gateway hiccup, and billing called it synchronously after processing a payment without an aggressive timeout. Billing requests piled up waiting on a service that wasn't even core, and it looked like billing itself was down even though billing's logic was fine. The fix was adding a strict timeout on that call and making the notification step fire-and-forget — log it and retry separately — instead of blocking the payment response on it.
