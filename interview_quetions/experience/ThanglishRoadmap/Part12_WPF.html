⚡ Part 12 Quick-Recall Cheat Sheet

⚠️ Remember: WPF is conceptual knowledge for you, not a shipped project — never claim otherwise, lean on your real WinForms experience instead.
  • 1. WPF vs WinForms: XAML-based declarative UI, DirectX retained-mode rendering, vs WinForms' code-heavy GDI+ immediate-mode drawing.
  • 2. XAML: XML-based markup that maps 1:1 to C# objects — <Button/> tag = new Button() under the hood.
  • 3. Layout panels: Grid (rows/cols), StackPanel (linear stack), DockPanel (edge-docking), WrapPanel (auto-wrap flow).
  • 4. Data Binding: {Binding Path} connects UI to a property; INotifyPropertyChanged + ObservableCollection keep UI in sync automatically.
  • 5. Dependency Properties: special properties with built-in change notification, binding, styling, animation support — plain CLR properties can't do this.
  • 6. Routed Events: bubbling (child→parent, most common) vs tunneling (parent→child, "Preview" events).
  • 7. Styles & Templates: Style = set properties in bulk; ControlTemplate = redefine a control's whole look; DataTemplate = define how data renders.
  • 8. Resources / ResourceDictionary: centralized, reusable styles/brushes/templates, mergeable across the app for consistent theming.
  • 9. MVVM: Model (data) / View (XAML) / ViewModel (state + commands) — View never touches code-behind logic, binds to ViewModel instead.
  • 10. Commands (ICommand): binds a button's "action" to a ViewModel method instead of a code-behind Click handler — RelayCommand/DelegateCommand is the common helper.
  • 11. IValueConverter: transforms bound data for display (bool→Visibility, DateTime→string) without polluting the ViewModel with UI concerns.
  • 12. Dispatcher: WPF's UI-thread marshaller — same "only UI thread touches controls" rule as WinForms' InvokeRequired/Invoke, solved via Dispatcher.Invoke or just await back on the UI thread.
  • 13. WPF vs WinForms choice: WPF for rich/animated/skinnable UI + MVVM testability; WinForms for fast, simple, low-overhead internal tools (like his real locker app).
  • 14. MVVM toolkits: CommunityToolkit.Mvvm (Microsoft, source generators, modern), Prism (modular, navigation, DI, enterprise apps) — awareness level.
  • 15. "Do you have WPF experience?": honest script — real WinForms hands-on + strong WPF/MVVM conceptual prep, no fabricated project.
12A WPF Fundamentals & XAML
1. WPF vs WinForms (architecture difference)
WinForms la namba UI ah mostly code la thaan build pannுவோம் — Button b = new Button(); b.Location = ... nu C# code la set pannுவோம். WPF la adhu maari illa — UI ah XAML nu oru separate markup file la declarative ah eழுதுவோம், design ah design pannறவங்க separate ah pannலாம், developer logic ah separate ah pannலாம். Rendering side um difference — WinForms GDI+ vechu "immediate mode" la draw pannும் (ovvoru frame ah manually redraw), WPF DirectX vechu "retained mode" la — oru scene graph vachu automatic ah re-render pannும், so animations, transforms, scaling ellam smooth ah varும். Simple ah sonna — WinForms Notepad madhiri basic, WPF oru modern skinnable app madhiri.
🗣️ Interview la eppadi sollanumWPF separates UI definition into declarative XAML markup, while WinForms builds UI mostly through imperative C# code. WPF uses DirectX-based retained-mode rendering with a vector graphics engine, so it scales, animates, and skins far more easily than WinForms' GDI+ immediate-mode drawing. That architectural split — markup for structure, code-behind or ViewModel for logic — is also what enables WPF's strong MVVM story.
🎯 Idha eppadi honest ah frame pannanumI'd say I've worked hands-on with WinForms in production — a locker management console — where I already pushed myself to keep business logic out of the Form class for testability. I've studied WPF's XAML-first, retained-mode architecture in depth conceptually, and I understand exactly why it's a step up for rich, data-driven UI. I haven't shipped a WPF app yet, but the transition wouldn't be starting from zero.
🔁 Follow-up Qs they might fire next
Q: Why does WPF use DirectX instead of GDI+? — For hardware-accelerated, resolution-independent vector rendering, so UI scales cleanly and animations stay smooth.
Q: Can you mix WinForms and WPF controls? — Yes, via ElementHost (WPF-in-WinForms) or WindowsFormsHost (WinForms-in-WPF) interop.
Q: Is WPF still relevant in 2026? — Yes for enterprise desktop/LOB apps needing rich UI on Windows; new cross-platform work often leans MAUI, but WPF has huge existing footprint.
2. XAML Basics (markup to objects)
XAML na "eXtensible Application Markup Language" — namma ஒரு XML madhiri file la UI ah describe pannுவோம். Idhu just "pretty markup" illa — ovvoru XAML tag um actual ah oru C# object ku direct ah map aagும். <Button Content="Save"/> nu eழுதினா, andha behind-the-scenes compiler adha Button b = new Button(); b.Content = "Save"; nu convert pannும் (via a generated .g.cs file + BAML). So XAML na "declarative object graph" nu nenachukalam — design ah easy ah edit panna mudiyும், data binding define panna easy ah irukும்.
🗣️ Interview la eppadi sollanumXAML is a declarative XML-based markup language where every element maps directly to a .NET class, and every attribute maps to a property or event on that class. At compile time it's parsed into BAML and combined with a generated partial class, so the tags you write in the designer are literally object instantiation and property-setting under the hood — just expressed declaratively instead of imperatively.
🎯 Idha eppadi honest ah frame pannanumI understand XAML conceptually well — that it's not "just markup" but compiles down to real object construction, and that's exactly why data binding and templating work so cleanly in it. I haven't built a production XAML tree at scale, but reading and reasoning about a XAML file wouldn't be unfamiliar territory for me — I'd be productive quickly given my C# and UI-event background from WinForms.
🔁 Follow-up Qs they might fire next
Q: What is BAML? — Binary Application Markup Language — the compiled, tokenized form of XAML embedded as a resource in the assembly, parsed faster at runtime than raw XML.
Q: Can you write WPF UI entirely in C# without XAML? — Yes, technically, by constructing the object tree in code — but you lose the designer/markup separation that's WPF's main productivity win.
Q: What connects XAML to code-behind? — The x:Class attribute on the root element ties the XAML-generated partial class to your code-behind .cs file.
3. Layout Containers (Grid, StackPanel, DockPanel, WrapPanel)
WPF la layout ah panel controls vechu manage pannுவோம் — WinForms la neenga pixel-by-pixel Location/Size set panra madhiri illa, idhu "flow-based" layout. Grid — rows/columns define panni, table madhiri arrange pannுவது, most flexible. StackPanel — items ah oru line la (horizontal illa vertical) stack pannுவது, simple forms ku nalla irukும். DockPanel — children ah edges (Top/Bottom/Left/Right/Fill) la dock pannுவது, classic "toolbar top, statusbar bottom, content fill" layout ku perfect. WrapPanel — items ah oru line la potu, space mudinja next line ku automatic ah wrap aagும் — gallery/tag-cloud madhiri UI ku use pannுவோம்.
🗣️ Interview la eppadi sollanumGrid is the most flexible — row and column definitions for table-like layouts. StackPanel arranges children in a single line, horizontal or vertical, good for simple forms. DockPanel docks children to an edge — top, bottom, left, right — with the last child typically filling remaining space, which is a common shell layout for toolbar-plus-content screens. WrapPanel flows children left to right and wraps to a new line when space runs out, useful for tag or thumbnail layouts.
🎯 Idha eppadi honest ah frame pannanumI haven't built production screens with these panels, but conceptually the mental shift from WinForms' absolute positioning to WPF's flow-based layout makes sense to me — it's actually closer to how I think about responsive web layout in ASP.NET Core views, just applied to desktop. Given a spec, I'd know which panel fits which layout problem without much ramp-up.
🔁 Follow-up Qs they might fire next
Q: How do you make a Grid responsive? — Use "*" star-sizing and "Auto" for rows/columns instead of fixed pixel widths, so space redistributes on resize.
Q: Can panels be nested? — Yes, freely — e.g. a DockPanel shell containing a Grid form inside its Fill area is very common.
Q: Difference between Grid and StackPanel performance-wise? — Grid does a two-pass measure/arrange and is heavier for very deeply nested layouts; StackPanel is lighter for simple linear content.

    <RowDefinition Height="Auto"/>
    <RowDefinition Height="*"/>
  </Grid.RowDefinitions>
  <TextBlock Grid.Row="0" Text="Student Name"/>

</Grid>
4. Data Binding (Binding, INotifyPropertyChanged, ObservableCollection)
WinForms la namba data ah UI la kaatanum na, manually textBox1.Text = student.Name; nu eழுதுவோம் — data change aana UI ah manually update pannanum. WPF la {Binding Path} vechu, control ah directly oru property kூட "link" pannிடலாம் — data change aana automatic ah UI update aagும் (and vice versa, two-way binding na UI change aana data um update aagும்). Idhu work aaganum na, andha class INotifyPropertyChanged interface implement pannanum — property set aagும்bodhu PropertyChanged event raise pannanum, adha WPF binding engine listen pannும். List data ku ObservableCollection<T> use pannுவோம் — item add/remove aanaவுடன் automatic ah UI (ListBox/DataGrid) refresh aagும், namba manually refresh pannத் தேவை இல்லை.
🗣️ Interview la eppadi sollanumData binding connects a UI element's property to a source property using {Binding Path}. For it to auto-update the UI, the source class implements INotifyPropertyChanged and raises PropertyChanged whenever a property setter changes the value. For collections, ObservableCollection<T> raises CollectionChanged automatically on add/remove, so a bound ListBox or DataGrid refreshes itself without any manual UI-update code.
🎯 Idha eppadi honest ah frame pannanumThis is one WPF concept I can bridge cleanly from real experience — in my WinForms locker app I manually wrote the "update UI when data changes" logic by calling control updates directly after every data change, which is exactly the boilerplate INotifyPropertyChanged and ObservableCollection eliminate. I haven't wired up live two-way bindings in a shipped app, but I understand precisely what problem they solve because I've solved it the manual way myself.
🔁 Follow-up Qs they might fire next
Q: Difference between OneWay, TwoWay, OneWayToSource binding modes? — OneWay: source→UI only; TwoWay: both directions; OneWayToSource: UI→source only.
Q: What if a class has 20 properties — do you write PropertyChanged manually for each? — Typically use a base class (ObservableObject / BindableBase) with a SetProperty helper, or source generators in CommunityToolkit.Mvvm.
Q: What happens if you bind to a plain List<T> instead of ObservableCollection? — Initial binding works, but add/remove after binding won't auto-refresh the UI since List doesn't raise change notifications.
<TextBox Text="{Binding StudentName, Mode=TwoWay, UpdateSourceTrigger=PropertyChanged}"/>

public class StudentVM : INotifyPropertyChanged {
    private string _name;
    public string StudentName {
        get => _name;
        set { _name = value; OnPropertyChanged(nameof(StudentName)); }
    }
    public event PropertyChangedEventHandler PropertyChanged;
    void OnPropertyChanged(string p) =>
        PropertyChanged?.Invoke(this, new PropertyChangedEventArgs(p));
}
5. Dependency Properties (vs plain CLR properties)
Normal C# property na, get/set kூட oru private field vachுruppோம் — adhu la binding, styling, animation, default value inheritance ellam free ah kidaikaathu, ovvondrukும் நாம் manual code eழுத vendும். Dependency Property na WPF-oda special property system — DependencyObject base class use panni register pannுவோம். Idhu vachா, automatic ah data binding support pannும், style/trigger la value set panna mudியும், animation apply panna mudியும், parent la irundhu value inherit pannும் (example: FontSize parent la set panninா child automatic ah get pannும்), and memory-efficient ah irukும் (default values ellam per-instance store aagாம், shared ah store aagும்). Athanaala thaan Button.Width, TextBox.Text lam ellam plain property illa, DependencyProperty ah declare pannirukanga.
🗣️ Interview la eppadi sollanumA Dependency Property is a property registered with WPF's property system via DependencyObject, giving it built-in support for data binding, styling, animation, value inheritance from the visual tree, and change notification — none of which a plain CLR auto-property gets for free. WPF needs this because its entire UI model — templating, triggers, animation, binding — relies on the framework being able to observe and override property values generically, which a plain field-backed property can't support.
🎯 Idha eppadi honest ah frame pannanumI understand dependency properties at the conceptual level — why WPF needed a richer property system than plain CLR properties to support binding, styling and animation uniformly across every control. I haven't authored a custom DependencyProperty in a shipped app, but I've built custom controls' worth of property/event plumbing in WinForms by hand, so I get the underlying problem it's solving, even without WPF-specific hands-on time.
🔁 Follow-up Qs they might fire next
Q: How do you declare a custom Dependency Property? — Use DependencyProperty.Register(name, type, ownerType, metadata) and expose a CLR wrapper property that calls GetValue/SetValue.
Q: What's a DependencyObject vs a regular object? — DependencyObject provides the GetValue/SetValue infrastructure and thread affinity that dependency properties need to function.
Q: Difference between Dependency Property and Attached Property? — An attached property is a dependency property usable on any element, not just the class that defines it — e.g. Grid.Row can be set on any child, not just Grid itself.
6. Routed Events (bubbling vs tunneling)
Normal C# event na, andha specific control mattum handle pannும். WPF la Routed Events na — event ah visual tree வழியா "route" pannலாம். Rendu types: Bubbling — event source (example oru Button click) la irundhu, parent parent ah mேலே mேலே போய் (Button → StackPanel → Window) event travel aagும் — idhu thaan most common, example Click event. Tunneling — opposite direction, root la irundhu source ku கீழே varும் — idhu "Preview" prefix vechu identify pannலாம், example PreviewMouseDown — root Window la இருந்து start aaகி, andha specific button varை varை pogும். Real use case: parent la ஒரே handler vechு, ella child button clicks ஐயும் ஒரே இடத்தில் catch pannலாம் — bubbling la thaan idhu easy ah possible.
🗣️ Interview la eppadi sollanumRouted events travel through the visual tree instead of being confined to a single element. Bubbling events, like Click, start at the source element and travel up through its ancestors — so I can attach one handler at a parent Panel or Window and catch clicks from any child button inside it. Tunneling events, prefixed with Preview, travel the opposite direction — from the root down to the source — which lets an ancestor intercept and potentially suppress an event before the target control ever sees it.
🎯 Idha eppadi honest ah frame pannanumI know routed events conceptually — the bubble-up-vs-tunnel-down model and why WPF needed it for scenarios like a parent panel centrally handling clicks from dynamically generated children. I haven't wired custom routed event handling in a real app, but the underlying idea — event propagation through a hierarchy — isn't new to me; it's conceptually similar to DOM event bubbling I'd have reasoned about on the web side too.
🔁 Follow-up Qs they might fire next
Q: How do you stop a routed event from bubbling further? — Set e.Handled = true in the event args inside the handler.
Q: Give an example of a tunneling event. — PreviewMouseDown, PreviewKeyDown — always paired with a bubbling counterpart (MouseDown, KeyDown).
Q: What's the third routed event strategy? — Direct events — behave like normal .NET events, no bubbling or tunneling, e.g. MouseEnter.
7. Styles & Templates (ControlTemplate, DataTemplate)
WinForms la oru button oda look ah change pannanum na, seriously limited — color konjam maathலாம், aana button oda fundamental shape/behavior ah maathradhu romba kashtam, custom drawing eழுதவேண்டியிருக்கும். WPF la athu illa — Style vechு, multiple controls ku common properties (color, font, padding) bulk ah apply pannலாம் — CSS class madhiri nenachுக்கலாம். ControlTemplate vechு, control oda entire visual structure ah — behavior maintain pannிக்கொண்டே — completely redesign pannலாம் (example: Button ah round shape ah, gradient background ah maathலாம், aana Click event still work pannும்). DataTemplate na — data ah எப்படி UI ல் render pannணும் nu define pannுவது (example: ListBox la ஒவ்வொரு Student object ஐயும் oru card layout ah காட்ட DataTemplate use pannலாம்).
🗣️ Interview la eppadi sollanumA Style bulk-applies property values across controls — similar to a CSS class. A ControlTemplate goes further and completely redefines a control's visual tree while preserving its behavior — so I can restyle a Button's entire look without losing its Click semantics. A DataTemplate defines how a piece of data should be visually rendered — for instance, how each Student object in a bound ListBox should look as a card instead of plain text. This is a huge jump from WinForms, where control appearance is largely fixed and true re-theming requires owner-draw code.
🎯 Idha eppadi honest ah frame pannanumI haven't built custom ControlTemplates or DataTemplates in a shipped app, and I'll say that plainly if asked. What I can speak to honestly is the contrast — in WinForms I hit real limits trying to customize control appearance, so I understand exactly why WPF's template system exists and what problem it's solving, even without hands-on WPF template authoring yet.
🔁 Follow-up Qs they might fire next
Q: Can a Style include triggers? — Yes, Style.Triggers can change property values based on conditions like IsMouseOver, IsPressed.
Q: Difference between ControlTemplate and DataTemplate? — ControlTemplate redefines a control's own look; DataTemplate defines how bound data (not a control) is rendered.
Q: Can Styles be inherited? — Yes, via BasedOn="{StaticResource BaseStyle}" for style inheritance.

8. Resources & Resource Dictionaries (sharing styles across app)
Style, Brush, DataTemplate ellam oru single window/page la mattum use pannினா okay, aana full app la 20-30 screens la consistent ah same theme வேணும் na, ovvொரு file la copy-paste pannுவது maintenance nightmare. ResourceDictionary na — idhu ella reusable resources ஐயும் ஒரே central file la define pannி, App.xaml la MergedDictionaries வழியா whole app kூட share pannலாம். Enterprise apps la "theme" nu solradhுkூட idhu thaan base — color palette, font sizes, common styles ஒரே இடத்தில் maintain pannினா, ஒரு color maathினா whole app maариும்.
🗣️ Interview la eppadi sollanumA ResourceDictionary centralizes reusable resources — brushes, styles, templates — in one XAML file, which you then merge into App.xaml's MergedDictionaries so every window in the application can reference them via StaticResource or DynamicResource. This is how WPF apps implement consistent theming — change a color or font size once in the shared dictionary, and it reflects across every screen that references it, instead of hunting through dozens of individual XAML files.
🎯 Idha eppadi honest ah frame pannanumI get the "single source of truth for shared styling" idea well from a different angle — in my ASP.NET Core work I've centralized shared CSS and layout partials for exactly the same reason, avoid duplicating styling across pages. ResourceDictionary is WPF's version of that same discipline. I haven't organized one in a real WPF app, but the underlying architectural instinct — don't repeat styling, centralize it — is one I already apply.
🔁 Follow-up Qs they might fire next
Q: StaticResource vs DynamicResource? — StaticResource resolves once at load time; DynamicResource re-resolves at runtime, useful for live theme switching.
Q: Where do you typically define app-wide resources? — App.xaml's Application.Resources, often merging separate dictionary files via MergedDictionaries.
Q: Can resources be scoped to just one window? — Yes, define them in that Window's own Resources section instead of App.xaml for narrower scope.

9. MVVM Pattern (Model-View-ViewModel)
MVVM na WPF-oda flagship architecture pattern. Model — pure data + business logic (example: Student class, Database access). View — XAML mattum, purely UI, no logic — just bindings. ViewModel — View kூட Model ஐ connect pannும் middle layer — properties (bindable), commands (button actions), and INotifyPropertyChanged ellam idhula thaan irukும். Idhu oda periya benefit na — View ah completely maathினாலும் (redesign), ViewModel touch pannத் தேவை இல்லை, and ViewModel ah unit test panna mudியும் UI touch pannாமலேயே — edhுனா ViewModel oru plain C# class, WPF control எதுவும் depend aaghirukaadhu.
🗣️ Interview la eppadi sollanumMVVM separates an app into three layers — Model for data and business rules, View for pure XAML UI with no logic, and ViewModel as the middle layer exposing bindable properties and ICommand-based actions that the View binds to. The View knows about the ViewModel through DataContext and data binding, but the ViewModel has zero reference to the View — which is what makes ViewModels unit-testable without spinning up any UI, and lets designers restyle the View without breaking application logic.
🎯 Idha eppadi honest ah frame pannanumThis is where I can honestly bridge from real experience — in my WinForms locker management project I deliberately kept business logic out of the Form class and pushed it into separate service/manager classes so the Form stayed thin and testable. That's the same instinct MVVM formalizes with the ViewModel layer and data binding instead of manual event-wiring. I haven't implemented full MVVM in a shipped WPF app, but the separation-of-concerns discipline behind it is already how I think about UI code.
🔁 Follow-up Qs they might fire next
Q: How does the View know which ViewModel to bind to? — Usually via DataContext, set either in code-behind or XAML, sometimes through a ViewModelLocator.
Q: Why is MVVM better than code-behind for testability? — ViewModel has no dependency on WPF UI classes, so you can unit test it directly without a UI thread or rendered controls.
Q: Is MVVM mandatory in WPF? — No, code-behind works fine for small apps, but MVVM scales much better for larger, testable, maintainable apps.
10. Commands in WPF (ICommand, RelayCommand/DelegateCommand)
WinForms la Button click ah handle pannணும் na, direct ah button1.Click += ... event handler eழுதுவோம் — idhu code-behind la tight ah irukும். WPF MVVM la adhை vேற mாதிரி pannுவோம் — Button oda Command property ah, ViewModel la irukka oru ICommand property kூட bind pannுவோம். ICommand-oda 2 main members — Execute() (action run pannுவது) and CanExecute() (button enable/disable aaha vேண்டுமா nu decide pannுவது — example: TextBox empty ah irukும்bodhu Save button automatic ah disable aahும்). RelayCommand illa DelegateCommand na, idha manual ah ovvொரு command class eழுதாம boilerplate குறைக்கும் helper classes.
🗣️ Interview la eppadi sollanumInstead of wiring a Click event handler in code-behind, WPF lets a Button's Command property bind to an ICommand exposed by the ViewModel. ICommand has Execute, which runs the action, and CanExecute, which WPF automatically uses to enable or disable the control — so a Save button can auto-disable when required fields are empty, without manually toggling IsEnabled. RelayCommand or DelegateCommand are lightweight ICommand implementations that take delegates in the constructor, avoiding writing a full command class for every action.
🎯 Idha eppadi honest ah frame pannanumI've done the WinForms equivalent for years — enabling/disabling Save or Submit buttons based on form validation state, wired through direct event handlers and manual IsEnabled toggling. ICommand's CanExecute is essentially automating that exact pattern I already implement by hand. I haven't used ICommand/RelayCommand in a real WPF project, but I understand precisely what manual problem it's replacing because I've solved that problem the WinForms way myself.
🔁 Follow-up Qs they might fire next
Q: How do you force CanExecute to re-evaluate? — Call CommandManager.InvalidateRequerySuggested(), or raise CanExecuteChanged manually in a custom RelayCommand.
Q: Can Commands take parameters? — Yes, via CommandParameter bound in XAML, passed into Execute(object parameter).
Q: Why not just use Click event in MVVM? — It would put logic in code-behind, breaking the View/ViewModel separation and hurting testability.
<Button Content="Save" Command="{Binding SaveCommand}"
        CommandParameter="{Binding SelectedStudent}"/>

public ICommand SaveCommand => new RelayCommand(
    execute: _ => SaveStudent(),
    canExecute: _ => !string.IsNullOrWhiteSpace(StudentName));
11. Value Converters (IValueConverter)
Sometimes namba data oda type, UI la kaananum na direct ah match aagaadhு — example: ViewModel la bool IsActive irukku, aana UI la adhை Visibility (Visible/Collapsed) ah convert pannணும், illa DateTime ah "dd-MMM-yyyy" format string ah convert pannணும். Idhை ViewModel kூட pannினா, ViewModel UI-specific concerns thெரிஞ்சுக்கும் (bad separation). Athுக்காக IValueConverter — Convert() (source→UI) and ConvertBack() (UI→source) methods vachு, oru small standalone class eழுதி, XAML binding la Converter= nu attach pannுவோம்.

12. Dispatcher & Async on UI Thread (same root problem as InvokeRequired)
WPF layoutum WinForms layoutum ஒரே fundamental rule follow pannும் — UI controls ஐ, andha UI thread mattum touch pannணும். Background thread la irundhே direct ah control update pannினா exception varும். WinForms la namba InvokeRequired check panni Invoke()/BeginInvoke() use pannுவோம். WPF la athுக்கு equivalent ah Dispatcher irukku — Dispatcher.Invoke() illa Dispatcher.BeginInvoke() vechு, background thread la இருந்து UI thread ku "வேலை" ஐ marshal pannலாம். Modern WPF code la async/await use pannினா idhு simple ah aagும் — await Task.Run(...) pண்ணி, andha await க்கு அப்புறம் வர code automatic ah UI thread ku back வந்து run aagும் (SynchronizationContext வழியா), so explicit Dispatcher call கூட வேணாம் பல cases la.
🗣️ Interview la eppadi sollanumWPF has the same constraint as WinForms — only the UI thread can touch UI controls directly. Where WinForms uses InvokeRequired and Invoke, WPF uses the Dispatcher object, via Dispatcher.Invoke or Dispatcher.BeginInvoke, to marshal a delegate back onto the UI thread from a background thread. With async/await, WPF's SynchronizationContext usually handles this automatically — after an awaited Task.Run completes, execution resumes back on the UI thread by default, so you often don't need to call the Dispatcher explicitly at all.
🎯 Idha eppadi honest ah frame pannanumThis one I can speak to with real confidence, because I've hit this exact problem in production — in my WinForms locker app, background polling/socket threads needed to update UI controls, and I used InvokeRequired plus Invoke to marshal those updates safely, and I've handled the async/await version of this too. WPF's Dispatcher solves the identical problem with a different API name; conceptually there's nothing new here for me to learn, just a new method name to reach for.
🔁 Follow-up Qs they might fire next
Q: Dispatcher.Invoke vs BeginInvoke? — Invoke is synchronous and blocks the calling thread until done; BeginInvoke is asynchronous/fire-and-forget, queued on the UI thread.
Q: What happens if you update a WPF control from a non-UI thread without Dispatcher? — Throws an InvalidOperationException — "The calling thread cannot access this object because a different thread owns it."
Q: Why does async/await avoid explicit Dispatcher calls? — Because WPF's SynchronizationContext captures the UI thread context before the await, and posts the continuation back to it automatically.
13. WPF vs WinForms — When to Use Which (clear-eyed comparison)

WPF — richer, skinnable, animatable UI, strong data-binding/MVVM story, better for customer-facing apps or apps with complex, evolving UI requirements, but steeper learning curve and more setup overhead for simple tools. 🗣️ Interview la eppadi sollanumWinForms is faster to build with and has a lower learning curve, which makes it a strong choice for internal tools, quick utilities, and admin consoles where UI polish isn't the priority — that's exactly the profile of my locker management application. 🎯 Idha eppadi honest ah frame pannanumI'd say this from real experience rather than theory — my WinForms locker app was exactly the "fast, functional, internal tool" case where WPF would have been overkill for the timeline we had. 🔁 Follow-up Qs they might fire next

I'd reach for CommunityToolkit.Mvvm for most apps, and consider Prism specifically when the app has real modular/navigation complexity.

🔁 Follow-up Qs they might fire next
Q: What do source generators actually do here? — Generate the INotifyPropertyChanged boilerplate and ICommand wrapper code at compile time from your attributed fields/methods, so you write less code by hand.
Q: Is Prism still actively maintained? — Yes, it has an active open-source community version and remains common in enterprise WPF codebases.
Q: Would you build MVVM without any toolkit? — For a small app, yes, plain INotifyPropertyChanged and a hand-rolled RelayCommand base class is enough; toolkits pay off as the app grows.
