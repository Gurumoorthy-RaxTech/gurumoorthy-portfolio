10A ADO.NET Core Concepts
1. ADO.NET Architecture Overview (Connection/Command/DataReader/DataAdapter/DataSet)
ADO.NET na — .NET la irundhu SQL Server (illa edha DB venaalum) kூடa pேசrathukku base-level toolkit. Namba oru restaurant nu vechukka — SqlConnection na restaurant kூடa phone la call pannuvathu (link create pannுவathு), SqlCommand na order sollுவathு ("enna venum" — SELECT/INSERT/UPDATE), SqlDataReader na waiter neradhi neradhi ஒரு dish kொடுத்துட்டே irukkா (fast, one-way, streaming), DataAdapter+DataSet na full order ah oru "parcel" ah pack panni edுthுkittu poga mudியும் (connection close pannிட்டாலும் data கூடவே இருக்கும்). Dapper, EF Core ellame idhே ADO.NET-oda mேலே thaan build aagirukku — ADO.NET thaan foundation layer.
🗣️ Interview la eppadi sollanum"ADO.NET is the foundational data-access layer in .NET — SqlConnection opens the link to the database, SqlCommand represents the SQL statement or stored procedure to execute, and then you either read results with a forward-only SqlDataReader for a connected, streaming approach, or use a DataAdapter to fill a disconnected DataSet/DataTable that you can work with even after the connection is closed. Every higher-level tool — Dapper, EF Core — is ultimately built on top of these same ADO.NET primitives."
🏗️ Project la eppadi implement pannirukeenIn the Bank Locker OTP system I directly used the connected model — a SqlConnection opened for the duration of the call, a SqlCommand carrying the SQL text or stored procedure name plus parameters, and either ExecuteNonQuery for writes like the audit-trail insert or a SqlDataReader for simple lookups — I didn't need the disconnected DataSet model since I wasn't showing that data in a grid, I just needed to hit the database and move on.
🔁 Follow-up Qs they might fire next
Q: Is ADO.NET still relevant when we have EF Core and Dapper? — Yes, Dapper and EF Core both use ADO.NET internally (SqlConnection/SqlCommand) under the hood; knowing it helps you understand what's actually happening.
Q: What namespace is ADO.NET SQL Server code in? — System.Data.SqlClient (or Microsoft.Data.SqlClient in newer .NET).
Q: Can ADO.NET work with non-SQL-Server databases? — Yes, via provider-specific classes like OleDbConnection, OdbcConnection, or MySqlConnection — same pattern, different provider.
2. Connected vs Disconnected Architecture (DataReader vs DataSet)
Connected architecture na — connection open ah irukkும் vareikkும் mattum data padikka mudiyும் (SqlDataReader). Namba call panniட்டு wait pannுрோம், waiter row-by-row data kொடுத்துட்டே irukkா. Disconnected architecture na — DataAdapter data ah full ah eduthுட்டு, connection ah close panniட்டு, apparam offline ah andha data vechு velai pannலாம் (DataSet/DataTable). Connected = fast but connection busy ah irukkும். Disconnected = slower/heavier but flexible — data ah UI la bind pannலாம், edit pannலாம், offline la kூடa vaikkலாம்.
🗣️ Interview la eppadi sollanum"In the connected architecture, the connection stays open while you read — SqlDataReader streams rows one at a time and is the fastest option for simple read-and-move-on scenarios. In the disconnected architecture, a DataAdapter fills a DataSet or DataTable in memory, then the connection can be closed while you keep working with that in-memory copy — this is useful when you need to bind data to a UI grid, edit it, and later push changes back."
🏗️ Project la eppadi implement pannirukeenFor the OTP APIs I stuck to the connected model — open connection, ExecuteReader or ExecuteNonQuery, close immediately in a using block, because it was a stateless REST API call, not something that needed to hold data around. The disconnected DataSet approach I actually used on the WinForms admin-tool side of that project, where I filled a DataTable via a DataAdapter and bound it straight to a DataGridView so an operator could view and refresh OTP audit logs without keeping the connection open the whole time.
🔁 Follow-up Qs they might fire next
Q: Why is DataReader considered "faster"? — No extra in-memory table build-up or change tracking, it just streams rows directly as you iterate.
Q: Can you use SqlDataReader after closing the connection? — No — DataReader needs the connection open the whole time you're reading; closing it early throws an exception.
Q: When would disconnected model be a bad choice? — For high-frequency API calls where you just need one value or a few rows — the DataSet overhead is unnecessary there.
3. SqlConnection & Connection Pooling (reuse, not recreate)
Ovvoru SQL Server connection open pannுрathு expensive operation — TCP handshake, authentication ellame aagும். So namba conn.Close() pannினalum, .NET andha physical connection ah actual ah kill pannாthu — oru "pool" la thirumba podும். Next time yaru vேணumalum putha connection kேttа, pool la irundhu already-open connection ah reuse pannும். Idhu vachu 1000+ concurrent users handle panna easy aagும் — puthu connection create panna time waste aagாthu.
🗣️ Interview la eppadi sollanum"Opening a physical database connection is expensive, so ADO.NET maintains a connection pool under the hood. When I call Close() or Dispose() on a SqlConnection, it doesn't actually tear down the TCP connection — it returns it to the pool keyed by the connection string, and the next request that needs a connection gets handed back an already-open one. This is what makes it feasible to serve high concurrent load without connection setup becoming a bottleneck."
🏗️ Project la eppadi implement pannirukeenIn the OTP APIs, every method opened a fresh SqlConnection inside a using block and closed it right after the query — I never held connections open longer than needed. Because pooling is on by default (Max Pool Size in the connection string), this pattern of open-use-close per call didn't cost real connection-setup time under load, the pool just handed back a live connection each time, which kept response times fast even during concurrent OTP validation bursts.
🔁 Follow-up Qs they might fire next
Q: How do you configure pool size? — Via connection string keys like Min Pool Size and Max Pool Size.
Q: What happens if pool is exhausted? — New requests wait for a connection to free up, or time out with an exception if the wait exceeds Connect Timeout.
Q: Does connection pooling happen per connection string? — Yes — connections are pooled separately per distinct connection string (including things like different Application Name).
4. SqlCommand — ExecuteReader vs ExecuteNonQuery vs ExecuteScalar (when to use which)
SqlCommand oda 3 main execute methods irukku, ovvoru vேலைkku ovvondru. ExecuteReader — SELECT panni multiple rows/columns thirumba venum na (returns SqlDataReader). ExecuteNonQuery — INSERT/UPDATE/DELETE panna, thirumba varathு number of rows affected mattum (example: 3 rows update aachu). ExecuteScalar — result la first row first column ஒரே ஒரு value mattum venum na (example: COUNT(*), MAX(Id)) — over-kill ah full reader use pannாma, idhே fast.
🗣️ Interview la eppadi sollanum"I pick based on what I need back. ExecuteReader when I need to iterate over a result set with multiple rows and columns. ExecuteNonQuery for INSERT, UPDATE, DELETE statements where I only care about the number of rows affected. And ExecuteScalar when I just need a single value — like a count, a max ID, or checking if a row exists — it's the cheapest option because it doesn't build a full reader for one value."
🏗️ Project la eppadi implement pannirukeenThe OTP audit-trail insert used ExecuteNonQuery since I only needed to confirm the row got written — I checked the returned rows-affected count to make sure it wasn't zero before treating the audit log as successful. For a quick existence check like "does an active OTP already exist for this locker ID" I used ExecuteScalar with a SELECT COUNT(1) query — no need to open a full reader just to get one number back.
🔁 Follow-up Qs they might fire next
Q: What does ExecuteNonQuery return for a SELECT statement? — Technically works but it's misused — it doesn't give you row data back, only ever an affected-rows-style count, so don't use it for SELECT.
Q: Is ExecuteScalar faster than ExecuteReader for a single value? — Yes — less overhead since it doesn't set up the full reader/cursor machinery.
Q: What does ExecuteNonQuery return for an INSERT with no matching WHERE affected rows? — Returns 0 — that's how I detect a failed/no-op write in code.
5. SqlDataReader — Forward-Only, Read-Only Cursor (firehose)
SqlDataReader ah "firehose cursor" nu solலுவாrgal — edhுkkா na adhu ஒரே direction mattum move pannும் (forward-only), pinnal pogaadhu, and read-only (edit panna mudியாthு). Full result set ah memory la load pannாma, row-by-row ah stream pannும் — so million rows irundhalum, memory efficient ah irukும். while(reader.Read()) loop la ஒரு row aprovm ஒரு row படிக்கனும், skip pannalam aana back pogaமுடியாது.
🗣️ Interview la eppadi sollanum"SqlDataReader is a forward-only, read-only stream over the result set — often called a firehose cursor. You call Read() to advance one row at a time, and you can't go backward or randomly access a row. Because it doesn't materialize the entire result set in memory, it's very efficient for large result sets — you're processing data as it arrives instead of waiting for everything to load."
🏗️ Project la eppadi implement pannirukeenI used SqlDataReader for lightweight single-row lookups in the OTP flow — like fetching the last generated OTP and its expiry timestamp for a given locker ID. The pattern was open connection, ExecuteReader, check reader.Read() once since I only expected one row, pull the OTP value and expiry with reader.GetString()/GetDateTime(), then let the using block close everything — no need for the full DataSet machinery for a single-row check.
🔁 Follow-up Qs they might fire next
Q: Can you have multiple result sets in one reader? — Yes, if the command returns multiple SELECTs, use reader.NextResult() to move to the next result set.
Q: Why must the connection stay open with a DataReader? — It's streaming live from the server — closing the connection mid-read breaks the stream and throws an exception.
Q: How do you read a column safely if it might be NULL? — Check reader.IsDBNull(ordinal) before calling the typed getter, or use reader["col"] as string.
6. DataSet vs DataTable vs DataAdapter (in-memory data containers)
DataTable — oru single table oda in-memory copy, rows and columns irukும், Excel sheet madhiri. DataSet — multiple DataTables ah oru container la vechுkittu irukும் (relationships kூடa define pannலாம் between tables) — full "mini in-memory database" madhiri. DataAdapter — DataSet/DataTable-kum actual database-kum idaiyila bridge — Fill() panniна DB-la irundhu data eduthுட்டு varும், Update() panniна in-memory changes ah thirumba DB ku push pannும்.
🗣️ Interview la eppadi sollanum"A DataTable holds one table's worth of data in memory — rows and columns, like a spreadsheet. A DataSet is a container that can hold multiple DataTables plus relationships between them, essentially an in-memory relational structure. A DataAdapter is the bridge between the database and these in-memory objects — SqlDataAdapter.Fill() populates a DataTable from a query, and Update() pushes any in-memory changes back to the database using auto-generated or custom insert/update/delete commands."
🏗️ Project la eppadi implement pannirukeenOn the WinForms admin utility for the locker project, I used a SqlDataAdapter with a SELECT query against the OTP audit-log table, called Fill() to populate a DataTable, and then bound that DataTable directly to a DataGridView so the support team could browse recent OTP attempts without me writing manual row-mapping code — I didn't use a full DataSet since I only needed one table, a plain DataTable was enough.
🔁 Follow-up Qs they might fire next
Q: When would you use DataSet over a plain DataTable? — When you need multiple related tables in memory together with parent-child relationships, like Master + Detail data pulled in one round trip.
Q: Does DataAdapter.Update() know what changed? — Yes — DataTable tracks row states (Added/Modified/Deleted) via DataRowState, and the adapter uses that to decide insert/update/delete.
Q: Is DataSet still relevant in modern .NET Core apps? — Rarely for new web APIs — it's more common in legacy WinForms/desktop apps or tools; modern web APIs return DTOs/JSON instead.
7. Parameterized Queries with SqlParameter (SQL injection prevention)
Namba user input ah direct ah SQL string kூடa + pannி join panniна, attacker adha misuse panni database ah hack pannலாம் — example ' OR 1=1 -- type input kொடுத்தா, whole table வெளியே varும். SqlParameter use panniна — input ah "pure data" ah mattum treat pannும், query structure ah maathாtu — so injection %100 block aagும். Idhu ADO.NET la basic rule — namba எப்போவும் @paramName placeholder use pannுவோm, string concatenation panna maاttேn.
🗣️ Interview la eppadi sollanum"I never build SQL by concatenating user input into the query string. Instead I use SqlParameter with placeholders like @LockerId — the database treats the parameter value purely as data, never as executable SQL, which eliminates SQL injection risk entirely. It also gives SQL Server a chance to reuse cached execution plans since the query shape stays constant across calls."
🏗️ Project la eppadi implement pannirukeen Every single ADO.NET call in the OTP system — especially the audit-trail insert that fires after every OTP action — used SqlParameter for every value. Here's roughly how the audit insert looked: 
using (var conn = new SqlConnection(connectionString))
using (var cmd = new SqlCommand(
    "INSERT INTO OtpAuditLog (LockerId, Action, OtpValue, Status, CreatedOn) " +
    "VALUES (@LockerId, @Action, @OtpValue, @Status, GETDATE())", conn))
{
    cmd.Parameters.Add("@LockerId", SqlDbType.Int).Value = lockerId;
    cmd.Parameters.Add("@Action", SqlDbType.VarChar, 50).Value = action;
    cmd.Parameters.Add("@OtpValue", SqlDbType.VarChar, 10).Value = otpValue;
    cmd.Parameters.Add("@Status", SqlDbType.VarChar, 20).Value = status;

    conn.Open();
    cmd.ExecuteNonQuery();
}
Every value goes in as a typed SqlParameter, never string-concatenated, so there was zero SQL-injection surface on the audit path even though this table got hit after every generate/deliver/validate/expire action.
🔁 Follow-up Qs they might fire next
Q: AddWithValue() vs Parameters.Add() — what's the difference? — AddWithValue infers the SqlDbType from the .NET value, which can cause implicit conversions/plan cache bloat; Add() with explicit SqlDbType and size is safer and more predictable.
Q: Does parameterization also help performance? — Yes — SQL Server can cache and reuse the execution plan since the query text doesn't change per call, only the parameter values.
Q: Is stored-procedure execution automatically safe from injection? — Only if the parameters passed into the proc are also parameterized — dynamic SQL built inside a proc can still be injectable.
8. Transactions in ADO.NET (SqlTransaction, Commit/Rollback)
Namba multiple related DB operations ஒரே unit ah success aaganum, illna எல்லாமே fail aaganum nu venum na, transaction use pannுрோம். Example: OTP validate pannумbodhu — "OTP status ah Used ah maathanum" + "audit log ah entry pannanum" rendும் ஒரே time la success aaganum, ஒண்ணு fail aana rendும் rollback aaganum. conn.BeginTransaction() panniна transaction start aagும், ovvoru command-கும் andha transaction ah assign pannуணும், எல்லாம் sari na Commit(), edhavadhu error na Rollback().
🗣️ Interview la eppadi sollanum"When multiple related operations must succeed or fail together, I wrap them in a SqlTransaction. I call BeginTransaction on the connection, assign that transaction to every SqlCommand involved, and if all operations succeed I call Commit — if any of them throws, I catch it and call Rollback so the database is left in a consistent state rather than half-updated."
🏗️ Project la eppadi implement pannirukeenIn the OTP validation flow, when a correct OTP came in, I needed to mark it "Used" in the OtpRequest table and write a corresponding audit-log row in the same operation — I wrapped both SqlCommands in one SqlTransaction so a crash or DB error between the two steps couldn't leave an OTP marked used without an audit trail, or vice versa. Wrapped in a try-catch, Commit() on success, Rollback() in the catch block, then dispose everything in the finally/using.
🔁 Follow-up Qs they might fire next
Q: What happens if you forget to assign the transaction to a command? — You get a runtime exception — "ExecuteNonQuery requires the command to have a transaction" — because the connection has an active transaction pending.
Q: Can you nest transactions in ADO.NET? — Not directly with SqlTransaction — you'd use savepoints (Save()) for partial rollback points within one transaction.
Q: How is this different from TransactionScope? — TransactionScope is a higher-level, ambient transaction API that can span multiple connections/resources; SqlTransaction is tied to one SqlConnection.

9. Executing Stored Procedures via ADO.NET (CommandType.StoredProcedure)
SQL text direct ah eludhுрathukku badhilா, database la already irukka stored procedure ah call pannலாம். cmd.CommandType = CommandType.StoredProcedure nு set pannிட்டு, cmd.CommandText la proc name mattum kொடுத்தா pothும். Parameters ah SqlParameter vाசி add pannуணும் (proc definition la irukka @Param names match aaganum). Idhu vachu complex logic DB level la irukும், reusable ஆகும்.
🗣️ Interview la eppadi sollanum"To call a stored procedure with ADO.NET, I set CommandType to CommandType.StoredProcedure, set CommandText to just the procedure name, and add SqlParameters matching the procedure's declared parameter names. ADO.NET then handles building the RPC call to SQL Server instead of sending raw SQL text."
🏗️ Project la eppadi implement pannirukeenSome of the reporting queries around OTP audit history — like "get all audit entries for a locker in a date range" — were wrapped in stored procedures so the logic lived close to the data and could be tuned by adjusting the proc without redeploying code. I called them with plain ADO.NET: cmd.CommandType = CommandType.StoredProcedure; cmd.CommandText = "usp_GetOtpAuditByLocker"; then added SqlParameters for @LockerId, @FromDate, @ToDate, and read results with a SqlDataReader.
🔁 Follow-up Qs they might fire next
Q: How do you get an output parameter back from a stored procedure? — Add a SqlParameter with Direction = ParameterDirection.Output, execute the command, then read parameter.Value afterward.
Q: Why prefer stored procedures for some queries? — Logic stays close to the data, can be tuned/indexed independently, and reduces network chatter for complex multi-step logic.
Q: Can a stored procedure return multiple result sets via ADO.NET? — Yes — use reader.NextResult() to step through each SELECT the proc returns.
10. Connection Lifecycle Management (using / IDisposable)
SqlConnection, SqlCommand, SqlDataReader ellame IDisposable implement pannукிறது — meaning idhுங்களை use panniட்டு, kடைசியில் clean ah dispose pannуணும். Namba manual ah Close() call panna maranu poனா — especially exception vandhu code andha line ku வராம போனா — connection open ah pool la stuck aagும். Loop kulla romba connections open panniட்டு close pannாமाл் இருந்தால், DB "max connections reached" nு error kொடுத்து whole system crash aagும். using(...) block use panniна, exception vandhalும் kூடa automatic ah Dispose() call aagும் guarantee — idhே correct pattern.
🗣️ Interview la eppadi sollanum"SqlConnection, SqlCommand, and SqlDataReader all implement IDisposable, so I always wrap them in using statements or using declarations. That guarantees Dispose() runs even if an exception is thrown mid-query, returning the connection cleanly to the pool. If you skip that and rely on manual Close() calls, an exception before that line leaves the connection open, and under load that exhausts the pool and the database starts rejecting new connections."
🏗️ Project la eppadi implement pannirukeenEvery ADO.NET method I wrote in the OTP system followed the same nested-using pattern — using (var conn = new SqlConnection(cs)) using (var cmd = new SqlCommand(sql, conn)) { ... } — so regardless of whether the query succeeded, threw a SqlException, or the caller cancelled, the connection and command always got disposed. This mattered a lot for the audit-trail insert specifically because it fired after every single OTP action — generate, deliver, validate, expire — so any connection leak there would have exhausted the pool fast under real traffic.
🔁 Follow-up Qs they might fire next
Q: What actually happens on Dispose() for SqlConnection? — It returns the physical connection to the pool (or truly closes it if pooling is off) rather than necessarily terminating the TCP session immediately.
Q: Do you need to explicitly call Close() if you use using? — No — Dispose() calls Close() internally for SqlConnection, using is enough.
Q: What's the risk of a connection leak in production? — Pool exhaustion — new requests start timing out or throwing "Timeout expired, the connection pool is exhausted" errors, effectively an outage.
11. Async ADO.NET (ExecuteReaderAsync, ExecuteNonQueryAsync)
Namba DB call pannумbodhu, adhு network round-trip — konjam time eduthுkும். Andha wait time la thread ah block panniна, adhу thread vera velaikkும் use panna mudியாthு. ExecuteReaderAsync(), ExecuteNonQueryAsync(), ExecuteScalarAsync() — idhуngа use panniна, await pannும்bodhு thread free aagும், DB reply வந்தা thaan andha continuation resume aagும். High concurrency APIs ku idhு mandatory — illna thread pool exhaust aagும்.
🗣️ Interview la eppadi sollanum"For any I/O-bound database call in an ASP.NET Core or high-concurrency context, I use the async variants — ExecuteReaderAsync, ExecuteNonQueryAsync, ExecuteScalarAsync — with await. That frees up the thread while waiting on the network round-trip to SQL Server instead of blocking it, which is essential for scaling to hundreds or thousands of concurrent requests without exhausting the thread pool."
🏗️ Project la eppadi implement pannirukeenThe OTP validate and deliver endpoints were REST APIs expected to handle bursts of concurrent requests, so I used the async ADO.NET methods there — await cmd.ExecuteNonQueryAsync() for the audit insert, await cmd.ExecuteReaderAsync() for the OTP lookup — instead of the blocking sync versions, so a slow database round-trip on one request didn't tie up a thread that could be serving another user's OTP request.
🔁 Follow-up Qs they might fire next
Q: Does async make a single query faster? — No — it doesn't reduce query latency, it improves throughput/scalability by freeing threads during the wait.
Q: Should you mix sync and async ADO.NET calls in the same method? — No — stick to async all the way through (async all the way) to avoid deadlocks, especially avoid calling .Result/.Wait() on async calls.
Q: Is there an async equivalent for opening a connection? — Yes — OpenAsync() instead of Open().
12. Error Handling & Connection Resiliency (SqlException, retries)
DB call panniна, palavidhamana errors varலாம் — connection timeout, deadlock, network blip, constraint violation. Namba SqlException ah specifically catch pannі, adhுla irukka Number property vечு andha error edhு specific type nு identify pannலாம் (example: 1205 = deadlock, -2 = timeout). Transient errors (temporary network issue) ku retry logic vेchுkondu, permanent errors (constraint violation) ku user ku proper error message kொடுக்கनum. finally block la connection dispose pannுрathு уறுதி pannуணும்.
🗣️ Interview la eppadi sollanum"I catch SqlException specifically rather than a generic Exception, so I can inspect the error Number and distinguish transient issues — like a timeout or deadlock — from permanent ones like a constraint violation. For transient errors I add a bounded retry with a small backoff; for permanent ones I fail fast and surface a meaningful error instead of retrying something that will never succeed. And regardless of the outcome, the connection cleanup happens in a using block or finally, so a failed query never leaves a connection dangling."
🏗️ Project la eppadi implement pannirukeenAround the audit-trail insert specifically, I wrapped the ExecuteNonQuery call in a try-catch for SqlException — since this insert fired after every OTP action, I didn't want a transient DB hiccup to bubble up and fail the actual OTP generate/validate response to the user. I logged the SqlException details and, for known-transient error numbers like timeouts, did a single retry before giving up and logging a failure, so a momentary DB blip on the audit path wouldn't break the core OTP flow.
🔁 Follow-up Qs they might fire next
Q: How do you tell a transient error from a permanent one? — Check SqlException.Number against known transient codes (timeout, deadlock victim, connection broken) vs constraint/permission errors that won't resolve on retry.
Q: Would you use Polly for retries in ADO.NET code? — Yes, in a modern .NET app I'd wrap calls with a Polly retry/circuit-breaker policy rather than hand-rolling retry loops.
Q: What's a deadlock, in one line? — Two transactions each holding a lock the other needs — SQL Server kills one as the "deadlock victim" and it throws a SqlException.
13. ADO.NET vs Dapper vs EF Core (the honest comparison)
Moonu um data access ku thaan, aana different levels of abstraction. ADO.NET — lowest level, full manual control, namba SqlConnection/SqlCommand ellame naame handle pannуணும் — verbose aana romba fast, zero overhead. Dapper — micro-ORM, ADO.NET-oda mேலே thin layer, SQL namba eludhуணும் aana result-mapping automatic — намба resume la irukka 40% query optimization, 50% OTP validation speedup ellame Dapper vечு achieve pannirukேn. EF Core — full ORM, change tracking, migrations, LINQ-to-SQL translation — fast development aana konjam heavy. Naan raw ADO.NET ah ippoவும் use pannுрேn — simple, single-table, high-frequency operations ku, ORM mapping overhead venaам் — example audit-trail insert.
🗣️ Interview la eppadi sollanum"They're all valid, just at different abstraction levels. Raw ADO.NET gives full manual control with zero mapping overhead — best for simple, single-statement, high-frequency operations. Dapper sits on top of ADO.NET as a micro-ORM — I still write the SQL, but it maps results to objects for me, which is what I used for the complex join-heavy OTP validation queries to cut response time by 50%. EF Core is the full ORM — change tracking, migrations, LINQ — great for productivity on standard CRUD, which is what I used across most of the Zen Campus ERP. I don't treat this as 'pick one' — in the Bank Locker project I used all three in the parts where each one actually earns its place."
🏗️ Project la eppadi implement pannirukeenConcretely: the OTP audit-trail insert — one table, one INSERT, fired constantly — went through raw ADO.NET with SqlCommand and SqlParameter because there was nothing to map back, no relationships, no tracking needed, just "write this row and move on" as fast as possible. The OTP validation logic — which needed to join the OTP request table with the locker/user tables and check status/expiry together — went through Dapper, and optimizing that path specifically is what got us the 50% response-time cut on OTP validation. EF Core handled the broader CRUD/business-entity side of that .NET Framework project where relationships and change tracking actually paid off.
🔁 Follow-up Qs they might fire next
Q: Isn't raw ADO.NET for a single insert overkill compared to Dapper? — Not really — Dapper's overhead is small but non-zero (reflection/IL-mapping setup), and for something this hot-path and this simple, raw ADO.NET is marginally leaner and I already have full control over exact parameter types.
Q: Would you ever rewrite that audit insert in Dapper? — I could, it's a one-line connection.Execute(sql, params) — I just never needed to since the raw ADO.NET version was already simple, fast, and working fine.
Q: How do you decide which of the three to reach for on a new task? — Single simple statement, no mapping needed → ADO.NET. SQL-heavy report/join → Dapper. Standard CRUD with relationships/need for migrations → EF Core.
14. Binding ADO.NET Data to a WinForms DataGridView (bridge to Part 11)
WinForms desktop tools la, DataGridView oru grid control — Excel madhiri rows/columns kaatuрாthu. ADO.NET vечு data eduthுட்டு, direct ah DataGridView.DataSource ku assign panniна pothும் — grid automatic ah columns generate pannும், rows populate pannும். Namba DataAdapter vечு DataTable fill pannிட்டு, adha DataSource ah kொடுthா, minimal code la full grid ready aagும். Idhு thaan Part 11 WinForms la detail ah paakப்போрோm.
🗣️ Interview la eppadi sollanum"For a WinForms admin tool, I fill a DataTable using a SqlDataAdapter's Fill() method, then set that DataTable directly as the DataGridView's DataSource. WinForms auto-generates the columns and rows from the DataTable's schema, so with just a few lines of ADO.NET code you get a fully working, editable grid — no manual row-by-row UI code needed."
🏗️ Project la eppadi implement pannirukeenThe internal support tool for the Bank Locker project had a WinForms screen where operators could review OTP audit logs by locker or date range — I used a SqlDataAdapter with a parameterized SELECT against the OtpAuditLog table, called Fill() into a DataTable, and bound it straight to a DataGridView. That gave the support team a sortable, filterable grid of who accessed which locker and when, without me writing any manual UI-population code — just the ADO.NET fetch and one DataSource assignment.
🔁 Follow-up Qs they might fire next
Q: How do you refresh the grid when data changes? — Re-run Fill() into the same DataTable (or a new one) and reassign DataSource, or call DataGridView.Refresh() if the underlying DataTable already updated.
Q: Can users edit data directly in the grid and save it back? — Yes — edits update the DataTable's row state, and calling DataAdapter.Update() pushes those changes back to the database.
Q: Would you use this DataTable-binding approach in a modern app? — For WinForms tooling, yes, it's still idiomatic; for a web app I'd never do this — that's what Web API + a JS grid or Razor view is for.
15. When I Personally Reach for Raw ADO.NET Over Dapper (the audit-trail-insert story)
Idhу romba specific question — interviewer "neenga Dapper иருக்கும்bodhு yeatуkku ADO.NET use panninga" nu kேttа, idhே story sollуணும். Bank locker OTP project la — OTP generate pannின apparam, deliver pannின apparam, validate pannின apparam, expire aana apparam — ஒவ்வொரு action-கும் apparam ஒரே ஒரு row "OtpAuditLog" table la insert pannуணும். Idhு single-table, no joins, no complex mapping — mattum "insert this row, confirm it wrote, move on" nu velai. Idhуkkா Dapper venам், straight SqlConnection+SqlCommand+SqlParameter thaan use pannினேn — konjam kூடa mapping overhead illama, code kூடa transparent ah irukும் — exactly enna query run aagுрathு nu clear ah теriуும். Dapper ah naan reserve pannினேn — OTP validate pannумbodhу locker table, user table, OTP table moonrayum join panни status/expiry check panна vேணும் — andha complex query ku thaan Dapper use panniна result mapping easy aagும், idhуnala thaan OTP validation response time 50% kammi aachu.
🗣️ Interview la eppadi sollanum"In the Bank Locker OTP system, every OTP action — generate, deliver, validate, expire — triggers an audit-trail insert into a single table, no joins, no relationships, just 'write this row and confirm it landed.' For that I used raw ADO.NET — SqlConnection, SqlCommand, SqlParameter — because there's nothing to map back and Dapper's mapping layer, however lightweight, is overhead I simply don't need for a fire-and-forget single-table insert. I reserved Dapper for the OTP validation query, which joins the OTP, locker, and user tables to check status and expiry together — there, Dapper's automatic object mapping over that multi-table result actually saves real code and time, and optimizing that specific path with Dapper is what got us the 50% cut in OTP validation response time. So it's not 'ADO.NET vs Dapper' as a blanket choice — I match the tool to how simple or complex the query and its result shape are."
🏗️ Project la eppadi implement pannirukeenConcretely, the audit method looked like a small reusable helper — LogOtpAudit(int lockerId, string action, string otpValue, string status) — that opened a SqlConnection in a using block, built an INSERT with named SqlParameters for every value (never string-concatenated), called ExecuteNonQuery, and checked the rows-affected result. This helper got called from every stage of the OTP lifecycle — right after OTP generation, right after the SMS gateway confirmed delivery, right after validation succeeded or failed, and when an OTP expired unused — so it had to be lean and predictable, which is exactly what plain ADO.NET gave me over pulling in Dapper for a one-line insert.
🔁 Follow-up Qs they might fire next
Q: Wouldn't Dapper's Execute() for that insert be just as simple? — Pretty close, yes — it's a judgment call on a very simple hot path; I kept it in raw ADO.NET partly because it was already there and working, and there was no measurable benefit to changing it.
Q: How did you verify the 50% improvement was real? — Compared before/after response times for the OTP validation endpoint using logged timestamps/APM data once the join query moved from the earlier approach to Dapper-optimized SQL with proper parameters and indexing.
Q: Did the audit-trail insert ever become a bottleneck itself? — No — single-table inserts with an indexed identity column are cheap; the real cost was in the multi-table validation join, which is why that's where the optimization effort went.
