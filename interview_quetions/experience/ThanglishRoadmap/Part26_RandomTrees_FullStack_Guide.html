1. SOAP vs REST, and what is WCF (Genuine gap — likely a 1-2 min rapid-fire question)
SOAP nu solradhu oru old-style, strict protocol — request/response ellame XML envelope kulla pack pannanum, WSDL file la contract define pannanum. REST adha vida light-weight — HTTP verbs (GET/POST/PUT/DELETE) direct ah use pannuvom, JSON return pannuvom, no strict envelope. WCF (Windows Communication Foundation) nu solradhu — .NET Framework kudutha oru framework, adhula neenga SOAP services um, REST-style services um, TCP/named-pipe services um — ella "binding" um build panna mudiyum, same code model la (ServiceContract/OperationContract attributes). Adhaal WCF ஒரு specific protocol illa, multiple protocol support panra ஒரு framework. Modern ASP.NET Core la WCF replace panradhu — gRPC (high performance, gRPC/Protobuf) அல்லது plain REST Web API. Naan production la mostly REST Web API mattum thaan use pannirukken, aana legacy systems (bank/insurance/government) integration pannumbodhu still SOAP/WCF endpoints consume pannanum — adhukku Connected Service add pannி, auto-generated proxy client mூlama call pannuvom.
🗣️ Interview la eppadi sollanum"SOAP is a strict, XML-based messaging protocol with a WSDL contract — heavier but very formal, still common in banking/insurance/government integrations. REST is lighter — just HTTP verbs and JSON, which is what I use in production. WCF is a .NET Framework technology that can host services over multiple protocols — SOAP, REST-style, TCP, named pipes — through the same ServiceContract/OperationContract model. In ASP.NET Core, WCF's role is generally taken over by REST Web API or gRPC for high-performance internal calls. I haven't built a WCF service from scratch, but I'm comfortable consuming a SOAP/WCF endpoint from a .NET client using a generated service reference or proxy."
🔁 Follow-up Qs they might fire next
Q: How would you call a SOAP endpoint from ASP.NET Core? — Add a connected service / WCF Web Service Reference (via the community WCF Connected Service tooling since ASP.NET Core dropped the built-in wizard), which generates a proxy client class; then call it like any injected service.
Q: Why did Microsoft move away from WCF in .NET Core? — WCF is tightly coupled to Windows/.NET Framework internals; the ecosystem moved to REST for interoperability and gRPC for performance, both of which are cross-platform and align with microservices better.
2. Blazor Server vs Blazor WebAssembly (WASM) (Genuine gap — completely new topic)
Blazor nu solradhu — C# vachu, JavaScript illama, interactive web UI build panra Microsoft oda framework. Rendu models irukku. Blazor Server — UI logic server la thaan run aagum, browser kூடа SignalR (WebSocket) connection vachu, ovvoru button click um server ku poyi, DOM diff calculate panni, browser ku patch anuppum. Fast startup (chinna download), aana constant network connection venum — connection poyita UI freeze aagum. Blazor WebAssembly — muzhu .NET runtime ah, C# code ah, WASM ah compile panni, browser kூllаЕ download panni run pannுрроம் — server dependency illa ஒரு trip ku appuram, offline-capable, aana initial download periசு (few MB), first load slow. Namba MVC project la irundhu Blazor ku switch pannanumna — Razor Components (.razor files) use pannuvom, adhula HTML + C# mix panna mudiyum (namba Razor view madhiriye, aana client-side interactivity kூடа).
🗣️ Interview la eppadi sollanum"Blazor lets you build interactive web UI in C# instead of JavaScript, using Razor Components. Blazor Server keeps the UI logic on the server and uses a SignalR/WebSocket connection to push DOM updates to the browser — fast initial load, but it needs a constant connection. Blazor WebAssembly compiles the actual .NET runtime to WASM and runs entirely in the browser — no server round-trip per interaction, and it works offline after load, but the first download is larger. I haven't shipped a production Blazor app yet, but I understand the component model well since it's built on Razor syntax, which I use daily in MVC views — and I'd be comfortable picking it up quickly."
🔁 Follow-up Qs they might fire next
Q: When would you pick Blazor Server over WASM? — When you want a fast first paint, don't want to expose app logic to the client, and can tolerate a live connection requirement — e.g. an internal admin dashboard on a reliable network.
Q: How does a Razor Component differ from an MVC Razor view? — An MVC Razor view (.cshtml) is rendered once per request on the server and sent as static HTML; a Razor Component (.razor) is a stateful, reusable UI unit with its own lifecycle (OnInitializedAsync, parameters, event handlers) that can re-render itself without a full page reload.

D Real project — EmployeeDeptApi (built this week, exact JD stack match)
🏗️ What it actually is
A 4-project solution built to sharpen exactly this JD's stack: one Web API with two swappable data-access implementations, plus two different front-end clients consuming the same API.
EmployeeDeptApi — ASP.NET Core 9 Web API, EF Core (SQL Server, code-first migrations), JWT authentication, Swagger with Bearer auth, generic repository + service layer, a custom exception-handling middleware, a request-logging middleware, and two MVC-style filters (ValidateModelFilter for 400s on bad input, LogActionFilter for per-action logging). EmployeeDeptApi.DapperAdo — the same Employee/Department CRUD, but implemented twice more: once with raw ADO.NET (SqlCommand, manual SqlDataReader mapping, output parameters) and once with Dapper (micro-ORM, calling the same stored procedures) — so you can speak to all three data-access technologies (EF Core, Dapper, ADO.NET) from one real codebase. EmployeeDeptApi.WebApp — a Razor Pages front-end (login page, dashboard) that consumes the API over a typed HttpClient with a Bearer token — the MVC-style web client. EmployeeDeptApi.WinForms — a desktop client (Login → Dashboard → Add Employee forms) that consumes the exact same REST API using .NET's Generic Host for DI, proving the API is a true reusable backend, not tied to one UI.
Talking point 1 — JWT auth, end to end
TokenService generates a signed JWT with username/role claims; Program.cs wires up AddJwtBearer with issuer/audience/lifetime validation; controllers are protected with [Authorize]. Both clients (WebApp, WinForms) store the token after login and attach it as a Bearer header on every subsequent call.
public string GenerateToken(string username, string role)
{
    var claims = new[] {
        new Claim(JwtRegisteredClaimNames.Sub, username),
        new Claim(ClaimTypes.Role, role)
    };
    var key = new SymmetricSecurityKey(Encoding.UTF8.GetBytes(_jwtSettings.Key));
    var token = new JwtSecurityToken(_jwtSettings.Issuer, _jwtSettings.Audience,
        claims, expires: DateTime.UtcNow.AddMinutes(_jwtSettings.ExpiryMinutes),
        signingCredentials: new SigningCredentials(key, SecurityAlgorithms.HmacSha256));
    return new JwtSecurityTokenHandler().WriteToken(token);
}
🗣️ Interview la eppadi sollanum"I generate a signed JWT on login with the user's identity and role as claims, and ASP.NET Core's JwtBearer middleware validates the signature, issuer, audience and expiry on every request — anything decorated with [Authorize] rejects the request before it reaches my controller if the token's invalid. Both my web client and desktop client just attach the token as a Bearer header, so the same auth model works identically for a browser and a WinForms app."
Talking point 2 — three data-access technologies, same contract
Same IEmployeeRepository interface, three implementations: EF Core (LINQ, DbSet, migrations), Dapper (stored proc + DynamicParameters for output params), ADO.NET (raw SqlCommand/SqlDataReader with manual column mapping and null-handling). This is the strongest interview talking point — it proves you understand what each layer is actually doing underneath, not just "I called EF Core."
// Dapper — output parameter pattern
var p = new DynamicParameters();
p.Add("FirstName", employee.FirstName);
p.Add("NewEmployeeId", dbType: DbType.Int32, direction: ParameterDirection.Output);
await connection.ExecuteAsync("sp_Employee_Insert", p, commandType: CommandType.StoredProcedure);

// ADO.NET — same operation, manual

🗣️ Interview la eppadi sollanum"To really understand the trade-offs, I implemented the same repository interface three ways against the same stored procedures — EF Core for productivity and LINQ, Dapper for a thin, fast micro-ORM layer that still lets me hand-write SQL, and raw ADO.NET so I understand exactly what's happening at the SqlCommand/SqlDataReader level underneath both of them — manual parameter binding, output parameters, null handling with IsDBNull. 🔁 Follow-up Qs they might fire next
Q: When would you pick Dapper over EF Core? — When I need tight control over the exact SQL executed and lower overhead — reporting queries, high-throughput endpoints — while still avoiding ADO.NET's manual mapping boilerplate.
Q: Why stored procedures instead of inline SQL in Dapper? — Precompiled execution plan reuse, centralizing data-access logic in the DB so it can be tuned/audited independently of app deploys, and a smaller SQL-injection surface since parameters are enforced at the proc boundary.
Talking point 3 — cross-cutting concerns done the ASP.NET Core way
Exception handling middleware wraps the whole pipeline and returns a consistent JSON error shape instead of leaking stack traces; ValidateModelFilter is an MVC action filter (runs per-action, not per-request like middleware) that short-circuits with 400 before the controller body runs; LogActionFilter logs action name + arguments in/out. Good example to explain the middleware vs filter distinction interviewers love asking.
public async Task InvokeAsync(HttpContext context)
{
    try { await _next(context); }
    catch (Exception ex)
    {
        _logger.LogError(ex, "Unhandled exception for {Method} {Path}", context.Request.Method, context.Request.Path);
        context.Response.StatusCode = 500;
        await context.Response.WriteAsync(JsonSerializer.Serialize(new {
            status = 500, message = "An unexpected error occurred.", traceId = context.TraceIdentifier
        }));
    }
}
🗣️ Interview la eppadi sollanum"I separated cross-cutting concerns by scope. Exception handling is middleware — it wraps the entire HTTP pipeline, so no matter which controller throws, the client always gets a consistent JSON error with a trace ID instead of a stack trace. Model validation is an action filter, not middleware, because it only needs to run around a specific action's execution and needs access to ModelState, which is an MVC-pipeline concept, not a raw HTTP-pipeline one."
Talking point 4 — same API, two very different front-ends
WebApp (Razor Pages) uses a typed HttpClient registered via AddHttpClient<IEmployeeApiClient, EmployeeApiClient> — DI-managed, pooled connections, no manual new HttpClient(). WinForms uses the exact same typed-client pattern, wired through .NET's Generic Host (Host.CreateApplicationBuilder()) so a desktop app gets the same DI container ASP.NET Core apps use. This is a strong point for a "full stack" JD — proves the API design is UI-agnostic.
public async Task<IReadOnlyList<EmployeeApiModel>> GetEmployeesAsync(string token)
{
    using var request = new HttpRequestMessage(HttpMethod.Get, "api/employees");
    request.Headers.Authorization = new AuthenticationHeaderValue("Bearer", token);
    var response = await _httpClient.SendAsync(request);
    response.EnsureSuccessStatusCode();
    return await response.Content.ReadFromJsonAsync<List<EmployeeApiModel>>() ?? new();
}
🗣️ Interview la eppadi sollanum"To prove the API was properly decoupled from any one UI, I built two consumers against it — a Razor Pages web client and a WinForms desktop client — both using ASP.NET Core's typed HttpClient pattern registered through dependency injection, so connection pooling and lifetime are handled for me instead of creating raw HttpClient instances. Both attach the same JWT Bearer token; the only difference is the UI layer on top."
