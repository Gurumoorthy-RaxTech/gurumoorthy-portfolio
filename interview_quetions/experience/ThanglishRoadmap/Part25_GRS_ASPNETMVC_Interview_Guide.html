C ASP.NET MVC (Classic) — Must-Know Gap-Fill
1. MVC Request Lifecycle (Routing → Controller → Action → View)
Browser request varும் pothu — Routing (RouteConfig.cs, {controller}/{action}/{id} default pattern) URL ah correct Controller+Action ku map pannும். Controller Factory andha controller ah instantiate pannும். Action Invoker, matching Action method ah find pannі, parameters ah bind pannі (Model Binding) call pannும். Action, ActionResult return pannும் (mostly ViewResult). View Engine (Razor), .cshtml file ah locate pannі, model data vачі HTML render pannும், response ah browser ku anuppрроадு.
🗣️ Interview la eppadi sollanum"A request first hits routing, which maps the URL to a controller and action — usually via the default {controller}/{action}/{id} pattern in RouteConfig. The controller factory instantiates the controller, the action invoker binds parameters via model binding and calls the matching action method, the action returns an ActionResult — typically a ViewResult — and the Razor view engine locates the .cshtml file, renders it with the model data, and sends the HTML response back."
🔁 Follow-up
Q: Where is the default route defined? — RouteConfig.cs, registered in Application_Start in Global.asax via RouteConfig.RegisterRoutes(RouteTable.Routes).
2. Difference between ASP.NET MVC and ASP.NET Web Forms (If Asked — Older Comparison)
Web Forms — event-driven (Button_Click server events), ViewState (heavy hidden state in page), Page Controller pattern, less control over rendered HTML, tightly coupled UI+logic (code-behind .aspx.cs). MVC — separation of concerns (Model/View/Controller distinct), no ViewState, full control over HTML output, easier unit testing (controllers testable without a web server), better for REST-style/AJAX-heavy apps.
🗣️ Interview la eppadi sollanum"Web Forms is event-driven with heavy ViewState and tightly couples UI and logic through code-behind files. MVC cleanly separates Model, View, and Controller, has no ViewState overhead, gives full control over rendered HTML, and is much easier to unit test since controllers don't need a running web server to test."
🔁 Follow-up
Q: Have you worked with Web Forms? — Be honest — if you haven't, say so directly and pivot to "but I understand the comparison and why MVC's separation of concerns is the better fit for testable, REST-friendly apps."
3. ViewData vs ViewBag vs TempData (Classic MVC Data-Passing — Very Likely Asked)
ViewData — Dictionary<string, object>, Controller la ViewData["Name"] = "Guru", View la cast pannі use pannанум், current request kூllа mattум் valid. ViewBag — ViewData மேலே dynamic wrapper, ViewBag.Name = "Guru", cast panна vேண்டам், same request-scope thaан. TempData — Dictionary<string, object>, session-backed, oru redirect kூда survive pannும் (next request kூда data кொண்டு pоокும், apparam automatic ah clear aагும்) — RedirectToAction appuram data pass panна thaан use pannрроேн்.
🗣️ Interview la eppadi sollanum"ViewData is a dictionary that only lives for the current request and needs casting when read in the view. ViewBag is a dynamic wrapper over the same ViewData dictionary, so no casting is needed, but it's still scoped to the current request only. TempData is session-backed and survives exactly one redirect — I use it specifically when I need to pass data, like a success message, across a RedirectToAction call, since ViewData and ViewBag don't survive a redirect."
🔁 Follow-up
Q: Why does TempData survive a redirect but ViewData doesn't? — TempData is backed by Session under the hood, so it persists server-side across the redirect's round trip; ViewData/ViewBag are just plain in-memory objects tied to a single controller-to-view render, gone once that response is sent.
4. Partial View, Layout, and _ViewStart (Razor View Composition)
Layout (_Layout.cshtml) — master page madhiri, common header/footer/nav ellame vачு, @RenderBody() mூlама child view content ah inject pannрроадு. Partial View — reusable UI chunk (example — employee row template), @Html.Partial("_EmployeeRow", model) illainа @Html.RenderPartial() mூlама render pannрроேн். _ViewStart.cshtml — every view ku default Layout ah specify pannра file (ovvoru view la Layout = ... repeat pannадha vекkрроадு).
🗣️ Interview la eppadi sollanum"Layout, in _Layout.cshtml, acts like a master page with shared header, footer, and navigation, using @RenderBody() to inject the child view's content. A Partial View is a reusable chunk of UI — I render one with @Html.Partial passing in a model, for something like a repeatable table row template. _ViewStart.cshtml sets the default Layout for every view automatically, so I don't have to specify it individually in every .cshtml file."
🔁 Follow-up
Q: Difference between Html.Partial and Html.RenderPartial? — Html.Partial returns a string (used with @), RenderPartial writes directly to the response stream (used as a statement, slightly more memory-efficient for large partials) — functionally similar, Partial is more commonly used today.
5. Common Razor Syntax & HTML Helpers (@Html.TextBoxFor, BeginForm, etc.)
Razor la @ symbol vачі C# code embed pannрроேн் — @model Employee, @foreach, @if. HTML Helpers — strongly-typed form controls generate panна — @Html.TextBoxFor(m => m.FirstName), @Html.LabelFor(...), @Html.DropDownListFor(...), @Html.ValidationMessageFor(...). Form wrap panна — @using (Html.BeginForm("Create", "Employees", FormMethod.Post)) { ... }.
🗣️ Interview la eppadi sollanum"Razor uses the @ symbol to embed C# in HTML — @model declares the strongly-typed model, @foreach and @if work as normal C#. HTML Helpers generate strongly-typed form controls, like @Html.TextBoxFor(m => m.FirstName), which gives compile-time safety over hardcoded input names. 🔁 Follow-up
Q: Why prefer TextBoxFor over a plain <input> tag with a hardcoded name? — Compile-time safety — if the property is renamed, TextBoxFor breaks the build immediately; a hardcoded string name would silently stop binding at runtime with no warning.
6. Action Filters in Classic MVC ([Authorize], [HandleError], [OutputCache])
Classic MVC la built-in filters — [Authorize] (authentication/authorization enforce pannрроадு), [HandleError] (unhandled exceptions ah catch pannі custom error view kaатрроадு), [OutputCache] (action result ah cache pannрроадு, duration specify pannலам்), [ValidateAntiForgeryToken] (CSRF protection — POST actions ku definitely add pannрроேն்). Custom filter — ActionFilterAttribute inherit pannі, OnActionExecuting/OnActionExecuted override pannрроேн்.
🗣️ Interview la eppadi sollanum"Classic MVC has built-in filters — [Authorize] for auth, [HandleError] to catch unhandled exceptions and show a custom error view, [OutputCache] to cache an action's result, and [ValidateAntiForgeryToken] for CSRF protection, which I always add on POST actions alongside @Html.AntiForgeryToken() in the form. For custom logic, I'd inherit ActionFilterAttribute and override OnActionExecuting or OnActionExecuted."
🔁 Follow-up
Q: What pairs with [ValidateAntiForgeryToken] in the view? — @Html.AntiForgeryToken() inside the form — it emits a hidden token field that the attribute validates on submit, preventing cross-site request forgery.
7. Bundling and Minification (BundleConfig.cs)
Bundling/Minification na — multiple CSS/JS files ah, ஒரே ஒரு file ah combine pannі (bundling), whitespace/comments remove pannі compress pannі (minification), page load faster ah pannрроadhु — fewer HTTP requests + smaller file size. 🗣️ Interview la eppadi sollanum"Bundling combines multiple CSS or JS files into one to reduce HTTP requests, and minification strips whitespace and comments to shrink file size — both speed up page load. Q: How do you disable bundling temporarily for debugging? — BundleTable.EnableOptimizations = false, or debug="true" in Web.config, so scripts load individually and unminified for easier browser debugging.
8. Areas in MVC (Organizing Large Applications)
Areas na — periya MVC application ah, functional modules ah split pannі organize pannра mechanism — ovvoru Area ku thaanoda Controllers/Views/Models folder structure iருkkும் (example — Admin area, Billing area). 🗣️ Interview la eppadi sollanum"Areas let you organize a large MVC application into functional modules, each with its own Controllers, Views, and Models folder structure — like a separate Admin area or Billing area within the same application. Q: How does routing know which Area to route to? — Each Area registers its own AreaRegistration class with area-specific routes, and the area name is included in the route, typically as a namespace-qualified route registered in Global.asax's AreaRegistration.RegisterAllAreas().
9. Client-Side vs Server-Side Validation (jquery.validate.unobtrusive)
Server-side — Data Annotations ([Required], [StringLength]) DTO/ViewModel la vачி, controller la ModelState.IsValid check pannрроேн். Client-side — jquery.validate.js + jquery.validate.unobtrusive.js scripts include pannінா, same Data Annotations ah, automatic ah client-side JS validation ku convert pannும் (server ku pоокாma, browser la thaан immediate ah error kाатрроадு) — @Html.ValidationMessageFor() mூlама error message display pannрроேն்.
🗣️ Interview la eppadi sollanum"Server-side validation uses Data Annotations checked via ModelState.IsValid in the controller — the authoritative source of truth, since client-side checks can be bypassed. Client-side validation, through jquery.validate and jquery.validate.unobtrusive, automatically converts those same Data Annotations into JavaScript validation, giving instant feedback in the browser without a round trip, displayed via Html.ValidationMessageFor()."
🔁 Follow-up
Q: Why is server-side validation still mandatory even with client-side validation working? — Client-side JS can be disabled or bypassed entirely by a malicious or unusual client — server-side is the only validation you can actually trust to enforce data integrity.
10. ActionResult Types (View, Json, PartialView, Redirect, File)
Common ActionResult return types — ViewResult (return View()), JsonResult (return Json(data, JsonRequestBehavior.AllowGet) — AJAX calls ku), PartialViewResult (return PartialView()), RedirectToRouteResult (return RedirectToAction("Index")), ContentResult (plain text/string), FileResult (file download), HttpNotFoundResult/HttpStatusCodeResult.
🗣️ Interview la eppadi sollanum"Common ActionResult types are ViewResult for rendering a full view, JsonResult for AJAX responses — remembering JsonRequestBehavior.AllowGet is needed for GET requests returning JSON, PartialViewResult for a partial, RedirectToRouteResult from RedirectToAction, ContentResult for plain text, and FileResult for downloads."
🔁 Follow-up
Q: Why does JsonResult need JsonRequestBehavior.AllowGet explicitly? — A security default — returning JSON from a GET request was historically vulnerable to a JSON hijacking attack, so ASP.NET MVC blocks it by default unless you explicitly opt in.

2. DOM selection & manipulation basics (jQuery selectors)

Q: Difference between .attr() and .prop()? — .prop() reflects the current runtime DOM property state (like a checkbox's actual checked state); .attr() reflects the original HTML attribute value — for booleans like checked/disabled, .prop() is more reliable.
3. Event handling & event delegation (.on(), dynamically added elements)
$('#btn').click(function(){...}) — direct event binding, page-load pothu already иருkка elements ku mattum் work aагும். Event Delegation — $(document).on('click', '.dynamic-row', function(){...}) — parent element (already иருkка) ku listener attach pannі, later AJAX/JS mூlама dynamically add pannра elements кூда (.dynamic-row class) event catch pannும் — namba AJAX-heavy MVC pages la idhу essential.
🗣️ Interview la eppadi sollanum"Direct binding like $('#btn').click() only works on elements that already exist when the page loads. Event delegation, using $(document).on('click', '.dynamic-row', handler), attaches the listener to an already-existing parent and catches events bubbling up from matching elements — including ones added dynamically later via AJAX, which is essential in AJAX-heavy MVC pages where I'm re-rendering table rows without a full page reload."
🔁 Follow-up
Q: Why not just re-bind the click handler after every AJAX update? — It's fragile and easy to forget, and can cause duplicate handlers if you re-bind without unbinding first — delegation solves it once, permanently, regardless of how the DOM changes later.
4. var vs let vs const (Modern JS Scoping)
var — function-scoped, hoisted, redeclare panna мudiyум் (legacy, avoid pannuрроேн்). let — block-scoped, redeclare panna мудியாthu (same scope la), reassign panna мудியும். const — block-scoped, reassign panna мудியாтху (aana object/array oда contents kூда modify panna мудியும் — reference mattum் fix).
🗣️ Interview la eppadi sollanum"var is function-scoped and hoisted, which causes subtle bugs, so I avoid it in new code. let is block-scoped and reassignable. const is block-scoped and can't be reassigned — though if it holds an object or array, the contents can still be mutated, only the reference itself is locked."
🔁 Follow-up
Q: What's a classic var bug in a loop? — A var-declared loop variable inside a setTimeout callback in a for loop captures the same shared variable, so all callbacks see its final value — let fixes this since each iteration gets its own block-scoped binding.
5. == vs === in JavaScript (Loose vs Strict Equality)
== — loose equality, comparing munна type coercion pannும் ("5" == 5 — true). === — strict equality, type coercion pannадhu, type + value rendும் same aагanum் ("5" === 5 — false). Naan always === use pannрроேн் — predictable, unexpected coercion bugs avoid pannрроேн்.
🗣️ Interview la eppadi sollanum"== does type coercion before comparing, so '5' == 5 is true. === is strict — no coercion, both type and value must match, so '5' === 5 is false. I always use === in my code to avoid unexpected coercion bugs."
🔁 Follow-up
Q: Give a surprising == coercion example. — [] == false is true — an empty array coerces to an empty string, which coerces to 0, which equals false's coerced 0 — exactly the kind of surprise === avoids entirely.
