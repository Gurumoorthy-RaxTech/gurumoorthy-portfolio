En resume la — Bank Locker project (.NET Framework, 2023 வரை legacy system continue aana project) old world ku belong pannும், Zen Campus (ASP.NET Core v5/6/8) — modern unified .NET ku belong pannும், so rendு vரலாறும் நான் நேரடியா experience பண்ணிருக்கேன்.
🗣️ Interview la eppadi sollanum".NET Framework was the original Windows-only runtime, closed-source, tied to IIS. .NET Core was a ground-up rewrite for cross-platform, open-source, high-performance scenarios — and starting with .NET 5, Microsoft unified both lines into a single yearly-released '.NET' going forward, with even-numbered releases like 6, 8, and 10 being LTS. 2. Generic Host vs Web Host (unified hosting model)
Old days la ASP.NET Core rendு separate host types kொடுத்திருந்துச்சு — WebHost (web apps ku), and non-web apps (console/background worker) ku வேற எதுவும் standard வழி இல்லை, தனியா wire பண்ணனும். Microsoft realize பண்ணாங்க — "எதுக்கு rendு separate hosting model, ஒரே DI container + configuration + logging pipeline-a web app-ும் background worker-ும் share பண்ணலாமே" — அதுthaan Generic Host (Host.CreateDefaultBuilder). 🗣️ Interview la eppadi sollanum"Before Generic Host, ASP.NET Core had a Web Host specifically for HTTP apps, and a separate setup for non-web console or worker processes — you couldn't easily share DI, configuration, and logging between them. The Generic Host unified that — now a single hosting model backs both a web API and a background worker, so I get consistent dependency injection and configuration whether I'm serving HTTP requests or running a scheduled background task in the same process."
🔁 Follow-up Qs they might fire next

Q: Where have you used a background worker alongside your API? — I'd run report-generation or notification-fan-out logic as an IHostedService inside the same process, sharing the same DI-registered services as the API.

4. Minimal APIs (MapGet/MapPost style)
Minimal API na — full MVC controller class, [ApiController] attribute, action methods ellame venaam, direct ah Program.cs kulla (illa oru separate file la) route-a lambda-va define pannலாம் — app.MapGet("/students/{id}", ...) madhiri. Namba enga use pannுவோம்? — chinna, focused microservice (example: ஒரு simple "health-check" endpoint, ஒரு simple lookup service) ku, full MVC ceremony (controller class, routing attributes, filters setup) தேவையில்லாம quick ah spin up பண்ணனும் na, Minimal APIs perfect. Aana namma Zen Campus main services (Student, Billing, Attendance) madhiri — validation, filters, model binding, Swagger grouping, dozens of endpoints — complex ah irukும்bodhu, MVC controllers thaan namba pick pannுவோம், because organization and testability better ah irukும்.

{
    var result = await svc.ValidateAsync(req.Code, req.LockerId);

🗣️ Interview la eppadi sollanum"Minimal APIs let me define an HTTP endpoint directly with app.MapGet or app.MapPost, without a full controller class or attributes — much less ceremony. 5. MVC Controllers vs Minimal APIs vs gRPC Endpoints (three-way comparison)
Moonu options — MVC Controllers: full-featured, filters, model binding, [ApiController], best for medium-large services with lots of endpoints and cross-cutting concerns (namba Zen Campus mostly idhu thaan). Minimal APIs: lightweight, low ceremony, best for small focused services or a handful of endpoints. gRPC: completely different protocol — Protobuf (binary, compact) oda contract-first (.proto file la contract define pannுрோம்), HTTP/2 base pண்ணி multiplexed streaming support pண்ணும், REST/JSON vida romba fast — best for internal service-to-service calls where speed matters more than human-readability. Honest ah sொல்லனும் na — gRPC en resume la இல்லை, aana concept theriyும் — namba Ocelot-based sync HTTP microservices ku, high-frequency internal calls (example StudentService → AttendanceService) irundhா, athுக்கு gRPC நல்ல fit ஆகும் nu conceptual ah puriyும்.
🗣️ Interview la eppadi sollanum"MVC controllers give the most structure — filters, model binding, attribute-based configuration — good for larger services. Minimal APIs trade structure for less boilerplate on small, focused endpoints. gRPC is a different protocol entirely — contract-first with Protobuf, running over HTTP/2, much faster and more compact than JSON-over-REST, and it's the go-to for internal, high-performance service-to-service calls. I haven't shipped gRPC in production, but I understand where it fits — for our internal microservice calls between, say, Student and Attendance services, gRPC would be a natural fit if raw throughput became a bottleneck."
🔁 Follow-up Qs they might fire next
Q: Why is gRPC not typically exposed directly to a browser frontend? — Browsers don't fully support raw HTTP/2 trailers gRPC needs; you'd need grpc-web or a gateway translation layer, so it's mainly for backend-to-backend.
Q: Would gRPC replace Ocelot in your architecture? — Not replace — Ocelot would still front external HTTP/REST traffic; gRPC would be an internal optimization for specific high-frequency service-to-service calls behind the gateway.
6. IConfiguration vs IOptions<T> (raw config vs strongly-typed options)
IConfiguration na — raw key-value access, string key vecha value eduka (config["Jwt:Secret"] madhiri) — simple aana, typo pண்ணினா compile time la கண்டுபிடிக்க முடியாது, validation இல்லை. IOptions<T> na — namba oru C# class define பண்ணி (JwtSettings madhiri), config section அதுக்கு bind panniடலாம் (builder.Services.Configure<JwtSettings>(config.GetSection("Jwt"))), apparam service la IOptions<JwtSettings> inject பண்ணி strongly-typed ah access பண்ணலாம் — IntelliSense kూடa kidaikும், DataAnnotations vச்சு validate பண்ணலாம் (missing key na startup fail aagும், runtime la surprise varாது). 9. Endpoint Routing (UseRouting/UseEndpoints separation)
Old ASP.NET Core la routing na — MVC middleware itself thaan "எந்த controller/action இந்த URL ku match ஆகும்" nu decide பண்ணும், adhுக்கு munadி வேற எந்த middleware-ும் andha route info ah பார்க்க முடியாது. Modern endpoint routing la — app.UseRouting() first URL-a "match" பண்ணி எந்த endpoint nு decide பண்ணும் (but execute பண்ணாது), apparam authentication/authorization/CORS madhiri middleware-கள் andha matched endpoint metadata-வ பார்த்து decisions எடுக்கலாம் (example: "இந்த endpoint [Authorize] podиருக்கா" nு check பண்ணலாம்), apparam app.MapControllers()/UseEndpoints() actual ah execute பண்ணும். Idhu vachu authorization middleware, route data-va early ah access பண்ண முடியும் — idhுthaan decouple பண்ண main reason.
🗣️ Interview la eppadi sollanum"Endpoint routing splits route matching from route execution. UseRouting matches the incoming request against configured endpoints and attaches that match to the HttpContext, without running it. That lets middleware in between — like authorization — inspect the matched endpoint's metadata, such as an [Authorize] attribute, and make a decision before the endpoint actually executes via MapControllers or UseEndpoints. This decoupling is what lets the authorization middleware work generically across MVC, Minimal APIs, and Razor Pages."

Q: Can you access route values in custom middleware? — Yes, via HttpContext.GetEndpoint() after UseRouting has run, to read RouteEndpoint metadata.
10. DI & the 3 Service Lifetimes (Transient/Scoped/Singleton + the classic bug)
Transient — every time namba ask பண்ணும்bodhு புது object. Scoped — ஒரே HTTP request kulla ஒரே object (multiple places la inject பண்ணினாலும் same instance). Singleton — whole app life ku ஒரே object, first request la create aagும், apparam reuse aagும். Classic bug na — Singleton service kulla, Scoped service-a (example: DbContext) inject பண்ணினா — first request வந்து DbContext create aagி Singleton kulla "captured" aagிடும், apparam andha DbContext never dispose aagாது, next requests எல்லாம் ஒரே stale/disposed context-a use பண்ண முயற்சி பண்ணி exception throw பண்ணும். Idhு thaan "captive dependency" problem.
🗣️ Interview la eppadi sollanum"Transient creates a new instance every time it's resolved, Scoped shares one instance per HTTP request, and Singleton shares one instance for the app's entire lifetime. The classic mistake is injecting a Scoped service — like a DbContext — into a Singleton. The DbContext gets captured at Singleton's creation time and then either gets reused stale across every future request or throws an ObjectDisposedException. I always double-check lifetime alignment when registering a new service, especially anything wrapping EF Core's DbContext."

🔁 Follow-up Qs they might fire next
Q: How would you safely use a Scoped dependency from a Singleton? — Inject IServiceScopeFactory into the Singleton and create a new scope per operation, resolving the Scoped dependency inside that short-lived scope.
Q: What's the default lifetime for DbContext registered via AddDbContext? — Scoped — one instance per HTTP request, which is why it should never be injected directly into a Singleton.

🗣️ Interview la eppadi sollanum"Authentication answers 'who are you' — verifying identity, typically via credentials. 23. PUT vs POST (idempotency & semantics)
