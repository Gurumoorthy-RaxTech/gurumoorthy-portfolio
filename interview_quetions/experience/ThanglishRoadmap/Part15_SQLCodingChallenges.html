⚡ Part 15 Quick-Recall Cheat Sheet

  • 1. Nth Highest Salary → DENSE_RANK() handles ties correctly; OFFSET-FETCH is simplest but wrong with duplicates
  • 2. Find Duplicate Rows → GROUP BY all matching columns, HAVING COUNT(*) > 1
  • 3. Delete Duplicates, Keep One → ROW_NUMBER() OVER (PARTITION BY dup-cols ORDER BY Id) in a CTE, DELETE WHERE rn > 1
  • 4. Employees Earning > Manager → self-join Employees e JOIN Employees m ON e.ManagerId = m.EmployeeId
  • 5. Dept with Highest Avg Salary → GROUP BY DepartmentId, AVG(Salary), ORDER BY DESC, TOP 1 (or TOP 1 WITH TIES)
  • 6. Employees With No Manager → WHERE ManagerId IS NULL (self-referencing FK, nullable)
  • 7. Swap Column Values, No Temp Var → single UPDATE with CASE WHEN Id=@Id1 THEN val2 WHEN Id=@Id2 THEN val1 END
  • 8. 2nd Highest Salary, No TOP/OFFSET → SELECT MAX(Salary) WHERE Salary < (SELECT MAX(Salary) FROM Employees)
  • 9. Top N Students Per Class → RANK()/DENSE_RANK() OVER (PARTITION BY ClassId ORDER BY Marks DESC) in a CTE, filter rnk <= N
  • 10. Running Total Per Department → SUM(Salary) OVER (PARTITION BY DepartmentId ORDER BY HireDate ROWS UNBOUNDED PRECEDING)
  • 11. Consecutive Attendance Days → gaps-and-islands: DATEDIFF(day,0,Date) - ROW_NUMBER() OVER(...) = constant for a streak
  • 12. Pivot Marks Rows→Columns → PIVOT operator OR conditional aggregation SUM(CASE WHEN Subject='Math' THEN Marks END) — prefer conditional agg
  • 13. Managers With > N Reports → GROUP BY ManagerId HAVING COUNT(*) > N, join back to Employees for manager name
  • 14. Avg Days Between Two Dates, Grouped → DATEDIFF(day, AdmissionDate, PaymentDate), AVG(...), GROUP BY ClassId
  • 15. Rows in Table A Not in B → NOT EXISTS preferred over LEFT JOIN...IS NULL and especially over NOT IN (NULL trap)
15A Classic Query-Writing Problems
1. Find the Nth Highest Salary (Difficulty: Medium · Window Functions / Subquery)
Problem: Given an Employees table with columns EmployeeId, Name, Salary, write a query to find the Nth highest salary (N passed as a parameter, e.g. 3rd highest). Handle the case where multiple employees share the same salary.
Idhu interview la mostly kekura first SQL question thaan. Naaka moonu approach kaatanum — OFFSET-FETCH use panradhu simple aana, duplicate salary irundha wrong answer tharum (rows count pannum, distinct salary count pannadhu). DENSE_RANK() thaan correct approach edhukku na adhu "rank" ah salary value mela base pannuthu, row mela illa — so 2 pேrukku same top salary irundhaalum, moonாவadhu unique salary correct ah varum. Old SQL Server (2005 ku munnadi, window function illama) versions ku correlated subquery use pannுவோம் — adhu konjam slow aana works pannும்.
-- Sample schema
-- CREATE TABLE Employees (EmployeeId INT PRIMARY KEY, Name VARCHAR(100), Salary DECIMAL(12,2));

DECLARE @N INT = 3;

-- (a) OFFSET-FETCH approach (simple, but WRONG if duplicate salaries exist above N)
SELECT DISTINCT Salary
FROM Employees
ORDER BY Salary DESC
OFFSET @N - 1 ROWS FETCH NEXT 1 ROW ONLY;

-- (b) DENSE_RANK() approach — BEST, correctly handles ties
;WITH RankedSalaries AS (
    SELECT Salary,
           DENSE_RANK() OVER (ORDER BY Salary DESC) AS SalaryRank
    FROM Employees
)
SELECT DISTINCT Salary
FROM RankedSalaries
WHERE SalaryRank = @N;

-- (c) Correlated subquery approach (works even on very old SQL Server, no window functions needed)
SELECT TOP 1 Salary
FROM (
    SELECT DISTINCT TOP (@N) Salary
    FROM Employees
    ORDER BY Salary DESC
) AS Top_N
ORDER BY Salary ASC;
🗣️ Interview la eppadi walk-through pannanumI'd lead with the DENSE_RANK() version because it's the one that's actually correct in production — with OFFSET-FETCH, if two people tie for 2nd highest, my "3rd highest" would silently be wrong since it counts rows, not distinct salary ranks. DENSE_RANK() ranks by value so ties share a rank and the next distinct salary correctly gets N+1. I'd mention the correlated subquery only as a fallback for legacy SQL Server versions without window function support. On an indexed Salary column this is a single index scan, so performance is fine even on large tables.
🔁 Follow-up variations they might ask
Q: What if you need Nth highest salary per department? — Add PARTITION BY DepartmentId to the DENSE_RANK() window: DENSE_RANK() OVER (PARTITION BY DepartmentId ORDER BY Salary DESC).
Q: Why not just use RANK() instead of DENSE_RANK()? — RANK() skips numbers after a tie (1,2,2,4), so "3rd highest" would return nothing if there's a tie at rank 2 — DENSE_RANK() keeps ranks consecutive (1,2,2,3), which matches how people actually mean "3rd highest."
Q: How would you get the Nth highest without DISTINCT skewing results if two employees genuinely have different Ids but same salary? — DENSE_RANK() already dedupes by value, so DISTINCT on the final SELECT is just a safety net; the ranking itself is already correct per-value.
2. Find Duplicate Rows in a Table (Difficulty: Easy · GROUP BY / HAVING)
Problem: A Students table has accidental duplicate entries — same Name and DOB inserted more than once due to a form double-submit bug. Write a query to list all duplicate groups along with how many times each appears.
Idhu straightforward — duplicate nu artham enna columns match aaganum nu decide pannanum first (business rule: Name + DOB same na duplicate). Apparam andha columns mela GROUP BY panni, HAVING COUNT(*) > 1 vachu filter pannanum. WHERE clause la filter panna mudiyaadhu edhukku na COUNT() oru aggregate — adhukku HAVING thaan use pannanum, group create aana APPRAM thaan filter aagum.
-- Sample schema
-- CREATE TABLE Students (StudentId INT PRIMARY KEY, Name VARCHAR(100), DOB DATE, ClassId INT);

SELECT Name, DOB, COUNT(*) AS DuplicateCount
FROM Students
GROUP BY Name, DOB
HAVING COUNT(*) > 1
ORDER BY DuplicateCount DESC;

-- If you also want to see the actual duplicate row IDs:
SELECT s.*
FROM Students s
INNER JOIN (
    SELECT Name, DOB
    FROM Students
    GROUP BY Name, DOB
    HAVING COUNT(*) > 1
) d ON s.Name = d.Name AND s.DOB = d.DOB
ORDER BY s.Name, s.DOB;
🗣️ Interview la eppadi walk-through pannanumThe key thing I explain is why HAVING and not WHERE — WHERE filters rows before grouping, but COUNT(*) doesn't exist until after grouping, so it has to be HAVING. I also always confirm with the interviewer what "duplicate" means for that table — is it the whole row, or a business key like Name+DOB — because that changes the GROUP BY column list entirely. If this table is large, I'd want an index on (Name, DOB) since that's exactly the columns being grouped and filtered on.
🔁 Follow-up variations they might ask
Q: How do you find duplicates across the ENTIRE row, not just two columns? — GROUP BY every column in the table (or CHECKSUM/HASHBYTES of the whole row for wide tables), HAVING COUNT(*) > 1.
Q: How would you get just the extra copies (not the first occurrence)? — That's problem 3 — use ROW_NUMBER() to number each duplicate group and treat rn > 1 as the "extra" rows.
3. Delete Duplicate Rows, Keeping Only One Copy (Difficulty: Medium · ROW_NUMBER / CTE)
Problem: Same Students table with duplicate (Name, DOB) rows from problem 2 — but now actually DELETE the extra copies, keeping exactly one row per duplicate group (keep the lowest StudentId).
Idhu real production la periya usage irukka pattern — data cleanup script eludhumbodhu exactly idhu thaan pannuvom. Trick enna na — direct ah DELETE panna mudiyaadhu "duplicate ah irukka rows" nu identify pannாma, edhukku na SQL rows ah individually address panna mudiyாthu unique key illama. So ROW_NUMBER() OVER (PARTITION BY Name, DOB ORDER BY StudentId) vachu ovvoru duplicate group kulla 1,2,3 nu number pannuvom — apparam CTE ah "updatable/deletable" ah treat panni, DELETE FROM CTE WHERE rn > 1 pannினா, rn=1 (first/lowest StudentId) mattum survive aagum.
-- Sample schema
-- CREATE TABLE Students (StudentId INT PRIMARY KEY, Name VARCHAR(100), DOB DATE, ClassId INT);

;WITH DupCTE AS (
    SELECT
        StudentId,
        Name,
        DOB,
        ROW_NUMBER() OVER (
            PARTITION BY Name, DOB
            ORDER BY StudentId ASC   -- keeps the row with the lowest StudentId
        ) AS rn
    FROM Students
)
DELETE FROM DupCTE
WHERE rn > 1;

-- Always test first with a SELECT before running the DELETE:
-- ;WITH DupCTE AS (
--     SELECT StudentId, Name, DOB,
--            ROW_NUMBER() OVER (PARTITION BY Name, DOB ORDER BY StudentId ASC) AS rn
--     FROM Students
-- )
-- SELECT * FROM DupCTE WHERE rn > 1;
🗣️ Interview la eppadi walk-through pannanumThe CTE here isn't just for readability — it's what makes the DELETE possible at all, since you can't put ROW_NUMBER() directly in a DELETE statement's WHERE clause. I always run the SELECT version of the CTE first to eyeball which rows will actually get deleted before flipping it to a DELETE — that's a habit from real incidents where a wrong PARTITION BY column silently deleted the wrong rows. In production I'd also wrap this in a transaction and take a backup of the affected rows into a temp table first.
🔁 Follow-up variations they might ask
Q: What if you want to keep the MOST RECENT duplicate instead of the oldest? — Change ORDER BY StudentId ASC to ORDER BY StudentId DESC (or order by a CreatedDate column DESC) inside the ROW_NUMBER window.
Q: Can you do this without a CTE? — Yes, with a derived table subquery in the DELETE's FROM/JOIN, but CTE is cleaner and more readable — most teams prefer it.
Q: How would you prevent this duplicate problem from happening again? — Add a unique constraint/index on (Name, DOB) if business rules allow, or fix the double-submit bug at the application layer with idempotency keys.
4. Employees Earning More Than Their Manager (Difficulty: Easy · Self-Join)
Problem: An Employees table has a self-referencing ManagerId column pointing to another row's EmployeeId. Write a query to list employees who earn more than their own manager.
Idhu classic self-join problem — same table ah rendu vera roles la treat pannanum: oru copy "employee" ah, vera copy "manager" ah. So Employees e nu oru alias, Employees m nu vera alias vachu, e.ManagerId = m.EmployeeId mூlama join pannுவோம் — idhu employee-oda manager row ah kandu pudikkும். Apparam simple ah WHERE e.Salary > m.Salary podunga, done.
-- Sample schema
-- CREATE TABLE Employees (EmployeeId INT PRIMARY KEY, Name VARCHAR(100), Salary DECIMAL(12,2), ManagerId INT NULL);

SELECT
    e.EmployeeId,
    e.Name       AS EmployeeName,
    e.Salary     AS EmployeeSalary,
    m.Name       AS ManagerName,
    m.Salary     AS ManagerSalary
FROM Employees e
INNER JOIN Employees m ON e.ManagerId = m.EmployeeId
WHERE e.Salary > m.Salary
ORDER BY (e.Salary - m.Salary) DESC;
🗣️ Interview la eppadi walk-through pannanumThis is the standard self-join pattern — same table, two aliases, one representing the "child" row and one the "parent" row it references. I use INNER JOIN here deliberately because employees with no manager (ManagerId is NULL) shouldn't show up in this comparison at all — an INNER JOIN naturally excludes them. If I wanted to also flag employees with no manager as "not applicable" rather than silently dropping them, I'd switch to a LEFT JOIN and add a CASE expression.
🔁 Follow-up variations they might ask
Q: What about the CEO who has no manager? — INNER JOIN automatically excludes them since ManagerId is NULL and there's nothing to join to — that's the correct behavior here.
Q: How would you find the salary gap for every manager-report pair, not just where employee earns more? — Just drop the WHERE clause, or keep the join and add a computed column for the gap without filtering.
Q: Can you do this without a self-join, using a subquery instead? — Yes: WHERE Salary > (SELECT Salary FROM Employees m WHERE m.EmployeeId = e.ManagerId), but that's a correlated subquery per row — the self-join is generally more efficient since the optimizer can use a hash or merge join.
5. Department with the Highest Average Salary (Difficulty: Easy · GROUP BY / Aggregate)
Problem: Given Employees(EmployeeId, Name, Salary, DepartmentId) and Departments(DepartmentId, DepartmentName), find the department with the highest average salary. Also show how to handle a tie between two departments.
Standard aggregate-and-sort problem. DepartmentId mela GROUP BY panni, AVG(Salary) calculate pannitu, DESC order la sort panni TOP 1 eduthுkalam. Aana "TOP 1" na — rendu department exact ah same average salary irundha, ஏதோ ஒண்ணு mattum arbitrary ah return aagum, adhu correct illa business-wise. So ties handle panna TOP 1 WITH TIES use pannுவோம் — adhu ORDER BY column mela same value irukka ellame return pannும்.
-- Sample schema
-- CREATE TABLE Departments (DepartmentId INT PRIMARY KEY, DepartmentName VARCHAR(100));
-- CREATE TABLE Employees (EmployeeId INT PRIMARY KEY, Name VARCHAR(100), Salary DECIMAL(12,2), DepartmentId INT);

-- Basic version — single winner
SELECT TOP 1
    d.DepartmentName,
    AVG(e.Salary) AS AvgSalary
FROM Employees e
INNER JOIN Departments d ON e.DepartmentId = d.DepartmentId
GROUP BY d.DepartmentId, d.DepartmentName
ORDER BY AvgSalary DESC;

-- Tie-safe version — returns ALL departments tied for the highest average
SELECT TOP 1 WITH TIES
    d.DepartmentName,
    AVG(e.Salary) AS AvgSalary
FROM Employees e
INNER JOIN Departments d ON e.DepartmentId = d.DepartmentId
GROUP BY d.DepartmentId, d.DepartmentName
ORDER BY AvgSalary DESC;
🗣️ Interview la eppadi walk-through pannanumI'd start with the plain TOP 1 version since that's what's usually asked, but I always mention the tie problem proactively — it shows I think about correctness, not just getting a query to run. TOP 1 WITH TIES uses the same ORDER BY to decide what counts as "tied," so if two departments land on exactly the same rounded average, both come back instead of one being silently dropped. In a real ERP report I'd also round AVG(Salary) with ROUND() since raw decimal averages can differ by fractions of a rupee and never actually "tie" in the strict sense.
🔁 Follow-up variations they might ask
Q: How would you find the top 3 departments by average salary? — Replace TOP 1 with TOP 3, or use DENSE_RANK() OVER (ORDER BY AVG(Salary) DESC) in a CTE and filter rnk <= 3 for tie-safe top-3.
Q: What if a department has no employees at all? — INNER JOIN excludes it automatically; switch to LEFT JOIN with COALESCE(AVG(e.Salary), 0) if you want empty departments to show up with a zero/null average.
6. Employees With No Manager Assigned (Difficulty: Easy · NULL Handling)
Problem: From the same self-referencing Employees table, list all employees who don't have a manager assigned (e.g. the CEO, or newly-onboarded staff whose manager hasn't been set yet in the ERP).
Idhu naaka over-think pannaadha simple question — aana palapper self-join la over-complicate pannिடுவாங்க. ManagerId nullable FK on the same table na, straight ah WHERE ManagerId IS NULL podі pothum, join enna venum. Join venum-nu varradhu enna scenario na — vera table la (example a separate ManagerAssignment mapping table) irundha mattum, adhula andha employee ku entry illama irundha "no manager" nu treat pannanum, appo LEFT JOIN + IS NULL pattern use pannுவோம்.
-- Sample schema
-- CREATE TABLE Employees (EmployeeId INT PRIMARY KEY, Name VARCHAR(100), Salary DECIMAL(12,2), ManagerId INT NULL);

-- Simple case: ManagerId is a nullable column on the same table
SELECT EmployeeId, Name, Salary
FROM Employees
WHERE ManagerId IS NULL;

-- If manager assignment lived in a separate mapping table instead:
-- CREATE TABLE ManagerAssignments (EmployeeId INT, ManagerId INT);
SELECT e.EmployeeId, e.Name
FROM Employees e
LEFT JOIN ManagerAssignments ma ON e.EmployeeId = ma.EmployeeId
WHERE ma.ManagerId IS NULL;
🗣️ Interview la eppadi walk-through pannanumI always ask a clarifying question first — is ManagerId a nullable column directly on Employees, or is management tracked through a separate junction table? That decides whether it's a one-line WHERE clause or a LEFT JOIN pattern. For the common case, WHERE ManagerId IS NULL is all you need, and a plain equality check like ManagerId = NULL never works in SQL — NULL comparisons always need IS NULL / IS NOT NULL because NULL isn't a value, it's the absence of one.
🔁 Follow-up variations they might ask
Q: Why doesn't WHERE ManagerId = NULL work? — Because NULL isn't comparable with =; any comparison against NULL evaluates to UNKNOWN, not TRUE, so those rows get filtered out — you must use IS NULL.
Q: How would you count how many employees report to no one, per department? — Add GROUP BY DepartmentId with COUNT(*) after filtering WHERE ManagerId IS NULL.
7. Swap Values of a Column Between Two Specific Rows (Difficulty: Medium · Single UPDATE / CASE)
Problem: In a Classes table with columns ClassId, RoomNumber, swap the RoomNumber values of two specific classes (Id = 5 and Id = 9) — without using a temporary variable or a temp table, in a single statement. (This is the SQL cousin of the classic "swap two variables without a temp variable" puzzle.)
Idhu C# side la irukka "swap without temp variable" question oda SQL version thaan — namba Part 14 la XOR/addition-subtraction trick use pannினோம், idhu la SQL specific trick use pannுவோம். Naive ah pannினா — first UPDATE Id=5's RoomNumber to Id=9's value, apparam UPDATE Id=9's RoomNumber to Id=5's value — aana rendாவadhu UPDATE run aagும் bodhu, Id=5 already overwrite aayiduchu, so Id=9 wrong value edுthுkkும். Fix enna na — rendும் updates ஒரே single UPDATE statement kulla, CASE expression vachu pannுவோம். SQL Server oru UPDATE statement kulla "old" values ah snapshot eduthுkkитు, appuram apply pannும், so intermediate overwrite problem varadhu.
-- Sample schema
-- CREATE TABLE Classes (ClassId INT PRIMARY KEY, ClassName VARCHAR(50), RoomNumber INT);

DECLARE @Id1 INT = 5, @Id2 INT = 9;

UPDATE Classes
SET RoomNumber = CASE ClassId
                    WHEN @Id1 THEN (SELECT RoomNumber FROM Classes WHERE ClassId = @Id2)
                    WHEN @Id2 THEN (SELECT RoomNumber FROM Classes WHERE ClassId = @Id1)
                  END
WHERE ClassId IN (@Id1, @Id2);
🗣️ Interview la eppadi walk-through pannanumThe reason this works in one statement is that SQL Server evaluates the SET expression for every row against the pre-update snapshot of the table before writing anything back — it's not sequential row-by-row like procedural code, so the subquery inside CASE still sees the original, un-swapped values even for the second row it processes. If I did this as two separate UPDATE statements instead, the first one would overwrite Id 5's value before I read it for Id 9, corrupting the swap — that's the exact bug I'd flag if I saw someone write it as two statements. This is a nice callback to the classic swap-without-temp-variable problem, just solved with SQL's set-based semantics instead of arithmetic tricks.
🔁 Follow-up variations they might ask
Q: Would this work correctly inside a transaction with concurrent writers? — Wrap it in a transaction with an appropriate isolation level (or use ROWLOCK/UPDLOCK hints) to prevent another session from changing RoomNumber between the read and write of this statement.
Q: Can you swap values across TWO DIFFERENT tables in one statement? — Not directly with a single UPDATE (UPDATE only targets one table) — you'd need a transaction wrapping two UPDATE statements, each reading the other table's original value before either write happens.
Q: What if you need to swap N rows in a rotation (row1→row2→row3→row1)? — Same CASE-based single UPDATE pattern generalizes — each WHEN branch pulls the "previous" row's original value, since the whole SET is evaluated against the pre-update state.
8. Second Highest Salary WITHOUT Using TOP/LIMIT/OFFSET-FETCH (Difficulty: Easy · Subquery Trick)
Problem: Find the second-highest salary from the Employees table using only a plain subquery — no TOP, no OFFSET-FETCH, no window functions. (A very old-school interview variant, still asked to test basic subquery thinking.)
Idhu oru classic "old school" trick question — idea enna na, "2nd highest" nu artham "the maximum salary that is still less than the overall maximum salary." So inner subquery la MAX(Salary) eduthுkkitு, outer query la adha vida less ah irukka salaries la mele MAX() edுthाल், adhu automatic ah 2nd highest tharும். Idhu neat aana oru limitation irukku — idha "3rd highest" ku extend panna try pannா, query nested ah nested ah pogum (MAX of MAX of MAX...), so real-world la Nth highest ku DENSE_RANK() (problem 1) thaan preferred, idhu 2nd highest interview trivia ku mattum nalla.
-- Sample schema
-- CREATE TABLE Employees (EmployeeId INT PRIMARY KEY, Name VARCHAR(100), Salary DECIMAL(12,2));

SELECT MAX(Salary) AS SecondHighestSalary
FROM Employees
WHERE Salary < (SELECT MAX(Salary) FROM Employees);

-- Extending to 3rd highest this way gets ugly fast (nested MAX-of-MAX):
SELECT MAX(Salary) AS ThirdHighestSalary
FROM Employees
WHERE Salary < (
    SELECT MAX(Salary) FROM Employees
    WHERE Salary < (SELECT MAX(Salary) FROM Employees)
);
🗣️ Interview la eppadi walk-through pannanumThe trick is reframing "second highest" as "the highest value among everything that's not the highest" — that single reframe is what makes the subquery-only solution possible. I'd answer this confidently since it's a well-known pattern, but I'd immediately add the honest caveat: this doesn't scale to "Nth highest" cleanly, you end up nesting a nested subquery per level, which is unreadable and slow beyond 3 or 4 levels. In real work I'd reach for DENSE_RANK() from problem 1 instead — I'm only using this trick here because the question specifically rules out TOP/OFFSET/window functions.
🔁 Follow-up variations they might ask
Q: What if there's only one distinct salary in the whole table? — This query correctly returns NULL, since there's no salary less than the max — that's actually the right answer, "there is no second-highest."
Q: Why do you prefer DENSE_RANK() in production over this trick? — This nested-MAX approach doesn't generalize past a couple of levels and is harder to read; DENSE_RANK() handles any N and ties correctly in one clean expression.

9. Top N Students Per Class by Marks (Difficulty: Medium · RANK/DENSE_RANK, Partitioned)
Problem: Given ExamMarks(StudentId, ClassId, Subject, TotalMarks), write a query to find the top 2 students per class based on TotalMarks.
Idhu problem 1 oda "partitioned" version — appuram namba WHOLE table mela rank pannினோம், idhu la ovvoru ClassId kulla separately rank pannanum. Adhukku thaan window function la PARTITION BY ClassId add pannுவோம் — idhu "ovvoru class ku separate ah oru mini-ranking start pannு" nu SQL Server ku sollுறாது. RANK() use panna ties irundha same rank kudukkum aana adhukku appuram gap varum; DENSE_RANK() na gap illama continuous ah irukkும் — "top 2" nு business rule na, DENSE_RANK() thaan usual ah correct choice, edhukku na 2 pேrு tie aana rendு perைyும் top-2 ah consider pannanum.
-- Sample schema
-- CREATE TABLE ExamMarks (StudentId INT, ClassId INT, Subject VARCHAR(50), TotalMarks INT);

;WITH RankedStudents AS (
    SELECT
        StudentId,
        ClassId,
        TotalMarks,
        DENSE_RANK() OVER (PARTITION BY ClassId ORDER BY TotalMarks DESC) AS rnk
    FROM ExamMarks
)
SELECT StudentId, ClassId, TotalMarks, rnk
FROM RankedStudents
WHERE rnk <= 2
ORDER BY ClassId, rnk;
🗣️ Interview la eppadi walk-through pannanumThe core idea to explain is that PARTITION BY resets the ranking counter for every group — so ClassId 1's top student gets rank 1, and ClassId 2's top student also independently gets rank 1, they don't share a global rank. I use DENSE_RANK() over RANK() for "top N" style filters because if two students tie for rank 1, RANK() jumps the next student straight to rank 3, which would silently exclude someone who genuinely earned a top-2 spot. I've written this exact pattern for the Zen Campus exam module — showing rank cards for top performers per class.
🔁 Follow-up variations they might ask
Q: How would you rank students across all subjects combined instead of by total? — Aggregate SUM(Marks) per StudentId, ClassId first in a subquery/CTE, then apply the same DENSE_RANK() pattern on that aggregated result.
Q: What's the difference if you used ROW_NUMBER() instead here? — ROW_NUMBER() never ties — it arbitrarily breaks ties by the ORDER BY's tiebreak order, so exactly 2 rows per class always come back, even if 3 students tied for 2nd place; use it only when you want a strict fixed count regardless of ties.
Q: How do you get the top student per class only, as a single flat result (not a CTE)? — Same query with rnk = 1 instead of <= 2, or use CROSS APPLY with TOP 1 per ClassId if the optimizer needs a nudge for large tables.
10. Running Total / Cumulative Sum Per Department (Difficulty: Medium · Window Aggregate)
Problem: For a payroll report, show each employee's salary along with a running (cumulative) total of salaries within their department, ordered by hire date — so HR can see how the department's payroll cost accumulated over time.
Idhu "running total" pattern — old school ah pannunа self-join or correlated subquery vachu O(n²) ah eluduvom, aana window function vachu ஒரே pass la clean ah pannலாம். SUM(Salary) OVER (PARTITION BY DepartmentId ORDER BY HireDate) — PARTITION BY dept-wise separate running total start pannும், ORDER BY HireDate "eந்த order la accumulate pannணும்" nு sollும். Default frame ROWS UNBOUNDED PRECEDING nu explicit ah podலாம் (illa na SQL Server automatically andha default vைchுkkும் oru ORDER BY irukும் bodhu).
-- Sample schema
-- CREATE TABLE Employees (EmployeeId INT PRIMARY KEY, Name VARCHAR(100), Salary DECIMAL(12,2), DepartmentId INT, HireDate DATE);

SELECT
    EmployeeId,
    Name,
    DepartmentId,
    HireDate,
    Salary,
    SUM(Salary) OVER (
        PARTITION BY DepartmentId
        ORDER BY HireDate
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS RunningDeptTotal
FROM Employees
ORDER BY DepartmentId, HireDate;
🗣️ Interview la eppadi walk-through pannanumI explain this as "each department gets its own independent running total, accumulating in hire-date order." The explicit frame clause — ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW — means "sum everything from the start of this partition up to and including this row," which is exactly a running total; I always write it explicitly rather than relying on the implicit default, because it makes the intent obvious to anyone reading the query later. Before window functions existed, this needed a self-join or correlated subquery per row which was O(n²) — the windowed version is a single sorted pass, far cheaper on a large payroll table.
🔁 Follow-up variations they might ask
Q: How would you get a running total across the WHOLE company, not per department? — Drop the PARTITION BY clause entirely — SUM(Salary) OVER (ORDER BY HireDate ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) treats the whole table as one partition.
Q: How would you show a moving average of the last 3 hires' salaries instead of a full running total? — Change the frame to ROWS BETWEEN 2 PRECEDING AND CURRENT ROW and use AVG() instead of SUM().
Q: What happens if two employees in the same department have the exact same HireDate? — The running total is still correct in total, but the intermediate row-by-row values for those tied rows can appear in either order — add a tiebreaker like EmployeeId to ORDER BY if row-level determinism matters.
11. Find Consecutive Login/Attendance Days for a Student (Difficulty: Hard · Gaps-and-Islands)
Problem: Given an Attendance(StudentId, AttendanceDate) table (one row per day the student was marked present), find the longest streak of consecutive present-days for each student.
Idhu SQL interviews la mostly kekura "hard" category question — "gaps and islands" nu oru famous technique. Idea enna na: ஒரு student consecutive ah present iருந்தா, andha dates oda "row number (in date order)" and "the date itself" oda difference ஒரே constant ah irukும். Example: dates 1,2,3 present na, ROW_NUMBER 1,2,3 — date minus row_number ella rowsукum same value tharும். Aana gap iருந்தா (example date 5 miss aachu, 6ல் resume aachु), andha difference maariடும் — so andha "difference" value ah oru "island group id" ஆ treat pannலாம். Apparam andha group mela GROUP BY panni COUNT(*) pannினா, streak length kidaikும்.
-- Sample schema
-- CREATE TABLE Attendance (StudentId INT, AttendanceDate DATE);

;WITH Islands AS (
    SELECT
        StudentId,
        AttendanceDate,
        DATEDIFF(DAY, 0, AttendanceDate)
          - ROW_NUMBER() OVER (PARTITION BY StudentId ORDER BY AttendanceDate) AS IslandGroup
    FROM Attendance
),
Streaks AS (
    SELECT
        StudentId,
        MIN(AttendanceDate) AS StreakStart,
        MAX(AttendanceDate) AS StreakEnd,
        COUNT(*) AS StreakLength
    FROM Islands
    GROUP BY StudentId, IslandGroup
)
SELECT StudentId, StreakStart, StreakEnd, StreakLength
FROM Streaks
WHERE StreakLength = (
    SELECT MAX(s2.StreakLength) FROM Streaks s2 WHERE s2.StudentId = Streaks.StudentId
)
ORDER BY StudentId;
🗣️ Interview la eppadi walk-through pannanumThis is the "gaps and islands" trick, and I'd walk through the intuition slowly since it's genuinely non-obvious the first time you see it — for a run of consecutive calendar dates, the row number (which increases by exactly 1 per row) and the date itself (which also increases by exactly 1 per day) drift apart by a constant amount, so subtracting them gives the same "group id" for every date in that unbroken run; the moment there's a gap, the date jumps ahead of the row number and the group id changes. Once you have that group id, it's just a GROUP BY and COUNT to get streak lengths, then filter for the max per student. I'd also mention DATEDIFF(DAY, 0, date) is just converting the date to an integer day-offset so the subtraction is pure arithmetic — this is exactly the kind of pattern I'd use for a "longest attendance streak" badge feature in the Zen Campus student app.
🔁 Follow-up variations they might ask
Q: How would you find ALL streaks of 5+ days, not just the longest one per student? — Same CTE, but filter the Streaks CTE with WHERE StreakLength >= 5 instead of matching MAX per student.
Q: Does this work if attendance is tracked with timestamps instead of dates? — Cast to DATE first (CAST(AttendanceTimestamp AS DATE)) and dedupe multiple same-day entries before applying the island trick, since the technique assumes one row per calendar day.
Q: How would this apply to the Bank Locker OTP audit-trail logs to find continuous access windows? — Same pattern but on a finer time grain — instead of DAY, use DATEDIFF(MINUTE/SECOND, 0, LogTimestamp) minus ROW_NUMBER() to find continuous blocks of activity with no gap beyond a threshold.
12. Pivot Exam Marks from Rows to Columns (Difficulty: Medium · PIVOT / Conditional Aggregation)
Problem: ExamMarks(StudentId, Subject, Marks) has one row per student per subject. Produce a report with one row per student and separate columns for Math, Science, and English marks.
Idhu report-generation ku romba common ah kekura pattern — "rows ah columns ah maathanum". SQL Server la built-in PIVOT operator irukku, aana syntax konjam awkward-ah irukும், and subjects hardcode pannणум் structure la. Palapper devs — including naan — CASE vecha conditional aggregation preferred pannுவோம்: SUM(CASE WHEN Subject='Math' THEN Marks END) madhiri ovvoru subject-ku column create pannலாம். Idhu PIVOT operator vida readable ஆ irukும், dynamic ah subjects add pannும் bodhு adjust panna easy ஆ irukும் (dynamic SQL vachு column list build panna mudியும்), so real projects la idhu thaan preferred.
-- Sample schema
-- CREATE TABLE ExamMarks (StudentId INT, Subject VARCHAR(50), Marks INT);

-- (a) Using the PIVOT operator
SELECT StudentId, [Math], [Science], [English]
FROM (
    SELECT StudentId, Subject, Marks
    FROM ExamMarks
) AS SourceTable
PIVOT (
    SUM(Marks)
    FOR Subject IN ([Math], [Science], [English])
) AS PivotTable;

-- (b) Conditional aggregation — usually preferred: more readable, easier to extend
SELECT
    StudentId,
    SUM(CASE WHEN Subject = 'Math'    THEN Marks END) AS Math,
    SUM(CASE WHEN Subject = 'Science' THEN Marks END) AS Science,
    SUM(CASE WHEN Subject = 'English' THEN Marks END) AS English
FROM ExamMarks
GROUP BY StudentId;
🗣️ Interview la eppadi walk-through pannanumI'd show both, but say upfront that I default to conditional aggregation in real projects. PIVOT's syntax has a rigid shape — you have to know all the column values (subject names) at query-write time and hardcode them in the FOR...IN clause, and it only supports a single aggregate function per pivot. Conditional aggregation with SUM(CASE WHEN...) does the exact same thing but reads like plain SQL anyone can follow, and it's trivial to add a fourth subject or combine it with other aggregates like AVG or COUNT in the same query. If the subject list itself is dynamic and unknown ahead of time, I'd build the column list with dynamic SQL rather than fight PIVOT's static syntax.
🔁 Follow-up variations they might ask
Q: What if a student has no marks recorded for a subject? — Both approaches return NULL for that student's column automatically since there's no matching row to aggregate — wrap in ISNULL/COALESCE if you want 0 instead.
Q: How would you handle an unknown/dynamic list of subjects? — Build the SELECT column list with dynamic SQL (STRING_AGG over DISTINCT Subject values to construct the CASE expressions or PIVOT IN list), then EXEC the generated string.
Q: Can you also get each student's total and average across subjects in the same query? — Yes with conditional aggregation — just add SUM(Marks) AS Total and AVG(Marks) AS AvgMarks as extra columns in the same GROUP BY StudentId query.
13. Managers With More Than N Direct Reports (Difficulty: Easy · Self-Reference GROUP BY / HAVING)
Problem: Using the self-referencing Employees(EmployeeId, Name, ManagerId) table, find all managers who have more than 3 direct reports, along with the manager's name and report count.
Idhu problem 4 (self-join) oda "count" version thaan. ManagerId mela GROUP BY pannினா, ovvoru manager ku evlo employees report pannுறாங்க nu count kidaikும். HAVING COUNT(*) > 3 vachu filter pannuvom. Apparam manager oda Name kaattanum na, andha ManagerId ah திரும்பி Employees table oடா join pannनணும் — edhukku na ManagerId oru Id mattum thaan, name illa.
-- Sample schema
-- CREATE TABLE Employees (EmployeeId INT PRIMARY KEY, Name VARCHAR(100), ManagerId INT NULL);

SELECT
    m.EmployeeId    AS ManagerId,
    m.Name          AS ManagerName,
    COUNT(e.EmployeeId) AS DirectReportCount
FROM Employees e
INNER JOIN Employees m ON e.ManagerId = m.EmployeeId
GROUP BY m.EmployeeId, m.Name
HAVING COUNT(e.EmployeeId) > 3
ORDER BY DirectReportCount DESC;
🗣️ Interview la eppadi walk-through pannanumThis combines two patterns I use constantly — the self-join to resolve ManagerId into an actual name, and GROUP BY/HAVING to filter on an aggregate count. I group by both EmployeeId and Name (not just Name) because two different managers could theoretically share the same name — grouping by the primary key first is the safe habit, Name just comes along as a dependent column. I'd flag N=3 as a parameter in real code — either a stored proc parameter or a query variable — rather than hardcoding it, since HR would likely want to tweak that threshold for different reports.
🔁 Follow-up variations they might ask
Q: How would you find managers with the MOST reports (top manager by team size)? — Same GROUP BY, drop the HAVING filter, ORDER BY COUNT(*) DESC, and add TOP 1 (or TOP 1 WITH TIES for ties).
Q: How do you count INDIRECT reports too — the whole reporting tree under a manager? — That needs a recursive CTE walking down ManagerId relationships level by level, then counting all descendants, not just direct GROUP BY.
Q: What if a manager has zero direct reports — will they show up? — No, INNER JOIN and the implicit GROUP BY only include managers who actually appear as someone's ManagerId; a manager with zero reports never appears as e.ManagerId, so they're naturally excluded — which is correct here since HAVING > 3 would exclude them anyway.
14. Difference in Days Between Two Dates, Grouped Report (Difficulty: Easy · DATEDIFF / AVG / GROUP BY)
Problem: Given Students(StudentId, ClassId, AdmissionDate) and FeePayments(StudentId, PaymentDate) (first fee payment per student), write a query showing the average number of days between admission and first fee payment, grouped by class — so the finance team can see which classes are slow to pay.
Idhu report-style question — join rendு tables, DATEDIFF(DAY, date1, date2) vachu difference calculate pannுறோம், apparam AVG() + GROUP BY ClassId vachu class-wise summarize pannுறோம். Simple aana romba practical — real ERP reports la இது madhiri "average delay" metrics romba common.
-- Sample schema
-- CREATE TABLE Students (StudentId INT PRIMARY KEY, ClassId INT, AdmissionDate DATE);
-- CREATE TABLE FeePayments (PaymentId INT PRIMARY KEY, StudentId INT, PaymentDate DATE);
-- Assumes FeePayments has one row per student representing their FIRST payment
-- (or use MIN(PaymentDate) per student if multiple payments exist)

SELECT
    s.ClassId,
    AVG(DATEDIFF(DAY, s.AdmissionDate, fp.FirstPaymentDate)) AS AvgDaysToFirstPayment,
    COUNT(*) AS StudentCount
FROM Students s
INNER JOIN (
    SELECT StudentId, MIN(PaymentDate) AS FirstPaymentDate
    FROM FeePayments
    GROUP BY StudentId
) fp ON s.StudentId = fp.StudentId
GROUP BY s.ClassId
ORDER BY AvgDaysToFirstPayment DESC;
🗣️ Interview la eppadi walk-through pannanumI always double-check the direction of DATEDIFF's arguments — DATEDIFF(DAY, start, end) gives a positive number when end is after start, and it's an easy mistake to swap them and get negative averages that confuse a report. I pre-aggregate FeePayments down to one row per student (first payment via MIN) before joining, rather than joining raw and risking double-counting a student with multiple payments in the average. This is the kind of report I'd actually build for the Zen Campus billing dashboard — flagging classes where parents are consistently slow to pay fees after admission.
🔁 Follow-up variations they might ask
Q: How would you find students who took more than 30 days to pay, individually not aggregated? — Same join without GROUP BY/AVG, just add WHERE DATEDIFF(DAY, s.AdmissionDate, fp.FirstPaymentDate) > 30.
Q: What about students who admitted but never paid at all? — Switch the inner join to a LEFT JOIN against the FeePayments subquery and they'll appear with a NULL FirstPaymentDate — decide whether to exclude them from the AVG or report them separately as "unpaid."
Q: How do you express this in months instead of days? — Use DATEDIFF(MONTH, ...) instead, though be aware DATEDIFF(MONTH,...) counts calendar-month boundaries crossed, not full 30-day periods — mention that nuance if precision matters.
15. Find Records That Exist in One Table But Not Another (Difficulty: Medium · LEFT JOIN vs NOT EXISTS vs EXCEPT)
Problem: Given Students(StudentId, Name) and Attendance(StudentId, AttendanceDate), find all students who have NEVER had an attendance record marked (i.e. students in Students but with no matching StudentId in Attendance at all). Compare three approaches.
Idhu very common "anti-join" pattern — 3 vazhi la pannலாம், ovvoru ஒண்ணுக்கும் trade-off irukku. LEFT JOIN ... WHERE B.Id IS NULL — popular aana, andha table la multiple matching rows irundhaal (duplicates) duplicate results வரலாம் na careful ah irukanum. NOT EXISTS — correlated subquery, semantically clean, correctly handle pannும் NULLs ah, and mostly SQL Server optimizer idhula best plan generate pannும். NOT IN namba mention pannாதagata try pannாதீங்க — adhu la ஒரு dangerous trap irukku: subquery result la ஒரே ஒரு NULL value இருந்தாலும், NOT IN ellame silently ஆ empty result தரும். EXCEPT — set-based, clean aana column count/types match aaganum rendு sides la.
-- Sample schema
-- CREATE TABLE Students (StudentId INT PRIMARY KEY, Name VARCHAR(100));
-- CREATE TABLE Attendance (StudentId INT, AttendanceDate DATE);

-- (a) LEFT JOIN ... WHERE IS NULL
SELECT s.StudentId, s.Name
FROM Students s
LEFT JOIN Attendance a ON s.StudentId = a.StudentId
WHERE a.StudentId IS NULL;

-- (b) NOT EXISTS — preferred: correct with NULLs, usually best execution plan
SELECT s.StudentId, s.Name
FROM Students s
WHERE NOT EXISTS (
    SELECT 1 FROM Attendance a WHERE a.StudentId = s.StudentId
);

-- (c) EXCEPT — set-based difference
SELECT StudentId, Name FROM Students
EXCEPT
SELECT s.StudentId, s.Name
FROM Students s
INNER JOIN Attendance a ON s.StudentId = a.StudentId;

-- AVOID THIS — NOT IN silently breaks if Attendance.StudentId ever contains NULL:
-- SELECT StudentId, Name FROM Students
-- WHERE StudentId NOT IN (SELECT StudentId FROM Attendance);  -- returns ZERO rows if any StudentId is NULL
🗣️ Interview la eppadi walk-through pannanumMy default is NOT EXISTS — it's semantically the clearest statement of intent ("give me students where no attendance row exists for them"), it handles NULLs in the subquery correctly, and the SQL Server optimizer typically turns it into an efficient anti-semi-join. LEFT JOIN with IS NULL gives the same result here, but I'm careful with it when the right-hand table could have duplicate StudentId rows, since that's not an issue for NOT EXISTS but can silently inflate a LEFT JOIN's row count before the NULL filter kicks in. I actively avoid NOT IN for this kind of check — if the Attendance.StudentId column ever contains even a single NULL, the entire NOT IN condition evaluates to UNKNOWN for every row and the whole query silently returns zero rows, which is a nasty production bug to track down. EXCEPT is clean too, but it dedupes and requires matching column lists/types, so I reach for it less often than NOT EXISTS.
🔁 Follow-up variations they might ask
Q: Why exactly does NOT IN break with a NULL in the subquery? — NOT IN expands to a chain of AND (x <> v1) AND (x <> v2)...; comparing against a NULL value always yields UNKNOWN, and UNKNOWN anywhere in an AND chain makes the whole condition UNKNOWN (not TRUE), so that row gets excluded — across all rows, the result set becomes empty.
Q: Which of these three has the best performance on a large table with a supporting index? — NOT EXISTS and LEFT JOIN...IS NULL both typically produce equivalent anti-semi-join plans in SQL Server when Attendance.StudentId is indexed — I'd check the actual execution plan rather than assume, but NOT EXISTS is my starting point.
Q: How would you find students who attended on SOME days but were absent on others (partial mismatch), not just zero-attendance students? — That's a different problem — compare COUNT of expected school days vs COUNT of actual attendance rows per student, or use a calendar table joined with LEFT JOIN to find the specific missing dates.
