⚡ Part 16 Quick-Recall Cheat Sheet

  1. Clarify — what's being tested, what's explicit vs your judgment, time budget, tech stack fixed or open.
  2. Design the data model first — 2 min sketch, catch missing-but-implied columns, don't over-add either.
  3. Build the API with clean contracts — match names literally to what was asked, validate, right status codes.
  4. Build UI simple & functional — plain and working beats fancy and broken. Happy path first.
  5. One shared API/DB for multiple UIs — never duplicate save/fetch logic across web + desktop clients.
  6. Ship with README + self-check — explain how to run it, state your assumptions, and actually build/run it before submitting.

16A General Playbook for "Build Me a Sample Project" Tasks
1. Clarify Requirements Before Writing Code (scope the task)
Task kudutha udane keyboard la type panna aarambikkaathey — first 2 nimisham yosi. Enna thaan test pannuranga — DB design ah, API design ah, UI polish ah, illa ellame together ah? Enga explicit ah solli irukanga, enga "un judgement ku vidraanga"? Time budget enna — 2 hours ah, 2 days ah? Tech stack fix pannirukangala illa namma choose pannalaama? Idha clarify pannama poi neenga assume pannina thing wrong ah irundha, whole task wasted aagidum.
🗣️ Interview la eppadi sollanum / pannanumBefore I write a single line, I read the task twice and separate what's explicitly specified from what's left to my judgment — table names, endpoint names, and UI fields if given, I follow literally; anything unspecified, like a missing foreign key or which client-server pattern to use, I decide based on what a real production system would need, and I document that decision.
🔁 Follow-up Qs / traps to avoid
Q: What if it's a live/pair-programming round and I can actually ask? — Ask 1-2 sharp clarifying questions upfront (e.g. "should Status be an enum or free text?") rather than guessing silently — it also shows communication skill.
Q: What if there's no one to ask (async take-home)? — Make a reasonable assumption, note it clearly in the README, and move on — don't stall the whole task over one ambiguous line.
Q: Common trap? — Over-reading the task and adding features nobody asked for "to impress" — that eats your time budget and can look like you didn't read carefully.
2. Design the Data Model First (sketch before code)
Chinna task ah irundhalum kuda, 2 nimisham eduthu tables/entities ah, avanga relationship ah paper la illa notepad la sketch pannunga — idhu thaan senior vs junior answer ah split pannura point. Task la solla column list ah literal ah eduthukama, "idhu functional ah irukka?" nu kekkanum — example, scan history na, edhu unit ah scan pannanga nu link panna column illama irundha, andha table use aagave aagathu. Aana over-engineer pannavum koodathu — kekkaadha 10 extra column add panna, over-thinking nu therium.
🗣️ Interview la eppadi sollanum / pannanumI always spend a couple of minutes sketching the tables and their relationships before touching code — I look for columns that are implied by the requirement even if not explicitly listed, like a foreign key that makes a "history" table actually traceable back to what it's history of. I also deliberately keep it to what's needed — no unrequested audit columns, no extra tables just to look thorough.
🔁 Follow-up Qs / traps to avoid
Q: How do you decide NOT NULL vs nullable? — If the business meaning of the row is broken without that value (like Barcode on a Unit), make it NOT NULL; optional metadata can stay nullable.
Q: Should you always add CreatedDate/ModifiedDate even if not asked? — Only if it's clearly implied or trivial to justify — don't add a full audit trail unless the task hints at needing history/traceability.
Q: What if I realize the model is wrong halfway through? — Fix it immediately rather than working around it in code — a bad schema costs more time the longer you leave it.
3. Build the API Layer With Clean Contracts (naming + validation)
Endpoint/DTO names ah task la solla madhiri exact ah vaikkanum — "SaveUnit" nu solranga na, adha "CreateUnitAsync" nu rename pannama, andha peru vachea vaikanum. Edhukku na, review pannuravanga ku "idhu enna kekkaango adha thaan pannirukan" nu clear ah therியanum, unnecessary friction varaadhu. Input validate pannanum, sensible status codes return pannanum (200/201/400/404), and repository/data-access layer ah controller kூடa mix pannama separate ah vachukanum.
🗣️ Interview la eppadi sollanum / pannanumI name my controller actions and DTOs to match the literal wording of the requirement whenever it's given — if they say SaveUnit and GetHistory, that's exactly what I call my endpoints, because it reduces friction for whoever reviews it against the spec. I validate the model, return 201 on create with the created resource, 400 on bad input, and keep data access behind a repository or service interface so the controller stays thin.
🔁 Follow-up Qs / traps to avoid
Q: Should you rename an awkwardly-named endpoint to something "more RESTful"? — Not if the task named it explicitly — literal alignment beats personal style preference here.
Q: What if the DB save and a related insert need to happen together? — Wrap them in a transaction so you never end up with a half-saved state.
Q: Common trap? — Returning 200 for everything, including creates and errors — graders notice sloppy status codes.
4. Build the UI Layer(s) Simple and Functional (happy path first)
Plain HTML table oru vela okay ah irundhalum, adhu velai seiyum na, adhu than best — broken fancy grid vida million times better. Textbox + button work aaguthu na pothum, over-engineered multi-step form venam. First priority — happy path rock solid ah irukkanum: correct data enter panna, correct ah save aagi, correct ah display aaganum. Time mişchu irundha mattum, polish (styling, loading spinner, client validation) add pannunga.
🗣️ Interview la eppadi sollanum / pannanumI get the core flow — enter data, submit, see the result reflected — working end to end first, using the simplest UI that does the job, like a server-rendered table. Only once that's solid do I spend remaining time on polish like styling or client-side validation, because a working plain UI always beats a half-finished fancy one in a time-boxed task.
🔁 Follow-up Qs / traps to avoid
Q: Isn't a plain table "unimpressive"? — Not if it's correct and fast — interviewers are grading working software, not visual flair, unless the task explicitly says "polish the UI."
Q: What if I have time left over? — Add small, safe polish — sorting, a loading state, basic client validation — never risky refactors right before submitting.
Q: Common trap? — Spending an hour on CSS before the save button even works — always sequence correctness before cosmetics.
5. Wire Multiple UIs to ONE Shared API/DB (single source of truth)
Task la web UI-um desktop UI-um kேkkumbodhu, save/fetch logic ah rendu idathula separate ah eluthanum nu thonum — adha resist pannunga. Rendu client-um same API ah call pannanum, andha API thaan single source of truth. Idhu vechu, business logic (validation, transaction) oru edathula mattum irundhaachum pothum, rendu idathula duplicate pannama irunthaal — idhu interview la "architectural maturity" nu strong ah signal pannum.
🗣️ Interview la eppadi sollanum / pannanumWhen a task needs both a web and a desktop UI, I never duplicate the save or fetch logic — both clients call the same Web API, so there's exactly one place where business rules and validation live. This is exactly how I built the barcode scanner sample — the MVC web app and the WinForms scanner both talk to the same API instead of each having their own copy of the save logic.
🔁 Follow-up Qs / traps to avoid
Q: Isn't it slower to add an API layer for a "quick" desktop tool? — Slightly more upfront work, but it avoids logic drift between clients — worth it any time there's more than one UI.
Q: What if the desktop client needs to work offline? — That's the one case where some local duplication is justified — but say so explicitly, don't silently duplicate logic by default.
Q: Common trap? — Writing separate SQL/Dapper calls directly from the desktop app "just to save time" — that's the exact anti-pattern interviewers are checking for.
6. Ship With a README and a Self-Check (don't submit unverified code)
Ellame mudinjadhum, short README eluthunga — eppadi run pannanum, enna assumptions pannineenga nu. Aana idhu vida mukkiyamana thing — submit pannradhukku munnadi neenga real ah build pannii run pannii paakanum. "Idhu compile aaguma nu try pannen ah" nu solradhu thaan — take-home tasks fail aaguravanga la mostest common reason idhu thaan, silly typo/missing using statement kூடa check pannaama submit pannuranga.
🗣️ Interview la eppadi sollanum / pannanumI always write a short README with run steps and any assumptions I made, and before I call anything done, I actually run it myself — start the API, hit it from the UI, confirm the data round-trips correctly. Skipping that self-check is the single biggest reason take-home submissions fail, so I treat it as a non-negotiable last step, not an optional nice-to-have.
🔁 Follow-up Qs / traps to avoid
Q: What goes in the README besides run steps? — Assumptions you made, any known limitations, and how to seed/reset test data if relevant.
Q: What if I run out of time before finishing everything? — Ship what works, and clearly note in the README what's incomplete — a smaller working thing beats a bigger broken thing.
Q: Common trap? — "It should work, I didn't get time to test it" — that sentence alone can sink an otherwise good submission.

8. The Database Design Decision — Adding UnitId to ScanHistory (requirements judgment)
Task la kudutha ScanHistory column list — HistoryId, OperatorName, ScanTime — mattum thaan irundhichu. Aana yosichu paakumbodhu, "yaar edha scan pannanga" nu therியाma, "எப்போ scan pannanga, யார் scan pannanga" nu mattum theriyum na, andha history table practically useless. So oru UnitId foreign key deliberately add pannineen, Unit table oda link panna. Idhu literal instruction-following ah vida, requirements ah puriinjukittu functional ah build pannradhukku example.
🗣️ Interview la explain pannanumThe given ScanHistory columns didn't include anything linking a scan back to which unit was scanned, so I added a UnitId foreign key to the Unit table — without it, the history table records who scanned and when, but not what, which isn't functional for a barcode scanner. If asked why I deviated from the literal column list, I'd say: I read the requirement for its intent, not just its letter, and a scan history table has to actually be traceable to be useful.
CREATE TABLE ScanHistory (
    HistoryId     INT IDENTITY PRIMARY KEY,
    UnitId        INT NOT NULL FOREIGN KEY REFERENCES Unit(UnitId),
    OperatorName  NVARCHAR(100) NOT NULL,
    ScanTime      DATETIME NOT NULL DEFAULT GETDATE()
);

🔁 Follow-up Qs / traps to avoid
Q: Isn't adding a column "not following instructions"? — Framed correctly it's the opposite — it's showing you understood the purpose of the table, not just copying a column list.
Q: Should UnitId be nullable in case the unit isn't found? — No — if a scan can't be linked to a real unit, that's a validation failure at save time, not a nullable FK to work around.
Q: What if the interviewer disagrees with the addition? — Explain the reasoning calmly and offer to revert if they prefer strict literal compliance — showing you can justify a decision matters more than winning the point.

10. The MVC Web UI Design — Server-Side HttpClient, No CORS (browser never calls the API)
MVC UI la, browser JavaScript direct ah Web API ah call pannala. Adha maathaa, HomeController thaan IHttpClientFactory vechu server-side la Web API ah call pannுthu, appuram result ah view ku pass pannuthu. Idhu vechu rendu benefit — CORS setup pannaveyae venam (edhukku na browser-server communication ellame same-origin ah irukum), and API URL/credentials ellame server la mattum irukும், browser la expose aagathu.
🗣️ Interview la explain pannanumThe browser only ever talks to the MVC app itself — the HomeController's ScanUnit action calls the Web API server-side through IHttpClientFactory, and the view just renders whatever the controller got back. If asked why I didn't call the API from JavaScript, I'd say it avoids CORS configuration entirely and keeps the API's location and any future auth details off the client, which is a cleaner boundary for a browser-facing app.
public async Task<IActionResult> ScanUnit(string barcode, string operatorName)
{
    var client = _httpClientFactory.CreateClient("UnitApi");
    var response = await client.PostAsJsonAsync("api/unit/save",
        new { Barcode = barcode, OperatorName = operatorName });
    return RedirectToAction("Index");
}

Real barcode scanner hardware yenna pannும் theriyுமா — adhu basically oru "fast keyboard" mாthiri, barcode ah type panniட்டு last la Enter key automatic ah anுppும். I also built the form's controls in code rather than the visual Designer, so the layout is explicit and reviewable in a diff instead of buried in Designer-generated markup.

16C Quick Playbooks for Other Common Sample-Project Prompts
13. "Build a CRUD API + UI for [Entity] with Search/Filter" (quick playbook)
Same pattern than — table, API, UI (16B madhiri) — mattum plus rendu extra thing: pagination and a search textbox. Search ku raw string concatenation panna koodathu, always parameterized WHERE ... LIKE @term (Dapper/ADO.NET) illna LINQ .Where(x => x.Name.Contains(term)) (EF Core) use pannunga.
🗣️ Interview la eppadi sollanum / pannanumI'd follow the exact same table-API-UI pattern as any sample project, then add a search textbox wired to a parameterized WHERE LIKE query or an EF Core .Where() clause — never string-concatenated SQL — plus basic pagination so the list endpoint doesn't return the whole table at once.
🔁 Follow-up Qs / traps to avoid
Q: Common trap? — Concatenating the search term directly into a SQL string — instant SQL injection risk, always parameterize.
Q: Where should the search filter run — client or server? — Server/database level always, for anything beyond a tiny in-memory list — don't fetch everything and filter in C#.
14. "Build a WinForms/Console Tool That Imports a CSV into SQL Server" (quick playbook)
OpenFileDialog vechu file pick pannunga, StreamReader + string.Split(',') (illana CsvHelper nuget package, mேலே clean ah irukும் na) vechு rows parse pannunga. Periya file na SqlBulkCopy use pannunga performance ku, chinna file na simple looped INSERT pothum. Insert panna munnadi ovvoru row-um validate pannunga (required fields empty ah irukka, data type match aaguthaa), and end la success/failure count kaatanum ("48 rows imported, 2 failed — see log").
🗣️ Interview la eppadi sollanum / pannanumI'd use OpenFileDialog to pick the file, parse rows with StreamReader or CsvHelper for anything with quoted fields or commas inside values, validate each row before touching the database, and use SqlBulkCopy for large files since row-by-row INSERTs don't scale — then report exact success and failure counts at the end instead of a silent pass/fail.
🔁 Follow-up Qs / traps to avoid
Q: When is looped INSERT acceptable? — For small files (a few hundred rows) where SqlBulkCopy's setup overhead isn't worth it.
Q: What about a malformed row in the middle of the file? — Log it with its row number and reason, skip it, and continue — don't let one bad row abort the whole import.
15. "Build an MVC Page With Server-Side Paging/Sorting on a Datagrid" (quick playbook)
Query params la page, pageSize, sortColumn, sortDirection pass pannunga. Repository query la .Skip((page-1)*pageSize).Take(pageSize) (EF Core) illna equivalent SQL OFFSET/FETCH use pannunga. Mukkiyamana rule — whole table ah memory ku load pannाma, appuram C# la page pannாthinga — adhu periya table ku disaster. Paging always SQL level ல தான் நடக்கனும்.
🗣️ Interview la eppadi sollanum / pannanumI pass page, pageSize, and sort column/direction as query parameters, then apply Skip/Take (or OFFSET/FETCH in raw SQL) directly in the database query — never load the entire table into memory and page it in C#, since that defeats the whole point of paging once the table grows.
🔁 Follow-up Qs / traps to avoid
Q: How do you prevent SQL injection on a dynamic sortColumn? — Whitelist allowed column names against a fixed list/switch — never interpolate the raw query-string value into SQL.
Q: Should you also return total row count? — Yes — the UI needs it to render page numbers/total pages correctly.
16. "Build a REST API With JWT-Secured Endpoints" (quick playbook — sequencing)
JWT mechanics ku (token generate pannرathு, validate pannுrathு, [Authorize] attribute) detail ah Part 8 (#8A) paakunga. Idhu quick playbook la mukkiyam — sequence. First API ah unsecured ah build pannii, working ah confirm pannuங்கள். Apparam mattum JWT bolt-on pannunga. Rendayum together debug panna try panninga na, "auth fail aaguthா, illa business logic fail aaguthா" nu confuse aagi time waste aagும்.
🗣️ Interview la eppadi sollanum / pannanumI always build and verify the API's business logic unsecured first, confirm every endpoint works with a plain HTTP client, and only then add JWT authentication on top — trying to debug both auth and business logic simultaneously means every failure is ambiguous about which layer broke.
🔁 Follow-up Qs / traps to avoid
Q: Isn't building it unsecured first a security risk if you forget to add auth later? — Track it explicitly as a task step, and never consider the task "done" until auth is verified — it's a sequencing choice, not a shortcut.
Q: How do you test auth once added? — Hit a protected endpoint without a token (expect 401), then with a valid token (expect 200) — both cases, not just the happy path.
17. "I've Never Built Exactly This Before, What Do I Do" (the meta-answer)
Task genuinely unfamiliar domain na kூடa panic aagаaதீங்க. Same 6 steps 16A la — clarify, data model, API, UI, shared backend, README+self-check — edhu domain ah irundhalும் apply aagும். Live/pair-programming round na, idha loud ah sollுங்க — "let me start by sketching the data model" nu solli start pannுங்க. Idhu process ah narrate pannுradhே, code eluthுவதுку munnadியே, strong ah signal aagும் — interviewer ku "idhu structured ah yosikkuரார்" nu therியும்.
🗣️ Interview la eppadi sollanum / pannanumEven in an unfamiliar domain, I fall back on the same six steps — clarify scope, sketch the data model, build a clean API contract, keep the UI simple, share one backend across clients, and finish with a README and a self-check. In a live round I say this out loud — "let me start by sketching the data model for this" — because narrating a structured process is itself a strong signal, even before I've written a line of code.
🔁 Follow-up Qs / traps to avoid
Q: What if you genuinely don't know a required technology (e.g., a new library)? — Say so honestly, then reason about how it likely works from similar tools you do know, and keep moving — don't fake confidence you don't have.
Q: Isn't narrating your thinking distracting in a timed round? — No — interviewers running live rounds usually want to see your process, not just a silent final answer; check the format if unsure.
Q: What's the single biggest mistake in an unfamiliar-domain task? — Jumping straight to code without the 2-minute data-model sketch — that's exactly the step most likely to save you from a costly restart later.
