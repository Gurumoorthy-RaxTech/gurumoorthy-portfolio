⚡ Part 23 Quick-Recall Cheat Sheet

  • First step for any slow query → Execution plan, check Scan vs Seek, look for the fattest arrow (biggest row estimate mismatch)
  • Clustered Index → table oda actual physical order (1 per table); Non-Clustered → separate pointer structure (many per table)
  • Covering Index → query ku venும் columns ellame index la irundhaа thaан, key lookup venும் illa — INCLUDE columns vачி achieve pannலam்
  • Fragmentation → 5-30% REORGANIZE, >30% REBUILD (rule of thumb)
  • Isolation levels (weakest → strongest) → Read Uncommitted → Read Committed (default) → Repeatable Read → Serializable → Snapshot
  • Deadlock fix → same order la resources access pannуம்படி standardize pannуроadhு, transactions சிறியதா vачு vечிроadhу
  • Optimistic vs Pessimistic → Optimistic = check at save time (rowversion/timestamp); Pessimistic = lock upfront (SELECT...FOR UPDATE style)
  • OLTP vs OLAP → OLTP = many small fast writes (normalized); OLAP = few large reads for analytics (denormalized/star schema)
  • CTE vs Temp Table → CTE = query-scoped, no index, re-evaluated each reference; Temp Table = session-scoped, can index, materialized once
  • ROW_NUMBER vs RANK vs DENSE_RANK → ROW_NUMBER unique sequential; RANK skips numbers after tie; DENSE_RANK no gaps after tie
  • CROSS APPLY vs OUTER APPLY → CROSS APPLY = INNER JOIN-like (drops no-match rows); OUTER APPLY = LEFT JOIN-like (keeps them with NULLs)
  • CPU spike triage → sys.dm_exec_query_stats ORDER BY total_worker_time DESC — find the top CPU-consuming query first

9. Difference between Clustered and Non-Clustered Indexes (Physical vs Logical)
Clustered Index na — table oda actual physical row order ah define pannрроадu — so ஒரு table ku ஒரே ஒரு clustered index mattum iருккும் (usually Primary Key mூlама auto create aагுрроадு). Table ku பல non-clustered indexes iருккலाம் (SQL Server 2016+ la max 999). Non-clustered index use pannінா, actual data venும்на "Key Lookup" nு extra step vேண்டியiருккும் (adhа avoid panна thaan Covering Index concept).
🗣️ Interview la eppadi sollanum"A clustered index defines the table's actual physical row order, so there can only be one per table — typically created automatically on the primary key. A non-clustered index is a separate structure holding the indexed columns plus a pointer back to the actual data row, and a table can have many of them. Using a non-clustered index that doesn't cover all needed columns requires an extra Key Lookup step to fetch the rest of the row, which is exactly what covering indexes are designed to eliminate."
🔁 Follow-up Qs they might fire next
Q: What does the pointer in a non-clustered index actually point to? — If the table has a clustered index, it points to the clustering key; if the table is a heap (no clustered index), it points directly to the physical row location (RID).
Q: Why is primary key usually the clustered index by default? — It's the natural, unique, stable identifier most queries filter or join on, so physically ordering the table by it tends to benefit the most common access patterns — though it's not mandatory, and sometimes a different column is a better clustering key.

🗣️ Interview la eppadi sollanum"A covering index contains every column a query needs — for filtering, joining, and selecting — so SQL Server can satisfy the entire query from the index itself, without a Key Lookup back to the clustered index. Q: Why not just put everything in the key instead of INCLUDE? — A wider key means a bigger index, more page splits, slower writes, and it affects sort order — INCLUDE keeps only what's needed for filtering/sorting in the key while cheaply carrying the rest.
Q: How do you spot a missing covering opportunity in an execution plan? — A Key Lookup operator next to an Index Seek — SSMS often even suggests the exact INCLUDE columns needed to eliminate it via a missing index hint.

🗣️ Interview la eppadi sollanum"Fragmentation happens when an index's logical page order drifts from its physical disk order, mainly from ongoing inserts, updates, and deletes, and it hurts read performance through extra I/O. Q: How would you automate this maintenance? — A scheduled SQL Agent job (or Maintenance Plan / Ola Hallengren's popular open-source scripts) that checks fragmentation per index nightly or weekly and applies REORGANIZE or REBUILD based on the threshold automatically.
