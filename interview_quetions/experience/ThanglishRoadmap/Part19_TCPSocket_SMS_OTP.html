19A TCP Socket & Hardware Communication
1. TCP vs UDP (why TCP for locker hardware)
TCP na connection-oriented — handshake pannitu, ella packet-um acknowledge pannitu, lost aana retransmit pannும், order guarantee pannுрோம். UDP na fire panni "poidum" nu vidura maadhiri — fast aana reliability illa, video streaming/gaming ku pothum. Locker system la, unlock command or hardware ack silent ah drop aana, locker open aagaama poidும் allathு worse ah clean audit trail illaama open aagidும் — idhு security system la accept panna mudியாthு risk. So andha guaranteed, ordered delivery ku TCP thaan correct choice — UDP "lighter" nu early discussion vandhalும், andha risk pesina udane TCP ku vandhu vittom.
🗣️ Interview la eppadi sollanumTCP is connection-oriented — it does a handshake and guarantees every packet is acknowledged and retransmitted if lost, in order. UDP just fires packets with no such guarantee, which is fine for video streaming but not for a security system. On the locker project every command between our server and the IoT board — open request, OTP validation result, hardware ack — went over TCP, because a silently dropped unlock ack could mean a locker that never opens, or worse, one that opens without a clean audit trail. We discussed UDP briefly for being "lighter," but the moment we talked through a missing-ack scenario, TCP was the obvious call.
🔁 Follow-up Qs they might fire next
Q: UDP eppo actually use pannalam? — Speed correctness ah vida mukkiyam nu irundha mattум் — video streaming, gaming, adhula frame drop aanalum periya problem illa.
Q: TCP handshake eppadi work aagum? — SYN, SYN-ACK, ACK moondru-step process, rendu side-um connection ready nu confirm pannikara varaikkum data pogaathu.
Q: TCP-oda guarantee vaangura cost enna? — Konjam extra latency and overhead (ack, retransmit) — aana locker madhiri correctness-critical system ku andha cost worth it.
2. .NET Socket Class (sync vs async)
System.Net.Sockets.Socket class vachu low-level ah Connect pannitu Send/Receive bytes directly pannலாம். Synchronous call na, data varra varaikkum thread block aagi nikkும் — oru quick script ku pothும் aana multiple lockers same server ku connect aana andha approach mudியாthு. Async — BeginReceive/EndReceive (pazhaya style) allathு ippo Task-based SendAsync/ReceiveAsync — vachu thread ah wait pannாma vidura, wire la data varra varaikkும் vera velai pannikittu irukkalாம். Oru thread per locker connection nu block panna, konjam lockers active aana udane thread pool exhaust aagi app choke aagidும்.
🗣️ Interview la eppadi sollanumI used the Socket class directly for low-level control — connect, then send and receive bytes with the hardware's protocol. Synchronous calls block the thread until data arrives, which works for a quick prototype but doesn't scale once multiple lockers are connected at once. So in production I used async socket calls so the backend could keep several locker connections open without dedicating a thread to each one — I started with a simpler synchronous version just to nail the byte format with the hardware vendor, then swapped to async once the protocol was confirmed working.
🔁 Follow-up Qs they might fire next
Q: Synchronous eppo ok? — One-off diagnostic tool or quick test client ku fine, production multi-connection system ku illa.
Q: Async vacha thread eppadi save aagum? — Wait panra neram thread pool thread ah release pannidும், data ready aana mattும் thaan thirும்ba pick pannும் — so oru thread pannirundhalும் pathhu connections handle pannலாம்.
Q: BeginReceive/EndReceive vs Task-based async — enna difference? — Rendум் same idea, aana Task-based (SendAsync/ReceiveAsync) modern C# la cleaner, async/await vachu readable ah eludhalாம்.
3. Bidirectional Real-Time Communication (persistent duplex socket)
Idhு straight ah REST madhiri request-response illa. Server "locker 14 open pannு" nu command push pannும், aana hardware kூடa tamper alert allathு door-open sensor trip madhiri events ah thaan aaga push pannанும் — server kekkatha nerathula kூடa. Adhukaaga socket ah open ah vachikkonde irukanum, rendு side-um edhு venалும் nerathула write pannலாம். Ella nerathum server hardware ah poll panni state kekkанும் na wasteful and latency adhigam aagும் — persistent duplex connection irundha, hardware change aana udane thaanaa sollும்.
🗣️ Interview la eppadi sollanumThis isn't a plain request-response pattern like REST — it's a persistent, duplex socket where both sides can write whenever something happens. On the locker board, once OTP validation passed, our server pushed the unlock command down that open socket, and the hardware pushed back a physical-state ack — like "door actually opened" — on the same connection, which we logged for the audit trail. That ack mattered a lot, because without it we'd only know we sent the command, not that the locker actually responded.
🔁 Follow-up Qs they might fire next
Q: Polling vachu idha replace panna mudiyுmа? — Technically mudiyும் aana server ovvoru second-um hardware ah kekkанும் — wasteful and slow, persistent socket better.
Q: Idhு WebSocket madhiri thaanaa? — Concept similar — duplex, persistent — aana idhு raw TCP socket, custom protocol namba design pannirukkом், WebSocket illa.
Q: Ack varaama poyina enna pannுрீнgа? — Andha command "pending/unconfirmed" nu mark pannitu, retry or alert flow ku pogும்.
4. Connection Drops & Reconnection (heartbeat)
TCP connection permanent-a irukkathு — flaky network switch, locker unit power blip, edhு venалும் socket ah clean close notification illама kollidும். Adhukaaga heartbeat mechanism venும் — periodic ah chinna ping packet anுppுрோம், oru window kulla response varாma na connection dead nu assume pannitu reconnect pannuрோம். Heartbeat illама na, server "socket fine" nu nenachikittu irukkும் aana adhு 10 minutes ah dead ah irundhirукkும் — customer complaint varra varaikkும் theriyாthு.
🗣️ Interview la eppadi sollanumTCP connections can silently die without a clean close — a network blip or a power issue on the locker unit is enough. So we built a periodic heartbeat ping, and if a locker missed a couple of beats in a row, the server flagged it and reconnected with a backoff instead of hammering it every second. I remember one locker unit kept flapping every few hours, which turned out to be a router issue on-site, not our code — but the reconnect logic meant it self-healed instead of needing someone to manually restart the service.
🔁 Follow-up Qs they might fire next
Q: Backoff nu edhு, en use pannirukeenga? — Retry interval ovvoru failure-kum konjam konjam increase pannuрathу — network ah spam pannாma, gradually recover panna vidுрathு.
Q: Heartbeat interval eppadi decide pannுрீнgа? — Too frequent na unnecessary traffic, too rare na dead connection detect panna late aagும் — balance ah test pannitu set pannанும்.
Q: Reconnect aana apparam pending commands enna aagும்? — Andha state ah track pannitu, reconnect aana udane pending/unconfirmed commands ah retry allathு re-check pannанும்.
5. Message Framing (partial & merged reads)
TCP bytes ah order-la, uncorrupted ah kudukkும் guarantee tharும் — aana oru Receive call ku full "message" kidaikkும் nu guarantee tharாthு. 50-byte command anுppina, OS adha 30+20 nu rendு reads ah kudukkalாм், allathு rendு chinna messages ah oru read ah batch pannikkalாм். Idha "one Receive = one message" nu assume pannina, message split aanadhalும், rendு messages merge aanadhalும் garbage parse pannுрோம். Fix — fixed length-prefix: first few bytes payload length sollும், appuram exact andha bytes count varaikkும் wait pannitu thaan complete message nu treat pannанும்.
🗣️ Interview la eppadi sollanumTCP guarantees ordered, uncorrupted bytes, but not that one Receive call equals one complete message — a single send can get split across two reads, or two small sends can get batched into one. On the locker protocol we hit exactly this — intermittent parsing failures that looked random until a packet capture showed two commands concatenated in a single read. We fixed it by adding a length-prefix — read the length first, then buffer until we have the full frame — and that class of bug basically disappeared after that.
🔁 Follow-up Qs they might fire next
Q: Length-prefix eppadi implement pannுрீнgа? — First N bytes (example 4 bytes) la payload length store pannுрோм், appuram exact andha count bytes varaikkும் read loop la wait pannுрோம்.
Q: Nagle's algorithm idhoda eppadi related? — Chinna sends ah OS batch panni anுppலாம் (Nagle), adhுnaale rendு messages oru read ah vandhurும் — framing illама idhு confusing aagும்.
Q: WebSocket use pannirundhaa idhு problem varுmа? — Illa, WebSocket already message framing built-in ah handle pannும் — raw TCP socket la namba manual ah handle pannанும்.
19B OTP Lifecycle & SMS Gateway
6. OTP Lifecycle Design (Generate → Deliver → Validate → Expire)
Nаngu stages — Generate: random code create pannitu, user session + locker request ku tie pannitu, hashed ah store pannанும், plaintext ah log pannакூடाthு. Deliver: SMS gateway mூlама push pannанும். Validate: user keypad/app la code type panna, stored value oda compare pannанும் — code mattум் illама request ID kூடa match aaganum (replay avoid panna). Expire: fixed window (2-5 nimisham) kazhichu allathு ஒரே use ku apparam, OTP automatically dead — use pannalaiyo pannirundhalும் sari.
🗣️ Interview la eppadi sollanumI structured it as four stages — generate, deliver, validate, expire — each writing a row into a SQL Server audit table. We built this as REST endpoints, and validate wasn't dependent purely on a background cleanup job — it re-checked the OTP's own timestamp on every attempt, so even if the expiry job lagged, a stale OTP still got rejected. That double-check saved us at least once when the cleanup job fell behind under load.
🔁 Follow-up Qs they might fire next
Q: Generate stage la enna precaution edukanum? — Cryptographically secure random use pannанும், plaintext ah store pannакூடाthு — hash pannitu vachukanum.
Q: Validate stage la code mattум் check panna pothумa? — Illa, request ID kூடa match pannанும், illana oru valid OTP ah vera request ku replay panna mudiyும்.
Q: Expire background job mattум் nambalामா? — Illa, validate call itself timestamp check pannанும் — background job lag aanalும் security hole varaккூடाthு.
7. Secure OTP Generation (RandomNumberGenerator vs Random.Next())
Random.Next() cryptographically secure illa — seed predictable ah irukkалாம், attacker seed ah guess pannina "random" numbers ah predict pannலாம். Security-sensitive edhуkkும் System.Security.Cryptography la irukka RandomNumberGenerator use pannанும் — OS oda crypto-secure random source la irundhு edukkும். OTP oda whole security value "unguessable ah irukkанும்" nu adhула thaan irukkு — predictable pattern irundha, OTP vachirukka point-e illама poidum.
🗣️ Interview la eppadi sollanumRandom.Next() isn't cryptographically secure — it's seeded predictably enough that an attacker could potentially predict values. For the locker OTPs I used RandomNumberGenerator to produce a 6-digit code, and stored a hash of it rather than the raw code, so even read access to that table wouldn't give usable OTPs. I'll be honest — I didn't get this right in the very first pass, we were storing it plaintext during early dev, and a code review flagged that we needed to hash it before it went anywhere near production.
🔁 Follow-up Qs they might fire next
Q: Random.Next() eppo use pannalாм்? — UI list shuffle pannuрathу madhiri non-security cases ku fine, auth-related edhуkumе kூடाthு.
Q: OTP ah hash pannுрathு enna benefit? — Table read access kidaichalும், raw OTP illама hash mattум் irukkும் — usable illa, adhуvும் short expiry vachirukkும்.
Q: 6-digit numeric code pothумa, athigama venumaa? — Expiry + rate limiting kூடavе irundha 6-digit pothும் — adhуkkு mela pona user experience affect aagும், security marginal ah mattум் improve aagும்.
8. OTP Expiry & Rate Limiting (anti brute-force)
Expiry na attack window ah chinna ah vachirukkுрathу — 2-5 nimisham apparam OTP database la dead, evlo attempts irundhaलும் sari. Rate limiting na oru OTP ku evlo guesses (3-5 tries) nu cap pannuрathу — andha limit mudinja, request lock pannitu pudhу OTP generate panna sollанும். 6-digit OTP ku million possible values irukkு nu tonna, rate limiting illама attacker script vachு fast ah try pannலாம் — expiry + attempt limit rendும் sernthu thaan 6-digit code ah practically safe ah aakkும்.
🗣️ Interview la eppadi sollanumExpiry keeps the attack window small — a few minutes — and rate limiting caps failed attempts before we invalidate the OTP and force a fresh one. On the bank locker system we capped it at a handful of failed attempts, and every failed try got logged with the locker ID and timestamp, which turned out to be as useful as the rate limit itself — it gave us a clean signal if someone was probing a specific locker repeatedly. Combined with the short expiry window, that's a big part of why we had zero unauthorized access incidents after go-live.
🔁 Follow-up Qs they might fire next
Q: Rate limit exceed aana apparam enna aagும்? — Andha OTP invalidate pannitu, user pudhу OTP request pannанும் — same code ku thirும்ba try panna vida maatom.
Q: Failed attempts log pannuрathу edhуkku? — Same locker mela repeated probing irukka nu pattern kaanamudiyும் — potential attack early ah spot pannலாம்.
Q: Expiry window evlo perfect ah irukkанும்? — Too long na attack window athigam, too short na legit user OTP miss pannitu retry panna vendiyathу varும் — 2-5 nimisham balance ah irukkும்.
9. Integrating a Third-Party SMS Gateway
Send endpoint call panradhу thaan easy 10% — bakki ellам் response codes properly handle pannuрadhula irukkு. "Success" nu vandha, adhу "delivered" nu artham illa, chila gateways "accepted for delivery" nu mattум் artham — delivery status callback allathு polling venும். Retry logic transient failures ku, timeout handling slow gateway namba flow ah hang panna vidாma irukka, and error codes ah namba oda meaningful logging ku map pannанும். API call succeed aachu nu "SMS delivered" nu assume pannina, user OTP kidaikkாma poyirukka, namакkு edhуkkு nu debug panna vazhiyும் illама poidும்.
🗣️ Interview la eppadi sollanumCalling the send endpoint is the easy part — the real work is handling their response codes properly, since a success response often just means "accepted", not "delivered". For the locker system I made sure every send call's response — including their message ID and status — got written to our audit log, not just a generic "sent" flag. That mattered later when we needed to trace a specific delayed-message complaint back to the gateway's own reference ID instead of just guessing.

Q: Delivery status eppadi confirm pannுрீнgа actual ah? — Gateway kudukura delivery callback webhook or status polling API vachু — "accepted" and "delivered" rendு vera state nu track pannанும்.
Q: Timeout handle pannाlaa na enna aagும்? — Slow gateway namba whole OTP generation flow ah block pannidும் — so outbound call ku explicit timeout vachு fail-fast pannанும்.

10. Getting SMS Delivery Under 3 Seconds
Delivery slow aaguрadhуkkу palah factors sera venும் — gateway ku network latency, namba code queue pannura request eludukka edukka neram, send call ku munnadi edhachум் synchronous blocking irukka, and gateway oda own processing time (namba control panna mudiyaathு). Namba control panna mudiya part — OTP generation, hashing, DB audit write ellам் SMS call ku munnadi slow blocking chain ah irundhuchaa nu check pannuрadhу. Profile pannina — DB audit write, SMS call ku munnadi nadakkuрadhу extra latency add pannirundhуchு.
🗣️ Interview la eppadi sollanumA few things stack up before an SMS actually sends — our own code's queueing time, any blocking work before the call, and the gateway's own processing. We profiled the generate-to-send path and found the DB audit write was happening before the SMS call rather than in parallel or after, adding unnecessary latency. Reordering so the SMS request fired first and the audit logging happened right after — plus picking a gateway with decent response times — got us consistently under 3 seconds. It wasn't one big fix, honestly, it was trimming a handful of small things that added up.
🔁 Follow-up Qs they might fire next
Q: Audit write ah SMS call apparam pannina data loss risk illaiyaa? — Namba andha write ah immediate ah, SMS response kitta oda pannом் — so risk minimal, aana benefit periyathу.
Q: Async ah SMS call fire panna mudiyуmа? — Mudiyum, aana user ku confirmation venும் na konjam wait pannанும் — namba critical path mattум் trim pannom, full fire-and-forget pannala.
Q: User "resend" multiple times click pannina enna aagும்? — Duplicate OTPs and gateway load athigam aagும் — adhуkaaga fast delivery mukkiyam, resend button ah konjam delay vachு rate-limit pannанும்.
19C Security, Audit, Concurrency & Testing
11. Audit Trail Logging (why it's not optional)
Idhு "nice to have" logging illa — bank locker ku "yaru edha access panna try panninanga, eppo, enna nadanthuchu" nu answer pannuрa record venும், success mattум் illама every event ku. Adhула OTP generation, delivery attempt + gateway response, ovvoru validation attempt (pass/fail), expiry events, actual unlock command and hardware ack ellам் varும். Dispute vandha — customer "enakku access kidaikala" nu solli, allathு "vera yaro locker ku access aachу" nu solli — audit trail thaan namma source of truth.
🗣️ Interview la eppadi sollanumEvery stage of the OTP lifecycle wrote a row to a SQL Server audit table — locker ID, masked phone number, timestamp, status, and for failures, the specific reason like expired or wrong code. If there's ever a dispute — a customer says they never got access — that audit trail is the only source of truth we have. Honestly, that level of detail is a big reason we could say zero unauthorized access incidents with confidence — we had logs to back it up, not just an assumption nothing bad happened.

Q: Failure reason specific ah log pannuрadhу edhуkkу mukkiyam? — "Failed" nu mattум் irundha debug panna mudiyaathу — "expired" vs "wrong code" vs "rate limited" nu theriyanум், pattern spot panna.
Q: Audit table kூடa security risk aagумаа? — Aama, adhуve sensitive data — so access control vachு, raw OTP illама hash/masked data mattум் store pannанும்.
12. SMS Gateway Downtime & Failover
Gateway unreachable aana, OTP generation call hang aagavo unhandled exception throw pannavo கூடாthு. Outbound call ku timeout vachு fail-fast pannанум், transient blips ku short backoff-oda konjam retries pannанும், ideal ah oru secondary gateway irukkanum, illaна user ku clear failure state kaatanum, silent ah hang aagaккூடாthு. Fallback illama na, third-party vendor outage aana mattум் namba system-oda availability kூடa direct ah affect aagidும் — users literally locked out.
🗣️ Interview la eppadi sollanumWe put a timeout and retry-with-backoff around the SMS send call, and if it still failed after retries, the user got a clear "try again" message instead of the request hanging silently. We didn't have a full secondary gateway provider on that project — budget and vendor contracts didn't allow it at the time. Looking back, that's honestly the one gap I'd push harder on if I did it again — a proper failover provider would've made it more resilient, and it's something I'd flag now in a design review.

Q: Secondary gateway illама enna risk irukkу? — Single point of failure — andha vendor down aana namba locker access completely stop aagidum.
Q: User ku enna message kaatuрீнgа failure aana? — Generic "system error" illама, "SMS anுppa mudiyala, konjam neram kazhichu try pannunga" nu clear, actionable message.
13. Securing a Raw TCP Channel (TLS via SslStream)
Plain socket na plaintext bytes thaan wire mela pogும் — encryption illa, other end authenticate pannuрa vazhi illa. Secure panna, TLS layer pannанும் — .NET la SslStream vachு underlying NetworkStream ah wrap pannுрோம், adhу certificate handshake pannitu, apparam ella data-um encrypt pannும். Hardware oda certificate ah validate pannанும், blind ah accept pannaккூடாthு — illaна man-in-the-middle vandhu network segment la sit pannitu commands sniff/replay panna mudiyум்.
🗣️ Interview la eppadi sollanumPlain sockets are just unencrypted bytes on the wire, so we layered TLS on top using SslStream wrapping the underlying NetworkStream, which handles the certificate handshake and encrypts everything after that. For a bank locker, an unlock command or OTP result going over the wire in plaintext is an obvious attack surface, so this was part of the stack alongside HTTPS for the REST side. Getting the hardware vendor's firmware to play nice with our TLS setup took some back-and-forth — their side had cert format quirks — but once sorted, no plaintext commands went over the wire.
🔁 Follow-up Qs they might fire next
Q: Certificate validate pannalaama enna risk? — Blind ah accept pannina, man-in-the-middle vera cert vachு impersonate panna mudiyум் — validation mandatory.
Q: SslStream eppadi NetworkStream vera? — SslStream adha wrap pannikittu, adhуkkு mela encryption layer add pannum் — application code same Send/Receive pattern thaan use pannும்.
Q: TLS handshake overhead socket connection ku affect aagумаа? — Connection establish panna oru chinna one-time cost irukkு, aana ongoing data transfer speed mela periya impact illa.
14. Concurrent Requests to the Same Locker
Vera vera lockers ku simultaneous requests vandha, adhu mostly fine — each socket connection independent, async I/O adha naturally handle pannும். Kashtaмa irukka case — rendு requests ஒரே locker ku race pannuрadhу — adhуkkு lock venும், DB-level check on request state, allathு locker ID vachு in-memory lock — so rendaavadhு request first one already in-progress nu paathu reject/queue aagум். Illaна, rendும் valid OTP-um ஒரே physical locker ah unlock panna try pannலாம் — audit trail confuse aagும், "yaru actual ah open pannanga" nu security ambiguity varும்.
🗣️ Interview la eppadi sollanumRequests to different lockers are fine on their own, since async I/O handles them independently. The trickier case is two requests racing for the same locker — we tracked each locker's active request state, so if a second OTP validation came in while one was already mid-flow, it got rejected with a clear "request already in progress" instead of silently racing. We didn't hit this often in practice, but we tested it deliberately by firing near-simultaneous requests in a test harness to make sure it behaved correctly rather than just hoping.

16. Toughest Bug (socket framing incident)
Munnadi sonna framing issue thaan best example — incoming hardware messages intermittent ah parse aagaama poidும், first look la totally random ah tonnும். First instinct hardware firmware bug nu — reproduce panna mudiyaama irundhathunala hardware vendor kூடa konjam neram pesi, appuram thaan namba oda receive-buffer handling ah seriously paathom். Rendு command messages ஒரே TCP read buffer la landing aagi, oru garbled message ah parse aagiduchu — length-prefix framing day-1 la illама poyadhула problem irundhuchu. Packet capture vachு raw bytes ah paathu, rendு JSON payloads concatenated ah irukka paathu clicked aachu.
🗣️ Interview la eppadi sollanumThis was intermittent parsing failures on incoming hardware messages that looked totally random at first — my first instinct was a hardware firmware bug since I couldn't reproduce it on demand, so I spent time going back and forth with the vendor before really digging into our own socket code. A packet capture showed two of our command messages had landed in the same TCP read buffer and gotten parsed as one garbled message, since we hadn't put length-prefix framing in from day one. Adding the length-prefix and a proper buffering loop that waits for the full frame fixed it completely — but yeah, I blamed the wrong thing first before looking at our own code.
