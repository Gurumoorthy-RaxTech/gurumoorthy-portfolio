⚡ Part 5 Quick-Recall Cheat Sheet

  • 1. Joins — INNER=matching only, LEFT=all left+match, RIGHT=all right+match, FULL=both sides, CROSS=cartesian, SELF=table joins itself
  • 2. WHERE vs HAVING — WHERE filters rows before grouping, HAVING filters groups after aggregation
  • 3. Subquery vs Join — Join when you need columns from both tables; EXISTS subquery when you just need existence check
  • 4. UNION vs UNION ALL — UNION removes duplicates (slower, implicit sort), UNION ALL just appends (faster)
  • 5. CTE — WITH clause named result set, readable, supports recursion, not materialized/indexed
  • 6. Window Functions — ROW_NUMBER/RANK/DENSE_RANK + PARTITION BY for top-N-per-group without collapsing rows
  • 7. Clustered vs Non-Clustered — Clustered = physical row order (1 per table), Non-Clustered = separate pointer structure (many per table)
  • 8. Finding slow queries — Execution plan: Table/Clustered Index Scan = bad, Index Seek = good; Query Store / sys.dm_exec_query_stats for prod
  • 9. Covering Index — Index has every column the query needs (key + INCLUDE), avoids Key Lookup
  • 10. Over-indexing — Every index slows INSERT/UPDATE/DELETE; drop unused ones via sys.dm_db_index_usage_stats
  • 11. Stored Procs vs inline SQL — Cached execution plan + parameterized + no direct table access = faster and safer
  • 12. The 40% story — Attendance report 5s → execution plan showed scans → added composite index + SP + Dapper mapping → 2-3s
  • 13. Optimization checklist — No SELECT *, sargable WHERE, right indexes, set-based not cursors, paginate, update stats
  • 14. ACID — Atomicity, Consistency, Isolation, Durability — all-or-nothing, valid states, no interference, survives crash
  • 15. Isolation levels — Read Committed = default; Snapshot for non-blocking reports; Serializable = strictest/slowest
  • 16. Deadlocks — Circular wait on locks; SQL Server kills the cheaper "victim"; fix = consistent access order, short transactions
  • 17. PK/FK/Unique/Check — PK=unique+not null (1 per table), FK=referential integrity, Unique=allows one null, Check=custom rule
  • 18. Optimistic vs Pessimistic — Pessimistic locks upfront (ATM-style); Optimistic checks rowversion at save time, rejects on conflict
  • 19. Triggers — Auto-fire on DML; useful for DB-level audit but hidden logic, hard to debug — use sparingly
  • 20. Normalization vs Denormalization — 1NF/2NF/3NF removes duplication; denormalize only for read-heavy dashboards/reports
  • 21. Temp table vs Table var vs CTE — #Temp = indexable/real stats (big data), @TableVar = no stats (small data), CTE = not materialized
  • 22. Pagination — OFFSET x ROWS FETCH NEXT y ROWS ONLY with mandatory ORDER BY, index the sort column
  • 23. SQL Injection prevention — Always parameterized queries (Dapper @param / EF Core / SqlParameter), never string concat
  • 24. Views — Saved SELECT, reuse complex joins, can restrict columns for security; indexed views physically store data
  • 25. Backup/Recovery — Full/Differential/Log backups, Full recovery model for point-in-time restore — DBA territory mostly, basics understood

5A Query Fundamentals & Joins
1. Types of Joins (INNER / LEFT / RIGHT / FULL / CROSS / SELF)
Namba rendு table ah oru common column (key) vachi connect pannுрதுக்கு thaan joins. School ERP example vachi paakalam — Students table oru side, Fees table vera side, StudentId thaan connecting key. INNER JOIN na — rendு table லும் match aana rows mattum thaan varும் (fees record irukka students mattum). LEFT JOIN na — Students table full ah eduthுட்டு, fees match aanaal add pannும், illana NULL kaatும் (fees pay pannாma irukka students um list la varும் — idhu thaan real reports ku useful). RIGHT JOIN na LEFT oda opposite. FULL JOIN na rendு side layum full ah edukும். CROSS JOIN na every row oda every row combine aagும் (cartesian product — carefully use pannanum, accidentally periya table la use pannினா millions of rows varும்). SELF JOIN na table ah adhே table kூடா join pannுрது — example Employee table la Manager um employee thaане, so EmployeeId = ManagerId vachi join pannலாம்.
SELECT s.Name, f.AmountDue
FROM Students s
LEFT JOIN Fees f ON s.StudentId = f.StudentId;
🗣️ Interview la eppadi sollanumI use joins to combine related tables through a common key. INNER JOIN gives only matching rows, LEFT JOIN gives everything from the left table with NULLs where there's no match, FULL JOIN gives both sides, and CROSS JOIN is a cartesian product I use rarely and carefully. I also use SELF JOIN for hierarchy-type data, like a manager-employee relationship in the same table.
🔁 Follow-up Qs they might fire next
Q: Difference between LEFT JOIN and LEFT OUTER JOIN? — Same thing, OUTER is just an optional keyword SQL Server lets you drop.
Q: When would you actually use CROSS JOIN in a real project? — Rarely — maybe generating all date-vs-class combinations for a timetable grid or report scaffold.
Q: If you filter on the right table's column in WHERE after a LEFT JOIN, what happens? — It silently turns your LEFT JOIN into an effective INNER JOIN, since NULL rows get filtered out — a classic mistake I watch out for.
2. GROUP BY + HAVING vs WHERE (filtering rows vs filtering groups)
WHERE na — grouping nadakura munnadi, individual rows ah filter pannுрது (example: Status='Active' irukka students mattum edukkanum). GROUP BY na — rows ah oru column basis la group pannுрது (example: ClassId wise group panni count எடுக்கनும்). HAVING na — group aana apparam, aggregate result mela filter pannுрது (example: 30-க்கு mேல் students irukka classes mattum kaatanum). Simple ah artham vechுக்க — WHERE gate la nikkும், oru oruthar ah check pannும், group aaganum munnaale; HAVING team form aana apparam, andha team size check pannும்.
SELECT ClassId, COUNT(*) AS TotalStudents
FROM Students
WHERE Status = 'Active'
GROUP BY ClassId
HAVING COUNT(*) > 30;
🗣️ Interview la eppadi sollanumWHERE filters individual rows before grouping happens, and it can't reference aggregate functions. HAVING filters after GROUP BY has produced aggregated results, so it's the right place to say something like "only classes with more than 30 active students." I always push non-aggregate filters into WHERE for performance, since it reduces the row count before the expensive grouping step.
🔁 Follow-up Qs they might fire next
Q: Can WHERE and HAVING be used together in one query? — Yes — WHERE filters rows first, then GROUP BY runs, then HAVING filters the grouped output.
Q: Can HAVING be used without GROUP BY? — Technically yes, treating the whole table as one group, but it's rare and usually a sign the query should just use WHERE.
Q: Why is WHERE cheaper than HAVING for non-aggregate filters? — Because WHERE removes rows before the grouping/aggregation cost even happens, so there's less data to process.
3. Subqueries vs Joins (when each is better)
Subquery na — query kulla vera oru query irukும் (nested). Join na — rendு table ah nேraaga side-by-side combine pannுрது. Generally, join namакки rendு table oda columns um oru result la venும் bodhu best — optimizer um join ah nல்லா optimize pannும். Aana நான் oru row exist pannுthா illaiya nu மட்டும் check pannா poдும் bodhu — full join pannாma EXISTS subquery use pannுрேன், அது first match kிடைச்ச உடனே stop aagும், duplicate rows வரவே varாthு. Correlated subquery na — outer query oda column ah refer pannும் subquery, adhu ovvoru outer row кki ஒரு தடவை run aagும் — periya table ku slow ah aagலாம், so careful ah use pannanum.
🗣️ Interview la eppadi sollanumI use joins when I need columns from both tables in the final result — the optimizer handles those well. I reach for an EXISTS subquery when I only need to check whether a related row exists, because it short-circuits at the first match and avoids the row-duplication issues a join can cause. I'm cautious with correlated subqueries since they can execute once per outer row and get slow on large tables.
🔁 Follow-up Qs they might fire next
Q: IN vs EXISTS — which is better for large subquery result sets? — EXISTS usually, since it stops at the first match; IN has to materialize the full list to compare against.
Q: What's a correlated subquery, in one line? — A subquery that references a column from the outer query, so it logically re-runs for every outer row.
Q: Can a subquery ever be faster than a join? — Yes, for pure existence checks — EXISTS avoids fetching and duplicating matched columns that a JOIN would produce.
4. UNION vs UNION ALL (combining result sets)
UNION na rendு query result ah combine pannிட்டு, duplicate rows ah automatic ah remove pannும் (internally sort/distinct pannும், so konjam slow). UNION ALL na duplicates remove pannாma, rendும் oda mattum append pannிடும் — fast. Example: current students list + alumni list combine pannும் bodhu, same person rendு listla um வந்தா duplicate வேணாம் na UNION use pannலாம். Aana rendு vera log tables ah combine panniட்டு export pannும் bodhu duplicate irundா problem illa na, UNION ALL use panni speed edukkலாம்.
🗣️ Interview la eppadi sollanumUNION removes duplicate rows across the combined result sets, which means an implicit sort/distinct operation and a performance cost. UNION ALL just concatenates both result sets without checking duplicates, so it's faster. I default to UNION ALL unless I specifically know duplicates are possible and unwanted — for example combining current and archived attendance records for an annual report.
🔁 Follow-up Qs they might fire next
Q: Why exactly is UNION ALL faster? — It skips the distinct/sort step UNION needs to eliminate duplicates.
Q: Do the columns need to match between the two queries? — Yes — same number of columns, and compatible data types in the same order.
Q: Where have you used UNION in a real report? — Combining current-year and archived attendance tables into one annual attendance report.
5. CTE — Common Table Expression (WITH clause)
CTE na — WITH clause vachi, oru temporary named result set define pannுрது, appuram அந்த query ah namal name vecha reference pannலாம். Readability romba improve aagும் — periya nested subquery ah instead of writing, step-by-step name kொடுத்து break pannலாம். Real power na — recursive CTE, hierarchy data ku (example: Manager tree, Class-Section hierarchy) — anchor query + recursive query serthu use pannலாம். ஆனா CTE ah namal materialize pannாthு, adhு index um panna mudியாthு — every time reference panна bodhு andha underlying query rerun aagலாம்.
🗣️ Interview la eppadi sollanumA CTE is a named, temporary result set defined with WITH, used to break a complex query into readable steps or to write recursive queries — like walking a manager-employee hierarchy. It's not materialized as a physical object, so it can't be indexed, and SQL Server may re-evaluate it each time it's referenced, so for large reused intermediate results I'd use a temp table instead.
🔁 Follow-up Qs they might fire next
Q: CTE vs subquery — any real performance difference? — Mostly the same execution plan; CTE's main win is readability, except for recursive CTEs which subqueries can't do at all.
Q: Can you use a CTE recursively? — Yes — define an anchor member and a recursive member that references the CTE itself, useful for org charts or class hierarchies.
Q: Can a CTE be indexed? — No, since it's never materialized as a table — that's the key difference from a temp table.
6. Window Functions — ROW_NUMBER, RANK, DENSE_RANK, PARTITION BY (top-N per group)
Window functions na — GROUP BY madhiri rows ah collapse pannாma, ovvoru row кkum கூட ஒரு calculated value (rank, row number, running total) add pannுрது. PARTITION BY na — data ah logical groups ah split pannுрது (example: ClassId wise), OVER() கூட use pannுрோம். Real use case — "top scorer per class" report: ovvoru class ku, marks அடிப்படையில் ROW_NUMBER kொடுத்து, rn=1 ah filter pannினா top scorer கிடைக்கும். RANK na tie irundா next number ah skip pannும் (1,1,3), DENSE_RANK na skip pannாma (1,1,2).
WITH RankedMarks AS (
  SELECT StudentName, ClassId, Marks,
    ROW_NUMBER() OVER (PARTITION BY ClassId ORDER BY Marks DESC) AS rn
  FROM Marks
)
SELECT StudentName, ClassId, Marks
FROM RankedMarks
WHERE rn = 1;
🗣️ Interview la eppadi sollanumWindow functions let me compute a rank or row number per row without collapsing the result set the way GROUP BY does. I use PARTITION BY to reset the calculation per group — for example, ROW_NUMBER over Marks partitioned by ClassId, ordered descending, wrapped in a CTE and filtered to rn=1, gives me the top scorer per class in a single query instead of a correlated subquery per class.
🔁 Follow-up Qs they might fire next
Q: Difference between RANK and DENSE_RANK on ties? — RANK leaves a gap after ties (1,1,3), DENSE_RANK doesn't (1,1,2).
Q: Can you filter directly on ROW_NUMBER() in the same SELECT's WHERE clause? — No, window functions can't be used in WHERE directly — you have to wrap the query in a CTE or subquery and filter the outer one.
Q: Where else have you used this pattern? — Deduplicating records, like picking the latest attendance entry per student per day when duplicates exist.
5B Indexing & Performance Tuning
7. Clustered vs Non-Clustered Index (physical order vs pointer structure)
Clustered index na — actual table data ah, physical ah, andha index order layே arrange pannிடும் — book oda pages themselves order la vecha madhiri. Adhанால ஒரு table кki ஒரே ஒரு clustered index thaan possible (usually Primary Key mேல் automatic ah varும்). Non-clustered index na — vேra separate structure, actual data illama, "இந்த value இருக்கும் row idhோ" nu pointer mattum vечிருக்கும் — book oda back-side index page madhiri, page number kொடுக்கும், data adhே page la irukும். Oru table ku multiple non-clustered indexes create pannலாம்.
🗣️ Interview la eppadi sollanumA clustered index defines the physical storage order of the table itself, so there can only be one per table — I usually put it on the primary key, an ever-increasing narrow column, to avoid page fragmentation. A non-clustered index is a separate B-tree structure that just points back to the row, and a table can have several of them tuned to different query patterns.
🔁 Follow-up Qs they might fire next
Q: How many clustered indexes can a table have? — Only one, because it physically orders the data — you can't order the same table two different ways at once.
Q: What happens if a table has no clustered index? — It's stored as a "heap" — unordered data, generally slower lookups without a good non-clustered index to help.
Q: Why pick a narrow, ever-increasing column for the clustered key? — A wide or random key causes page splits and fragmentation every time a new row is inserted out of order; a narrow, increasing key like an identity column keeps inserts efficient.
8. How he finds slow queries (execution plan, scan vs seek)
Naan first SSMS la "Include Actual Execution Plan" on pannிடுவேன், query run pannினа apparam plan ah paakுவேன். Table Scan illana Clustered Index Scan na — full table ah padikுthु nu artham (periya table ku romba bad). Index Seek na — directly index வெச்சு venும் rows ah seek pannிடும் (good). Estimated vs Actual row count la periya difference irundா, statistics outdated ah irukும் nu suspect pannுவேன். Production la — Query Store illana sys.dm_exec_query_stats vечி, top CPU/duration edukkும் queries ah kandுpiடிப்பேன், overhead illama.
🗣️ Interview la eppadi sollanumI start with the actual execution plan in SSMS — a Table Scan or Clustered Index Scan means SQL Server is reading every row, while an Index Seek means it's using an index efficiently. I also check for a big gap between estimated and actual row counts, which usually points to stale statistics. In production, I use Query Store or sys.dm_exec_query_stats to find the top time-consuming queries without adding tracing overhead.
🔁 Follow-up Qs they might fire next
Q: Difference between Table Scan and Index Seek in plain terms? — Scan reads every row to find matches; Seek navigates directly to the matching rows using the index's B-tree structure.
Q: What tool finds the top slow queries in production without adding overhead? — Query Store, or sys.dm_exec_query_stats — both use existing SQL Server metadata rather than live tracing.
Q: What does a "Key Lookup" in the plan mean, and is it bad? — It means the non-clustered index found the matching rows but had to go back to the clustered index for extra columns — a sign the index isn't covering the query; I'd consider adding INCLUDE columns.
9. Covering Index (no key lookup needed)
Oru query ku venும் ella columns um (SELECT la வேணும், WHERE la வேணும், ORDER BY la வேணும்) — andha ellame ஒரே non-clustered index kூடа இருந்தா, adha "covering index" nu sொல்லுவோம். Adhு irundா SQL Server clustered index ku back pona (Key Lookup) தேவை இல்லை, andha index onnுல மட்டும் full answer கிடைக்கும். INCLUDE clause vachi, key columns ah touch pannாma, extra columns ah leaf level la mattum add pannலாம் — sort order affect aagாthु.
🗣️ Interview la eppadi sollanumA covering index contains every column a query needs — filter, join, and select columns — so SQL Server can answer entirely from the index without a Key Lookup back to the clustered index. I use the INCLUDE clause to add extra output columns to the index leaf level without making them part of the key, keeping the seek predicate narrow while still avoiding the lookup.
🔁 Follow-up Qs they might fire next
Q: Difference between key columns and INCLUDE columns? — Key columns define the index's sort order and seek predicate; INCLUDE columns just ride along at the leaf level for covering, without adding sort/storage overhead in the tree.
Q: Any downside to covering indexes? — A wider index means more storage and more write overhead on every insert/update, so I only cover the queries that actually matter.
Q: How do you spot a covering index working in the plan? — No Key Lookup operator shows up — just a clean Index Seek.
10. Over-indexing tradeoffs (read speed vs write cost)
Every index um, ஒரு INSERT/UPDATE/DELETE nadakும் bodhு, adhуkkும் extra ah update aaganum — so index adhikama irundா, writes slow aagும். Attendance table madhiri, dhinamும் thousands of rows insert aагும் table la over-indexing pannினா, andha inserts romba slow aagும். So balance venும் — WHERE/JOIN/ORDER BY la frequently use aagுrа columns kku mattum index podுவேன், unused indexes ah sys.dm_db_index_usage_stats vечி kandுpiடிச்சு drop pannுவேன்.
🗣️ Interview la eppadi sollanumEvery index adds overhead on writes, since it has to be maintained on every insert, update, and delete — so I don't index everything blindly. I check sys.dm_db_index_usage_stats to see which indexes actually get seeks or scans versus which just soak up update cost with zero reads, and drop the unused ones. On write-heavy tables like attendance, I keep indexes minimal; on read-heavy report tables, I add more.
🔁 Follow-up Qs they might fire next
Q: How do you find unused indexes? — sys.dm_db_index_usage_stats — compare seeks/scans/lookups against updates; high updates with zero reads is a drop candidate.
Q: Does over-indexing hurt SELECT performance too? — Indirectly — the optimizer has more index options to evaluate — but the bigger, more direct cost is always on the write side.
Q: Real example from your project? — Attendance table gets heavy daily inserts, so I kept it to essential indexes only; billing/report tables that are read far more than written got more indexes.
11. Stored Procedures vs inline SQL (performance + security angle)
Performance angle — stored procedure oda execution plan cache aagும், next time call pannும் bodhு andha plan reuse aagும், parse/compile overhead kammi. Security angle — SP la mattum parameterized calls aagும், application ku direct table access தேவையில்லை, so SQL injection risk um kammiaagும். Business logic ah DB level la centralize pannினா, maintain panna easy aagும் — especially heavy report/aggregation logic ku.
🗣️ Interview la eppadi sollanumStored procedures get a cached, reusable execution plan and avoid the repeated parse/compile cost of ad-hoc inline SQL. They also naturally use parameterized calls instead of direct table access from application code, which reduces SQL injection surface. I moved our heaviest reporting queries — attendance and fee summaries — into stored procedures so the logic executes close to the data instead of being built dynamically in the application layer.
🔁 Follow-up Qs they might fire next
Q: Doesn't EF Core/Dapper with parameters also get plan caching? — Yes, parameterized queries from either get cached plans too — but a stored procedure gives more explicit control and can bundle multiple statements or transaction logic together server-side.
Q: Any downside to stored procedures? — Splitting logic between the app and DB layer makes it harder to version-control and unit test, so I reserve them for heavy reporting/aggregation, not routine CRUD.
Q: Where did you use SPs in Zen Campus specifically? — Attendance and fee-collection reports — both involve heavy joins and aggregation across several tables.
12. Real Story — the 40% query time reduction (STAR walkthrough)
Situation: Zen Campus la report screens — especially attendance report — romba slow ah irundhுchு, dashboard load pannும் bodhு users wait pannуவaanga. Task: namба team ku andha response time ah kammi pannuрது ஒரு priority ஆanachू. 🗣️ Interview la eppadi sollanumOne of our attendance reports initially took around 5 seconds. I profiled it with the execution plan and found a table scan caused by a missing composite index on the filter columns. Across similar reports and queries, that combination of indexing, stored procedures, and Dapper/EF Core tuning is what drove the overall 40% query execution time reduction I mention on my resume.
🔁 Follow-up Qs they might fire next

Q: What if the index alone hadn't been enough? — I'd also look at rewriting the query to be sargable, avoiding SELECT *, and pre-aggregating or caching data that doesn't change often, like static lookup lists.
13. Query optimization checklist (day-to-day habits)

(2) WHERE/JOIN/ORDER BY la use aagுрa columns ku correct index irukுthா nu check pannுрேன். (3) Execution plan la scan varுthா nu paakுрேன். (8) Heavy logic ku stored procedures. (9) Periodic ah statistics update pannுрேன், fragmented indexes rebuild/reorganize pannுрேன்.
🗣️ Interview la eppadi sollanumBefore writing a query, I avoid SELECT * and only pull what's needed, make sure WHERE/JOIN/ORDER BY columns are indexed, and keep predicates sargable so the optimizer can actually use those indexes — no functions wrapped around indexed columns. I prefer set-based logic over cursors, always parameterize, paginate large result sets, push heavy aggregation into stored procedures, and periodically check for index fragmentation and outdated statistics.
🔁 Follow-up Qs they might fire next
Q: What's wrong with using a function on an indexed column in WHERE? — It makes the predicate non-sargable, so SQL Server can't seek using the index and falls back to a scan.
Q: What does "sargable" mean? — Search ARGument ABLE — a predicate written so the optimizer can directly use an index seek instead of evaluating a function per row.
Q: How often do you rebuild or reorganize indexes? — Depends on fragmentation from sys.dm_db_index_physical_stats — I reorganize around 10-30% fragmentation and rebuild above that, usually scheduled during low-traffic hours.
5C Transactions, Concurrency & Constraints
14. ACID Properties (Atomicity, Consistency, Isolation, Durability)
Atomicity na — oru transaction full ah succeed aаганум illana full ah rollback aаганும், nடுவில் oru part mattum apply aагக்கூடாது (fee payment insert pannினா, receipt entry um serthу insert aагனும், rendும் aагனும் illana rendும் poganும்). Consistency na — DB ஒரு valid state la irundு vera valid state кku mattum pogும், constraints violate aагாmல். Isolation na — concurrent ah run aагுрa transactions ஒன்னை ஒன்னு affect pannாma irukanum. Durability na — commit ஆன apparam, server crash ஆனாலும் andha data poganаthु.
🗣️ Interview la eppadi sollanumACID stands for Atomicity — a transaction either fully completes or fully rolls back, Consistency — the database moves between valid states without violating constraints, Isolation — concurrent transactions don't interfere with each other, and Durability — once committed, the change survives a crash because SQL Server writes it to the transaction log first. A real example from my locker project: OTP validation and its audit log insert are wrapped in one transaction, so both succeed or both roll back.
🔁 Follow-up Qs they might fire next
Q: Give a real example where atomicity mattered in your work. — OTP validation plus the audit trail insert in the bank locker system — both had to succeed together in one transaction.
Q: Which ACID property does the isolation level control? — Isolation — it determines how much of another transaction's uncommitted or committed changes a transaction can see.
Q: How does SQL Server guarantee durability? — Write-ahead logging — changes are written to the transaction log before the commit is acknowledged, so a crash can be recovered from the log.
15. Transaction Isolation Levels (Read Committed default + when he'd change it)
Read Committed — SQL Server default, dirty reads (uncommitted data padikுрது) illa, aana non-repeatable reads possible. Read Uncommitted — dirty reads even allow pannும், fastest but risky, rarely use pannுрேன். Repeatable Read — transaction full ஆаgும் varைilла andha read pannина rows ah lock pannும். Serializable — strictest, phantom reads கூட block pannும், aаna locking அதிகம் so throughput kammiaагும். Snapshot — row versioning vачி, blocking illама consistent read கொடுக்கும். Naan default Read Committed la thaan velai pannுрேன் aаna heavy reports run aагும் bodhு, ongoing writes ah block pannாma/block aагாma irukка Snapshot isolation ku switch pannுவேன்.
🗣️ Interview la eppadi sollanumRead Committed is SQL Server's default, and it's fine for most of my day-to-day CRUD — no dirty reads, decent concurrency. I'd move to Snapshot isolation for heavy reporting queries so they get a consistent view without blocking or being blocked by ongoing writes, and I'd only reach for Serializable in rare cases where phantom rows during a transaction would actually cause a correctness problem, knowing it costs throughput.
🔁 Follow-up Qs they might fire next
Q: What's a dirty read? — Reading another transaction's uncommitted data, which might later get rolled back — so you read something that never officially existed.
Q: When would you use Snapshot isolation specifically? — For reporting/dashboard queries that shouldn't block or be blocked by concurrent writes — it uses row versioning instead of locks.
Q: What's a phantom read? — New rows matching your query's criteria appear when you re-run the same query within one transaction — Serializable is the level that prevents it.
16. Deadlocks (what causes them, how he's handled one)
Deadlock na — rendு transactions, ஒன்னு vेра ஒன்னு vech irukка resource ah lock panniட்டு, மறு transaction release panна wait pannிட்டு irukும் — circular wait situation. SQL Server automatic ah idha detect pannிட்டு, roll back panna kammi cost aагுра transaction ah "deadlock victim" ah kill pannும் (error 1205). Naan handle panninadhu eppadi na — ella stored procedures லும் tables ah consistent order la access pannும்படி maатrினேn (example: always Students first, appuram Fees), transactions ah short ah vечேன், appropriate indexes vechு lock scope kammi pannினேn, and application code la deadlock error ah catch panni retry logic add pannினேn.
🗣️ Interview la eppadi sollanumA deadlock happens when two transactions each hold a lock the other needs, creating a circular wait — SQL Server detects this and kills the cheaper transaction as the "victim." I've handled these by making sure all procedures touch tables in a consistent order, keeping transactions as short as possible, adding indexes to shrink the lock footprint, and wrapping critical calls in retry logic that catches error 1205 and retries once.
🔁 Follow-up Qs they might fire next
Q: How does SQL Server pick the deadlock victim? — Usually whichever transaction is cheapest to roll back, unless you've explicitly set a deadlock priority.
Q: How do you debug a deadlock after it's happened? — SQL Server Profiler or Extended Events deadlock graph shows exactly which resources and queries were involved.
Q: What's a practical code-level way to prevent deadlocks? — Access tables/rows in the same order everywhere, keep transactions short, and never wait on user input or an external call inside an open transaction.
17. Primary Key vs Foreign Key vs Unique vs Check (constraints)
Primary Key — row ah uniquely identify pannும், NULL allow pannாthу, oru table ku ஒரே ஒரு PK. Foreign Key — vேrе table oda PK ah reference pannும், referential integrity maintain pannும் (Fees table la StudentId irundா, andha StudentId Students table la exist pannанும்). Unique constraint — duplicate values allow pannаthu, aаna PK madhiri illама oru NULL row allow pannலாm. Check constraint — specific condition satisfy pannанум (example: Marks BETWEEN 0 AND 100, FeeAmount >= 0).
🗣️ Interview la eppadi sollanumPrimary Key uniquely identifies each row and never allows NULL — one per table. Foreign Key enforces referential integrity by pointing to a Primary Key in another table, so I can't insert a fee record for a student that doesn't exist. Unique constraints prevent duplicate values but do allow one NULL, unlike a Primary Key. Check constraints enforce a business rule directly in the schema, like marks staying between 0 and 100.
🔁 Follow-up Qs they might fire next
Q: Can a table have multiple unique constraints but only one primary key? — Yes exactly — one PK per table, but any number of unique constraints.
Q: What happens deleting a parent row that's referenced by a foreign key? — Fails by default, unless ON DELETE CASCADE (or SET NULL) is configured — protects referential integrity.
Q: Give a Check constraint example from your project. — Fee amount must be greater than or equal to zero, and attendance status must be one of a fixed set of allowed values.
18. Optimistic vs Pessimistic Concurrency Control (lock now vs check later)
Pessimistic concurrency — row ah edit pannும் bodhே lock pannிடுவோम், vேrе யாरும் andha row ah touch panna mудியாது வரерை — ATM machine madhiri, oru transaction use pannும்बोदhु, vேrе யாरும் andha account ah touch panna mудியாthу. Optimistic concurrency — lock pannாma edit pannுрோm, save pannும் bodhு mattum "idha edhavadhு vera oruthar change pannினार்களா" nu rowversion/timestamp column vechு check pannுрோm — conflict irundா reject pannிடுவोm, user ku retry sொல்லுவोम். EF Core default ah optimistic concurrency use pannுрதु, concurrency token vечि.
🗣️ Interview la eppadi sollanumPessimistic concurrency locks a row the moment you start editing, so no one else can touch it until you're done — like an ATM session. Optimistic concurrency doesn't lock upfront; it lets edits happen freely and checks a rowversion/timestamp at save time — if the row changed since you read it, the update is rejected and the user retries. I generally prefer optimistic since most of our data has low contention, and EF Core supports it directly with a concurrency token.
🔁 Follow-up Qs they might fire next
Q: How do you implement optimistic concurrency in SQL Server directly? — Add a rowversion column, include it in the UPDATE's WHERE clause, and if zero rows are affected, that means someone else already changed the row.
Q: When would you prefer pessimistic locking instead? — High-contention scenarios where conflicts are frequent and retries would be wasteful — like a seat or time-slot booking system.
Q: Does EF Core support optimistic concurrency out of the box? — Yes — mark a property with [Timestamp] or configure it as a concurrency token, and EF Core throws a DbUpdateConcurrencyException on conflict.
19. Triggers (what they are, why he's cautious)
Trigger na — oru event (INSERT/UPDATE/DELETE) nadaकும் bodhு automatic ah fire aагுра oru piece of code. Audit logging ku useful, example — fee record edit aаgும் bodhு, old value ah automatic ah audit table la log pannலாm, edhа application-level code miss pannினалும். Aаna naan careful ah use pannுрேn, edhуnала — trigger hidden ah run aагுрhu, developer manual ah paartha thavira teriyатு, debug panna kastam, performance hit varа vaाippу irukку, appuram nested triggers/recursive triggers ellame confusion aагум். So explicit ah application layer la audit logging pannுрதே prefer pannுрேन், unless DB level integrity ku hard requirement irukка.
🗣️ Interview la eppadi sollanumA trigger auto-fires on an INSERT, UPDATE, or DELETE without anyone explicitly calling it — I've used them for DB-level audit logging so no code path can bypass it. But I'm cautious with them because that logic runs silently — it's easy to forget it exists, hard to debug when something behaves unexpectedly, and nested triggers can cause confusing cascading effects. I prefer explicit logging in the application or stored procedure layer unless there's a hard requirement that the audit must fire no matter what touches the data.
🔁 Follow-up Qs they might fire next
Q: Why exactly are you cautious with triggers? — They run silently on every DML statement, are easy to overlook, and can cause hidden performance issues or cascading trigger chains that are hard to trace.
Q: Where would a trigger genuinely be the right tool? — Enforcing an audit trail at the database level that must fire regardless of which application or path modifies the data.
Q: What do you use instead when possible? — Explicit logging in the application/stored procedure layer, or SQL Server's built-in Change Data Capture or Temporal Tables for audit history.
5D Design, Scale & Real-World Ops
20. Normalization (1NF/2NF/3NF) vs Denormalization (tradeoffs)
Normalization na — data duplicate aагாmал், related data ah separate tables ku split pannுртு. 1NF — ovvoru column лум atomic value irукanum, repeating groups иருkkக்கூடாdhு. 2NF — composite key la, partial dependency irукக்கூடாthु (part of key mேлाமே depend aагுра column иருkkக்கூடாdhு). 3NF — transitive dependency иருkkக்கூடாthु, non-key column key mேлा mattum depend aаганum, vera non-key column mேла illa. Denormalization na — intentional ah konjam duplicate data vечுкkuрту, read-heavy dashboards/reports fast ah irukкaнும் nu — 5 table join pannாma, oru pre-joined summary table use pannுрோm.
🗣️ Interview la eppadi sollanumNormalization avoids data duplication by splitting related data into separate tables — 1NF means atomic column values, 2NF removes partial dependency on part of a composite key, and 3NF removes transitive dependency so non-key columns depend only on the key. I do intentionally denormalize for read-heavy dashboards — precomputed summary tables so I'm not joining four or five tables on every page load, but I keep transactional data fully normalized.
🔁 Follow-up Qs they might fire next
Q: Give a 3NF violation example. — Storing ClassTeacherName directly in the Students table — it depends on ClassId, not on StudentId, so that's a transitive dependency.
Q: When did you denormalize in Zen Campus? — Dashboard summary tables where joining 4-5 tables on every load was too slow — I used a periodically refreshed, precomputed summary table instead.
Q: Is denormalization always purely about performance? — Mostly yes for reads, but it trades off write complexity and risk of inconsistency, so I only apply it to read-heavy reporting data, never core transactional tables.
21. Temp Tables vs Table Variables vs CTEs (when to use which)
Temp table (#Temp) — actual table tempdb la create aагும், index podலாm, real statistics irукும் — periya intermediate result sets ku best. Table variable (@Table) — similar aаna SQL Server statistics maintain pannாthу (optimizer 1 row nu assume pannிடும்), so periya data ku bad execution plan varа vaаippу — small dataset/lookup ku fine. CTE — materialize aagаthу, just named subquery, oru query кku mattum exist pannும், readability/recursion ku best aаna repeated reference aana ovvoru тடवैум் re-evaluate aаgும் vaаippу irukку.
🗣️ Interview la eppadi sollanumFor a large intermediate result set in a report procedure, I use a temp table because it can be indexed and SQL Server keeps real statistics on it. Table variables skip statistics entirely — the optimizer assumes a small fixed row count — so they're fine for small lookup data but risky for anything large. A CTE isn't materialized at all — it's just a named subquery scoped to one statement, great for readability and recursion but re-evaluated on each reference.
🔁 Follow-up Qs they might fire next
Q: Which do you pick for a large intermediate result set in a report SP? — A temp table — it can be indexed and has real statistics, which matters a lot once row counts get big.
Q: Why can table variables cause bad execution plans? — No statistics maintained on them, so the optimizer assumes a low fixed row count, which can lead to poor plan choices on larger data.
Q: Can you index a CTE? — No — it's not a physical or temporary object, it only exists within that single query's scope.
22. Pagination in SQL Server (OFFSET-FETCH for large report lists)
Periya report list (thousands of students/fee records) full ah oru shot la load pannாma, page-by-page ah fetch pannுрту thaan pagination. SQL Server 2012+ la OFFSET-FETCH clause irукку, aana andha vech ORDER BY mandatory. Page 3, page size 20 na — first 40 rows skip panni, next 20 edukkanum. Important — ORDER BY column ku always index poданум, illana ovvoru page ku um full table scan panni sort pannிडும், deep pages ku slow aагும்.

FROM AttendanceReport
ORDER BY AttendanceDate DESC
OFFSET 40 ROWS FETCH NEXT 20 ROWS ONLY;
🗣️ Interview la eppadi sollanumFor large report grids I use OFFSET-FETCH with a mandatory ORDER BY — OFFSET skips the earlier pages' rows and FETCH NEXT pulls just the current page. I make sure the ORDER BY column is indexed, otherwise SQL Server still has to scan and sort the whole table for every page. I've used this for server-side paging in attendance and fee report grids instead of loading thousands of rows into the browser at once.
🔁 Follow-up Qs they might fire next
Q: Why is ORDER BY mandatory with OFFSET-FETCH? — Without a defined order, pagination isn't deterministic — the same "page 2" could return different rows on different calls.
Q: Downside of OFFSET-FETCH for very large offsets, like page 10000? — SQL Server still has to skip past all the prior rows, so deep pages get progressively slower — keyset pagination (WHERE Id > lastSeenId) scales better there.
Q: Where did you actually use this? — Attendance and fee report grids in the UI — server-side paging with OFFSET-FETCH instead of dumping thousands of rows to the client.
