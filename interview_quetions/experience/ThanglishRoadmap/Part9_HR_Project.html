18. How did you achieve zero unauthorized access in the locker system — security layers
Idhu "defense in depth" concept ah explain pannanum — oru layer mattum illama, multiple layers of security irundhaa thaan "zero incidents" nu confident ah sollalam. Layers: password + OTP (MFA), time-expiry on OTP, TLS/HTTPS encryption, parameterized queries (SQL injection prevention), audit logging (accountability). Ella layers um list pannu.
🗣️ Interview la eppadi sollanum"It wasn't one single thing — it was layered security. First, we required OTP-based multi-factor authentication on top of the base credential, so a password alone was never enough. Third, all communication — OTP delivery, validation, and the TCP link to the locker hardware — ran over TLS/HTTPS, so nothing was exposed in transit. Fourth, every database call used parameterized queries through Dapper and EF, so SQL injection wasn't a viable attack path. And finally, every access attempt — successful or failed — was logged with a full audit trail in SQL Server, so anomalies could be caught and reviewed. It's that combination, not any single control, that gave us zero unauthorized access incidents post-deployment."

2️⃣0️⃣ "Developed centralized logging & exception-handling middleware, reducing production incident resolution time by 35%"
Resume line

Before: ovvoru controller/service la separate try-catch blocks, log format consistent illa (oru place la plain text, vera place la different format), production la bug vandha, edho oru random log file scroll panni manual ah search pannanum, adhu time-consuming. After: oru single global exception-handling middleware ella microservices layum use pannurathu, structured log format (Serilog/similar) use pannuvom, correlation ID vachu ஒரே request full journey ah multiple services kudaakkum trace pannalam. 2️⃣1️⃣ "Built TCP socket communication for real-time bi-directional exchange between IoT locker hardware and backend server"
Resume line
Idhu la "en TCP, en HTTP illa" nu direct ah kekkalam — idhu answer panna ready ah irukanum. HTTP na request-response model — client kekkanum, apparam mattum server reply pannum, connection ஒரு quick round-trip ku mattum thiranthu irukkum. Aana locker hardware kuda — server "unlock pannu" nu command anுppanum, hardware immediate ah "unlock aagiten" nu confirm anுppanum, idha continuous ah, both directions la real-time ah pannanum. Adhukku HTTP ah baar baar puthu connection open/close panradhu overhead adhigam, latency um adhigam. TCP socket na ஒரே connection continuously open ah vachu, rendu side um edhu vேணuma anுppikittu irukkalam — real IoT hardware communication ku idhu thaan standard approach. Idha "why not HTTP" angle la explain pannunga, adhu thaan interviewer expect pannuvanga.
