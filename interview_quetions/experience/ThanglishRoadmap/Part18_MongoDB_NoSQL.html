18A Core Concepts & Data Modeling
1. MongoDB vs SQL Server (when to actually pick which)
Idhu mostly depend aagும் data shape eppadi irukku, appuram adha eppadi query pannுрோம் nu vachு thaan. Data clean relational ah irundhு, fixed columns, joins, foreign keys, multiple tables la ACID guarantee venும் na — adhு SQL Server territory. Aana data document madhiri — consistent shape illама, high volume writes (logs, activity feeds) na, MongoDB nalla fit aagும், edhуkки na schema fight pannама structure change panna mudiyும். Reasoning mostly "future pain" gurunchi thaan — relational DB la schema change aana every new field oru migration, aana Mongo la document ah evolve panna existing rows touch pannama mudiyum.
🗣️ Interview la eppadi sollanumI mostly look at how rigid the data shape is and the read/write pattern before picking. If it's relational and needs joins with ACID across tables — like payments or student records — I stay with SQL Server. If it's document-shaped and high-volume, like logs or analytics feeds, Mongo fits better because I'm not fighting a schema every time the structure changes. In Zen Campus, admissions, fee structures, and payroll stayed in SQL Server, while activity logs, device pings, and analytics data feeding dashboards moved to MongoDB — that split came out of actual pain, not a day-one plan, since we'd crammed too much into SQL Server first and the query load got messy.
🔁 Follow-up Qs they might fire next
Q: Idhу day-one decision ah irundhுchа? — Illa, ella first SQL Server la thaan potом், query load messy aana apparam MongoDB specifically andha unstructured, high-volume side ku introduce pannом்.
Q: Payment data ah Mongo la vaikkalама? — Kandippa vendaм், payment ku multiple tables mela ACID guarantee mukkiyaм், adhு SQL Server oda strength.
Q: Rendு database same app la use panniрathуkки name enna? — Adhуthан polyglot persistence — relational core SQL Server la, unstructured/high-volume side Mongo la.
2. Document Model & BSON (vs rows in a SQL table)
Document na — JSON madhiri object, aana adhу actual ah store aaguрathу BSON (binary JSON) form la. BSON la JSON-a vida athigaма data types support pannும் — dates, binary data, ObjectId ellам் proper type ah store aagும் (string ah convert pannி thirும்பা parse panna vendaм்). Innும் oru key difference — same collection kulla rendு documents same fields kூடa irukka vendியathில்ла, SQL table la ella rows-um same columns follow pannанும், aana Mongo la ovvoru document ku thani structure irukkலாம்.
🗣️ Interview la eppadi sollanumA document is basically a JSON-like object but stored as BSON, the binary form, which supports more types than plain JSON — dates stay actual dates, not strings I have to reparse. Each document can have its own set of fields, unlike SQL rows which all follow the same schema. We used this for analytics snapshots in Zen Campus — one document per event with nested metadata, timestamps, device info — if we'd normalized that into SQL Server it would've been four or five tables with joins just to reconstruct one event.
🔁 Follow-up Qs they might fire next
Q: BSON vs plain JSON — enna extra benefit? — BSON type information store pannும் (date is date, not string) + faster ah parse/traverse panna mudiyும், plain JSON text ah thaan.
Q: ObjectId nu edhу? — Mongo automatic ah generate pannுрa unique identifier, ovvoru document ku default primary key madhiri.
Q: Nested object எப்போ prefer pannுрிnga vs separate table? — Data naturally nest aagi, object + array of sub-objects madhiri irundhа, 3-4 tables la split pannாma document ah use pannуведн் நல்லது.
3. Embedding vs Referencing (schema design decision)
Embedding na — related data ah parent document kullaye nest pannுрathу (example: blog post kulla comments array). Referencing na — foreign key madhiri, ObjectId ah store pannitு separate ah lookup pannுрathу. Child data eppovум் parent kூடave read aagும், unbounded ah grow aagாthு na embed pannалாம் — one query, no join. Aana andha data periyathாка irundhா, shared aagi irundhா, illама independent ah update aagும் na reference pannанும் — illana document size limit hit aagுm் risk irukkு.
🗣️ Interview la eppadi sollanumEmbedding nests related data inside the parent document for fast, single-query reads — good when the child data is small, bounded, and always read together. Referencing stores an ObjectId and looks it up separately — better for large or independently-changing data. Honestly I got this wrong once on Zen Campus — I embedded log entries directly inside a parent event document thinking it'd be quick to read, but it grew way faster than expected and the documents got bloated, so I restructured it into its own collection with a reference back. That taught me to think about growth rate before embedding, not just read convenience.
🔁 Follow-up Qs they might fire next
Q: Andha mistake eppadi katupidichінga? — Log entries collection valaraadhу nu nenachін், aana unbounded ah append aagi document bloat aagсு, apparam separate collection ku migrate pannінеn்.
Q: Document size limit nu edhу? — Mongo la per-document 16MB limit irukkு, unbounded array embed pannina eventually adha hit pannலam்.
Q: Reference use pannina extra cost enna? — Oru extra round trip venும் andha related data fetch panna, embed pannina madhiri single query la kidaikка.
4. Indexing in MongoDB (how to decide what needs one)
SQL Server la eppadi indexing decide pannுрோмோ, adhе instinct thaan — filters, sorts, lookups la athikamа use aagுрa fields ku index podanum். Mongo la single-field, compound, multi-key (array ku) index ellам் support pannும். Index illама na, "collection scan" nadakkும் — chinna collection ku problem illa, aana scale aana bodhu romba slow. explain() run pannitு, query COLLSCAN (bad, full scan) or IXSCAN (good, index used) nu check pannுрோм் — SQL Server oda execution plan equivalent thaan idhு.
🗣️ Interview la eppadi sollanumSame instinct as SQL Server — I look at what fields are hit in filters, sorts, and lookups the most, and index those, whether single-field or compound. I use explain() to check if a query is doing a COLLSCAN versus an IXSCAN, which is basically Mongo's version of checking an execution plan. When we were optimizing the analytics side of Zen Campus, dashboard queries filtering by date range and category were slow because there was no compound index covering both — adding one cut the query time down noticeably.
🔁 Follow-up Qs they might fire next
Q: Everything index pannradhilalа edhukку problem? — Ovvoru index-um write speed konjам் slow pannுрோм் + storage edukkும், so actual query pattern paathu thaan index pannaனும், guesswork illa.
Q: Compound index la field order mukkiyамா? — Aama, most selective/equality field first, range field appuram — order matter pannும் query match aaguрathуkки.
Q: COLLSCAN paartha udane enna pannuvinga? — Andha filter fields mela index illa na add pannுрோм், appuram explain() thirும்ba run pannitு IXSCAN varும் varaikkும் confirm pannுрோм்.
5. Aggregation Pipeline ($match, $group, $project, $sort)
Aggregation pipeline na — oru series of stages ah chain pannுрathу — $match, $group, $project, $sort, $lookup — ovvoru stage previous stage oda output ah eduthу innум் transform pannும். Unix pipe madhiri, illама LINQ operations chain pannுрathу madhiri, aana idhу database side la thaan run aagும், memory ku ella data-yум் eduthу vaaraдhу. Raw documents ah pull pannitу C# la process pannுрathуkки badhाla idhу use pannுрathу performance ku thaan — filtering/grouping data mela close ah nadakkும்.
🗣️ Interview la eppadi sollanumIt's a series of chained stages — match, group, project, sort — where each stage transforms the output of the previous one, and it runs on the database side instead of pulling everything into memory. For an analytics view in Zen Campus, we needed counts of events grouped by day and category for a dashboard chart — I wrote a $match to narrow the date range, a $group to bucket by day, and a $project to shape the output. First version had too broad a $match and pulled way more documents than needed — fixing that taught me the biggest aggregation lesson: put your filters as early as possible in the pipeline.
🔁 Follow-up Qs they might fire next
Q: $match adhikaмா early la vaikка edhukку mukkiyaм்? — Documents ah early ah filter pannina, next stages (group, project) kammi documents mattும் process pannும், adhu performance ku romba help pannும்.
Q: $lookup nu edhу? — SQL join madhiri, rendு collections ah oru common field vechு combine pannுрோм் pipeline kulla.
Q: Aggregation vs plain find() query — eppo edhу use pannுрிнga? — Simple filter/lookup ku find() pothும், grouping/sum/count/report-style output venும் na aggregation pipeline thaan correct choice.
6. CRUD from .NET (MongoDB.Driver code pattern)

🗣️ Interview la eppadi sollanumI pull in the MongoDB.Driver package, get an IMongoDatabase from a MongoClient, then a typed IMongoCollection<T> for the document class — from there it's InsertOneAsync, Find with a filter builder, UpdateOneAsync, DeleteOneAsync. In Zen Campus, one of our microservices behind the Ocelot gateway talks to MongoDB for analytics and log data — I registered MongoClient as a singleton in Program.cs since it's thread-safe and meant to be reused, injected a typed repository wrapping the collection, and kept the structure close to how I'd built the Dapper repositories on the SQL Server side, just swapping the driver underneath.

7. CAP Theorem (where MongoDB fits)
CAP theorem sollுрathу enna na — distributed system la Consistency, Availability, Partition tolerance mூnrum் ஒரே நேரத்தில் full ah irukka mudியாthு. Network partition vandhа, consistency ah stay pannрathா illa availability ah stay pannрathா nu choose pannanum். MongoDB default ah consistency side ku thaan lean pannும் — reads default ah primary node kku thaan pогும், so latest data varும், aana read preference tune pannitு secondaries kூடa (konjum stale data) allow pannலаம்.
🗣️ Interview la eppadi sollanumCAP theorem says you can't fully have Consistency, Availability, and Partition tolerance all at once during a network partition — you have to pick between staying consistent or staying available. MongoDB defaults toward consistency since reads go to the primary by default, but you can tune read preferences toward availability if slightly stale reads are okay. I'll be honest — this is more theory for me than something I've actively tuned in production, our Zen Campus setup runs a fairly standard replica set and hasn't hit real partition scenarios, but I understand why the driver exposes read preference and write concern settings and I'd know where to look if we ever needed a multi-region setup.
🔁 Follow-up Qs they might fire next
Q: Write concern nu edhу? — Write operation eppo "successful" nu confirm pannுрроинga nu control pannுрathу — primary mattум் ack pannின்ா pothும் a, secondaries kூடa replicate aagும் varaikkும் wait pannанumа nu decide pannலam்.
Q: Zen Campus la CAP tradeoff face pannirукींga-а? — Ilai, standard replica set setup, real partition scenario force panniрillа — aana concept theriyும் dependent settings eppadi tune pannanum nu.
Q: Secondary node la read panna eppo good idea? — Read-heavy, slight staleness okay-na scenario ku (analytics dashboard madhiri), primary mela load kammi panna help pannும்.

8. Real-time Analytics/Logging (why not just log to SQL Server)
High-volume events — activity logs, request traces, continuous write stream — capture pannі dashboard ku feed pannанும் na, transactional database ah slow pannாma pannанும். Ovvoru log event same shape ah irukkату — sometimes extra metadata irukkும், sometimes illa — adha rigid SQL table kulла force pannina, nullable columns romba வேணும் illama separate metadata table + joins venum, adhу extra friction thaan mostly write-once, read-for-analytics data ku.
🗣️ Interview la eppadi sollanumEvery log event isn't identical — some have extra metadata, some don't — and forcing that into a rigid SQL table means a lot of nullable columns or a separate metadata table with joins, which adds friction for data that's mostly write-once and read-for-analytics. In Zen Campus, we integrated MongoDB exactly for this — unstructured, high-volume data feeding real-time analytics dashboards, plus centralized logging and exception-handling middleware across the microservices. Keeping that separate from the transactional load actually helped cut down incident resolution time too, since we could query logs fast without worrying about locking up production tables.

9. Polyglot Persistence (different DB per use case)
Polyglot persistence na — oru database ah ella velaikрum use pannாма, application kulla use-case ku thakka storage engine pick pannуведн். Relational data ku transactional, structured — SQL Server; flexible, high-volume data ku document store — MongoDB. No single database ella jobskூடa best ah irukаthу — SQL Server relational integrity, complex joins ku excellent, aana loosely-structured fast-growing event data ku athу natural fit illa.
🗣️ Interview la eppadi sollanumIt's the idea that no single database is great at everything, so you pick the right storage engine per use case within the same app — relational for transactional data, document store for flexible, high-volume data. That's basically the setup in Zen Campus — SQL Server for admissions, billing, payroll, all the relational core, and MongoDB alongside it for the unstructured, high-volume side like analytics and logs. It did add overhead — two connection strings, two sets of repositories, coordinating deployments for both in Docker Compose — but it genuinely solved a problem we were having when everything was crammed into SQL Server alone.

10. MongoDB Transactions (vs SQL Server transaction scope)
MongoDB version 4.0 la irundhு thaan multi-document ACID transaction support pannும் — session start pannitு, transaction begin pannitு, operations run pannitу, commit/abort pannுрோम். Adhуkки munnadi single-document level la mattум் atomicity guarantee irundhுchу. SQL Server transaction scope oda exact feel illa, aana concept similar thaan — multiple writes ah wrap pannitு ella-um success or ella-um rollback nu pannுрோм். Multi-document transaction Mongo la more overhead irukкும், so schema ah restructure pannitу cross-document transaction avoid pannuрathே general advice.
🗣️ Interview la eppadi sollanumMongoDB supports multi-document ACID transactions since version 4.0 using a session — start a session, begin the transaction, run operations, commit or abort — conceptually similar to a SQL Server transaction scope, though not the exact same feel. In Zen Campus, most of our transactional needs — payment processing, fee updates — stayed on the SQL Server side where we already had solid transaction handling through Dapper. We didn't lean on multi-document Mongo transactions much because the data we put in Mongo was mostly append-only analytics and log data, where single-document atomicity was already enough.

11. Sharding vs Replication
Replication na — same data oda multiple copies vaikkuрathу — replica set la oru primary, konjம் secondaries — mainly high availability + read scaling ku. Primary down aana, oru secondary automatic ah election mூlама primary aagும். Sharding na completely vera — data ah itself multiple servers kூடa split pannуведн் oru shard key vechு, so oru server kூடa full dataset irukaathу — adhу horizontal scaling ku, oru machine handle panna mudியாma poна.
🗣️ Interview la eppadi sollanumReplication is multiple copies of the same data via a replica set for high availability — if the primary goes down, a secondary gets elected automatically. Sharding is different — it splits the data itself across multiple servers using a shard key for horizontal scaling when one machine can't handle the volume. For Zen Campus, our MongoDB runs as a replica set for availability, but we haven't needed to shard yet — if usage kept growing across more institutions using the ERP, sharding by something like tenant or institution ID would probably be the next step, but we're not there yet.

12. Schema-less Risk (data quality issues)
Schema rigid ah illама irundhа, flexibility rendு side kூடa cut aagும் — document drift aaga chance irukku, oru field oru document la string, vera document la number, illама field-е missing (older code path adha set pannама irundhitа). 🗣️ Interview la eppadi sollanumWithout a rigid schema, documents can drift — a field being a string in one document and a number in another, or missing entirely if an older code path didn't set it. We hit exactly that on Zen Campus, where a couple of analytics documents had an inconsistent field type and it broke a query that assumed a consistent shape. After that incident, I added stricter model validation in the C# layer before writing to Mongo, and set up JSON schema validation on the collection itself, so Mongo rejects documents that don't match the expected structure at the database level, not just hoping the application layer catches it.

13. MongoDB Performance Tuning (real incident)
Zen Campus la analytics queries feeding oru dashboard, event volume valaruрa madhiri konjам் konjaм் slow aагіту — dramatic illa, aana instant-ah feel aana queries konjум் seconds edukкும் aachi. explain() run pannitு, filter fields index illama full collection scan nadakkும் nu paarthом். Compound index add pannінеn் actual filter pattern (date range + category) mela, appuram aggregation pipeline la $match stage ah early ah move pannінеn் — group pannрathуkки munnadi thaan filter aagும் madhiri.
🗣️ Interview la eppadi sollanumDashboard queries in Zen Campus started getting noticeably slower as the log and event volume grew — I ran explain() on the slow ones and saw full collection scans because the filter fields weren't indexed. I added a compound index on the fields the dashboard filters were actually using, date range plus category, similar to the SQL Server indexing work I'd already done with Dapper — that brought the query time down significantly. I also moved the $match stage earlier in the aggregation pipeline so it filtered before grouping, which cut down how many documents even had to be processed.
