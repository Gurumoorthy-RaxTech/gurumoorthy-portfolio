22A Fundamentals — Definitions & Core Differences
1. What is an Interface? (Pure Contract)
Interface na — oru class enna enna method/property/event kொடுக்கணும் nu ஒரு contract mattum define pannுрathு, implementation edhுவும் kொடுக்காது (C# 8+ la default methods possible aanaalum, andha style rare ah thaan use pannுрோம்). Interface la irukka every member ah, adha implement panра class compulsory ah define pannanum. Example — Zen Campus la IPaymentService interface la ProcessPayment() nu method signature mattum irukும், actual logic OnlinePaymentService and OfflinePaymentService classes la separate ah irukும்.
🗣️ Interview la eppadi sollanum"An interface is a pure contract — it defines what members a class must have, without providing any implementation. Any class that implements the interface is required to provide the actual logic for every member. In my project, IPaymentService just declares a ProcessPayment method signature — OnlinePaymentService and OfflinePaymentService each implement it with completely different logic."
🔁 Follow-up Qs they might fire next
Q: Can an interface have a body for a method in modern C#? — Yes, C# 8 introduced default interface methods with a body, mainly for adding new members to an existing interface without breaking implementers, but it's used rarely and not a replacement for abstract classes.
Q: What access modifier do interface members have by default? — Public by default and implicitly — you can't mark them private/protected in the classic style (default interface methods allow more modifiers now, but plain members stay implicitly public).
2. What is an Abstract Class? (Partial Implementation)
Abstract Class na — oru class direct ah instantiate panna mudiyaadha class, aana adhуla some methods fully implemented, some methods abstract (implementation illama, sub-class implement pannanum) nu irukும். Idhு "common code share pannі, specific parts mattum child class ku vittuduрathு" nு design pattern. En project la BaseRepository<T> abstract class la SaveChangesAsync() madhiri common logic already implement panniruken், aana GetTableName() madhiri abstract method ah ovvoru derived repository (StudentRepository, BillRepository) override pannanum.
🗣️ Interview la eppadi sollanum"An abstract class is a class that can't be instantiated directly and can mix fully implemented methods with abstract methods that derived classes must override. It's useful when related classes share common logic but each needs to customize a specific part. My BaseRepository<T> implements shared logic like SaveChangesAsync, while an abstract GetTableName method is overridden differently by StudentRepository and BillRepository."
🔁 Follow-up Qs they might fire next
Q: Can an abstract class have zero abstract methods? — Yes, technically valid — it just still can't be instantiated because of the abstract keyword on the class, even if every method is already implemented.
Q: Is it mandatory for a derived class to override every abstract method? — Yes, unless the derived class is itself declared abstract, in which case it can defer the obligation further down the chain.
3. Interface vs Abstract Class (The Core Difference Table)
Idhu thaan most-asked question — answer ah oru table madhiri clear ah nenachுkoonga, adhுkkаppuram sொllுngа. Implementation: Interface la illa (mostly), Abstract Class la partial iருkும். Multiple inheritance: Interface la multiple implement panna mudiyும், Abstract Class la ஒரே ஒரு base mattum. Fields: Interface la state/fields illa, Abstract Class la fully allowed. Constructor: Interface la illa, Abstract Class la iருkும் (aana direct instantiate panna mudiyaathu). Access Modifiers: Interface members default public, Abstract Class members private/protected/public எதுவும் ஆகலாம். Purpose: Interface — "can-do" capability contract; Abstract Class — "is-a" relationship + shared code.
🗣️ Interview la eppadi sollanum"The core differences are: an interface is a pure contract with no implementation and no state, and a class can implement multiple interfaces. An abstract class can mix implemented and abstract methods, has fields and a constructor, but a class can only inherit from one abstract class. I use interfaces for capability contracts across unrelated classes — like IPaymentService — and abstract classes when related classes share real code — like my BaseRepository."
🔁 Follow-up Qs they might fire next
Q: If you had to pick just one rule of thumb, what would it be? — "Is-a plus shared code" points to abstract class; "can-do across otherwise unrelated types" points to interface.
Q: Has this difference gotten blurrier with C# 8 default interface methods? — A bit — interfaces can now carry implementation — but interfaces still can't hold instance state (fields), which remains the sharpest practical distinction.
4. When would you use an Interface? (Decision Criteria)
Naan Interface ah use pannуரேன் — enakku unrelated classes (payment types, notification channels, repositories) mூlama same contract mattum venும், code share panna vேண்டாm nா. Especially DI (Dependency Injection) la, unit testing la mock panna venும் nā, and future la multiple implementations வேணும் nа (today Dapper, tomorrow EF Core maathanum nா) — andha ellame Interface use panра scenarios thaan.
🗣️ Interview la eppadi sollanum"I reach for an interface when I need a contract shared across otherwise unrelated classes without sharing implementation — for dependency injection so I can swap implementations at runtime, for unit testing so I can mock the dependency, and whenever I expect multiple, independent implementations of the same capability, like my IStudentRepository being backed by either Dapper or EF Core."
🔁 Follow-up Qs they might fire next
Q: Give one interface from your project and why it's an interface, not an abstract class. — IPaymentService — OnlinePaymentService and OfflinePaymentService share no common code at all, just the same contract, so an interface is the right fit.
Q: Do you always define an interface even for a class with only one implementation? — Mostly yes for anything crossing a service boundary or needing to be mockable in tests, even with one implementation today — it keeps the door open for DI and testing without a later refactor.
5. When would you use an Abstract Class? (Decision Criteria)
Abstract Class ah naan use pannுрேன் — related classes group ஒரே common code (fields, fully implemented methods) share pannуম் scenario la, aana சில specific parts ovvoru class ku வேறு வேறு ஆக இருக்கும் pothу. Example — namba ellame repository ku common ah SaveChangesAsync(), LogAudit() madhiri methods venும், aana GetTableName() mattum ovvoru repository ku வேறு. Andha "90% same, 10% different" scenario ku Abstract Class perfect fit.
🗣️ Interview la eppadi sollanum"I use an abstract class when a family of related classes shares a large amount of common code, but each needs to customize a specific piece. My BaseRepository<T> is a good example — SaveChangesAsync and audit logging are implemented once and shared, while GetTableName is abstract and different for every concrete repository. It avoids duplicating the shared 90% across every derived class."
🔁 Follow-up Qs they might fire next
Q: What breaks if you use an interface instead in this scenario? — You'd have to duplicate the shared implementation (SaveChangesAsync, logging) in every single derived class, since interfaces traditionally can't hold real shared code — a maintenance nightmare.
Q: Could you combine both — abstract class and interface — for the same hierarchy? — Yes, and it's common — an abstract class can implement an interface, giving you shared code plus a contract other unrelated types can also implement.
6. Can an Interface contain method implementation? (Default Interface Methods, C# 8+)
Classic rule — "Interface la implementation illa" — aana C# 8.0 (.NET Core 3.0+) la irundhு, default interface methods nu feature வந்திருக்கு, adhுla body kொடுக்கலாம். Idhu mainly library authors ku — existing interface ku pudhu method add pannினா, adhу implement panрра ellame classes ah break pannாma, default implementation kொடுத்து backward compatibility maintain pannуррதுкku thaan. Naan project la idha rarely thaan use pannுрேன் — most cases la interface la implementation vekkама, plain contract ah thaan வைக்கிறேன்.
🗣️ Interview la eppadi sollanum"Traditionally no, but since C# 8 and .NET Core 3.0, interfaces can have default interface methods with a body. It's mainly meant for library authors to add new members to an existing interface without breaking every class that already implements it. I rarely use this in application code — I keep interfaces as pure contracts and put real implementation in classes."
🔁 Follow-up Qs they might fire next
Q: If a class doesn't override a default interface method, what happens? — It just uses the interface's default implementation automatically — no compile error.
Q: Why don't you use default interface methods more in your own code? — I want interfaces to stay clear contracts — mixing in real logic blurs the interface-vs-abstract-class distinction and makes it harder to reason about what a class actually does.
7. Can an Abstract Class contain implemented methods? (Yes — Core Feature)
Ithuஎல்லேயே Abstract Class oda main purpose thaan — fully implemented (concrete) methods and abstract (no implementation) methods rendும் ஒரே class la mix panna mudiyும். Example — BaseApiController abstract class la HandleError() full ah implement panniருக்கேன் (common error response format), aana ValidateRequest() abstract ah vachु, ovvoru controller adha own validation logic ku override pannுрому.
🗣️ Interview la eppadi sollanum"Yes, that's the whole point of an abstract class — it can freely mix fully implemented methods with abstract ones that have no body. In my BaseApiController, HandleError is fully implemented with a standard error response shape, while ValidateRequest is abstract, and each derived controller provides its own validation logic."
🔁 Follow-up Qs they might fire next
Q: Can a derived class call the base class's implemented method directly? — Yes, via base.MethodName() — it's a normal implemented method, fully callable and reusable.
Q: Can it also have virtual (not abstract) methods with a default implementation that CAN be overridden? — Yes — abstract classes commonly mix all three: fully implemented (non-virtual), virtual with a default that can optionally be overridden, and abstract that must be overridden.
8. Can an Interface have constructors? (No)
Illaiiye — Interface ku constructor kிடையவே கிடையாது. Reason simple — Interface ku state (fields) illa, so initialize panna edhуvும் illa. Constructor concept "object create aagумbodhu state ah set panradhu" — Interface direct ah object aaga mudiyாthu, adhுla state um illa, so constructor concept idhுkku applicable illave illa.
🗣️ Interview la eppadi sollanum"No, interfaces cannot have constructors. A constructor's job is to initialize an object's state, but interfaces have no fields or state to initialize, and they can never be instantiated directly — so a constructor simply has nothing to do there."
🔁 Follow-up Qs they might fire next
Q: Can an interface have static members in modern C#? — Yes, C# 8+ allows static members including static constructors on interfaces for things like constants or default implementation helpers, but instance constructors are still not allowed.
Q: How do implementing classes then get initialized if the interface has no constructor? — Entirely through the implementing class's own constructor — the interface just guarantees the shape, the class owns all initialization.
9. Can an Abstract Class have constructors? (Yes — but never called directly)
Ambadiyும் — Abstract Class ku constructor iருkும், because adhуkku fields/state iருkும், adha initialize pannanum. Aana neenga new AbstractClassName() nа nேrடியaaga call panna mudiyாthu (compile error). Adha instead — derived (child) class oda constructor : base(...) mூlama automatic ah call pannும் pothу, andha abstract class constructor run aagும், common fields initialize aagும்.
🗣️ Interview la eppadi sollanum"Yes, an abstract class can have a constructor, because it can have fields that need initializing. You just can't call 'new' on the abstract class directly — the constructor only runs when a derived class's constructor implicitly or explicitly calls base(...), which initializes the shared state before the derived class's own constructor body runs."
🔁 Follow-up Qs they might fire next
Q: What's the execution order — base constructor or derived constructor first? — The base (abstract) class constructor always runs first, then the derived class constructor body executes.
Q: Can an abstract class constructor be private? — It can be, but then only nested classes within it could derive from it in a meaningful way — in practice abstract class constructors are almost always protected, since only derived classes should ever invoke them.
10. Can we create an object of an Abstract Class? (No — Direct Instantiation Blocked)
Illaiiye — new AbstractClassName() nu try pannінா compile-time error kொடுக்கும் — "Cannot create an instance of the abstract class". Aana neenga adha reference variable ah use panna mудியும் — BaseRepository<Student> repo = new StudentRepository(); madhiri, right side la concrete derived class object create pannінா, left side la abstract type reference ah vachуkkалাম். Idhу polymorphism oda basic thaan.
🗣️ Interview la eppadi sollanum"No, you can't directly instantiate an abstract class — new AbstractClassName() is a compile-time error. But you absolutely can use it as a reference type — BaseRepository<Student> repo = new StudentRepository(); — where the object on the right is a concrete derived class, and the abstract type on the left is just the reference variable's declared type, enabling polymorphism."
🔁 Follow-up Qs they might fire next
Q: How does the compiler enforce this? — The 'abstract' keyword on the class is a compiler-level flag — any 'new' expression targeting that type is rejected at compile time, not runtime.
Q: Is there any reflection-based way around this restriction? — No — Activator.CreateInstance on an abstract type also throws at runtime; the restriction is enforced by the CLR itself, not just the C# compiler.
22B Instantiation, Inheritance & Multiplicity
11. Can we create an object of an Interface? (No — Same Logic as Abstract Class)
Illaiiye — Interface ku definitely object create panna mudiyாthu, because adhуla implementation edhуvும் illa, so runtime la "enna code run panramும்" nu edhуvும் ille. Aana Abstract Class madhiriye, Interface ah reference type ah use pannலाம் — IPaymentService svc = new OnlinePaymentService();.
🗣️ Interview la eppadi sollanum"No — an interface can never be instantiated directly, since it has no implementation at all for the runtime to execute. Just like an abstract class though, it can absolutely be used as a reference type — IPaymentService svc = new OnlinePaymentService(); — the concrete implementing class is what actually gets created."
🔁 Follow-up Qs they might fire next
Q: Why is using the interface as the reference type useful here? — It lets calling code depend only on the contract, so I can swap OnlinePaymentService for OfflinePaymentService without changing any code that consumes IPaymentService.
Q: What does 'svc.GetType()' return at runtime in that example? — The actual concrete type — OnlinePaymentService — reflection always reports the real runtime type, not the compile-time reference type.
12. Can a class inherit multiple Interfaces? (Yes)
Aamा — C# la oru class multiple interfaces implement pannலாம் — class OnlinePaymentService : IPaymentService, IAuditable, IRetryable madhiri. Idhу thaan "multiple inheritance of type" — C# classes ku single class inheritance mattum allow pannினாலும், interfaces ku andha restriction illa. Idhу exactly namba ISP (Interface Segregation Principle) follow panna help pannும் — periya interface ku pathila, chinna chinna focused interfaces ah combine pannிக்கலாம்.
🗣️ Interview la eppadi sollanum"Yes, a class can implement any number of interfaces — class OnlinePaymentService : IPaymentService, IAuditable, IRetryable is perfectly valid. This is how C# gives you multiple inheritance of type even though it restricts you to a single base class. It also directly supports the Interface Segregation Principle — I can compose several small, focused interfaces instead of forcing everything through one bloated one."
🔁 Follow-up Qs they might fire next
Q: What if two interfaces have a method with the exact same signature? — The class provides one implementation and it satisfies both interfaces, unless you need different behavior per interface, in which case you use explicit interface implementation.
Q: Is there a practical limit to how many interfaces you'd implement on one class? — No hard limit, but if a single class is implementing many unrelated interfaces it's often a Single Responsibility Principle smell worth reconsidering.
13. Can a class inherit multiple Abstract Classes? (No)
Illaiiye — C# la (Java madhiriye) oru class ஒரே ஒரு base class mattum inherit panna mудியும் — adhу abstract class aayirundhalum, normal class aayirundhalum. "Multiple class inheritance" C# design ah avoid panniருக்கு — reason, "Diamond Problem" nа oru issue வரும் (rendு parent classes ku ஒரே method name irundhா, yaаru version ah use pannуррathு nु ambiguity). Andha problem ah avoid panna thaan, C# — single class inheritance + multiple interface implementation combination ah choose pannіருக்கு.
🗣️ Interview la eppadi sollanum"No — C# only allows single inheritance from one class, abstract or not. This is a deliberate design decision to avoid the diamond problem, where two parent classes might have conflicting implementations of the same member and the compiler wouldn't know which one to use. Instead, C# lets you inherit one base class and implement as many interfaces as you need to compose behavior."
🔁 Follow-up Qs they might fire next
Q: How would you work around needing 'two sets' of shared behavior? — Compose instead of inherit — bring in a second abstract class's behavior via a member field/interface (composition), or restructure the hierarchy so one abstract class covers both concerns.
Q: Does C# 8's default interface methods reintroduce the diamond problem? — Partially — if two interfaces provide conflicting default implementations, the implementing class is forced to explicitly resolve the conflict itself, so the compiler still won't silently pick one.
14. What is the purpose of an Interface? (Contract for Polymorphism)
Interface oda main purpose — loose coupling and polymorphism achieve panradhu. Adhу "type ah pathi kவலை padாma, capability ah pathi mattум் pேசு" nு design pannа help pannும். En code la controller IStudentRepository ah mattум் theriyum் — adhу Dapper vachा iருக்கா, EF Core vachа iருக்கா nu theriyaathu, theriyavum் venum் illa. Idhу thaan testability, extensibility, and DI ellame possible ah panрродhு.
🗣️ Interview la eppadi sollanum"An interface's core purpose is to enable loose coupling and polymorphism by letting code depend on a capability contract instead of a concrete type. My controllers only know about IStudentRepository — not whether it's backed by Dapper or EF Core underneath — which is exactly what makes dependency injection, unit testing with mocks, and swapping implementations later all possible without touching consumer code."
🔁 Follow-up Qs they might fire next
Q: Is 'loose coupling' the same idea as Dependency Inversion Principle? — They're closely tied — DIP is the design rule (depend on abstractions), and interfaces are the primary C# mechanism used to actually achieve that loose coupling in practice.
Q: What's the cost of overusing interfaces? — Extra indirection — for a class that will realistically only ever have one implementation and no test-mocking need, an interface can be unnecessary ceremony; I add one when there's a real reason (DI, testing, multiple implementations).
15. What is the purpose of an Abstract Class? (Shared Code + Partial Contract)
Abstract Class oda main purpose — code duplication ah avoid panni, related classes kூllа common logic ah ஒரே இடத்தில் share pannுрathு, aana specific parts ah customize panna room kொடுкkuррathு. Idhу "template method" madhiri thaan work pannும் — base class la overall flow/skeleton ah fix pannі, specific steps ah abstract methods ah vittu, derived classes அந்த steps ah override pannும்.
🗣️ Interview la eppadi sollanum"An abstract class's purpose is to centralize shared code across a family of related classes, while still leaving room for each derived class to customize specific parts. It's essentially the Template Method pattern — the abstract class fixes the overall skeleton and shared logic, while abstract methods are the customizable steps each derived class fills in differently."
🔁 Follow-up Qs they might fire next
Q: How is that different from just using composition/helper classes? — Composition avoids the tight is-a coupling of inheritance, but abstract classes are simpler when the shared logic genuinely belongs to a common lineage of related types, not just reusable utility code.
Q: Can you name the Template Method pattern explicitly in an interview? — Yes — it's worth naming directly: "this is the Template Method pattern" signals you know the formal design pattern behind the abstract class usage, not just the C# mechanics.
16. Which is faster, Interface or Abstract Class? (Performance Nuance)
Technically — Abstract Class method calls oru tiny bit fast aaganum (virtual table lookup neraiya direct), Interface method calls oru extra level of indirection (interface dispatch table) irukkалாம். Aana idhु — real-world applications la (web APIs, business logic) measurable ah irundhе illa — microseconds/nanoseconds level difference thaan, adha nееनga profile pannினாலும் kவலை படலाam। Idha main deciding factor ah nенachukoodadhu — design correctness (loose coupling vs shared code) thaan main criteria, performance illa.
🗣️ Interview la eppadi sollanum"Technically, abstract class method calls can be marginally faster than interface calls, because interface dispatch has one extra layer of indirection through an interface method table. But in practice, for typical business application code — web APIs, services, repositories — this difference is nanoseconds and never shows up as a real bottleneck. I choose between them based on design fit — loose coupling versus shared code — not micro-benchmarks."
🔁 Follow-up Qs they might fire next
Q: Have you ever actually had to care about this in production? — No — in three-plus years of production .NET work, this specific difference has never been the cause of a real performance issue I've had to fix; database and network calls dominate every time.
Q: When would this micro-difference actually matter? — Extremely hot paths called millions of times per second in tight loops — think low-level game engines or high-frequency trading — not typical CRUD/web API code.
17. Can an Abstract Class have private methods? (Yes)
Aamа — Abstract Class ku private methods fully allowed. Idhу useful — internal helper logic ah vачі, adha derived classes kூда expose panна vேண்டாm nа. Example — BaseRepository la private string BuildConnectionString() madhiri internal helper vачаலाம், derived classes adha access panna mудியாthu, aana public/protected methods internally adha call pannலाம்.
🗣️ Interview la eppadi sollanum"Yes, absolutely — an abstract class can have private methods, just like any normal class. I use them for internal helper logic that shouldn't be exposed or overridden by derived classes — for example, a private BuildConnectionString helper inside BaseRepository that only the base class's own methods call internally."
🔁 Follow-up Qs they might fire next
Q: Can a private method in an abstract class be abstract itself? — No — 'private abstract' is a contradiction and a compile error; abstract methods must be overridable, which requires at least protected visibility.
Q: Can derived classes call a base class's private method indirectly? — Only indirectly, by calling a public/protected method in the base class that internally uses the private helper — the private method itself stays completely inaccessible to derived classes.
18. Can an Interface contain properties? (Yes — but no backing field)
Aamа — Interface la properties define panna mудியும், aana adhуkku backing field kொடுக்க mудியாthু (state illaiye). interface IAuditable { DateTime CreatedAt { get; set; } } madhiri declare pannலाம், aana implement panра class thaan actual storage decide pannуரроது — auto-property ah, computed property ah, whatever. Interface just "idha property irukanum" nு specify pannுрродu mattум்.
🗣️ Interview la eppadi sollanum"Yes, interfaces can declare properties — like a CreatedAt DateTime property with a getter and setter — but they can't have a backing field, since interfaces hold no state. The implementing class decides how to actually back that property, whether that's an auto-property or a computed value."
🔁 Follow-up Qs they might fire next
Q: Can an interface property have only a getter, no setter? — Yes — you can declare a read-only property with just { get; }, and the implementing class can still choose to back it however it wants, including a computed expression-bodied property.
Q: Real example from your project of an interface property? — IAuditable exposing CreatedBy and CreatedAt so any entity implementing it — Student, Bill, Employee — guarantees those audit fields exist and can be handled generically in a shared audit-logging layer.
19. Can an Abstract Class contain fields/variables? (Yes — Full State Support)
Aamа — Abstract Class la normal fields/variables fully allowed — private, protected, public evanga vениumаналும் irukalaам். Idhу thaan Interface ah vida oru periya difference — Abstract Class ku real state iருkkuम், Interface ku illa. Example — BaseRepository abstract class la protected readonly AppDbContext _context field vачு, derived classes ellame andha same context ah access pannும்.
🗣️ Interview la eppadi sollanum"Yes — abstract classes fully support fields, with any access modifier, private, protected, or public. That's one of the sharpest differences from an interface, which can never hold state. My BaseRepository holds a protected readonly AppDbContext field that every derived repository inherits and shares."
🔁 Follow-up Qs they might fire next
Q: Why protected instead of private for that context field? — So derived classes can directly use it in their own query logic — private would hide it entirely, forcing an extra property/method just to expose it back out.

22. What happens if a class implements an Interface but doesn't define its methods? (Compile Error)
Compile-time error thaan varும் — "does not implement interface member". C# compiler strict ah check pannும் — interface la irukka every method/property ah, implement panра class definitely provide pannанும், illainaа build fail aagум். Exception — andha class kூда abstract nа declare panniருந்தா, remaining implementation ah adhу derived class ku pass pannலाam், adhуkku exception கிடையாது.
🗣️ Interview la eppadi sollanum"It's a compile-time error — the compiler enforces that every member declared in the interface must be implemented by the class. The only exception is if the implementing class itself is declared abstract, in which case it's allowed to leave the interface members unimplemented and defer that obligation to whichever concrete class eventually derives from it."
🔁 Follow-up Qs they might fire next
Q: Does this happen at runtime or compile time? — Strictly compile time — the code won't even build, unlike some languages where a missing method might only fail when actually called at runtime.
Q: What if the interface has a default implementation (C# 8+)? — Then it's fine to not override it — the class automatically inherits the interface's default implementation, and only members without a default must be explicitly implemented.
23. Difference between Abstract Methods and Virtual Methods (Must-Override vs Can-Override)
Abstract Method — implementation illaiye illa, derived class compulsory ah override pannанும். Virtual Method — default implementation already irukку, derived class venуманா override pannலam், venும் இல்லைனா andha default logic ah use pannலाam். Example — abstract void ValidateRequest(); — ovvoru controller thanoda own validation kொடுக்க vேண்டியathu. virtual void LogRequest() — base class la already default logging irukку, specific controller ku venும்நா mattum override pannலaाम்.
🗣️ Interview la eppadi sollanum"An abstract method has no implementation at all and forces every non-abstract derived class to override it. A virtual method has a default implementation already, and overriding it in a derived class is optional. I use abstract for things that genuinely differ per class, like ValidateRequest, and virtual for things with a sensible default that only occasionally needs customizing, like a base LogRequest method."
🔁 Follow-up Qs they might fire next
Q: Can an abstract method exist in a non-abstract class? — No — a class containing any abstract member must itself be declared abstract; the compiler enforces this pairing.
Q: What keyword does the derived class need when overriding either kind? — Both require the 'override' keyword in the derived class — the difference is entirely in the base declaration (abstract vs virtual), not in how it's overridden.

🔁 Follow-up Qs they might fire next

Q: Can the derived concrete class also implement additional interfaces beyond what the abstract base already implements? — Yes — a derived class can add any number of extra interfaces on top of whatever its abstract base class already implements.
25. Why do we use Interfaces in Dependency Injection? (DI's Core Mechanism)

Container la services.AddScoped<IStudentRepository, StudentRepository>() register pannினா, consumer classes IStudentRepository ah mattум் ask pannும், actual concrete class ah theriyaathu. 🗣️ Interview la eppadi sollanum"Interfaces are what actually makes dependency injection useful — the container maps an abstraction to a concrete type, like AddScoped<IStudentRepository, StudentRepository>(), and every consumer only ever asks for the interface. That gives me two big wins: I can inject a mock implementation in unit tests without touching a real database, and I can swap the underlying implementation in production without changing any consuming code."
🔁 Follow-up Qs they might fire next
Q: Could you do DI with an abstract class instead of an interface? — Yes, technically the container can inject an abstract-class reference too, but interfaces are preferred because they don't force an inheritance hierarchy and support multiple contracts per class.
Q: What happens if two implementations are registered for the same interface? — By default the last registration wins for a simple GetRequiredService call — if you genuinely need both side by side, .NET 8+ keyed services let you register and resolve each by an explicit key.
26. Which SOLID principle is related to Interfaces? (ISP + DIP)

Dependency Inversion Principle (DIP) — high-level modules concrete class ku pathila, Interface (abstraction) ku depend aaganum். Rendும் real projects la interfaces mூlама thaan practically implement pannрроம்.
🗣️ Interview la eppadi sollanum"Two SOLID principles directly involve interfaces — Interface Segregation Principle, which says split large interfaces into small, focused ones so clients aren't forced to implement methods they don't need, and Dependency Inversion Principle, which says high-level modules should depend on abstractions, not concrete classes. In practice, interfaces are the concrete C# mechanism I use to satisfy both."
🔁 Follow-up Qs they might fire next
Q: Give an ISP example from your project. — Instead of one giant IRepository with 15 methods, I split into IReadRepository and IWriteRepository so a read-only reporting service only depends on the query methods it actually needs.
Q: How do ISP and DIP work together in practice? — ISP shapes the interface to be small and focused, and DIP is the rule that says depend on that shaped interface rather than a concrete class — they reinforce each other.
27. What are the advantages of using Interfaces? (Summary)
Interface oda advantages — 1) Loose coupling — code concrete class ku pathila contract ku depend aagும். 2) Multiple inheritance of type — oru class palavidha contracts follow pannலam். 3) Testability — mock objects easy ah inject panna mудியும். 4) Flexibility — implementation maathinaаlum consumer code touch pannадha venடியirукkaathu. 5) Team collaboration — interface ah muன்na define pannінா, different developers parallel ah different implementations vela pannலाम்.
🗣️ Interview la eppadi sollanum"Interfaces give you loose coupling since code depends on a contract, not a concrete class; multiple inheritance of type, since a class can implement several interfaces; strong testability through mocking; flexibility to swap implementations without touching consumers; and better team collaboration — once an interface contract is agreed on, different developers can build separate implementations in parallel."
🔁 Follow-up Qs they might fire next
Q: Which of these has mattered most in your actual work? — Testability and swappability — being able to mock IStudentRepository in unit tests, and having actually swapped a Dapper implementation for EF Core once without touching any controller.

28. What are the advantages of using Abstract Classes? (Summary)
Abstract Class oda advantages — 1) Code reuse — common logic ஒரே இடத்தில் implement pannі, duplication avoid pannலाம். 2) Partial abstraction — some methods fully define pannі, some mattум் derived class ku vitrudalaам். 3) Fields/state support — shared state ah maintain pannலaाम். 4) Constructor logic — common initialization ஒரே இடத்தில் pannலaाम். 5) Template Method pattern — overall algorithm skeleton ah fix pannі, specific steps ah customize panna room kொடுкkalaам்.
🗣️ Interview la eppadi sollanum"Abstract classes give real code reuse across related classes, support partial abstraction where some methods are fully implemented and others deferred, allow shared state through fields, centralize common constructor initialization, and naturally support the Template Method pattern — fixing a shared algorithm's skeleton while letting derived classes customize specific steps."
🔁 Follow-up Qs they might fire next
Q: When has code reuse via abstract class saved you real effort? — BaseRepository<T> — adding a new entity's repository takes minutes now, since save/audit logic is already there; before that pattern, every new repository meant copy-pasting the same boilerplate.

29. What is Interface Segregation Principle? (SOLID "I")
ISP sொல்лுрродு enna na — client ah force pannакூடадு, adhуvுку thேவைப்படாத methods ah implement panна. Periya "fat" interface ku pathila, chinna chinna focused interfaces ah split pannі, ovvoru class thanoda thேவைya interfaces mattum implement pannанум். Example — namba IStudentRepository la read matum் write ellame ஒரே இடத்தில் irundhа, oru read-only reporting service kூда Delete() method implement panna force aagும் (empty body vачு, meaningless ah). Adha thead-off pannuрродுкку, IStudentReadRepository and IStudentWriteRepository nu split pannலaаம்.
🗣️ Interview la eppadi sollanum"Interface Segregation Principle says clients shouldn't be forced to depend on or implement methods they don't use. Instead of one large interface with everything, you split it into smaller, focused ones. If IStudentRepository had both read and write methods together, a read-only reporting service would still be forced to implement Delete with a meaningless empty body — splitting into IStudentReadRepository and IStudentWriteRepository avoids that."
🔁 Follow-up Qs they might fire next
Q: How is ISP different from SRP? — SRP is about a class having one reason to change; ISP is specifically about interface contracts not forcing unrelated methods onto every implementer — related, but ISP is scoped to contracts.
Q: What's a sign an interface violates ISP? — If an implementing class routinely throws NotImplementedException or leaves empty bodies for some methods, the interface is doing too much and needs splitting.

31. Can method overloading differ only by return type? (No)
Illaiiye — C# la method overloading ah decide pannுрроது, method signature (name + parameter types/count/order) vачு mattум், return type vачு illa. So int GetValue() and string GetValue() — parameter same ah (none) irundhа, rendும் return type mattум் வேறுபடும்பொழுது — compile error thaan varும், "already defines a member with the same parameter types".
🗣️ Interview la eppadi sollanum"No — method overloading in C# is resolved purely by the method signature, meaning the parameter types, count, and order. The return type is not part of the signature for overload resolution, so having two methods that differ only by return type is a compile-time error."
🔁 Follow-up Qs they might fire next
Q: How would you achieve two 'versions' that only differ in return type? — Give them different names, or add a differentiating parameter — e.g. TryGetValue(out int) alongside GetValue() — or use generics with a type parameter driving the return type.
Q: Is this true for overriding too? — Overriding is different — an override must match the base method's signature AND return type exactly (except for covariant return types added in C# 9 for reference types).
32. Can an abstract method exist in a non-abstract class? (No)
Illaiiye — oru class la abstract method iருந்தа, andha class definitely abstract nу declare pannाnும். Compiler rule idhу — "'ClassName.MethodName()' is abstract but it is contained in non-abstract class 'ClassName'". Reason logical ah nенachukoongа — abstract method ku implementation illa, adha class instantiate pannінா, andha method call pannினா enna nadaккும் nு clarity illa — so compiler already andha risk ah block pannуррродு.
🗣️ Interview la eppadi sollanum"No — if a class contains even one abstract method, the class itself must be declared abstract, or it's a compile error. This makes sense logically — an abstract method has no implementation, so if the class could be instantiated, calling that method would have nothing to execute, which the compiler simply won't allow."
🔁 Follow-up Qs they might fire next
Q: Can a sealed class have abstract methods? — No — sealed and abstract are mutually exclusive on a class; sealed prevents further inheritance while abstract requires it to complete the type.
Q: If a class is abstract, must it contain at least one abstract method? — No — a class can be marked abstract purely to prevent direct instantiation, even with zero abstract members, all its methods fully implemented.
33. How do you implement two Interfaces having the same method? (Explicit Interface Implementation)
Rendு scenario iருkkuम். Same behavior venும்na — oru normal method implement pannінா pothum், rendு interfaces ku同 same method ah automatically satisfy pannும். Different behavior venум்na (each interface ku வேறு logic) — explicit interface implementation use pannанும் — void IPrintable.Print() { ... } and void IExportable.Print() { ... } madhiri, method ah interface name vachu qualify pannанும். Idha call panна, object ah andha specific interface type ku cast pannі call pannанும் — ((IPrintable)obj).Print().
🗣️ Interview la eppadi sollanum"If both interfaces should share the same behavior, one plain method implementation satisfies both. If they genuinely need different behavior for the same method name, I use explicit interface implementation — qualifying the method with the interface name, like void IPrintable.Print(). To call it, you cast the object to that specific interface type first, since the method isn't visible on the class's own public surface."
🔁 Follow-up Qs they might fire next
Q: Why is the explicitly implemented method hidden from the class's public API? — By design — it forces callers to be explicit about which interface's version they mean, avoiding ambiguity when the same method name means two different things.
Q: Have you used explicit interface implementation in real code? — It's rare in day-to-day application code — mostly comes up when adapting a class to satisfy two third-party or framework interfaces that happen to collide on a method name.
34. What is a private constructor? (Controlling Instantiation)
Private constructor na — class ku constructor iருkும், aana adhуkku private access modifier vачி, class oda outside la iருந்து (or even derived classes la iருந்தும்) new ClassName() nு direct ah call panна mудியாtha maathi block pannுрроது. Use cases — 1) Singleton pattern — ஒரே ஒரு instance mattум் iருkka venும் nа. 2) Static-only classes — utility/helper class la static methods mattум் iருkkanum் nа, object create panna venడाм் nа. 3) Factory pattern — object creation ah oru static factory method mூlама mattум் control pannанும் nа.
🗣️ Interview la eppadi sollanum"A private constructor is a constructor marked private, which blocks any code outside the class from calling 'new' on it directly. I've used it for a couple of things — enforcing the Singleton pattern so only one instance can ever exist, static-only utility classes where instantiation makes no sense, and factory patterns where object creation should only happen through a controlled static factory method, not directly."
🔁 Follow-up Qs they might fire next
Q: Can an abstract class have a private constructor? — Technically yes, but it becomes uninstantiable even by derived classes unless they're nested within it, since derived classes need at least protected access to call base() — so it's rarely useful outside very specific sealed-hierarchy designs.
Q: How is a private constructor different from a static class? — A static class can never be instantiated at all and can't have instance members; a class with only a private constructor CAN still be instantiated from within itself, which is exactly what Singleton and factory patterns rely on.
35. If A obj = new B(); which overridden method gets called? (Runtime Polymorphism)
Idhу classic polymorphism question. If B class A ah inherit pannі, oru method ah override pannினா — obj.MethodName() call pannும்பொழுது, B oda overridden version thaan run aagும், A oda original version illa — even though compile-time reference type A thaan. Idhу thaan "runtime polymorphism" / "dynamic method dispatch" — actual method call decide aaguрродு, compile time type ah vачी இல்லை, runtime object type ah vачி. Aana andha method virtual/abstract/override keywords illama, normal method ah irundhа — adhу "method hiding" (new keyword), adhула compile-time type (A) thaan decide pannும், B oda version illa.
🗣️ Interview la eppadi sollanum"If B overrides A's method using the virtual/override keywords, then obj.MethodName() calls B's overridden version at runtime, even though the compile-time reference type is A — that's runtime polymorphism, resolved by the actual object type, not the reference type. But if the method isn't declared virtual/override — say B just uses 'new' to hide A's method — then it's method hiding, and the reference type A determines which version runs, calling A's version instead."
🔁 Follow-up Qs they might fire next
Q: Why does this matter practically? — It's exactly why my controllers can hold an IPaymentService reference and get the correct OnlinePaymentService or OfflinePaymentService behavior automatically at runtime — polymorphism is what makes DI-based swapping actually work.
Q: What's the difference between 'override' and 'new' on a derived method in one line? — 'override' replaces the base behavior polymorphically for all reference types; 'new' just hides it and only applies when accessed through the derived type's own reference.
