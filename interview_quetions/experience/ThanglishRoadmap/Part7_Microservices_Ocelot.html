7A Microservices Fundamentals
1. Microservices Architecture (What & Why)
Namba ZenCampus project ah first single project ah build pannirundha na, ellame ஒரே codebase la StudentService, BillService, HostelService, TransportService ellame kalandhu irundhurukum. Adha maathi, ovvoru domain kum thani thani service ah split pannirukom — StudentService student data mattum paakkum, BillService billing mattum paakkum, HostelService hostel mattum. Idhu oru "restaurant kitchen" madhiri nenachukonga — oru periya chef ella dish um oruthanaale panradhu maathi, separate stations (tandoor, curry, dessert) vachu, ovvoru station thani ah work pannum, ஒரு station busy na vera stations affect aagaathu. Idhu vachu namba team ku bill fix panna, student module ah touch pannama redeploy panna mudiyum — that's the real win.
🗣️ Interview la eppadi sollanumI designed ZenCampus as a microservices ecosystem instead of one large ASP.NET Core application — separate services for student, billing, hostel, transport, timetable, exam, payment, parent, and reporting, each with its own codebase, database access, and deployment pipeline. This gave us clean domain boundaries — a change in billing logic doesn't require touching or redeploying the student or hostel service. It also means each service can scale independently based on its own traffic pattern.
🔁 Follow-up Qs they might fire next
Q: How many microservices did you build? — Around 11-12: student, bill, transport, timetable, payment, hostel, dvr, exam, parent, report, staff, plus the gateway itself.
Q: Who decided the service boundaries — was it purely by domain? — Yes, domain-driven — each service maps to one ERP module/business capability, not by technical layer.
Q: Did you start with microservices from day one or migrate? — Started with the architecture already planned as microservices since the ERP had clearly separable modules from the requirement stage itself.
2. Monolith vs Microservices (honest tradeoffs)
Interview la microservices ah over-glorify panna koodathu — interviewer ku andha trap theriyum. Monolith na — ஒரே app, ஒரே deploy, debug panna easy (ஒரே log file, ஒரே process), aana chinna change ku kooda full app ah redeploy pannanum, oru module heavy traffic vandha whole app resource eat pannum. Microservices na — independent deploy/scale pannalam, aana network calls athigam aagum, oru request 4-5 services touch panna vaaipu irukku, so debug panna "distributed" ah aagidum — namma ஒரு log file paartha pothaadhu, correlation ID vachu trace pannanum. Chinna team ku monolith better, namba ERP madhiri periya multi-module system ku microservices worth panna complexity.
🗣️ Interview la eppadi sollanumMicroservices aren't automatically "better" — they trade simplicity for flexibility. A monolith is easier to debug since everything runs in one process with one log stream, and deployment is a single unit. Microservices give independent scaling and deployment, but you pay for it with network latency between services, distributed debugging, and the operational overhead of running and monitoring many services — which is why we invested in centralized logging and a gateway to manage that complexity.
🔁 Follow-up Qs they might fire next
Q: When would you NOT recommend microservices? — For a small team or a simple CRUD app with no independent scaling need — the operational overhead isn't worth it there.
Q: What's the hardest part of microservices in your experience? — Tracing a bug that spans multiple services — that's exactly why I built the correlation-ID-based centralized logging.
Q: Did you ever consider a modular monolith instead? — It's a valid middle ground, but given the ERP had clear independently-scalable modules like billing and attendance, full microservices fit better for us.
3. Database-per-Service Pattern (data ownership)
Microservices oda oru golden rule — ovvoru service kum thani DB/schema irukkanum, vera service andha DB ah direct ah touch pannakoodathu. Namba StudentService kitta student data irukku, BillService kitta bill data — BillService ku student name venum na, direct ah StudentService oda DB ku query pannama, StudentService oda API ah call pannanum. Idhu oru "office department" madhiri — HR department oda employee file cabinet ah, Finance department direct ah open panna koodathu, HR kitta request pannanum. Idhu vachu ஒரு service oda schema change pannalum, vera service break aagathu.
🗣️ Interview la eppadi sollanumEach service in our architecture owns its own data — StudentService's tables aren't queried directly by BillService or HostelService. If a service needs data owned by another service, it calls that service's API rather than reaching into its database. This keeps services loosely coupled — I can change StudentService's schema without breaking BillService, as long as the API contract stays the same.
🔁 Follow-up Qs they might fire next
Q: Do all your services use the same database engine? — Mostly SQL Server per service for relational ERP data, with MongoDB used where the data was more flexible/high-volume, like logs or analytics.
Q: What if two services need the same reference data, like a list of classes? — Either duplicate the minimal read-only copy per service, or have one service own it and others call its API — we lean toward the owning-service API approach.
Q: Isn't calling another service's API for every lookup slow? — It can add latency, so for frequently needed reference data we cache it locally in the calling service rather than calling on every request.
4. Inter-Service Communication (sync vs async)
Services ஒன்றுக்கொன்று pesanum na rendu vazhi irukku — synchronous (HTTP/REST — call panni immediate ah response wait pannurathu) illa asynchronous (message queue mூlama — call pannitu, andha service busy irundhalum namba block aagama continue pannurathu). Namba ZenCampus la — mostly synchronous HTTP calls thaan, adhuvum Ocelot gateway வழி route aagும். Example — PaymentServices oru fee payment process pannumbodhu, StudentService ah call panni "இந்த student valid ah?" nu check pannும். Message queue (RabbitMQ/Kafka) namba currently use pannala, aana notification/SMS madhiri "fire and forget" operations ku async approach best fit nu artham pannirukom.
🗣️ Interview la eppadi sollanumIn my project, services communicate primarily over synchronous HTTP/REST, routed through the Ocelot gateway — for example, PaymentServices calling StudentService to validate a student before processing a fee payment. We didn't have an async message broker like RabbitMQ or Kafka in this project, but I'm aware that for scenarios needing decoupling — notifications, SMS delivery, report generation — an async, event-driven approach would reduce tight coupling and improve resilience if one service is temporarily down.
🔁 Follow-up Qs they might fire next
Q: What's the downside of sync-only communication? — If the called service is slow or down, the caller blocks or fails too — it creates a chain of dependency, which is why circuit breakers matter.
Q: Would you introduce a message queue if you redesigned this today? — Yes, for events like "payment completed" or "student admitted" that multiple services care about, a pub/sub queue would decouple them better than direct HTTP calls.
Q: How do services find each other's addresses? — Through configuration in ocelot.json on the gateway side — downstream services are registered there rather than each service hardcoding others' URLs.
5. Independent Deployment & Horizontal Scaling (admission-season example)
Idhu microservices oda periya selling point. Namba school ERP la, admission season (June-July madhiri) varumbodhu, StudentService kitta huge traffic varum — parents online form fill pannuvanga, documents upload pannuvanga. Andha time la StudentService mattum extra containers ah spin up pannalam (horizontal scaling), TransportService illa ExamService ah touch pannama. Monolith irundha, full application ah scale pannanum — server resources waste aagum, edhukku na exam module ku andha traffic illa. Idhu oru "bus route" madhiri — rush hour la ஒரே route ku extra buses add pannலாம், vேறு routes ku touch pannama.
🗣️ Interview la eppadi sollanumBecause each service is independently deployable and containerized, during admission season when StudentService gets a traffic spike, I can scale just that service horizontally — run more container instances behind Ocelot's load balancer — without touching or redeploying TransportService, ExamService, or anything else. That's a direct cost and reliability advantage over a monolith, where you'd have to scale the entire application even though only one module is under load.
🔁 Follow-up Qs they might fire next
Q: How do you scale a service in practice — manually or automated? — In our setup it was manual scaling by running additional container instances during known high-traffic periods; auto-scaling with an orchestrator like Kubernetes would be the next step.
Q: Does Ocelot automatically detect new instances? — Ocelot routes to instances configured in its downstream host/port list, or via a service discovery integration — we managed it through configuration for known instances.
Q: What metric told you StudentService needed scaling? — Response time degradation and request volume from our centralized logging/monitoring during peak admission windows.
6. Data Consistency Across Services (eventual consistency)
Idhu oru tricky topic — monolith la ஒரே DB transaction kulla ellame commit/rollback pannalam. Microservices la ஒவ்வொரு service kும் thani DB irukka na, oru operation multiple services ah touch panna, "distributed transaction" nu oru periya headache varum. Example — student admission pannumbodhu StudentService la record create aagணும், appuram BillService la fee record create aagணும் — rendும் ஒரே "all or nothing" transaction ah panna mudியாது cross-service ah. So namba "eventual consistency" nu approach follow pannுrom — StudentService first create pannும், appuram BillService ah call pannும், adhu fail aana, retry logic illa compensating action (rollback student record illa flag panni "pending billing" nu vachu, manual/automated retry) vachu handle pannுrom.
🗣️ Interview la eppadi sollanumI don't rely on distributed transactions across services — there's no two-phase commit spanning StudentService and BillService's separate databases. Instead I accept eventual consistency: an operation like student admission first commits in StudentService, then calls BillService to create the fee record; if that downstream call fails, I handle it with retries and status flags rather than trying to roll back atomically across two databases. It's honestly one of the harder parts of this architecture, and I'm aware that a proper saga pattern or outbox pattern would make this more robust than what we currently have.
🔁 Follow-up Qs they might fire next
Q: What's a saga pattern? — A way to manage a multi-step business transaction across services using a sequence of local transactions plus compensating actions if a later step fails — I know the concept but haven't fully implemented it here.
Q: How do you currently detect a failed downstream step? — The calling service checks the response and logs failure with correlation ID; currently it's closer to manual reconciliation than a fully automated saga.
Q: Isn't eventual consistency risky for billing data? — Yes, that's exactly why I'd prioritize a proper outbox/retry mechanism for money-related flows if I were to harden this further.
7. Versioning Microservices (without breaking consumers)
Namba ஒரு service ah update pannுrom na, adha use pannுra vera services illa frontend break aagakoodathu. Example — StudentService oda API response la oru field remove pannினோm, aana BillService andha field expect pannுthu na — production la crash aagும். So namba URL level versioning follow pannுrom — /api/v1/students, /api/v2/students — old consumers v1 use panniட்டே irukanga, pudhu consumers v2 ku migrate aagலаம். Idhu oru "phone app update" madhiri — Play Store la force update pannாma, old version um konjam nாள் work aagும் madhiri vaikanum.
🗣️ Interview la eppadi sollanumI version APIs at the route level — for example /api/v1/students versus /api/v2/students — through Ocelot's routing, so a breaking change doesn't force every consumer to update simultaneously. Old consumers keep hitting v1 while new features roll out under v2, and I deprecate v1 only after confirming nothing depends on it anymore. For non-breaking changes like adding an optional field, I don't bump the version at all — I keep it additive.
🔁 Follow-up Qs they might fire next
Q: How do you decide if a change is breaking or not? — Removing or renaming a field, or changing a data type, is breaking; adding a new optional field is not.
Q: Do you version at the gateway or at each service? — Both stay in sync — Ocelot routes reflect the version, and the downstream service itself exposes matching versioned endpoints.
Q: How long do you keep an old version alive? — Until we confirm through logging that no client is calling it anymore, then we deprecate it with advance notice.
7B Ocelot API Gateway & Cross-Cutting Concerns
8. What is Ocelot (why it fronts everything)
Namba kitta 11+ microservices irundha, frontend/mobile app ovvoru service ku separate URL vachu call panna koodathu — confuse aagும், security exposure um athigam. Ocelot oru API Gateway — namba ellame SSMSGateWay nu ஒரே entry point வெச்சிருக்கோம், client Ocelot ku mattum call pannும், Ocelot andha request ah correct downstream service ku route pannும். Idhu oru "hotel reception" madhiri — guest direct ah kitchen/housekeeping ku pesamattanga, reception (front desk) ku pesuvanga, reception thaan andha request ah correct department ku pass pannும். Ocelot la routing, load balancing, rate limiting, authentication — ellame ஒரே இடத்தில் centralize aagும்.
🗣️ Interview la eppadi sollanumOcelot is our API gateway sitting in front of all 11-plus microservices — StudentService, BillService, HostelService, and the rest. The frontend never calls any downstream service directly; it only ever talks to the gateway's URL, and Ocelot routes each request to the right service based on configured routes. This gives us one place to handle cross-cutting concerns — authentication, rate limiting, load balancing, logging — instead of duplicating that logic in every service.
🔁 Follow-up Qs they might fire next
Q: Why Ocelot specifically and not something like YARP or Kong? — Ocelot is a lightweight, .NET-native gateway that integrates cleanly with ASP.NET Core and our JWT setup — it fit our stack without adding a separate infra dependency like Kong would.
Q: Is Ocelot itself a single point of failure? — It can be, so we'd run multiple gateway instances behind a load balancer in a fully hardened production setup — that's an area I'm aware needs attention.
Q: Does Ocelot do any business logic? — No, it stays thin — routing, auth checks, rate limiting — actual business logic always lives in the downstream services.
9. ocelot.json Route Configuration (Upstream/Downstream)
ocelot.json file la, ovvoru route ku rendு main part irukku — UpstreamPathTemplate (client namba gateway ku enna URL kூப்பிடுவாரோ) and DownstreamPathTemplate (andha request actual ah endha service/port ku pogaணும்). Example — client /students/5 nu gateway ku kூப்பிட்டா, Ocelot config paathu adha StudentService oda internal port ku forward pannும். Idhu oru "call center IVR" madhiri — neenga "1 for billing" press pannினீங்க na, andha call automatic ah billing department ku route aagும், neenga direct ah billing number theriinjukanum nu venaam.
🗣️ Interview la eppadi sollanumEvery route in ocelot.json maps an UpstreamPathTemplate — the URL the client calls on the gateway — to a DownstreamPathTemplate plus a downstream host and port for the actual microservice. So a client hitting /students/5 on the gateway gets transparently routed to StudentService running on its own port, without the client ever needing to know that internal address.
{
  "Routes": [
    {
      "UpstreamPathTemplate": "/students/{everything}",
      "UpstreamHttpMethod": [ "Get", "Post", "Put", "Delete" ],
      "DownstreamPathTemplate": "/api/students/{everything}",
      "DownstreamScheme": "https",
      "DownstreamHostAndPorts": [
        { "Host": "studentservice", "Port": 5001 }
      ],
      "AuthenticationOptions": {
        "AuthenticationProviderKey": "Bearer"
      }
    }
  ]
}
🔁 Follow-up Qs they might fire next
Q: What does {everything} mean in the template? — It's a catch-all placeholder that forwards any remaining path segments and route values from upstream to downstream unchanged.
Q: How do you manage separate ocelot.json per environment? — We use environment-specific config files, like ocelot.Production.json, merged at startup similar to appsettings pattern.
Q: What happens if a route isn't defined in ocelot.json? — The gateway returns a 404 — Ocelot only forwards requests that match a configured route, which is also a safety boundary against unintended access.
10. Load Balancing via Ocelot (multiple instances)
Oru service kitta traffic athigam irundha, adha ஒரே instance handle panna kashtam. So namba andha service ah 2-3 instances ah run pannுrom (Docker containers), Ocelot config la ellame instances ah list pannுrom, Ocelot round-robin (illa least connection) algorithm vachu traffic ah split pannும். Idhu oru "toll booth" madhiri — ஒரே lane la ellame vandi wait panna vaikama, multiple lanes open panni, traffic evenly distribute pannுவது.
🗣️ Interview la eppadi sollanumWhen a service has multiple running instances, I list all of them under DownstreamHostAndPorts in ocelot.json and set a LoadBalancerOptions type — Ocelot supports round-robin and least-connection strategies out of the box. This spreads incoming traffic across instances instead of overloading a single one, which matters a lot for a heavily-hit service like StudentService during peak load.
🔁 Follow-up Qs they might fire next
Q: Which load balancing strategy did you use? — Round-robin for most services — simple and predictable given our instance counts were small.
Q: What if one of the instances goes down? — Ocelot's built-in load balancer doesn't do active health checks by default in the community edition, so a downed instance can still get traffic and fail — that's a known gap I'd address with a proper health-check-aware balancer or orchestrator.
Q: Does load balancing work with Docker Compose scaling? — Yes conceptually — you'd scale replicas via Compose/Kubernetes and update Ocelot's host list, though a service discovery mechanism handles this more dynamically than static config.
11. Rate Limiting in Ocelot (protecting services)
Namba API ah oru client (bug irundha script illa attacker) over-use pannா, andha service down aagிடும், adhுla நம்ப ella users ku ம் affect aagும். Rate limiting na — "ஒரு client ku per minute/second இவ்வளவு requests mattum" nு limit vaikுவது. Ocelot la RateLimitOptions config pannி, Limit, Period, PeriodTimespan set pannலாம் — over aana Ocelot 429 (Too Many Requests) thaninu client ku reply pannும், downstream service ku andha request pogादhu.
🗣️ Interview la eppadi sollanumI configured RateLimitOptions per route in ocelot.json — setting a request limit within a time period, like 100 requests per minute per client. If a client exceeds that, Ocelot returns a 429 Too Many Requests response directly, without the request ever reaching the downstream service. This protects services from being overwhelmed by a misbehaving client, a retry storm, or an abusive script.
🔁 Follow-up Qs they might fire next
Q: How do you identify a "client" for rate limiting — IP or token? — Ocelot can key by client ID header or by the caller's IP; for authenticated endpoints, tying it to the client identity is more accurate than IP alone.
Q: What limit did you set and why? — It's tuned per endpoint — a heavier reporting endpoint gets a stricter limit than a lightweight lookup endpoint, based on expected legitimate usage.
Q: Does rate limiting replace the need for authentication? — No, they solve different problems — auth controls who can access, rate limiting controls how much even a legitimate authenticated caller can hit in a window.
12. Authentication/Authorization at Gateway vs Per-Service (where to check)
Namba JWT token validation ah Ocelot gateway level la centralize pannுрom — ellame route ku AuthenticationOptions la AuthenticationProviderKey vachு, token valid ah nu first gateway thaான் check pannும். Token invalid na, downstream service ah kூடa touch pannாma gateway அங்கேயே 401 kudukкும். Aana role-based authorization (example — "இந்த endpoint Admin mattum access pannலாம்") — adha namba downstream service level la [Authorize(Roles="Admin")] vachு check pannுрom, edhуkки na andha business-specific rule ovvoru service ku வேறு வேறு ah irukும். So — authentication centralized (gateway), fine-grained authorization decentralized (per-service).
🗣️ Interview la eppadi sollanumToken validation — checking the JWT signature, issuer, and expiry — happens centrally at the Ocelot gateway using its AuthenticationOptions, so an invalid or missing token never even reaches a downstream service. Fine-grained, role-based authorization — deciding whether this specific user's role can hit this specific endpoint — I handle at the individual service level with [Authorize(Roles = "...")], since those rules differ per service and per endpoint. That split keeps the gateway generic while services own their own business-specific access rules.
🔁 Follow-up Qs they might fire next
Q: If the gateway already validates the token, why re-check anything downstream? — The gateway just confirms the token is valid; the downstream service still needs to decide what that specific valid user is allowed to do there.
Q: Can a downstream service be called bypassing the gateway? — In a properly locked-down deployment, downstream services would only accept traffic from the gateway's internal network, not be publicly exposed — that's the ideal, and something worth double-checking in any production setup.
Q: How do claims like role flow from gateway to service? — The JWT itself carries the claims; Ocelot forwards the token as-is to the downstream service, which reads the claims from the same token.
13. Centralized/Correlated Logging Across Microservices (the 35% story)
Idhu namba resume la irukka biggest number — "35% incident resolution time reduction". Story sollுрோm: Munnadi, ovvoru service kூடa thani thani ah exception handle pannிக்கிட்டு, thani log format la log pannிக்கிட்டு irundhுchu. Production la oru bug வந்தா, "இந்த request எங்கே fail aachu" nு StudentService log paakணும், appuram BillService log paakணும், appuram PaymentServices log paakணும் — ரொம்ப time waste aagும். Fix pannினோம் — ovvoru service kூடaम் shared middleware vachு, ஒரே structured log format la log pannுрோம், appuram Ocelot gateway la request varும்bodhு ஒரு unique Correlation ID generate panни, adha header la vечு ella downstream calls kூடaம் pass pannுрோம். So oru request StudentService → BillService → PaymentServices nu 3 services touch pannினா, ella logs kூடaம் same correlation ID irukும் — adha search pannினா, full journey ஒரே place la kidaikும். Idhu thaான் 35% reduction ku main reason.
🗣️ Interview la eppadi sollanumBefore this, every service had its own scattered try-catch blocks with inconsistent log formats, so tracing one failing request across three or four services meant manually stitching together logs from each service separately — that ate a lot of debugging time. I built a shared exception-handling and logging middleware used across every microservice, and paired it with a correlation ID generated at the gateway and passed through every downstream call. So when a production issue comes in, I search by that one correlation ID and get the entire request's journey across services in a structured format — that's what cut our incident resolution time by 35%.
🔁 Follow-up Qs they might fire next
Q: Where does the correlation ID get generated and how is it propagated? — Ocelot generates or forwards it at the gateway as a custom header, and each service's middleware reads it, includes it in every log line, and forwards it on any further downstream calls it makes.
Q: What logging library/format did you use? — Structured logging with a consistent JSON-ish format so logs are searchable and filterable by fields like correlation ID, service name, and severity, rather than parsing free-text.
Q: How did you actually measure the 35% number? — By comparing average time from incident report to root-cause identification before and after rolling out the centralized middleware, tracked against our incident log history.
14. Handling Partial Failures (one service down)
Microservices la oru periya risk — oru service down aana, adha depend aagுра vேறு services கூட fail aaka koodathு, illena முழு system கூட viழுந்துடும். Example — PaymentServices, StudentService ah call panнி student valid ah nு check pannுрара, StudentService down aana, PaymentServices andha call kூடa fail aagும் — namba proper timeout illama irundha, PaymentServices ம் kூடa hang aagி whole system slow aagும். Honest ah sollанும்na — namba currently full circuit-breaker pattern (Polly library) implement pannிடல, aana timeout config pannி, downstream call fail aana graceful error return pannуரோம். Idhு oru "traffic signal" madhiri — signal work pannாதுna, andha junction la ellame stop aagாma, police manual ah handle pannுрார் — full system deadlock aagாma oru fallback irukаணும்.
🗣️ Interview la eppadi sollanumHonestly, we handled this at a basic level — HTTP client timeouts so a downstream service being slow or down doesn't hang the calling service indefinitely, and graceful error responses instead of unhandled exceptions bubbling up. We haven't implemented a full circuit-breaker pattern with something like Polly in this project, but I'm aware that's the right next step — wrapping downstream calls with retry-with-backoff and a circuit breaker so a struggling service gets given time to recover instead of being hammered with retries, and the caller fails fast instead of hanging.
🔁 Follow-up Qs they might fire next
Q: What is a circuit breaker, in your own words? — It's a pattern that stops calling a failing downstream service after a threshold of failures, "opens the circuit" to fail fast for a cooldown period, then tests with a few calls before fully closing again.
Q: How would you add Polly to your existing HttpClient calls? — Register Polly policies — retry, circuit breaker, timeout — on the HttpClientFactory registration for each service's outgoing client, so it wraps every call automatically.
Q: What's your fallback when a downstream service is unavailable right now? — The caller returns a clear error to the client rather than hanging, and we rely on alerting from the centralized logs to catch it quickly for manual intervention.

🗣️ Interview la eppadi sollanumEvery microservice has its own Dockerfile, and I use Docker Compose to define all of them — plus SQL Server and MongoDB — as services in a single YAML file with a shared network. That means the entire stack, all 11-plus microservices and databases, comes up with one docker-compose up command, both on my local machine and in staging. It eliminated environment inconsistency issues, since every environment runs the exact same images with the exact same dependency versions, and it made onboarding a new developer to the project dramatically faster.
🔁 Follow-up Qs they might fire next
Q: How do services in Compose talk to each other by name? — Docker Compose creates a shared network where each service is reachable by its service name as the hostname, which is what we reference in ocelot.json's downstream host settings.
Q: How do you manage environment-specific settings like connection strings in Compose? — Through environment variables and .env files referenced in the compose file, so the same compose definition works across environments by swapping variable values.
Q: Would you move from Docker Compose to Kubernetes eventually? — Yes — Compose is great for local dev and small deployments, but for auto-scaling, self-healing, and rolling updates in production, Kubernetes would be the natural next step.
