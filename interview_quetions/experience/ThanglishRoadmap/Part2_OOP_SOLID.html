2A OOP Core — 4 Pillars, Inheritance, Polymorphism, Interfaces
1. 4 Pillars of OOP (Overview)
OOP nu solradhu edhuvo periya theory illa — namba real world la irukka objects ah (Student, Payment, Employee) nenaichu, adhukku matching ah code ah design panradhu thaan. 4 pillars irukku: Encapsulation — data ah safe ah private ah vachu, controlled ah access panradhu. Abstraction — complex logic ah hide panni, simple interface mattum kaataradhu. Inheritance — oru class vera class oda code ah reuse pannuradhu. Polymorphism — same method name, but different classes la different ah behave pannuradhu. Naan Zen Campus la ivanga 4 pillars ah daily use pannuren — especially IPaymentService interface la Polymorphism, Repository pattern la Abstraction.
🗣️ Interview la eppadi sollanum"OOP has four pillars — Encapsulation for protecting internal state, Abstraction for hiding complexity behind a simple contract, Inheritance for reusing behavior across related classes, and Polymorphism for letting one interface behave differently across implementations. In my ERP project, I use all four together — for example my IPaymentService interface with OnlinePayment and OfflinePayment implementations is a live example of abstraction plus polymorphism."
🔁 Follow-up Qs they might fire next
Q: Which pillar do you use the most in your project? — Polymorphism and Abstraction together, mostly through interfaces — like IPaymentService and repository interfaces.
Q: Is C# a 100% pure OOP language? — Not fully — it has primitive types (int, bool) which aren't objects in the pure sense, and static members/methods break strict OOP too, so it's "mostly OOP" not pure.
Q: Can you name a real class from your project for each pillar? — Encapsulation: private fields in my repository classes; Abstraction: IStudentRepository; Inheritance: base ApiController; Polymorphism: IPaymentService implementations.
2. Encapsulation (Deep-Dive with Example)
Encapsulation na — data (fields) ah private ah vachu, outside world adha direct ah touch panna mudiyaathu, property or method mூlama மட்டும் access pannanum nu artham. ATM machine madhiri nenachukonga — உங்க balance amount ah neenga direct ah edit panna mudiyaathu, "Withdraw" button (method) mூlama மட்டும் touch panna mudiyும், adhுக்குள்ள validation (sufficient balance iruka nu check) irukkும். Zen Campus la BillService la, invoice amount field ah private ah vachu, ApplyPayment() method mூlama மட்டும் update pannurom — so negative amount or invalid state la balance போகாது.
🗣️ Interview la eppadi sollanum"Encapsulation means keeping fields private and exposing controlled access through properties or methods, so the class itself enforces its invariants. In my billing microservice, I never expose the balance field directly — I expose an ApplyPayment method that validates the amount before mutating state, so it's impossible to push the object into an invalid state from outside."
🔁 Follow-up Qs they might fire next
Q: Why not just make everything public and simple? — Because then any code anywhere could set invalid values directly — no validation, no control, bugs become very hard to trace back.
Q: Difference between private field and public property with private setter? — Both restrict external writes, but a property can add validation logic in the setter, expose computed values, or be part of an interface — a raw private field can't do any of that.
Q: Is encapsulation only about access modifiers? — Mostly about access control, but also about bundling data and the methods that operate on it together in one unit — that's the real "capsule" idea.
3. Abstraction vs Encapsulation (Commonly Confused)
Idhu interview la ellarukkum confuse aagum topic. Simple ah sollanum na — Abstraction is about "what" vs "how" — user ku enna panna mudiyum nu mattum kaatti, adhu eppadi internal ah work aaguthu nu hide pannuradhu (example: car drive panradhu, steering/accelerator use pannalam, aana engine internal ah eppadi work aaguthu nu therinja thevai illa). Encapsulation is about protecting data — access ah restrict panni, controlled ah mattum modify panna vidradhu (wallet la PIN irukka madhiri). Abstraction = design level la complexity hide panradhu (interface/abstract class vechu). Encapsulation = implementation level la data safety (private fields vechu). Rendume ஒண்ணா தெரியும் aana purpose வேற வேற.
🗣️ Interview la eppadi sollanum"Abstraction is about hiding implementation complexity and exposing only what's needed — it's a design-level concept, typically done through interfaces or abstract classes. Encapsulation is about protecting data by controlling access to it — it's an implementation-level concept, done with access modifiers. I'd say abstraction answers 'what can this do', encapsulation answers 'who can touch the internals'."
🔁 Follow-up Qs they might fire next
Q: Can you have abstraction without encapsulation? — Technically yes if fields are public, but that's bad practice — real-world abstraction usually relies on encapsulation underneath to actually hide the internals.
Q: Give one line to remember the difference forever. — Abstraction hides complexity from the user, encapsulation hides data from the outside world.
Q: Which one does an interface achieve? — Mainly abstraction — it declares "what" without exposing "how", it has no data to encapsulate.
4. Inheritance — Types (Single / Multilevel / Hierarchical)
Inheritance na — oru class (child), vera class (parent) oda properties/methods ah reuse pannuradhu, "is-a" relationship. 3 types: Single — oru parent, oru child (Employee → Manager). Multilevel — chain madhiri (Employee → Manager → SeniorManager) — grandparent to grandchild. Hierarchical — oru parent-ku multiple children (Employee → Teacher, Employee → Driver, Employee → Accountant). C# la classes ku multiple inheritance kidayathu (rendu parent class inherit panna mudiyathu) — confusion (diamond problem) avoid panradhukku — aana interfaces ah multiple implement pannalam.
🗣️ Interview la eppadi sollanum"C# supports single, multilevel, and hierarchical inheritance for classes. Single is one parent-one child, multilevel is a chain across generations, and hierarchical is one base class with multiple derived classes — like a base Employee class with Teacher, Driver, and Accountant derived from it. C# deliberately doesn't allow multiple class inheritance to avoid the diamond ambiguity problem, but a class can implement multiple interfaces instead."
🔁 Follow-up Qs they might fire next
Q: Why doesn't C# support multiple inheritance for classes? — To avoid the diamond problem — if two parent classes have a method with the same signature, the compiler can't decide which one the child should use.
Q: How do you achieve multiple-inheritance-like behavior in C#? — By implementing multiple interfaces on one class, or using composition — injecting multiple dependencies instead of inheriting from multiple classes.
Q: What's the base keyword used for? — To explicitly call the parent class's constructor or a specific overridden method from the child class.
5. Polymorphism (Compile-time vs Runtime)
Polymorphism na "many forms" — same method name, but different ah behave pannuradhu. Rendu types: Compile-time (Overloading) — same method name, different parameters, compiler thaan decide pannum which method call aaganum nu (example: Calculate(int a, int b) vs Calculate(int a, int b, int c)). Runtime (Overriding) — base class la virtual method irukkும், derived class la override panradhu, actual object type based ah runtime la decide aagும் (example: IPaymentService.Pay() — OnlinePaymentService and OfflinePaymentService rendும் Pay() ah different ah implement pannும், aana call panra code ku "IPaymentService.Pay() call panren" nu mattum theriyும், actual eppadi run aagும் nu runtime la decide aagும்).
🗣️ Interview la eppadi sollanum"Compile-time polymorphism is method overloading — same method name, different signatures, resolved by the compiler at build time. Runtime polymorphism is method overriding through virtual and override, resolved based on the actual object type at runtime. My IPaymentService interface is a real example — OnlinePaymentService and OfflinePaymentService both implement Pay() differently, and my controller just calls IPaymentService.Pay() without caring which one runs — that decision happens at runtime based on what got injected."
🔁 Follow-up Qs they might fire next
Q: Difference between overriding and hiding (new keyword)? — Override participates in runtime polymorphism through the virtual table; new just hides the base method at compile time based on the reference type, not the actual object.
Q: Can you overload based on return type alone? — No — C# requires the parameter list to differ; return type alone isn't enough to distinguish overloads.
Q: Why is runtime polymorphism also called dynamic binding? — Because the actual method to execute is determined dynamically at runtime by looking at the object's real type, not the variable's declared type.
6. Interface vs Abstract Class (When to Pick Which)
Interface na — "enna panna vendum" nu contract mattum sollும், "eppadi panna vendum" nu implementation kொடுக்காது, oru class multiple interfaces implement pannலாம். Abstract class na — konjam common logic already implement pannirukும், konjam abstract ah vittu child class fill pannanum, aana single inheritance mattum. Naan எப்போ interface pick pannுவேன் na — IPaymentService madhiri, completely different implementations irukkும்போது (Online vs Offline payment — logic vera vera, common code onnum illa). Abstract class pick pannுவேன் na — common code share pannanum, but konjam customization venum (example: BaseMicroserviceController la common error-handling logic already irukkும், specific controllers adha inherit panni extra logic add pannும்).
🗣️ Interview la eppadi sollanum"I pick an interface when I need a pure contract with no shared implementation, and the class might need to implement multiple contracts — that's exactly why IPaymentService is an interface, since online and offline payment logic have nothing in common. I pick an abstract class when there's real shared logic to inherit along with the contract, and single inheritance is fine — like a base controller with common exception handling that specific controllers extend."
🔁 Follow-up Qs they might fire next
Q: Can an interface have a method body in C#? — Since C# 8, interfaces can have default implementations, but I still treat interfaces as pure contracts and put real shared logic in abstract classes or helper services.
Q: Can an abstract class have a constructor? — Yes, and it runs when a derived class is instantiated, mainly used to initialize shared fields — but you still can't instantiate the abstract class directly.
Q: If I have to choose one default, which would you pick? — I'd lean interface-first, since it keeps things loosely coupled and testable — I only reach for an abstract class when there's genuine shared code to avoid duplicating.
7. Composition vs Inheritance ("Favor Composition" Principle)
Inheritance is "is-a" relationship (Manager is-a Employee). Composition is "has-a" relationship (Car has-a Engine — Engine ah separate object ah vachu, Car adha use pannும், Car Engine இல்ல illa). Problem enna na — inheritance over-use pannினா, "fragile base class" problem varும் — base class la maathinā, ellame derived classes um affect aagும், tight coupling. So industry la "favor composition over inheritance" nu sollுவாங்க. Zen Campus la naan idha follow pannுрேன் — example, NotificationService ku SMS send panna venும் na, SmsGateway class ah inherit pannாma, adha oru dependency ah inject pannுрேன் (composition) — so tomorrow email gateway venும் na, easy ah swap pannலாம், inheritance chain touch pannாma.
🗣️ Interview la eppadi sollanum"Inheritance models an is-a relationship, composition models a has-a relationship by injecting one object into another instead of extending it. I generally favor composition because inheritance creates tight coupling to the base class — any change there ripples into every derived class. In my project, services depend on other services or gateways through constructor injection rather than inheriting from them, which keeps things flexible and easy to swap or mock in tests."
🔁 Follow-up Qs they might fire next
Q: What's the fragile base class problem exactly? — When changing a base class's internal behavior unintentionally breaks derived classes that depended on the old behavior, even though the public contract looks unchanged.
Q: Does that mean inheritance is bad? — No, it's still right for genuine is-a hierarchies with stable shared behavior — the guidance is to default to composition and reach for inheritance only when it's a clear, stable is-a relationship.
Q: How does composition help with unit testing? — Since dependencies are injected as interfaces rather than baked in through inheritance, I can substitute a mock or fake implementation easily in tests without touching the class under test.
2B SOLID Principles + Design Patterns
8. Single Responsibility Principle (SRP)
SRP nu solradhu — oru class ku oru vela mattum irukanum, adha change panna oru reason mattum irukanum. Idha explain panna best example enna na, naan direct ah Zen Campus microservices ah sollுவேன் — StudentService student data mattum handle pannும், BillService billing mattum, AttendanceService attendance mattum. Ovvoru service ku ovvoru specific responsibility. Billing logic maathina, attendance service ah touch panna தேவையில்லை — adhுதான் SRP oda real benefit. Class level la kூடa idha follow pannுрேன் — example StudentValidator class validation mattum pannும், DB save pannுрthу vera StudentRepository class dhaan pannும்.
🗣️ Interview la eppadi sollanum"SRP means a class should have only one reason to change. I apply this both at the architecture level and the class level — my microservices architecture is a direct example, where StudentService only owns student data, BillService only owns billing, so a billing change never forces a redeploy of the student module. Within a service too, I separate validation logic, repository logic, and business logic into different classes so each one changes for exactly one reason."
🔁 Follow-up Qs they might fire next
Q: How do you identify an SRP violation in a class? — If I can list two unrelated reasons someone might need to modify the class — like "changing validation rules" and "changing how data is saved" — that's a sign it's doing too much.
Q: Isn't splitting classes too much also a problem? — Yes, over-splitting adds unnecessary indirection — the goal is cohesive, focused responsibility, not the smallest possible class.
Q: How does SRP relate to your microservices split? — Each microservice is essentially SRP applied at the architecture level — one bounded business capability per service instead of per class.
9. Open/Closed Principle (OCP)
OCP na — class/module "open for extension, closed for modification" nu irukanum, artham enna na, pudhu functionality add panna venum na, existing tested code ah edit panna koodadhu, pudhusa oru class add panni extend pannanum. En best example — IPaymentService interface. Ippo enakku Online, Offline payment irukku. Nalaikku "Wallet payment" nu pudhu type add panna venum na, existing OnlinePaymentService or OfflinePaymentService code ah edit panna தேவையில்லை — புதுசா WalletPaymentService : IPaymentService nu oru class create panni, DI container la register pannினா pothும். Existing tested code touch aagல, so existing bugs varra chance illai.
🗣️ Interview la eppadi sollanum"Open/Closed means a class should be extendable without modifying its existing, already-tested code. My IPaymentService setup demonstrates this well — when we needed to add a new payment mode, I just created a new class implementing the interface and registered it in DI, without touching OnlinePaymentService or OfflinePaymentService at all. That keeps the existing, already-verified logic untouched and reduces regression risk."
🔁 Follow-up Qs they might fire next
Q: How do you achieve OCP practically in C#? — Mainly through interfaces, abstract classes, and polymorphism — code against the abstraction, then add new implementations for new behavior instead of branching with if-else on type.
Q: What's a code smell that violates OCP? — A big switch or if-else chain checking a type or enum to decide behavior — every new case means editing that same method again and again.
Q: Does OCP mean you never touch old code? — Not literally never — it means the design should minimize the need to, especially for adding new variants of existing behavior.
10. Liskov Substitution Principle (LSP)
LSP na — child class ah, parent class oda place la substitute pannினalum, app oda behavior correct ah irukanum, break aagakூடாது. Classic example — Rectangle class ku Square ah inherit panna try pannினா problem varும் (Square la width/height set panna, rendும் ஒண்ணா maara vendியthu, Rectangle expect panra behavior break aagும்). En project context la sollанumnа — IPaymentService-oda ellame implementations um Pay() call pannினா same "contract" ah follow pannanum — amount deduct pannி, success/failure return pannனும். Oru implementation matum unexpected ah exception throw pannினா, illa different meaning kொடுத்தா, andha class ah IPaymentService nu substitute panna mudியாது — LSP violate aagும்.
🗣️ Interview la eppadi sollanum"LSP says a derived class should be substitutable for its base type without breaking the caller's expectations. The classic example is Square inheriting from Rectangle — it looks like an is-a relationship, but Square breaks the contract that width and height can vary independently. In my code, every IPaymentService implementation must honor the same contract — take an amount, return a consistent success or failure result — if one implementation throws unexpectedly or behaves differently, code that depends on the interface can no longer trust it, and that's an LSP violation."
🔁 Follow-up Qs they might fire next
Q: Why does the Rectangle-Square example break LSP? — Because setting Square's width also changes its height to keep it square, which violates the Rectangle contract that width and height are independent — code written against Rectangle breaks when given a Square.
Q: How do you catch LSP violations in code review? — Look for a derived class that throws NotImplementedException, weakens a base guarantee, or needs type-checking (is/as) in the calling code to work correctly.
Q: Is LSP only about inheritance? — It applies to any substitutable abstraction, including interface implementations, not just class inheritance — the "IPaymentService" example is interface-based LSP.
11. Interface Segregation Principle (ISP)
ISP na — client ku thevaipadatha methods ah, oru periya "fat" interface la vachi force pannakூடாது, adha chinna chinna focused interfaces ah split pannanum. Example — naan oru IRepository interface la GetAll(), GetById(), Add(), Update(), Delete() ellame vachিட்டேன் nu vachுkonga. Aana namba ReportService ku read mattum venும் (GetAll/GetById), Delete/Update ellame venaam — ana andha service um force ah full interface implement pannும், adhுla thevaiyillatha methods ku dummy/NotImplementedException throw panna vendியidum — idhு ISP violation. Fix — IReadRepository and IWriteRepository nu rendு chinna interfaces ah split pannினா, ovvoru client தனக்கு தேவையான mattum implement pannும்.
🗣️ Interview la eppadi sollanum"ISP says clients shouldn't be forced to depend on methods they don't use. If I have one fat IRepository with Get, Add, Update, and Delete, but a reporting service only ever reads data, forcing it to implement Update and Delete is a violation — those methods end up unimplemented or throwing exceptions. I'd split that into smaller interfaces like IReadRepository and IWriteRepository, so a read-only consumer only depends on the read contract."
🔁 Follow-up Qs they might fire next
Q: How is ISP different from SRP? — SRP is about a class having one reason to change; ISP is specifically about interfaces not forcing clients to implement methods they don't need — related but ISP is scoped to contracts.
Q: What's a sign an interface violates ISP? — If implementing classes routinely throw NotImplementedException or leave empty method bodies for some methods, the interface is too broad.
Q: Does splitting interfaces hurt discoverability? — A little, but you can still combine smaller interfaces into a composite one for classes that genuinely need everything — you don't lose that option, you just don't force it on everyone.
12. Dependency Inversion Principle (DIP)
DIP na — high-level modules (business logic, controllers) low-level modules (DB access, external services) ah direct ah depend aagakூடாது, rendும் ஒரு abstraction (interface) ah depend aaganum. En project la exact ah idha implement pannirukேன் — Repository pattern over Dapper/EF Core. Controller SqlStudentRepository nu concrete class ah nேrடியaaga depend aagாma, IStudentRepository interface ah mattும் depend aagும். Implementation (Dapper vachா, EF Core vachா) inject aagும் at runtime via DI. Idhु nala benefit enna na, nalaikku Dapper vittu vera data access technology ku maara venும் na, controller code ah touch pannாma, pudusa oru IStudentRepository implementation mattum swap pannінா pothும்.
🗣️ Interview la eppadi sollanum"DIP says high-level modules shouldn't depend on low-level modules directly — both should depend on abstractions. In my project, controllers and business logic depend on IStudentRepository, not on a concrete SqlStudentRepository or DapperStudentRepository — the concrete implementation gets injected at runtime through DI. That means I can swap the underlying data-access technology, or plug in a mock for unit testing, without touching a single line in the controller."
🔁 Follow-up Qs they might fire next
Q: Isn't DIP the same as Dependency Injection? — No — DIP is the design principle (depend on abstractions), Dependency Injection is one common technique used to implement it by supplying dependencies from outside.
Q: What breaks if you skip DIP? — Your business logic becomes tightly coupled to a specific database or library, making it hard to unit test and hard to swap implementations later without rewriting the high-level code.
Q: Where exactly do you define the interface — in the high-level or low-level project? — The abstraction should live close to or owned by the high-level module/consumer, with the low-level implementation depending on and fulfilling that contract — that's the "inversion" part.
13. IoC Container & DI-DIP Connection (How They Tie Together)
DIP oru principle — "abstraction ah depend aagu" nு sொல்லும். Aana andha abstraction (interface) ku actual concrete implementation ah eppa, yaru kொடுக்கும்? Adhுक்கு thaan Dependency Injection + IoC Container use pannுрோம். ASP.NET Core la Program.cs la builder.Services.AddScoped<IStudentRepository, StudentRepository>() nu register pannினா, IoC Container (built-in DI container) oru "map" vachiருக்கும் — "yaaru IStudentRepository kேட்டாலும், StudentRepository object கொடு" nு. Controller constructor la IStudentRepository parameter kேட்டா, container automatic ah correct object ah create panni inject pannும். So DIP theory ah, DI + IoC Container thaan practical ah runtime la implement pannுрathு.
🗣️ Interview la eppadi sollanum"DIP is the design principle, and the IoC container is what makes it practical at runtime. I register my interfaces and their implementations in Program.cs — like AddScoped() — and the built-in ASP.NET Core container resolves and injects the right concrete object into any constructor that asks for the interface. So the container is the piece of infrastructure that actually wires the abstraction to a real implementation, without my business logic ever knowing or caring which concrete class it got."
🔁 Follow-up Qs they might fire next
Q: What are the DI lifetimes and when do you use each? — Transient for lightweight, stateless services created fresh every time; Scoped for per-HTTP-request objects like a DbContext or repository; Singleton for shared, thread-safe state across the app's lifetime, like a config or cache service.
Q: What happens if you forget to register an interface in DI? — You get a runtime InvalidOperationException at startup or first resolution saying the service isn't registered — it fails fast rather than silently.
Q: Can you inject a Singleton into a Scoped service safely? — Yes, that direction is fine; the problem is the reverse — injecting a Scoped or Transient service into a Singleton causes a captive dependency bug where short-lived state gets pinned for the app's whole life.
14. Coupling vs Cohesion (Design Quality)
Coupling na — rendு classes/modules eவ்வளவு ஒண்ணை ஒன்று depend aagirukku nu அளவு. Cohesion na — ஒரு class/module oda internal responsibilities எவ்வளவு closely related ah irukku nு அளவு. Namba goal always — low coupling, high cohesion. Zen Campus microservices architecture idhukku exact ah oru example — StudentService and BillService rendும் independent ah deploy pannலாm (low coupling — API contract mூlama மட்டும் communicate pannுрom, direct DB access illa). Ovvoru service kூllaа irukka code, ஒரே purpose ku (student management, or billing) related ah irukும் (high cohesion). Tight coupling irundhா, oru service change panna, vera services ellame redeploy panna vendியidும் — adhу thaan namba microservices split panradhுкku main reason.
🗣️ Interview la eppadi sollanum"Coupling is how much one module depends on the internals of another, and cohesion is how focused a module's own responsibilities are. The goal is always low coupling and high cohesion. My microservices architecture is a direct example — StudentService and BillService only talk through well-defined API contracts, not shared databases, so they're loosely coupled and can be deployed independently, while each service internally stays highly cohesive because it only deals with one business capability."
🔁 Follow-up Qs they might fire next
Q: How does DI reduce coupling? — By depending on interfaces rather than concrete classes, a consumer is coupled only to a contract, not to a specific implementation, so implementations can change without affecting the consumer.
Q: Can you have high cohesion but still tight coupling? — Yes — a class can do one focused job well but still reach directly into another class's internals or concrete type, which keeps coupling high even though cohesion is fine.
Q: How would you measure or spot low cohesion in a class? — If the class's methods use only a small subset of its own fields each, or the class name needs "and" to describe it (like "ValidatorAndLogger"), that's usually low cohesion.

Singleton, mostly through ASP.NET Core's built-in Singleton DI lifetime rather than hand-rolling the classic pattern, for things like shared config or cache services. 🔁 Follow-up Qs they might fire next
Q: Isn't Strategy pattern the same as just using an interface? — Strategy is essentially the design pattern name for exactly that — an interface with interchangeable implementations selected at runtime — IPaymentService is a textbook Strategy example in disguise.
Q: Why use DI Singleton instead of the classic Singleton pattern with a private static instance? — The classic pattern is hard to unit test and creates hidden global state; the DI container's Singleton lifetime gives the same one-instance-per-app-lifetime behavior but through injection, so it's mockable and testable.
Q: When would you introduce a Factory instead of just injecting the interface directly? — When the decision of which implementation to use depends on runtime data, like an incoming payment type in the request, rather than being fixed at startup registration time.
