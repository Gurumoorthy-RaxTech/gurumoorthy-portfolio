⚡ Part 4 Quick-Recall Cheat Sheet

  • 1. REST: Resource = noun in URL, HTTP verb = action, stateless, no verbs in URL like /getStudent.
  • 2. Status codes: Success = 2xx, client mistake = 4xx, server mistake = 5xx — table below.
  • 3. [ApiController]: auto 400 on invalid model, auto infers [FromBody]/[FromQuery]/[FromRoute], returns ProblemDetails.
  • 4. IActionResult vs ActionResult<T>: IActionResult = flexible return type, ActionResult<T> = flexible + strongly typed + Swagger-friendly.
  • 5. DTO vs Entity: never return EF entity directly — DTO controls what's exposed, avoids over-posting & serialization loops.
  • 6. API Versioning: URL segment (/api/v1/) most common, header/query param alternatives, never break v1 when adding v2.
  • 7. Content negotiation: Accept header decides JSON/XML — JSON is default and 99% of the time what's used.
  • 8. Swagger: Swashbuckle auto-docs + AddSecurityDefinition("Bearer") for JWT Authorize button.
  • 9. Pagination: pageNumber + pageSize + total count in response, never return full table.
  • 10. Rate limiting: AspNetCoreRateLimit / built-in .NET 7+ RateLimiter — protect from abuse/DDoS.
  • 11. Async controllers: async/await on every I/O call frees the thread — key to 1,000+ concurrent users.
  • 12. Global exception handling: UseExceptionHandler / custom middleware → consistent ProblemDetails response, no scattered try-catch.
  • 13. File upload/download: IFormFile for upload, FileStreamResult/FileContentResult for download (PDF/Excel reports).
  • 14. Idempotency: GET/PUT/DELETE = idempotent (same result on repeat), POST = not idempotent (creates new each time).
  • 15. Security end-to-end: HTTPS + JWT + input validation + rate limiting + parameterized queries — defense in layers.
  • 16. Response caching/ETag: cache GET-heavy read-only endpoints, ETag for "did data change" check, saves bandwidth.
  • 17. Background work: BackgroundService/IHostedService for long report generation — don't block the API request thread.
  • 18. API Gateway: Ocelot sits in front, single entry point, routes to right microservice (full detail in Part 7).
  • 19. Testing: Swagger UI for quick manual check, Postman collections + environments for real workflow testing.
  • 20. Logging/monitoring: structured logging (Serilog) + correlation ID per request = fast root-cause = 35% faster incident resolution.

4A REST Design & Contracts
1. REST Principles & Resource-Based URL Design (REST fundamentals)
REST na simple ah sonna — URL la "enna" nu iruka vendum (resource, noun), verb la "eppadi" nu sollanum (action). Namba oda thappu pannura mistake enna na — /getStudentById, /createStudent madhiri URL la verb kalakkurathu. REST way na — GET /students/5, POST /students, PUT /students/5, DELETE /students/5 — same URL, different verb, different meaning. Namba Zen Campus la ovvoru module (Student, Bill, Attendance) ku idhe pattern follow pannurom — /api/students, /api/bills/{id} madhiri. Statusless nu sollradhu — server oru request ah next request ku "memory" la vachukka koodathu, ovvoru request um tha independent ah, token/params ellam andha request la thaane kudukkanum.
🗣️ Interview la eppadi sollanumI design my APIs around resources, not actions — the URL is always a noun like /students or /bills/{id}, and the HTTP verb tells the server what to do with it: GET to read, POST to create, PUT to update, DELETE to remove. I keep every request stateless — no session dependency on the server side — which is also what makes horizontal scaling across multiple instances easy behind our Ocelot gateway.
🔁 Follow-up Qs they might fire next
Q: PATCH vs PUT difference? — PUT replaces the whole resource, PATCH updates only specific fields sent in the request.
Q: Why should URLs be plural nouns like /students not /student? — Convention — it represents a collection; /students/5 then reads naturally as "item 5 from the students collection."
Q: What makes an API "RESTful" vs just "HTTP API"? — RESTful strictly follows resource-based URLs, stateless requests, and proper verb/status-code usage; a lot of "REST APIs" in practice are just HTTP APIs that bend these rules a bit, which is fine as long as it's consistent.
2. HTTP Status Codes (2xx/4xx/5xx usage)
Status code na — server client kitta "enna nadanthichu" nu oru short code la sollradhu, body padikkama. Namba daily use pannura ones: 200 OK — normal success (GET/PUT). 201 Created — POST panni pudhu record create aana odane, response header la andha new resource oda location um kudukkanum. 204 No Content — DELETE success, but return panna body onnum illa. 400 Bad Request — client anuppina data thappu (validation fail). 401 Unauthorized — "neenga yaaru nu enakku theriyala" (token illa/invalid). 403 Forbidden — "neenga yaaru nu theriyum, aana intha action panna permission illa" (role mismatch). 404 Not Found — andha ID ku matching record illa. 409 Conflict — duplicate create pannuratha try pannuraanga (already existing email). 500 — namba code la bug, unexpected exception.
🗣️ Interview la eppadi sollanumI'm consistent about status codes across all my APIs — 200 and 201 for successful reads and creates, 204 for successful deletes with no body, 400 for validation failures, 401 when the token is missing or invalid, 403 when the token is valid but the role doesn't have permission, 404 when the resource doesn't exist, 409 for conflicts like duplicate unique fields, and 500 only for genuine unhandled exceptions. This consistency is what lets the frontend team write generic error handling instead of guessing per endpoint.
🔁 Follow-up Qs they might fire next
Q: 401 vs 403 — real difference? — 401 means "I don't know who you are" (auth failed), 403 means "I know who you are, but you can't do this" (authorization failed).
Q: When do you use 422 instead of 400? — 422 Unprocessable Entity is for semantically invalid data (syntax fine, business rule broken) — I mostly stick to 400 with [ApiController] auto-validation, but 422 fits well for business-rule violations.
Q: Should a "record not found" on update return 404 or 400? — 404 — the URL points to a specific resource ID that doesn't exist, so the resource itself wasn't found, not the request body.
3. [ApiController] Attribute Deep-Dive ([FromBody]/[FromQuery]/[FromRoute], ProblemDetails)
Controller mela [ApiController] attribute podradhu na, ASP.NET Core konjam "smart" ah maarudhu. Mudhal benefit — automatic model validation. Namba DTO la [Required] madhiri Data Annotation potta, adhu fail aana odane, namba manual ah if(!ModelState.IsValid) return BadRequest() eludha vendam — automatic ah 400 kudukkum, adhuvum ProblemDetails format la (standard error JSON structure — title, status, errors dictionary). Rendavadhu benefit — parameter source auto-infer pannum: route la irukka {id} ah [FromRoute] nu automatic ah edukkum, complex object ah [FromBody] nu edukkum, simple query string values ah [FromQuery] nu edukkum — namba explicit ah attribute podama vittalum, mostly correct ah guess pannidum. Namba explicit ah podradhu — code padikkuravanga ku clarity kudukkum nu.
🗣️ Interview la eppadi sollanum[ApiController] gives me automatic model-state validation — if the DTO fails a data annotation, the framework short-circuits and returns a 400 with a ProblemDetails body before my action method even runs. It also enables inference of binding sources — route values as [FromRoute], complex types as [FromBody], simple types from the query string as [FromQuery] — though I still annotate explicitly for readability. This removed a lot of repetitive validation boilerplate across all my controllers.
🔁 Follow-up Qs they might fire next
Q: What is ProblemDetails exactly? — It's the RFC 7807 standard error response shape — type, title, status, detail, and an errors dictionary — so every failure across the API looks the same to the client.
Q: Can you customize the automatic 400 response? — Yes, via ApiBehaviorOptions.InvalidModelStateResponseFactory in Program.cs — I can shape my own error object if ProblemDetails default doesn't fit.
Q: What happens if you mix [FromQuery] on a complex object? — It binds each public property of that object from matching query string keys — useful for filter/pagination request objects like PagedRequest with PageNumber, PageSize, SortBy.
4. IActionResult vs ActionResult<T> vs Raw Return Types (controller return types)
Moonu options irukku action method return panna. Raw type (example public Student Get()) — simple ah, aana Ok()/NotFound() madhiri different status codes return panna mudiyathu, always 200 mattum. IActionResult — flexible, Ok(data), NotFound(), BadRequest() — edhu venalum return pannalam, aana Swagger ku andha method oda actual return type enna nu clear ah theriyathu (object nu mattum kaamikum). ActionResult<T> — best of both — flexible ah status codes return pannalam, adhே samayam Swagger/OpenAPI ku "idhu Student type return pannum" nu clear ah document aagum. Adhunala namba mostly ActionResult<StudentDto> madhiri use pannurom.
🗣️ Interview la eppadi sollanumI default to ActionResult<T> on most endpoints because it gives me both worlds — I can return Ok(dto), NotFound(), or BadRequest() as needed, and Swagger still picks up the concrete DTO type for accurate API documentation. Raw types work only when the response is always a success with no variation, and plain IActionResult is fine when the response shape genuinely varies or there's no meaningful single return type.
🔁 Follow-up Qs they might fire next
Q: Can ActionResult<T> implicitly convert from T? — Yes — return studentDto; works directly, it implicitly wraps into an ActionResult<T> equivalent to Ok(studentDto).
Q: Why not always use IActionResult everywhere? — It works, but Swagger/OpenAPI can't infer the response schema from it alone, so I'd have to add [ProducesResponseType] manually on every action — ActionResult<T> gives that for free.
Q: How do you document multiple possible response types? — [ProducesResponseType(typeof(StudentDto), 200)] and [ProducesResponseType(404)] etc. on the action, so Swagger shows all possible outcomes.
5. DTOs vs Entities (never expose EF entities, mapping)
Entity na — EF Core oda class, database table structure exact ah match aagum, navigation properties, internal fields ellam irukkum. DTO (Data Transfer Object) na — API oda "public face", client ku edhu venumo adhu mattum kudukkura simple class. Namba never entity ah direct ah return pannakoodathu — reason rendu: Over-posting attack — client PasswordHash, IsAdmin madhiri field ah kooda request la anupinaa, model binding adha automatic ah entity la set pannidum, adhu security hole. Serialization loop — Student entity la Course navigation property, Course la thirumba Students list irundha, JSON serialize pannumbodhu infinite loop/circular reference error varum. Adhunala Zen Campus la StudentDto, BillDto madhiri separate classes vachu, AutoMapper or manual mapping (new StudentDto { Name = entity.Name }) vachu convert pannurom.
🗣️ Interview la eppadi sollanumI never return EF entities directly from an API — every endpoint has a dedicated DTO. This prevents over-posting attacks where a client could set fields like IsAdmin through model binding, avoids circular reference errors from navigation properties during JSON serialization, and lets me shape the response independently of the database schema. I use AutoMapper for straightforward field mapping and drop to manual mapping when the projection needs custom logic, like combining first and last name.
🔁 Follow-up Qs they might fire next
Q: AutoMapper vs manual mapping — which do you prefer and why? — For simple one-to-one field mapping I use AutoMapper to cut boilerplate; for complex projections or performance-critical read paths I map manually or project directly in the EF query with Select() to avoid loading unneeded columns.
Q: What's over-posting/mass assignment? — When a client sends extra fields in the request body that happen to match entity properties, and naive model binding sets them — DTOs restrict binding to only the fields I explicitly declared.
Q: Do you use separate DTOs for request and response? — Yes when they differ — a CreateStudentRequest won't have an Id, but the response StudentDto will, so I don't reuse one class for both directions.
6. API Versioning Strategies (URL/header/query param)
API live la irundhuchu nu vechukonga, appuram business requirement change aagi, response structure maathanum nu vandhaa — existing clients (mobile app, frontend already deployed) break aagama, pudhusa version add pannanum. Moonu common ways: URL segment — /api/v1/students, /api/v2/students — mudhal la easy ah puriyum, most common, Ocelot route configs la kooda easy ah match pannalam. Query param — /api/students?api-version=2. Header-based — X-Api-Version: 2 header la anuppuvanga, URL clean ah irukkum aana testing konjam kashtam (Postman la header add pannanum ovvoru time). Zen Campus scale la namba URL segment versioning pannuvom, edhukku na simple, clear, gateway routing kooda easy.
🗣️ Interview la eppadi sollanumI use URL-segment versioning — /api/v1/... — because it's explicit, easy to route through the Ocelot gateway, and easy for consumers to test manually in Postman or a browser. When a breaking change is needed, I add a v2 route alongside v1 instead of modifying v1, so any client still on the old version keeps working until they migrate on their own schedule. Query-param and header-based versioning are cleaner URLs but harder to route and debug, so I've stuck with URL versioning across our microservices.
🔁 Follow-up Qs they might fire next
Q: When do you actually need a new version vs just extending the DTO? — Adding an optional new field is non-breaking, no version bump needed; removing or renaming a field, or changing its type, is breaking and needs a new version.
Q: How do you avoid duplicating controller logic across versions? — Share the business logic in a service layer, and only the controller/DTO shape differs per version — the version-specific controller is a thin adapter.
Q: Is Microsoft.AspNetCore.Mvc.Versioning package needed? — It helps with automatic version resolution and deprecation headers, but for simple URL-segment versioning I've also just used separate route prefixes without the package.
7. Content Negotiation (JSON vs XML, Accept header)
Content negotiation na — client "enna format la data venum" nu Accept header la sollradhu (example Accept: application/json or application/xml), server adha padichu appropriate format la response kudukkum. ASP.NET Core default ah JSON mattum support pannum (System.Text.Json) — practical ah, namba entha project layum XML ku venum nu request vanjadhu illa, frontend/mobile ellame JSON expect pannuranga. XML support venum na AddXmlSerializerFormatters() add pannanum Program.cs la. Namba theoretical ah theriyum nu sollanum, aana real world la 99% JSON mattum thaan use pannurom.
🗣️ Interview la eppadi sollanumContent negotiation lets a client specify its preferred response format via the Accept header, and ASP.NET Core supports this out of the box with JSON as the default formatter. In practice, every API I've built serves JSON only — that's what the frontend, mobile clients, and Postman testing all expect — but if XML support were needed, it's a one-line AddXmlSerializerFormatters() call in Program.cs to add it alongside JSON.
🔁 Follow-up Qs they might fire next
Q: What happens if the client asks for a format the server doesn't support? — By default ASP.NET Core falls back to the default formatter (JSON) rather than returning 406, unless you explicitly configure RespectBrowserAcceptHeader and strict matching.
Q: Is content negotiation the same as API versioning via header? — No — content negotiation is about response format (JSON/XML), versioning via header is about which version of the resource schema to return; both use headers but for different purposes.
Q: Why JSON over XML for modern APIs? — Smaller payload, native fit with JavaScript, faster to parse, and virtually every client library — mobile, frontend, Postman — defaults to JSON.
4B Cross-Cutting API Concerns
8. Swagger/OpenAPI Setup with JWT Bearer Auth (Swashbuckle + Authorize button)
Swagger na namba API endpoints ah automatic ah document panni, browser la interactive ah test panna oru tool — Swashbuckle.AspNetCore NuGet package. Program.cs la AddSwaggerGen() vechu setup pannurom. Namba API la JWT protected endpoints irukku nu vechukonga — adha Swagger UI la test panna, AddSecurityDefinition("Bearer", ...) config panni "Authorize" button add pannanum. Andha button click panni Bearer <token> paste pannitta, appuram edho protected endpoint try pannalum, andha token automatic ah header la sethuku pogum. Idhu vechu Postman open pannama, Swagger UI kulla thaane login → token generate → protected APIs verify panna mudiyum — namba daily workflow idhu thaan.
🗣️ Interview la eppadi sollanumI configure Swashbuckle to auto-generate OpenAPI docs for every microservice, and I add a Bearer security definition with AddSecurityDefinition and AddSecurityRequirement so an Authorize button shows up in the Swagger UI. I paste the JWT there once, and it automatically gets attached to every subsequent request I try from Swagger — which makes it my first stop for quickly sanity-checking a protected endpoint before writing a full Postman test.
🔁 Follow-up Qs they might fire next
Q: Do you disable Swagger in production? — Yes, typically I only enable it for Development/Staging environments via app.Environment.IsDevelopment() checks, or gate it behind auth in higher environments, so it's not a public attack surface.
Q: How do you document response types in Swagger beyond the defaults? — [ProducesResponseType] attributes per status code, and XML comments enabled via GenerateDocumentationFile so summary/remarks show up in the UI.
Q: Swagger vs Postman — when do you use which? — Swagger for quick one-off checks straight from the browser during development; Postman for saved collections, environment variables, chained requests, and repeatable regression testing.
9. Pagination, Filtering & Sorting (large dataset lists)
Attendance report la 10,000 records irundha, oru API call la ellame return pannina — response size periusu aagi, frontend slow aagum, DB kooda unnecessary load. Adhunala pagination mandatory — client pageNumber and pageSize anuppuvaanga, namba .Skip((pageNumber-1)*pageSize).Take(pageSize) vechu DB level layae andha page mattum fetch pannurom, plus response la total record count um kudukkurom (frontend ku "total pages evlo" nu kanakku podanum). Filtering — client date range, status madhiri filter values anuppuvaanga, namba WHERE clause ah dynamic ah build pannurom. Sorting — sortBy=name&sortDir=asc madhiri, EF Core la OrderBy() dynamically apply pannurom. Zen Campus la Billing reports, Attendance lists ellame idhe pattern follow pannurom — namba never "select * and filter in memory" pannamattom, DB level layae filter/page/sort pannurom, edhukku na thousands of rows ah memory ku fetch panna vendam.
🗣️ Interview la eppadi sollanumFor any list endpoint that can return large volumes — attendance, billing history — I never return the full table. I accept pageNumber, pageSize, plus optional filter and sort parameters, and apply Skip/Take, Where, and OrderBy at the database level with EF Core or in the SQL itself with Dapper, so only the exact page of rows needed comes across the wire. The response also carries the total record count so the frontend can render correct pagination controls without a separate count call.
🔁 Follow-up Qs they might fire next
Q: Skip/Take vs keyset pagination — which is more efficient for large offsets? — Skip/Take degrades on very large offsets because SQL Server still scans past rows; keyset pagination (WHERE Id > lastSeenId) avoids that, but I've used Skip/Take since our page counts stayed reasonable in practice.
Q: How do you prevent a client from requesting pageSize=100000? — I cap it server-side — if pageSize exceeds a max like 100, I clamp it rather than trusting the client value directly.
Q: Where does dynamic sorting happen without SQL injection risk? — I whitelist allowed sort column names against a known list before building the OrderBy, never concatenate the raw client string into SQL or a dynamic LINQ expression unchecked.
10. Rate Limiting at the API Level (protecting endpoints from abuse)
Rate limiting na — oru client (IP or user) X seconds/minute kulla evlo requests anuppalam nu limit vachukradhu. Idhu illama irundha, oru bad actor or buggy frontend loop OTP generate API ah repeatedly hit pannina, namba SMS gateway cost explode aagum, illa DDoS madhiri server ah overload pannidum. Namba Ocelot gateway level layae rate limiting config pannurom (ovvoru downstream route ku separate limit) — but individual API kulla layum .NET 7+ built-in AddRateLimiter() middleware or AspNetCoreRateLimit package vechu add pannalam, especially sensitive endpoints ku (OTP generate, login) — example "5 requests per minute per IP" madhiri policy.
🗣️ Interview la eppadi sollanumI apply rate limiting both at the Ocelot gateway level for general traffic shaping, and at the individual API level for sensitive endpoints like OTP generation or login, using the built-in .NET RateLimiter middleware with a fixed-window or sliding-window policy — something like 5 requests per minute per client. This protects downstream resources like the SMS gateway from being hammered by a retry loop or an actual abuse attempt, and it's a big part of how the bank locker OTP system stayed reliable under load.
🔁 Follow-up Qs they might fire next
Q: Fixed window vs sliding window vs token bucket — differences? — Fixed window resets counts at a clock boundary (can burst at the edge), sliding window smooths that out, token bucket allows short bursts while enforcing an average rate — I've mostly used fixed/sliding window for simplicity.
Q: What response do you return when rate-limited? — 429 Too Many Requests, often with a Retry-After header telling the client when to try again.
Q: Rate limit by IP or by authenticated user? — By user/API key when authenticated, since IP can be shared behind NAT or proxies; by IP for anonymous endpoints like login where there's no user identity yet.
11. Async Controllers — Why Every I/O Action Should Be Async (scalability for concurrent users)
Namba resume la "1,000+ concurrent users" nu irukku — idhu achieve panna async/await romba key. Vishayam enna na, ASP.NET Core oru thread pool vechu requests handle pannum. Oru controller action DB call pannumbodhu, sync ah eludhina (.Result or blocking call), andha thread andha DB call complete aaganum nu "wait" pannikittu block aagi nikkum — andha thread vera edhukum use panna mudiyathu. 1000 users same time la request anuppina, 1000 threads block aagi, thread pool exhaust aagum, server slow aayidum/hang aayidum. async/await vechina — DB call pogumbodhu thread free aagi vera request handle panna pogum, DB response vandha appuram thaan andha task continue aagum. Adhunala Zen Campus la ovvoru DB call, HTTP call, file I/O — ellame async ah dhaan eludhurom.
🗣️ Interview la eppadi sollanumEvery controller action that touches I/O — a database call, an external HTTP call, file read/write — I write as async all the way down, using await instead of blocking calls like .Result or .Wait(). This means the thread isn't held hostage while waiting for a database or SMS gateway response — it goes back to the thread pool to serve another request, and resumes once the awaited task completes. That's a direct reason we could handle 1,000+ concurrent users on the ERP without exhausting the thread pool.
🔁 Follow-up Qs they might fire next
Q: What's the danger of calling .Result or .Wait() on an async method? — It blocks the calling thread synchronously and can deadlock in certain contexts (like ASP.NET's old SynchronizationContext), plus it defeats the whole purpose of freeing the thread.
Q: Does async make a single request faster? — Not necessarily faster for one request — it improves throughput/scalability under concurrent load by freeing threads, not the latency of an individual call.
Q: Should CPU-bound code be async too? — No — async is for I/O-bound work; for CPU-bound work, Task.Run to offload to a background thread makes more sense, but I rarely need that in a typical CRUD API.
12. Global Exception Handling / ProblemDetails Middleware (consistent error responses)
Munnadi namba ovvoru controller la separate try-catch potu irundhom — inconsistent error format, and edhavadhu exception miss aana app crash aagum. Ippa namba oru global exception-handling middleware vechirukom — pipeline la ellathukum mela vechurukom, so entha controller layum unhandled exception thrown aana, idhu catch pannum. Idhu enna pannum na — exception ah structured ah log pannum (correlation ID sethu), client ku consistent ProblemDetails JSON format la error return pannum (status, title, detail), and internal stack trace madhiri sensitive info client ku leak aagama paakum. Idhu vechu thaan namba 35% production incident-resolution time cut pannirukom — edhukku na, ippa ellame same place la, same format la log aagudhu, so debug panna easy.
🗣️ Interview la eppadi sollanumI built a global exception-handling middleware — registered early in the pipeline with app.UseExceptionHandler — that catches any unhandled exception across all controllers, logs it with a correlation ID using structured logging, and returns a consistent ProblemDetails response instead of leaking a stack trace to the client. This replaced scattered per-controller try-catch blocks, standardized every error response across all our microservices, and was a big contributor to the 35% cut in production incident-resolution time, since I could trace one request end-to-end with the correlation ID.
🔁 Follow-up Qs they might fire next
Q: UseExceptionHandler vs custom middleware — which did you use? — I've used both — UseExceptionHandler with a dedicated error-handling endpoint for the standard case, and a custom middleware when I needed finer control over the ProblemDetails shape per exception type.
Q: How do you avoid leaking sensitive info in error responses? — In production, the detail field is generic ("An error occurred"), the real exception message and stack trace go only to the logs, gated by environment check.
Q: How do you map specific exception types to specific status codes? — I catch known exception types (like a custom NotFoundException or ValidationException) and map them to the right status code — 404, 400 — while anything unrecognized falls back to 500.
13. File Upload/Download Endpoints (PDF/Excel report generation)
Upload side — client oru file (example: student photo, bulk-import Excel) anuppumbodhu, [FromForm] IFormFile file parameter vechu receive pannurom — multipart/form-data content type venum idhukku, JSON illa. File size validate pannurom (max size limit), extension check pannurom (only .xlsx/.pdf/.jpg allow pannurathu) — security ku romba important, illana yaaru venalum .exe file upload panni server ah compromise pannidalam. Download side — namba resume la "PDF/Excel report generation endpoints" irukku — fee receipt, attendance report generate pannumbodhu, namba iTextSharp/QuestPDF (PDF ku) or ClosedXML/EPPlus (Excel ku) vechu file bytes generate pannurom, appuram File(bytes, "application/pdf", "receipt.pdf") madhiri FileContentResult return pannurom — browser adha automatic ah download pannikkum.
🗣️ Interview la eppadi sollanumFor uploads, I accept an IFormFile bound via [FromForm] with multipart/form-data, and I always validate file size and extension server-side before saving — never trust the client-provided content type alone. For downloads, like the fee receipt PDFs and attendance Excel reports in Zen Campus, I generate the file bytes in memory using libraries like QuestPDF or ClosedXML, and return them as a FileContentResult with the right content type and filename, so the browser handles the download directly.
🔁 Follow-up Qs they might fire next
Q: How do you handle very large file uploads? — I set a request size limit explicitly (RequestSizeLimit attribute or Kestrel MaxRequestBodySize), and for genuinely large files I'd stream to storage rather than buffering the whole file in memory.
Q: Where do uploaded files get stored — DB or disk/blob? — Not in the database as a blob — I store them on disk or cloud storage and keep only the file path/reference in SQL Server, to keep the database lean and fast.
Q: How do you generate a large report without blocking the request thread? — For genuinely heavy reports I'd offload generation to a background job and let the client poll or get notified, rather than generating synchronously inside the request — ties into background/hosted services.
14. Idempotency in APIs (why PUT is idempotent, POST usually isn't)
Idempotent na — same request ah 1 thadava anuppinalum, 10 thadava anuppinalum, result same ah irukkanum — server state mela extra side-effect varakoodadhu. PUT /students/5 with same body — 5 thadava call pannalum, andha student record final ah same values la thaan irukkum, so PUT idempotent. DELETE /students/5 um idempotent — first call delete pannidum, next calls "already deleted"/404 return pannum, aana end state same. Aana POST /students — ovvoru call oru pudhu record create pannum, 5 thadava call pannina 5 students create aagidum — so POST idempotent illa. Idhu real world la romba matter pannum — network timeout aagi client retry pannumbodhu, PUT retry panna safe, aana POST retry panna duplicate records create aagum risk irukku. Idhukaga sensitive POST operations ku (payment, OTP generate) namba "idempotency key" pattern use pannalam — client oru unique key anuppuvaanga, namba andha key already process pannirundha, same response return pannuvom, pudhusa process pannamattom.
🗣️ Interview la eppadi sollanumIdempotent means calling the same request multiple times produces the same end state as calling it once. PUT and DELETE are idempotent by design — updating a resource to the same values or deleting an already-deleted resource leaves the system in the same state. POST typically isn't, because each call creates a new resource. This matters most for network retries — if a payment POST times out and the client retries blindly, you risk a duplicate charge, so for critical POST operations I'd use an idempotency key the client generates once, and the server checks before reprocessing.
🔁 Follow-up Qs they might fire next
Q: Is GET idempotent? — Yes, and it's also safe — it shouldn't change server state at all, unlike PUT/DELETE which are idempotent but do modify state.
Q: How would you implement an idempotency key in practice? — Client sends a unique key in a header, server stores processed keys with their response for a time window, and if the same key arrives again, it returns the cached response instead of reprocessing.
Q: Is PATCH idempotent? — Generally not guaranteed — it depends on the operation; a PATCH that says "increment count by 1" is not idempotent, while one that says "set status to Active" is.
4C Security, Performance & Testing
15. Securing Web APIs End-to-End (JWT + HTTPS + validation + rate limiting)
Security oru single feature illa — layers la thaan design pannanum. Namba bank locker OTP project la follow pannina layers: 1) HTTPS/TLS — transport level la data encrypt, man-in-the-middle attack thadukkum. 2) JWT auth — token valid ah irukka nu check, expiry/signature verify. 3) RBAC — role check, Admin-only endpoint ah Parent access panna mudiyathu. 4) Input validation — Data Annotations + FluentValidation, DTO level layae garbage/malicious input reject pannurom. 5) Parameterized queries — SQL injection thadukkum (EF Core/Dapper default ah safe). 6) Rate limiting — OTP generate API ah brute-force pannama thadukkum. Idhu ellam sernthu thaan namba "zero unauthorized access incidents" achieve pannirukom — oru layer fail aana, next layer catch pannum, single point of failure illa.
🗣️ Interview la eppadi sollanumI think of API security in layers, not a single control. Transport is HTTPS/TLS everywhere. Authentication is JWT with signature and expiry validation, authorization is role-based on top of that. Every DTO goes through input validation before it touches business logic, all database access is parameterized so SQL injection isn't possible, and sensitive endpoints like OTP generation are rate-limited to prevent brute-force attempts. In the bank locker system, this layered approach — combined with a full audit trail — is what got us zero unauthorized access incidents post-deployment.
🔁 Follow-up Qs they might fire next
Q: What's the risk of storing JWT in localStorage vs a cookie? — localStorage is accessible to JavaScript, so it's vulnerable to XSS token theft; an HttpOnly secure cookie isn't readable by JS, though it needs CSRF protection instead — trade-off depends on the app.
Q: How do you handle token expiry gracefully? — Short-lived access tokens plus a refresh token flow, so the client can silently get a new access token without forcing a full re-login.
Q: Where does input validation happen — client, API, or both? — Both — client-side for UX (instant feedback), but I never trust it; the API always re-validates because client-side checks can be bypassed entirely.
16. Response Caching / ETag for GET-Heavy Endpoints (reducing repeated load)
Konjam data (example: course list, dropdown master data) frequently request aagum, aana rarely change aagum. Idha ovvoru request kum DB hit panni fetch panradhu waste — response caching use pannalam. [ResponseCache(Duration = 60)] madhiri attribute vechu, andha response ah 60 seconds ku cache pannalam — andha window kulla same request vandha, DB kuda pogama cached response return aagum. ETag vera oru technique — server response kooda oru "version tag" (hash) anuppum, client next time same resource kekkumbodhu If-None-Match header la andha tag anuppuvaanga — data change aagalanaa, server "304 Not Modified" return pannum (body illama), client already irukka cached copy ah use pannalam. Idhu bandwidth save pannum, especially mobile clients ku.
🗣️ Interview la eppadi sollanumFor endpoints serving fairly static data — like master/lookup lists — I use response caching with a short duration so repeated requests within that window don't hit the database at all. For data that changes occasionally but I still want strong correctness, ETag is a better fit — the server sends a hash of the current response, the client sends it back as If-None-Match on the next call, and if nothing changed, I return 304 Not Modified with no body, saving bandwidth without risking stale data being served indefinitely like a blind cache would.
🔁 Follow-up Qs they might fire next
Q: Where would you NOT use response caching? — Anything user-specific or frequently changing — attendance marked today, live billing status — caching there risks showing stale data to the user.
Q: Client-side cache vs server-side cache (Redis) — difference in use case? — Response caching/ETag controls what the browser or HTTP client caches; a server-side cache like Redis avoids repeated DB hits across all users for shared data — I'd reach for Redis for something like a dashboard summary hit by many users.
Q: How do you invalidate a cache when the underlying data changes? — Short duration windows handle it naturally for response caching; for Redis I'd explicitly evict or update the key on the write path that changes the data.
17. Background/Long-Running Work from an API (IHostedService/BackgroundService)
Vishayam enna na, oru API request "fast" ah respond pannanum — client wait pannikittu irukkanum. Aana some work romba time edukkum — example, 5000 students ku bulk Excel report generate pannradhu, or bulk SMS anuppradhu. Idha request thread layae synchronously pannina, client 30-40 seconds wait pannanum, timeout aagalam. Adhukaga BackgroundService (IHostedService implement pannura base class) use pannurom — API endpoint request ah accept pannitu, odane "202 Accepted, job ID: xyz" return pannidum, actual heavy work ah background la separate thread/queue la process aagum. Client appuram andha job ID vechu status poll pannalam ("GET /reports/status/xyz") illa notification/email vaangalam. Idhu vechu API responsive ah irukkum, user ah block pannama.
🗣️ Interview la eppadi sollanumFor genuinely long-running work — bulk report generation, bulk SMS batches — I don't do it synchronously inside the request. I implement a BackgroundService that runs as a hosted service in the app, queue the job when the API receives the request, return 202 Accepted immediately with a job reference, and let the client poll a status endpoint or get notified once it's done. This keeps the API responsive and avoids client-side timeouts for anything that would otherwise take tens of seconds or more.
🔁 Follow-up Qs they might fire next
Q: How does the background service get the job details — shared memory or a queue? — I'd use a lightweight in-memory queue (Channel<T>) for a single-instance setup, or a real message queue if the service is scaled across multiple instances, so no job gets lost or double-processed.
Q: What if the app restarts mid-job? — In-memory queues lose unprocessed jobs on restart — for anything critical I'd persist job state in the database so it can be picked up again, rather than relying purely on memory.
Q: IHostedService vs BackgroundService — difference? — BackgroundService is an abstract base class that implements IHostedService and gives you a simple ExecuteAsync override — I use BackgroundService directly rather than implementing IHostedService from scratch.
18. API Gateway Relationship — How Ocelot Sits in Front (brief — full detail in Part 7)
Zen Campus la 10+ microservices irukku — StudentService, BillService, AttendanceService, HostelService, etc. Frontend ovvoru service ku direct ah call pannina, ovvoru service oda URL/port therinjirukkanum, CORS separately configure pannanum — mess aagidum. Ocelot API Gateway namba idha simplify pannudhu — client single URL (Ocelot) ku mattum call pannum, Ocelot ocelot.json config paathu, "idhu /api/students/* na StudentService ku pogum" nu correct downstream service ku route pannum. Namba individual Web APIs level layae namba idha epdi design pannurom nu paathom (REST design, status codes, DTOs) — adhu ellam Ocelot back side la irukkura ovvoru microservice kulla apply aagum. Ocelot mela authentication, rate limiting, load balancing kooda config panna mudiyum — but idhu deep dive Part 7 la paakalam.
🗣️ Interview la eppadi sollanumEvery Web API I build follows the same design principles we've covered — resource-based routes, consistent status codes, DTOs — but individually, none of these services are exposed directly to the frontend. Ocelot sits in front as the single entry point; the frontend only knows Ocelot's URL, and Ocelot's route configuration maps incoming paths to the right downstream microservice. I'll go deeper into Ocelot's routing, load balancing, and rate-limiting configuration in the microservices section, but at the Web API level, the key thing is each service is self-contained and gateway-agnostic — it doesn't need to know it's sitting behind a gateway.
🔁 Follow-up Qs they might fire next
Q: Does JWT validation happen at the gateway or the individual service? — Both can validate it, but I typically validate at the downstream service too, not just the gateway, so a service isn't blindly trusting gateway-forwarded requests.
Q: Can a Web API be called directly bypassing Ocelot? — Technically yes if its port is reachable, so in production I'd restrict direct access at the network level and only expose the gateway publicly.
Q: What happens to versioning when Ocelot is in front? — Ocelot route config just maps the versioned upstream path (/api/v1/students) to the correct downstream path — versioning logic itself still lives in the individual Web API.
19. Testing APIs with Postman/Swagger (real workflow he follows)
Namba daily workflow enna na — code eludhi run pannina odane, first Swagger UI open pannuvom, quick ah "compile aayiducha, basic response varudha" nu check pannuvom — Authorize button la token paste panni protected endpoint um try pannuvom. Aana real testing ku Postman use pannuvom — edhukku na Postman la collections create pannalam (StudentService collection, BillService collection), environment variables vechukalam (dev/staging/prod base URL, token — hardcode pannama), oru request oda response ah next request la use pannalam (example: login response la vara token ah automatic ah next request header ku set pannradhu — Postman scripts/tests tab vechu). Edge cases test pannuvom — invalid token, missing required field, wrong data type, boundary values (pageSize=0 madhiri) — namba status codes correct ah varudha nu verify pannuvom. Team la share pannradhukum Postman collection export panni git la vechurupom.
🗣️ Interview la eppadi sollanumMy typical flow is Swagger first for a quick sanity check right after writing an endpoint — does it compile, does the happy path respond correctly. Then I move to Postman for real testing, where I maintain collections per microservice with environment variables for dev, staging, and production base URLs and tokens, and I chain requests — for example, capturing the login response's token into an environment variable automatically for subsequent authenticated calls. I always test edge cases too — invalid or expired tokens, missing required fields, boundary values like pageSize zero — to make sure the status codes and error responses behave exactly as designed, and I keep the collections version-controlled so the team can reuse them.
🔁 Follow-up Qs they might fire next
Q: Do you write automated tests for APIs, or is it all manual Postman? — Postman for exploratory/manual verification during development; for regression I've also used Postman's Collection Runner/Newman in a pipeline, plus unit/integration tests at the code level for critical business logic.
Q: How do you test a rate-limited endpoint? — Fire repeated requests quickly (Postman Runner or a small script) past the configured threshold and confirm I get a 429 with the expected Retry-After behavior.
Q: How do you keep Postman collections in sync with API changes? — I update the collection alongside the code change in the same PR when possible, or regenerate parts of it from the Swagger/OpenAPI spec using Postman's import feature.
20. Logging/Monitoring API Requests (structured logging + correlation ID)
Plain Console.WriteLine madhiri logging production la use pannamudiyathu — search panna kashtam, structure illama irukkum. Namba structured logging (Serilog madhiri) use pannurom — log entries JSON-like key-value format la irukkum (timestamp, level, service name, requestPath, userId), so log aggregation tool la easy ah query panna mudiyum ("last 1 hour la BillService la irundha 500 errors evlo" madhiri). Idhula romba important piece — correlation ID. Oru request Ocelot gateway vazhi StudentService ku pogum, andha service BillService ah call pannum nu vechukonga — namba gateway level layae oru unique correlation ID generate pannitu, ovvoru downstream call layum andha ID ah header la forward pannurom. Production la bug report vandha, andha correlation ID vechu, andha one request full journey ah — evlo services touch pannichu, edhu la fail aachu nu — trace panna mudiyum. Idhu thaan namba 35% incident-resolution time cut pannradhukku key reason — munnadi ovvoru service log separate ah paakanum, ippa oru ID vechu ellame link aagum.
🗣️ Interview la eppadi sollanumI use structured logging with Serilog instead of plain text logs — every log entry carries consistent fields like timestamp, log level, service name, and request path, which makes it queryable in a log aggregation tool. The key piece is a correlation ID generated at the gateway and propagated through every downstream service call in a request's chain, so when there's a production issue, I can pull every log line tied to that one correlation ID across multiple microservices and see exactly where in the chain it failed. That end-to-end traceability, combined with the global exception middleware, is what got our incident-resolution time down by 35%.
🔁 Follow-up Qs they might fire next
Q: Where does the correlation ID get generated and how does it propagate? — Generated at the Ocelot gateway (or the first entry point) if not already present on an incoming request, then passed as a custom header — like X-Correlation-Id — to every downstream service call, and each service includes it in its own log entries.
Q: What log levels do you use and when? — Information for normal request flow, Warning for recoverable issues, Error for unhandled exceptions, and I avoid excessive Debug-level logs in production since they add noise and cost.
Q: Do you log request/response bodies? — Selectively — never for endpoints handling sensitive data like passwords, OTPs, or tokens; I log metadata (path, status code, duration) always, and bodies only for non-sensitive debugging scenarios.
