8A JWT & Security
1. JWT Structure & Validation (header.payload.signature)
JWT nu sonna oru courier parcel madhiri nenachukо — outside oru label irukku (header — enna algorithm use pannirukom nu sollum), ulla contents irukku (payload — userId, role, expiry ellame plain ah readable ah irukkum, but edit panna mudiyathu), and oru tamper-proof seal irukku (signature). Server andha seal ah, thanoda secret key vachu recalculate pannunu, original signature-oda compare pannum. Match aana "sari idhu namma than kudutha token, evanum edit pannala" nu confirm aagum, match aagala na 401 kudukum. Payload encode pannirukom (base64) — encrypt pannala, so andha part la password madhiri sensitive data vaikka koodathu, adha thaan naan role/userId/expiry mattum vachiruken.
🗣️ Interview la eppadi sollanumA JWT has three parts — header, payload, and signature, separated by dots. The header names the signing algorithm, the payload carries claims like user id, role, and expiry, and the signature is generated by hashing the header and payload with a secret key. On every request my middleware recomputes that signature server-side and compares it — if it doesn't match, or the token has expired, the request is rejected before it ever reaches my controller.
🔁 Follow-up Qs they might fire next
Q: Is the payload encrypted? — No, it's just base64-encoded, so anyone can read it — I never put passwords or secrets in there, only non-sensitive claims.
Q: What happens if the token is tampered with? — Even changing one character breaks the signature match, so the middleware rejects it with 401 before hitting business logic.
Q: Where do you store the secret key used to sign tokens? — In configuration/secrets store, never hardcoded in source — same key the API uses to validate incoming tokens.
2. RBAC — Role Based Access Control ([Authorize(Roles=...)])
Login aana odane, JWT payload kulla oru "role" claim (Admin, Teacher, Parent) embed pannuren. Apparam ovvoru endpoint mela [Authorize(Roles="Admin")] nu attribute podren — idhu oru security guard madhiri, "Admin ID card irundha mattum ulla vidu" nu check pannum. Valid token irundhalum, role match aagala na 403 Forbidden kudukkum — login aagirukan nu artham illa, andha specific door open pannalam nu artham. Namma Zen Campus la ellame API endpoints la idha exactly implement pannirukom.
🗣️ Interview la eppadi sollanumI embed the user's role as a claim in the JWT at login, and then decorate each endpoint with [Authorize(Roles="...")]. ASP.NET Core's authorization middleware reads that claim on every request — so even with a perfectly valid token, a Parent role gets a 403 the moment they try hitting an Admin-only endpoint. I used this across every API endpoint in Zen Campus.
🔁 Follow-up Qs they might fire next
Q: What if a user has multiple roles? — Claims support multiple role values for one user, and Authorize(Roles="A,B") checks for any match.
Q: Policy-based authorization vs role-based — which do you use? — Mostly role-based since it covered our needs; for finer rules like "only the owner can edit their own record" I'd add a custom policy or handler.
Q: Can roles change without re-login? — Not with the current token — the role is baked in at issue time, so a role change only takes effect on next login or token refresh.
3. Access Token vs Refresh Token (honest gap + design awareness)
Access token na — namma request ku attach panra short-lived token, expiry konjam kammi ah vaipom (example 30-60 mins) so leak aana kooda risk window sila. Refresh token na — access token expire aana apparam, user again password kekkama, oru long-lived (aana securely stored) token vachu pudhu access token vaanga help pannum. Naan honest ah sollanum na — ippa vare namma projects la simple ah short-expiry access token mattum use pannirukom, full refresh-token rotation flow implement pannala. Aana idhu eppadi design pannanum nu clear ah theriyum — refresh token ah DB la (hashed ah) store pannanum, HttpOnly secure cookie la vaikkanum, and use pannina odane rotate pannanum (old one invalidate).
🗣️ Interview la eppadi sollanumCurrently I issue short-lived access tokens with a reasonable expiry, which forces re-authentication periodically — I haven't yet implemented a full refresh-token rotation flow in production. But I understand the pattern well: store a long-lived refresh token server-side, usually hashed, deliver it via an HttpOnly secure cookie, and rotate it on every use so a stolen refresh token can't be replayed indefinitely. That's the next hardening step I'd bring to a system like this.
🔁 Follow-up Qs they might fire next
Q: Why not just make access tokens live longer instead of adding refresh tokens? — Longer-lived tokens increase the blast radius if stolen; short access + refresh gives both security and a smooth UX.
Q: Where would you store the refresh token on the client? — HttpOnly, Secure, SameSite cookie — not localStorage, since JS-accessible storage is vulnerable to XSS token theft.
Q: How do you revoke a compromised refresh token? — Keep a server-side record (or token family) so you can invalidate it immediately, forcing re-login.
4. OTP-based MFA Full Flow (bank locker project)
Bank locker project la, password mattum pothadhu nu decide pannitom — edhukna locker access na sensitive, so second factor venum. Flow ah step by step sollren: user credentials kudukkiran → system oru random numeric OTP generate pannum → adha DB la (hashed illa, but expiry timestamp vechu) store pannitu, SMS gateway API ku call pannuven, sub-3-second la user phone ku poidum. User andha OTP ah enter pannum bodhu, server-side la compare pannuven — correct ah irukka, expire aagala nu rendu check pannanum (naan 2 minutes expiry vechen). Sari na locker unlock command anuppuven, illa na attempt count track pannitu, over-try pannina temporarily lock pannuven. Ellame audit trail ah SQL Server la log pannirukom — evan eppo access try pannana nu.
🗣️ Interview la eppadi sollanumIn the bank locker system, after primary login I generate a time-limited, single-use OTP, deliver it through an SMS gateway API within about three seconds, and validate it server-side against both correctness and expiry before unlocking. I also track failed attempts to prevent brute-forcing and write every access attempt into an audit log table in SQL Server. This whole lifecycle — generate, deliver, validate, expire — is what got us zero unauthorized access incidents after deployment.
🔁 Follow-up Qs they might fire next
Q: What if the SMS gateway is slow or fails? — I'd add a retry with backoff and a fallback delivery channel; also surface a clear error to the user instead of a silent hang.
Q: How do you prevent OTP brute-force? — Limit attempts per OTP (say 3-5), lock out or cool-down after that, and keep expiry short.
Q: Do you store the OTP in plain text? — I store it short-lived with an expiry and treat it like any sensitive value — minimal retention, cleared after validation or expiry.
5. SQL Injection Prevention (parameterized queries)
SQL Injection na — user oru textbox la ' OR 1=1 -- madhiri type pannitu, namma query structure ah maathi, ellame data ah dump pannitu poiduvan. Idhu varradhu, namma user input ah direct ah SQL string kulla concatenate panninom na than. Naan EF Core and Dapper rendayum use pannுrathunala, ivanga default ah parameterized queries generate pannuranga — input ah "just a value" nu mattum treat pannuvanga, query oda logic ah maathamatanga. So @Email madhiri parameter vachu pass pannuren, never string.Format illa string concat panni query build pannala.
🗣️ Interview la eppadi sollanumI never build SQL by concatenating user input into a string. Both EF Core and Dapper use parameterized queries by default — the input is bound as a parameter value, not spliced into the query text, so a malicious string like ' OR 1=1 -- is treated purely as data and can never change the query's structure. That eliminates injection risk without any extra effort on my part, as long as I stick to parameterized APIs consistently.
🔁 Follow-up Qs they might fire next
Q: What about dynamic queries — like a search with optional filters? — I still build them with parameters, just conditionally adding WHERE clauses with placeholders, never inlining raw values.
Q: Are stored procedures automatically safe from injection? — Only if you pass values as parameters to the proc — if you build dynamic SQL inside the proc by concatenation, it's still vulnerable.
Q: Any other defense layer besides parameterization? — Least-privilege DB accounts and input validation as a second layer, but parameterization is the real fix.
6. HTTPS / TLS — Why Every Request Encrypted
HTTP la data plain text ah travel pannum — namma home wifi, office network, edho public wifi edhu use pannalum, in between irukka evanume oru "packet sniffer" vachu password, token, OTP ellame padikka mudiyum. HTTPS/TLS podra bodhu, client-server between oru encrypted tunnel create aagudhu — data travel pannumbodhu scramble ah irukkum, correct key irukkarava mattum decrypt panna mudiyum. Bank locker project la — OTP, locker unlock command, token ellame HTTPS mela than travel aagudhu, edhukna idhu compromise aana serious problem.
🗣️ Interview la eppadi sollanumEvery API in both projects runs strictly over HTTPS. TLS encrypts the data in transit, so even if someone intercepts the traffic on a shared network, they can't read the JWT, OTP, or any payload — they'd just see encrypted bytes. For a security-sensitive system like the bank locker, where OTPs and unlock commands are flowing over the network, that's non-negotiable.
🔁 Follow-up Qs they might fire next
Q: What's the difference between SSL and TLS? — SSL is the older, deprecated protocol; TLS is its modern successor — people still say "SSL certificate" loosely but it's TLS in practice.
Q: How do you enforce HTTPS in ASP.NET Core? — UseHttpsRedirection() middleware plus HSTS, so even an accidental HTTP request gets redirected to HTTPS.
Q: Does HTTPS alone make an API secure? — No — it protects data in transit only; you still need auth, input validation, and injection defenses on top.
7. Common Web Attacks — XSS & CSRF (honest exposure level)
Naan deep pentest expert illa nu honest ah sollanum, aana basics clear ah theriyum. XSS na — attacker oru malicious script ah input field mூlama inject pannitu, adha vera user browser la run pannavaikaraan (example: comment box la <script> tag). Adha thadukka — user input ah render pannumbodhu output encode pannanum, React/Razor madhiri frameworks default ah idha handle pannum. CSRF na — user login aagi irundha bodhu, oru malicious site andha user cookie use panni, avaru arivilama oru request namma server ku anuppa try pannum. Namma APIs cookie-based session illa, JWT bearer token (Authorization header la explicit ah anுpanum) use panradhunala, CSRF risk naturally kammi — aana form-based/cookie flow irundha anti-forgery token venum.
🗣️ Interview la eppadi sollanumI'm not a security specialist, but I build with these in mind. For XSS, I rely on the framework's default output encoding and validate/sanitize any user-supplied input before it's rendered or stored. For CSRF, since our APIs use JWT bearer tokens in the Authorization header rather than cookies, the classic CSRF vector is largely mitigated — the attacker's page can't silently attach that header. If I did have a cookie-based or Razor form flow, I'd add the built-in anti-forgery token validation ASP.NET Core provides.
🔁 Follow-up Qs they might fire next
Q: If you did store JWT in a cookie, would CSRF be a concern again? — Yes — then I'd need SameSite cookie attributes plus anti-forgery tokens to close that gap.
Q: How do you prevent stored XSS from user-generated content? — Sanitize/encode on output, and if rich text is allowed, use a whitelist-based HTML sanitizer rather than trusting raw input.
Q: Any tooling you use to catch these issues? — I lean on framework defaults and code review; I haven't run a dedicated security scanner in these projects yet, that'd be a good addition.
8B Docker & Containerization
8. Why Docker ("works on my machine" problem)
Namba ellorukkum theriyum andha pain — namma laptop la .NET version, DB driver, config ellame perfect ah work aagudhu, aana andha same code ah server ku deploy pannumbodhu, "works on my machine but not on yours" nu problem varum. Reason — server la vera .NET runtime version irukkalam, vera library missing irukkalam, vera OS config irukkalam. Docker idha permanent ah fix pannudhu — namma application code, adhoda exact dependencies (runtime, libraries, config) ellame ஒரு "container" nu package pannitu, andha container edhu machine la venalum — namma laptop, staging server, production server — same ah exact ah run aagum, edhukna ulla irukka environment thaan travel pannudhu, host machine oda difference matter pannadhu.
🗣️ Interview la eppadi sollanumBefore Docker, we'd hit the classic problem of code behaving differently across dev, staging, and production because of environment drift — different runtime versions, missing dependencies. Docker packages the app together with its exact runtime and dependencies into an image, so the same container runs identically everywhere. That's exactly why I containerized every microservice in Zen Campus — it removed environment inconsistency as a source of production bugs entirely.
🔁 Follow-up Qs they might fire next
Q: Isn't a VM the same idea? — Conceptually similar but VMs virtualize a whole OS, which is heavy; containers share the host OS kernel, so they're much lighter and start faster.
Q: Does Docker guarantee zero deployment issues? — No, it removes environment drift, but you still need to manage config, secrets, and infra correctly.
Q: What if the host OS is different (Windows vs Linux)? — Linux containers need a Linux kernel — on Windows dev machines that's handled via WSL2/Docker Desktop, and production usually runs Linux containers on Linux hosts.
9. Docker Image vs Container
Idha oru simple analogy vachu explain pannalam — Image na oru "recipe" madhiri, freeze pannirukka, edit panna mudiyathu. Container na andha recipe vachu actual ah cook panniruka "dish" — running, live instance. Same image (recipe) vachu, 5 containers (5 dishes) start pannalam — ovvondrukum tanaa memory, tanaa process, but ellame same code base. Namba StudentService traffic athigama irundha, same StudentService image vachu innoru container start pannitu, load balance pannalam.
🗣️ Interview la eppadi sollanumAn image is the immutable, packaged blueprint — code, runtime, dependencies, all frozen at build time. A container is a running instance of that image, with its own process and writable layer on top. I can spin up multiple containers from the same image to scale a service horizontally, which is exactly what I'd do if StudentService needed extra capacity during admission season.
🔁 Follow-up Qs they might fire next
Q: If a container writes data, does it persist after it's removed? — No, by default the writable layer is lost when the container is deleted — for persistence you mount a volume.
Q: How do you tag and version images? — Using tags like myservice:1.2.0 or a git-commit-sha, so I can roll back to a known-good image if a deploy goes wrong.
Q: Where do images get stored/shared? — A container registry — Docker Hub or a private registry like Azure Container Registry.
10. Dockerfile — Base Image & Multi-Stage Builds
Dockerfile na — "eppadi image ah build pannanum" nu step by step instructions vaikka file. .NET service ku typical ah rendu stage vaikkuvom — first stage la full SDK image (heavy, but build tools ellame irukkum) vachu, code restore pannitu, build panni publish pannuvom. Second stage la lightweight runtime-only image (SDK illama) vachu, first stage la irundhu compiled output ah mattum copy pannuvom. Idha "multi-stage build" nu sollுவோம் — final image la SDK, source code, intermediate build files ethுவும் illama, just runtime + published DLLs mattum irukkum, so image size chinnama, deploy pannradhu fast ah, attack surface um kammi ah irukkum.
🗣️ Interview la eppadi sollanumMy Dockerfiles use a multi-stage build — the first stage uses the full .NET SDK image to restore, build, and publish the app, and the second stage starts fresh from a lean ASP.NET runtime image and just copies over the published output. That keeps the final image small since it doesn't carry the SDK, source code, or intermediate build artifacts — smaller images pull faster and have a smaller attack surface.
🔁 Follow-up Qs they might fire next
Q: Why not just use the SDK image for everything? — It works, but it's much larger and includes build tooling you don't need at runtime — unnecessary size and surface area.
Q: What goes into a .dockerignore? — bin/obj folders, .git, local config — anything that would bloat the build context or leak local artifacts into the image.
Q: How do you keep image layers cache-friendly? — Copy csproj files and restore dependencies before copying the rest of the source, so Docker can cache the restore layer when only code changes.
11. Docker Compose — Orchestrating Multiple Microservices + DB
Zen Campus la StudentService, BillService, HostelService, TransportService, TimeTableService — pattern ah 10+ microservices + DB irukku. Ovvondraiyum tanaa docker run command type panni start pannradhu practical illa — port mismatch, dependency order ellame manual ah handle panna vendiyathu. Docker Compose oru single YAML file (docker-compose.yml) la, ovvoru service — adhoda image, ports, environment variables, depends_on ellame define pannitu, docker-compose up ஒரே command la full stack ah — ellame microservices + SQL Server container um sேरndhu — start pannalam. Local dev la kூda, oru pudhu team member laptop la clone pannitu, ஒரே command la full environment ready.
🗣️ Interview la eppadi sollanumWith 10+ microservices plus a database, starting each one individually with docker run isn't practical. I defined the entire stack in a single docker-compose.yml — each service's image, ports, environment variables, and startup dependencies — so the whole system, including the SQL Server container, comes up with one docker-compose up command. That gave us consistent, repeatable environments locally and across staging.
🔁 Follow-up Qs they might fire next
Q: How do you make sure the DB container is ready before the API starts? — depends_on controls start order, but for true readiness I'd add a healthcheck or retry logic in the app's DB connection startup.
Q: Is Docker Compose used in production? — I mainly used it for local/dev consistency; for real production orchestration you'd typically move to something like Kubernetes or Azure Container Apps.
Q: How do services in Compose find each other? — By service name as hostname — Compose sets up a default network where each service is DNS-resolvable by the name given in the YAML.
12. Container Networking Basics (services talk by name)
Container networking konjam confuse aagalam first time — but simple ah sollanum na, Docker Compose default ah oru internal network create pannum, andha network la irukka ellaa containers-um oru chinna "office" la irukka team members madhiri, ovvorutharukum peru (service name) vechu call pannalam. So StudentService, BillService ah call pannanum na, IP address theriya theva illa — direct ah http://billservice:8080 nu URL vachu call pannalam, Docker internal DNS andha name ah correct container ku resolve pannikum. Namma laptop (host) la irundhu andha container ah access pannanum na mattum, port map pannirukka external port use pannanum (example localhost:5001).
🗣️ Interview la eppadi sollanumWithin a Docker Compose network, containers can reach each other using the service name as the hostname — Docker's internal DNS resolves it automatically, so my StudentService just calls http://billservice:8080 instead of hardcoding an IP. From outside the Docker network, on the host machine, you go through the mapped port, like localhost:5001. This made inter-service configuration simple and portable across environments.
🔁 Follow-up Qs they might fire next
Q: What if two services are on different Compose files/networks? — They can't resolve each other by name unless you explicitly join them to a shared external network.
Q: Is this networking approach production-grade? — For simple setups yes, but at real scale you'd use a proper orchestrator's service discovery, like Kubernetes Services.
Q: How does the Ocelot gateway fit into this network? — Ocelot is itself a container on the same network, configured with the downstream service names as its routing targets.
13. Environment Variables & Secrets in Docker
Beginner mistake enna na — connection string, API key, JWT secret ellame directly Dockerfile-la or appsettings.json-la hardcode pannitu, image build pannradhu. Idhu romba risky — image ah edhavadhu registry ku push pannina, andha secrets um exposed aagidum. Correct way — Docker Compose la environment: section vachu, illa .env file (git ignore pannirukanum) vachu, runtime la mattum inject pannanum, image build time la illa. Production la idha innum secure ah pannanum na, Azure Key Vault madhiri secret manager use pannalam — app start aagumbodhu Key Vault-la irundhu fetch pannikum, image la edhuvume store aagathu.
🗣️ Interview la eppadi sollanumI never bake secrets like connection strings or API keys into the Docker image itself. They're passed in as environment variables at container runtime — through Compose's environment section or an untracked .env file locally — so the same image can move across dev, staging, and production with different secrets injected per environment. For production, the ideal next step is pulling secrets from something like Azure Key Vault at startup instead of environment variables on disk.
🔁 Follow-up Qs they might fire next
Q: What's wrong with putting secrets in appsettings.json even if it's not in the image? — It's still risky if committed to source control — I use appsettings for non-sensitive config and environment variables/secret managers for anything sensitive.
Q: How do you handle different secrets for dev vs prod? — Different .env files or Compose overrides locally, and environment-specific configuration in the deployment pipeline for higher environments.
Q: Have you used Docker secrets or Kubernetes secrets? — Not hands-on yet — I've used environment-variable injection; secret-store integrations like Key Vault are the next level I'm building toward.

15. Centralized/Structured Logging (35% incident-resolution story)
Idhu than namba resume la irukka signature achievement — full detail ah sollanum. Before: ovvoru microservice, ovvoru controller la separate ah try-catch pota irundhom, log format kூda consistent illa — oru service Console.WriteLine pannum, vera oru service vera format la log pannum. Production la bug வந்தா, andha specific service-oda log file ah manual ah தேடி, vேறு service log-oda correlate pannradhu romba time eduthுchu. Fix: naan ellaa microservices kூடavum common ah oru global exception-handling middleware and structured logging library (Serilog madhiri) build panninen — ellaa services-um same JSON-based log format la log pannும், centralized ah oru place (log file/sink) la collect aagும். Idhுனாலே — production la issue வந்தா, log ah search pannினா, exact stack trace, timestamp, request details ellame ஒரே format la kidaikும், developer manual ah scatter aana logs ah correlate panna வேண்டியதில்லை. Idhு exactly than incident resolution time ah 35% கமிа pannுchi.
🗣️ Interview la eppadi sollanumBefore this, each microservice had scattered try-catch blocks with inconsistent log formats, so debugging a production issue meant manually hunting through different services' logs and mentally stitching them together. I built a shared exception-handling middleware and structured logging setup — using Serilog — used consistently across every microservice, so all exceptions get caught centrally, logged in a uniform structured format with timestamps, stack traces, and context, and written to a centralized sink. This consistency and centralization is what cut our production incident-resolution time by 35% — engineers could search one place instead of chasing logs service by service.

16. Correlation ID — Tracing a Request Across Services
Microservices la oru periya problem enna na — oru single user action (example: fee payment), Ocelot gateway வழியா StudentService, BillService, PaymentService — moonu service-a touch pannும். Edhavadhu ஒரு service la fail aana, andha specific request ah trace pannradhu kashtam, edhukna ovvoru service-um tanaa logs vekkும். Fix: gateway level la, ovvoru incoming request ku ஒரு unique "Correlation ID" generate pannுவேன் (illa client-tha kudukuma na accept pannுவேன்), andha ID ah header மூலம் downstream ellaa service calls-லும் pass pannுவேன். Ovvoru service andha same correlation ID vachu than log pannும். So production la issue வந்தா, andha ஒரே correlation ID ah search panninா, StudentService la enna aachu, BillService la enna aachu, PaymentService la enna aachu — full journey ஒரே place la kidaikும்.
🗣️ Interview la eppadi sollanumFor a request that touches multiple microservices — say, a fee payment flowing through StudentService, BillService, and PaymentService — I generate a correlation ID at the gateway and propagate it through a header on every downstream call. Each service logs using that same ID, so when I'm debugging a production issue, I search once by correlation ID and see the request's entire journey across all services in order, instead of piecing together separate logs manually.
🔁 Follow-up Qs they might fire next
Q: Where do you generate the correlation ID? — At the gateway/edge, or accept one if the client already sends it — then it's attached to the logging context for the whole request lifecycle.
Q: How is it propagated between services? — Via a custom HTTP header, forwarded on every outgoing call so downstream services pick it up and log with it too.
Q: Is this related to distributed tracing tools like OpenTelemetry? — Yes, conceptually — correlation ID is the manual version of what tools like OpenTelemetry/Application Insights automate with full distributed traces and spans.

18. Deployment Strategies Awareness (blue-green / rolling)
Hands-on ah naan idha production la configure pannala nu honest ah sollanum, aana concept clear ah theriyum, edhukna zero-downtime deployment romba important topic. Blue-Green na — rendு identical environments vечிருப்போம் (Blue = current live, Green = new version). Green la deploy pannitu test pannுவோம், sari na traffic ah одரடி ah Blue-ல் இருந்து Green-ku switch pannுவோம் — downtime illama instant cutover. Rolling deployment na — ஒரே நேரத்தில் ellaa instances-a update pannாமல், ஒவ்வொன்றா ஒன்றா update pannுவோம் (example 10 instances la, 2-2 ah update panni traffic verify pannிட்டே pogும்) — edhavadhu problem aana, majority instances ஸ்டில் old stable version la irukும்.
🗣️ Interview la eppadi sollanumI haven't personally executed blue-green or rolling deployments in production, but I understand both patterns well since they matter for zero-downtime releases. Blue-green keeps two identical environments and switches traffic to the new one instantly once it's verified, giving an easy rollback by switching back. Rolling deployment updates instances gradually, so if something's wrong, only a fraction of traffic is affected and you catch it before the whole fleet is on the bad version. Given our Docker Compose setup, moving to either strategy would be a natural next step with a proper orchestrator.
🔁 Follow-up Qs they might fire next
Q: What's the tradeoff of blue-green over rolling? — Blue-green needs double the infrastructure temporarily but gives instant, clean rollback; rolling is more resource-efficient but rollback is gradual.
Q: How does this relate to Azure Deployment Slots? — Same idea — deploy to a staging slot, verify, then swap into production for near-zero downtime, which is Azure's managed take on blue-green.
Q: What would you monitor right after a deployment to catch issues early? — Error rate and latency in the centralized logs/APM immediately post-deploy, so a bad release is caught within minutes, not hours.

20. Debugging a Slow Production Issue — End to End
Idhu oru "put it all together" question — interviewer real scenario kudukuவார், "production la oru endpoint suddenly slow aagirukku, eppadi debug pannுвீங்க" nu. Step by step approach sollanum: முதலில் centralized logs/APM la போய், andha time range la எந்த requests slow ah irundhுchு nu identify pannுவேன் — correlation ID vачு andha specific request full journey (which service, edhu step la delay) trace pannுவேன். Adhு oru DB call la delay na, SQL Server execution plan paar்тது — index missing ah, table scan aагுதா, statistics stale ah nu check pannுவேன். Idhு vera microservice call-la delay na, andha downstream service oda own logs/health ah paar்тது root cause pannуவேன். Fix pannிட்ட apparam — same metric (response time, DB call duration) ah monitor panni confirm pannுவேன், idhு exactly recur aagாmல் irukka future ku alert/threshold um set pannуவேன்.
🗣️ Interview la eppadi sollanumI'd start in the centralized logs to identify which requests were slow in that time window, then use the correlation ID to trace that specific request's journey across services and pinpoint exactly where the time was spent. If it's a database call, I'd pull the execution plan to check for missing indexes, table scans, or stale statistics. If it's a downstream service call, I'd check that service's own logs and health. Once I identify and fix the root cause, I'd verify by watching the same latency metric afterward, and ideally add alerting so a regression like that gets caught automatically next time instead of waiting for a user complaint.
