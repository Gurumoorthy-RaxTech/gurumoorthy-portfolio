6A Dapper Deep-Dive
1. Dapper — Ithu enna, "micro-ORM" nu ஏன் sollுவோம் (What is Dapper)
Dapper ah oru "micro-ORM" nu sollுவோம் — edhukku nu na, full ORM (EF Core madhiri) automatic ah entities track pannradhu, migrations kudukradhu, LINQ ah SQL ah convert pannradhu — idhellam Dapper panradhu illa. Dapper basically IDbConnection mela irukka konjam extension methods (Query<T>(), Execute()) — namma raw SQL eludhanum, Dapper adha run panni result ah C# object ku fast ah map pannikkodukkும். ADO.NET mேlaye thaan build aagi irukku, so extra abstraction layer kammi — adhான Dapper fast.
🗣️ Interview la eppadi sollanumDapper is a lightweight object mapper built directly on top of ADO.NET. Unlike a full ORM, it doesn't do change tracking, query translation, or migrations — I write the raw SQL myself, and Dapper's only job is to map the result set to my C# POCOs efficiently, using IL-generated code close to ADO.NET speed. That's why it's called a micro-ORM.
🔁 Follow-up Qs they might fire next
Q: Dapper ஏன் EF Core-a vida fast? — It skips the heavy machinery — change tracking, LINQ-to-SQL translation — and maps results with compiled IL code, nearly as fast as raw ADO.NET.
Q: Dapper install pannanuma separate ah? — Yes, it's just a NuGet package that adds extension methods onto IDbConnection — no context, no configuration classes.
Q: Connection open/close pannradhu yaaru? — Namba thaan responsible — Dapper doesn't manage connection lifecycle, we open and dispose the connection ourselves (usually in a using block).
2. Dapper vs EF Core — eppo edha pick pannுவேன் (when to use which)
Idhu enakku romba common ah varra question, edhukku na naan project la rendayும் use panniருக்கேன். Naan clear ah sollுவேன் — simple CRUD, entity relationships automatic ah manage pannanum na EF Core easy, productivity fast. Aana high-volume dashboard/report queries, complex joins, aggregate calculations irukka idangal la Dapper mattum use pannுவேன், edhукки na andha idangal la EF Core oda overhead (change tracking, LINQ translation) namakku venaam. Real example — Zen Campus la attendance report page irundhичи, EF Core vecha load panna almost 5 seconds edhுthuchu, multiple tables join aagi, unnecessary tracking kூடa panniрундhichu. Naan andha logic ah stored procedure ku maathi, required indexes add panni, Dapper vecha adha call panna — result 2-3 seconds ku vandhuchu.
🗣️ Interview la eppadi sollanumI use EF Core for standard CRUD modules where productivity and maintainability matter more than raw speed — LINQ, change tracking, and migrations save a lot of boilerplate there. But for performance-critical, high-volume queries — like our attendance dashboard — I switch to Dapper with a tuned stored procedure. That one change, along with proper indexing, took that report from about 5 seconds down to 2 to 3 seconds.
🔁 Follow-up Qs they might fire next
Q: EF Core ah completely replace pannalaama Dapper ah vecha? — No — for simple CRUD, EF Core is actually faster to build and safer to maintain, so I keep both and choose per use case.
Q: Evvளவு complexity irundha Dapper ku switch pannுவீங்க? — Once a query profiles slow, has multiple joins/aggregations, or runs frequently on a dashboard, I first check the execution plan, then move it to Dapper if EF's overhead is the actual bottleneck.
Q: Same project la rendayும் together use pannalaama? — Yes — that's exactly what I did, EF Core for CRUD modules, Dapper for reports, billing, and payroll hot paths.
3. Dapper query result ah C# objects ku eppadi map pannும் (result-to-object mapping)
Dapper query run panni edukkra DataReader result ah, namma specify pannina C# class ku automatic ah map pannikkodukkும் — condition enna na, column name um property name um match aaganum (case-insensitive ah check pannும்). Example: SQL la "StudentName" column irundha, class la "StudentName" property irundha, adha direct ah assign pannிடும். Column name match aagalana, SQL la AS alias vecha match pannலாம், illa manual mapping pannலாம்.
🗣️ Interview la eppadi sollanumDapper maps each row to my POCO by matching column names to property names case-insensitively, using reflection-backed IL generation under the hood for speed. If a column name doesn't line up with a property, I alias it in SQL with AS, or map it manually — there's no separate mapping configuration file needed like EF Core's fluent API.
🔁 Follow-up Qs they might fire next
Q: Column name property name match aagalana enna aagும்? — That property just stays at its default value — Dapper doesn't throw an error unless you configure strict mapping.
Q: Dapper dynamic objects ah return panna mudியுமா? — Yes, calling connection.Query(sql) without a generic type returns dynamic rows — handy for quick ad-hoc queries.
Q: Nested object (Address inside Student madhiri) ah eppadi map pannுவீங்க? — For a real join with two tables I use multi-mapping instead of trying to force it into a single flat class.
4. Dapper Multi-Mapping (join two tables into nested objects)
Namma oru query la rendு tables ah join panni, result ah rendு separate C# objects ah nested ah map pannanum na — example: Student oda Fee details ah ஒரே query la eduthu, Student object kulla Fee object ah nest panni vaikkanum na — Dapper multi-mapping use pannுவோம். Query<TFirst, TSecond, TReturn>() generic overload irukku, adhula splitOn parameter kudukkanum — andha column irundhu next object oda columns start aagும் nu Dapper ku sollradhு.
string sql = @"SELECT s.Id, s.Name, s.Class,
                      f.Id, f.Amount, f.DueDate
               FROM Students s
               INNER JOIN Fees f ON f.StudentId = s.Id
               WHERE s.Id = @Id";

var student = connection.Query<Student, Fee, Student>(
    sql,
    (s, f) => { s.Fee = f; return s; },
    param: new { Id = studentId },
    splitOn: "Id"   // Fee's "Id" column is where the second object starts
).FirstOrDefault();
🗣️ Interview la eppadi sollanumWhen a join query returns columns from two tables, Dapper's multi-mapping overload lets me map them into two separate objects and stitch them together in a callback — for example joining Students and Fees, and setting student.Fee = fee. I tell Dapper where the second object's columns begin using the splitOn parameter, which usually maps to that table's Id column.
🔁 Follow-up Qs they might fire next
Q: splitOn default value enna? — It defaults to "Id" — so if both tables' split column is literally named Id, you don't even need to pass it explicitly.
Q: Multi-mapping evlo tables varaikkum support pannும்? — Overloads go up to TFirst..TSeventh, so up to 7 types in a single query — beyond that I'd usually restructure or use QueryMultiple.
Q: Rendு tables kூடaவும் "Id" column irundha confusion varாதா? — I alias each table's Id column uniquely in the SQL, or make sure the splitOn value matches exactly where the second table's columns genuinely start.
5. QueryMultiple — ஒரே round trip la palapadi result sets (multiple result sets, one call)
Oru dashboard page ku palapadi different queries venum na (example: total students count, total fee collected, recent admissions list) — idha ellam separate ah database ku call panna, network round trip pala thadava serum. QueryMultiple() vecha, ஒரே database call la multiple SELECT statements ah semicolon vecha ஒரே batch ah run panni, result ah oru "grid" (GridReader) object ah thirumba vaangalाம், apparam Read<T>() call panni ovvoru result set ah order ah edukkalாம்.
string sql = @"SELECT COUNT(*) FROM Students;
               SELECT SUM(Amount) FROM Fees WHERE Paid = 1;
               SELECT TOP 5 * FROM Students ORDER BY AdmissionDate DESC;";

using var grid = connection.QueryMultiple(sql);
int totalStudents   = grid.Read<int>().Single();
decimal totalFee     = grid.Read<decimal>().Single();
var recentAdmissions = grid.Read<Student>().ToList();
🗣️ Interview la eppadi sollanumQueryMultiple lets me send several SELECT statements in one database round trip and read each result set back sequentially with grid.Read<T>(). I use this heavily for dashboard endpoints — instead of five separate queries and five round trips, it's one connection open, one network hop, and I pull counts, lists, and summaries out of the same GridReader.
🔁 Follow-up Qs they might fire next
Q: Read() call order matter aagுமா? — Yes — you must call Read<T>() in the exact same order the SELECT statements appear in the SQL, otherwise you get the wrong result set.
Q: QueryMultiple stored procedure kூடa use pannலாமா? — Yes — a stored procedure that returns multiple result sets works identically with CommandType.StoredProcedure.
Q: Transaction kூடa combine pannலாமா? — Yes, you can pass the same IDbTransaction into QueryMultiple like any other Dapper call.
6. Stored Procedures Dapper vecha call pannradhu (CommandType.StoredProcedure)
Namba heavy report/billing logic ah database level la stored procedure ah eludhi vachirukkும் na, adha Dapper la call panna easy — connection.Query<T>() ku procedure name kudukkanum, parameters ah anonymous object ah pass pannanum, commandType: CommandType.StoredProcedure nu explicit ah sollanum. Idhu thaan naan attendance report optimization la panna main change — raw complex SQL ah application layer la eludhradhுkku pathila, stored proc ku move panni, Dapper adha call pannradhu mattum pannuchu.
var result = connection.Query<AttendanceReportDto>(
    "usp_GetAttendanceReport",
    new { ClassId = classId, FromDate = from, ToDate = to },
    commandType: CommandType.StoredProcedure,
    commandTimeout: 60
).ToList();
🗣️ Interview la eppadi sollanumFor anything heavy — reports, payroll calculations — I move the logic into a stored procedure and call it from Dapper with CommandType.StoredProcedure, passing parameters as an anonymous object. That keeps the complex logic close to the data so the DB engine can optimize execution, and Dapper's job is just sending parameters and mapping the result back.
🔁 Follow-up Qs they might fire next
Q: Output parameters eppadi handle pannுவீங்க? — I use a DynamicParameters object, add an output parameter to it, call Execute, then read the value back afterward.
Q: Long-running proc timeout aayidumo? — Dapper accepts a commandTimeout argument I bump up for long reports instead of hitting the default 30-second SQL timeout.
Q: SQL injection risk irukkுமா stored proc call la? — No — parameters go through Dapper's proper parameterization, never string-concatenated, so it's safe by default.
7. Transactions in Dapper (IDbTransaction)
Namba multiple INSERT/UPDATE ஒரே operation ah part ah pannanum na (example: fee payment record pannradhu + student balance update pannradhu) — rendும் succeed aaganum illa rendும் fail aaganum nu ensure pannanum na, IDbTransaction use pannுவோம். Connection open panni, BeginTransaction() call panni, andha transaction object ah ovvoru Dapper call kூடaவும் (Execute, Query) pass pannanum, ellам success ah irundha Commit(), edhavadhு fail ah irundha Rollback() pannанум்.
🗣️ Interview la eppadi sollanumFor multi-step writes that must succeed or fail together — like recording a fee payment and updating the student's balance — I open a connection, call BeginTransaction(), and pass that transaction object into every Dapper Execute call. If everything succeeds I commit; if any step throws, I catch it and roll back so the database never ends up in a half-updated state.
🔁 Follow-up Qs they might fire next
Q: try-catch illama transaction use pannலாமா? — Technically yes but risky — I always wrap it in try-catch-finally so rollback fires on any exception and the connection is disposed properly.
Q: TransactionScope um IDbTransaction um difference enna? — TransactionScope is ambient and can span multiple connections/resources including distributed transactions; IDbTransaction is tied to one connection — simpler, and what I use for single-DB Dapper work.
Q: Nested transactions Dapper la supported aa? — Not directly — SQL Server has savepoints but there's no special nested-transaction API in Dapper, so I keep transactions flat and tightly scoped.
8. Dapper oru ORM-a? — nuanced answer (Is Dapper an ORM)
Idhu oru tricky question — interviewer "Dapper ORM-a illa illaya" nu directly kேkkலாம். Naan clear ah sollுவேன் — Dapper ah "micro-ORM" nu classify pannுவோம், edhukku nu na adhu object mapping (query result → C# object) mattum panрadhு, aana oru full ORM panra veрey velaigal — change tracking, LINQ-to-SQL query generation, migrations, lazy loading — idhellam Dapper panрadhு illa. So "ORM" nu strict ah sollradhுkku pathila, "it's an object mapper, not a full ORM" nu sollradhு correct answer.
🗣️ Interview la eppadi sollanumI'd call Dapper a micro-ORM rather than a full ORM. It does one thing — mapping query results to objects — really well and fast, but it doesn't do query generation from LINQ, change tracking, or schema migrations the way EF Core does. You write the SQL, Dapper hands you back typed objects. That's a deliberate trade-off for speed and control over convenience.
🔁 Follow-up Qs they might fire next
Q: Enga full ORM venum, enga micro-ORM podhும்? — Full ORM when I want productivity and don't want to hand-write SQL for every CRUD operation; micro-ORM when I need maximum control and performance on specific hot-path queries.
Q: Dapper la LINQ query eludhலாமா? — No, Dapper never translates LINQ to SQL — you always write raw SQL yourself; that's actually the core design choice that keeps it fast and predictable.
Q: Dapper vera micro-ORMs irukkா? — Yes — PetaPoco and ServiceStack.OrmLite follow a similar philosophy, but Dapper is by far the most widely adopted in the .NET ecosystem.
6B EF Core Deep-Dive
9. EF Core Migrations — Add-Migration/Update-Database workflow (schema versioning)
Namba DB schema ah code la version control pannுவathukku migrations use pannுவோம். Model class la field add panninen na, Add-Migration <Name> command run pannினா, EF Core andha changes ah detect panni oru migration file (Up/Down methods கூடa) generate pannும். Apparam Update-Database run pannினா andha migration ah actual DB ku apply panni column/table create pannும். Aana production la app start aagும்bodhu neraaga Migrate() call panрадhu naan pannmatten — edhukku na adhu automatic ah run aaguthu, review pannாma edhavadhு destructive change (column drop madhiri) accidentally apply aaidum bhayam irukку. Adhukku pathila, migration ah SQL script ah generate panni, team review panni, apparam controlled ah deployment pipeline la run pannுрோம்.
🗣️ Interview la eppadi sollanumI generate migrations locally with Add-Migration, review the generated Up and Down methods, and apply them to dev with Update-Database. In production I never call context.Database.Migrate() at app startup — instead I generate an idempotent SQL script from the migration, get it reviewed, and run it as a controlled step in the deployment pipeline. That avoids an accidental destructive change silently running the moment the app boots.
🔁 Follow-up Qs they might fire next
Q: Migration rollback pannанum na enna pannுவீங்க? — The Down() method reverses that specific migration, or I apply an earlier migration name with Update-Database <PreviousMigrationName>.
Q: Team la migration conflict aana enna pannுவீங்க? — I rebase against the latest model, regenerate the migration, and manually reconcile if two people added migrations in parallel.
Q: Seed data eppadi handle pannுவீங்க migrations la? — Either via HasData() in OnModelCreating for static reference data, or a separate seeding script run after migration for larger datasets.
10. Change Tracking & AsNoTracking() (read-only query optimization)
EF Core query run panna, default ah andha entity ah "track" panни vaikkும் — adhு na, oru internal snapshot vachiруkkும், apparam SaveChanges() call pannினா andha entity la enna maari irukku nu compare panни UPDATE query generate pannும். Idhு update/insert/delete panna venum na useful, aana read-only report/list page ku idhு waste — unnecessary memory, CPU overhead. Adhukku AsNoTracking() add pannினா, EF andha snapshot ah maintain pannாdhு, query fast aagும், memory kammi aagும்.
🗣️ Interview la eppadi sollanumBy default EF Core keeps a change-tracking snapshot of every entity it loads, so SaveChanges knows what changed. That's essential when I'm going to modify and save an entity, but for read-only endpoints — a report, a listing page — I add AsNoTracking(), which skips that snapshot entirely and noticeably reduces memory usage and query time.
🔁 Follow-up Qs they might fire next
Q: AsNoTracking() panна entity ah edit panна mudியுமா? — You can modify it in memory but SaveChanges won't pick up the change automatically — you'd need to re-attach it and mark it modified explicitly.
Q: Default tracking behavior globally off panна mudியுமா? — Yes — set context.ChangeTracker.QueryTrackingBehavior = NoTracking at the DbContext level if most queries in that context are read-only.
Q: AsNoTrackingWithIdentityResolution() enna? — It's a variant that still avoids full tracking overhead but ensures the same entity instance is reused for a repeated key within one query result — useful when a join produces duplicate related entities.
11. N+1 Query Problem & Include()/Select() Fix (eager loading / projection)
N+1 problem na — namba oru list eduthu, andha list kulla loop pannி, ovvoru item கூடaவும் related data ah lazy ah access pannினா, ovvoru iteration கூடaவும் separate ah DB ku query pogும். Example: 50 students eduthaa (1 query), apparam ovvoru student oda Fees ah loop la access pannினா — 50 more queries = 51 total. Idha fix panna rendு vazhi — Include() vecha eager loading pannி, related data ah single JOIN query la munnadiye eduthுkkalaam, illa Select() vecha exactly required fields ah oru projection ah single query la fetch pannலாம் (idhу best — unnecessary columns கூடa varாdhு).
// N+1 problem — bad, fires 1 + N queries
var students = context.Students.ToList();
foreach (var s in students)
    Console.WriteLine(s.Fees.Count); // lazy load per student = separate query

// Fixed with eager loading — 1 query total
var students = context.Students
    .Include(s => s.Fees)
    .AsNoTracking()
    .ToList();

// Even better with projection — only needed columns, 1 query
var summary = context.Students
    .Select(s => new { s.Name, FeeCount = s.Fees.Count })
    .ToList();
🗣️ Interview la eppadi sollanumN+1 happens when you access a navigation property inside a loop with lazy loading on — one query for the parent list, then one more per item for the related data. I avoid it by eager-loading with Include() when I genuinely need the full related entity, or better, projecting exactly the fields I need with Select() into a DTO — that turns N+1 queries into a single query with a join.
🔁 Follow-up Qs they might fire next
Q: Include() um Select() um edhு better? — Select() projection is usually better for read paths since it fetches only needed columns and avoids over-fetching; Include() is simpler when I genuinely need the full related entity graph.
Q: N+1 eppadi kandupidikkuவீங்க production la? — I check SQL Server Profiler or EF Core's logging/Application Insights dependency calls — a burst of near-identical queries per request is the giveaway.
Q: ThenInclude() use panрадhு eppo? — When I need to eager-load a second level of nested relationship, like Student → Fees → Payments.
12. Lazy vs Eager vs Explicit Loading (loading strategies)
Moonu vazhi related data ah load pannலாம் EF Core la. Lazy loading na — navigation property access panрадhு varaikkும் query fire aagாdhு, first access pannும்bodhu automatic ah background la query pogும் (idhு thaan N+1 ku root cause, careful ah use pannанும், proxy package வேணும்). Eager loading na — Include() vecha, main query கூடaவே related data ah முன்னாடியே JOIN pannி fetch pannுவோம். Explicit loading na — entity ah first load pannிட்டு, apparam explicit ah context.Entry(entity).Collection(x => x.Fees).Load() nu manual ah trigger pannுவோம், on-demand mattum.
🗣️ Interview la eppadi sollanumLazy loading fetches related data automatically the first time you touch the navigation property — convenient but the classic source of N+1 if you're not careful. Eager loading with Include() fetches everything upfront in one query, which is what I default to for known access patterns. Explicit loading is when I load the main entity first and then deliberately call context.Entry().Collection().Load() only when I actually need that related data — useful when loading it depends on a runtime condition.
🔁 Follow-up Qs they might fire next
Q: Lazy loading default ah on ah irukkுமா EF Core la? — No, it's off by default — you need Microsoft.EntityFrameworkCore.Proxies and virtual navigation properties to enable it.
Q: Neenga lazy loading use pannுவீங்களா production la? — Rarely — I prefer explicit Include()/Select() because lazy loading hides DB calls and makes performance unpredictable.
Q: Explicit loading eppo useful aagும்? — When loading the related data depends on a runtime condition, so I don't want to always eager-load it in the base query.
13. DbContext Lifecycle — ஏன் Scoped (DI lifetime, Singleton risk)
DbContext ah Program.cs la AddDbContext() pannும்bodhு default ah "Scoped" lifetime la register aaguது — idhு oru artham na, ஒரு HTTP request ku ஒரு DbContext instance mattum create aagும், request முடிஞ்சவுடனே dispose aagும். Idhு edhுkки correct nu na, DbContext thread-safe illa — same instance ah multiple threads/requests share pannினா, change tracker corrupt aagும், data race varும். Singleton ah panнинோm na enna aagும் nu kேkkலாம் — andha same context instance ellா requests கூடaவும் share aagும், memory la ellா entities um accumulate aagும் (memory leak), concurrent requests same context ah touch pannினா exceptions/data corruption varும்.
🗣️ Interview la eppadi sollanumAddDbContext registers the context as Scoped by default, so a new instance is created per HTTP request and disposed at the end of it. That matters because DbContext isn't thread-safe — if it were Singleton, every concurrent request would share the same context and change tracker, causing race conditions, stale tracked entities piling up in memory, and unpredictable exceptions when two requests touch it at once. Scoped keeps each request isolated with its own unit of work.
🔁 Follow-up Qs they might fire next
Q: Transient ah panнинா enna difference? — Transient creates a new instance every time it's injected, even multiple times within the same request — usually unnecessary overhead compared to Scoped for a typical web request.
Q: Background service la DbContext eppadi use pannுவீங்க, adhு Singleton alavаа? — I inject IServiceScopeFactory into the singleton background service and create a new scope (and DbContext) per unit of work instead of holding one context long-term.
Q: DbContextPool enna, edhுkки use pannுவீங்க? — AddDbContextPool reuses context instances from a pool instead of allocating new ones each request, reducing allocation overhead in high-throughput scenarios — I'd consider it if context construction became a measurable bottleneck.
14. Optimistic Concurrency — RowVersion/Concurrency Tokens (handling concurrent edits)
Rendு users ஒரே record ah ஒரே time la edit panнинанга na, யாروடa change last ah save aagும் nu problem varும் (namba data silently overwrite aagும் risk). Idha handle panна, table la oru "RowVersion" (timestamp) column vaikkும், EF Core model la adha [Timestamp] attribute illa Fluent API la IsRowVersion() nu configure pannுவோம். Update pannும்bodhu, EF andha WHERE clause la old RowVersion value ah include pannும் — DB la andha row already வேрே யாராவது update pannிருந்தா, RowVersion match aagாdhு, 0 rows affected aagும், EF adha DbUpdateConcurrencyException ah throw pannும் — apparam namba latest data ah reload panнி, user ku "வேрே யாராவது already update pannிருக்காங்க, refresh pannுங்க" nu kaatalaam.
🗣️ Interview la eppadi sollanumFor scenarios where two users might edit the same record concurrently, I add a RowVersion column and mark it as a concurrency token in EF Core. On update, EF includes the original RowVersion value in the WHERE clause — if someone else already changed that row, zero rows match, and EF throws a DbUpdateConcurrencyException. I catch that, reload the latest data, and let the user decide whether to overwrite or merge instead of silently losing someone else's change.
🔁 Follow-up Qs they might fire next
Q: RowVersion illama concurrency check pannலாமா? — Yes, you can mark any column as a concurrency token with Fluent API's IsConcurrencyToken(), but a dedicated rowversion/timestamp column is the standard SQL Server approach since it auto-updates on every write.
Q: Optimistic um pessimistic concurrency um difference enna? — Optimistic assumes conflicts are rare and checks at save time; pessimistic locks the row upfront so no one else can touch it until you're done — I use optimistic since it doesn't block other users.
Q: DbUpdateConcurrencyException vandha exact ah enna pannுவீங்க? — I reload the current database values, show the conflict to the caller (or merge non-conflicting fields), and let them resubmit with the fresh RowVersion.
15. Fluent API vs Data Annotations (model configuration approaches)
Model configuration rendு vazhi pannலாம் — Data Annotations na, entity class மேலேயே attribute pottுடுவோம் (example [Required], [MaxLength(100)], [Key]) — simple, quick, aana konjam limited (complex relationships, composite keys, indexes ellам attribute la full ah panна mudியாdhு). Fluent API na, DbContext oda OnModelCreating() method la, modelBuilder.Entity<Student>() vecha explicit ah configure pannுவோம் — idhு la every option available, complex composite keys, many-to-many custom join tables, check constraints, indexes ellам full control கிடைக்கும். Naan simple validation ku Data Annotations, real relationship/constraint configuration ku Fluent API use pannுவேன்.
🗣️ Interview la eppadi sollanumData Annotations are quick attributes right on the entity — Required, MaxLength, Key — good for simple, obvious rules. Fluent API in OnModelCreating gives full control — composite keys, precise relationship configuration, indexes, check constraints, things Data Annotations can't express at all. I generally use Data Annotations for basic validation and Fluent API for anything relational or database-specific, since it also keeps the entity classes clean of infrastructure concerns.
🔁 Follow-up Qs they might fire next
Q: Rendayும் together use pannலாமா? — Yes, they combine — Fluent API always wins if there's a conflict between the two on the same property.
Q: Composite key eppadi configure pannுவீங்க? — Only through Fluent API — modelBuilder.Entity<T>().HasKey(x => new { x.Prop1, x.Prop2 }) — Data Annotations can't define composite keys at all.
Q: Entity class ah "clean" ah vaikkanum nu solringale, adhு edhுkки matter aagும்? — Keeping DB-specific config out of the entity class follows separation of concerns — the entity stays a plain POCO, and all persistence details live in configuration classes (IEntityTypeConfiguration<T>), which is easier to maintain as the schema grows.
