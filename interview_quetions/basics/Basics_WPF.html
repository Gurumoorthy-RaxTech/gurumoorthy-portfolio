📐 What is XAML — and why WPF uses it

🎭
Anniyan (2005) — ஒரே DNA, மூணு தோற்றம்
Ambi/Remo/Anniyan — ஒரே ஆளு (same underlying "data"), aana 3 வேற வேற தோற்றம் (appearance). WPF-la idhu exactly antha maari — DataContext (the person/data) same-a irukum, aana DataTemplate maathi different-a "render" pannalam. UI (XAML) = character makeup. Data (ViewModel) = the real person underneath. Same ViewModel-a, list view-la summary-a kaatalam, detail view-la full card-a kaatalam — template mathum change aagum, data mathala.

XAML (eXtensible Application Markup Language) is a declarative XML-based markup used to describe UI structure — layout, controls, styling — separately from the C# code that drives behaviour. Each .xaml file pairs with a .xaml.cs code-behind via a generated partial class and InitializeComponent().

<!-- StudentSearchWindow.xaml -->
<Window x:Class="ZenCampus.Desktop.StudentSearchWindow"
        xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
        xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml"
        Title="Student Search" Height="450" Width="700">
    <Grid>
        <TextBox Text="{Binding SearchText, UpdateSourceTrigger=PropertyChanged}" Width="250" />
        <ListView ItemsSource="{Binding Results}" />
    </Grid>
</Window>
// StudentSearchWindow.xaml.cs — code-behind, kept thin
public partial class StudentSearchWindow : Window
{
    public StudentSearchWindow()
    {
        InitializeComponent();               // generated from the XAML above
        DataContext = new StudentSearchViewModel(); // see MVVM section
    }
}
Thanglish tip: Interview-la "Why XAML, why not just build UI in C# code?" kekalam. Answer: XAML declarative-a irukum (structure clear-a paaru mudiyum, like HTML), designer tools (Visual Studio/Blend) drag-drop generate pannalam, and it cleanly separates look (XAML) from behaviour (C#) — that separation is what makes MVVM possible.

🔗 Data Binding — the heartbeat of WPF

A binding connects a target property (e.g. TextBox.Text) to a source property (a ViewModel property) via a path. For the UI to react automatically when the source changes, the source class must implement INotifyPropertyChanged.

public class StudentSearchViewModel : INotifyPropertyChanged
{
    private string _searchText = "";
    public string SearchText
    {
        get => _searchText;
        set
        {
            if (_searchText == value) return;
            _searchText = value;
            OnPropertyChanged();             // tells the TextBox to update, and can trigger search
        }
    }

    public ObservableCollection<Student> Results { get; } = new();

    public event PropertyChangedEventHandler? PropertyChanged;
    protected void OnPropertyChanged([CallerMemberName] string? name = null)
        => PropertyChanged?.Invoke(this, new PropertyChangedEventArgs(name));
}
Binding ModeDirectionZenCampus Example
OneWaySource → Target onlyLabel showing Student.AdmissionNumber (read-only display)
TwoWaySource ↔ TargetTextBox editing Student.Phone in an edit form
OneWayToSourceTarget → Source onlyRare — a slider that only writes a filter value, never reads it back
OneTimeSource → Target, onceA dropdown of ClassSection options loaded once at window open

ObservableCollection<T> is the collection-level equivalent of INotifyPropertyChanged — it raises CollectionChanged automatically on Add/Remove, so a bound ListView updates itself without you touching the UI directly. Ithu than the whole point — ViewModel-la data update pannunga, UI automatic-a refresh aagum, listView.Items.Add(...) nu manual-a nee sollave venaam.

🧩 Dependency Properties

👻
Chandramukhi (2005) — Trigger vandha character மாறுது
Chandra character oru "trigger" condition vandha udane (night, palace room) transform aaguthu — same person, but behaviour/appearance changes based on a condition. WPF Style Trigger exactly idhu than: AttendanceStatus property "Absent" aana udane, badge color automatic-a red-a maarum — nee manual-a if vachu color set panna vendaam. That "automatic reaction to a property value" is only possible because the property is a DependencyProperty, not a plain C# field.

A regular CLR auto-property (public int Foo { get; set; }) is just a backing field. A DependencyProperty plugs into WPF's property system, unlocking things a plain property can't do: data binding, styling, animation, value coercion, and property-value inheritance down the visual tree.

// AttendanceStatusBadge.xaml.cs — custom UserControl
public partial class AttendanceStatusBadge : UserControl
{
    public static readonly DependencyProperty StatusProperty =
        DependencyProperty.Register(
            nameof(Status), typeof(string), typeof(AttendanceStatusBadge),
            new PropertyMetadata("Present"));

    public string Status
    {
        get => (string)GetValue(StatusProperty);
        set => SetValue(StatusProperty, value);
    }

    public AttendanceStatusBadge() => InitializeComponent();
}
<!-- Style trigger reacts to the DependencyProperty automatically -->
<Style TargetType="{x:Type local:AttendanceStatusBadge}">
    <Style.Triggers>
        <Trigger Property="Status" Value="Absent">
            <Setter Property="Background" Value="Red" />
        </Trigger>
    </Style.Triggers>
</Style>
Attached properties are the cousin concept — Grid.Row="1", Grid.Column="0" — a property defined by one class (Grid) but set on a completely different element (any child). Same DependencyProperty machinery underneath, just registered as RegisterAttached instead of a normal instance property.

🏛️ MVVM — Model / View / ViewModel

🎯
Thani Oruvan (2015) — Back office plan vs field execution
Mithran (hero) plan pannுவான் antha "logic" oru layer-la irukum, field-la execute panna weapon vera layer. MVVM-um idhu than: Model = raw truth (DB entity, no UI knowledge). ViewModel = the "planning room" — decides what to show, formats it, exposes Commands. View (XAML) = pure field execution — just renders whatever the ViewModel says, zero business logic inside. Testing venumna, ViewModel-a mattum test pannalam — no window open aaganum.

Model — the raw data / domain entity

Plain class, no WPF types, no UI knowledge. Usually your EF entity or DTO.

public class Student
{
    public int Id { get; set; }
    public string Name { get; set; } = "";
    public string AdmissionNumber { get; set; } = "";
}

ViewModel — state + behaviour, zero UI types

Implements INotifyPropertyChanged, exposes bindable properties and ICommands. The View never appears here — no MessageBox.Show, no Window reference.

public class StudentSearchViewModel : INotifyPropertyChanged
{
    private readonly IStudentRepository _repo;
    public StudentSearchViewModel(IStudentRepository repo)
    {
        _repo = repo;
        SearchCommand = new RelayCommand(async () => await SearchAsync(), () => !string.IsNullOrWhiteSpace(SearchText));
    }

    public ICommand SearchCommand { get; }
    public ObservableCollection<Student> Results { get; } = new();

    private async Task SearchAsync()
    {
        Results.Clear();
        foreach (var s in await _repo.SearchAsync(SearchText))
            Results.Add(s);
    }
    // SearchText property + INotifyPropertyChanged same as earlier example
}

View — XAML only, binds to the ViewModel

No business logic in code-behind. Just DataContext = new StudentSearchViewModel(...) and bindings do the rest (see the XAML section above).

RelayCommand — the glue that makes ICommand practical

WPF doesn't ship a ready-made ICommand you can new-up with a lambda (unlike modern MVVM toolkits' [RelayCommand] generator) — you write this small wrapper once and reuse it everywhere.

public class RelayCommand : ICommand
{
    private readonly Func<Task> _execute;
    private readonly Func<bool>? _canExecute;

    public RelayCommand(Func<Task> execute, Func<bool>? canExecute = null)
    {
        _execute = execute;
        _canExecute = canExecute;
    }

    public bool CanExecute(object? parameter) => _canExecute?.Invoke() ?? true;
    public async void Execute(object? parameter) => await _execute();
    public event EventHandler? CanExecuteChanged
    {
        add => CommandManager.RequerySuggested += value;
        remove => CommandManager.RequerySuggested -= value;
    }
}

⚡ Commands & Routed Events

📻
Kaithi (2019) — Walkie-talkie message relay
Dilli oru message pass pannுவான், adhu station to station relay aaghi mele varaikum poyi serum — each checkpoint kekum, react pannalam, illana next-ku pass pannும். WPF Bubbling routed event exactly idhu than: Button click event andha button-la start aagi, parent Grid, parent Window varaikum "bubble" pannும் — every ancestor gets a chance to handle it. Tunneling (Preview events) reverse direction — root-ல start aagi kulla down pogum.
ConceptDirection / BehaviourExample
Bubbling eventChild → Parent, up the visual treeClick, MouseDown
Tunneling eventParent → Child, down the visual treePreviewMouseDown, PreviewKeyDown
Direct eventFires only on the source element, no travelMouseEnter, GotFocus
ICommandDecoupled action — ViewModel owns CanExecute + ExecuteCommand="{Binding SearchCommand}"
<!-- Command binding — no Click event handler in code-behind at all -->
<Button Content="Search" Command="{Binding SearchCommand}" />

Why prefer Command over a Click event handler: the button's enabled state and its behaviour both live in the testable ViewModel — CanExecute automatically grays the button out when the search box is empty, with zero code-behind.

🗺️ Layout Panels

PanelArranges ChildrenZenCampus Example
GridRows & columns, like a tableStudent edit form — label column + input column
StackPanelSingle row or column, one after anotherVertical toolbar of action buttons (Add, Edit, Delete)
DockPanelDocks children to Top/Bottom/Left/Right, last fills remaining spaceMain window shell — menu docked Top, status bar Bottom, content fills center

Interview trap: "Which panel is fastest?" — Canvas is cheapest to lay out since it does zero measurement logic, but that's rarely the right reason to pick it. Pick the panel that matches the layout semantics you need (table → Grid, shell → DockPanel) — performance only matters at a scale desktop apps rarely hit.

🎯 10 Interview Scenarios

1
What is XAML and why does WPF use it instead of plain C# UI code?Basics
XAML is a declarative XML markup describing UI structure (controls, layout, styling) separately from behaviour code. WPF uses it because: 1) declarative structure is easier to read/scan than nested constructor calls, 2) design tools (Visual Studio/Blend) can generate and preview it visually, 3) it enforces separation between "look" and "logic," which is what makes MVVM practical. Every XAML file compiles to a partial class; InitializeComponent() wires it to the code-behind.
2
What is data binding, and what binding modes exist?Binding
Data binding wires a target UI property to a source (usually ViewModel) property via a path, resolved through DataContext. Modes: OneWay (source→target only), TwoWay (both directions, e.g. an editable TextBox), OneWayToSource (target→source only, rare), OneTime (read once, never again). Default mode depends on the property — TextBox.Text defaults to TwoWay, most others default to OneWay.
3
What is INotifyPropertyChanged and why does the UI need it?Binding

An interface with a single event, PropertyChanged. WPF's binding engine subscribes to it — when your ViewModel property setter raises this event, every bound control refreshes automatically. Without it, a TextBox bound to a ViewModel property will show the initial value correctly but never update again after the ViewModel changes it from code. It's the single most common reason a "binding doesn't update" bug happens in interviews and in real code.

4
What is a DependencyProperty, and how is it different from a normal C# property?DependencyProperty

A DependencyProperty is registered with WPF's property system via DependencyProperty.Register, giving it capabilities a plain auto-property doesn't have: data binding as a target, styling/templating, animation, value coercion, and property-value inheritance down the element tree. The cost is boilerplate — a static field plus a CLR wrapper property. Use it whenever you build a custom control (like a NumericTextBox or a status badge) that needs to participate in styling or binding.

5
What is MVVM, and why is it the standard pattern for WPF?MVVM

Model (plain data), View (XAML, no logic), ViewModel (state + Commands, exposes what the View binds to, implements INotifyPropertyChanged). It's the standard because WPF's binding engine makes the connection nearly free — you don't need a framework to wire a TextBox to a property. The payoff: ViewModels are plain C# classes you can unit-test with zero UI involved, and designers/developers can work on XAML and logic somewhat independently.

6
ICommand vs a Click event handler — what's actually different?Commands
<!-- event handler approach — logic ends up in code-behind -->
<Button Click="OnSearchClick" />

<!-- command approach — logic stays in the testable ViewModel -->
<Button Command="{Binding SearchCommand}" />

A Click handler lives in code-behind and has no way to disable the button declaratively. An ICommand bundles both the action (Execute) and its enabled state (CanExecute) in the ViewModel — WPF automatically grays out the button when CanExecute returns false, and none of that logic touches the View.

7
Explain routed events — bubbling vs tunneling, with an example.Events

Bubbling: the event starts at the source element and travels up through every ancestor in the visual tree (e.g. Click — a Button inside a Grid inside a Window; the Window can handle a click that happened on the Button). Tunneling: the event starts at the root and travels down to the source first (e.g. PreviewMouseDown fires on the Window before it fires on the Button beneath the cursor). Most tunneling events have a bubbling counterpart with the same name minus "Preview."

8
What is a DataTemplate and when would you use one?Templates

A DataTemplate defines how a bound data object should be visually rendered — e.g. how each Student in an ItemsSource should look inside a ListView, without changing the Student class itself. Use it whenever a list of plain data objects needs a custom visual layout (a photo + name + status badge per row) instead of the default ToString() rendering.

9
StaticResource vs DynamicResource — what's the difference?Resources

StaticResource resolves the resource once, at load time — faster, but if the resource changes later (e.g. a runtime theme swap), the control won't pick up the change. DynamicResource resolves lazily and re-resolves whenever the resource value changes — slightly slower, but required for anything that needs to update live, like a light/dark theme toggle.

10
You're asked to build a new internal desktop tool — WPF or WinForms?Decision

Depends on scope. A throwaway internal utility with a couple of forms — WinForms is faster to bang out and has almost no learning curve. Anything with dense/evolving UI, live data updates, theming, or where you expect requirements to grow — WPF's binding and templating pay for their extra setup cost quickly. For a CAD/CAM-style product with rich property panels and live previews (the kind of product this interview guide is aimed at), WPF is the standard answer.
