🎯 10 Interview Scenarios

1
Azure SQL Database vs SQL Managed Instance vs SQL on a VM — when would you pick each?Data

Azure SQL Database: fully managed PaaS, single-database or elastic pool, no OS access, fastest to provision — pick this for a new cloud-native app like Zen Campus where you don't need SQL Agent, cross-database queries, or OS-level access. SQL Managed Instance: near-100% SQL Server compatibility (SQL Agent, cross-DB queries, linked servers) with less management than a VM — pick this when migrating a legacy on-prem app that relies on instance-level features. SQL on a VM: full IaaS, you patch the OS and SQL Server yourself — pick this only when you need full control (specific SQL Server version, custom OS configuration, or software that must run alongside SQL Server).

2
Your API currently reads a SQL connection string from appsettings.json. How would you secure it in Azure?Security

Move the connection string into Azure Key Vault as a secret. Enable System-Assigned Managed Identity on the App Service. Grant that identity "Get" permission on the Key Vault secret (via access policy or RBAC). Reference the secret in App Settings using a Key Vault reference (@Microsoft.KeyVault(SecretUri=...)), or fetch it at startup with Azure.Identity + Azure.Security.KeyVault.Secrets. Result: zero credentials in source control or config files, and secrets can be rotated without redeploying code.

3
You have 6 Dockerized microservices behind Ocelot on-prem. How do you move this to Azure with minimal re-architecture?Compute

Push each service's image to Azure Container Registry. Deploy each as an Azure Container App (or Web App for Containers if simpler) — this gives managed scaling and HTTPS without learning full Kubernetes. Replace Ocelot with Azure API Management in front of the services for routing, rate limiting, and auth policies — or keep Ocelot running as one of the containers if you want to migrate incrementally rather than rewrite the gateway config immediately. Either way, the Docker images themselves barely change; it's the hosting and gateway layer that moves.

4
What's the difference between Azure App Service, Azure Container Apps, and AKS — and how do you choose?Compute

App Service: simplest, best for a single web app or API, PaaS — Azure fully manages the runtime, minimal config. Container Apps: built on Kubernetes under the hood but abstracts it away — good for multiple microservices needing scale-to-zero, revisions, and Dapr-style service-to-service calls, without managing a cluster. AKS: full Kubernetes control — pick it when you need custom scheduling, service meshes, complex networking, or your team already knows Kubernetes operations. Rule of thumb: start with App Service or Container Apps; only reach for AKS when you have a concrete reason.

5
How would you migrate your MongoDB collections to Azure with the least code change?Data

Use Azure Cosmos DB with the API for MongoDB (formerly "Mongo API"). It speaks the MongoDB wire protocol, so your existing MongoDB.Driver C# code and connection string pattern work with only the connection string swapped. You get Cosmos DB's global distribution, automatic indexing, and SLA-backed throughput without rewriting queries. The trade-off: some advanced native MongoDB features/aggregation operators may have partial compatibility, so check the compatibility docs for anything unusual you use.

6
Explain the difference between horizontal and vertical scaling in Azure App Service.Fundamentals

Vertical scaling ("scale up"): move to a bigger App Service Plan tier — more CPU/RAM per instance, same number of instances. Good for a single-instance bottleneck. Horizontal scaling ("scale out"): increase the instance count — more copies of the same app running behind the load balancer, handling more concurrent requests. Can be manual or autoscale-based on CPU/memory/queue-length rules. For a stateless ASP.NET Core Web API (no in-process session state), horizontal scaling is usually the better first move because it also improves availability.

7
How would you replace your hand-rolled JWT issuing/validation with Microsoft Entra ID?Security

Register the API and client app in Microsoft Entra ID (App Registrations). Entra ID issues the JWT access tokens instead of your own code — it handles login, MFA, token refresh, and revocation. Your ASP.NET Core API just validates incoming tokens using the standard JWT Bearer middleware configured with Entra ID's authority/issuer and audience — AddMicrosoftIdentityWebApi() from Microsoft.Identity.Web does most of this. Role-based authorization then maps Entra ID app roles or groups to your existing [Authorize(Roles="...")] attributes with little change to controller code.

8
What's the difference between Service Bus, Event Grid, and Event Hub?Messaging

Service Bus: reliable, ordered, transactional message queuing/pub-sub between services — closest to what you'd want for "send this fee-payment confirmation, guarantee it's processed exactly once." Event Grid: lightweight, reactive event routing (e.g. "a blob was uploaded," "a resource was created") — near-real-time, fire-and-forget notifications to many subscribers. Event Hub: high-throughput streaming ingestion for large volumes of telemetry/log data (think IoT locker sensor data at scale), built for analytics pipelines, not per-message guaranteed processing. For your bank-locker TCP/SMS flow, Service Bus is the closest match.

9
What is a Managed Identity and why is it better than a service principal with a stored secret?Security

A Managed Identity is an identity Azure automatically creates and manages for a resource (like an App Service), with no credentials for you to store, rotate, or leak. System-assigned: tied to the resource's lifecycle, deleted when the resource is deleted, one-to-one. User-assigned: a standalone identity you create once and attach to multiple resources, useful when several App Services need the same permissions. Compare to a manually created service principal with a client secret — you'd have to store that secret somewhere (defeating the purpose) and remember to rotate it. Managed Identity removes that entire class of risk.
