🤔 Why CQRS? The Problem It Solves

Traditional service: StudentService has GetStudent() + CreateStudent() + UpdateStudent() + GetStudentReport() + GetAttendanceSummary() — 20 methods, single class, one DB model for everything. At scale, read and write have DIFFERENT needs.

🎬
Thalapathy (1991) — Surya Writes, Dev Reads
Thalapathy-la Surya (Rajini) = action man, executes commands. Dev (Mammootty) = gathers information, reads. CQRS same: Commands = state changes (create, update, delete). Queries = data reads. Different classes, different handlers, different DB models if needed!
Traditional ServiceCQRS Approach
One model for read + writeWrite model (entities) + Read model (DTOs)
All in StudentService — one classEach operation = separate Command/Query + Handler
Hard to scale reads independentlyRead handlers can use read replicas / projections
Business logic mixed with query logicCommands have validation + domain logic; Queries are simple
Controller knows service methods directlyController only knows mediator.Send() — no coupling

📤 Commands — State-Changing Operations

// NuGet: MediatR
// Command — contains all data needed to change state
public record CreateStudentCommand(
    string Name,
    string AdmissionNumber,
    DateOnly DateOfBirth,
    int    ClassId,
    int    SchoolId,
    string ParentPhone
) : IRequest<Result<int>>;  // IRequest<TResponse>

// Handler — single responsibility: handle this one command
public class CreateStudentCommandHandler(
    IStudentRepository _repo,
    IUnitOfWork        _uow,
    ILogger<CreateStudentCommandHandler> _log)
    : IRequestHandler<CreateStudentCommand, Result<int>>
{
    public async Task<Result<int>> Handle(
        CreateStudentCommand cmd, CancellationToken ct)
    {
        if (await _repo.ExistsAsync(cmd.SchoolId, cmd.AdmissionNumber, ct))
            return Result<int>.Failure("Admission number already exists");

        var student = Student.Create(cmd.Name, cmd.AdmissionNumber, cmd.DateOfBirth,
                                     cmd.ClassId, cmd.SchoolId);
        await _repo.AddAsync(student, ct);
        await _uow.CommitAsync(ct);

        _log.LogInformation("Student {Id} admitted to school {SchoolId}", student.Id, cmd.SchoolId);
        return Result<int>.Success(student.Id);
    }
}

// Controller — thin, only mediator.Send
[HttpPost]
public async Task<IActionResult> Create(CreateStudentCommand cmd, CancellationToken ct)
{
    var result = await _mediator.Send(cmd, ct);
    return result.IsSuccess ? Ok(result.Value) : BadRequest(result.Error);
}

📥 Queries — Read Operations (No State Change)

// Query — parameters only, returns DTO never domain entity
public record GetStudentQuery(int StudentId, int SchoolId) : IRequest<StudentDetailDto?>;

public class GetStudentQueryHandler(ZenCampusDbContext _db)
    : IRequestHandler<GetStudentQuery, StudentDetailDto?>
{
    public async Task<StudentDetailDto?> Handle(
        GetStudentQuery qry, CancellationToken ct)
        => await _db.Students
            .Where(s => s.Id == qry.StudentId && s.SchoolId == qry.SchoolId)
            .Select(s => new StudentDetailDto
            {
                Id              = s.Id,
                Name            = s.Name,
                AdmissionNumber = s.AdmissionNumber,
                ClassName       = s.Class.Name,
                AttendancePct   = s.AttendanceRecords.Count == 0 ? 0 :
                    s.AttendanceRecords.Count(a => a.IsPresent) * 100.0 / s.AttendanceRecords.Count,
                PendingFees     = s.FeeRecords.Where(f => !f.IsPaid).Sum(f => f.Amount)
            })
            .AsNoTracking()
            .FirstOrDefaultAsync(ct);
}

// Paged student list query
public record GetStudentsPagedQuery(int SchoolId, int Page, int PageSize, string? SearchName)
    : IRequest<PagedList<StudentSummaryDto>>;
Query Rule: Queries NEVER modify state. Query handlers can be read-optimised — AsNoTracking, read replicas, cached projections. Commands validate domain rules. This separation lets you scale reads and writes independently.

⚙️ Pipeline Behaviors — Cross-Cutting Concerns

// Logging behavior — wraps EVERY command/query
public class LoggingBehavior<TRequest, TResponse>(
    ILogger<LoggingBehavior<TRequest, TResponse>> _log)
    : IPipelineBehavior<TRequest, TResponse> where TRequest : notnull
{
    public async Task<TResponse> Handle(TRequest req,
        RequestHandlerDelegate<TResponse> next, CancellationToken ct)
    {
        var name = typeof(TRequest).Name;
        _log.LogInformation("Handling {Request}", name);
        var sw = Stopwatch.StartNew();
        var response = await next();  // call the actual handler
        sw.Stop();
        _log.LogInformation("Handled {Request} in {Ms}ms", name, sw.ElapsedMilliseconds);
        return response;
    }
}

// Validation behavior — runs FluentValidation before handler
public class ValidationBehavior<TRequest, TResponse>(
    IEnumerable<IValidator<TRequest>> _validators)
    : IPipelineBehavior<TRequest, TResponse> where TRequest : notnull
{
    public async Task<TResponse> Handle(TRequest req,
        RequestHandlerDelegate<TResponse> next, CancellationToken ct)
    {
        var context = new ValidationContext<TRequest>(req);
        var failures = _validators
            .Select(v => v.Validate(context))
            .SelectMany(r => r.Errors)
            .Where(e => e != null)
            .ToList();
        if (failures.Count > 0)
            throw new ValidationException(failures);
        return await next();
    }
}

// Register MediatR + Behaviors in Program.cs
builder.Services.AddMediatR(cfg =>
{
    cfg.RegisterServicesFromAssembly(typeof(Program).Assembly);
    cfg.AddBehavior(typeof(IPipelineBehavior<,>), typeof(LoggingBehavior<,>));
    cfg.AddBehavior(typeof(IPipelineBehavior<,>), typeof(ValidationBehavior<,>));
});

📡 Domain Events — Decoupled Side Effects

// Domain Event — raised when student admitted
public record StudentAdmittedEvent(int StudentId, string StudentName,
    string ParentPhone, int SchoolId) : INotification;

// Handler 1 — Send welcome SMS
public class SendWelcomeSmsHandler(IMsg91Client _sms)
    : INotificationHandler<StudentAdmittedEvent>
{
    public async Task Handle(StudentAdmittedEvent e, CancellationToken ct)
        => await _sms.SendAsync(e.ParentPhone,
            $"Welcome! {e.StudentName} admitted. Student ID: {e.StudentId}", ct);
}

// Handler 2 — Create audit log
public class AuditAdmissionHandler(IAuditRepository _audit)
    : INotificationHandler<StudentAdmittedEvent>
{
    public async Task Handle(StudentAdmittedEvent e, CancellationToken ct)
        => await _audit.LogAsync("StudentAdmitted", e.StudentId, e.SchoolId, ct);
}

// In CreateStudentCommandHandler — publish after commit
await _uow.CommitAsync(ct);
await _mediator.Publish(new StudentAdmittedEvent(
    student.Id, student.Name, cmd.ParentPhone, cmd.SchoolId), ct);
// Both handlers called automatically — decoupled!

🎯 10 Interview Scenarios

1
What is CQRS and why would you use it in ZenCampus?Concept

CQRS separates read (Query) and write (Command) operations. ZenCampus: student dashboard reads (attendance %, fees, class rank) have different performance requirements than writes (admit student, record fee). With CQRS: queries use optimised projections with AsNoTracking, can hit read replicas. Commands go through validation and domain logic. Scales independently.

2
What is MediatR and what problem does it solve?Concept

MediatR is an in-process mediator — controllers don't call services directly, they send requests to MediatR, which routes to the appropriate handler. Solves: tight coupling (controller knows service), fat services (20 methods). Each command/query has one handler. Adding new operation = new class, zero changes to controller or existing handlers.

3
What is a Pipeline Behavior in MediatR? Give two ZenCampus uses.Architecture

Pipeline Behavior wraps every command/query — like middleware for MediatR. Runs before/after every handler. ZenCampus uses: (1) LoggingBehavior: log request name + elapsed time for every operation — no code change needed in handlers. (2) ValidationBehavior: run FluentValidation on every command before it reaches the handler — centralized validation, handlers assume valid input.

4
Should a Query handler return domain entities or DTOs?Best Practice

Always DTOs — never domain entities from query handlers. Domain entities are for command-side (change tracking, business rules). Returning entities from queries: (1) accidentally exposes unnecessary data, (2) tempts callers to modify and expect SaveChanges, (3) loads full entity when only 3 columns needed. Queries should project directly to DTOs via Select() — optimal SQL, minimal memory.

5
What is the difference between IRequest and INotification in MediatR?Concept

IRequest: exactly ONE handler. Send() method. Returns a response. Used for Commands and Queries. INotification: MULTIPLE handlers can subscribe. Publish() method. No response. Used for Domain Events — student admitted triggers SMS + audit + email all as separate handlers, all fire when published.

6
How does CQRS help with the Single Responsibility Principle?SOLID

Traditional StudentService has 15+ methods — one class, many responsibilities. With CQRS: CreateStudentCommandHandler has ONE job: handle student creation. GetStudentQueryHandler has ONE job: read student data. Each class has a single, clear reason to change. New feature = new command+handler class. SRP naturally enforced by CQRS structure.

7
How do you handle transactions in CQRS? What if SMS fails after student created?Production Bug

Separate DB commit from side effects. Student creation: (1) Create student + commit (DB transaction). (2) Publish StudentAdmittedEvent. (3) SMS handler fires. If SMS fails → student already committed. Solution: Outbox Pattern — store event in same DB transaction as the entity change. Background worker reads outbox and sends SMS with retry. Guarantees at-least-once delivery.

8
Is CQRS necessary for small projects like a 5-page admin panel?Architecture Decision

No. CQRS adds complexity: more classes, more files, more patterns to understand. For simple CRUD with 5-10 operations, traditional service pattern is fine. CQRS benefits appear at scale: many features, team of 5+ developers, need to scale reads/writes differently, complex domain logic. ZenCampus with 50k students and reporting needs = good CQRS candidate.

9
Can CQRS use the same database for reads and writes?Clarification

Yes — CQRS is primarily about separating CODE (command handlers vs query handlers), not necessarily separate databases. Same DB, different access patterns. Advanced: read replica for queries (Azure SQL read replica for ZenCampus reports), write to primary. Even more advanced: separate read store (Redis, Elasticsearch). Start with same DB, optimise later.
