1 Why Docker? — "Works on My Machine" Problem Solved
🎬 Kaaka Muttai — Same vadai, same recipe, same box. Different city-la delivery. Taste same! Docker = same container, any server, same behavior.

Dev machine-la ZenCampus run ஆகுது. Production-la crash. "Works on my machine!" = classic problem. Docker = package app + all dependencies + runtime into one image. Run anywhere — dev laptop, staging, production, k8s. Behaviour identical!

❌ Without Docker
# Dev machine: .NET 8.0.5
# Prod server: .NET 6.0.2  ← different!
# Dev: SQL Server 2022
# Prod: SQL Server 2019 ← different!
# Dev: Redis 7.2
# Prod: Redis 6.0 ← different!
# Result: "Works on my machine!" 😭
✅ With Docker
# Dockerfile pins exact versions
FROM mcr.microsoft.com/dotnet/aspnet:8.0
# Same image on all environments!
# Dev, staging, prod = identical runtime
# docker run zencampus:latest
# → same behavior everywhere ✅
Docker vs VM: VM: Hardware → Hypervisor → Guest OS (2GB) → App Container: Hardware → OS → Docker Engine → App (share host OS kernel!) Container = lightweight (MB not GB), starts in seconds not minutes Same isolation as VM for app dependencies, NOT for OS kernel
2 Dockerfile — Multi-Stage Build for .NET
# Single-stage (BAD) — image includes SDK (1.8GB!)
FROM mcr.microsoft.com/dotnet/sdk:8.0
WORKDIR /app
COPY . .
RUN dotnet publish -c Release -o out
ENTRYPOINT ["dotnet", "out/ZenCampusWeb.dll"]
# Image size: ~1.8GB  ← huge! SDK not needed at runtime!
# Multi-stage (GOOD) — SDK for build, ASP.NET runtime for run
####  Stage 1: Build  ####
FROM mcr.microsoft.com/dotnet/sdk:8.0 AS build
WORKDIR /src

# Copy csproj first — layer cache! (packages only re-download if .csproj changes)
COPY ["ZenCampusWeb/ZenCampusWeb.csproj", "ZenCampusWeb/"]
COPY ["ZenCampus.Core/ZenCampus.Core.csproj", "ZenCampus.Core/"]
RUN dotnet restore "ZenCampusWeb/ZenCampusWeb.csproj"

# Copy everything else and build
COPY . .
WORKDIR /src/ZenCampusWeb
RUN dotnet publish -c Release -o /app/publish --no-restore

####  Stage 2: Runtime  ####
FROM mcr.microsoft.com/dotnet/aspnet:8.0 AS runtime
WORKDIR /app

# Security: run as non-root user!
RUN adduser --disabled-password --gecos "" appuser
USER appuser

# Copy only published output from build stage
COPY --from=build --chown=appuser /app/publish .

# Health check — Docker knows if container is healthy
HEALTHCHECK --interval=30s --timeout=3s --retries=3 \
    CMD curl -f http://localhost:8080/health || exit 1

EXPOSE 8080
ENTRYPOINT ["dotnet", "ZenCampusWeb.dll"]
# Image size: ~250MB  ← 7x smaller! Only runtime, no SDK
# .dockerignore — don't copy these into build context
**/bin
**/obj
**/.vs
**/node_modules
**/*.user
.git
.gitignore
README.md
# Without .dockerignore: COPY . . copies 500MB of build artifacts!
# With .dockerignore: only source files, fast build
💡 Layer Caching — The Key to Fast Builds

Docker caches each layer. If a layer hasn't changed, it uses cache. Copy .csproj first + restore → this layer cached unless NuGet packages change. Copy source code after → only invalidates from that layer down. Result: code change = only compile layer rebuilds (seconds, not minutes)!

$ docker build -t zencampus-web:latest .
$ docker run -p 8080:8080 -e "ConnectionStrings__ZenCampus=Server=db..." zencampus-web:latest
$ docker images zencampus-web # check image size
$ docker history zencampus-web:latest # see layers

4 Kubernetes Concepts — The Vocabulary
🎬 Baahubali — Amarendra commands the army. Kubernetes = Amarendra. Pods = soldiers. Node = battalion. Deployment = battle strategy. Service = communication channel. HPA = call reinforcements!

K8s orchestrates containers. Amarendra doesn't fight — he commands. K8s doesn't run code — it manages which containers run where, restarts failed ones, scales up when load increases. Master plan (Desired State) → K8s makes it happen.

Kubernetes Architecture — ZenCampus: Control Plane (Master): ├── API Server — kubectl talks here ├── Scheduler — decides which Node runs a Pod ├── Controller — ensures desired state (restart dead pods) └── etcd — cluster state storage (like DB for k8s) Worker Nodes (where containers run): Node 1 (4 CPU, 16GB) Node 2 (4 CPU, 16GB) ├── Pod: student-service-abc ├── Pod: student-service-xyz ├── Pod: fee-service-def ├── Pod: fee-service-ghi └── Pod: redis-aaa └── Pod: hangfire-worker-bbb Pod = 1 or more containers running together (usually 1) Node = VM or physical machine where pods run Deployment = manages pod replicas, rolling updates Service = stable DNS name + load balancing for pods Namespace = virtual cluster — isolate school-a from school-b ConfigMap = non-secret config (URLs, feature flags) Secret = sensitive data (DB passwords, API keys) Ingress = HTTP routing rules → routes /api → services HPA = auto-scale pods based on CPU/memory
ResourcePurposeZenCampus Use

5 Kubernetes YAML — Deploy ZenCampus StudentService
# deployment.yaml — manages pod replicas
apiVersion: apps/v1
kind: Deployment
metadata:
  name: student-service
  namespace: zencampus
  labels:
    app: student-service
spec:
  replicas: 3                    # 3 pods always running
  selector:
    matchLabels:
      app: student-service
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 1           # at most 1 pod down during deploy
      maxSurge: 1                 # 1 extra pod during deploy = 4 total
  template:
    metadata:
      labels:
        app: student-service
    spec:
      containers:
      - name: student-service
        image: zencampus/student-service:1.2.0    # pin version, not "latest"!
        ports:
        - containerPort: 5001
        resources:
          requests:                               # guaranteed minimum
            cpu: "250m"                            # 0.25 CPU cores
            memory: "256Mi"
          limits:                                 # hard maximum
            cpu: "500m"
            memory: "512Mi"                       # exceed → OOMKilled!
        env:
        - name: ASPNETCORE_ENVIRONMENT
          value: Production
        - name: ConnectionStrings__ZenCampus      # from Secret!
          valueFrom:
            secretKeyRef:
              name: zencampus-secrets
              key: db-connection-string
        - name: Redis__Connection                 # from ConfigMap
          valueFrom:
            configMapKeyRef:
              name: zencampus-config
              key: redis-connection
        livenessProbe:
          httpGet:
            path: /health/live
            port: 5001
          initialDelaySeconds: 15
          periodSeconds: 20
        readinessProbe:
          httpGet:
            path: /health/ready
            port: 5001
          initialDelaySeconds: 10
          periodSeconds: 10
# service.yaml — stable DNS name for StudentService pods
apiVersion: v1
kind: Service
metadata:
  name: student-service         # DNS: student-service.zencampus.svc.cluster.local
  namespace: zencampus
spec:
  selector:
    app: student-service        # routes to pods with this label
  ports:
  - port: 5001
    targetPort: 5001
  type: ClusterIP               # only internal cluster access

---  three types of Service  ---
ClusterIP   — internal only (service-to-service)   ← use this for microservices
NodePort    — exposes on each node IP:port (dev/testing)
LoadBalancer— creates cloud LB (AWS/Azure) with public IP  ← use for API Gateway
# ingress.yaml — HTTP routing to services
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: zencampus-ingress
  namespace: zencampus
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
    nginx.ingress.kubernetes.io/rate-limit: "100"
spec:
  tls:                                    # HTTPS!
  - hosts: [api.zencampus.com]
    secretName: zencampus-tls-secret
  rules:
  - host: api.zencampus.com
    http:
      paths:
      - path: /api/students
        pathType: Prefix
        backend:
          service:
            name: student-service
            port:
              number: 5001
      - path: /api/fees
        pathType: Prefix
        backend:
          service:
            name: fee-service
            port:
              number: 5002
6 ConfigMap + Secret — Never Hardcode Config!
⛔ NEVER do this — hardcoded secrets in Dockerfile or appsettings.json committed to Git!

Connection strings, JWT secrets, API keys in appsettings.json → committed to Git → exposed to all developers, CI logs, Docker image history. Real incident: AWS keys in GitHub repo = $50,000 bill in 2 hours (crypto miners).

# configmap.yaml — non-sensitive configuration
apiVersion: v1
kind: ConfigMap
metadata:
  name: zencampus-config
  namespace: zencampus
data:
  redis-connection: "redis-service:6379"
  rabbitmq-host:    "rabbitmq-service"
  sms-gateway-url:  "https://sms.gateway.com/api"
  max-students-per-school: "5000"
  feature-flag-new-ui: "true"

---
# secret.yaml — sensitive data (base64 encoded, not encrypted by default!)
apiVersion: v1
kind: Secret
metadata:
  name: zencampus-secrets
  namespace: zencampus
type: Opaque
stringData:                              # k8s base64-encodes automatically
  db-connection-string: "Server=sqlserver;Database=ZenCampusDB;User=sa;Password=Str0ngP@ss!"
  jwt-secret:           "zencampus-super-secret-jwt-key-min-32-chars"
  sms-api-key:          "sk_live_abc123"
  redis-password:       "redisSecretPass"
// In .NET — environment variables override appsettings.json automatically
// k8s injects Secret/ConfigMap as env vars → app reads via IConfiguration

// appsettings.json — NO secrets, only structure + dev defaults
{
  "ConnectionStrings": {
    "ZenCampus": "Server=localhost;..."  // dev default only
  }
}

// Production: k8s env var ConnectionStrings__ZenCampus overrides this!
// Double underscore __ = colon : in .NET config hierarchy

// Program.cs — reads config from env vars automatically
var connStr = builder.Configuration.GetConnectionString("ZenCampus");
// In k8s: reads env var ConnectionStrings__ZenCampus (from Secret)
// Locally: reads appsettings.Development.json
💡 Production Secret Management

K8s Secrets = base64, NOT encrypted. Store real secrets in: Azure Key Vault + CSI driver (mounts as files), AWS Secrets Manager, HashiCorp Vault. Never commit Secret YAML with real values — use kubectl create secret from CLI or CI/CD pipeline env vars. In interview: "Secrets in Azure Key Vault, mounted via CSI driver — never in YAML files."

7 HPA — Auto-Scale for Month-End Fee Rush
🎬 Bigil — Normal match 11 players போதும். Final match = extra reserves call! HPA = same. Normal day 3 pods. Month-end fee rush → 10 pods auto-scale!

ZenCampus: 1st of month, 50,000 parents paying fees simultaneously. CPU spikes to 90%. HPA notices → scales FeeService from 3 → 10 pods in 60 seconds. Rush passes → scales back to 3. Pay only for what you use!

# hpa.yaml — HorizontalPodAutoscaler
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: fee-service-hpa
  namespace: zencampus
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: fee-service
  minReplicas: 3                # always at least 3
  maxReplicas: 15               # month-end scale limit
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 60   # scale when CPU > 60%
  - type: Resource
    resource:
      name: memory
      target:
        type: Utilization
        averageUtilization: 70
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 30    # wait 30s before scaling up again
      policies:
      - type: Pods
        value: 3                          # add max 3 pods at once
        periodSeconds: 60
    scaleDown:
      stabilizationWindowSeconds: 300   # wait 5 min before scaling down

Pod CrashLoopBackOff — StudentService keeps restarting! 🔥 Production Hard
📋 Scenario: "kubectl get pods shows: student-service-abc 0/1 CrashLoopBackOff 7 12m. How do you debug this? Walk through your exact investigation steps."

# Step 1: Previous pod logs — crash reason is in the CRASHED container

# OOMKilled     → increase limits.memory in deployment.yaml

Rolling update — zero downtime deploy in ZenCampus Medium Real Interview
📋 Scenario: "'How do you deploy a new version of ZenCampus StudentService with zero downtime on Kubernetes?'"
✅ ROLLING UPDATE STRATEGY

Setup: Deployment with strategy.type: RollingUpdate, maxUnavailable: 1, maxSurge: 1. Readiness probe on /health/ready.

What happens: 3 pods running v1.2. Deploy v1.3: k8s starts 1 new v1.3 pod (now 4 total). Waits for readiness probe to pass (DB connected, Redis connected). Only then terminates 1 old v1.2 pod. Repeats. During entire process: always ≥2 healthy pods serving traffic. Zero downtime.

Critical: Readiness probe MUST check real dependencies (DB, Redis). Without it, k8s sends traffic to v1.3 pod before it's ready → 502 errors for users.

Rollback: kubectl rollout undo deployment/student-service → instantly rolls back to previous revision. Takes 30 seconds — previous image already cached on nodes.

# Watch rolling update in real-time
kubectl rollout status deployment/student-service -n zencampus
# Waiting for deployment to finish: 1 out of 3 new replicas have been updated...
# Waiting for deployment to finish: 2 out of 3 new replicas have been updated...
# Waiting for deployment to finish: 3 out of 3 new replicas have been updated...
# deployment "student-service" successfully rolled out

OOMKilled — pod killed due to memory limit Medium
📋 Scenario: "FeeService pod killed every night at 2 AM during batch report generation. kubectl logs shows process killed. No exception in logs. kubectl describe pod shows 'OOMKilled'. How to fix?"
✅ OOM — DIAGNOSE + FIX LIMITS
# OOMKilled = Linux kernel killed process for exceeding memory limit
# Container says limits.memory: 512Mi, report generation uses 600Mi → killed!

# Step 1: Measure actual memory during batch
kubectl top pods -n zencampus --sort-by=memory

# Step 2: Fix in deployment.yaml — increase limits
resources:

    memory: "256Mi"    # guaranteed minimum
    cpu: "250m"
  limits:
    memory: "1Gi"      # increased from 512Mi
    cpu: "500m"

# Step 3: Optimize code — report loads all 50k rows at once!
# Fix: stream/paginate report generation
await foreach (var batch in _repo.GetStudentsBatchedAsync(1000))
{
    await WriteReportBatchAsync(batch);
    // GC can collect each batch after processing — constant memory!
}
# Result: Memory usage: 598Mi → 80Mi (constant regardless of student count)

HPA not scaling — month-end fee rush, pods stuck at 3! 🔥 Production Hard
📋 Scenario: "1st of month, 10,000 parents hitting fee payment simultaneously. FeeService CPU at 90%+. HPA configured minReplicas:3 maxReplicas:15. But kubectl get hpa shows TARGETS: unknown/60%. Pods stuck at 3. Why?"
✅ HPA TROUBLESHOOTING — MISSING METRICS SERVER
# TARGETS: unknown = HPA can't read CPU metrics!
# Root cause: metrics-server not installed in cluster

# NAME           REFERENCE         TARGETS         MINPODS  MAXPODS  REPLICAS

    cpu: "500m"

# Fix 3: Verify metrics-server working

kubectl describe hpa fee-service-hpa -n zencampus
# Events: SuccessfulRescale: New size: 8, reason: cpu resource utilization above target

apiVersion: v1

metadata:

Q1: Multi-stage Docker build — main benefit?

A) Faster container startup time
B) Final image contains only runtime (no SDK) — 7x smaller, no build tools in production
C) Allows building multiple apps in one Dockerfile
D) Enables parallel layer caching
✅ Multi-stage = SDK image (1.8GB) for building, aspnet runtime image (250MB) for running. Final image has ONLY the published DLLs + runtime — no source code, no compiler, no build tools. Smaller = faster pull, smaller attack surface. Also: source code not in production image (security benefit)!
❌ Main benefit = image size! SDK = 1.8GB (includes compiler, build tools). ASP.NET runtime = 250MB (just runtime). Multi-stage: Stage 1 builds using SDK. Stage 2 starts fresh from runtime, copies only /app/publish output. 7x smaller. Also: source code not in final image!

Q2: Pod TARGETS: <unknown>/60% in HPA — what's the most likely cause?

A) HPA YAML is misconfigured
B) CPU limit not set in deployment
C) metrics-server not installed, OR resource requests not set in deployment
D) HPA needs restart after deployment change
✅ HPA reads CPU metrics from metrics-server. Without metrics-server → can't get metrics → TARGETS: unknown. Also: HPA calculates utilization% as (currentCPU / requests.cpu × 100) — if requests.cpu not set → can't calculate % → unknown. Fix: install metrics-server + always set resource requests!
❌ TARGETS unknown = HPA can't get metrics! Two causes: (1) metrics-server not installed in cluster — HPA can't read CPU data. (2) resource requests not set — HPA can't calculate utilization percentage (needs requests.cpu as the baseline). Fix both!

Q3: image:latest production-la use பண்ணக்கூடாது — why?

A) :latest pulls the wrong OS version
B) Pod restarts silently pull new :latest image — broken builds deploy without explicit deployment!
C) k8s doesn't support :latest tag
D) :latest makes rollback impossible
✅ :latest = non-deterministic! CI pushes broken build tagged :latest. Next pod restart (node failure, OOMKilled, etc.) → k8s pulls new :latest → broken code in production without anyone doing a deploy! Use git SHA tags: image: zencampus/student-service:sha-a1b2c3d. Every image = traceable to exact commit.
❌ :latest danger = silent unexpected deploys! imagePullPolicy: Always (default for :latest) → every pod restart pulls new image from registry. If CI pushed broken :latest, pod restarts after node failure = broken code. Solution: versioned tags (git SHA). Explicit, deterministic, rollback-able.

Q4: Kubernetes Secret-la data என்ன format-la store ஆகும்?

A) AES-256 encrypted
B) Base64 encoded (NOT encrypted by default!)
C) Plain text with RBAC access control
D) SHA-256 hashed
✅ K8s Secrets = Base64 in etcd — NOT encrypted! echo "mypassword" | base64 → bXlwYXNzd29yZA== (easily decoded!). Must enable etcd encryption at rest separately (EncryptionConfiguration). Better: use Azure Key Vault / AWS Secrets Manager + CSI driver. Never commit Secret YAML with real values!
❌ K8s Secrets are Base64 encoded, NOT encrypted! kubectl get secret zencampus-secrets -o jsonpath='{.data.db-connection-string}' | base64 -d → plain text password! Anyone with kubectl access can decode. Production: enable etcd encryption at rest + use Azure Key Vault CSI driver for real security.
