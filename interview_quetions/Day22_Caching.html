200 schools, 50,000 students — every page load DB-a hit panna — DB collapses. Fee structure data? Same for entire year. School list? Changes once a month. Cache panna — memory-la irukku, microseconds-la return aagum.

IMemoryCache — In-Process Cache

🧠 Fastest cache — lives in app memory

Single server-la irundha — IMemoryCache best. No network hop, RAM-la store, sub-millisecond read. But: 3 server instances irundha — each has its own cache = inconsistency!

// Program.cs
builder.Services.AddMemoryCache(opts =>
{
    opts.SizeLimit            = 1024;     // Max 1024 units (arbitrary — you define "size")
    opts.CompactionPercentage = 0.25;    // Remove 25% when limit hit
});
// GetOrCreate — the standard pattern
public async Task<FeeStructure> GetFeeStructureAsync(int schoolId, CancellationToken ct)
{
    var key = $"fee-structure:school:{schoolId}";

    return await _cache.GetOrCreateAsync(key, async entry =>
    {
        entry.AbsoluteExpirationRelativeToNow = TimeSpan.FromHours(6);  // 6h — fee structure rarely changes
        entry.SlidingExpiration               = TimeSpan.FromMinutes(30); // Reset on each access
        entry.Priority                        = CacheItemPriority.High;      // Don't evict easily
        entry.Size                            = 1;                          // 1 unit of SizeLimit

        var structure = await _db.FeeStructures
            .FirstOrDefaultAsync(f => f.SchoolId == schoolId, ct);

        _log.LogInformation("Cache MISS fee-structure for school {Id}", schoolId);
        return structure!;
    }) ?? throw new NotFoundException($"Fee structure not found for school {schoolId}");
}

// Explicit invalidation — when fee structure is updated
public async Task UpdateFeeStructureAsync(int schoolId, UpdateFeeDto dto, CancellationToken ct)
{
    await _db.FeeStructures.UpdateAsync(schoolId, dto, ct);
    await _db.SaveChangesAsync(ct);

    _cache.Remove($"fee-structure:school:{schoolId}"); // ← Invalidate immediately
    _log.LogInformation("Cache invalidated: fee-structure school {Id}", schoolId);
}

Absolute vs Sliding Expiration:

Absolute: Expires after fixed time from creation (e.g., 6h from when cached). Use for data that must refresh periodically regardless of usage.

Sliding: Expires after X time from last access. Popular items stay alive. Use for user session data, active student profiles.

Both together: Sliding resets on each access, BUT absolute is the hard ceiling. Common combo: sliding 30min + absolute 6h.

CacheItemPriority:

Low — first to evict under memory pressure

Normal — default

High — evict only if necessary

NeverRemove — never evict (use carefully!)

ZenCampus: Fee structure = High, student list = Normal, report preview = Low

IDistributedCache — Shared Cache (Redis)

🔴 Cross-server cache — all instances share same data

ZenCampus-la 3 server instances irukku (load balanced) — IMemoryCache = 3 separate caches = inconsistency. Server 1-la student admitted, Server 2-la cache still shows old count. IDistributedCache (Redis) = one shared cache = consistency!

❌ IMemoryCache with 3 server instances:
Server 1 cache: studentCount = 1250 (updated)
Server 2 cache: studentCount = 1249 (stale — load balancer might route parent here)
Server 3 cache: studentCount = 1249 (stale)
Parent sees different numbers on refresh!
✅ Redis (IDistributedCache):
Redis: studentCount = 1250 (updated once)
All 3 servers read from Redis → always consistent
// NuGet: Microsoft.Extensions.Caching.StackExchangeRedis
// Program.cs
builder.Services.AddStackExchangeRedisCache(opts =>
{
    opts.Configuration = builder.Configuration["Redis:ConnectionString"];
    // Example: "zencampus-redis.redis.cache.windows.net:6380,password=xxx,ssl=True"
    opts.InstanceName  = "zencampus:";  // Key prefix — all keys become "zencampus:students:school:1"
});
// IDistributedCache — string/byte[] based (unlike IMemoryCache which accepts any object)
public class StudentCacheService(IDistributedCache _cache, ZenCampusDbContext _db)
{
    private static readonly DistributedCacheEntryOptions _opts5Min = new()
    {
        AbsoluteExpirationRelativeToNow = TimeSpan.FromMinutes(5)
    };

    public async Task<IReadOnlyList<StudentDto>> GetBySchoolAsync(int schoolId, CancellationToken ct)
    {
        var key    = $"students:school:{schoolId}";
        var cached = await _cache.GetStringAsync(key, ct);

        if (cached is not null)
        {
            _log.LogDebug("Cache HIT: {Key}", key);
            return JsonSerializer.Deserialize<List<StudentDto>>(cached)!;
        }

        _log.LogInformation("Cache MISS: {Key} — querying DB", key);
        var students = await _db.Students
            .Where(s => s.SchoolId == schoolId && !s.IsDeleted)
            .Select(s => new StudentDto(s.Id, s.Name, s.Class, s.RollNo))
            .ToListAsync(ct);

        await _cache.SetStringAsync(key, JsonSerializer.Serialize(students), _opts5Min, ct);
        return students;
    }

    public async Task InvalidateSchoolAsync(int schoolId, CancellationToken ct)
        => await _cache.RemoveAsync($"students:school:{schoolId}", ct);
}

Redis — Direct StackExchange.Redis

🔑 Beyond IDistributedCache — pub/sub, atomic ops, sorted sets

IDistributedCache only string/bytes support pannum — complex operations-ku direct Redis client use pannrom. Atomic increment (concurrent fee payment count), pub/sub (cache invalidation signal across servers), sorted sets (leaderboard).

// NuGet: StackExchange.Redis
builder.Services.AddSingleton<IConnectionMultiplexer>(
    ConnectionMultiplexer.Connect(new ConfigurationOptions
    {
        EndPoints        = { builder.Configuration["Redis:Host"]! },
        Password         = builder.Configuration["Redis:Password"],
        Ssl              = true,
        SslProtocols     = System.Security.Authentication.SslProtocols.Tls12,
        AbortOnConnectFail = false,    // Don't crash app if Redis down at startup
        ConnectRetry     = 3,
        ReconnectRetryPolicy = new ExponentialRetry(5000)
    }));
// Direct Redis operations
public class RedisService(IConnectionMultiplexer _redis)
{
    private IDatabase Db() => _redis.GetDatabase();

    // Atomic increment — concurrent-safe school student counter
    public async Task<long> IncrementStudentCountAsync(int schoolId)
        => await Db().StringIncrementAsync($"school:{schoolId}:student-count");

    // Set with TTL in one atomic op
    public async Task SetWithExpiryAsync(string key, string value, TimeSpan ttl)
        => await Db().StringSetAsync(key, value, ttl);

    // Distributed lock — prevent concurrent admission in same school
    public async Task<bool> AcquireLockAsync(string resource, TimeSpan expiry)
    {
        var lockKey = $"lock:{resource}";
        return await Db().StringSetAsync(lockKey, "1", expiry, When.NotExists);
        // When.NotExists = SET NX — only set if key doesn't exist (atomic!)
    }

    // Pub/Sub — notify all servers to invalidate cache
    public async Task PublishCacheInvalidationAsync(string key)
        => await _redis.GetSubscriber()
            .PublishAsync(RedisChannel.Literal("cache-invalidate"), key);

    // Subscribe to invalidation (in BackgroundService startup)
    public void SubscribeToCacheInvalidation(IMemoryCache localCache)
        => _redis.GetSubscriber().Subscribe(RedisChannel.Literal("cache-invalidate"),
            (_, key) => localCache.Remove((string)key!));
    // Multi-level cache: local IMemoryCache + Redis — invalidate both!
}

Cache-Aside Pattern

🔄 The most common pattern — application manages cache

Cache-Aside = Lazy loading. Application itself cache manage pannum — DB-a read pannum bo cache-la store pannum. Cache miss aana — DB-la poa, return pannum, cache-la save pannum.

Client Request │ ▼ Application ──1. Check cache──► Redis/Memory │ │ │ <─── 2a. CACHE HIT ─────────┘ ↩ return immediately (~1ms) │ │ 2b. CACHE MISS │ ▼ Application ──3. Query DB──────► SQL Server (~45ms) │ ▼ Application ──4. Store in cache─► Redis/Memory │ ▼ Return to Client 
// Cache-Aside as a generic helper extension
public static class DistributedCacheExtensions
{
    public static async Task<T?> GetOrSetAsync<T>(
        this IDistributedCache cache,
        string                   key,
        Func<Task<T?>>          factory,
        DistributedCacheEntryOptions opts,
        CancellationToken        ct = default)
    {
        var cached = await cache.GetStringAsync(key, ct);
        if (cached is not null)
            return JsonSerializer.Deserialize<T>(cached);

        var value = await factory();
        if (value is not null)
            await cache.SetStringAsync(key, JsonSerializer.Serialize(value), opts, ct);

        return value;
    }
}

// Usage — clean one-liner
var students = await _cache.GetOrSetAsync(
    key:     $"students:school:{schoolId}",
    factory: () => _db.Students.Where(s => s.SchoolId == schoolId).ToListAsync(ct),
    opts:    new() { AbsoluteExpirationRelativeToNow = TimeSpan.FromMinutes(5) },
    ct:      ct);
Cache-Aside vs Write-Through vs Read-Through:
Cache-Aside (Lazy): App reads cache, on miss reads DB + populates cache. App manages writes. Most common. ✅ ZenCampus standard.
Write-Through: Write to cache AND DB simultaneously. Always consistent but adds write latency. Use for high-read/frequent-write data.
Read-Through: Cache layer automatically fetches from DB on miss. Framework handles it (NCache, etc.). Less common in .NET world.

Cache Invalidation — The Hard Problem

🗑️ "There are only two hard things in CS: cache invalidation and naming things" — Phil Karlton

Cache stale aana — user wrong data kaanuvanga. ZenCampus-la fee structure update aana, 6 hours until expiry — all 200 schools see old fee! Invalidation = puriyave vendiya concept.

3 Invalidation Strategies:

// Strategy 1: TTL-based (passive) — simplest, some staleness accepted
new DistributedCacheEntryOptions
{
    AbsoluteExpirationRelativeToNow = TimeSpan.FromMinutes(5) // Auto-expire
}
// ✅ Simple. ❌ Stale for up to 5 min. ZenCampus: fee balance (1min TTL)

// Strategy 2: Explicit invalidation on write — immediate consistency
public async Task AdmitStudentAsync(AdmitStudentCommand cmd, CancellationToken ct)
{
    var student = new Student { Name = cmd.Name, SchoolId = cmd.SchoolId };
    _db.Students.Add(student);
    await _db.SaveChangesAsync(ct);

    // Invalidate ALL affected keys immediately
    await _cache.RemoveAsync($"students:school:{cmd.SchoolId}", ct);
    await _cache.RemoveAsync($"school:{cmd.SchoolId}:student-count", ct);
    // ✅ Immediate consistency. ❌ Must remember all keys to invalidate.
}

// Strategy 3: Tag-based invalidation (Output Cache or custom)
// Group keys by tag → invalidate entire tag at once
public async Task InvalidateSchoolDataAsync(int schoolId, CancellationToken ct)
{
    // Store tag → key mapping in Redis Set
    var tagKey  = $"tag:school:{schoolId}";
    var members = await _db.SetMembersAsync(tagKey);

    foreach (var cacheKey in members)
        await _cache.RemoveAsync(cacheKey, ct);
    await _db.KeyDeleteAsync(tagKey);
    // ✅ One call invalidates all school-related keys
}

Output Cache — ASP.NET Core 7+ built-in:

// Program.cs
builder.Services.AddOutputCache(opts =>
{
    opts.AddPolicy("SchoolDashboard", policy => policy
        .Expire(TimeSpan.FromMinutes(5))
        .SetVaryByQuery("schoolId")           // Different cache per schoolId
        .Tag("school-data"));                  // Tag for group invalidation

    opts.AddPolicy("StaticLookup", policy => policy
        .Expire(TimeSpan.FromHours(1))
        .SetVaryByHeader("Accept-Language")); // Different cache per language
});

app.UseOutputCache();

// Controller — cache entire HTTP response
[OutputCache(PolicyName = "SchoolDashboard")]
[HttpGet("{schoolId}/dashboard")]
public async Task<IActionResult> GetDashboard(int schoolId) { ... }

// Invalidate by tag — called after any school data change
[HttpPost("{schoolId}/clear-cache")]
public async Task<IActionResult> ClearCache(int schoolId, [FromServices] IOutputCacheStore store)
{
    await store.EvictByTagAsync("school-data", HttpContext.RequestAborted);
    return NoContent();
}

Cache Stampede — Thundering Herd

🐘 200 schools hit cache simultaneously at expiry — all miss, all hit DB

Morning 8AM-la 200 schools simultaneously login pannuvanga — fee structure cache expire aana — 200 simultaneous cache misses — 200 DB queries at same time — DB crash! Cache stampede / thundering herd problem.

❌ The Stampede:
8:00:00.000 — Cache expires for school list
8:00:00.001 — Request 1: cache miss → DB query started
8:00:00.002 — Request 2: cache miss → DB query started (not yet cached!)
8:00:00.003 — Request 3: cache miss → DB query started
... 200 more queries ... DB CPU 100%, timeout!
// Fix 1: SemaphoreSlim per key — only 1 thread fetches DB, rest wait
private static readonly ConcurrentDictionary<string, SemaphoreSlim> _locks = new();

public async Task<T?> GetOrSetWithLockAsync<T>(
    string key, Func<Task<T?>> factory,
    DistributedCacheEntryOptions opts, CancellationToken ct)
{
    // 1. Fast path — check cache without lock
    var cached = await _cache.GetStringAsync(key, ct);
    if (cached is not null) return JsonSerializer.Deserialize<T>(cached);

    // 2. Acquire per-key semaphore
    var sem = _locks.GetOrAdd(key, _ => new SemaphoreSlim(1, 1));
    await sem.WaitAsync(ct);
    try
    {
        // 3. Double-check after acquiring lock — might be populated now
        cached = await _cache.GetStringAsync(key, ct);
        if (cached is not null) return JsonSerializer.Deserialize<T>(cached);

        // 4. Only ONE thread reaches here — fetches from DB
        var value = await factory();
        if (value is not null)
            await _cache.SetStringAsync(key, JsonSerializer.Serialize(value), opts, ct);
        return value;
    }
    finally
    {
        sem.Release();
        _locks.TryRemove(key, out _);
    }
}
// Result: 200 concurrent misses → 1 DB query, 199 wait → serve from cache
// Fix 2: Jitter (randomize TTL) — prevents synchronized expiry
private DistributedCacheEntryOptions WithJitter(TimeSpan baseTtl)
{
    var jitter = TimeSpan.FromSeconds(Random.Shared.Next(0, 60)); // 0-60s random
    return new() { AbsoluteExpirationRelativeToNow = baseTtl + jitter };
}
// 200 schools each get different expiry → never all expire simultaneously
// School 1: expires at 8:04:37, School 2: 8:06:12, School 3: 8:03:55...

🎯 10 Interview Scenarios

Q1. IMemoryCache vs IDistributedCache — when to use which? ZenCampus has 3 load-balanced server instances.
Concept

IMemoryCache — single server, or data that can be different per server:

  • Lookup tables (classes, subjects, districts) — same data, ok if replicated
  • User-specific transient state — computation results, temp objects
  • Sub-millisecond requirement — no network hop

IDistributedCache (Redis) — 3 server instances:

  • Student lists — parent on Server 1 sees same data as parent on Server 2
  • Fee balance — must be consistent across all servers
  • Session state — user logs in via Server 1, next request might hit Server 2
  • Rate limiting counters — shared across servers

ZenCampus hybrid: "We use both. IMemoryCache for school config (20 schools' configs loaded once per server — tiny, ok to duplicate). Redis for student lists and fee balances where a parent must see consistent data regardless of which server handles their request."

Q2. Production Bug: Fee structure was updated by admin, but parents still see old fee amounts for 6 hours. Root cause and fix?
Production Bug

Root cause: Cache-Aside with 6h TTL only — no explicit invalidation on write. When admin updates fee structure in DB, the Redis cache still holds the old value until TTL expires.

// ❌ Original code — update DB but forget cache
public async Task UpdateFeeStructureAsync(int schoolId, UpdateFeeDto dto, CancellationToken ct)
{
    await _db.UpdateAsync(schoolId, dto, ct); // DB updated
    await _db.SaveChangesAsync(ct);
    // ← Cache NOT invalidated! Parents see stale data for up to 6h
}

// ✅ Fix — explicit invalidation after write
public async Task UpdateFeeStructureAsync(int schoolId, UpdateFeeDto dto, CancellationToken ct)
{
    await _db.UpdateAsync(schoolId, dto, ct);
    await _db.SaveChangesAsync(ct);
    await _cache.RemoveAsync($"fees:structure:school:{schoolId}", ct); // ← MUST add
    // Next request = cache miss → fresh DB read → cached with new TTL
}

Prevention: Document all cache keys each service reads/writes. Code review checklist: "Does this write invalidate all relevant cache keys?" Use Decorator pattern (CachedStudentRepository from Day 19) — cache logic co-located with write logic.

Q3. What is the Cache Stampede / Thundering Herd problem? How do you prevent it in ZenCampus?
Performance

Cache Stampede: Many requests arrive simultaneously for the same expired cache key. All get cache miss, all hit DB concurrently — DB gets overwhelmed.

Two complementary fixes:

  1. SemaphoreSlim lock per key: Only 1 request fetches from DB. Others wait, then serve from freshly populated cache. "Double-check" pattern after acquiring lock is critical.
  2. Jitter: Add random 0-60s to TTL so 200 schools don't all expire simultaneously. School 1 expires at 8:04:37, School 2 at 8:06:12 — spread the load.

Additional option — Stale-while-revalidate: Return stale cache immediately, update cache in background. Complex but zero latency spikes.

Q5. What data should you NEVER cache? What are the security implications?
Concept

Never cache:

  • Passwords / hashes — never cache, always fetch fresh (BCrypt verification needs DB)
  • JWT tokens — cache compromises revocation (need to check blacklist in DB)
  • Per-user sensitive data mixed together — cache key must always include userId. Never cache "all students" without schoolId — cross-tenant data leak risk
  • Financial transactions — fee receipt details must be from DB for audit trails
  • OTP codes — use Redis with exact TTL, but never extend sliding expiry on failed attempt
  • Permission/role data — if parent's account is suspended, stale cache might allow access

// ✅ ALWAYS scope by tenant
var key = $"students:school:{schoolId}";  // Tenant-isolated

Q7. How does Output Caching differ from IDistributedCache? When would you use each?
Concept

IDistributedCache: You manually cache data objects (DTOs, lists). Application controls when to read/write/invalidate. Fine-grained control. Works in any layer (service, repository).

Output Cache: ASP.NET Core caches the entire HTTP response (headers + body). Zero code in business logic. Works at middleware level before controller even executes. Declarative with [OutputCache] attribute.

ZenCampus: Output Cache for school dashboard GET (entire response is same for all users of same school). IDistributedCache for student list (used by multiple endpoints, need to invalidate on admission).

Q8. Redis is down. Should ZenCampus crash? How do you make caching resilient?
Resilience

Cache should NEVER be a single point of failure — it's a performance optimization, not the source of truth. If Redis is down, fall back to DB (slower but correct).

// Resilient cache-aside — Redis failure → DB fallback
public async Task<IReadOnlyList<StudentDto>> GetBySchoolAsync(int schoolId, CancellationToken ct)
{
    var key = $"students:school:{schoolId}";

    try
    {
        var cached = await _cache.GetStringAsync(key, ct);
        if (cached is not null)
            return JsonSerializer.Deserialize<List<StudentDto>>(cached)!;
    }
    catch (Exception ex)
    {
        // Redis down — log and continue (cache miss, DB fallback)
        _log.LogWarning(ex, "Redis unavailable, falling back to DB for {Key}", key);
    }

    var students = await _db.Students
        .Where(s => s.SchoolId == schoolId).ToListAsync(ct);

    try
    {
        await _cache.SetStringAsync(key, JsonSerializer.Serialize(students),
            new() { AbsoluteExpirationRelativeToNow = TimeSpan.FromMinutes(5) }, ct);
    }
    catch (Exception ex)
    {
        _log.LogWarning(ex, "Redis unavailable, could not cache {Key}", key);
    }

    return students; // Always returns data — just slower when Redis is down
}
// Also: AbortOnConnectFail = false in ConnectionMultiplexer config (Day 22 setup)

Q10. "Tell me about your caching experience in ZenCampus." — Full structured answer
Experience

"In ZenCampus — 50,000 students, 200 schools — we hit a performance wall when our school dashboard queries ran on every page load for all schools simultaneously at 8 AM. DB CPU hit 90%.

Multi-level caching strategy: IMemoryCache for static lookup tables (district names, class list — tiny, ok to be duplicated per server). Redis IDistributedCache for student lists and fee balances — we have 3 load-balanced servers, so data must be consistent regardless of which server handles a parent's request.

Cache-Aside pattern: Check Redis → hit returns immediately (~1ms), miss queries DB (~45ms) → stores result. I built a generic GetOrSetAsync extension method so every repository uses the same pattern.

Invalidation: We had a bug where fee structure updates didn't invalidate cache — parents saw old fee amounts for up to 6 hours. Fixed by adding explicit cache.RemoveAsync calls in every write command handler. Code review now checks this explicitly.

Stampede prevention: Added jitter (±60s random) to all TTLs so 200 schools don't expire simultaneously. SemaphoreSlim lock for high-traffic endpoints.

Result: DB CPU dropped from 90% to 12%. Dashboard response time: 450ms → 8ms average. Redis adds 99th percentile cache hit rate."

🧠 Quick Quiz

Q1. ZenCampus has 3 load-balanced servers. Student list is cached with IMemoryCache. Student admitted on Server 1. Server 2 still shows old count. Fix?
IDistributedCache (Redis) — the only correct fix. IMemoryCache is per-process; 3 servers = 3 separate caches = inconsistency. Sticky sessions don't scale and still break. Reducing TTL to 10s is a band-aid. Redis is the single shared cache for all instances.
Q2. Admin updates fee structure. Which cache strategy ensures parents see the new fee IMMEDIATELY?
Explicit invalidation — remove the key immediately after DB write. Next request = cache miss → fresh DB read → cache populated with new value. 1-minute TTL still has up to 1 min staleness. Write-through is correct too but requires different setup. Disabling cache defeats the purpose and DB gets hammered.
Q3. Morning 8 AM — fee structure cache for all 200 schools expires simultaneously. 200 requests hit DB at once. What prevents this?
TTL jitter — randomize each school's expiry by ±60 seconds so 200 schools expire at different times (spread over 2 minutes instead of all at once). Combine with SemaphoreSlim for the remaining concurrent misses. Output Cache doesn't solve synchronized expiry either. 24h TTL means stale data all day.
Q4. Redis goes down at 2 PM. What should happen to ZenCampus fee collection?
Fall back to DB — cache is a performance optimization, NOT the source of truth. Redis down = slower responses (~45ms instead of ~1ms) but fee collection must continue. Wrap cache reads in try-catch, log warning, fall through to DB. AbortOnConnectFail=false in ConnectionMultiplexer prevents startup crash when Redis is unavailable.
