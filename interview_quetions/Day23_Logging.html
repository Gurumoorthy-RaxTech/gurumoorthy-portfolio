Why Structured Logging?

📋 String logs vs Structured logs — searchable vs noise

Parent call panranga — "fee payment failed" — log-la search pannrom. String log-la? 50,000 lines grep panna vendum. Structured log-la? StudentId = 42, SchoolId = 7 directly query panna mudiyum. Night and day difference in production debugging.

❌ String logging (unstructured) 
// Can't query "all failures for school 7"
_log.LogError(
  "Fee payment failed for student "
  + studentId + " in school "
  + schoolId);

// App Insights stores as raw string:
// "Fee payment failed for student 42 in school 7"
// → grep-only, no filtering by property
✅ Structured logging 
// Query: StudentId = 42 OR SchoolId = 7
_log.LogError(
  "Fee payment failed for {StudentId} in {SchoolId}",
  studentId, schoolId);

// Stored as JSON object:
// { "StudentId": 42, "SchoolId": 7,
//   "Level": "Error", "Timestamp": "..." }
// → filter, aggregate, alert on properties
The Pillars of Observability:
Logs — discrete events ("fee collected", "student admitted") — what happened
Metrics — numeric measurements over time (request count, error rate, latency) — how much
Traces — end-to-end request journey across services (correlation ID) — where and how long
ZenCampus uses all three: Serilog (logs), App Insights (metrics + traces), Health Checks (probes)

Serilog Setup

📝 Serilog — structured logging with multiple sinks

ILogger interface Microsoft-oda — but Serilog-a provider-a use pannrom. NuGet packages install panna, Program.cs-la configure panna — done. All ILogger<T> calls automatically go through Serilog.

// NuGet: Serilog.AspNetCore, Serilog.Sinks.Console, Serilog.Sinks.File,
//        Serilog.Sinks.ApplicationInsights, Serilog.Enrichers.Environment,
//        Serilog.Enrichers.Thread, Serilog.Expressions

// Program.cs — before builder.Build()
builder.Host.UseSerilog((ctx, services, config) => config
    .ReadFrom.Configuration(ctx.Configuration)  // appsettings.json overrides
    .ReadFrom.Services(services)              // DI-injected enrichers
    .Enrich.FromLogContext()                   // LogContext.PushProperty() support
    .Enrich.WithMachineName()                  // Which server (Server1 vs Server2)
    .Enrich.WithEnvironmentName()              // Production / Staging
    .Enrich.WithProperty("Application", "ZenCampusAPI")  // App tag on every log
    .MinimumLevel.Information()               // Global minimum
    .MinimumLevel.Override("Microsoft", LogEventLevel.Warning)     // Reduce MS noise
    .MinimumLevel.Override("System", LogEventLevel.Warning)         // Reduce system noise
    .MinimumLevel.Override("Microsoft.EntityFrameworkCore", LogEventLevel.Warning)
    .WriteTo.Console(outputTemplate:
        "[{Timestamp:HH:mm:ss} {Level:u3}] {SourceContext}: {Message:lj} {Properties}{NewLine}{Exception}")
    .WriteTo.File(
        path:          "logs/zencampus-.log",
        rollingInterval: RollingInterval.Day,          // New file each day
        retainedFileCountLimit: 30,                  // Keep 30 days
        outputTemplate: "{Timestamp:yyyy-MM-dd HH:mm:ss} [{Level:u3}] {Message:lj} {Properties}{NewLine}{Exception}")
    .WriteTo.ApplicationInsights(
        services.GetRequiredService<TelemetryConfiguration>(),
        TelemetryConverter.Traces));

{

public class CollectFeeCommandHandler
{
    public async Task<CollectFeeResult> Handle(CollectFeeCommand cmd, CancellationToken ct)
    {
        _log.LogDebug("Processing fee for {StudentId}, amount {Amount}", cmd.StudentId, cmd.Amount);
        // ↑ DEBUG: internal processing detail, Dev/Staging only

        var receipt = await _repo.SaveReceiptAsync(cmd, ct);
        _log.LogInformation(
            "Fee collected: ReceiptNo={ReceiptNo} StudentId={StudentId} Amount={Amount} SchoolId={SchoolId}",
            receipt.ReceiptNo, cmd.StudentId, cmd.Amount, cmd.SchoolId);
        // ↑ INFO: business event — always log successful collections

        try
        {
            await _sms.SendReceiptAsync(receipt);
        }
        catch (SmsDeliveryException ex)
        {
            _log.LogWarning(ex,
                "SMS receipt not sent for {ReceiptNo} — will retry via Hangfire",
                receipt.ReceiptNo);
            // ↑ WARNING: SMS failed but fee is saved — degraded, not failed
            BackgroundJob.Enqueue(() => _sms.SendReceiptAsync(receipt));
        }
        catch (Exception ex)
        {
            _log.LogError(ex,
                "Fee collection failed for {StudentId} Amount={Amount}",
                cmd.StudentId, cmd.Amount);
            // ↑ ERROR: fee not saved — user impacted — alert fires
            throw;
        }
        return new CollectFeeResult(receipt.ReceiptNo);
    }
}

Correlation ID — Trace Requests End-to-End

🔗 One ID connects all log lines for one request

Parent call pannuvanga — "10 nimisham munnaadi fee pay panna try pannean, failed". Without correlation ID — 50,000 log lines from 200 schools — needle in haystack. With CorrelationId? One query, all logs for that exact request — controller, service, DB, SMS — all visible.

// Correlation ID Middleware — generates/propagates ID for every request
public class CorrelationIdMiddleware(RequestDelegate _next)
{
    private const string HeaderName = "X-Correlation-Id";

    public async Task InvokeAsync(HttpContext ctx)
    {
        // Accept from client (mobile app passing its own ID) or generate new
        var correlationId = ctx.Request.Headers[HeaderName].FirstOrDefault()
                         ?? Activity.Current?.Id           // From W3C trace context
                         ?? Guid.NewGuid().ToString("N");

        ctx.Items[HeaderName] = correlationId;
        ctx.Response.Headers.Append(HeaderName, correlationId); // Echo back to client

        // Push into Serilog's LogContext — all logs in this request get this property
        using (LogContext.PushProperty("CorrelationId", correlationId))
        using (LogContext.PushProperty("RequestPath", ctx.Request.Path))
        {
            await _next(ctx);
        }
    }
}

// Program.cs — register BEFORE all other middleware
app.UseMiddleware<CorrelationIdMiddleware>();
app.UseSerilogRequestLogging(opts =>
{
    opts.MessageTemplate =
        "HTTP {RequestMethod} {RequestPath} → {StatusCode} in {Elapsed:0.0}ms | {CorrelationId}";
    opts.EnrichDiagnosticContext = (diag, http) =>
    {
        diag.Set("SchoolId",   http.User.FindFirst("schoolId")?.Value);
        diag.Set("UserAgent", http.Request.Headers.UserAgent.ToString());
    };
});
// Every log line now includes CorrelationId automatically:

// [14:23:07 INF] Fee collected: ReceiptNo=ZC-2024-0042 Amount=5000 | CorrelationId=a3f9c2d1
// [14:23:07 DBG] Cache MISS: fees:balance:student:42 | CorrelationId=a3f9c2d1
// [14:23:07 WRN] SMS receipt not sent for ZC-2024-0042 — retry queued | CorrelationId=a3f9c2d1

// Support query in App Insights: customDimensions.CorrelationId = "a3f9c2d1"
// → ALL 4 lines surface immediately — trace the whole story

// Pass CorrelationId to downstream HTTP calls (Service Bus already propagates it)
public class CorrelationIdDelegatingHandler(IHttpContextAccessor _http) : DelegatingHandler
{
    protected override Task<HttpResponseMessage> SendAsync(
        HttpRequestMessage req, CancellationToken ct)
    {
        if (_http.HttpContext?.Items["X-Correlation-Id"] is string id)
            req.Headers.TryAddWithoutValidation("X-Correlation-Id", id);
        return base.SendAsync(req, ct);
    }
}

Health Checks

💚 /health — is ZenCampus alive and ready?

Azure load balancer / Kubernetes-la — healthy pod-ku traffic route pannuvom, unhealthy pod-ku route pannoma. Health check = "can this instance serve traffic?" DB down aana — unhealthy return pannuvom — traffic automatically shifted to other pods.

// NuGet: AspNetCore.HealthChecks.SqlServer, AspNetCore.HealthChecks.Redis,
//        AspNetCore.HealthChecks.UI

// Program.cs
builder.Services.AddHealthChecks()
    // Liveness — is the app running?
    .AddCheck("self", () => HealthCheckResult.Healthy(), tags: ["live"])
    // Readiness — can it serve traffic?
    .AddSqlServer(
        connectionString: builder.Configuration.GetConnectionString("ZenCampusDb")!,
        name:             "sql-db",
        failureStatus:    HealthStatus.Unhealthy,
        tags:             ["ready", "db"])
    .AddRedis(
        redisConnectionString: builder.Configuration["Redis:ConnectionString"]!,
        name:          "redis-cache",
        failureStatus: HealthStatus.Degraded,      // Redis down = degraded (not unhealthy)
        tags:          ["ready", "cache"])
    .AddCheck<Msg91HealthCheck>("sms-provider", tags: ["ready", "external"]);

// Map endpoints
app.MapHealthChecks("/health/live", new() { Predicate = c => c.Tags.Contains("live") });
app.MapHealthChecks("/health/ready", new()
{
    Predicate       = c => c.Tags.Contains("ready"),
    ResponseWriter  = UIResponseWriter.WriteHealthCheckUIResponse  // JSON detail
});
// Custom health check — ping Msg91 SMS provider
public class Msg91HealthCheck(IMsg91Client _client) : IHealthCheck
{
    public async Task<HealthCheckResult> CheckHealthAsync(
        HealthCheckContext ctx, CancellationToken ct = default)
    {
        try
        {
            var sw = Stopwatch.StartNew();
            var ok = await _client.PingAsync(ct);
            sw.Stop();

            if (!ok)
                return HealthCheckResult.Degraded("Msg91 ping returned unhealthy");

            return HealthCheckResult.Healthy($"Msg91 OK in {sw.ElapsedMilliseconds}ms");
        }
        catch (Exception ex)
        {
            return HealthCheckResult.Unhealthy("Msg91 unreachable", ex);
        }
    }
}
// /health/ready response (HealthChecks.UI format):
{
  "status":   "Degraded",
  "totalDuration": "00:00:00.0432",
  "entries": {
    "sql-db":      { "status": "Healthy",  "duration": "00:00:00.0310" },
    "redis-cache": { "status": "Degraded", "duration": "00:00:00.0089",
                     "description": "Redis timeout after 1s" },
    "sms-provider": { "status": "Healthy",  "duration": "00:00:00.0033" }

Healthy vs Degraded vs Unhealthy:
Healthy — all checks pass, serve traffic normally
Degraded — Redis down (Day 22 fallback to DB!) — still serve traffic but at reduced performance
Unhealthy — DB down — cannot serve meaningful traffic → load balancer removes from rotation

public class HealthCheckFilter(ITelemetryProcessor next) : ITelemetryProcessor
{
    public void Process(ITelemetry item)
    {
        if (item is RequestTelemetry req &&
            req.Url?.AbsolutePath.StartsWith("/health") == true)
            return; // Drop health check requests — they pollute metrics
        next.Process(item);
    }
}
// Custom events + metrics in command handlers
public class CollectFeeCommandHandler(TelemetryClient _telemetry)
{
    public async Task<CollectFeeResult> Handle(CollectFeeCommand cmd, CancellationToken ct)
    {
        var sw = Stopwatch.StartNew();
        try
        {
            var receipt = await _repo.SaveReceiptAsync(cmd, ct);
            sw.Stop();

            // Custom event — queryable in App Insights Analytics
            _telemetry.TrackEvent("FeeCollected", new Dictionary<string, string>
            {
                ["SchoolId"]    = cmd.SchoolId.ToString(),
                ["ReceiptNo"]  = receipt.ReceiptNo,
                ["PaymentMode"] = cmd.PaymentMode
            },
            new Dictionary<string, double>
            {
                ["Amount"]       = (double)cmd.Amount,
                ["ProcessTimeMs"] = sw.ElapsedMilliseconds
            });

            // Custom metric — dashboard chart: daily fee collection per school
            _telemetry.TrackMetric("FeeAmount", (double)cmd.Amount,
                new Dictionary<string, string> { ["SchoolId"] = cmd.SchoolId.ToString() });

            return new CollectFeeResult(receipt.ReceiptNo);
        }
        catch (Exception ex)
        {
            sw.Stop();
            // Exception telemetry with custom properties — auto-grouped in Failures blade
            _telemetry.TrackException(ex, new Dictionary<string, string>
            {
                ["StudentId"] = cmd.StudentId.ToString(),
                ["SchoolId"]  = cmd.SchoolId.ToString(),
                ["Amount"]    = cmd.Amount.ToString()
            });
            throw;
        }
    }
}

🎯 10 Interview Scenarios

Q1. Why is _log.LogError("Fee failed for student " + id) bad? What's the correct approach?
Concept

String concatenation problems:

  1. Not queryable: Log stored as a flat string — can't filter "all errors for school 7" in App Insights
  2. Performance: String allocation happens even if log level is disabled. With template: LogInformation("...", id) — if INFO is disabled, no string is created
  3. No structure: Kibana/Seq/App Insights can't extract StudentId as a field to group, aggregate, or alert on
  4. Security risk: String interpolation with user input can log sensitive data accidentally
// ❌ String concatenation
_log.LogError("Fee failed for student " + studentId + " school " + schoolId);

// ✅ Message template — named parameters become JSON properties
_log.LogError(ex, "Fee collection failed {StudentId} {SchoolId} {Amount}",
    studentId, schoolId, amount);
// Stored: { StudentId: 42, SchoolId: 7, Amount: 5000, Level: "Error", ... }
// Query: customDimensions.StudentId = "42" → instant result
Q2. Production: Parent calls saying "fee payment failed 10 minutes ago." No error visible in dashboard. How do you find it?
Production Debug

Step-by-step investigation using correlation ID + structured logging:

  1. Ask parent for the X-Correlation-Id header value from their failed request (mobile app should show it in error screen)
  2. Query App Insights: traces | where customDimensions.CorrelationId == "a3f9c2d1"
  3. See all log lines for that request — controller, handler, DB, SMS
  4. If no CorrelationId: query by approximate time + StudentId: where timestamp between (ago(15m)..ago(5m)) and customDimensions.StudentId == "42"
  5. Check exceptions blade — filter by timestamp window
// App Insights KQL — find the failed request
traces
| where timestamp > ago(30m)
| where customDimensions.StudentId == "42"
| where severityLevel >= 3  // Warning and above
| project timestamp, message, customDimensions, severityLevel
| order by timestamp asc

// Alternative — search all request telemetry for school
requests
| where timestamp > ago(30m)
| where customDimensions.SchoolId == "7"
| where success == "False"
| project timestamp, url, resultCode, duration, operation_Id

Key insight: Without structured logging + correlation ID, this investigation takes hours. With both, minutes.

Q3. What is a Correlation ID? How do you propagate it across microservices in ZenCampus?
Concept

Correlation ID is a unique identifier assigned to each incoming request and attached to every log line produced while handling that request. When a request calls multiple services, the ID is forwarded in HTTP headers so logs across all services can be joined.

Propagation in ZenCampus:

  1. Mobile app generates CorrelationId → sends in X-Correlation-Id header
  2. CorrelationIdMiddleware reads/generates, pushes to Serilog LogContext
  3. All log lines in that request automatically include CorrelationId
  4. Outbound HTTP calls (to Msg91 SMS API): CorrelationIdDelegatingHandler adds header
  5. Service Bus messages: add CorrelationId to ApplicationProperties
  6. Response echoes it back: client can show it in error messages for support

W3C Trace Context (OpenTelemetry standard): traceparent header — auto-propagated by .NET HttpClient. App Insights uses this for end-to-end distributed traces (Application Map).

Q4. What should log level be for: (a) EF Core SQL queries, (b) Successful fee collection, (c) Msg91 timeout after 3 retries, (d) Null reference in fee handler?
Concept

(a) EF Core SQL queries → DEBUG
Internal diagnostic — too verbose for production. Enable in Staging only when investigating query performance. Override: "Microsoft.EntityFrameworkCore": "Debug" in appsettings.Staging.json.

(b) Successful fee collection → INFO
Important business event — always log. Include structured properties: ReceiptNo, Amount, SchoolId, PaymentMode. Useful for audit, analytics, and debugging without noise.

(c) Msg91 timeout after 3 retries → WARNING
Fee was saved to DB — the core operation succeeded. SMS is a side effect that will be retried via Hangfire. Degraded state — log as Warning to alert ops but not trigger incident. If all retries permanently fail → ERROR.

(d) Null reference in fee handler → ERROR
Unhandled exception — fee was NOT collected — user impacted. Log with exception object (stack trace). Should trigger alert. The IExceptionHandler (Day 20) catches and logs this automatically.

Q5. What is the difference between Liveness and Readiness health checks? How does Kubernetes use them?
Concept

Liveness probe (/health/live): "Is the process alive and not deadlocked?" Only checks the app itself — not its dependencies. If fails → Kubernetes restarts the pod. Check: memory not exhausted, no deadlock, application started.

Readiness probe (/health/ready): "Can this pod serve traffic?" Checks all dependencies. If fails → Kubernetes removes pod from load balancer rotation (but does NOT restart). Check: DB reachable, Redis reachable, downstream APIs up.

// Kubernetes deployment yaml
livenessProbe:
  httpGet: { path: /health/live, port: 8080 }
  initialDelaySeconds: 10    # Wait 10s for app to start
  periodSeconds: 15           # Check every 15s
  failureThreshold: 3         # 3 failures → restart

readinessProbe:
  httpGet: { path: /health/ready, port: 8080 }
  periodSeconds: 10
  failureThreshold: 2         # 2 failures → remove from LB

ZenCampus: "During DB maintenance, readiness returns Unhealthy → traffic routes to other pods. After maintenance, readiness recovers → pod rejoins rotation. Liveness still passes throughout (app itself didn't crash)."

Q7. What is OpenTelemetry? How does it differ from Serilog + App Insights?
Concept

OpenTelemetry (OTel) is a vendor-neutral standard for collecting logs, metrics, and traces. Single instrumentation → export to any backend (App Insights, Jaeger, Grafana, Datadog, Seq).

Q8. What should you NEVER log? What are the GDPR/compliance risks?
Security

Never log (PII / sensitive data):

  • Passwords (plain or hashed) — even BCrypt hash must not be logged
  • JWT tokens — attacker with log access can impersonate any user
  • Full phone numbers — mask to +91******7890
  • Aadhaar/PAN numbers (Indian law: IT Act, DPDP Act 2023)
  • Student health/medical records (especially DPDP Act sensitive data)
  • Bank account / payment card details
  • Session/refresh tokens

Q9. ZenCampus morning peak: 500 errors/minute suddenly. How do you set up automated alerting?
Observability

Alert layers in Azure:

  1. App Insights Smart Detection — auto-alerts on anomalies (sudden error rate spike). No setup needed, ML-based.
  2. Metric Alerts — threshold-based: "error rate > 5% for 5 minutes → PagerDuty"
  3. Log Alerts — KQL query-based: "count of FATAL logs > 0 in 1 minute → immediate page"
  4. Custom metric alert — "FeeCollected events = 0 for 30min during 8AM-6PM → Slack warning"
// App Insights KQL — alert query
// "Error rate" alert — fires when > 5% of requests fail
requests
| where timestamp > ago(5m)
| summarize
    total   = count(),
    failed  = countif(success == "False"),
    errRate = round(100.0 * countif(success == "False") / count(), 2)
| where errRate > 5

// "Dead school" alert — school 7 has zero activity for 30 min during business hours
customEvents
| where timestamp > ago(30m)
| where name in ("FeeCollected", "StudentAdmitted", "AttendanceMarked")
| where customDimensions["SchoolId"] == "7"
| count
| where Count == 0  // No events → alert!
Q10. "Tell me about your logging and observability experience." — Full ZenCampus answer
Experience

"In ZenCampus — 50,000 students, 200 schools — debugging production issues without good observability was nearly impossible. We had incidents where parents called support about failed payments and we'd spend hours grepping log files with no context.

Serilog with structured logging: Replaced all string concatenation with message templates — "Fee collected {ReceiptNo} {StudentId} {Amount} {SchoolId}". App Insights stores these as queryable JSON properties. Added a custom SchoolContextEnricher that pulls SchoolId and UserId from the JWT claim — every log line automatically tagged with tenant context.

Correlation ID middleware: Generates a unique ID per request, pushes to Serilog LogContext, echoes in response header. We display it in the mobile app's error screen. Support team takes that ID → App Insights query → see controller log, handler log, DB log, SMS log — all for that one request. Investigation time went from hours to 2 minutes.

Health checks: Three endpoints — /health/live (app alive), /health/ready (DB + Redis + Msg91 reachable). Azure Load Balancer routes based on readiness. During a DB maintenance window, readiness went Unhealthy → traffic auto-shifted to remaining pods — zero user impact.

App Insights: Custom events for business KPIs — FeeCollected with amount, school, payment mode. KQL dashboards show daily collection per school. Alert fires if FeeCollected count drops to zero during business hours — caught a DB connectivity issue before any parent called."

🧠 Quick Quiz

Q1. You write: _log.LogDebug("Processing " + studentId + " amount " + amount); — what are the two main problems?
Both problems: (1) Unstructured — studentId stored as part of the string, not a queryable property — App Insights can't filter "all debug logs for student 42". (2) String allocation — concatenation runs even if DEBUG level is disabled (filtered before writing). Fix: _log.LogDebug("Processing {StudentId} amount {Amount}", studentId, amount);
Q2. Msg91 SMS fails after 3 retries. Fee was saved to DB. The Hangfire retry job is queued. What log level?
LogWarning — the core operation (fee collection) succeeded. SMS is a side effect currently in retry queue. System is degraded but functional. LogError should be reserved for operations where the user's goal (fee collection) failed. If all Hangfire retries exhaust → then LogError. Error alerts fire incidents; warnings are monitored but don't page on-call.
Q3. During DB maintenance, /health/ready returns Unhealthy. /health/live returns Healthy. What does Kubernetes do?
Removes pod from LB rotation (no restart). Liveness = "is app alive?" (restart if no). Readiness = "can it serve traffic?" (remove from LB if no, but keep running). During DB maintenance, app is alive (liveness pass) but can't serve traffic (readiness fail) → correct behavior: take it out of rotation until DB recovers.
Q4. Parent calls support: "Fee payment failed at 2:15 PM." What's the fastest way to find the logs using observability best practices?
Correlation ID query — if the mobile app shows the CorrelationId in the error, one App Insights query surfaces all logs (controller, handler, DB, SMS) for that exact request. Without CorrelationId: query by StudentId + timestamp range in App Insights structured logs. SSH grep is slow, unstructured, and requires server access. DB shows the result, not the cause.
