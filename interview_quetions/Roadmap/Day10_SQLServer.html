1 Indexes — Clustered vs Non-Clustered vs Covering
🎬 Thuppakki — Sniper shoots exact target (Index Seek) vs spraying everywhere (Table Scan)

Index Seek = Jagadish Kumar — direct shot, exact row. Table Scan = AK-47 spray — scan every row in table. 50,000 students table-la student name search — no index = scan all 50k rows. Index = jump directly!

TypeStructureCount per tableZenCampus example
ClusteredData rows sorted by key — B-Tree leaf = actual rowOnly 1Students.StudentId (PK auto-creates CI)
Non-ClusteredSeparate B-Tree, leaf = row pointer (RID or CI key)Up to 999Students.AdmissionNo, FeeRecords.SchoolId

-- ZenCampus: FeeRecords table — common query patterns

-- Clustered Index (auto-created by PRIMARY KEY)
CREATE TABLE FeeRecords (
    FeeRecordId   INT IDENTITY(1,1) PRIMARY KEY,  -- CI on FeeRecordId
    StudentId     INT NOT NULL,
    SchoolId      INT NOT NULL,
    AcademicYear  NVARCHAR(10),
    FeeType       NVARCHAR(50),
    Amount        DECIMAL(10,2),
    PaidDate      DATE,
    Status        NVARCHAR(20)
);

-- Non-Clustered Index — frequent WHERE SchoolId + AcademicYear
CREATE NONCLUSTERED INDEX IX_FeeRecords_School_Year
    ON FeeRecords (SchoolId, AcademicYear);

-- Covering Index — query needs Name, Amount too → add INCLUDE!
CREATE NONCLUSTERED INDEX IX_FeeRecords_School_Year_Covering
    ON FeeRecords (SchoolId, AcademicYear)
    INCLUDE (StudentId, FeeType, Amount, Status);
-- Now: SELECT StudentId, FeeType, Amount, Status
--       WHERE SchoolId = 42 AND AcademicYear = '2026-27'
-- → Index Seek only! No Key Lookup → fastest!

-- Filtered Index — only active students (most queries filter this)
CREATE NONCLUSTERED INDEX IX_Students_Active
    ON Students (SchoolId, AdmissionNo)
    WHERE Status = 'Active';
-- Smaller index — only active rows (80% of queries!)

-- Check index usage — is your index being used?
SELECT
    i.name AS IndexName,
    ius.user_seeks, ius.user_scans, ius.user_lookups, ius.user_updates
FROM sys.indexes i
JOIN sys.dm_db_index_usage_stats ius
    ON i.object_id = ius.object_id AND i.index_id = ius.index_id
WHERE OBJECT_NAME(i.object_id) = 'FeeRecords';
⚠️ Index Column Order Matters — Composite Index Left-Prefix Rule!

Index on (SchoolId, AcademicYear): WHERE SchoolId = 42 ✅ (leftmost prefix — seeks). WHERE AcademicYear = '2026' ❌ (skip leftmost — scan). WHERE SchoolId = 42 AND AcademicYear = '2026' ✅ (both — seeks). Design composite index with most selective + most frequently filtered column FIRST.

📋 Execution Plan — Know these operators!

3 SARGable Queries — Make Your Index Work!
🎬 Kaththi — Villain uses villagers' weakness directly. SARGable = index weakness directly exploited (in a good way!)

SARGable = Search ARGument ABLE. Query-a index use பண்ண allow பண்றதா? Column-a function-la wrap பண்ணா — index bypass ஆகும். Column naked-a இருக்கணும் WHERE-la!

❌ NON-SARGable (Index Not Used!)
-- Function on column = index bypass!
WHERE YEAR(PaidDate) = 2026
-- SQL must call YEAR() on every row!

WHERE UPPER(StudentName) = 'RAMESH'
-- Calls UPPER() on each row!

WHERE Amount + 100 > 5000
-- Math on column = scan!

WHERE SchoolId = '42'  -- int vs varchar
-- Implicit conversion = index ignored!

WHERE Name LIKE '%Ramesh%'
-- Leading % = full scan!
✅ SARGable (Index Seek!)
-- Move function to right side!
WHERE PaidDate >= '2026-01-01'
  AND PaidDate <  '2027-01-01'
-- Column naked — range seek!

WHERE StudentName = 'RAMESH'
-- Case-insensitive collation handles it

WHERE Amount > 4900
-- Move math to value side!

WHERE SchoolId = 42  -- int literal
-- Types match — no conversion!

WHERE Name LIKE 'Ramesh%'
-- Trailing % only = seek possible!
-- ZenCampus real bug: attendance query was slow (Table Scan!)

-- WRONG (Non-SARGable)
SELECT * FROM Attendance
WHERE CAST(AttendanceDate AS DATE) = '2026-06-15'
  AND CONVERT(VARCHAR, SchoolId) = '42'  -- implicit conversion!

-- FIXED (SARGable — Index Seek!)
SELECT StudentId, IsPresent, Remarks
FROM   Attendance
WHERE  AttendanceDate >= '2026-06-15'
  AND  AttendanceDate <  '2026-06-16'
  AND  SchoolId = 42   -- int, no conversion!
-- Result: 8420 logical reads → 14 reads. 600x improvement!
💡 SELECT * — Two Problems

1. Covering index useless — * needs ALL columns → Key Lookup back to clustered index. 2. Network overhead — sending columns you don't need. Always SELECT only needed columns = SARGable + covering index friendly!

4 Stored Procedures — When & Why
🎬 Baasha — Trusted lieutenant, pre-trained, executes perfectly every time. Stored Proc = pre-compiled, cached plan!

Inline SQL = hire new worker every time, train them, then work. SP = trained lieutenant — ready, plan cached, faster execution. ZenCampus fee report SP — first call compiles, next 10,000 calls use cached plan!

-- ZenCampus: Fee collection report stored procedure
CREATE OR ALTER PROCEDURE usp_GetFeeReport
    @SchoolId     INT,
    @AcademicYear NVARCHAR(10),
    @Status       NVARCHAR(20) = NULL  -- optional filter
AS
BEGIN
    SET NOCOUNT ON;  -- suppress "X rows affected" — improves perf!

    SELECT
        s.StudentId,
        s.Name         AS StudentName,
        s.AdmissionNo,
        s.Grade,
        f.FeeType,
        f.Amount,
        f.PaidDate,
        f.Status,
        f.ReceiptNo

    WHERE  f.SchoolId     = @SchoolId
      AND  f.AcademicYear  = @AcademicYear
      AND  (@Status IS NULL OR f.Status = @Status)
    ORDER BY s.Grade, s.Name;
END

5 Transactions — ACID + BEGIN/COMMIT/ROLLBACK
🎬 Vikram — Multi-agent mission: all agents complete OR mission aborts. Partial success = failure!

Fee payment = Vikram mission. Debit student account + Add FeeRecord + Update Balance + Generate Receipt — all 4 must succeed. One fails = ROLLBACK all. Half-paid fee = data corruption!

-- ZenCampus: Fee payment transaction
CREATE OR ALTER PROCEDURE usp_CollectFee
    @StudentId  INT,
    @SchoolId   INT,
    @Amount     DECIMAL(10,2),
    @FeeType    NVARCHAR(50),
    @CollectedBy INT
AS
BEGIN
    SET NOCOUNT ON;

    DECLARE @ReceiptNo  NVARCHAR(20);
    DECLARE @ErrorMsg   NVARCHAR(500);

    BEGIN TRY
        BEGIN TRANSACTION;  -- Start transaction!

        -- 1. Generate receipt number
        SET @ReceiptNo = 'ZEN' + FORMAT(GETDATE(), 'yyyyMMdd')
                       + RIGHT('0000' + CAST(@StudentId AS VARCHAR), 4);

        -- 2. Insert fee record
        INSERT INTO FeeRecords
            (StudentId, SchoolId, FeeType, Amount, PaidDate, Status, ReceiptNo, CollectedBy)
        VALUES
            (@StudentId, @SchoolId, @FeeType, @Amount,
             GETDATE(), 'Paid', @ReceiptNo, @CollectedBy);

        -- 3. Update student balance
        UPDATE StudentFeeBalance
        SET    PaidAmount = PaidAmount + @Amount,
               LastPaidDate = GETDATE()
        WHERE  StudentId = @StudentId AND SchoolId = @SchoolId;

        -- 4. Verify balance was updated (else something wrong!)
        IF @@ROWCOUNT = 0
            THROW 50001, 'Student fee balance record not found', 1;

        -- 5. Log audit
        INSERT INTO AuditLog (TableName, Action, RecordId, UserId, ActionDate)
        VALUES ('FeeRecords', 'INSERT', SCOPE_IDENTITY(), @CollectedBy, GETDATE());

        COMMIT TRANSACTION;  -- All good — commit!

        SELECT @ReceiptNo AS ReceiptNo, 'Success' AS Status;

    END TRY
    BEGIN CATCH
        IF @@TRANCOUNT > 0
            ROLLBACK TRANSACTION;  -- Undo EVERYTHING!

        SET @ErrorMsg = ERROR_MESSAGE();
        INSERT INTO ErrorLog (ErrorMessage, ErrorDate, UserId)
        VALUES (@ErrorMsg, GETDATE(), @CollectedBy);

        SELECT NULL AS ReceiptNo, @ErrorMsg AS Status;
    END CATCH
END
💡 ACID Properties — Remember via ZenCampus Fee:

Atomic — Fee insert + Balance update = one unit. Half success = fail.
Consistent — Before: Balance=5000. After: Balance=6000. Always valid state.
Isolated — Concurrent fee collections don't interfere.
Durable — Committed fee payment survives server crash.

7 Deadlocks — Detect + Prevent
🎬 Kaithi — Dilli trapped — can't move forward, can't go back. Two sessions, each waiting for the other!

Session A holds lock on Students, wants FeeRecords lock. Session B holds lock on FeeRecords, wants Students lock. Deadlock! SQL Server chooses one as "victim" — kills it. App gets error 1205!

-- Deadlock scenario — classic cross-resource lock!

-- Session A (fee collection)
BEGIN TRAN;
UPDATE Students        SET Status = 'Updated' WHERE StudentId = 101; -- locks Students row
UPDATE FeeRecords      SET Amount = 5000       WHERE StudentId = 101; -- WAITS for B!

-- Session B (attendance update)
BEGIN TRAN;
UPDATE FeeRecords      SET Status = 'Paid'    WHERE StudentId = 101; -- locks FeeRecords
UPDATE Students        SET FeeStatus = 'OK'   WHERE StudentId = 101; -- WAITS for A!
-- DEADLOCK! SQL Server kills one → Error 1205

-- FIX 1: Consistent access order — ALWAYS lock tables in same order!
-- Both sessions: Students first, then FeeRecords
BEGIN TRAN;
UPDATE Students   ... ;  -- always first
UPDATE FeeRecords ... ;  -- always second
COMMIT;

-- FIX 2: Keep transactions short — don't hold locks long!

-- FIX 3: Index on join/WHERE columns — reduces lock contention

-- FIX 4: READ_COMMITTED_SNAPSHOT — readers don't block writers!

-- Detect deadlocks — trace flag or Extended Events
DBCC TRACEON (1222, -1); -- log deadlock info to SQL Server error log
// C# — handle deadlock retry (Error 1205)
public async Task CollectFeeWithRetryAsync(FeeDto dto)
{
    const int maxRetries = 3;
    for (int attempt = 1; attempt <= maxRetries; attempt++)
    {
        try
        {
            await CollectFeeAsync(dto);
            return;
        }
        catch (SqlException ex) when (ex.Number == 1205) // Deadlock!
        {
            if (attempt == maxRetries) throw;
            await Task.Delay(attempt * 100); // 100ms, 200ms, 300ms
            _log.LogWarning("Deadlock victim, retry {attempt}", attempt);
        }
    }
}

Index add panna slow aachu — too many indexes! Medium Real Interview
📋 Scenario: "ZenCampus FeeRecords table-la reports fast ஆகணும்னு 15 indexes add பண்ணாங்க. Now INSERT பண்றா 3 seconds ஆகுது! Why? How many indexes is too many?"
✅ INDEX OVERHEAD TRAP

Why slow: Every INSERT/UPDATE/DELETE must update ALL 15 indexes + the clustered index. One row inserted = 16 B-Tree updates. FeeRecords = 200 inserts/second during school hours = massive write amplification!

Rule of thumb: OLTP tables (frequent writes): 5-7 indexes max. Reporting tables (few writes): 10-15 OK. Review unused indexes — they add write cost with zero read benefit.

-- Find unused indexes (high updates, low seeks = useless weight!)
SELECT
    i.name AS IndexName,
    ius.user_seeks    AS Seeks,     -- reads using this index
    ius.user_scans    AS Scans,
    ius.user_updates  AS Updates    -- maintenance cost!
FROM   sys.indexes i
LEFT JOIN sys.dm_db_index_usage_stats ius
    ON i.object_id = ius.object_id AND i.index_id = ius.index_id
WHERE  OBJECT_NAME(i.object_id) = 'FeeRecords'
  AND  i.index_id > 1  -- skip clustered
  AND  ISNULL(ius.user_seeks, 0) + ISNULL(ius.user_scans, 0) = 0
ORDER BY ISNULL(ius.user_updates, 0) DESC;
-- Indexes with Updates > 0 but Seeks/Scans = 0 → DELETE THEM!

-- After cleanup: 15 indexes → 6 covering indexes
-- INSERT time: 3 seconds → 0.4 seconds

Fee payment — server crash during transaction. Data corrupt? 💥 Critical Senior
📋 Scenario: "ZenCampus fee collection-la: FeeRecord INSERT ஆச்சு, StudentFeeBalance UPDATE ஆகுமுன்னே server crash ஆச்சு. Student fee paid, balance update ஆகல. Data corrupt ஆச்சா? SQL Server எப்படி handle பண்றது?"
✅ TRANSACTION LOG + WAL PROTOCOL

No corruption — WAL saves you!

Write-Ahead Logging (WAL): SQL Server writes to Transaction Log BEFORE writing data pages. Log is sequential, fast I/O.

On crash: SQL Server restarts → reads Transaction Log → sees incomplete TX (no COMMIT) → automatically ROLLBACKS it. FeeRecord INSERT undone. Student not charged. Consistent state!

Durability: COMMITTED data → log was flushed to disk before COMMIT returned. Survives crash.

-- Transaction Log flow:
-- 1. BEGIN TRAN  → Log: BEGIN_TRAN
-- 2. INSERT FeeRecords → Log: INSERT undo info (before image)
-- 3. UPDATE Balance    → Log: UPDATE undo info
-- [CRASH HERE — no COMMIT logged]
-- Restart: Log shows no COMMIT → ROLLBACK using before images
-- Database consistent!

-- That's why ACID Durability works:
-- COMMIT doesn't return until log record flushed to disk (fsync)

-- Check transaction log
SELECT operation, context, [Transaction ID], [Begin Time]
FROM   fn_dblog(NULL, NULL)
WHERE  operation IN ('LOP_BEGIN_XACT', 'LOP_COMMIT_XACT', 'LOP_ABORT_XACT')
ORDER BY [Current LSN] DESC;

Stored Procedure vs EF Core — when to use what? Easy Real Interview
📋 Scenario: "Interview la kekkuran: 'ZenCampus-la EF Core use pannuva, SP use pannuva — when to use what? Design philosophy sollu.'"
✅ BALANCED ANSWER

EF Core for: Simple CRUD (Add, Update, Delete students), relationship navigation, business logic queries, testable code, change tracking. EF Core generates optimized SQL for most cases.

Stored Procedures for: Complex multi-table reports (fee summary, term-wise attendance, exam rank), bulk operations (year-end promotion for 50k students), operations requiring fine-tuned query plan control, DBA-managed performance-critical queries.

ZenCampus approach: EF Core for all CRUD + simple business queries. SPs for 5-6 complex reports that process >10k rows. Dapper for intermediate ad-hoc queries where EF generates suboptimal SQL.

📋 Scenario: "March year-end — 50,000 students Grade promote பண்ணணும் (Grade 1 → Grade 2, etc). One UPDATE statement panrom — blocking issue, log file full, DBA panic. How to handle safely?"

-- WRONG: One giant update — locks entire table for minutes!

DECLARE @TotalUpdated  INT  = 0;
DECLARE @RowsAffected  INT  = 1;

WHILE @RowsAffected > 0
BEGIN
    BEGIN TRAN;

    UPDATE TOP (@BatchSize) Students
    SET    Grade        = Grade + 1,
           AcademicYear = '2026-27',
           PromotedDate = GETDATE()
    WHERE  Status = 'Active'
      AND  PromotedDate IS NULL; -- flag: not yet promoted

    SET @RowsAffected = @@ROWCOUNT;
    SET @TotalUpdated  = @TotalUpdated + @RowsAffected;

    COMMIT TRAN;  -- commit each batch = log checkpoint, no log bloat!

    WAITFOR DELAY '00:00:00.1'; -- 100ms pause = other queries breathe!

    PRINT 'Updated ' + CAST(@TotalUpdated AS VARCHAR) + ' students';
END
-- 50k rows: 100 batches × 500 rows × 100ms = 10 seconds total
-- Max lock duration: 500 rows × ~1ms = negligible blocking!

MERGE — Upsert attendance without INSERT + UPDATE separately Medium
📋 Scenario: "Attendance sync API — SMS gateway sends attendance data, some students already have attendance (UPDATE), some new (INSERT). C# la IF EXISTS → UPDATE ELSE INSERT pattern-a use pannuvan. Better approach?"
✅ SQL MERGE (UPSERT) Statement
-- MERGE = UPDATE if exists, INSERT if not. Atomic, one statement!
MERGE INTO Attendance AS Target
USING (
    -- Source: incoming attendance from SMS gateway (TVP)
    SELECT StudentId, AttendDate, IsPresent, SchoolId, Remarks
    FROM   @AttendanceData  -- TVP parameter
) AS Source
ON (Target.StudentId  = Source.StudentId
AND  Target.AttendDate = Source.AttendDate
AND  Target.SchoolId  = Source.SchoolId)

WHEN MATCHED THEN                 -- Row exists → UPDATE
    UPDATE SET
        Target.IsPresent = Source.IsPresent,
        Target.Remarks   = Source.Remarks,
        Target.UpdatedAt = GETDATE()

WHEN NOT MATCHED BY TARGET THEN   -- New row → INSERT
    INSERT (StudentId, AttendDate, IsPresent, SchoolId, Remarks, CreatedAt)
    VALUES (Source.StudentId, Source.AttendDate,
             Source.IsPresent, Source.SchoolId, Source.Remarks, GETDATE())

WHEN NOT MATCHED BY SOURCE        -- Row in DB but not in source → optional
    AND Target.AttendDate = '2026-06-23'
    THEN DELETE;                   -- remove removed students

-- OUTPUT shows what happened
OUTPUT $action AS Action, Inserted.StudentId;
-- Returns: 'INSERT' or 'UPDATE' per row

9 ⚡ Quick Quiz

Q1: WHERE YEAR(PaidDate) = 2026 — இது SARGable-aa?

A) Yes — YEAR() is a simple function, SQL optimizes it
B) No — function on column = SQL evaluates every row = Table Scan!
C) Depends on index type
D) Only for DATETIME columns, not DATE
✅ Non-SARGable! YEAR(PaidDate) = SQL must call YEAR() on each row = can't use index. Fix: WHERE PaidDate >= '2026-01-01' AND PaidDate < '2027-01-01' → Index Seek!
❌ Function on column = index bypass! SQL can't seek into an index when column is wrapped in function. Always rewrite to bare column = value pattern!

Q2: Table-la 20 Non-Clustered Index add பண்றா — INSERT performance?

A) No change — reads improve, writes unaffected
B) Improves — more indexes, faster searches
C) Degrades — each INSERT must update all 20 index B-Trees!
D) Depends on index size
✅ Write amplification! 1 INSERT = 20 index B-Tree updates + 1 clustered index update. OLTP tables: 5-7 indexes max. Find unused indexes (seeks=0, updates>0) and drop them!
❌ Indexes hurt writes! Every INSERT/UPDATE/DELETE must maintain ALL indexes. 20 indexes = 21 B-Tree updates per row. Find and drop unused indexes!

Q3: @@TRANCOUNT = 2 — BEGIN TRAN எத்தனை times call ஆச்சு?

A) 1
B) 2 — nested transactions. COMMIT once = @@TRANCOUNT 2→1. Need 2 COMMITs!
C) @@TRANCOUNT always 0 or 1
D) 0 — outer transaction not started yet
✅ @@TRANCOUNT counts nested BEGIN TRANs. Each BEGIN TRAN = +1. Each COMMIT = -1. Only when @@TRANCOUNT reaches 0 does the actual transaction commit to disk. ROLLBACK always goes to 0 (full rollback)!
❌ @@TRANCOUNT = 2 means 2 nested BEGIN TRANs. Need 2 COMMITs to actually commit. But 1 ROLLBACK kills everything (goes to 0)! Check @@TRANCOUNT > 0 before ROLLBACK in CATCH blocks!

Q4: Composite index (SchoolId, Grade) — WHERE Grade = 5 AND SchoolId = 42 — uses index?

A) Yes — SQL optimizer can use leftmost prefix even if WHERE order differs
B) No — WHERE must match index column order exactly
C) Only if Grade = 5 comes first in WHERE
D) Only clustered index can be used here
✅ SQL optimizer is smart — it rearranges WHERE clause to match index column order. WHERE Grade = 5 AND SchoolId = 42 → optimizer reads as SchoolId = 42 AND Grade = 5 → leftmost prefix (SchoolId) matched → Index Seek!
❌ SQL optimizer reorders WHERE predicates! The critical thing is which COLUMNS are present, not their WHERE clause order. (SchoolId, Grade) index: WHERE Grade=5 AND SchoolId=42 → optimizer uses Index Seek on SchoolId first!
