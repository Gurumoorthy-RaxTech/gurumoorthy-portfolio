Case number kudukkuvan — exact evidence box kittum instantly. Linear search pannala! Dictionary exactly same — key kudukkuva → hash → exact bucket → value. O(1) lookup!

6 Concurrent Collections — Thread-Safe Senior Level!
🎬 Enthiran — Multiple Chittis, same resource

1000 Chittis same file access panna — race condition! Regular Dictionary multi-thread access = corruption. ConcurrentDictionary = built-in thread safety — no manual lock needed!

⚠️ Regular Dictionary — NOT thread safe!

Multiple threads simultaneously read/write → data corruption, exceptions. Never use regular Dictionary in concurrent scenarios without locking!

// ZenCampus — Concurrent student session cache
private readonly ConcurrentDictionary<string, SessionInfo> _sessions = new();

// Thread-safe add or update
_sessions.TryAdd(sessionId, new SessionInfo(userId));

// GetOrAdd — atomic! No race condition
var session = _sessions.GetOrAdd(sessionId,
    key => new SessionInfo(userId)); // factory only called if missing

// AddOrUpdate — atomic update
_sessions.AddOrUpdate(
    sessionId,
    new SessionInfo(userId),                    // add value
    (key, existing) => { existing.Refresh(); return existing; } // update fn
);

// TryRemove
_sessions.TryRemove(sessionId, out var removed);

Multi-thread Cache Corruption Hard
Topic: Dictionary vs ConcurrentDictionary race condition
📋 Scenario: ZenCampus-la school settings in-memory cache — multiple API requests parallel access pannuthu. Occasionally "Index out of range" exception, "KeyNotFoundException" varuthu — reproducible illai! Root cause enna, fix pannu? 
private static Dictionary<int, SchoolSettings> _cache = new();

{

        _cache[schoolId] = _repo.GetSettings(schoolId);

✅ ROOT CAUSE + FIX

Race Condition: Two threads simultaneously check ContainsKey → both see "not found" → both try to add → Dictionary internal state corrupt → random exceptions!

// ✅ Fix 1 — ConcurrentDictionary.GetOrAdd (atomic!)
private static readonly ConcurrentDictionary<int, SchoolSettings>
    _cache = new();

public SchoolSettings GetSettings(int schoolId)
    => _cache.GetOrAdd(schoolId,
        id => _repo.GetSettings(id)); // thread-safe!

// ✅ Fix 2 — lock (when factory should run ONCE only)
private readonly object _lock = new();

public SchoolSettings GetSettings(int schoolId)
{
    if (_cache.TryGetValue(schoolId, out var cached)) return cached;
    lock (_lock)
    {
        if (_cache.TryGetValue(schoolId, out cached)) return cached;
        var settings = _repo.GetSettings(schoolId);
        _cache[schoolId] = settings; // double-check pattern
        return settings;
    }
}
⭐ Note: GetOrAdd factory may run multiple times!

ConcurrentDictionary.GetOrAdd — factory not guaranteed to run once under concurrent load. If factory = expensive DB call, use Lazy<T> pattern or lock. For cheap factories, GetOrAdd is fine.

SMS Queue — Process 5000 Fee Reminders Medium
Topic: Queue + producer-consumer pattern
📋 Scenario: Fee reminder day — 5000 parents-ku SMS pathaikanom. SMS API rate limit: 100/minute. Queue use panni design pannu — producer thread adds, consumer thread processes 100/minute.
✅ ANSWER — BlockingCollection Producer-Consumer
public class SmsReminderService
{
    private readonly BlockingCollection<string> _queue
        = new(new ConcurrentQueue<string>(), 500); // max 500 buffered

    // Producer — add all mobile numbers
    public async Task ProduceAsync(List<string> mobiles)
    {
        foreach (var mobile in mobiles)
            _queue.Add(mobile); // blocks if queue full (backpressure!)
        _queue.CompleteAdding(); // signal done
    }

    // Consumer — process 100/minute
    public async Task ConsumeAsync()
    {
        int count = 0;
        foreach (var mobile in _queue.GetConsumingEnumerable())
        {
            await _sms.SendAsync(mobile, "Fee reminder");
            if (++count % 100 == 0)
                await Task.Delay(TimeSpan.FromMinutes(1)); // rate limit
        }
    }
}

// Run both

    svc.ProduceAsync(mobileNumbers),

Undo Feature — Admission Form Easy
Topic: Stack for undo/redo
📋 Scenario: ZenCampus admission form-la Ctrl+Z (undo) + Ctrl+Y (redo) implement pannanum. Stack use panni design pannu.
✅ ANSWER
public class FormUndoManager<T>
{
    private readonly Stack<T> _undoStack = new();
    private readonly Stack<T> _redoStack = new();

    public void SaveState(T state)
    {
        _undoStack.Push(state);
        _redoStack.Clear(); // new action clears redo
    }

    public T? Undo(T current)
    {
        if (!_undoStack.TryPop(out var prev)) return default;
        _redoStack.Push(current); // save for redo
        return prev;
    }

    public T? Redo(T current)
    {
        if (!_redoStack.TryPop(out var next)) return default;
        _undoStack.Push(current);
        return next;
    }

    public bool CanUndo => _undoStack.Count > 0;
    public bool CanRedo => _redoStack.Count > 0;
}

// Usage
var undo = new FormUndoManager<StudentFormState>();
undo.SaveState(currentState);  // before each change
if (undo.CanUndo) form.Restore(undo.Undo(current));

10
Word Frequency Counter — Dictionary trick Medium
Topic: Dictionary grouping — common coding interview
📋 Scenario: ZenCampus parent feedback form — free text varuthu. Top 5 most mentioned words find pannanum. (Common coding round question!) Efficient-a solve pannu.
✅ ANSWER
public List<(string Word, int Count)>
    GetTopWords(IEnumerable<string> feedbacks, int top = 5)
{
    var freq = new Dictionary<string, int>(StringComparer.OrdinalIgnoreCase);

    foreach (var feedback in feedbacks)
    {
        foreach (var word in feedback.Split(' ', StringSplitOptions.RemoveEmptyEntries))
        {
            // Elegant increment — GetValueOrDefault returns 0 if not found
            freq[word] = freq.GetValueOrDefault(word) + 1;
        }
    }

    return freq
        .OrderByDescending(kv => kv.Value)
        .Take(top)
        .Select(kv => (kv.Key, kv.Value))
        .ToList();
}

// StringComparer.OrdinalIgnoreCase — "Good" and "good" = same word!
⭐ Elegant Pattern — Dictionary increment

freq[word] = freq.GetValueOrDefault(word) + 1 — one liner! No TryGetValue needed. If key missing → 0 + 1 = 1. Existing → old + 1. ZenCampus-la subject feedback analysis-la use panninen!

Q3: List<T> middle-la Insert — time complexity?

A) O(1) — linked list maari
B) O(n) — all elements after shift pannanum
C) O(log n)
D) O(1) amortized
✅ List = internal array. Middle insert → shift all elements right → O(n). End insert = O(1) amortized. LinkedList middle insert = O(1) if you have the node!
❌ O(n)! List internally array — insert position-ku right-la uluaa elements shift aaganum. n elements = n shifts. Add to end = O(1) amortized only!

Q4: ConcurrentDictionary.GetOrAdd() — factory eppoluthe run aagum?

A) Always exactly once
B) Never more than once
C) May run multiple times under concurrent load — not guaranteed once
D) Throws if called twice
✅ GetOrAdd factory = NOT atomic! Under high concurrency, multiple threads may invoke factory simultaneously — only one result stored, others discarded. For expensive factory (DB call) use Lazy<T> or lock!
❌ GetOrAdd factory NOT guaranteed once! Multiple threads may call it — race on factory. One winner stored, others discarded. Cheap factory = fine. Expensive factory (DB) = use lock or Lazy<T>!
