1 DbContext + DbSet — ZenCampus Setup
🎬 Ratsasan — Investigation board with all evidence connected

DbContext = investigation board. DbSet = each evidence category (suspects, clues, weapons). Context saves everything together. Change Tracker = board-la enna change aachu track பண்றான்!

// ZenCampus DbContext
public class ZenCampusContext : DbContext
{
    public ZenCampusContext(DbContextOptions<ZenCampusContext> opts) : base(opts) { }

    public DbSet<Student>     Students     { get; set; }
    public DbSet<FeeRecord>   FeeRecords   { get; set; }
    public DbSet<Attendance>  Attendances  { get; set; }
    public DbSet<School>      Schools      { get; set; }

    protected override void OnModelCreating(ModelBuilder mb)
    {
        // Student → School relationship
        mb.Entity<Student>()
          .HasOne(s => s.School)
          .WithMany(sc => sc.Students)
          .HasForeignKey(s => s.SchoolId)
          .OnDelete(DeleteBehavior.Restrict); // school delete → students block

        // Composite index — speed up school+class queries
        mb.Entity<Student>()
          .HasIndex(s => new { s.SchoolId, s.ClassName });

        // Value conversion — store enum as string in DB
        mb.Entity<Student>()
          .Property(s => s.Status)
          .HasConversion<string>();
    }
}

// Entity models
public class Student
{
    public int      Id        { get; set; }
    public string   Name      { get; set; } = "";
    public string   ClassName { get; set; } = "";
    public int      SchoolId  { get; set; }
    public School   School    { get; set; } = null!; // nav property
    public List<FeeRecord> FeeRecords { get; set; } = [];
    public StudentStatus Status { get; set; }
}

// Program.cs registration
builder.Services.AddDbContext<ZenCampusContext>(opts =>
    opts.UseSqlServer(builder.Configuration.GetConnectionString("ZenCampus"))
        .EnableSensitiveDataLogging() // dev only!
        .LogTo(Console.WriteLine, LogLevel.Information));
💡 DbContext Lifetime — ALWAYS Scoped (per request)

DbContext = Scoped. NEVER Singleton (data races, stale data). NEVER Transient (performance, connection waste). If background service needs it — use IDbContextFactory<T>.

2 🔴 N+1 Problem — Biggest EF Core Interview Topic
🎬 Kaithi — Dilli one trip plan, ஆனா 100 trips ஆச்சு

Oru trip-la all evidence collect பண்ணலாம். ஆனா N+1 = 1 query + ஒவ்வொரு record-ku 1 more query. 1000 students → 1001 queries! API slow ஆகும் reason NUMBER ONE!

❌ N+1 Problem — 1001 SQL queries fired!
SELECT * FROM Students WHERE SchoolId = 1 -- 1 query → 1000 students
SELECT * FROM Schools WHERE Id = 1 -- for student[0]
SELECT * FROM Schools WHERE Id = 1 -- for student[1]
SELECT * FROM Schools WHERE Id = 1 -- for student[2]
... 997 more queries!
SELECT * FROM Schools WHERE Id = 1 -- for student[999]
Total: 1001 round trips to DB = API 8-10 seconds!
❌ N+1 TRIGGER CODE
// Looks innocent — IS NOT!
var students = await _ctx.Students
    .Where(s => s.SchoolId == id)
    .ToListAsync();

// Each .School access = NEW query!
foreach (var s in students)
{
    Console.WriteLine(s.School.Name);
    // ↑ triggers SELECT * FROM Schools!
}
✅ INCLUDE FIX — 1 SQL query (JOIN)
// Include = SQL JOIN = 1 query!
var students = await _ctx.Students
    .Where(s => s.SchoolId == id)
    .Include(s => s.School)
    .ToListAsync();

// .School already loaded — no new queries!
foreach (var s in students)
{
    Console.WriteLine(s.School.Name);
    // ↑ in-memory! 0 extra queries!
}
// ThenInclude — deeper navigation
var students = await _ctx.Students
    .Include(s => s.School)          // JOIN Schools
    .Include(s => s.FeeRecords)      // JOIN FeeRecords
    .ThenInclude(f => f.PaymentMethod) // JOIN PaymentMethods (from FeeRecord)
    .Where(s => s.SchoolId == id)
    .ToListAsync();
// All data in 1 query with multiple JOINs

// BUT — Include can return too much data!
// BEST PRACTICE: Include only when you NEED navigation. Use Select projection otherwise!
var result = await _ctx.Students
    .Where(s => s.SchoolId == id)
    .Select(s => new
    {
        s.Id, s.Name,
        SchoolName = s.School.Name  // EF auto-JOINs, no Include needed!
    })
    .ToListAsync();
// Returns ONLY needed columns — fastest!
⚠️ Lazy Loading — Silent N+1 Generator

Lazy loading (virtual nav properties + UseLazyLoadingProxies()) = automatic N+1. Convenient in dev, disaster in prod. ZenCampus-la lazy loading OFF — always explicit Include/Select.

3 Loading Strategies — Eager vs Explicit vs Lazy

Explicit .Entry().Collection().LoadAsync() Separate query, on-demand Fee records — load only when user opens fee tab Controlled

// Explicit loading — load on demand
var student = await _ctx.Students.FindAsync(studentId);

// User opens fee tab — load now!
await _ctx.Entry(student)
          .Collection(s => s.FeeRecords)
          .LoadAsync();

// User opens school info tab — load reference
await _ctx.Entry(student)
          .Reference(s => s.School)
          .LoadAsync();

// With filter — explicit load only pending fees!
await _ctx.Entry(student)
          .Collection(s => s.FeeRecords)
          .Query()
          .Where(f => !f.IsPaid)
          .LoadAsync();
4 AsNoTracking — Read-only Query Boost
🎬 Vikram — Undercover agent கண்காணிக்கிறான், ஆனா CBI case-la unnecessary surveillance waste

EF Core normally tracks EVERY entity (Change Tracker). Read-only dashboard-ku tracking வேண்டாம் — extra memory, extra CPU, zero benefit. AsNoTracking = tracking OFF = 30-40% faster reads!

❌ WITH TRACKING (Read-only use)
// EF tracks ALL 1000 entities
// Change Tracker snapshots every property
// Memory: ~2x, CPU: extra comparison work
var students = await _ctx.Students
    .Where(s => s.SchoolId == id)
    .ToListAsync();
// Dashboard display — never modifying!
// Tracking = useless overhead here
✅ ASNOTRACKING — Read-only fast!
// No tracking = no snapshot = less memory!
var students = await _ctx.Students
    .AsNoTracking()  // ← just ONE method!
    .Where(s => s.SchoolId == id)
    .ToListAsync();
// 30-40% faster for read-only scenarios
// Use for: GET endpoints, reports, dashboards
// Global AsNoTracking — all queries in context
builder.Services.AddDbContext<ZenCampusContext>(opts =>
{
    opts.UseSqlServer(cs);
    opts.UseQueryTrackingBehavior(QueryTrackingBehavior.NoTracking); // global!
});
// Then use AsTracking() ONLY when you want to update

// UPDATE flow — must track!
var student = await _ctx.Students
    .AsTracking()  // explicit tracking for update
    .FirstOrDefaultAsync(s => s.Id == studentId);
student!.Name = "Updated Name";
await _ctx.SaveChangesAsync(); // Change Tracker detects, generates UPDATE SQL
5 Migrations — DB Schema Version Control
🎬 Vikram Vedha — Every case has version, chain of events

Migration = DB schema-ku Git commit மாதிரி. ஒவ்வொரு change-um tracked, rollback பண்ணலாம். Production-la migration ஒண்ணு fail ஆச்சுன்னா — previous version-ku roll back பண்ணலாம்!

// Migration Commands
dotnet ef migrations add InitialCreate         // create first migration
dotnet ef migrations add AddStudentPhoneColumn  // schema change
dotnet ef database update                        // apply to DB
dotnet ef database update InitialCreate        // rollback to specific!
dotnet ef migrations remove                      // undo last migration (dev only)
dotnet ef migrations list                        // list all migrations

// Script — for production (never run dotnet ef in prod directly!)
dotnet ef migrations script --output schema.sql  // generate SQL script
dotnet ef migrations script Migration1 Migration2 // diff script
// Migration file — auto-generated, can customize
public partial class AddStudentPhoneColumn : Migration
{
    protected override void Up(MigrationBuilder mb)
    {
        mb.AddColumn<string>(
            name:      "PhoneNumber",
            table:     "Students",
            nullable:  true,
            maxLength: 15);

        // Custom: add index in migration
        mb.CreateIndex(
            name:    "IX_Students_PhoneNumber",
            table:   "Students",
            column:  "PhoneNumber");
    }

    protected override void Down(MigrationBuilder mb)
    {
        mb.DropIndex("IX_Students_PhoneNumber", "Students");
        mb.DropColumn("PhoneNumber", "Students"); // rollback!
    }
}

// Auto-migrate on startup (dev/staging only!)
using var scope = app.Services.CreateScope();
var db = scope.ServiceProvider.GetRequiredService<ZenCampusContext>();
await db.Database.MigrateAsync(); // applies pending migrations
⚠️ Production Migration Rules

1. NEVER run dotnet ef database update in production directly
2. Generate SQL script, review with DBA, then apply
3. Additive changes (add column nullable) = safe. Destructive (drop column, rename) = dangerous!
4. Large tables — add column nullable first, backfill data, then add NOT NULL constraint in separate migration

6 ⚡ Query Performance — API Slow Fix Toolkit
🎬 Bigil — Teamwork optimize பண்ணுது, one player slow ஆனா whole team loses

API performance = weakest DB query-a fix பண்று. One slow query = entire request slow. Bigil coach = profiler. Players = queries. Identify slow player → fix → team wins!

1. Select Projection — Don't fetch what you don't need

❌ Fetching 50 columns for 2
var students = await _ctx.Students
    .ToListAsync();
// SELECT * — all 50 columns!
// 1000 students × 50 cols = huge payload
✅ Select only needed
var students = await _ctx.Students
    .Select(s => new StudentListDto
    {
        Id   = s.Id,
        Name = s.Name
    }).ToListAsync();
// SELECT Id, Name — only 2 columns!

2. Pagination — Never return all records

// ❌ WRONG — loading 50,000 students!
var all = await _ctx.Students.ToListAsync();

// ✅ CORRECT — Keyset pagination (fastest!)
public async Task<PagedResult<StudentDto>> GetStudentsAsync(int page, int size = 20)
{
    var total = await _ctx.Students.CountAsync();

    var items = await _ctx.Students
        .AsNoTracking()
        .OrderBy(s => s.Id)   // MUST order for stable pagination!
        .Skip((page - 1) * size)
        .Take(size)
        .Select(s => new StudentDto { Id = s.Id, Name = s.Name })
        .ToListAsync();

    return new PagedResult<StudentDto>
    {
        Items      = items,
        TotalCount = total,
        PageNumber = page,
        PageSize   = size
    };
}

3. Compiled Queries — Reuse query plan

// EF Core compiles LINQ → SQL every call (parsing + translation overhead)
// Compiled query: parse once, reuse!

private static readonly Func<ZenCampusContext, int, Task<Student?>>
    GetStudentById = EF.CompileAsyncQuery(
        (ZenCampusContext ctx, int id) =>
            ctx.Students.FirstOrDefault(s => s.Id == id));

// Usage — zero compilation overhead on repeated calls
var student = await GetStudentById(_ctx, studentId);

4. Raw SQL — when LINQ can't express it

// Stored procedure or complex SQL — use FromSqlRaw
var students = await _ctx.Students
    .FromSqlRaw(@"
        SELECT s.* FROM Students s
        INNER JOIN FeeRecords f ON f.StudentId = s.Id
        WHERE f.DueDate < GETDATE() AND f.IsPaid = 0
        GROUP BY s.Id HAVING COUNT(f.Id) > 2")
    .AsNoTracking()
    .ToListAsync();

// NEVER concatenate params — SQL injection!
// ❌ WRONG:
_ctx.Students.FromSqlRaw($"SELECT * FROM Students WHERE Name = '{name}'");

// ✅ CORRECT — parameterized:
_ctx.Students.FromSqlInterpolated($"SELECT * FROM Students WHERE Name = {name}");
// OR:
_ctx.Students.FromSqlRaw("SELECT * WHERE Name = @p0", name);
7 Bulk Operations — Handle Thousands of Records
🎬 Sarpatta Parambarai — Mass training camp, not one-by-one

1000 boxers ஒவ்வொருத்தரையும் ஒவ்வொரு day train பண்றது slow. Batch training = fast. EF Core SaveChanges per record = slow. AddRange + single SaveChanges = fast. Bulk library = fastest!

❌ Loop Save — 1000 DB round trips!
foreach (var student in importList)
{
    _ctx.Students.Add(student);
    await _ctx.SaveChangesAsync(); // 1000 calls!
}
// 1000 INSERT statements, 1000 round trips!
✅ AddRange — 1 SaveChanges
await _ctx.Students.AddRangeAsync(importList);
await _ctx.SaveChangesAsync(); // 1 call!
// Still 1000 INSERTs but 1 round trip!
// Good for < 500 records
// ExecuteUpdate / ExecuteDelete — EF Core 7+ (zero tracking overhead!)

// ✅ Bulk update — single UPDATE statement, no entity load!
await _ctx.Students
    .Where(s => s.SchoolId == schoolId && s.Status == StudentStatus.Active)
    .ExecuteUpdateAsync(calls =>
        calls.SetProperty(s => s.Status, StudentStatus.Graduated));
// Generates: UPDATE Students SET Status='Graduated' WHERE SchoolId=@p0 AND Status='Active'
// Zero entity loading! FASTEST bulk update.

// ✅ Bulk delete — single DELETE statement!
await _ctx.Attendances
    .Where(a => a.Date < DateTime.Now.AddYears(-3))
    .ExecuteDeleteAsync();
// DELETE FROM Attendances WHERE Date < @p0
// No load, no track, no loop — single SQL!

// Chunk import — for very large files (10,000+ records)
const int chunkSize = 500;
for (int i = 0; i < importList.Count; i += chunkSize)
{
    var chunk = importList.GetRange(i, Math.Min(chunkSize, importList.Count - i));
    await _ctx.Students.AddRangeAsync(chunk);
    await _ctx.SaveChangesAsync();
    _ctx.ChangeTracker.Clear(); // clear after each chunk — prevents memory bloat!
}

Loop la matikichuna — foreach DB call pattern Real Interview Hard

✅ FIX — Single query replaces 5000!
// ✅ Pull all pending fees in ONE query, then group in memory
var pendingByStudent = await _ctx.FeeRecords
    .AsNoTracking()
    .Where(f => !f.IsPaid)                       // all unpaid — 1 query!
    .Select(f => new { f.StudentId, f.Amount, f.DueDate })
    .ToListAsync();                                 // 1 DB round trip

var grouped = pendingByStudent
    .GroupBy(f => f.StudentId)                    // group in memory
    .ToDictionary(g => g.Key, g => g.ToList());

// Now send SMS for students who have pending fees
var tasks = grouped.Select(kvp =>
    _sms.SendReminderAsync(kvp.Key, kvp.Value));

// Parallel SMS — but rate-limit to 50 at a time!
var semaphore = new SemaphoreSlim(50);
await Task.WhenAll(grouped.Select(async kvp =>
{
    await semaphore.WaitAsync();
    try   { await _sms.SendReminderAsync(kvp.Key, kvp.Value); }
    finally { semaphore.Release(); }
}));

Migration fail aachu production la — ena panuvaa? Real Interview Hard
📋 Scenario: "Friday evening production migration apply பண்ணோம் — AddColumn NOT NULL without default. Migration half-way fail ஆச்சு. DB inconsistent state-la இருக்கு. Monday morning client வருவாங்க. என் பண்ணுவ?"
✅ ANSWER — Rollback + Prevention

Immediate — Rollback

SQL Server — wrap migration in transaction. If failed → auto rollback. EF Core migrations use transactions by default for DDL in SQL Server.

Manual rollback: dotnet ef database update PreviousMigrationName → runs Down() method.

If DB in inconsistent state

Restore from backup (always take backup before migration!). Fix manually if no backup. Update __EFMigrationsHistory table to reflect actual state.

Prevention — Safe Pattern

// Safe NOT NULL column addition — 2 migrations!

// Migration 1: Add nullable first

// Safe — all rows have value before NOT NULL applied!

// Always: backup → script → review → apply
dotnet ef migrations script LastGoodMigration CurrentMigration --output delta.sql

📋 Scenario: "Two teachers same time student grade update பண்றாங்க. Teacher 1 save பண்றான். Teacher 2 save பண்றான். Teacher 1 changes lost! Optimistic concurrency explain பண்ணு."

// Add RowVersion (timestamp) to entity
public class Student
{
    public int     Id         { get; set; }
    public string  Grade      { get; set; } = "";

    [Timestamp]  // EF auto-increments on every update
    public byte[] RowVersion { get; set; } = [];
}

// On update — EF includes RowVersion in WHERE!
// SQL: UPDATE Students SET Grade=@g WHERE Id=@id AND RowVersion=@rv
// If RowVersion changed (someone else saved) → 0 rows affected
// → EF throws DbUpdateConcurrencyException!

try
{
    await _ctx.SaveChangesAsync();
}
catch (DbUpdateConcurrencyException ex)
{
    // Conflict! Tell user to refresh and retry
    var entry = ex.Entries[0];
    var dbValues = await entry.GetDatabaseValuesAsync();
    throw new ConflictException("Another user updated this record. Please refresh.");
}
⭐ Pessimistic vs Optimistic

Optimistic = no lock, detect conflict on save (RowVersion). Use when conflicts rare.
Pessimistic = lock row on read (WITH (UPDLOCK)). Use when conflicts frequent. Higher contention.

9 ⚡ Quick Quiz

Q1: N+1 problem — exact cause enna?

A) Missing index in DB
B) Using async/await incorrectly
C) Loading collection of entities then accessing navigation property for each — triggers 1 query per entity
D) DbContext registered as Singleton
✅ N+1 = 1 query to get N entities + N more queries to get navigation property for each. Fix: .Include() (JOIN) or .Select() (projection with auto-JOIN)!
❌ N+1 = 1 main query + N per-record queries when accessing navigation property in loop. Include() = single JOIN query = fix!

Q2: AsNoTracking — when NOT to use it?

A) List endpoints
B) Dashboard queries
C) When you're going to update the entity and call SaveChanges
D) Reports
✅ AsNoTracking = EF won't track changes. If you modify entity and SaveChanges, no UPDATE generated (no tracking = no change detection). Use tracking only for update/delete flows!
❌ AsNoTracking is wrong for updates! Without tracking, EF can't detect property changes → SaveChanges generates no UPDATE SQL → update silently lost!

Q3: ExecuteUpdateAsync — advantage over load + modify + save?

A) Zero entity loading — single UPDATE SQL, no Change Tracker overhead, fastest for bulk updates
B) Better error handling
C) Works with Singleton DbContext
D) Supports transactions better
✅ ExecuteUpdateAsync = no entity load, no Change Tracker, single UPDATE SQL for all matching rows. Bulk update 1000 rows = 1 SQL instead of 1000 loads + 1000 tracks + 1000 updates!
❌ ExecuteUpdateAsync skips loading entirely — generates single bulk UPDATE SQL. No entity hydration, no change tracking. Fastest for bulk updates by far!

Q4: Migration — "dotnet ef database update" production-la run panna safe aa?

A) Yes, always safe
B) Safe if you test in staging first
C) No — generate SQL script, review with DBA, run manually. Never CLI directly on production.
D) Safe for small changes only
✅ Production migration = generate SQL script (dotnet ef migrations script), DBA reviews for data loss / locking risks, run manually with ability to stop. CLI direct = no review, no rollback plan!
❌ Never run dotnet ef database update on production directly! Generate SQL script → review → manual apply. Large table DDL can lock table for minutes, causing downtime!
