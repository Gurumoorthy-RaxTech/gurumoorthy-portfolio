🏛️ The 4 Layers — ZenCampus Structure

🎬
Vikram Vedha (2017) — Layers of Truth
Vikram Vedha-la story layers-la reveal aagum — outer layer nee theriyum, but inner moral truth different. Clean Architecture same: outer layers (Web, Infrastructure) change aagalaam — inner Domain layer never changes! Dependency always points INWARD.
🟣 Domain Layer (Core — never changes)
Entities, Value Objects, Domain Events, Interfaces. NO dependency on any other layer. Pure C# — no EF, no HTTP, no external packages.
🔵 Application Layer
Use Cases (Commands, Queries), Application Services, DTOs, Validators. Depends on Domain only. No EF, no HTTP — only domain interfaces.
🟢 Infrastructure Layer
EF Core implementations, Dapper repos, SMS clients, Email, Azure Blob. Implements Domain interfaces. Depends on Application + Domain.
🟡 Presentation Layer (Web/API)
Controllers, Minimal API endpoints, Program.cs, DI registration. Depends on Application layer (sends commands/queries). Never directly calls Infrastructure.

🎯 Domain Layer — Rich Entities, Not Anemic

// Anemic Domain Model — WRONG (just data bags)
public class Student
{
    public int    Id      { get; set; }
    public string Name    { get; set; } = "";
    public bool   IsActive { get; set; }
}
// Service does all the logic — entity is dumb

// Rich Domain Model — CORRECT
public class Student
{
    public int    Id              { get; private set; }
    public string Name            { get; private set; }
    public bool   IsActive        { get; private set; }
    public int    SchoolId        { get; private set; }
    private readonly List<IDomainEvent> _events = [];

    // Factory method — business rules enforced at creation
    public static Student Create(string name, string admNo, int classId, int schoolId)
    {
        if (string.IsNullOrWhiteSpace(name))
            throw new DomainException("Student name cannot be empty");
        var s = new Student { Name = name, AdmissionNumber = admNo, ClassId = classId, SchoolId = schoolId, IsActive = true };
        s._events.Add(new StudentAdmittedEvent(s.Id, s.Name, s.SchoolId));
        return s;
    }

    // Domain behaviour — logic lives in entity
    public void Deactivate(string reason)
    {
        if (!IsActive) throw new DomainException("Student already inactive");
        IsActive = false;
        _events.Add(new StudentDeactivatedEvent(Id, reason));
    }

    public IReadOnlyList<IDomainEvent> GetDomainEvents() => _events.AsReadOnly();
    public void ClearDomainEvents() => _events.Clear();
}

⚙️ Application Layer — Orchestration

// Application layer depends on Domain interfaces only — no EF, no HTTP
public class AdmitStudentUseCase(
    IStudentRepository    _repo,    // Domain interface
    IUnitOfWork           _uow,     // Domain interface
    ISmsService           _sms,     // Domain interface (impl in Infrastructure)
    ILogger<AdmitStudentUseCase> _log)
{
    public async Task<int> ExecuteAsync(AdmitStudentDto dto, CancellationToken ct)
    {
        if (await _repo.ExistsAsync(dto.SchoolId, dto.AdmissionNumber, ct))
            throw new ConflictException("Admission number already taken");

        var student = Student.Create(dto.Name, dto.AdmissionNumber, dto.ClassId, dto.SchoolId);
        await _repo.AddAsync(student, ct);
        await _uow.CommitAsync(ct);

        // Publish domain events
        foreach (var evt in student.GetDomainEvents())
            await _mediator.Publish(evt, ct);

        return student.Id;
    }
}

🔧 Infrastructure — Implements Domain Contracts

// Infrastructure DI extension — self-contained registration
public static class InfrastructureServiceExtensions
{
    public static IServiceCollection AddInfrastructure(
        this IServiceCollection services, IConfiguration config)
    {
        services.AddDbContext<ZenCampusDbContext>(opts =>
            opts.UseSqlServer(config.GetConnectionString("ZenCampusDb")));

        services.AddScoped<IStudentRepository, EfStudentRepository>();
        services.AddScoped<IFeeRepository, EfFeeRepository>();
        services.AddScoped<IUnitOfWork, EfUnitOfWork>();
        services.AddSingleton<ISmsService, Msg91SmsService>();
        services.AddSingleton<IFileStorage, AzureBlobStorageService>();
        return services;
    }
}

🎯 10 Interview Scenarios

1
What is the Dependency Rule in Clean Architecture?Concept

Dependencies only point inward. Domain layer has zero external dependencies. Application depends on Domain only. Infrastructure implements Domain interfaces — the dependency flows toward Domain via Dependency Inversion. Web/API depends on Application. No outer layer is referenced by an inner layer. If you change the database (Infrastructure), Domain and Application need zero changes.

2
Where should IStudentRepository be defined — Domain or Infrastructure?Architecture

Domain layer. The interface represents what the application needs from persistence — it's a domain concern. EfStudentRepository (the implementation) lives in Infrastructure. This is Dependency Inversion: Infrastructure depends on Domain interface, not the other way around. Swapping from EF Core to Dapper means only Infrastructure changes.

3
What is the difference between Rich Domain Model and Anemic Domain Model?Concept

Anemic: entities are data bags (public get/set, no methods). All business logic in service layer — Student class has no behaviour. Rich: entities encapsulate behaviour. student.Deactivate(reason) — business rules validated, events raised, state changed. All in the entity. Rich model = self-validating, event-emitting, true object orientation. Clean Architecture favours rich domain models.

4
Can a query handler use EF Core DbContext directly, bypassing the repository?Practical

In practice, yes — many teams inject DbContext directly into query handlers for flexibility (AsNoTracking, Select projections, Split queries). The Dependency Rule is slightly bent but justified for reads: query handlers never change state. Strict Clean Architecture would use a read-only repository interface. Pragmatic choice: query handlers use DbContext, command handlers use repositories with UoW.

5
How is Clean Architecture different from N-Tier Architecture?Comparison

N-Tier (classic): UI → Business Logic → Data Access → DB. Dependencies flow downward — UI depends on BL, BL depends on DAL, DAL depends on DB. Clean Architecture: Domain at center, all dependencies point inward via interfaces. Business logic (Domain) has ZERO external dependencies. N-Tier BL typically depends on DAL — tight coupling. Clean Arch BL never knows about EF Core.

6
Where does FluentValidation live in Clean Architecture?Placement

Application layer — validators for Commands/Queries live alongside the commands in Application. Domain validation (business invariants) lives in Domain entities/value objects (throw DomainException). Application validation (input format, required fields) lives in Application validators. Infrastructure has no validators.

7
You need to switch ZenCampus from SQL Server to PostgreSQL. What changes in Clean Architecture?Test of Understanding

Only Infrastructure changes: (1) Change EF Core provider: UseNpgsql instead of UseSqlServer. (2) Update migrations. (3) Any SQL-Server-specific raw SQL. Domain, Application, and Web layers: ZERO changes. This is the proof that Clean Architecture works — the business rules don't care which database is underneath.

8
Explain the difference between Domain Events and Application Events.Architecture

Domain Events: raised by domain entities, represent something that happened in the domain (StudentAdmitted, FeeRecorded). Dispatched in Application layer after commit. Application Events: raised between application use cases (integration events), may cross bounded contexts. Domain Events = synchronous, in-process. Integration Events = async, via message bus (RabbitMQ, Azure Service Bus).

9
Why should entities have private setters in Clean Architecture?Design

Private setters enforce that state changes only happen through domain methods. With public setters: student.IsActive = false anywhere — business rules bypassed. With private setters + student.Deactivate(reason): business rules (check currently active, raise event, log reason) always enforced. EF Core can still set private properties via constructor or reflection.

10
What is a Value Object and how does it differ from an Entity?DDD

Entity: has identity (Id) — two students with same name are different entities. Value Object: no identity — defined by its values alone. Address(Street="Main St", City="Chennai") == Address(Street="Main St", City="Chennai") — same. In ZenCampus: Address, Money, ContactNumber are Value Objects. Immutable (private setters or record). No separate table if OwnsOne. Equality by value, not ID.
