Change Tracker — How EF Knows What Changed

Change Tracker is EF Core's memory. Every entity you load into a DbContext gets tracked — Unchanged, Modified, Added, Deleted states. On SaveChangesAsync() EF generates SQL only for changed entities.

StateMeaningSQL Generated
DetachedNot tracked at allNone
UnchangedLoaded, nothing changedNone
AddedNew entity via Add()INSERT
ModifiedProperty changedUPDATE (only changed columns)
DeletedRemove() calledDELETE
// Change tracker inspection — useful in debugging
var entries = _db.ChangeTracker.Entries<Student>()
    .Where(e => e.State == EntityState.Modified);

foreach (var entry in entries)
{
    foreach (var prop in entry.Properties.Where(p => p.IsModified))
    {
        _log.LogInformation("Student {Id} {Prop}: {Old} -> {New}",
            entry.Entity.Id, prop.Metadata.Name,
            prop.OriginalValue, prop.CurrentValue);
    }
}

⚡ N+1 Problem — ZenCampus's #1 Performance Killer

Production Bug Pattern: "Fee report page 3 minutes load aagudhu" — almost always N+1. 200 schools x 250 students = 50,001 DB queries for ONE page.

The Bug — N+1 in ZenCampus

// WRONG — N+1 problem
// 1 query for students + N queries (one per student) for fees
var students = await _db.Students
    .Where(s => s.SchoolId == schoolId)
    .ToListAsync();  // Query 1: SELECT * FROM Students WHERE SchoolId=@p0

var report = new List<FeeReportRow>();
foreach (var student in students)
{
    // Query 2, 3, 4... N+1: SELECT * FROM FeeRecords WHERE StudentId=@p0
    var fees = await _db.FeeRecords
        .Where(f => f.StudentId == student.Id)
        .ToListAsync();  // 250 separate queries for 250 students!
    report.Add(new { student.Name, TotalFee = fees.Sum(f => f.Amount) });
}

Fix 1 — Include (Eager Loading)

// FIX: Include — 1 JOIN query instead of N+1
var students = await _db.Students
    .Include(s => s.FeeRecords)               // LEFT JOIN FeeRecords
    .ThenInclude(f => f.FeeCategory)          // JOIN FeeCategories
    .Include(s => s.AttendanceRecords)        // multiple Include chains allowed
    .Where(s => s.SchoolId == schoolId && s.IsActive)
    .ToListAsync();
// Generates: SELECT s.*, f.*, fc.*, a.*
//            FROM Students s
//            LEFT JOIN FeeRecords f ON f.StudentId = s.Id
//            LEFT JOIN FeeCategories fc ON fc.Id = f.FeeCategoryId
//            LEFT JOIN AttendanceRecords a ON a.StudentId = s.Id
//            WHERE s.SchoolId = @p0 AND s.IsActive = 1

Fix 2 — AsSplitQuery (Cartesian Explosion Fix)

// PROBLEM with Include: Cartesian Product!
// Student has 50 FeeRecords AND 200 AttendanceRecords
// JOIN produces 50 x 200 = 10,000 rows per student — wasteful!

var students = await _db.Students
    .Include(s => s.FeeRecords)
    .Include(s => s.AttendanceRecords)
    .Where(s => s.SchoolId == schoolId)
    .AsSplitQuery()       // EF Core 5+ — runs SEPARATE queries per Include
    .ToListAsync();
// Generates 3 separate queries — no cartesian explosion:
// Query 1: SELECT * FROM Students WHERE SchoolId=@p0
// Query 2: SELECT * FROM FeeRecords WHERE StudentId IN (1,2,3...)
// Query 3: SELECT * FROM AttendanceRecords WHERE StudentId IN (1,2,3...)
// EF stitches results in memory

// Set globally for all queries in DbContext:
b.UseSqlServer(cs, opts => opts.UseQuerySplittingBehavior(
    QuerySplittingBehavior.SplitQuery));

Fix 3 — Projection (Best Performance)

// BEST: Project to DTO — load only needed columns
var report = await _db.Students
    .Where(s => s.SchoolId == schoolId && s.IsActive)
    .Select(s => new FeeReportRow
    {
        StudentId   = s.Id,
        StudentName = s.Name,
        ClassName   = s.Class.Name,
        TotalFee    = s.FeeRecords.Sum(f => f.Amount),
        PaidAmount  = s.FeeRecords.Where(f => f.IsPaid).Sum(f => f.Amount),
        Balance     = s.FeeRecords.Sum(f => f.Amount)
                    - s.FeeRecords.Where(f => f.IsPaid).Sum(f => f.Amount)
    })
    .OrderBy(r => r.StudentName)
    .ToListAsync();
// EF translates to a SINGLE optimised SELECT with SUM aggregates
// No navigation properties loaded — minimal memory, max speed

🚀 AsNoTracking — Read-Only Query Optimisation

Tracking is expensive — EF Core creates snapshots for change detection. For GET requests jo data modify panna maataanga, tracking waste of memory and CPU.

// SLOW — tracked by default (for GET APIs this is waste)
var students = await _db.Students.Where(s => s.SchoolId == schoolId).ToListAsync();

// FAST — no snapshot, no change tracking overhead
var students = await _db.Students
    .AsNoTracking()         // 20-40% faster on large result sets
    .Where(s => s.SchoolId == schoolId)
    .ToListAsync();

// AsNoTrackingWithIdentityResolution — deduplicates related entities in memory
// Use when same entity appears in multiple Include paths
var students = await _db.Students
    .Include(s => s.School)
    .AsNoTrackingWithIdentityResolution()
    .Where(s => s.SchoolId == schoolId)
    .ToListAsync();

// Set globally for QueryTrackingBehavior on read-only repositories
public class ReadOnlyStudentRepository
{
    private readonly IQueryable<Student> _students;

    public ReadOnlyStudentRepository(ZenCampusDbContext db)
    {
        _students = db.Students.AsNoTracking();  // all queries are no-tracking
    }
}
Rule: GET endpoints → always AsNoTracking(). POST/PUT/DELETE → use tracked queries (default). Never call Update() on a no-tracking entity without re-attaching.

🔒 Global Query Filters — Multi-Tenant + Soft Delete

ZenCampus-la 200 schools. School A data, School B-ku show aagakudathu. Manual WHERE clause everywhere = bug risk. Global Query Filter = once configure, every query-la automatically apply.

Multi-Tenant + Soft Delete Filter

// ICurrentSchoolContext — resolves schoolId from JWT
public interface ICurrentSchoolContext
{
    int SchoolId { get; }
}

public class JwtSchoolContext(IHttpContextAccessor _http) : ICurrentSchoolContext
{
    public int SchoolId =>
        int.Parse(_http.HttpContext!.User.FindFirst("schoolId")!.Value);
}

// DbContext with injected tenant context
public class ZenCampusDbContext(DbContextOptions<ZenCampusDbContext> opts,
                              ICurrentSchoolContext _school) : DbContext(opts)
{
    protected override void OnModelCreating(ModelBuilder mb)
    {
        // Applies to ALL queries on Students — zero chance of data leak
        mb.Entity<Student>().HasQueryFilter(s =>
            !s.IsDeleted && s.SchoolId == _school.SchoolId);

        mb.Entity<FeeRecord>().HasQueryFilter(f =>
            !f.IsDeleted && f.Student.SchoolId == _school.SchoolId);

        mb.Entity<AttendanceRecord>().HasQueryFilter(a =>
            a.Student.SchoolId == _school.SchoolId);
    }
}

// Usage — filter applied automatically, no manual WHERE needed
var students = await _db.Students.ToListAsync();
// SQL: SELECT * FROM Students WHERE IsDeleted=0 AND SchoolId=@currentSchoolId

// Bypass filter when needed (admin-only endpoints)
var allStudents = await _db.Students
    .IgnoreQueryFilters()   // Super admin: see all schools' data
    .Where(s => s.Name.Contains(searchTerm))
    .ToListAsync();
Security Risk: IgnoreQueryFilters() bypasses BOTH soft-delete AND tenant isolation. Only use in admin-only services. Never expose via student/teacher API endpoints.

Soft Delete Pattern

public interface ISoftDeletable
{
    bool IsDeleted { get; set; }
    DateTime? DeletedAt { get; set; }
    string? DeletedBy { get; set; }
}

// Override SaveChanges to intercept Delete and convert to soft-delete
public override Task<int> SaveChangesAsync(CancellationToken ct = default)
{
    foreach (var entry in ChangeTracker.Entries<ISoftDeletable>()
        .Where(e => e.State == EntityState.Deleted))
    {
        entry.State          = EntityState.Modified;
        entry.Entity.IsDeleted = true;
        entry.Entity.DeletedAt = DateTime.UtcNow;
        entry.Entity.DeletedBy = _currentUser.UserId;
    }
    return base.SaveChangesAsync(ct);
}

FromSqlRaw — Entity Queries

// Complex report query LINQ-la express panna mudiyathu
var schoolId = 42;
var month    = 6;

var summary = await _db.Students
    .FromSqlInterpolated($@"
        SELECT s.Id, s.Name, s.AdmissionNumber,
               COUNT(a.Id) as TotalDays,
               SUM(CASE WHEN a.IsPresent=1 THEN 1 ELSE 0 END) as PresentDays,
               SUM(f.Amount) as TotalFee,
               SUM(CASE WHEN f.IsPaid=1 THEN f.Amount ELSE 0 END) as PaidFee
        FROM Students s
        LEFT JOIN AttendanceRecords a
               ON a.StudentId = s.Id AND MONTH(a.Date) = {month}
        LEFT JOIN FeeRecords f ON f.StudentId = s.Id
        WHERE s.SchoolId = {schoolId} AND s.IsDeleted = 0
        GROUP BY s.Id, s.Name, s.AdmissionNumber")  // Safe: interpolated = parameterized
    .AsNoTracking()
    .ToListAsync();
NEVER use FromSqlRaw with string concatenation: 
// DANGER: SQL Injection!
var query = $"SELECT * FROM Students WHERE Name = '{name}'";
_db.Students.FromSqlRaw(query); // name = "'; DROP TABLE Students; --"

// SAFE: Always use FromSqlInterpolated OR SqlParameter
_db.Students.FromSqlRaw(
    "SELECT * FROM Students WHERE Name = {0}", name);  // SqlParameter under the hood

// Dapper alongside EF Core — best of both worlds
public class FeeReportRepository(ZenCampusDbContext _db)
{
    public async Task<IEnumerable<FeeReportRow>> GetTermReportAsync(int schoolId, int termId)
    {
        // Get the raw SqlConnection from EF's DbContext
        var conn = _db.Database.GetDbConnection();

        return await conn.QueryAsync<FeeReportRow>(
            @"SELECT s.Name, s.AdmissionNumber, c.Name AS ClassName,
                     SUM(f.Amount) AS TotalFee,
                     SUM(CASE WHEN f.IsPaid=1 THEN f.Amount ELSE 0 END) AS Paid
              FROM Students s
              JOIN Classes c ON c.Id = s.ClassId
              JOIN FeeRecords f ON f.StudentId = s.Id AND f.TermId = @TermId
              WHERE s.SchoolId = @SchoolId AND s.IsDeleted = 0
              GROUP BY s.Name, s.AdmissionNumber, c.Name
              ORDER BY c.Name, s.Name",
            new { SchoolId = schoolId, TermId = termId });
    }
}

public partial class AddDefaultFeeCategories : Migration
{
    protected override void Up(MigrationBuilder migrationBuilder)
    {
        migrationBuilder.CreateTable(
            name: "FeeCategories",
            columns: table => new
            {

                                   .Annotation("SqlServer:Identity", "1, 1"),
                Name        = table.Column<string>(maxLength: 100),
                IsActive    = table.Column<bool>(defaultValue: true)
            },
            constraints: t => t.PrimaryKey("PK_FeeCategories", x => x.Id));

        // Seed default data in the same migration
        migrationBuilder.InsertData(
            table: "FeeCategories",
            columns: new[] { "Name", "IsActive" },
            values: new object[,]
            {
                { "Tuition Fee", true },
                { "Transport Fee", true },
                { "Hostel Fee", true },
                { "Exam Fee", true }
            });
    }

    protected override void Down(MigrationBuilder migrationBuilder)
        => migrationBuilder.DropTable(name: "FeeCategories");
}
Migration Best Practices for ZenCampus Production:
  • Always generate --idempotent scripts for deployment — safe to re-run
  • Never modify an already-applied migration — create a new one
  • Destructive changes (DROP column) — add new column first, migrate data, then drop old in separate migration
  • Backup before every prod migration — 50k student records, no jokes
  • Run migrations at startup via db.Database.MigrateAsync() only in dev/staging — not production

📦 Owned Types — Value Objects in EF Core

Owned Types = DDD Value Objects EF Core-la map panrom. Separate table create panna theva illa — parent table-la columns-a store pannum.

// Value Object — Address is not an entity (no identity of its own)
public record Address(
    string Street,
    string City,
    string State,
    string PinCode,
    string District);

public class Student
{
    public int        Id              { get; set; }
    public string     Name            { get; set; } = default!;
    public Address    PermanentAddress { get; set; } = default!;
    public Address?   CurrentAddress   { get; set; }  // optional
}

// EF Configuration — Owned Types
public class StudentConfiguration : IEntityTypeConfiguration<Student>
{
    public void Configure(EntityTypeBuilder<Student> b)
    {
        b.OwnsOne(s => s.PermanentAddress, addr =>
        {
            addr.Property(a => a.Street).HasColumnName("PermanentStreet").HasMaxLength(300);
            addr.Property(a => a.City).HasColumnName("PermanentCity").HasMaxLength(100);
            addr.Property(a => a.PinCode).HasColumnName("PermanentPinCode").HasMaxLength(6);
            addr.Property(a => a.District).HasColumnName("PermanentDistrict").HasMaxLength(100);
        });

        b.OwnsOne(s => s.CurrentAddress, addr =>
        {
            addr.Property(a => a.Street).HasColumnName("CurrentStreet").IsRequired(false);
            addr.Property(a => a.City).HasColumnName("CurrentCity").IsRequired(false);
        });
    }
}
// Result: Students table has columns:
// PermanentStreet, PermanentCity, PermanentPinCode, PermanentDistrict,
// CurrentStreet, CurrentCity (nullable)
// No separate Address table!

⚔️ Concurrency Tokens — Optimistic Concurrency

ZenCampus-la fee payment: two staff simultaneously update same student record — last write wins = data loss. Concurrency token = detect conflict BEFORE saving.

public class FeeRecord
{
    public int      Id        { get; set; }
    public decimal Amount    { get; set; }
    public bool    IsPaid    { get; set; }

    // Option 1: RowVersion (SQL Server timestamp — auto-updated on every UPDATE)
    [Timestamp]
    public byte[] RowVersion { get; set; } = default!;

    // Option 2: Manual concurrency token
    [ConcurrencyCheck]
    public DateTime UpdatedAt { get; set; }
}
// Handling concurrency conflict in FeePaymentService
public async Task<Result> MarkFeeAsPaidAsync(int feeId, byte[] rowVersion, CancellationToken ct)
{
    var maxRetries = 3;
    for (var attempt = 0; attempt < maxRetries; attempt++)
    {
        try
        {
            var fee = await _db.FeeRecords.FindAsync(new object[] { feeId }, ct)
                      ?? throw new NotFoundException($"FeeRecord {feeId} not found");

            // Attach the client's rowVersion so EF includes it in WHERE clause
            _db.Entry(fee).Property(f => f.RowVersion).OriginalValue = rowVersion;

            fee.IsPaid   = true;
            fee.PaidAt   = DateTime.UtcNow;
            fee.PaidBy   = _currentUser.UserId;

            await _db.SaveChangesAsync(ct);
            // EF generates: UPDATE FeeRecords SET IsPaid=1, PaidAt=@p0
            //               WHERE Id=@p1 AND RowVersion=@p2  ← concurrency check!
            return Result.Success();
        }
        catch (DbUpdateConcurrencyException ex)
        {
            if (attempt == maxRetries - 1) throw;

            // Reload fresh data and retry
            var entry = ex.Entries.Single();
            await entry.ReloadAsync(ct);
            rowVersion = ((FeeRecord)entry.Entity).RowVersion;
        }
    }
    return Result.Failure("Concurrent modification — please retry");
}

🚀 Compiled Queries & Performance Tips

EF.CompileAsyncQuery — Hot Path Optimisation

// Normal LINQ: EF translates to SQL EVERY call (parse tree, validate, generate SQL)
// Compiled: translate ONCE, reuse the plan — perfect for hot paths

public static class CompiledStudentQueries
{
    // Compiled once at app startup — stored as static field
    public static readonly Func<ZenCampusDbContext, int, string, IAsyncEnumerable<Student>>
        GetBySchoolAndClass = EF.CompileAsyncQuery(
            (ZenCampusDbContext db, int schoolId, string className) =>
                db.Students
                  .Include(s => s.Class)
                  .Where(s => s.SchoolId == schoolId && s.Class.Name == className)
                  .OrderBy(s => s.Name)
                  .AsNoTracking());

    // Usage (dashboard loads this 1000x per second)
    public static async Task<List<Student>> GetStudentsAsync(
        ZenCampusDbContext db, int schoolId, string className)
    {
        var result = new List<Student>();
        await foreach (var s in GetBySchoolAndClass(db, schoolId, className))
            result.Add(s);
        return result;
    }
}

🎯 10 Scenario-Based Interview Questions

1
ZenCampus fee report page takes 3 minutes to load for a 250-student school. What's the root cause and how do you fix it? Production Bug

Root cause: Classic N+1 problem. LINQ query loads all students, then loops and queries FeeRecords per student — 251 DB round trips.

Fix 1: .Include(s => s.FeeRecords) to eager-load — single JOIN query.

Fix 2 (better): Project to DTO via .Select(s => new FeeReportRow { Total = s.FeeRecords.Sum(...) }) — DB does aggregation, network transfer minimal.

Fix 3: If multiple large collections, add .AsSplitQuery() to prevent cartesian explosion.

Verify: Enable EF Core logging in dev — log all SQL + query count. Any query showing "N" identical queries with different IDs = N+1.

2
What is DbUpdateConcurrencyException and when does it occur in ZenCampus? Concept

Thrown when EF's UPDATE WHERE clause matches zero rows — meaning the record was modified by another user/process between your load and save (optimistic concurrency conflict).

ZenCampus scenario: Two fee clerks open the same student's fee record. Clerk A marks it paid and saves. Clerk B also marks it paid and saves — EF generates UPDATE FeeRecords SET ... WHERE Id=@id AND RowVersion=@oldVersion. The RowVersion now doesn't match → 0 rows affected → DbUpdateConcurrencyException.

Fix: Catch the exception, call entry.ReloadAsync() to get fresh data, re-apply business logic, retry save. Add [Timestamp] to the entity for auto-managed RowVersion.

3
Why should you never run dotnet ef database update in production? What should you do instead? Best Practice

dotnet ef database update runs migrations dynamically — no review, no rollback safety, no record of what SQL ran. For ZenCampus with 50k student records, a bad migration = production data loss.

Correct approach:

  • Generate idempotent SQL script: dotnet ef migrations script --idempotent --output deploy.sql
  • DBA reviews the SQL
  • Backup database before execution
  • Execute script in a transaction (so rollback on failure)
  • Verify in staging first with production data copy
4
What is Global Query Filter and how does it help with multi-tenancy in ZenCampus? Architecture

Global Query Filter adds a WHERE clause to every LINQ query on that entity — automatically, without the developer having to remember it.

ZenCampus use: HasQueryFilter(s => !s.IsDeleted && s.SchoolId == _currentSchoolContext.SchoolId)

Benefits: (1) Zero chance of cross-school data leak, (2) Soft-delete works transparently, (3) New developers can't accidentally forget the WHERE clause.

Trade-off: Must use IgnoreQueryFilters() for super-admin operations — this bypasses BOTH filters simultaneously, so guard those endpoints with [Authorize(Roles = "SuperAdmin")].

5
When would you choose AsSplitQuery over Include in ZenCampus? Performance

Use AsSplitQuery when: Loading a student with both FeeRecords (100 rows) AND AttendanceRecords (200 rows). A JOIN produces 100x200 = 20,000 rows per student — Cartesian explosion. Split Query runs 3 separate queries and joins in memory = much less data transferred.

Stick with Include when: Only one navigation property with small count, OR when you need atomic consistency (Split Query reads at different moments — possible slight inconsistency under high write load).

Rule of thumb: If result row count > 1000 due to JOIN multiplication → consider AsSplitQuery.

6
A developer used FromSqlRaw with string interpolation for a student search. What is the risk and how do you fix it? Security

Risk: SQL Injection. FromSqlRaw($"SELECT * FROM Students WHERE Name = '{searchTerm}'") — if searchTerm = '; DROP TABLE Students; --, it executes as two SQL statements.

Fix: Use FromSqlInterpolated (not Raw) — it parameterizes all {variable} interpolations:

_db.Students.FromSqlInterpolated($"SELECT * FROM Students WHERE Name = {searchTerm}")

Or use FromSqlRaw with explicit SqlParameter: FromSqlRaw("SELECT * FROM Students WHERE Name = {0}", searchTerm)

7
What is an Owned Type in EF Core? Give a ZenCampus example. How is it different from a normal entity? Concept

Owned Type = DDD Value Object mapped by EF Core. It has no identity of its own — its identity is the owning entity's identity. Stored as columns in the owner's table (OwnsOne) or a separate table (OwnsMany).

ZenCampus example: Address record owned by Student. Address has no meaning outside a Student. Using OwnsOne, the Student table gets PermanentStreet, PermanentCity, PermanentPinCode columns — no join needed to load address.

vs normal entity: Normal entity has its own primary key, can be referenced independently. Owned type cannot be queried or referenced by other entities — only via its owner.

8
You need to mark 15,000 Class 12 students as graduated across 200 schools. What is the most efficient EF Core approach? Performance Design

Wrong approach: Load all 15,000 students, set property, SaveChanges — EF generates 15,000 individual UPDATEs, loads all entities into memory.

Correct approach: EF Core 7+ ExecuteUpdateAsync:

await _db.Students
    .Where(s => s.Class.Name == "Class 12" && !s.HasGraduated)
    .ExecuteUpdateAsync(s => s
        .SetProperty(x => x.HasGraduated, true)
        .SetProperty(x => x.GraduatedAt, DateTime.UtcNow));

Generates single UPDATE ... WHERE ... — no entities loaded, no change tracking, minimal memory. Run in batch if needed (e.g., school by school) with Task.Delay between batches to avoid locking.

9
When should you use Dapper alongside EF Core in ZenCampus? Architecture Decision

EF Core is ideal for CRUD and business logic where change tracking helps. Dapper shines for:

  • Complex reports — multi-join aggregation LINQ can't express cleanly
  • Dynamic column queries — report builder with user-selected columns
  • Stored procedures with multiple result sets
  • Raw performance-critical reads — Dapper is 2-5x faster than EF for pure reads

Use EF's _db.Database.GetDbConnection() to share the same connection (and transaction) between EF and Dapper in one request — no second connection overhead.
