Why Background Jobs?

⚡ HTTP Request-Response vs Background Processing

Fee collect panna click pannum po — student-ku SMS send pannanum, receipt PDF generate pannanum, fee balance update pannanum. All these in the HTTP request? That's 3+ seconds. User thinks it's frozen!

❌ Doing everything in HTTP request

2. Send SMS via Msg91 (800ms)
3. Generate PDF receipt (500ms)
4. Update analytics (200ms)
Total: ~1.5s+ timeout risk
✅ Background job approach
1. Save to DB (50ms)
2. Enqueue job → return 201 (60ms)

— Background worker generates PDF

IHostedService / BackgroundService

🔄 Built-in ASP.NET Core — No External Packages

ASP.NET Core built-in-a irukku — NuGet package வேண்டாம். Long-running background work or scheduled periodic tasks-ku use pannrom.

// Option 1: IHostedService — manual start/stop
public class FeeReminderHostedService : IHostedService, IDisposable
{
    private Timer? _timer;

    public Task StartAsync(CancellationToken ct)
    {
        // Run immediately, then every 24 hours
        _timer = new Timer(SendFeeReminders, null,
            TimeSpan.Zero, TimeSpan.FromHours(24));
        return Task.CompletedTask;
    }

    private async void SendFeeReminders(object? state)
    {
        // ⚠️ Can't inject scoped services directly — must create scope!
        using var scope = _sp.CreateScope();
        var repo = scope.ServiceProvider.GetRequiredService<IStudentRepository>();
        var sms  = scope.ServiceProvider.GetRequiredService<ISmsService>();

        var overdue = await repo.GetOverdueFeeStudentsAsync();
        foreach (var student in overdue)
            await sms.SendReminderAsync(student.ParentPhone, student.FeeDue);
    }

    public Task StopAsync(CancellationToken ct)
    {
        _timer?.Change(Timeout.Infinite, 0);
        return Task.CompletedTask;
    }

    public void Dispose() => _timer?.Dispose();
}
// Option 2: BackgroundService — cleaner, abstract class with ExecuteAsync
public class AttendanceSyncService : BackgroundService
{
    private readonly IServiceProvider _sp;
    private readonly ILogger<AttendanceSyncService> _log;

    public AttendanceSyncService(IServiceProvider sp, ILogger<AttendanceSyncService> log)
        => (_sp, _log) = (sp, log);

    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        _log.LogInformation("Attendance sync service started");

        while (!stoppingToken.IsCancellationRequested)
        {
            try
            {
                await SyncAttendanceAsync(stoppingToken);
            }
            catch (Exception ex) when (ex is not OperationCanceledException)
            {
                _log.LogError(ex, "Attendance sync failed, will retry in 5 minutes");
            }

            // Wait until midnight or 5 min on error
            var delay = TimeUntilMidnight();
            await Task.Delay(delay, stoppingToken);
        }
    }

    private async Task SyncAttendanceAsync(CancellationToken ct)
    {
        using var scope = _sp.CreateScope();
        var service = scope.ServiceProvider.GetRequiredService<IAttendanceService>();
        await service.SyncAllSchoolsAsync(ct);
    }
}

// Program.cs
builder.Services.AddHostedService<AttendanceSyncService>();
builder.Services.AddHostedService<FeeReminderHostedService>();
Critical: IHostedService is singleton — you cannot inject scoped services (DbContext, repositories) directly! Always use IServiceProvider.CreateScope() inside the job method. Forgetting this causes "Cannot consume scoped service from singleton" exception at startup.

Hangfire — Persistent Background Jobs

🔧 Fire-and-Forget, Scheduled, Recurring — Persisted in SQL

IHostedService-la job fail aana — server restart aana — job gone! Hangfire persists jobs to SQL Server. Auto-retry 10 times. Dashboard-la all job status kaanalam.

// NuGet: Hangfire, Hangfire.SqlServer, Hangfire.AspNetCore
// Program.cs
builder.Services.AddHangfire(config => config
    .SetDataCompatibilityLevel(CompatibilityLevel.Version_180)
    .UseSimpleAssemblyNameTypeSerializer()
    .UseRecommendedSerializerSettings()
    .UseSqlServerStorage(builder.Configuration.GetConnectionString("HangfireDb")));

builder.Services.AddHangfireServer(opts =>
{
    opts.WorkerCount = 5;             // 5 parallel workers
    opts.Queues = ["critical", "default", "low"]; // Priority queues
});

// Dashboard — Dev only! Exposes job details
if (app.Environment.IsDevelopment())
    app.UseHangfireDashboard("/hangfire");
// 4 types of Hangfire jobs

// 1. Fire-and-forget — enqueue, run once ASAP
var jobId = BackgroundJob.Enqueue(
    () => _smsService.SendWelcomeSmsAsync(student.ParentPhone, student.Name));
// Called in AdmitStudentCommandHandler → HTTP request returns immediately

// 2. Delayed — run after X time
BackgroundJob.Schedule(
    () => _emailService.SendFeeReminderAsync(student.Id),
    TimeSpan.FromDays(7));  // 7 days after admission

// 3. Recurring — cron schedule
RecurringJob.AddOrUpdate(
    recurringJobId: "midnight-fee-reminder",
    methodCall:     () => _feeReminderService.SendAllRemindersAsync(),
    cronExpression: "0 0 * * *");  // Every midnight (cron)

// 4. Continuation — run B only if A succeeds
var admitJobId = BackgroundJob.Enqueue(
    () => _admissionService.CreateFeeStructureAsync(student.Id));
BackgroundJob.ContinueJobWith(admitJobId,
    () => _emailService.SendWelcomePackAsync(student.Id));
// Welcome pack email only if fee structure was created successfully
// Priority queues — [Queue] attribute
public class SmsJobService
{
    [Queue("critical")]  // Processed first
    public async Task SendPaymentReceiptSmsAsync(int receiptId) { ... }

    [Queue("default")]
    public async Task SendWelcomeSmsAsync(string phone, string name) { ... }

    [Queue("low")]      // Processed last
    public async Task SendMonthlyNewsletterAsync(int schoolId) { ... }
}

// Retry policy — default 10 retries with exponential backoff
[AutomaticRetry(Attempts = 5, OnAttemptsExceeded = AttemptsExceededAction.Fail)]
public async Task SendSmsAsync(string phone, string message)
{
    await _msg91Client.SendAsync(phone, message);
    // If Msg91 is down: retry at 1m, 10m, 30m, 1h, 2h intervals
}
Production setup: Use separate connection string for Hangfire DB — "HangfireDb". This prevents Hangfire's job tables from polluting your main ZenCampus DB. On Azure: use Redis as Hangfire storage (Hangfire.Pro.Redis) for better performance with 200 schools enqueuing simultaneously.

Azure Service Bus

☁️ Cross-Service Async Messaging — Fan-Out, Dead-Letter Queue

Fee payment aana po — SMS service, balance update service, analytics service, parent app notification — all must know. Service Bus = one message, multiple consumers. Decoupled! Service down-a irundha — message waits in queue, not lost.

Fee Payment API ──► publishes ──► Service Bus Topic (fee-payment-events) │ ┌──────────────┴──────────────┐──────────────────┐ ▼ ▼ ▼ SMS Subscription Analytics Subscription ParentApp Subscription (send receipt SMS) (update fee analytics) (push notification) 
// NuGet: Azure.Messaging.ServiceBus
// Program.cs
builder.Services.AddSingleton(new ServiceBusClient(
    builder.Configuration["ServiceBus:ConnectionString"]!));

builder.Services.AddSingleton<IEventPublisher, ServiceBusEventPublisher>();
builder.Services.AddHostedService<FeePaymentEventConsumer>();
// Publisher — sends message to Topic
public class ServiceBusEventPublisher(ServiceBusClient _client) : IEventPublisher
{
    public async Task PublishAsync<T>(T @event, string topicName, CancellationToken ct)
    {
        await using var sender = _client.CreateSender(topicName);

        var json    = JsonSerializer.Serialize(@event);
        var message = new ServiceBusMessage(json)
        {
            MessageId      = Guid.NewGuid().ToString(), // Dedup key
            ContentType    = "application/json",
            Subject        = typeof(T).Name,
            SessionId      = @event is ISchoolEvent se ? se.SchoolId.ToString() : null
        };
        await sender.SendMessageAsync(message, ct);
    }
}

// Usage in CollectFeeCommandHandler
await _publisher.PublishAsync(
    new FeePaymentCollectedEvent(receipt.ReceiptNo, student.ParentPhone, receipt.Amount),
    topicName: "fee-payment-events",
    ct);
// Consumer — BackgroundService processing messages
public class FeePaymentEventConsumer(ServiceBusClient _client, IServiceProvider _sp)
    : BackgroundService
{
    protected override async Task ExecuteAsync(CancellationToken ct)
    {
        await using var processor = _client.CreateProcessor(
            topicName:        "fee-payment-events",
            subscriptionName: "sms-subscription",
            new ServiceBusProcessorOptions
            {
                MaxConcurrentCalls = 5,
                AutoCompleteMessages = false  // Manual ack — we control when to complete
            });

        processor.ProcessMessageAsync += HandleMessageAsync;
        processor.ProcessErrorAsync   += HandleErrorAsync;

        await processor.StartProcessingAsync(ct);
        await Task.Delay(Timeout.Infinite, ct); // Keep alive
        await processor.StopProcessingAsync(ct);
    }

    private async Task HandleMessageAsync(ProcessMessageEventArgs args)
    {
        var @event = JsonSerializer.Deserialize<FeePaymentCollectedEvent>(
            args.Message.Body.ToString())!;

        using var scope = _sp.CreateScope();
        var sms = scope.ServiceProvider.GetRequiredService<ISmsService>();

        await sms.SendReceiptAsync(@event.ParentPhone, @event.ReceiptNo, @event.Amount);

        await args.CompleteMessageAsync(args.Message); // ✅ Remove from queue
        // If exception thrown → message goes back to queue → retry up to MaxDeliveryCount
        // After MaxDeliveryCount → Dead Letter Queue (DLQ) for manual inspection
    }
}
Dead Letter Queue (DLQ): After 10 failed deliveries (configurable), message moves to DLQ. Our ops team monitors DLQ — manually inspect why SMS failed (number invalid? Msg91 down?). Fix issue, re-process from DLQ. Never silently drop failed messages!

Outbox Pattern — Guaranteed Message Delivery

📬 The Problem: DB Saved but Event Not Published

Student admit panna — DB-la save achu, but Service Bus publish panna before server crash! Student DB-la irukku but fee record create aagala, SMS pogala. Data inconsistency! Outbox pattern solves this.

❌ The race condition without Outbox: 
await _db.SaveChangesAsync();      // ← Student saved ✅
await _bus.PublishAsync(event);     // ← CRASH HERE! Event lost ❌
// Student exists in DB, but FeeCreatedEvent never reached Service Bus
// Fee record never created → student has no fee structure → billing broken
✅ Outbox Pattern — same DB transaction:
Save domain event to OutboxMessages table in SAME transaction as domain changes. Separate background worker reads outbox table and publishes to Service Bus. If publish fails — retry from outbox. Atomic by design.
// Outbox message table
public class OutboxMessage
{
    public Guid     Id          { get; set; } = Guid.NewGuid();
    public string   EventType   { get; set; } = string.Empty; // "StudentAdmittedEvent"
    public string   Payload     { get; set; } = string.Empty; // JSON
    public DateTime OccurredAt  { get; set; } = DateTime.UtcNow;
    public DateTime? ProcessedAt { get; set; }  // null = not yet published
    public string?  Error       { get; set; }   // last failure reason
}

// EF Core entity + migration for outbox table
modelBuilder.Entity<OutboxMessage>(b =>
{
    b.HasIndex(x => x.ProcessedAt);  // Query unprocessed fast
    b.HasIndex(x => x.OccurredAt);
});
// Command handler — save student + outbox in ONE transaction
public async Task<AdmitStudentResult> Handle(AdmitStudentCommand cmd, CancellationToken ct)
{
    var student = new Student { Name = cmd.Name, SchoolId = cmd.SchoolId };
    _db.Students.Add(student);

    // Save domain event to Outbox — SAME transaction!
    var outboxMsg = new OutboxMessage
    {
        EventType = nameof(StudentAdmittedEvent),
        Payload   = JsonSerializer.Serialize(
            new StudentAdmittedEvent(student.Id, student.SchoolId, cmd.ParentPhone))
    };
    _db.OutboxMessages.Add(outboxMsg);

    await _db.SaveChangesAsync(ct);
    // ↑ Both student AND outbox message committed atomically
    // If this fails → neither saved → consistent
    // If this succeeds → both saved → background worker will publish event

    return new AdmitStudentResult(student.Id);
}
// Outbox processor — BackgroundService polling outbox table
public class OutboxProcessorService(IServiceProvider _sp, IEventPublisher _publisher)
    : BackgroundService
{
    protected override async Task ExecuteAsync(CancellationToken ct)
    {
        while (!ct.IsCancellationRequested)
        {
            await ProcessPendingMessagesAsync(ct);
            await Task.Delay(TimeSpan.FromSeconds(5), ct); // Poll every 5s
        }
    }

    private async Task ProcessPendingMessagesAsync(CancellationToken ct)
    {
        using var scope = _sp.CreateScope();
        var db = scope.ServiceProvider.GetRequiredService<ZenCampusDbContext>();

        var pending = await db.OutboxMessages
            .Where(m => m.ProcessedAt == null)
            .OrderBy(m => m.OccurredAt)
            .Take(20)          // Batch of 20
            .ToListAsync(ct);

        foreach (var msg in pending)
        {
            try
            {
                await _publisher.PublishRawAsync(msg.EventType, msg.Payload, ct);
                msg.ProcessedAt = DateTime.UtcNow;
            }
            catch (Exception ex)
            {
                msg.Error = ex.Message;
                // ProcessedAt stays null → will retry next poll
            }
        }

        await db.SaveChangesAsync(ct);
    }
}
Idempotency: Service Bus consumer must handle duplicate messages (at-least-once delivery). Store processed MessageId in ProcessedMessages table. Check before processing: if (await IsAlreadyProcessed(messageId)) { complete; return; } — this makes consumer idempotent.

🎯 10 Interview Scenarios

Q1. What is IHostedService vs BackgroundService? What's the difference?
Concept

IHostedService is the interface — you implement StartAsync and StopAsync manually.

BackgroundService is an abstract class that implements IHostedService — you override only ExecuteAsync(CancellationToken). It handles the boilerplate: CancellationToken wiring, exception handling skeleton, proper lifecycle.

// IHostedService — more control, more boilerplate
public class MyService : IHostedService
{
    public Task StartAsync(CancellationToken ct) { /* start timer/thread */ }
    public Task StopAsync(CancellationToken ct)  { /* cleanup */ }
}

// BackgroundService — cleaner, recommended
public class MyService : BackgroundService
{
    protected override async Task ExecuteAsync(CancellationToken ct)
    {
        while (!ct.IsCancellationRequested) { /* work */ }
    }
}

ZenCampus: "We use BackgroundService for all our hosted services — AttendanceSyncService, OutboxProcessorService. It's simpler and the CancellationToken propagation is handled correctly."

Q2. Production Bug: Fee reminder SMS scheduled job is failing silently — parents never get reminders, but no error in logs. How do you diagnose?
Production Bug

Most likely causes (check in order):

  1. Scoped service in singleton: If IHostedService directly injects IStudentRepository (scoped) — DI throws at startup but may be swallowed. Check: use IServiceProvider.CreateScope()
  2. async void timer callback: private async void TimerCallback() — exceptions in async void are unobserved and swallowed! Change to async Task and await properly
  3. Hangfire workers not running: Check /hangfire dashboard — are workers active? Is HangfireDb connection string valid?
  4. RecurringJob not registered: RecurringJob.AddOrUpdate must be called after app builds — not during builder.Services phase
// ❌ Silent exception swallower
private async void OnTimer(object? state)  // async void = exception disappears!
{
    await _smsService.SendRemindersAsync(); // throws → nobody hears it
}

// ✅ Fix — catch and log
private async void OnTimer(object? state)
{
    try { await _smsService.SendRemindersAsync(); }
    catch (Exception ex) { _log.LogError(ex, "Fee reminder job failed"); }
}
// Better: use Hangfire — it logs all job failures automatically
Q3. When do you choose Hangfire over IHostedService? When do you choose Azure Service Bus over Hangfire?
Design

Hangfire over IHostedService when:

  • Job must survive server restarts (IHostedService = in-memory)
  • Need dashboard to see job status, retry history, failures
  • Need auto-retry with backoff (Hangfire: 10 retries, exponential delay)
  • Need to enqueue jobs from within a request (fire-and-forget)
  • Need scheduled/delayed jobs dynamically (BackgroundJob.Schedule(delay))

Azure Service Bus over Hangfire when:

  • Multiple different services must react to same event (fan-out)
  • Producer and consumer are in separate microservices/processes
  • Need ordering guarantees (Sessions: per-school ordered processing)
  • Need Dead Letter Queue for poison messages
  • Scale: millions of messages/day (Service Bus handles unlimited; Hangfire limited by SQL)
  • Event-driven architecture (EDA) between domains

ZenCampus: "We use all three: IHostedService for daily sync (simple, in-process), Hangfire for SMS/PDF after admission (fire-and-forget with retry), Service Bus for fee payment events (multiple consumers across modules)."

Q4. Explain the Outbox Pattern. Why do you need it? What problem does it solve?
Concept

The Problem: You save to DB, then publish to Service Bus. If the server crashes between these two operations — DB has the data but the event was never published. Fee record not created, SMS not sent — data inconsistency.

Two-phase commit (distributed transactions) could solve it but is slow and complex.

Outbox Pattern solution:

  1. Save domain data + outbox message in same DB transaction
  2. Background OutboxProcessor polls outbox table, publishes to Service Bus
  3. Marks message as processed (ProcessedAt = DateTime.UtcNow)
  4. If publish fails → ProcessedAt stays null → retry next poll

Guarantees: At-least-once delivery (not exactly-once — consumer must be idempotent).
Library: MassTransit + EntityFramework outbox, or Quartz.NET — both implement this pattern built-in.

Q5. Service Bus consumer processes a fee payment message, sends SMS, then DB update fails. Message is completed. Parent gets SMS but DB has no record. How to fix?
Production Bug

Root cause: SMS sent and message completed before DB update. Operations not atomic.

Fix — correct order + idempotency:

private async Task HandleMessageAsync(ProcessMessageEventArgs args)
{
    var @event = JsonSerializer.Deserialize<FeePaymentEvent>(args.Message.Body.ToString())!;

    // 1. Check idempotency — already processed?
    if (await _db.ProcessedMessages.AnyAsync(m => m.MessageId == args.Message.MessageId))
    {
        await args.CompleteMessageAsync(args.Message);
        return; // Already done — safe to complete
    }

    // 2. DB update FIRST (if this fails → message goes back to queue → retry OK)
    await _feeRepo.UpdateBalanceAsync(@event.StudentId, @event.Amount);
    _db.ProcessedMessages.Add(new ProcessedMessage { MessageId = args.Message.MessageId });
    await _db.SaveChangesAsync();

    // 3. SMS after DB commit (SMS failure won't affect DB — idempotent retry is fine for SMS)
    await _sms.SendReceiptAsync(@event.Phone, @event.ReceiptNo);

    // 4. Complete LAST — only after everything succeeded
    await args.CompleteMessageAsync(args.Message);
}
Q6. How does Hangfire handle retries? What happens after all retries fail?
Concept

Default behavior (AutomaticRetry attribute):

  • Default: 10 retry attempts
  • Backoff: ~1min, 10min, 30min, 1h, 2h, 4h, 8h, 16h, 24h, 24h
  • Job state after attempt: Failed → scheduled retry → Processing → Failed again
  • After all retries: job moves to Failed final state (stays in DB for inspection)
// Custom retry policy
[AutomaticRetry(Attempts = 3, DelaysInSeconds = new[] { 60, 300, 3600 },
    OnAttemptsExceeded = AttemptsExceededAction.Delete)] // Or .Fail
public async Task SendSmsAsync(string phone, string message) { ... }

// Monitor failed jobs
var monitor = JobStorage.Current.GetMonitoringApi();
var failed = monitor.FailedJobs(0, 50);
// Or check /hangfire dashboard in dev

Production alert: Set up alert when failed job count > 0 — use Azure Monitor or Hangfire's GlobalJobFilters to send alert when jobs hit final failed state.

Q7. Service Bus sends duplicate messages sometimes (at-least-once delivery). How do you handle duplicates in ZenCampus?
Design

At-least-once delivery is guaranteed — exactly-once is not (and is expensive). Solution: Idempotent consumer.

  1. Every message has unique MessageId (we set it to Guid.NewGuid())
  2. Consumer checks ProcessedMessages table before processing
  3. If already processed → complete message and return (skip)
  4. If new → process, save to ProcessedMessages, complete

Alternatively: Enable Service Bus duplicate detection at Topic level — deduplicate window (5min to 7 days). Service Bus itself discards duplicates within that window. Combine both for belt-and-suspenders.

// Idempotency check table
public class ProcessedMessage
{
    public string   MessageId   { get; set; } = string.Empty;
    public string   EventType   { get; set; } = string.Empty;
    public DateTime ProcessedAt { get; set; } = DateTime.UtcNow;
}
// Index on MessageId for fast dedup check
Q8. Student admitted at 11:59 PM, server crashes at midnight during Hangfire job processing. Will the welcome SMS be sent?
Resilience

With Hangfire: Yes, eventually.

  1. Job enqueued → saved to HangfireDb (SQL)
  2. Server crashes while job is Processing
  3. Server restarts → Hangfire checks for jobs stuck in Processing state
  4. Jobs stuck > 30 minutes (configurable) → moved back to Enqueued
  5. Worker picks it up and processes → SMS sent

Without Hangfire (fire-and-forget thread): No — in-memory thread is lost on crash.

With IHostedService timer: No — next trigger is at next timer tick, job state lost.

Hangfire config: opts.InvisibilityTimeout = TimeSpan.FromMinutes(30) — how long before stuck jobs are re-queued.

Q9. 200 schools all collect fees simultaneously. Service Bus consumer falls behind — messages pile up. How do you scale?
Performance

Multiple scaling options:

  1. Increase MaxConcurrentCalls in ServiceBusProcessorOptions: MaxConcurrentCalls = 20 — process 20 messages in parallel per consumer instance
  2. Multiple consumer instances: Deploy 3 instances of ZenCampus API — each has its own BackgroundService consumer. Service Bus distributes messages across all instances (competing consumers pattern)
  3. Partitioned Queue/Topic: Enable partitioning in Service Bus — messages distributed across 16 partitions, multiple consumers process in parallel
  4. Scale out on Azure Container Apps / AKS: KEDA (Kubernetes Event-Driven Autoscaling) — auto-scale consumer pods based on Service Bus queue depth
// Consumer with higher concurrency
var processor = _client.CreateProcessor("fee-payment-events", "sms-subscription",
    new ServiceBusProcessorOptions
    {
        MaxConcurrentCalls   = 20,   // 20 parallel
        PrefetchCount        = 50,   // Prefetch for throughput
        AutoCompleteMessages = false
    });

🧠 Quick Quiz

Q1. You inject IStudentRepository (Scoped) directly into a BackgroundService (Singleton). What happens?
InvalidOperationException at startup — ASP.NET Core validates service scope at startup by default. Fix: inject IServiceProvider and call CreateScope() inside ExecuteAsync. This is one of the most common IHostedService mistakes in interviews.
Q2. Hangfire job fails (Msg91 throws exception). What happens by default?
Auto-retry 10 times with exponential backoff — this is Hangfire's default [AutomaticRetry] behavior. After 10 failures, job moves to Failed state (visible in dashboard). Backoff: ~1min, 10min, 30min, 1h, 2h... This is why Hangfire is better than manual fire-and-forget for SMS jobs.
Q3. Service Bus delivers a fee payment message TWICE (at-least-once delivery). Your consumer processes it twice — parent gets 2 SMS. How to prevent?
Idempotent consumer — store MessageId in ProcessedMessages table after first processing. Check before processing: if already exists, complete and return. Service Bus exactly-once is not available (only duplicate detection within a time window). Sessions provide ordering, not deduplication.
Q4. Student admitted → save to DB → server crashes before Service Bus publish. Fee record never created. Which pattern prevents this?
Outbox Pattern — saves domain event to OutboxMessages table in same DB transaction as domain data. Background OutboxProcessor polls and publishes. If publish fails, retry next poll. Atomic by design. 2PC is slow/complex. Polly retry doesn't help if server crashes after DB save. Saga is for orchestrating multi-step workflows, not guaranteed publishing.
