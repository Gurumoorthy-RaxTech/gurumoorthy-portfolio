📄 Full ZenCampus CI/CD Workflow

# .github/workflows/zencampus-cicd.yml
name: ZenCampus CI/CD

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

env:
  DOTNET_VERSION: '8.0.x'
  AZURE_WEBAPP_NAME: zencampus-api
  ACR_REGISTRY: zencampusacr.azurecr.io
  IMAGE_NAME: zencampus-api

jobs:
  # ── Job 1: Build + Test ────────────────────────────
  build-and-test:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Setup .NET
        uses: actions/setup-dotnet@v4
        with:
          dotnet-version: ${{ env.DOTNET_VERSION }}

      - name: Restore dependencies
        run: dotnet restore ZenCampusWeb/ZenCampusWeb.csproj

      - name: Build
        run: dotnet build ZenCampusWeb/ZenCampusWeb.csproj \
          -c Release --no-restore

      - name: Run tests
        run: dotnet test ZenCampusWeb.Tests/ZenCampusWeb.Tests.csproj \
          --no-build -c Release \
          --logger "trx;LogFileName=test-results.trx" \
          --collect:"XPlat Code Coverage"

      - name: Upload test results
        uses: actions/upload-artifact@v4
        if: always()
        with:
          name: test-results
          path: '**/test-results.trx'

      - name: Check code coverage
        run: |
          COVERAGE=$(dotnet tool run reportgenerator -reports:"**/coverage.cobertura.xml" \
            -targetdir:"coverage" -reporttypes:TextSummary 2>&1 | grep "Line coverage")
          echo "$COVERAGE"

  # ── Job 2: Docker Build + Push to ACR ──────────────
  docker-build-push:
    runs-on: ubuntu-latest
    needs: build-and-test
    if: github.ref == 'refs/heads/main' || github.ref == 'refs/heads/develop'
    steps:
      - uses: actions/checkout@v4

      - name: Log in to Azure Container Registry
        uses: azure/docker-login@v1
        with:
          login-server: ${{ env.ACR_REGISTRY }}
          username: ${{ secrets.ACR_USERNAME }}
          password: ${{ secrets.ACR_PASSWORD }}

      - name: Build and push Docker image
        run: |
          IMAGE_TAG=${{ env.ACR_REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }}
          LATEST_TAG=${{ env.ACR_REGISTRY }}/${{ env.IMAGE_NAME }}:latest

          docker build -t $IMAGE_TAG -t $LATEST_TAG \
            -f ZenCampusWeb/Dockerfile .

          docker push $IMAGE_TAG
          docker push $LATEST_TAG

          echo "IMAGE_TAG=$IMAGE_TAG" >> $GITHUB_ENV

  # ── Job 3: Deploy to Staging ────────────────────────
  deploy-staging:
    runs-on: ubuntu-latest
    needs: docker-build-push
    if: github.ref == 'refs/heads/develop'
    environment: staging
    steps:
      - name: Deploy to Azure App Service (staging slot)
        uses: azure/webapps-deploy@v3
        with:
          app-name: ${{ env.AZURE_WEBAPP_NAME }}
          slot-name: staging
          images: ${{ env.ACR_REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }}
          publish-profile: ${{ secrets.AZURE_WEBAPP_PUBLISH_PROFILE_STAGING }}

  # ── Job 4: Deploy to Production ─────────────────────
  deploy-production:
    runs-on: ubuntu-latest
    needs: docker-build-push
    if: github.ref == 'refs/heads/main'
    environment: production  # requires manual approval
    steps:
      - name: Azure Login (OIDC / Workload Identity — no password!)
        uses: azure/login@v2
        with:
          client-id: ${{ secrets.AZURE_CLIENT_ID }}
          tenant-id: ${{ secrets.AZURE_TENANT_ID }}
          subscription-id: ${{ secrets.AZURE_SUBSCRIPTION_ID }}

      - name: Deploy to Azure App Service (production)
        uses: azure/webapps-deploy@v3
        with:
          app-name: ${{ env.AZURE_WEBAPP_NAME }}
          images: ${{ env.ACR_REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }}

🎯 10 Interview Scenarios

1
What is the difference between CI and CD?Concept

CI (Continuous Integration): automatically build and test every code push. Catches integration bugs early. CD (Continuous Deployment): automatically deploy to production after tests pass. Or Continuous Delivery: deploy to staging automatically, manual gate for production. For ZenCampus: CI = every PR triggers build + tests. CD = main branch automatically deploys to staging, production requires approval.

2
Why use a Docker image tag with git SHA instead of "latest"?Practical

SHA tag (zencampus-api:abc123f) is immutable — always points to exactly the code at that commit. "latest" is mutable — overwritten by every new build. With SHA tags: you can redeploy any previous version by specifying its SHA. You can trace which exact code is running in production. If bug found, rollback = redeploy the previous SHA tag. "latest" is fine for local dev, but production needs SHA tags.

3
What is OIDC/Workload Identity Federation in GitHub Actions?Security

Instead of storing Azure service principal password in GitHub Secrets: configure Azure to trust GitHub's identity provider. When workflow runs, GitHub gets a short-lived JWT. Azure validates the JWT and issues a short-lived access token. No long-lived secrets stored anywhere. Zero credential rotation. If GitHub is compromised, the token expires in minutes — no persistent access.

4
What does "needs: build-and-test" do in a GitHub Actions workflow?Code

Creates a dependency between jobs. The docker-build-push job will not start until build-and-test job completes successfully. If tests fail, the build job never runs — no Docker image built, no deployment happens. Without needs: jobs run in parallel. With needs: forms a directed acyclic graph (DAG) of dependencies. Critical for ZenCampus: never deploy code that has failing tests.

5
How do you use GitHub Environments for production deployment approval?Practical

Create environment "production" in GitHub Settings. Add required reviewers. Add branch protection: only main. In workflow: specify "environment: production" on deploy job. When workflow reaches that job, GitHub pauses and notifies reviewers. Reviewer sees what's being deployed (SHA, message, who triggered). Approve = deployment proceeds. Reject = workflow fails. Perfect for ZenCampus: automated pipeline + human gate for 50,000 students' data.

6
Why do tests run before the Docker image is built in the CI pipeline?Concept

Fail fast: tests are faster than Docker build. If tests fail (20-30 seconds), save the 5-minute Docker build and registry push. Don't waste CI minutes building and pushing an image that's known-broken. For ZenCampus: unit tests catch fee calculation bugs before they're packaged and deployed. Order: restore → build → test → docker build → push → deploy. Each step gates the next.

7
How would you handle EF Core migrations in a CI/CD pipeline?Practical

Option 1: Generate migration script in CI (dotnet ef migrations script --idempotent -o migration.sql), upload as artifact, DBA reviews and applies before deployment. Option 2: Add a migration step before deploy job that runs the script against staging DB. Option 3: App applies migrations on startup (Program.cs: dbContext.Database.MigrateAsync()). Production: Option 1 or 2 — migrations reviewed before 50,000-student database is touched.

8
What is the purpose of "if: always()" on an upload-artifact step?Code

By default, steps are skipped if a previous step fails. "if: always()" means this step runs regardless — even if tests failed. Critical for test result artifacts: if tests FAIL, you especially need the test result files to diagnose what failed. Without always(): test fails → artifact step skipped → no report → you can't see what failed. With always(): test results always uploaded for review.

9
How do you handle different config per environment (staging vs production) in GitHub Actions?Practical

Environment-level secrets in GitHub: staging environment has STAGING_SQL_CONN, production environment has PROD_SQL_CONN. When job specifies "environment: staging", it gets staging secrets. Job with "environment: production" gets production secrets. App Service config (connection strings, Key Vault URIs) set separately per slot. Image is the same — only runtime config differs between environments.

10
What is a branch protection rule and why should main require CI to pass?Concept

Branch protection: rules enforced before code can merge to protected branch. Required status checks: "build-and-test" job must pass before PR can merge to main. No direct push to main (all changes via PR). Code owners review required. For ZenCampus: developer can't merge fee calculation change until tests pass. Can't push directly to main. Senior developer reviews before merge. Prevents broken code from reaching production pipeline.
