2 Strategy Pattern — Algorithm swap at runtime
🎬 Anniyan (2005) — Same person, different punishment style

Ambi/Anniyan/Remo — same goal, different approach. Strategy Pattern = same interface, different algorithms. FeeCalculation, NotificationStrategy, ReportFormat — all interchangeable!

// Strategy — ZenCampus notification system
public interface INotificationStrategy
{
    Task NotifyAsync(Student student, string message);
    string Channel { get; }
}

public class SmsStrategy : INotificationStrategy
{
    public string Channel => "SMS";
    public async Task NotifyAsync(Student s, string msg)
        => await _smsGateway.SendAsync(s.Mobile, msg);
}

public class EmailStrategy : INotificationStrategy
{
    public string Channel => "Email";
    public async Task NotifyAsync(Student s, string msg)
        => await _emailSvc.SendAsync(s.Email, msg);
}

public class WhatsAppStrategy : INotificationStrategy
{
    public string Channel => "WhatsApp";
    public async Task NotifyAsync(Student s, string msg)
        => await _waSvc.SendAsync(s.Mobile, msg);
}

// Context — uses any strategy
public class NotificationService
{
    private readonly Dictionary<string, INotificationStrategy> _strategies;

    public NotificationService(IEnumerable<INotificationStrategy> strategies)
        => _strategies = strategies.ToDictionary(s => s.Channel);

    public async Task NotifyAsync(Student student, string channel, string msg)
    {
        if (!_strategies.TryGetValue(channel, out var strategy))
            throw new ArgumentException($"Unknown channel: {channel}");
        await strategy.NotifyAsync(student, msg);
    }

    // Multi-channel blast
    public async Task NotifyAllAsync(Student student, string msg)
        => await Task.WhenAll(_strategies.Values.Select(s => s.NotifyAsync(student, msg)));
}

// DI: register all strategies — auto-discovered!
services.AddScoped<INotificationStrategy, SmsStrategy>();
services.AddScoped<INotificationStrategy, EmailStrategy>();
services.AddScoped<INotificationStrategy, WhatsAppStrategy>();
3 Factory Pattern — Object creation centralize பண்ணு
🎬 Enthiran — Chitti factory-la உருவாகுது

Robot directly new() பண்றோம் இல்ல — factory assembly line create பண்ணுது. Factory Pattern = object creation logic centralize, caller doesn't need to know HOW to create!

// Factory Method — ZenCampus fee receipt generator
public interface IReceiptGenerator
{
    byte[] Generate(FeeRecord record);
    string ContentType { get; }
}

public class PdfReceiptGenerator  : IReceiptGenerator
{
    public string ContentType => "application/pdf";
    public byte[] Generate(FeeRecord r) { /* iTextSharp PDF */ return null; }
}
public class ExcelReceiptGenerator: IReceiptGenerator
{
    public string ContentType => "application/vnd.openxmlformats-officedocument.spreadsheetml.sheet";
    public byte[] Generate(FeeRecord r) { /* EPPlus Excel */ return null; }
}

// Factory — caller asks for type, factory handles creation
public static class ReceiptGeneratorFactory
{
    public static IReceiptGenerator Create(string format) => format.ToLower() switch
    {
        "pdf"   => new PdfReceiptGenerator(),
        "excel" => new ExcelReceiptGenerator(),
        _       => throw new ArgumentException($"Unknown format: {format}")
    };
}

// Usage — caller doesn't know PDF vs Excel internals
var gen = ReceiptGeneratorFactory.Create(userPreference);
var bytes = gen.Generate(feeRecord);
return File(bytes, gen.ContentType);
💡 Abstract Factory vs Factory Method

Factory Method = one product, one factory method (PDF or Excel).
Abstract Factory = family of related products (Report + Receipt + Certificate — all PDF together OR all Excel together).

4 Repository Pattern — Data access abstraction
🎬 Ratsasan — Evidence room, one person manages

Direct evidence access பண்றோம் இல்ல — Evidence room officer மூலம்! Repository = DB access one layer. Business code SQL தெரியாம StudentRepository use பண்ணும். Testable, swappable!

// Repository with Unit of Work — ZenCampus
public interface IStudentRepository
{
    Task<Student?> GetByIdAsync(int id);
    Task<List<Student>> GetByClassAsync(string className);
    Task<List<Student>> GetFeePendingAsync();
    Task AddAsync(Student student);
    Task UpdateAsync(Student student);
}

public interface IUnitOfWork : IDisposable
{
    IStudentRepository Students { get; }
    IFeeRepository     Fees     { get; }
    Task<int> SaveChangesAsync();
}

// Service uses UoW — no DB details!
public class FeePaymentService
{
    private readonly IUnitOfWork _uow;

    public async Task ProcessPaymentAsync(int studentId, decimal amount)
    {
        var student = await _uow.Students.GetByIdAsync(studentId)
            ?? throw new StudentNotFoundException(studentId);

        await _uow.Fees.AddAsync(new FeeRecord
        {
            StudentId = studentId,
            Amount    = amount,
            PaidOn    = DateTime.Now
        });
        student.FeesPaid = true;
        await _uow.Students.UpdateAsync(student);
        await _uow.SaveChangesAsync(); // atomic transaction!
    }
}

// Event dispatcher
public class EventDispatcher
{
    private readonly IServiceProvider _sp;

    public async Task DispatchAsync<T>(T eventData)
    {

    }

7 Decorator Pattern — Add behavior without modifying class
🎬 Dasavathaaram — Kamal wears different costumes, same actor underneath

Costume (decorator) adds behavior — same actor underneath. Decorator wraps existing class, adds new behavior. Logging, Caching, Retry — add without touching original!

// Decorator — add caching to StudentRepository
public class CachedStudentRepository : IStudentRepository
{
    private readonly IStudentRepository _inner;  // wrapped!
    private readonly IMemoryCache       _cache;

    public CachedStudentRepository(
        IStudentRepository inner,
        IMemoryCache cache)
    {
        _inner = inner;
        _cache = cache;
    }

    public async Task<Student?> GetByIdAsync(int id)
    {
        var key = $"student:{id}";
        if (_cache.TryGetValue(key, out Student? cached))
            return cached; // cache hit!

        var student = await _inner.GetByIdAsync(id); // delegate
        _cache.Set(key, student, TimeSpan.FromMinutes(5));
        return student;
    }

    // Other methods delegate directly to _inner
    public async Task<List<Student>> GetByClassAsync(string c)
        => await _inner.GetByClassAsync(c);
}

// DI — Decorator wires up transparently
services.AddScoped<SqlStudentRepository>();
services.AddScoped<IStudentRepository>(sp =>
    new CachedStudentRepository(
        sp.GetRequiredService<SqlStudentRepository>(),
        sp.GetRequiredService<IMemoryCache>()));

❌ Strategy = algorithm/behavior swap. Factory = creation abstraction. Different concerns, often used together!

Q3: Singleton scoped dependency trap — what happens?

A) Works fine
B) Performance issue only
C) Scoped service captured in Singleton — wrong scope, stale data, runtime error
D) Compile error
✅ Captive dependency! Singleton (app lifetime) captures Scoped (request lifetime) → same scoped instance reused across all requests → stale data, potential bugs. ASP.NET Core detects and throws!
❌ Captive dependency trap! Scoped service in Singleton = same instance used for all requests = wrong scope = stale DbContext = data corruption!

Q4: Decorator pattern — how different from inheritance?

A) No difference
B) Decorator wraps existing instance (composition). Inheritance extends compile-time (static). Decorator swappable at runtime.
C) Decorator is faster
D) Decorator needs abstract class
✅ Decorator = composition over inheritance. Wraps object, adds behavior. Can chain decorators (Cache → Log → SQL). Inheritance = static, single parent, compile-time only.
❌ Decorator = runtime composition! Wrap instance, add behavior, chain multiple. Inheritance = static, one parent. "Favor composition over inheritance" = Decorator!
