3 Service Discovery — Find Where Services Are
🎬 Vikram — Rolex paaru exactly which agent is active. Service Discovery = same. Which StudentService instance is alive? Consul knows!

Without discovery: hardcode IPs → new instance spins up → gateway doesn't know. With Consul: each service registers on startup, sends heartbeat every 10s, Consul deregisters if heartbeat stops. Gateway asks Consul "StudentService available?" → gets live IPs.

// NuGet: Consul

// Service registers itself with Consul on startup
public static class ConsulRegistration
{
    public static IApplicationBuilder UseConsul(this IApplicationBuilder app)
    {
        var consul  = app.ApplicationServices.GetRequiredService<IConsulClient>();
        var life    = app.ApplicationServices.GetRequiredService<IHostApplicationLifetime>();
        var cfg     = app.ApplicationServices.GetRequiredService<IConfiguration>();

        var serviceId = $"student-service-{Guid.NewGuid()}"; // unique per instance

        life.ApplicationStarted.Register(() =>
        {
            consul.Agent.ServiceRegister(new AgentServiceRegistration
            {
                ID      = serviceId,
                Name    = "student-service",
                Address = cfg["Service:Host"],
                Port    = int.Parse(cfg["Service:Port"]!),
                Check   = new AgentServiceCheck
                {
                    HTTP                        = $"http://{cfg["Service:Host"]}:{cfg["Service:Port"]}/health",
                    Interval                    = TimeSpan.FromSeconds(10),
                    DeregisterCriticalServiceAfter = TimeSpan.FromMinutes(1)
                }
            });
        });

        // Deregister on shutdown — clean exit
        life.ApplicationStopping.Register(() =>
            consul.Agent.ServiceDeregister(serviceId).Wait());

        return app;
    }
}

4 Polly — Resilience Policies
// NuGet: Microsoft.Extensions.Http.Resilience (Polly v8 — .NET 8+)

// Program.cs — configure resilient HttpClient for StudentService calls
builder.Services.AddHttpClient<IStudentServiceClient, StudentServiceClient>(c =>
{
    c.BaseAddress = new Uri("http://student-service");
    c.Timeout     = TimeSpan.FromSeconds(30);
})
.AddResilienceHandler("student-pipeline", pipeline =>
{
    // 1. Retry — 3 attempts, exponential backoff
    pipeline.AddRetry(new HttpRetryStrategyOptions
    {
        MaxRetryAttempts = 3,
        BackoffType      = DelayBackoffType.Exponential,
        UseJitter        = true,  // avoids thundering herd
        Delay            = TimeSpan.FromMilliseconds(500)
    });

    // 2. Circuit Breaker — open after 5 failures
    pipeline.AddCircuitBreaker(new HttpCircuitBreakerStrategyOptions
    {
        SamplingDuration       = TimeSpan.FromSeconds(30),
        FailureRatio           = 0.5,  // 50% failure rate → open
        MinimumThroughput      = 10,   // need at least 10 calls to evaluate
        BreakDuration          = TimeSpan.FromSeconds(30),
        OnOpened               = args =>
        {
            _log.LogWarning("StudentService circuit OPENED — fast-fail mode");
            return ValueTask.CompletedTask;
        }
    });

    // 3. Timeout per attempt
    pipeline.AddTimeout(TimeSpan.FromSeconds(5));
});
5 Circuit Breaker — States + ZenCampus Fallback
🎬 96 — When connection fails, protect yourself. Don't keep trying. Circuit Breaker = same. Stop hammering a dead service. Protect cascading failure.

96 film-la Ram keeps calling Janu. After N failures — stop. Circuit opens. Wait. Try again later (half-open). This is exactly the Circuit Breaker pattern. FeeService calling StudentService 1000 times when it's down = kills both services. CB prevents cascade!

🟢 CLOSED — normal, requests flow → failures exceed threshold → 🔴 OPEN — fast-fail, no calls to service → break duration expires → 🟡 HALF-OPEN — test call, if success → CLOSED, if fail → OPEN 
// Fallback when circuit is OPEN — serve stale data instead of error
public async Task<StudentDto?> GetStudentAsync(int studentId)
{
    try
    {
        return await _httpClient.GetFromJsonAsync<StudentDto>(
            $"/api/students/{studentId}");
    }
    catch (BrokenCircuitException)
    {
        // Circuit is OPEN — don't wait, fail fast!
        _log.LogWarning("StudentService unavailable — returning cached data");

        // Fallback 1: Redis stale cache
        var cached = await _redis.GetAsync<StudentDto>($"student:{studentId}");
        if (cached is not null) return cached;

        // Fallback 2: return null (fee service shows "Student unavailable")
        return null;
    }
    catch (HttpRequestException ex)
    {
        _log.LogError(ex, "StudentService HTTP error for student {Id}", studentId);
        throw;
    }
}

7 gRPC — High-Performance Inter-service Calls

gRPC = Google RPC. Binary (Protocol Buffers), typed contracts, bidirectional streaming. Faster than REST for service-to-service. HTTP/2 multiplexing. NOT human-readable — use for internal service calls only.

// student.proto — contract-first API definition
syntax = "proto3";
package students;

service StudentService {
    rpc GetStudent     (StudentRequest) returns (StudentResponse);
    rpc GetStudentsBulk(BulkRequest)    returns (BulkResponse);
    rpc StreamStudents (SchoolRequest)  returns (stream StudentResponse); // streaming!
}

message StudentRequest  { int32 student_id = 1; }
message StudentResponse {
    int32  student_id  = 1;
    string name        = 2;
    int32  school_id   = 3;
    string grade       = 4;
}
message BulkRequest  { repeated int32 student_ids = 1; }
message BulkResponse { repeated StudentResponse students = 1; }
message SchoolRequest { int32 school_id = 1; }
// gRPC Server — StudentService
public class StudentGrpcService : StudentService.StudentServiceBase
{
    public override async Task<StudentResponse> GetStudent(
        StudentRequest request, ServerCallContext context)
    {
        var s = await _repo.GetByIdAsync(request.StudentId);
        return new StudentResponse
        {
            StudentId = s.Id,
            Name      = s.Name,
            SchoolId  = s.SchoolId,
            Grade     = s.Grade
        };
    }

    // Server-side streaming — stream all students for a school
    public override async Task StreamStudents(
        SchoolRequest request, IServerStreamWriter<StudentResponse> stream,
        ServerCallContext context)
    {
        var students = _repo.GetBySchoolAsync(request.SchoolId, context.CancellationToken);
        await foreach (var s in students)
            await stream.WriteAsync(new StudentResponse { StudentId = s.Id, Name = s.Name });
    }
}

// gRPC Client — AttendanceService calling StudentService
builder.Services.AddGrpcClient<StudentService.StudentServiceClient>(o =>

    .AddResilienceHandler("grpc-pipeline", p => { /* retry + CB */ });

// Usage
var response = await _grpcClient.GetStudentAsync(new StudentRequest { StudentId = 101 });

2
API Gateway — how does JWT validation work at scale? Medium Real Interview
📋 Scenario: "'ZenCampus microservices — 5 services. Each service validates JWT? Or gateway validates? What if teacher's role changes — how does it propagate?'"
✅ GATEWAY VALIDATES — SERVICES TRUST CLAIMS

Architecture: JWT validated ONCE at API Gateway (Ocelot/YARP). If valid, gateway forwards request with claims as headers (X-UserId, X-Role, X-SchoolId). Downstream services trust these headers — no re-validation needed. Saves CPU across all services.

Role change propagation: JWT has 15-min TTL. Role change → old JWT still valid until expiry. To revoke immediately: (1) JTI blacklist in Redis — gateway checks blacklist on every request (O(1)). (2) Or force logout via refresh token revocation — next token refresh fails, user must re-login with new role.

Service-to-service: When StudentService calls FeeService internally, it passes a service-account JWT (not user's JWT). Use client credentials grant (OAuth2 machine-to-machine). FeeService validates this service JWT to ensure only StudentService can call it — not any external actor.

Duplicate message — fee paid event processed twice! 🔥 Production Hard
📋 Scenario: "RabbitMQ redelivers fee.paid event because consumer crashed after processing but before ACK. Student gets 2 SMS. NotificationService sends duplicate receipt. Parent angry — 'amount debited twice!'"

// Message must carry unique ID — event ID or correlation ID
public record FeePaidEvent(
    Guid    EventId,      // unique per event — used for dedup!
    int     StudentId,
    decimal Amount,
    string  ReceiptNo
);

public class FeePaidHandler : IConsumer<FeePaidEvent>
{
    public async Task Consume(ConsumeContext<FeePaidEvent> ctx)
    {
        var evt    = ctx.Message;
        var dedupKey = $"processed:event:{evt.EventId}";

        // Redis NX — only set if NOT already processed
        var isNew = await _redis.StringSetAsync(
            dedupKey, "1", TimeSpan.FromDays(7), When.NotExists);

        if (!isNew)
        {
            _log.LogWarning("Duplicate event {Id} — skipping", evt.EventId);
            return; // ACK but don't process — idempotent!
        }

        // Process only if new
        await _sms.SendAsync(evt.StudentId, evt.ReceiptNo);
    }
}
// Redis NX = atomic check-and-set. Even if 2 consumers race → only 1 processes!

Monolith to microservices — strangler fig pattern Medium Real Interview
📋 Scenario: "'How would you migrate ZenCampus monolith to microservices without breaking production?'"
✅ STRANGLER FIG PATTERN — INCREMENTAL MIGRATION

Strangler Fig Pattern: Don't rewrite all at once — strangle the monolith gradually.

Step 1 — API Gateway first: Put YARP in front of monolith. All traffic → Gateway → Monolith (unchanged). Zero downtime.

Step 2 — Extract least-coupled service: NotificationService first — pure output, no shared DB. Route /api/notifications/* → new NotificationService. Gateway handles routing. Monolith unaffected.

Step 3 — Extract next service (FeeService): Create FeeService with its own DB. Copy fee data. Switch reads to new service. Validate. Switch writes. Monolith fee module becomes dead code.

Step 4 — Repeat: StudentService, AttendanceService. Monolith shrinks. Eventually empty → decommission.

Anti-pattern: Never do big-bang rewrite. Strangler Fig = zero-downtime, reversible at each step.

Circuit Breaker states — explain OPEN/CLOSED/HALF-OPEN Easy Real Interview
📋 Scenario: "'Circuit Breaker pattern — explain states with ZenCampus real example.'"
✅ STATES WITH REAL EXAMPLE

🟢 CLOSED — Normal operations. FeeService calls StudentService. All succeeds. Failure counter resets on success. This is the happy path.

🔴 OPEN — Fast-fail mode. StudentService crashed. 5 failures in 10s → circuit opens. Next 1000 calls to StudentService get BrokenCircuitException INSTANTLY — no HTTP call made. Polly serves fallback (Redis cache or 503). FeeService protected!

🟡 HALF-OPEN — Testing recovery. After 30s break → 1 test call allowed. If StudentService responds OK → circuit CLOSES (normal). If still fails → OPENS again for another 30s. Prevents premature recovery.

Analogy: Electrical circuit breaker at home. Overload → breaker trips (OPEN). Wait → manually reset (HALF-OPEN). If stable → stays reset (CLOSED). If overloads again → trips again.

9 ⚡ Quick Quiz

Q1: Circuit Breaker HALF-OPEN state — how many requests are allowed?

A) All requests — same as CLOSED state
B) One test request — success → CLOSED, failure → back to OPEN
C) Five requests, then evaluate
D) Random percentage of requests
✅ HALF-OPEN = probe state. 1 test request allowed after break duration. Success = service recovered → CLOSED. Failure = still broken → OPEN for another break period. This prevents premature recovery that would just flood a recovering service!
❌ HALF-OPEN allows exactly 1 test request! If that succeeds → CLOSED (normal). If it fails → back to OPEN for another break period. This cautious approach protects a recovering service from immediate flood.

Q2: Kubernetes liveness probe fails — what happens?

A) Pod removed from load balancer, but still running
B) Pod is restarted by Kubernetes
C) Pod traffic reduced to 50%
D) Alert sent but no action taken
✅ Liveness fail → k8s RESTARTS the pod. It means "process is stuck/dead". Readiness fail → removes from load balancer but doesn't restart (dependencies might just be temporarily unavailable). ZenCampus: liveness = just "I'm alive" check. Readiness = "I can handle traffic" (DB + Redis ok).
❌ Liveness = "is process alive?" Fail → k8s RESTARTS pod. Readiness = "can I handle traffic?" Fail → removed from LB (no restart). That's the key difference! Map healthchecks to /health/live (self) and /health/ready (DB, Redis, dependencies).

Q3: Microservices-la each service has own database — why?

B) Loose coupling — service can change its DB schema without impacting others. Independent deploy/scale.
A) Reduces cost — multiple small DBs cheaper than one big DB

Q4: IHttpClientFactory vs new HttpClient() — main difference?

A) IHttpClientFactory is async, HttpClient is synchronous
B) No difference — IHttpClientFactory just provides HttpClient
C) IHttpClientFactory manages connection pool — prevents socket exhaustion. new HttpClient() per request = too many sockets!
D) IHttpClientFactory only works in ASP.NET Core, not Worker Services
✅ new HttpClient() per request = socket exhaustion (each creates new TCP connection, not released immediately due to TIME_WAIT). IHttpClientFactory pools and reuses connections. Also handles lifetime properly — avoids DNS caching issue where long-lived HttpClient ignores DNS updates (k8s pod IP changes!).
❌ new HttpClient() per request = socket exhaustion! Each opens a socket, socket stays in TIME_WAIT after close (2 minutes). Under load = "port exhaustion" error. IHttpClientFactory pools connections. Also solves DNS caching issue — IHttpClientFactory rotates handlers so DNS changes propagate (important in k8s!)
