🗄️ Database Scaling Strategies

Read Replica — ZenCampus Report Queries

// DbContext factory — route reads to replica, writes to primary
public class ZenCampusDbContextFactory
{
    public ZenCampusDbContext CreateForRead()   => CreateContext(_replicaConnStr);
    public ZenCampusDbContext CreateForWrite()  => CreateContext(_primaryConnStr);
}

// Report service always uses read replica
public class AttendanceReportService(ZenCampusDbContextFactory _factory)
{
    public async Task<AttendanceReportDto> GetMonthlyReportAsync(int schoolId)
    {
        using var db = _factory.CreateForRead();  // read replica
        return await db.Attendances
            .AsNoTracking()
            .Where(a => a.SchoolId == schoolId)
            .GroupBy(a => a.StudentId)
            .Select(g => new { StudentId = g.Key, Present = g.Count(a => a.IsPresent) })
            .ToListAsync();
    }
}

📨 Message Queue — Azure Service Bus

// Send message to queue (non-blocking)
public class SmsQueuePublisher(ServiceBusSender _sender)
{
    public async Task EnqueueSmsAsync(string phone, string message)
    {
        var msg = new ServiceBusMessage(JsonSerializer.Serialize(new { phone, message }))
        {
            ContentType = "application/json",
            MessageId   = Guid.NewGuid().ToString(),
            TimeToLive  = TimeSpan.FromHours(24)
        };
        await _sender.SendMessageAsync(msg);
        // Returns immediately — SMS sent by worker process, not HTTP request
    }
}

// Worker reads from queue
public class SmsWorker(ServiceBusProcessor _processor, ISmsService _sms) : BackgroundService
{
    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        _processor.ProcessMessageAsync += async args =>
        {
            var payload = args.Message.Body.ToObjectFromJson<SmsPayload>();
            await _sms.SendAsync(payload.Phone, payload.Message);
            await args.CompleteMessageAsync(args.Message); // remove from queue
        };
        _processor.ProcessErrorAsync += args =>
        {
            // Message goes to dead-letter queue after max retry attempts
            return Task.CompletedTask;
        };
        await _processor.StartProcessingAsync(stoppingToken);
    }
}

ServiceReason to SplitTechnology

Notification ServiceHigh volume, separate scaling, multiple providers.NET + Azure Service Bus + Redis

Auth ServiceShared across all services.NET + Azure AD B2C or custom JWT
Don't split too early. ZenCampus at 200 schools? Monolith with good domain separation is fine. At 2,000 schools with different teams per domain? Microservices. The cost: distributed transactions, service discovery, network overhead, deployment complexity. Start monolith, split when team/scale demands it.

🎯 10 Interview Scenarios

1
ZenCampus has 50,000 students, exam results are released — 500 concurrent users. How do you handle this?System Design

Multi-layer approach: CDN serves static React app (zero API load). Azure App Service scales out to 5 instances automatically. Redis caches exam results per student (30 min TTL). Azure SQL read replica for result queries. Output Cache on API endpoint (30 sec). Rate limiting on result endpoint. Pre-generate PDF results in Hangfire before release time. Most users served from cache — DB barely touched.

2
Explain database sharding vs partitioning vs read replica.Concept

Read Replica: copy of primary, only for read queries. Reduces read load on primary. Replication lag = slightly stale data. Partitioning: split large table across multiple filegroups/files (same DB server). Attendance table partitioned by month — queries for "this month" hit only one partition. Sharding: distribute data across multiple database servers by shard key. ZenCampus: shard by SchoolId — school 1-100 on DB server 1, 101-200 on DB server 2. Complex to implement, requires shard-aware app code.

3
What is eventual consistency and when is it acceptable in ZenCampus?Concept

Eventual consistency: writes propagate through system eventually, not instantly. Acceptable: parent sees attendance SMS 2 minutes after teacher marks it — fine. Not acceptable: fee payment marked but balance not updated immediately — parent charged twice. Rule: for money/critical data, strong consistency. For notifications, reports, audit logs — eventual consistency acceptable. Service Bus gives reliable at-least-once delivery with eventual consistency.

4
How would you ensure ZenCampus has 99.9% uptime?Design

Multiple layers: Azure App Service with min 2 instances (single instance = planned downtime for updates). Azure SQL with geo-redundant backup + active geo-replication to secondary region. Azure Redis with geo-replication. Deployment slots for zero-downtime deploys. Health endpoints + App Service health probes. Application Insights alerts for p95 latency spikes. Runbook for failover. 99.9% = 8.7 hours downtime per year allowed.

5
What is the CAP theorem and how does it apply to ZenCampus?Concept

CAP: distributed system can guarantee only 2 of 3: Consistency, Availability, Partition tolerance. Azure SQL: CP (consistent + partition tolerant, may be briefly unavailable during failover). Redis: AP (available + partition tolerant, may serve slightly stale data). For ZenCampus: fee payments = CP (consistency over availability). Attendance dashboard = AP (available even if 30 sec stale). Choose based on business criticality.

6
Why use a message queue instead of direct HTTP calls between services?Concept

Direct HTTP: if SMS service is down when teacher marks attendance, the HTTP call fails, teacher gets error, attendance may be lost. Message queue: attendance saved → event enqueued. If SMS service is down, message waits in queue. When SMS service recovers, processes backlog. Decoupling = services can fail independently. Also: queue absorbs bursts (10,000 SMS at 6 PM). No thundering herd on SMS provider.

7
How do you handle multi-tenancy in ZenCampus — 200 schools, data isolation?Design

Options: 1) Row-level isolation: all schools in one DB, every query has WHERE SchoolId = @CurrentSchool. Risk: missing filter = data leak. EF Global Query Filters mitigate this. 2) Schema per school: isolated schemas in same DB server. Better isolation, complex migrations. 3) DB per school in Elastic Pool: full isolation, compliance-friendly, cost-effective with pool. For ZenCampus at scale: DB per school + Elastic Pool. Auth layer extracts SchoolId from JWT — impossible to cross-school.

8
What is the difference between a monolith, SOA, and microservices?Concept

Monolith: one deployable unit. Simple to develop, debug, deploy. Scales as a unit. ZenCampus current state. SOA: large services communicating via enterprise bus. Heavyweight, XML/SOAP. Microservices: small, independently deployable services with their own DB. Each team owns a service. Complex: distributed transactions, service discovery, network overhead. Best fit: microservices when teams > services and independent scaling is needed per domain.

9
How would you implement rate limiting for ZenCampus API?Code

.NET 7+ built-in: builder.Services.AddRateLimiter(opts => { opts.AddFixedWindowLimiter("school", o => { o.Window = TimeSpan.FromMinutes(1); o.PermitLimit = 1000; o.QueueLimit = 0; }); }). app.UseRateLimiter(); Apply [EnableRateLimiting("school")] on controllers. For distributed (multi-instance): use Redis-backed rate limiter (StackExchange.Redis.Extensions.RateLimiter). Rate limit by SchoolId (extracted from JWT) — prevents one school from starving others.

10
Design the ZenCampus fee payment flow end-to-end with high availability in mind.Design

1. Parent initiates payment → API validates student/amount → creates pending FeeTransaction record (idempotency key = UUID). 2. Redirect to payment gateway. 3. Webhook from gateway → API updates FeeTransaction to Completed/Failed (idempotent check — reject duplicate webhooks). 4. Publish FeePaymentCompleted event to Service Bus (fire-and-forget). 5. Worker sends receipt SMS + email. 6. Worker updates parent portal balance. Idempotency prevents double payment. Outbox pattern ensures event published even if API restarts mid-request.
