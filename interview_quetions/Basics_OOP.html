🧱 Class vs Object — The Foundation

🎬
Baasha (1995) — Blueprint and the real thing
Baasha-la Manickam oru normal auto driver-a vazhkiran — but his "class definition" (history, skills, power) is different. Object = antha specific manickam. Class = blueprint. ZenCampus-la Student class = blueprint. Raju, Priya, Basha — they are OBJECTS (instances) of that blueprint. Create pannina time-la each object has its own data (name, admNo) but same structure!
// Class = blueprint (schema) — defined once
public class Student
{
    // Fields — data the object holds
    public int Id { get; set; }
    public string Name { get; set; } = string.Empty;
    public string AdmissionNumber { get; set; } = string.Empty;
    public int ClassId { get; set; }

    // Constructor — runs when object is created
    public Student(string name, string admNo, int classId)
    {
        Name = name;
        AdmissionNumber = admNo;
        ClassId = classId;
    }

    // Method — behaviour of the object
    public string GetDisplayName()
    {
        return $"{Name} ({AdmissionNumber})";
    }
}

// Object = actual instance (created with 'new')
var student1 = new Student("Basha Kumar", "ADM001", 5);
var student2 = new Student("Priya Devi", "ADM002", 6);

// Each object is independent
Console.WriteLine(student1.GetDisplayName()); // Basha Kumar (ADM001)
Console.WriteLine(student2.GetDisplayName()); // Priya Devi (ADM002)
Console.WriteLine(student1.Name == student2.Name); // False

🏛️ 4 Pillars of OOP

1. Encapsulation — Hide Internal Details

Private fields + public properties. Control HOW data is accessed/changed. Like a capsule — protects internal state.

public class FeeAccount
{
    private decimal _balance;  // private — hidden from outside

    public decimal Balance => _balance;  // read-only property

    public void Credit(decimal amount)
    {
        if (amount <= 0) throw new ArgumentException("Amount must be positive");
        _balance += amount;  // only this method can modify _balance
    }

    public void Debit(decimal amount)
    {
        if (amount > _balance) throw new InvalidOperationException("Insufficient balance");
        _balance -= amount;
    }
}
// fee.Balance = 5000; ← ERROR! Can't set directly — encapsulated
// fee.Credit(5000); ← CORRECT — goes through validation

2. Inheritance — Reuse & Extend

Child class gets all properties/methods of parent. "IS-A" relationship. ZenCampus: Teacher IS-A Person. Student IS-A Person.

public class Person         // Parent / Base class
{
    public string Name { get; set; } = "";
    public string Phone { get; set; } = "";
    public virtual string GetRole() => "Person";
}

public class Student : Person   // Child class — inherits from Person
{
    public string AdmissionNumber { get; set; } = "";
    public override string GetRole() => "Student";  // override parent method
}

public class Teacher : Person
{
    public string EmployeeId { get; set; } = "";
    public string Subject { get; set; } = "";
    public override string GetRole() => "Teacher";
}

var s = new Student { Name = "Basha", AdmissionNumber = "ADM001" };
Console.WriteLine(s.Name);    // inherited from Person
Console.WriteLine(s.GetRole()); // "Student" — overridden

3. Polymorphism — Same Method, Different Behaviour

"Many forms." Same method name, different implementation. ZenCampus: SendNotification() — SMS for teacher, email for admin, push for parent.

public abstract class NotificationSender
{
    public abstract void Send(string message, string recipient);
}

public class SmsNotification : NotificationSender
{
    public override void Send(string message, string recipient)
        => Console.WriteLine($"SMS to {recipient}: {message}");
}

public class EmailNotification : NotificationSender
{
    public override void Send(string message, string recipient)
        => Console.WriteLine($"Email to {recipient}: {message}");
}

// Polymorphic usage — same method call, different behavior
List<NotificationSender> senders = [new SmsNotification(), new EmailNotification()];
foreach (var sender in senders)
    sender.Send("Fee due tomorrow", "parent@example.com");
// SMS to parent@example.com: Fee due tomorrow
// Email to parent@example.com: Fee due tomorrow

4. Abstraction — Show Only What's Needed

Hide complexity, expose simple interface. Like a car — you press accelerator, don't need to know engine internals.

public interface IFeeCalculator
{
    decimal Calculate(int studentId, int term);  // abstraction — what it does
}

// Implementation hidden from callers
public class GeneralFeeCalculator : IFeeCalculator
{
    public decimal Calculate(int studentId, int term)
    {
        // Complex logic hidden: base fee + transport + lab fee + etc
        return 5000m + (500m * term);
    }
}

// Caller only knows interface — doesn't care about internals
IFeeCalculator calc = new GeneralFeeCalculator();
var fee = calc.Calculate(1, 2);  // simple call, complex logic hidden

🎯 10 Interview Scenarios

1
What is the difference between a class and an object?Basics
Class = blueprint/template (Student). Object = actual instance of that blueprint (Basha Kumar, ADM001). Class is defined once. Objects (instances) can be created many times. Memory allocated for object, not for class. new keyword creates object from class. Analogy: Class = house plan, Object = actual built house.
2
What is encapsulation? Why use private fields?OOP
Encapsulation: bundle data (fields) and methods in one class AND restrict direct access. Private fields prevent outside code from setting invalid values. FeeAccount._balance is private — only Credit() and Debit() can change it, with validation. Without encapsulation: fee.balance = -9999 would be possible. With: validation is enforced. "Data hiding."
3
What is the difference between method overloading and method overriding?OOP

Overloading: same method name, different parameters, SAME class. Compile-time polymorphism. Example: Calculate(int term) and Calculate(int term, bool isHostel) in same class. Overriding: child class replaces parent's virtual method. Runtime polymorphism. Example: Teacher.GetRole() overrides Person.GetRole(). Overloading = same class, different signatures. Overriding = inheritance, same signature, different class.

4
Can you inherit from multiple classes in C#?OOP

No. C# does not support multiple class inheritance. "Diamond problem" — if both parents have same method, child doesn't know which to use. Solution: multiple interface implementation. class Student : Person, IPayable, IAttendable — one base class + many interfaces. Interfaces define what you CAN DO, base class defines what you ARE.

5
What is an abstract class and when to use it vs interface?OOP

Abstract class: partially implemented, cannot be instantiated directly. Mix of abstract (must override) and concrete (already implemented) methods. Use when: sharing code between related classes (BaseReport with common PDF rendering code). Interface: contract with no implementation (or default in C#8+). Use when: unrelated classes should share capability (IDisposable, IComparable, IFeeCalculator). Rule: "IS-A" relationship → abstract class. "CAN-DO" capability → interface.

6
What is the 'is' and 'as' keyword in C#?Code
Person person = new Student { Name = "Basha" };

// 'is' — check type (returns bool)
if (person is Student)
    Console.WriteLine("Is a student");

// 'is' with pattern matching (C# 7+) — recommended
if (person is Student student)
    Console.WriteLine(student.AdmissionNumber);

// 'as' — try cast, returns null if fails (not exception)
var s = person as Student;  // null if person is not Student
if (s != null) Console.WriteLine(s.Name);
7
What is a constructor and what types are there?OOP

Constructor: special method with same name as class, no return type. Runs when object is created with 'new'. Types: 1) Default constructor — no parameters, provided if you define none. 2) Parameterized — takes arguments (Student(string name, string admNo)). 3) Copy constructor — creates new object from existing. 4) Static constructor — runs once per type, for static initialization. 5) Primary constructor (C# 12): class Student(string Name, string AdmNo).

8
What is the difference between a property and a field?OOP

Field: raw variable in class. public string name; — direct access. Property: controlled access via get/set accessors. public string Name { get; set; }. Properties can: validate on set, compute on get, be read-only (get only), raise events. Fields: typically private. Properties: public interface. EF Core, JSON serializer, data binding — all work with properties, not fields. Always use properties for public members.

9
Explain static class and static members.OOP

Static member: belongs to the CLASS, not an instance. Shared across all objects. Student.TotalCount++ (shared counter). Static class: cannot be instantiated. Only static members. Use for: utility methods (StringHelper.ToTitleCase()), constants, extension methods. ZenCampus: CacheKeys.Student(id) — static method. DateHelper.IsSchoolDay(date) — static utility. Don't store mutable state in static — thread safety issues.

10
What is 'this' keyword in C#?OOP

this: refers to the CURRENT INSTANCE of the class. Uses: 1) Disambiguate when parameter name == field name: this.name = name; 2) Call another constructor: public Student(string name) : this(name, "N/A") { } 3) Pass current object: repository.Save(this) 4) Extension methods: the 'this' parameter defines what type it extends. In static methods: 'this' not available — no instance.
