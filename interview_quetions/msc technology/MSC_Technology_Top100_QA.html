1 C# Fundamentals & OOP
Q1. What are the four pillars of OOP and how does C# implement each?
Encapsulation — properties/access modifiers hide internal state. Abstraction — abstract classes and interfaces expose only what's needed. Inheritance — : BaseClass reuses and extends behavior. Polymorphism — virtual/override and interface implementations let derived types behave differently through the same contract.
Q2. Difference between an abstract class and an interface in modern C#?
An abstract class can hold state (fields), constructors, and a mix of implemented/abstract members; a class can inherit only one. An interface traditionally held only member signatures, but since C# 8 can include default implementations — it still can't hold instance fields, and a class can implement many interfaces. Use abstract classes for a shared base with common state; interfaces for a contract multiple unrelated types can fulfill.
Q3. Difference between IEnumerable, IQueryable, and List<T>?
IEnumerable iterates in-memory with deferred, item-by-item execution (LINQ-to-Objects). IQueryable builds an expression tree and defers execution until enumerated — with EF Core this translates to SQL, so filters composed before .ToList() get pushed to the database. List<T> is a concrete, already-materialized in-memory collection.
Q4. Explain value types vs reference types and where they live in memory.
Value types (struct, int, bool, enum) are copied by value and typically live on the stack (or inline within a containing object). Reference types (class, string, array) live on the managed heap, and variables hold a reference/pointer to that heap location — assigning one variable to another copies the reference, not the data.
Q5. Difference between ref, out, and in parameters?
ref passes by reference and requires the variable to be initialized before the call. out also passes by reference but doesn't require prior initialization — the callee must assign it. in passes by reference as read-only, avoiding a copy for large structs while preventing the callee from modifying it.
Q6. Explain boxing and unboxing, and why it's costly.
Boxing wraps a value type in an object on the heap (e.g., assigning an int to an object); unboxing extracts it back. Each boxing operation allocates heap memory and adds GC pressure, so it's avoided in hot paths — generics (List<int> instead of ArrayList) sidestep it entirely.
Q7. Difference between const, readonly, and static readonly?
const is a compile-time constant, inlined at compile time, and must be a primitive/string. readonly is set at runtime, only in the constructor, and can differ per instance. static readonly is the same but shared across all instances — useful when the value is computed at startup (e.g., from config) rather than known at compile time.
Q8. Explain async/await and how it differs from Task.Run.
async/await frees the calling thread while waiting on genuinely asynchronous I/O (DB calls, HTTP requests) — no extra thread is consumed while waiting. Task.Run queues CPU-bound work onto a thread-pool thread; using it to wrap I/O-bound calls just burns a thread pointlessly. Rule of thumb: await I/O directly, use Task.Run only for CPU-bound work you want off the current thread.
Q9. What are nullable reference types (C# 8+) and how do they help?
With <Nullable>enable</Nullable>, the compiler tracks whether a reference type is expected to be non-null and flags places where a possibly-null value is dereferenced without a check — turning a class of runtime NullReferenceExceptions into compile-time warnings.
Q10. Explain extension methods and when you'd use one.
An extension method is a static method in a static class, with this on the first parameter, that appears to "extend" an existing type without modifying it — LINQ (.Where(), .Select()) is built entirely this way. Use them to add reusable helper behavior to types you don't own (e.g., a string.ToSlug() helper) instead of a static utility class with awkward call syntax.
2 Design Patterns & Architecture
Q11. Which design patterns have you used in production, and why?
Repository/Unit of Work for data-access abstraction (often debated as redundant over EF Core, which is already a UoW). Factory for creating strategy implementations at runtime. Strategy for swappable behavior (e.g., payment or notification providers). Singleton via DI for shared config/cache. Decorator for cross-cutting concerns like logging/caching around a service. Mediator (MediatR) for CQRS-style request/handler separation.
Q12. Explain SOLID with one practical example.
Single Responsibility — a class handling both order calculation and email sending should be split so each has one reason to change. Open/Closed — new discount types should be added via a new IDiscountStrategy implementation, not by editing an existing switch. Liskov Substitution — a derived class shouldn't break behavior the base type promises. Interface Segregation — prefer several small interfaces over one bloated one. Dependency Inversion — depend on IEmailSender, not a concrete SmtpEmailSender, so it can be mocked/swapped.
Q13. What is Dependency Injection, and what are the three lifetimes in ASP.NET Core?
DI supplies a class's dependencies from outside rather than having it construct them, enabling loose coupling and testability. Transient creates a new instance every time it's requested; Scoped creates one instance per HTTP request; Singleton creates one instance for the app's lifetime. Common bug: injecting a Scoped service like DbContext into a Singleton — it captures the first request's context and reuses it forever, causing exceptions or cross-request data corruption under load.
Q14. Is the Repository pattern over EF Core redundant?
DbContext already implements Unit of Work and DbSet<T> already behaves like a repository, so a thin wrapper can just add indirection without real benefit. A Repository layer earns its keep when you need to hide EF-specific query composition behind a simpler, mockable contract, support swapping data sources, or centralize complex query logic reused across services — not as a blanket rule for every CRUD entity.
Q15. Explain CQRS and when you'd introduce MediatR.
CQRS separates the models used for writes (Commands) from those used for reads (Queries), allowing each to be optimized independently — reads can use lean projections while writes enforce full business rules. MediatR is introduced when controllers are getting bloated with orchestration logic; each request becomes a small, independently testable handler, and cross-cutting concerns (validation, logging) can be added via pipeline behaviors.
3 ASP.NET Core MVC
Q16. Explain the ASP.NET Core middleware pipeline and ordering.
Middleware executes in registration order on the way in and reverse order on the way out, forming a chain via next(). Typical order: exception handling → HTTPS redirection → static files → routing → CORS → authentication → authorization → custom business middleware → endpoints. Placing authorization before authentication, for example, would break every request.
Q17. Difference between MVC and Razor Pages?
MVC separates Controller (logic), Model (data), and View (markup) as distinct pieces, which suits larger apps with shared logic across many views. Razor Pages colocates a page's markup (.cshtml) with its page-model class (.cshtml.cs) in a page-focused, more self-contained structure — often faster to build simple CRUD/form-driven pages.
Q18. How does model binding work in ASP.NET Core MVC?
The framework matches incoming request data (route values, query string, form fields, JSON body, headers) to action method parameters or a bound model's properties by name, using value providers in a defined precedence order. Explicit attributes ([FromBody], [FromQuery], [FromRoute]) override the default inference when it would otherwise be ambiguous.
Q19. What does [ApiController] add over a plain Controller?
Automatic 400 responses on invalid ModelState without manual checks, automatic binding-source inference, automatic ProblemDetails error responses, and a requirement to use attribute routing — it's meant specifically for API controllers, not view-rendering ones.
Q20. ViewModels vs Models vs DTOs — why not pass EF entities straight to a view?
Passing entities to a view over-exposes internal data (and can accidentally trigger lazy-loading queries during rendering) and couples the UI to the database schema. A ViewModel/DTO shapes exactly what a view or API consumer needs, decoupling the persistence model from the presentation contract and avoiding over-posting vulnerabilities on the way back in.
Q21. How do you implement validation beyond Data Annotations?
FluentValidation for complex, conditional, or cross-field rules with better testability and separation from the model; custom ValidationAttribute classes for reusable simple rules; IValidatableObject for validation that depends on multiple properties together.
Q22. TempData vs ViewBag vs ViewData — lifetimes?
ViewData/ViewBag (same store, different access syntax) live only for the current request and are cleared afterward. TempData persists across one redirect (backed by session or cookies) and is typically used for post-redirect-get flash messages, like a "saved successfully" banner shown after a redirect.
Q23. Conventional routing vs attribute routing — when to use each?
Conventional routing (defined once in Program.cs as a URL pattern) suits traditional MVC apps with predictable, uniform URL structures. Attribute routing ([Route], [HttpGet] on actions) gives per-action control and is effectively required for Web APIs, where URL shapes vary widely and need to be explicit and versionable.
Q24. How do you implement global exception handling in MVC?
Exception-handling middleware (or an IExceptionHandler in .NET 8) catches unhandled exceptions app-wide, logs them, and returns a consistent error page or ProblemDetails response. Action/exception filters are an alternative scoped to MVC pipeline execution specifically, useful when you need access to MVC context (like the action being executed) that middleware doesn't have.
Q25. What is the Options pattern (IOptions<T>) and why use it?
It binds a strongly-typed class to a configuration section, injected via DI, instead of scattering IConfiguration["Section:Key"] magic-string lookups throughout the code. It also supports IOptionsSnapshot for scoped/reloadable config and IOptionsMonitor for live-reloading singleton consumers.
4 Web API & REST
Q26. What makes an API RESTful?
Statelessness (each request carries all context needed), a uniform resource-based interface (nouns as URLs, verbs as HTTP methods), client-server separation, cacheability, and a layered system. In practice this means GET /orders/5 not GET /getOrder?id=5, and using status codes and HTTP verbs meaningfully instead of always returning 200 with a body-level error flag.
Q27. How do you version a Web API — trade-offs of URL, header, query-string?
URL versioning (/api/v1/orders) is simplest, most cache- and Swagger-friendly, but pollutes the route. Header versioning keeps URLs clean but is harder to discover/test/debug. Query-string versioning is easy to add but easy to omit by accident. Most public APIs default to URL versioning; the Asp.Versioning package supports all three in ASP.NET Core.
Q28. Explain idempotency — which HTTP verbs are idempotent and why does it matter?
GET, PUT, DELETE are idempotent — repeating the same request produces the same end state. POST is not. This matters for retries: a client (or gateway) can safely retry a timed-out PUT/DELETE without risk, but retrying a POST could create duplicate records unless you add an idempotency key.
Q29. How do you implement pagination, filtering, and sorting in a Web API?
Accept query parameters (?page=2&pageSize=20&sort=-createdAt&status=active), apply them as IQueryable filters before materializing, and return metadata (total count, page, pageSize) alongside the data — either in the body or in headers like X-Total-Count — so clients can build pagination controls without a second round-trip.
Q30. Difference between 401 and 403 — when do you return each?
401 Unauthorized means the request lacks valid authentication (missing/expired/invalid token) — "who are you?". 403 Forbidden means the caller is authenticated but not allowed to perform this action — "I know who you are, but no."
Q31. How do you implement rate limiting in ASP.NET Core?
.NET 7+ ships a built-in Microsoft.AspNetCore.RateLimiting middleware supporting fixed-window, sliding-window, token-bucket, and concurrency limiter algorithms, configured per endpoint or globally. In a gateway-fronted architecture, rate limiting is often enforced at Azure APIM instead, so every backend service doesn't reimplement it.
Q32. Explain content negotiation in ASP.NET Core Web API.
The framework selects the response format (JSON by default, XML if configured) based on the client's Accept header, via output formatters. If no formatter matches the requested type, it falls back to the default or returns 406 Not Acceptable if strict negotiation is enabled.
Q33. How would you design an endpoint for a long-running operation?
Return 202 Accepted immediately with a Location header pointing to a status resource, queue the actual work (e.g., to a background job or message queue), and let the client poll the status endpoint (or use a webhook/SignalR notification) instead of holding the HTTP connection open.
Q34. What is HATEOAS?
Hypermedia As The Engine Of Application State — responses include links describing valid next actions (e.g., an order response includes a "cancel" link only if it's still cancellable), so clients discover available operations from the API itself rather than hard-coding URLs. It's the truest form of REST maturity but is rarely fully implemented in practice due to added complexity.
Q35. How do you handle API error responses consistently?
Use the ProblemDetails (RFC 7807) standard — a consistent JSON shape with type, title, status, detail, and a trace/correlation ID — produced by global exception-handling middleware so every error, regardless of where it's thrown, looks the same to consumers.
Q36. Explain CORS — why does the browser block a request?
CORS is a browser-enforced security policy preventing a page from one origin (scheme+host+port) from calling an API on a different origin unless the server explicitly allows it via response headers like Access-Control-Allow-Origin. In ASP.NET Core it's configured with AddCors()/UseCors(), scoped to specific origins rather than a wildcard in production.
Q37. How do you prevent over-posting/mass assignment?
Bind requests to a dedicated input DTO containing only the fields a client is allowed to set, rather than binding directly to the EF entity — so a client can't sneak in fields like IsAdmin or Balance that weren't meant to be user-editable.
5 Entity Framework Core & LINQ
Q38. What causes the N+1 query problem in EF Core, and how do you fix it?
Accessing a lazily-loaded navigation property inside a loop triggers a separate query per iteration instead of one batched query. Fix with .Include()/.ThenInclude() for eager loading, or better, a .Select() projection that fetches exactly the needed shape in a single query.
Q39. When and why use AsNoTracking()?
It disables the change tracker for a query, skipping the snapshot EF normally keeps for detecting changes — reducing memory and CPU overhead. Use it for read-only queries (GET endpoints, reports) where you'll never call SaveChanges() on the returned entities.
Q40. Eager loading vs lazy loading vs explicit loading?
Eager loading (.Include()) fetches related data in the same query up front. Lazy loading fetches related data automatically the first time a navigation property is accessed, one query at a time — convenient but the classic source of N+1. Explicit loading calls .Entry(x).Collection(...).Load() on demand, giving control over exactly when a related query fires.
Q41. How does EF Core handle optimistic concurrency?
A [Timestamp]/RowVersion column is included in the UPDATE's WHERE clause; if another transaction changed the row first, zero rows match and EF Core throws DbUpdateConcurrencyException, which the app catches and resolves (client-wins, store-wins, or prompts the user to merge).
Q42. .Select() projection vs .Include() — when to prefer projection?
.Include() loads full related entities, which is convenient when you need to modify and save them back. A .Select() projection into a DTO fetches only the columns actually needed, translates to a leaner SQL query, and avoids loading unnecessary data — preferred for read-only API responses.
Q43. How do you manage EF Core migrations safely in a team and in production?
Each developer generates migrations against a shared model; conflicts are resolved by reviewing the generated migration diff, not blindly regenerating. In production, migrations are applied via a controlled CI/CD pipeline step (not Database.Migrate() at app startup), with review for destructive operations like column drops before deployment, and a rollback plan.
Q44. How does IQueryable translate LINQ to SQL, and where do bugs typically happen?
EF Core translates the expression tree into SQL only when it can — calling a non-translatable method (a custom C# function, some string methods) mid-query throws or silently forces client-side evaluation, pulling more data than intended before filtering in memory. Bugs happen when a filter is written after an accidental .ToList()/.AsEnumerable() call, silently switching to slow in-memory filtering.
Q45. Explain deferred execution with a common bug it causes.
A LINQ query isn't executed when defined, only when enumerated (foreach, .ToList(), etc.). Common bug: building a query inside a loop that captures a loop variable by reference, then enumerating it later — every deferred execution sees the loop variable's final value instead of the value at definition time.
Q46. First() vs FirstOrDefault() vs Single() vs SingleOrDefault()?
First() throws if empty, returns the first match. FirstOrDefault() returns default (null/0) instead of throwing if empty. Single() throws if there isn't exactly one match — use it when exactly-one is a business invariant you want enforced. SingleOrDefault() returns default if empty but still throws if more than one match exists.
Q47. How do you perform bulk updates/deletes efficiently in EF Core?
EF Core 7+ added ExecuteUpdate()/ExecuteDelete(), which translate directly to a single UPDATE/DELETE SQL statement without loading entities into memory or the change tracker — far more efficient than loading rows, mutating them, and calling SaveChanges() for bulk operations.
Q48. What is a compiled query in EF Core?
EF.CompileQuery() pre-compiles a LINQ query's translation to SQL once, skipping re-translation on every call — useful for hot-path queries executed very frequently with different parameter values, though EF Core's own query caching already covers most cases without needing this manually.
Q49. How do you handle transactions spanning multiple DbContext operations?
A single SaveChanges() call is already atomic across all tracked changes. For operations spanning multiple SaveChanges() calls or multiple contexts, wrap them in an explicit transaction via context.Database.BeginTransaction() (or IExecutionStrategy-aware retry logic when using a resiliency policy) and commit/rollback manually.
Q50. Performance considerations for GroupBy in LINQ-to-Entities?
Simple GroupBy with aggregate functions (Count(), Sum()) translates well to SQL GROUP BY. More complex groupings (grouping by a computed client-side expression, or projecting the whole group) may force EF Core to pull data client-side first, which is far slower on large tables — worth checking the generated SQL when performance matters.
6 SQL Server
Q51. Clustered vs non-clustered index?
A clustered index physically orders the table's data rows by the index key — a table can have only one. A non-clustered index is a separate structure holding key values and pointers back to the data rows — a table can have many, useful for speeding up lookups on columns other than the clustering key.
Q52. What is SQL injection and how do parameterized queries prevent it?
SQL injection occurs when untrusted input is concatenated directly into a SQL string, letting an attacker alter the query's structure (e.g., ' OR 1=1 --). Parameterized queries send the input as a separate data value, not executable SQL text, so it can never change the query's shape — EF Core and Dapper's @param syntax do this by default.
Q53. Explain ACID properties.
Atomicity — a transaction's operations all succeed or all roll back together. Consistency — a transaction moves the database from one valid state to another. Isolation — concurrent transactions don't see each other's uncommitted changes (governed by isolation level). Durability — once committed, changes survive a crash.
Q54. WHERE vs HAVING?
WHERE filters individual rows before grouping/aggregation happens. HAVING filters groups after a GROUP BY has produced aggregate results — you can't filter on an aggregate like COUNT(*) in a WHERE clause because aggregation hasn't happened yet at that point.
Q55. Explain JOIN types with a use case.
INNER JOIN returns only matching rows in both tables (students with an enrollment). LEFT JOIN returns all rows from the left table plus matches from the right, nulls where none exist (all students, even those with no enrollment). RIGHT JOIN is the mirror of LEFT. FULL JOIN returns all rows from both sides. CROSS JOIN produces a Cartesian product, rarely used outside generating combinations.
Q56. How do you diagnose and fix a slow query?
Look at the actual execution plan (not estimated) for table scans where an index seek is expected, high-cost operators, or missing index suggestions. Common fixes: add a covering index on the filter/join/sort columns, rewrite a non-sargable predicate (e.g., a function applied to a column in WHERE), or update outdated statistics.
Q57. What is a deadlock in SQL Server, and how do you prevent it?
Two transactions each hold a lock the other needs, so neither can proceed — SQL Server detects this and kills one as the deadlock victim. Prevent by accessing tables/rows in a consistent order across all code paths, keeping transactions short, and using an appropriate isolation level (or READ COMMITTED SNAPSHOT) to reduce lock contention.
Q58. DELETE vs TRUNCATE vs DROP?
DELETE removes rows individually, is logged per-row, can be filtered with WHERE, and fires triggers. TRUNCATE deallocates all rows at once, is minimally logged and much faster, but can't be filtered and resets identity columns. DROP removes the entire table structure and data permanently.
7 JavaScript / jQuery
Q59. var vs let vs const — scoping differences?
var is function-scoped and hoisted with an undefined initial value, which causes classic loop-variable-capture bugs. let is block-scoped and not accessible before its declaration line (temporal dead zone). const is block-scoped like let but can't be reassigned after initialization (though object/array contents it references can still be mutated).
Q60. What is event delegation and why is it useful?
Instead of binding a handler to every element, you bind one handler to a stable parent and use $(parent).on('click', '.child-selector', handler) (or check event.target in vanilla JS) — the handler fires for matching descendants even ones added to the DOM after the binding, which direct binding would miss entirely.
Q61. == vs ===?
== compares after type coercion ('5' == 5 is true), which can produce surprising results. === compares value and type without coercion ('5' === 5 is false) — the safer default for almost all comparisons.
Q62. What is a Promise, and how does async/await help?
A Promise represents a value that will resolve or reject in the future, replacing nested callbacks with chainable .then()/.catch(). async/await is syntactic sugar over Promises that lets asynchronous code read top-to-bottom like synchronous code, avoiding deeply nested .then() chains ("callback hell").
Q63. fetch vs jQuery's $.ajax() — trade-offs?
fetch is a native browser API, Promise-based, and doesn't require jQuery as a dependency, but doesn't reject on HTTP error status codes (only network failures) and needs manual JSON parsing/error handling. $.ajax() handles more of this out of the box (auto error rejection, easy abort, broader legacy browser support) but requires jQuery and is being phased out of newer codebases in favor of fetch/axios.
Q64. Explain closures with a practical example.
A closure is a function that retains access to variables from its enclosing scope even after that scope has returned — commonly used to create private state, like a counter function returning an increment closure that keeps its own count without exposing a global variable.
Q65. $(document).ready() vs script at the bottom of the body?
$(document).ready() waits for the full DOM to be parsed before running, regardless of where the script tag is placed in the head. Placing a plain <script> at the end of <body> achieves a similar effect without jQuery, by ensuring the DOM above it is already parsed by the time it runs, and also avoids blocking page rendering.
Q66. How do you debounce a search input?
Wrap the API call in a debounce function that resets a timer on every keystroke and only fires after the user pauses typing for a set delay (e.g., 300ms) — implemented with setTimeout/clearTimeout, or via a utility like Lodash's _.debounce(), to avoid firing a request on every single character.
8 Security — OAuth2 / JWT / API Gateway
Q67. Walk through the JWT authentication flow in an ASP.NET Core Web API.
The client authenticates against an identity provider (Azure AD/Entra ID, IdentityServer, or a custom STS) and receives a signed JWT. The API validates the token via AddJwtBearer() middleware — checking issuer, audience, signature (against the IdP's public key/JWKS), and expiry — then populates HttpContext.User claims for downstream [Authorize] checks.
Q68. What's inside a JWT, and why can the payload be read but not modified?
A JWT has three Base64URL-encoded parts — header (algorithm/type), payload (claims), and signature. The payload is only encoded, not encrypted, so anyone can decode and read it, but the signature (computed over header+payload with a secret/private key) lets the server detect any tampering — a modified payload won't match the original signature.
Q69. Authentication vs Authorization; role-based vs policy-based?
Authentication verifies who you are; Authorization decides what you're allowed to do. Role-based authorization uses [Authorize(Roles = "Admin")] for simple role checks. Policy-based authorization (AddAuthorization(options => options.AddPolicy(...))) supports richer, composable requirements — claims-based checks, resource-based checks (does this user own this specific record) — and is more testable in isolation.
Q70. Which OAuth2 grant type fits which scenario?
Authorization Code (with PKCE) for server-side web apps and SPAs — the user authenticates via redirect, and the app exchanges a code for tokens without exposing credentials. Client Credentials for machine-to-machine/service-to-service calls with no user involved. Implicit grant is deprecated in favor of Authorization Code + PKCE even for SPAs.
Q71. What role does an API Gateway play (e.g., Azure APIM)?
It centralizes cross-cutting concerns — token validation, rate limiting, request routing/aggregation, response caching, and logging — so individual backend services don't each reimplement them. Azure APIM specifically adds policy-based request/response transformation, subscription keys, and a developer portal for API documentation and discovery.
Q72. How do you implement refresh tokens securely?
Issue a short-lived access token (minutes) plus a long-lived refresh token, stored securely (HttpOnly, Secure cookie — not localStorage, which is vulnerable to XSS theft). The refresh token is exchanged for a new access token via a dedicated endpoint, and should be rotated (invalidated and reissued) on each use to limit the damage if one is ever leaked.
Q73. How do you securely store secrets in an Azure-hosted app?
Azure Key Vault, accessed via Managed Identity so no credential is stored anywhere in code or config. appsettings.json holds only non-sensitive config, and secrets are never committed to source control; locally, dotnet user-secrets keeps them out of the repo during development.
Q74. What is CSRF and how does ASP.NET Core protect against it?
Cross-Site Request Forgery tricks an authenticated user's browser into submitting an unwanted request to your site (relying on cookies being sent automatically). ASP.NET Core MVC guards against it with anti-forgery tokens (@Html.AntiForgeryToken() + [ValidateAntiForgeryToken]) tied to the user's session, which an attacker's page can't forge. Token-based APIs (JWT in an Authorization header rather than a cookie) are inherently less exposed since the browser won't attach the header automatically.
Q75. Explain "Broken Access Control" (OWASP A01) with a multi-tenant example.
A user authenticated as Parent A can request GET /api/students/102 and receive Parent B's child's data because the endpoint only checks "is this caller authenticated," not "does this caller own this specific resource." Fix: enforce resource-level (ownership) authorization on every request, not just role-level checks.
Q76. How do you prevent XSS in an MVC application?
Razor HTML-encodes output by default, which covers most cases — the risk appears when using @Html.Raw() on untrusted input, or when rendering user content into JavaScript contexts without proper encoding. Always validate/sanitize input at the boundary and encode output for the specific context (HTML, JS, URL) it's rendered into, and set a Content-Security-Policy header as defense in depth.
9 Azure Cloud
Q77. What Azure services would you use to host a production .NET Web API?
Azure App Service (or Container Apps/AKS for containerized workloads) for hosting, Azure SQL Database for data, Key Vault for secrets, Application Insights for telemetry/monitoring, Blob Storage for files, Service Bus/Event Grid for async messaging, and Entra ID for identity.
Q78. What is a deployment slot and how does slot swap enable zero-downtime deployment?
A slot is a separate live instance of the app (e.g., "staging") sharing the same App Service Plan but with its own hostname. You deploy and warm up the staging slot, run smoke tests against it, then swap — traffic switches to the new version essentially instantly, and you can swap back immediately if something's wrong.
Q79. How do Key Vault and Managed Identity work together?
Managed Identity gives the App Service/Function an automatically-managed Azure AD identity with no credentials to store. That identity is granted access-policy permissions on the Key Vault, so the app can retrieve secrets at runtime via the SDK without ever holding a connection string or client secret in config.
Q80. What is Application Insights used for?
Application Performance Monitoring — request/dependency telemetry, exception tracking, custom event/metric logging, and live metrics. In production you'd monitor request latency and failure rate, dependency call duration (DB/external APIs), exception volume/trends, and set alerts on thresholds like error rate spikes.
Q81. App Service vs Azure Functions vs Container Apps/AKS — when to choose each?
App Service for a standard always-on web app/API with predictable load. Azure Functions for event-driven, short-lived, or bursty workloads billed per execution (e.g., processing a queue message). Container Apps/AKS when you need containerized microservices, fine-grained orchestration, or portability across environments — AKS specifically when you need full Kubernetes control.
Q82. How do you manage config differences across Dev/QA/Prod?
Layered appsettings.{Environment}.json files combined with environment variables or App Service application settings injected per deployment slot/environment, plus Azure App Configuration for centrally-managed feature flags and shared settings — never hard-coding environment-specific values in code.
Q83. What is Azure Service Bus, and when would you use it over a direct API call?
A managed message broker supporting queues and pub/sub topics. Use it when the caller shouldn't block waiting for the receiver to process (e.g., triggering a report generation job), when you need guaranteed delivery/retry semantics, or to decouple services so one going down doesn't cascade-fail the caller.
Q84. Horizontal vs vertical scaling in App Service, and auto-scale rules?
Vertical scaling (scale up) increases the resources of a single instance (more CPU/RAM via a higher App Service Plan tier). Horizontal scaling (scale out) adds more instances behind the load balancer. Auto-scale rules trigger scale-out/in automatically based on metrics like CPU percentage, memory, or request queue length, within a configured min/max instance count.
Q85. How do you troubleshoot a production issue using Application Insights?
Start with the Failures blade to find the failing request/exception and its rate, drill into the End-to-End Transaction Details to see the full dependency chain (DB call, downstream API) for a specific failed request, correlate by operation/correlation ID across services, and cross-check Log Analytics (KQL queries) for custom logged events around the same timestamp.
Q86. What is Entra ID (Azure AD) and how does it integrate with JWT-based API auth?
Entra ID is Microsoft's cloud identity provider — it issues JWTs after authenticating users or applications (via OAuth2/OpenID Connect flows). An ASP.NET Core API configures AddMicrosoftIdentityWebApi()/JWT bearer options pointing at Entra ID's issuer and signing keys, so incoming tokens issued by Entra ID are validated without the API managing its own user store.
10 Azure DevOps CI/CD
Q87. Describe a CI/CD pipeline you've built in Azure DevOps for a .NET Core app.
A YAML pipeline with a build stage (restore, build, run unit tests, publish artifacts, optionally a SonarQube quality gate) followed by release stages that deploy automatically to Dev, then require manual approval gates to promote to QA and Prod, with post-deploy smoke tests confirming the new version is healthy.
Q88. Build pipeline vs release pipeline — classic vs YAML?
The build pipeline compiles, tests, and produces deployable artifacts. The release pipeline takes those artifacts and deploys them across environments with approvals/gates in between. Classic pipelines are configured through the UI; YAML pipelines define everything as code in the repo, versioned alongside the app — YAML is the modern default since it's reviewable and reproducible.
Q89. How do you manage secrets/variables securely in a pipeline?
Variable groups linked to Azure Key Vault, marking sensitive variables as secret so they're masked in logs, and using service connections backed by a least-privilege service principal rather than embedding credentials directly in YAML.
Q90. What are approval gates, and how do you use them for Prod?
Approval gates pause a deployment stage until a designated approver signs off, typically placed before Prod (and sometimes QA) to prevent automatic, unreviewed deployment to a sensitive environment. Automated gates can also check external conditions (e.g., no active incident, work item state) before allowing the stage to proceed.
Q91. How do you integrate automated tests and SonarQube into a pipeline?
Add a test task (dotnet test with coverage collection) in the build stage, publish test results and coverage reports, then add a SonarQube scan task before/after build that fails the pipeline if the configured quality gate (coverage threshold, bug/code-smell/vulnerability count) isn't met.
Q92. How do you run EF Core migrations safely as part of deployment?
Generate a SQL script from the migration (dotnet ef migrations script) rather than calling Database.Migrate() at app startup, review it for destructive changes, and run it as an explicit, auditable pipeline step against the target database before or during deployment — decoupled from application startup so migration failures don't take down a running app.
11 Swagger / OpenAPI
Q93. How do you document a Web API using Swagger/OpenAPI in ASP.NET Core?
Swashbuckle.AspNetCore auto-generates the OpenAPI spec from controller actions, XML doc comments, and attributes like [ProducesResponseType]. Enabling XML documentation output in the project file lets summary comments on actions and DTOs flow directly into the generated Swagger UI.
Q94. How do you configure Swagger UI to test JWT-secured endpoints?
Add a security definition for the Bearer scheme in AddSwaggerGen() and apply it as a security requirement — this makes an "Authorize" button appear in Swagger UI where a tester pastes a JWT, which is then automatically attached as the Authorization header on every subsequent "Try it out" request.
Q95. Swagger UI vs the underlying OpenAPI spec — what's the difference?
The OpenAPI spec (swagger.json/swagger.yaml) is the machine-readable contract describing every endpoint, schema, and response — usable to generate client SDKs, run contract tests, or feed API gateways. Swagger UI is just one human-friendly renderer of that spec; the spec itself is the actual artifact of value.
