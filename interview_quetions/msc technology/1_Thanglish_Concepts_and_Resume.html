A1 C# & OOP Basics
OOP — Object Oriented Programming (4 Pillars)
Namba code ah "objects" vachu design pannurom nu artham. 4 pillars: Encapsulation — data ah private ah vachu, property mூlama மட்டும் access panna vidurathu (namba ATM machine madhiri — pinni show pannama money kudukkum). Abstraction — complex logic ah hide panni, simple interface mattum kaatarathu. Inheritance — oru class vera class oda properties/methods ah reuse pannuvathu (Employee class-a base vachu Manager class create panradhu). Polymorphism — same method name, but different classes la different ah behave pannum (Animal.Speak() — Dog sonna "Bark", Cat sonna "Meow").
🗣️ Interview la eppadi sollanum"OOP has four pillars — encapsulation for hiding internal data, abstraction for exposing only what's needed, inheritance for code reuse, and polymorphism for letting the same method behave differently across derived classes. I use this daily — for example, my IPaymentService interface has different implementations for online and offline payments."
Interface vs Abstract Class
Interface na — "enna panna vendum" nu சொல்லும், "eppadi panna vendum" nu solladhu. Abstract class na — konjam logic already irukkum, konjam class fill pannanum. Real life example: Interface = job description ("must do coding"). Abstract class = half-done template (structure already irukku, konjam customize pannanum).
🗣️ Answer Script"An interface only defines a contract — method signatures, no implementation, and a class can implement multiple interfaces. An abstract class can have both implemented and abstract members, plus fields and constructors, but a class can inherit only one."
Async / Await
Namba API call pannumbodhu, DB query eduthukkumbodhu, adhu complete aaganum nu wait pannikittu irukka koodathu — andha thread ah block pannama vera velai panna vidanum. async/await vechu, "இது நேரம் ஆகும் velai, wait pannu, aana thread ah free ah vidu" nu sollurom. Idhu thaan namba 1000+ concurrent users ah handle panna help pannudhu (un resume la irukka point).
🗣️ Answer Script"Async/await lets the thread stay free while waiting on I/O like a database call or an external API, instead of blocking. This is exactly how I supported 1,000+ concurrent users in my ERP project without threads getting exhausted."
Value Type vs Reference Type
Value type (int, bool, struct) — data direct ah stack la irukkum, copy pannumbodhu original value copy aagum. Reference type (class, string, array) — heap la data irukkum, variable oru "address" mattum vachirukkum. So oru reference type variable ah vera variable ku assign panna, rendu variables um same object ah point pannum.
🗣️ Answer Script"Value types are copied by value and live on the stack; reference types live on the heap and variables just hold a reference — so assigning one reference variable to another means both point to the same object in memory."
Exception Handling — Try/Catch/Finally
Namba resume la "centralized logging & exception-handling middleware" nu irukku. Idhu na — ovvoru controller la separate ah try-catch pota koodathu, oru middleware level la globally catch pannanum, so error consistent ah log aagum, incident resolution fast ah aagum.
🗣️ Answer Script"Instead of scattering try-catch in every controller, I built a global exception-handling middleware that catches unhandled exceptions app-wide, logs them with a correlation ID, and returns a consistent error response — this is what cut our incident resolution time by 35%."
LINQ
LINQ na Language Integrated Query — C# code kulla directly SQL-madhiri query eluthalam. Example: students.Where(s => s.Age > 18).Select(s => s.Name). Idhu database, list, array — edhu mela venalum work aagum.
🗣️ Answer Script"LINQ lets me write query logic directly in C# instead of raw SQL strings — filtering, projecting, grouping — and with EF Core it translates to SQL automatically, so the filtering happens in the database, not in memory."
A2 ASP.NET Core MVC & Web API
Middleware Pipeline
Request oru API ku varumbodhu, adhu oru "pipeline" mூlama pogum — first authentication check aagum, apparam authorization, apparam routing, appram namma actual controller code. Ovvoru middleware "next" nu next step ku pass pannum. Order thavaru na — example authorization ah authentication ku munnadi vecha, ellame fail aagum.
🗣️ Answer Script"Middleware runs in a defined pipeline order — exception handling, HTTPS redirect, routing, authentication, authorization, then the endpoint. I've written custom middleware for centralized logging and JWT validation in my ERP project."
Web API vs MVC Controller
MVC Controller — HTML view render pannum (webpage kaatta). Web API Controller — data mattum kudukkum (JSON/XML), no HTML. [ApiController] attribute podra bodhu automatic ah model validation, automatic 400 error response ellam free ah kidaikkum.
🗣️ Answer Script"MVC controllers return views (HTML) while Web API controllers return data, typically JSON. Marking a controller with [ApiController] gives automatic model-state validation and consistent error responses without writing that boilerplate myself."
REST API Principles
REST na — URL la resource (noun) irukkanum, HTTP verb (GET/POST/PUT/DELETE) action ah sollanum. Example: GET /students/5 — student 5-oda details eduka. POST /students — pudhu student create panna. Statusless nu சொல்றது na — ovvoru request tha independent ah irukkanum, server oru request-a next request ku memory la vecha koodathu.
🗣️ Answer Script"A RESTful API uses resource-based URLs and HTTP verbs meaningfully — GET to read, POST to create, PUT to update, DELETE to remove — and each request is stateless, carrying all the context it needs, which is how I designed the SmartSchool-Web APIs."
Model Binding & Validation
Client anuppura JSON data ah, C# object ah automatic ah convert panradhu thaan Model Binding. [FromBody], [FromQuery], [FromRoute] — data engirundhu varudhu nu sollradhukku use pannuvom. Validation — Data Annotations ([Required], [MaxLength]) mூlama basic check, FluentValidation mூlama complex rules.
🗣️ Answer Script"Model binding auto-maps incoming JSON to my request DTOs. For validation I use Data Annotations for simple rules and FluentValidation when the rule depends on multiple fields together."
Dependency Injection (DI)
Class ku venum na dependencies (example: IEmailService) — adha class thaane create pannama, outside irundhu "inject" pannuvom. Idhu vachu unit testing easy aagum (fake/mock service kudukkalam), code loosely coupled aagum. 3 lifetimes: Transient (every time new object), Scoped (per request one object), Singleton (whole app life one object).
🗣️ Answer Script"Dependency Injection means a class receives its dependencies from outside instead of creating them itself — I register services with Transient, Scoped, or Singleton lifetime in Program.cs depending on whether they should be per-call, per-request, or shared app-wide."
Swagger / OpenAPI
Namba API endpoints ellam automatic ah document pannitu, browser la testing pannalam nu oru tool. Swashbuckle.AspNetCore package use pannurom. JWT token vecha secured endpoint ah test panna, Swagger UI la "Authorize" button click panni token paste pannalam.
🗣️ Answer Script"I use Swashbuckle to auto-generate Swagger docs for every API, and configure the Bearer security scheme so I can test JWT-protected endpoints directly from Swagger UI without needing Postman."
A3 Entity Framework Core & Dapper
EF Core vs Dapper — Ithu rendayum ethuku use pannurom?
EF Core na — full ORM, entities ah automatic ah track pannum, migrations kudukkum, aana konjam slow (heavy). Dapper na — "micro ORM", raw SQL query eludhanum, aana speed miga fast. Un resume la rendu um irukku — heavy business logic ku EF Core, high-performance/high-volume query ku Dapper use pannirukeenga nu sollalam.
🗣️ Answer Script"I used EF Core for standard CRUD and business logic where change tracking and migrations help productivity, and Dapper for performance-critical, high-volume queries where I need raw SQL speed — that combination is what got us the 40% query time reduction."
N+1 Query Problem
Oru loop kulla, ovvoru iteration kum separate ah DB query pona — adha N+1 problem nu sollுவோம். Example: 100 students ku, ovvoru student oda marks ah loop la eduthaa — 1 query (students) + 100 query (marks) = 101 queries! Fix: .Include() or .Select() use panni ஒரே query la ellame eduthukanum.
🗣️ Answer Script"N+1 happens when you lazily load a related entity inside a loop — one query per iteration. I avoid it with eager loading via Include(), or better, a projection with Select() that fetches exactly what's needed in a single query."
Query Optimization — Indexing & Stored Procedures
Un resume la "cutting query execution time by 40%" nu irukku — idhu eppadi achieve pannirukeenga nu kekka vaaipu irukku. Answer: frequently filter/search panra columns ku index podradhu (example: StudentId, Email), complex reports ku stored procedure use panradhu (application layer la logic vaikkama, DB level la optimize pannirathu), and unnecessary columns ah fetch pannama .Select() vachu projection pannurathu.
🗣️ Answer Script"I profiled slow queries using SQL Server execution plans, added covering indexes on frequently filtered columns, moved heavy reporting logic into stored procedures, and used Dapper for those hot paths instead of EF Core's overhead — together that cut query time by 40%."
Migrations
DB schema ah code la track pannurathukku EF Core migrations use pannுவோம். Add-Migration command puதிய column/table add pannumbodhu oru "migration file" create pannum, Update-Database adha actual DB ku apply pannum. Production la direct ah Migrate() app start aagumbodhu run pannama, script generate panni review panniட்டு deploy pannanum — adhu safe.
🗣️ Answer Script"I generate migrations locally, review the diff with the team before merging, and in production I generate a SQL script from the migration and run it as a controlled pipeline step rather than auto-migrating at app startup, to avoid accidental destructive changes."
AsNoTracking()
Read-only query (report kaanumbodhu, list kaatumbodhu) — EF Core adha entity ah change track panna vendam. AsNoTracking() podra bodhu, EF memory kammi use pannும், query fast aagum, edhukku na SaveChanges() panna poura entity illa.
🗣️ Answer Script"For any read-only GET endpoint I add AsNoTracking() — it skips the change-tracking snapshot EF Core normally keeps, reducing memory and improving query speed."
A4 Microservices & Ocelot API Gateway
Microservices Architecture
Oru periya application ah, small small independent services ah split pannurathu — example: StudentService, BillService, AttendanceService, TransportService — ovvoru service um separately deploy panna mudiyum, separately scale panna mudiyum. Namba SSMS/ZenCampus project structure exactly idhu thaan — StudentService, HostelService, TimeTableService ellame vera vera microservices.
🗣️ Answer Script"Instead of one large monolith, I split the ERP into independent services — student, billing, attendance, hostel, transport — each with its own database access and deployment. This let us deploy a fix to billing without touching or redeploying attendance."
Ocelot API Gateway
Namba kitta 10 microservices irundha, client (frontend) ovvoru service ku separate ah call panna koodathu — confuse aagum. Ocelot oru "gateway" — single entry point, client Ocelot ku mattum call pannும், Ocelot andha request ah correct microservice ku route pannும். Idhula load balancing (multiple instance kitta traffic split pannurathு) and rate limiting (ஒரு client over-use pannamal thadukkurathு) um configure pannalam.
🗣️ Answer Script"Ocelot sits in front of all my microservices as a single entry point. The frontend calls Ocelot's URL, and Ocelot routes the request to the right downstream service based on configured routes — I also used it for load balancing across instances and rate limiting to protect services from overload."
Inter-Service Communication
Microservices ku ஒன்றுக்கொன்று pesanum na — synchronous ah (HTTP/REST call — direct ah call panni immediate response wait pannுவது) illa asynchronous ah (message queue mூlama — call pannிட்டு andha service busy irundhaalum namba block aagama continue pannலாம்) pannலாம். Namba resume la explicit ah message queue mention pannala na, HTTP-based sync communication nu sollலாம்.
🗣️ Answer Script"In my project, services primarily communicate synchronously over HTTP through the Ocelot gateway. For scenarios needing decoupling — like OTP delivery — I'd reach for an async approach, but the core CRUD flows were request/response over REST."
Independent Deployment / Scalability
Microservices oda periya benefit — StudentService la traffic athigama irundha, adha mattum scale pannலாம் (extra instances add pannலாம்), vera services ah touch pannாம. Monolith la ஆனா — full application ah scale pannanum, waste of resources.
🗣️ Answer Script"Because each service is independently deployable, if StudentService gets heavy traffic during admission season, I can scale just that service horizontally without touching or redeploying the rest of the system."
Centralized Logging Across Microservices
Multiple services la error/exception varumbodhu, ovvoru service log separate ah irundha debug panna kashtam. So oru common middleware/logging library vachu, ellame ஒரே format la log pannuvom (correlation ID vachu — oru request full flow ah trace panna mudiyும்).
🗣️ Answer Script"I built shared logging and exception-handling middleware used consistently across all microservices, with a correlation ID passed through the gateway, so I could trace one request's journey across services when debugging a production issue — this is what reduced our incident resolution time by 35%."
A5 JWT / Security
JWT — JSON Web Token
User login pannumbodhu, server oru "token" kudukkும் — adhu la user oda info (userId, role, expiry time) encode aagi irukkும். Client andha token ah every request kூடa Authorization: Bearer <token> header la anுppுவார். Server token ah verify pannிட்டு, "sari idhu valid user thaan" nu confirm pannிட்டு access kudukkும். Token 3 parts — header.payload.signature — signature thaan tamper aagalaana nu confirm pannும்.
🗣️ Answer Script"When a user logs in, my API issues a signed JWT containing their identity and role claims. Every subsequent request carries that token in the Authorization header, and my JWT bearer middleware validates the signature, issuer, audience, and expiry before allowing access."
RBAC — Role Based Access Control
Namba resume la "JWT RBAC" nu irukku. Idhu na — token la irukka "role" claim (Admin, Teacher, Parent) vechu, "இந்த endpoint ah Admin mattum access panna mudiyும்" nu control pannuvathu. C# la [Authorize(Roles = "Admin")] attribute vechu easy ah implement pannலாம்.
🗣️ Answer Script"I implemented Role-Based Access Control by embedding role claims in the JWT at login, then decorating endpoints with [Authorize(Roles = "Admin")] or similar — so a Parent role can never hit an Admin-only endpoint even with a valid token."
OTP-based MFA (Multi-Factor Authentication)
Un bank locker project la — password mattum pothaadhu, oru OTP kuda verify pannanum nu security add pannirukeenga. Flow: user login → system oru random OTP generate pannும் → SMS gateway mூlama phone ku anுppும் → user OTP enter pannும் → server verify pannும் → access kodukkும். OTP oru limited time ku mattum valid ah irukkும் (example 2 minutes), expire aana apparam use panna mudியாது.
🗣️ Answer Script"For the bank locker system I built a two-factor flow — after password login, the system generates a time-limited OTP, delivers it via an SMS gateway API, and validates it server-side with an audit trail. This gave us zero unauthorized access incidents post-deployment."
SQL Injection Prevention
Namba user input ah direct ah SQL query string kூடa join panna, attacker adha misuse panni database ah hack pannலாம் (example: ' OR 1=1 --). Parameterized queries (EF Core, Dapper @param) use pannினா, input data ah mattum consider pannும், query structure ah maathாmல் — so injection sாத்தியமே இல்லை.
🗣️ Answer Script"I never concatenate user input into SQL strings — both EF Core and Dapper use parameterized queries by default, which treats input purely as data, not executable SQL, eliminating injection risk."
HTTPS / TLS
Data client-um server-um edhukkidaiyில் travel pannும்bodhu, adha encrypt pannிட்டு anுppுவathu thaan HTTPS. Idhu illna, edhavadhu middle-man (attacker) data ah padikkலாம் (password, token ellame). Namba resume la "TLS/HTTPS" mentioned — bank locker project la sensitive data (OTP, locker access) transfer aagும்bodhu idhu mandatory.
🗣️ Answer Script"All API traffic in the locker system runs over HTTPS/TLS, so OTPs and access tokens can't be intercepted in transit — critical for a security-sensitive system like a bank locker."
A6 Docker
Docker & Containerization
Namba code namba laptop la work aagும், aana server la deploy pannும்bodhu "works on my machine aana server la work aagல" problem varும். Docker idha fix pannும் — application-a, adhoda dependencies (runtime, libraries) ellame ஒரு "container" nu package pannிடுவோம். Andha container edhu machine la venalum run pannும் (same result).
🗣️ Answer Script"I containerized every microservice with Docker so each service runs with its exact dependencies bundled in — eliminating environment inconsistencies between my local machine, staging, and production."
Docker Compose
Namba single microservice ku single container pothum. Aana ZenCampus la 10+ microservices + DB — ellame ஒரே time la start pannanum na, ovvoru docker run command tanியா type pannுvathu kashtam. Docker Compose oru YAML file la ellame define pannிட்டு, docker-compose up ஒரே command la ellame start pannலாம்.
🗣️ Answer Script"I used Docker Compose to define all our microservices and their dependencies — like the database — in one YAML file, so the entire stack spins up with a single command, both locally and in CI."
Dockerfile
Container ah eppadi build pannanum nu step-by-step instructions vaikkும் file thaan Dockerfile. Example: base image select pannுvathு (.NET runtime), code copy pannுvathு, dependencies restore pannுvathு, build pannுvathு, final ah run pannுvathு eppadi nu ellame idhula irukும்.
🗣️ Answer Script"Each service has its own Dockerfile — starting from the official .NET SDK image for build, then a lighter runtime image for the final container to keep the image size small."
Docker Image vs Container
Image na — oru "blueprint"/template, code ellame freeze panni package pannirukka oru file. Container na — andha image ah actual ah run panniட்டு irukka oru "live instance". Same image use panni multiple containers run pannலாம்.
🗣️ Answer Script"An image is the packaged, immutable blueprint; a container is a running instance of that image. I can spin up multiple containers from the same image to scale a service horizontally."
A7 MongoDB & SQL Server
MongoDB — engа use pannalam, SQL Server engа use pannalam?
SQL Server na — structured data (fixed columns, relationships) ku perfect — example: Student, Fee, Attendance tables, relationships clear ah irukும். MongoDB na — unstructured/flexible schema data ku (document-based) — example: logs, notifications, real-time analytics data, edhukku na fields change aagலாம், strict schema venam. Un resume la "unstructured/high-volume data, real-time analytics" nu specific ah mentioned.
🗣️ Answer Script"I used SQL Server for structured, relational data like student, fee, and attendance records where integrity and relationships matter. MongoDB handled high-volume, flexible-schema data like activity logs and analytics events, where the document structure could evolve without a migration."
Indexing (SQL Server)
Index na — oru book oda "index page" madhiri, adha vecha fast ah data find pannலாம், full table ah scan pannாma. Frequently search/filter/join pannра columns ku index poduvom (example: StudentId, Email). Aana over-indexing pannினா, INSERT/UPDATE speed kammi aagும் — so balance venum.
🗣️ Answer Script"I add indexes on columns frequently used in WHERE, JOIN, or ORDER BY clauses to avoid full table scans — but I'm careful not to over-index, since every extra index slows down inserts and updates."
Stored Procedures
Complex SQL logic ah, application code la vaikkாma, database level la ஒரு "procedure" ah define pannிட்டு, adha call pannுvathு. Benefits: reusable, DB level la execute aagும் (fast), security (direct table access illama procedure mூlama மட்டும் access).
🗣️ Answer Script"For heavy reporting queries — like fee collection summaries — I moved the logic into stored procedures so it executes close to the data, and application code just calls it with parameters instead of building complex SQL client-side."
Normalization (Database Design)
Data ah duplicate aagாma, related tables ah split pannுvathு thaan normalization. Example: Student table la Course name ah repeat panna, veraaga Course table create pannிட்டு, Student table la CourseId mattum reference pannுவோம் (foreign key). Idhu vachu data consistency maintain aagும்.
🗣️ Answer Script"I normalize the schema to avoid data duplication — for example, course details live in a separate Course table referenced by CourseId from Student, so updating a course name doesn't require touching thousands of student rows."
A8 Azure Cloud
Azure App Service
.NET Web API ah host pannுvathukku Azure kudukkும் oru managed hosting service. Server manage panna வேண்டியதில்லை (patching, OS updates) — Microsoft thaan handle pannும், நாம் code deploy panna மட்டும் paakanum.
🗣️ Answer Script"For hosting a .NET Web API in Azure, App Service is the go-to — it's a managed platform, so I don't manage the underlying OS or patching, just deploy and configure the app."
Azure Key Vault
Connection strings, API keys, passwords — ellame code/config file la direct ah vaikkаma, Key Vault nu ஒரு secure place la vechுக்கலாம். App ku "Managed Identity" கொடுத்து, adha vachு Key Vault access pannலாம் — credential edhுவும் store panna தேவையில்லை.
🗣️ Answer Script"Sensitive values like connection strings and API keys go into Azure Key Vault, accessed through a Managed Identity — so no secret ever sits in appsettings.json or gets committed to source control."
Deployment Slots
Production ku direct ah deploy panna risky. Adhaanaаla — oru "staging" slot la first deploy panni, test panniட்டு, sari na "swap" panни production ku matturathu. Zero downtime — traffic instant ah switch aagும், problem irundha உடனே திரும்ப swap pannலாம்.
🗣️ Answer Script"I deploy to a staging slot first, run smoke tests there, and then swap it into production — that gives zero-downtime deployment, and if something's wrong I can swap back instantly."
Application Insights
Production la app eppadi perform pannுthu nu monitor pannуவதுкки use pannுром. Errors, slow requests, dependency calls (DB call evlo time eduthuchu) ellame track pannலாம். Production la ஒரு bug வந்தா, idhு வெச்சு root cause fast ah kandupidikkலாம்.
🗣️ Answer Script"Application Insights tracks request latency, failure rates, and dependency calls in production — when there's an incident, I use the end-to-end transaction view to trace exactly which downstream call failed, instead of guessing from logs."
